# 영구 메시지 저장소

> 서버가 꺼져도 대화가 사라지지 않게 — Redis, SQL, MongoDB, 파일 기반 영구 저장소로 대화 히스토리를 보존하는 방법을 배웁니다.

## 개요

이 섹션에서는 [세션 10.1: 메시지 히스토리 기초](ch10/session_10_1.md)에서 배운 `InMemoryChatMessageHistory`의 한계를 넘어, 대화 히스토리를 영구 저장소(Persistent Storage)에 보관하는 방법을 배웁니다. [세션 10.2: RunnableWithMessageHistory](ch10/session_10_2.md)에서 익힌 `get_session_history` 팩토리 함수를 교체하는 것만으로 다양한 백엔드를 연동할 수 있다는 점이 핵심입니다.

**선수 지식**: `BaseChatMessageHistory` 인터페이스, `RunnableWithMessageHistory`의 Load-Run-Save 패턴, `session_id` 기반 멀티 세션 관리
**학습 목표**:
- 파일, SQLite/PostgreSQL, Redis, MongoDB 네 가지 영구 저장소의 특성과 적용 시나리오를 이해한다
- 각 저장소별 `ChatMessageHistory` 구현체를 설정하고 `RunnableWithMessageHistory`와 연동할 수 있다
- 프로덕션 환경에서 저장소 선택 기준을 판단할 수 있다

## 왜 알아야 할까?

여러분이 열심히 만든 AI 챗봇을 상상해 보세요. 사용자가 30분 동안 깊은 대화를 나눴는데, 서버를 재시작하니 모든 대화가 사라져 버립니다. "아까 말했던 그거 기억나?" 하는 질문에 "죄송합니다, 이전 대화 내용을 찾을 수 없습니다"라고 답한다면요? 사용자 경험은 바닥을 치겠죠.

`InMemoryChatMessageHistory`는 프로토타이핑에는 훌륭하지만, **프로세스가 종료되면 모든 데이터가 날아갑니다**. 실제 서비스에서는 다음과 같은 요구사항이 생기거든요:

- **서버 재시작** 후에도 대화를 이어가야 한다
- **여러 서버 인스턴스**가 동일한 대화 히스토리에 접근해야 한다
- 대화 데이터를 **분석하거나 감사(audit)**해야 한다
- 오래된 대화는 **자동 만료(TTL)**시켜야 한다

LangChain은 `BaseChatMessageHistory`라는 통일된 인터페이스 덕분에, 저장소를 교체할 때 **팩토리 함수 한 줄만 바꾸면** 됩니다. 이번 섹션에서 그 실전 방법을 모두 다뤄보겠습니다.

## 핵심 개념

### 개념 1: 영구 저장소의 구조 — 옷장 비유

> 💡 **비유**: `InMemoryChatMessageHistory`가 책상 위에 올려둔 메모지라면, 영구 저장소는 **옷장 서랍**과 같습니다. 메모지는 바람이 불면 날아가지만(프로세스 종료), 옷장 서랍에 넣어둔 메모는 집을 비워도 그대로 남아 있죠. Redis는 빠르게 여닫을 수 있는 **서랍장**, SQL 데이터베이스는 꼼꼼하게 정리된 **파일 캐비닛**, MongoDB는 유연한 크기의 **수납 박스**, 파일 시스템은 가장 단순한 **봉투**에 비유할 수 있습니다.

모든 영구 저장소 구현체는 동일한 `BaseChatMessageHistory` 인터페이스를 따릅니다. 앞서 [세션 10.1](ch10/session_10_1.md)에서 배운 것처럼, `add_message()`, `messages`, `clear()` — 이 세 가지 핵심 메서드가 어떤 저장소든 동일하게 동작합니다. 덕분에 [세션 10.2](ch10/session_10_2.md)의 `RunnableWithMessageHistory`에서 `get_session_history` 함수만 교체하면 저장소를 자유롭게 바꿀 수 있는 거죠.

```
┌─────────────────────────────────────────────┐
│        RunnableWithMessageHistory           │
│  ┌───────────────────────────────────────┐  │
│  │  get_session_history(session_id)      │  │
│  │       ↓ 팩토리 함수 교체만으로 전환     │  │
│  └───────────────────────────────────────┘  │
│       ↓              ↓            ↓         │
│  ┌────────┐   ┌──────────┐  ┌─────────┐   │
│  │ Redis  │   │ SQL(ite) │  │ MongoDB │   │
│  │ 저장소  │   │ 저장소    │  │ 저장소   │   │
│  └────────┘   └──────────┘  └─────────┘   │
└─────────────────────────────────────────────┘
```

### 개념 2: FileChatMessageHistory — 가장 단순한 영구 저장소

> 💡 **비유**: 대화 내용을 노트에 적어서 책장에 꽂아두는 것과 같습니다. 별도의 장비가 필요 없고, 노트(JSON 파일)만 있으면 됩니다.

`FileChatMessageHistory`는 대화 히스토리를 **로컬 JSON 파일**에 저장하는 가장 단순한 구현체입니다. 외부 데이터베이스가 전혀 필요 없기 때문에 개발 환경이나 단일 사용자 애플리케이션에 적합합니다.

```python
# pip install langchain-community

from langchain_community.chat_message_histories import FileChatMessageHistory

# 세션별로 별도의 JSON 파일에 저장
history = FileChatMessageHistory(file_path="./chat_histories/user_123.json")

# 메시지 추가 — 파일에 자동으로 기록됨
history.add_user_message("LangChain에서 메모리는 어떻게 동작하나요?")
history.add_ai_message("LangChain의 메모리는 BaseChatMessageHistory 인터페이스를 통해...")

# 저장된 메시지 조회
print(history.messages)
# [HumanMessage(content='LangChain에서 메모리는 어떻게 동작하나요?'),
#  AIMessage(content='LangChain의 메모리는 BaseChatMessageHistory 인터페이스를 통해...')]

# 히스토리 초기화 (파일 내용 삭제)
history.clear()
```

`RunnableWithMessageHistory`와 연동할 때는 `get_session_history` 팩토리 함수만 교체하면 됩니다:

```python
from pathlib import Path
from langchain_community.chat_message_histories import FileChatMessageHistory
from langchain_core.runnables.history import RunnableWithMessageHistory
from langchain_core.prompts import ChatPromptTemplate, MessagesPlaceholder
from langchain_openai import ChatOpenAI

# 저장 디렉토리 생성
HISTORY_DIR = Path("./chat_histories")
HISTORY_DIR.mkdir(exist_ok=True)

# 팩토리 함수: session_id → FileChatMessageHistory
def get_session_history(session_id: str) -> FileChatMessageHistory:
    file_path = HISTORY_DIR / f"{session_id}.json"
    return FileChatMessageHistory(str(file_path))

# 체인 구성
prompt = ChatPromptTemplate.from_messages([
    ("system", "당신은 친절한 AI 어시스턴트입니다."),
    MessagesPlaceholder(variable_name="history"),
    ("human", "{input}"),
])
chain = prompt | ChatOpenAI(model="gpt-4o")

# 히스토리 자동 관리 래퍼 적용
chain_with_history = RunnableWithMessageHistory(
    chain,
    get_session_history,               # ← 이 함수만 교체하면 저장소 변경!
    input_messages_key="input",
    history_messages_key="history",
)
```

> ⚠️ **흔한 오해**: FileChatMessageHistory는 파일 잠금(file locking)을 지원하지 않습니다. 여러 프로세스가 동시에 같은 파일에 쓰면 데이터가 깨질 수 있어요. **단일 프로세스** 환경에서만 사용하세요.

### 개념 3: SQLChatMessageHistory — SQL 데이터베이스 연동

> 💡 **비유**: 파일이 노트라면, SQL 데이터베이스는 **도서관의 카탈로그 시스템**입니다. 목록이 체계적으로 정리되어 있어서 "3일 전 대화"나 "특정 사용자의 모든 대화"를 빠르게 찾을 수 있죠.

`SQLChatMessageHistory`는 **SQLAlchemy**를 통해 SQLite, PostgreSQL, MySQL 등 다양한 관계형 데이터베이스에 대화를 저장합니다. 별도 서버 설치 없이 SQLite 파일 하나로 시작할 수 있어서 프로토타입에서 프로덕션까지 점진적으로 확장하기에 좋습니다.

```python
# pip install langchain-community sqlalchemy

from langchain_community.chat_message_histories import SQLChatMessageHistory

# SQLite — 파일 하나로 바로 시작 (서버 불필요)
history = SQLChatMessageHistory(
    session_id="user_123",
    connection="sqlite:///chat_memory.db",   # SQLAlchemy 연결 문자열
    table_name="message_store",              # 테이블 이름 (기본값: message_store)
)

# 메시지 추가
history.add_user_message("RAG가 뭔가요?")
history.add_ai_message("RAG는 Retrieval-Augmented Generation의 약자로...")

# 저장된 메시지 조회
print(history.messages)
# [HumanMessage(content='RAG가 뭔가요?'),
#  AIMessage(content='RAG는 Retrieval-Augmented Generation의 약자로...')]
```

PostgreSQL로 전환할 때는 **연결 문자열만 바꾸면** 됩니다:

```python
# PostgreSQL 연결 (별도 패키지: pip install psycopg2-binary)
history = SQLChatMessageHistory(
    session_id="user_123",
    connection="postgresql://user:password@localhost:5432/chatdb",
    table_name="message_store",
)
```

비동기(async) 환경에서는 `create_async_engine`을 활용합니다:

```python
# pip install aiosqlite
from sqlalchemy.ext.asyncio import create_async_engine

async_engine = create_async_engine("sqlite+aiosqlite:///chat_memory.db")

async_history = SQLChatMessageHistory(
    session_id="user_123",
    connection=async_engine,
)

# 비동기 메서드 사용
await async_history.aadd_message(HumanMessage(content="비동기로 저장합니다"))
messages = await async_history.aget_messages()
```

### 개념 4: RedisChatMessageHistory — 초고속 인메모리 영구 저장소

> 💡 **비유**: Redis는 **냉장고 문에 붙인 자석 메모판**과 같습니다. 눈에 바로 보이고(초고속 읽기/쓰기), 메모를 붙이고 떼기가 쉽고(TTL 자동 만료), 가족 모두가 동시에 볼 수 있죠(멀티 인스턴스 공유). 단, 냉장고를 버리면 메모도 사라질 수 있으니 주기적으로 사진을 찍어두는 것(RDB/AOF 영속화)이 중요합니다.

Redis는 인메모리 데이터 스토어이면서도 디스크 영속화를 지원하는 독특한 위치의 데이터베이스입니다. LangChain에서는 두 가지 패키지가 있는데, 최신 공식 파트너 패키지인 `langchain-redis`를 사용하는 것이 권장됩니다.

```python
# 최신 공식 패키지 설치
# pip install langchain-redis

from langchain_redis import RedisChatMessageHistory

history = RedisChatMessageHistory(
    session_id="user_123",
    redis_url="redis://localhost:6379",
    ttl=3600,  # 1시간 후 자동 만료 (초 단위, 선택사항)
)

# 메시지 추가
history.add_user_message("LangGraph란 뭔가요?")
history.add_ai_message("LangGraph는 상태 기반 그래프를 사용하여...")

# 저장된 메시지 조회
print(history.messages)

# 히스토리 삭제
history.clear()
```

TTL(Time-To-Live) 기능은 Redis만의 강력한 장점입니다. 대화가 일정 시간 후 자동으로 만료되므로, 임시 고객 상담이나 세션 기반 서비스에 특히 유용합니다.

```python
# TTL 활용 예: 고객 상담 시스템
# 상담 종료 후 24시간이 지나면 자동 삭제
support_history = RedisChatMessageHistory(
    session_id="support_ticket_456",
    redis_url="redis://localhost:6379",
    ttl=86400,  # 24시간 = 86,400초
)
```

### 개념 5: MongoDBChatMessageHistory — 유연한 문서 기반 저장소

> 💡 **비유**: MongoDB는 **만능 수납 박스**와 같습니다. 서랍 크기가 정해진 파일 캐비닛(SQL)과 달리, 어떤 모양이든 들어가는 유연한 수납함이죠. 대화 메시지에 메타데이터(사용자 감정 분석 결과, 태그 등)를 자유롭게 붙이고 싶을 때 빛을 발합니다.

MongoDB는 JSON과 유사한 BSON 형식으로 데이터를 저장하는 문서 지향 데이터베이스입니다. 스키마가 유연하기 때문에 메시지에 추가 메타데이터를 자유롭게 첨부할 수 있습니다.

```python
# pip install langchain-mongodb pymongo

from langchain_mongodb.chat_message_histories import MongoDBChatMessageHistory

history = MongoDBChatMessageHistory(
    session_id="user_123",
    connection_string="mongodb://localhost:27017",
    database_name="chatbot_db",
    collection_name="chat_histories",
)

# 메시지 추가
history.add_user_message("에이전트와 체인의 차이점이 뭔가요?")
history.add_ai_message("체인은 미리 정해진 순서대로 실행되지만, 에이전트는...")

# 최근 대화만 가져오기 (history_size 파라미터)
# MongoDBChatMessageHistory는 history_size 옵션을 지원합니다
recent_history = MongoDBChatMessageHistory(
    session_id="user_123",
    connection_string="mongodb://localhost:27017",
    database_name="chatbot_db",
    collection_name="chat_histories",
    history_size=10,  # 최근 10개 메시지만 로드
)
```

### 개념 6: 저장소 선택 가이드 — 어떤 상황에 뭘 써야 할까?

어떤 저장소를 선택할지는 프로젝트의 규모, 성능 요구사항, 인프라 환경에 따라 달라집니다:

| 저장소 | 설치 복잡도 | 속도 | 확장성 | TTL | 추천 시나리오 |
|--------|-----------|------|-------|-----|-------------|
| **File** | ⭐ (없음) | 느림 | 단일 프로세스 | ✗ | 개발/프로토타입, 단일 사용자 |
| **SQLite** | ⭐ (없음) | 보통 | 단일 서버 | ✗ | 소규모 서비스, 데이터 분석 필요 |
| **PostgreSQL** | ⭐⭐⭐ | 보통 | 멀티 서버 | ✗ (직접 구현) | 중대규모 서비스, 트랜잭션 중요 |
| **Redis** | ⭐⭐ | 매우 빠름 | 멀티 서버 | ✓ (내장) | 실시간 채팅, 높은 동시성 |
| **MongoDB** | ⭐⭐ | 빠름 | 멀티 서버 | ✓ (TTL 인덱스) | 유연한 스키마, 메타데이터 풍부 |

> 🔥 **실무 팁**: "뭘 써야 할지 모르겠다"면, **SQLite로 시작**하세요. 코드 한 줄도 안 바꾸고 나중에 PostgreSQL로 마이그레이션할 수 있고, 별도 서버 설치도 필요 없습니다. Redis는 "수천 명이 동시에 채팅"하는 상황이 되었을 때 도입해도 늦지 않습니다.

## 실습: 직접 해보기

아래 코드는 **하나의 체인에 저장소만 교체하며** 4가지 백엔드를 모두 실험하는 완전한 예제입니다. `get_session_history` 함수 하나만 바꾸면 저장소가 전환되는 LangChain의 설계 철학을 직접 체험해 보세요.

```python
"""
영구 메시지 저장소 실습
- 4가지 저장소 백엔드를 교체하며 동일한 체인에 연결
- 필요 패키지:
  pip install langchain-core langchain-openai langchain-community sqlalchemy
  pip install langchain-redis  # Redis 사용 시
  pip install langchain-mongodb pymongo  # MongoDB 사용 시
"""

import os
from pathlib import Path
from dotenv import load_dotenv

from langchain_core.prompts import ChatPromptTemplate, MessagesPlaceholder
from langchain_core.runnables.history import RunnableWithMessageHistory
from langchain_openai import ChatOpenAI

load_dotenv()  # .env 파일에서 OPENAI_API_KEY 로드

# ──────────────────────────────────────
# 1단계: 공통 체인 구성 (저장소와 무관)
# ──────────────────────────────────────
prompt = ChatPromptTemplate.from_messages([
    ("system", "당신은 LangChain 전문가입니다. 친절하게 답변하세요."),
    MessagesPlaceholder(variable_name="history"),  # 히스토리 삽입 위치
    ("human", "{input}"),
])
llm = ChatOpenAI(model="gpt-4o", temperature=0.7)
chain = prompt | llm


# ──────────────────────────────────────
# 2단계: 저장소별 팩토리 함수 정의
# ──────────────────────────────────────

# (A) 파일 기반 저장소
def get_file_history(session_id: str):
    """JSON 파일에 대화를 저장합니다."""
    from langchain_community.chat_message_histories import FileChatMessageHistory
    history_dir = Path("./chat_histories")
    history_dir.mkdir(exist_ok=True)
    return FileChatMessageHistory(str(history_dir / f"{session_id}.json"))


# (B) SQLite 저장소
def get_sqlite_history(session_id: str):
    """SQLite 데이터베이스에 대화를 저장합니다."""
    from langchain_community.chat_message_histories import SQLChatMessageHistory
    return SQLChatMessageHistory(
        session_id=session_id,
        connection="sqlite:///chat_memory.db",
        table_name="message_store",
    )


# (C) Redis 저장소 (Redis 서버 필요)
def get_redis_history(session_id: str):
```

## 더 깊이 알아보기

### Redis의 탄생 — 이탈리아 개발자의 "진짜 빠른 목록"

Redis는 2009년 이탈리아 개발자 **살바토레 산필리포(Salvatore Sanfilippo)**가 만들었습니다. 당시 그는 실시간 웹 로그 분석 스타트업을 운영하고 있었는데, MySQL로는 초당 수천 건의 쓰기 작업을 감당할 수 없었거든요. "디스크에 쓰지 않고 메모리에만 저장하면 얼마나 빠를까?"라는 단순한 질문에서 Redis(**RE**mote **DI**ctionary **S**erver)가 탄생했습니다.

처음에는 정말 단순한 키-값 저장소였지만, 리스트, 해시, 집합 같은 자료구조가 추가되면서 "데이터 구조 서버"로 진화했습니다. 채팅 메시지를 시간순 리스트로 저장하는 것은 Redis의 가장 자연스러운 사용 사례 중 하나인데요, LangChain의 `RedisChatMessageHistory`가 바로 이 패턴을 활용합니다.

흥미로운 점은 Redis의 이름에서 "Remote"가 붙은 이유입니다. 산필리포는 처음부터 여러 애플리케이션이 네트워크를 통해 공유하는 중앙 저장소를 구상했거든요. 이것이 바로 오늘날 멀티 서버 환경에서 Redis가 세션 저장소로 각광받는 이유이기도 합니다.

### SQLAlchemy — "관계형 DB를 파이썬답게"

`SQLChatMessageHistory`의 핵심에는 **SQLAlchemy**가 있습니다. 2006년 마이클 바이어(Michael Bayer)가 만든 이 라이브러리는 "SQL을 감추자"가 아니라 "SQL을 파이썬답게 표현하자"는 철학으로 설계되었습니다. 덕분에 LangChain은 연결 문자열(`sqlite:///`, `postgresql://` 등)만 바꾸면 어떤 관계형 데이터베이스든 지원할 수 있는 거죠. 하나의 코드베이스로 SQLite에서 개발하고 PostgreSQL로 배포하는 것이 가능한 이유입니다.

## 흔한 오해와 팁

> ⚠️ **흔한 오해**: "Redis는 메모리에만 저장하니까 영구 저장소가 아니다?" — 틀렸습니다! Redis는 **RDB 스냅샷**과 **AOF(Append Only File)** 두 가지 디스크 영속화 전략을 제공합니다. 적절히 설정하면 서버 재시작 후에도 데이터가 보존됩니다. 다만 기본 설정에서는 일부 데이터 손실이 가능하므로, 프로덕션에서는 반드시 영속화 설정을 확인하세요.

> 💡 **알고 계셨나요?**: LangChain의 모든 `ChatMessageHistory` 구현체는 `BaseChatMessageHistory`의 `add_messages()`(복수형) 메서드를 오버라이드하면 **벌크 삽입 최적화**를 할 수 있습니다. 예를 들어 MongoDB의 `insertMany()`나 Redis의 파이프라인을 활용하면 수십 개 메시지를 한 번의 네트워크 왕복으로 저장할 수 있어요. 공식 문서에서도 대량 메시지 추가 시 `add_messages()` 인터페이스를 권장합니다.

> 🔥 **실무 팁**: 프로덕션에서는 `get_session_history` 팩토리 함수 안에서 **커넥션 풀링**을 꼭 고려하세요. 매번 새 연결을 만들면 성능이 급격히 떨어집니다. SQLAlchemy의 `create_engine(pool_size=10)`, Redis의 `ConnectionPool`, MongoDB의 `MongoClient` 내장 풀링을 활용하면 됩니다. 팩토리 함수 밖에서 엔진/클라이언트를 한 번만 생성하고 재사용하는 패턴이 핵심입니다:
>
> ```python
> from sqlalchemy import create_engine
> # 모듈 레벨에서 한 번만 생성 (커넥션 풀 포함)
> engine = create_engine("sqlite:///chat.db", pool_size=5)
>
> def get_session_history(session_id: str):
>     return SQLChatMessageHistory(session_id=session_id, connection=engine)
> ```

## 핵심 정리

| 개념 | 설명 |
|------|------|
| `FileChatMessageHistory` | JSON 파일에 대화 저장. 설치 불필요, 단일 프로세스용 |
| `SQLChatMessageHistory` | SQLAlchemy 기반. SQLite~PostgreSQL까지 연결 문자열만 변경 |
| `RedisChatMessageHistory` | 초고속 인메모리 저장. TTL 내장, 멀티 서버 공유 가능 |
| `MongoDBChatMessageHistory` | 문서 DB 기반. 유연한 스키마, `history_size`로 최근 N개만 로드 |
| `get_session_history` 교체 | `RunnableWithMessageHistory`의 팩토리 함수만 바꾸면 저장소 전환 |
| TTL (Time-To-Live) | Redis/MongoDB에서 메시지 자동 만료 설정 가능 |
| 커넥션 풀링 | 프로덕션에서 엔진/클라이언트를 모듈 레벨에서 재사용하여 성능 확보 |
