# 메모리 최적화 전략

> 대화가 길어져도 토큰 폭발 없이 핵심 맥락을 유지하는 실전 메모리 관리 기법

## 개요

이 섹션에서는 대화 히스토리가 무한히 쌓일 때 발생하는 토큰 초과 문제를 해결하는 세 가지 핵심 전략을 학습합니다. `trim_messages`를 활용한 정밀 트리밍, 슬라이딩 윈도우 패턴, 그리고 LLM 기반 대화 요약까지 — 프로덕션 챗봇이 안정적으로 동작하기 위한 필수 기법들을 다룹니다.

[세션 9.3: 대화형 RAG](09-ragretrieval-augmented-generation-구축/03-대화형-rag.md)에서 RAG 맥락의 컨텍스트 윈도우 관리를 간략히 소개했었죠. 이 섹션에서는 RAG에 국한되지 않는 **범용적 메모리 최적화 기법**으로서 각 전략의 내부 동작, 세밀한 파라미터 튜닝, 그리고 전략 간 조합 패턴까지 본격적으로 파고듭니다.

**선수 지식**: [세션 10.1: 메시지 히스토리 기초](./01-메시지-히스토리-기초.md)에서 배운 `BaseChatMessageHistory`와 메시지 생명주기, [세션 10.2: RunnableWithMessageHistory](./02-runnablewithmessagehistory.md)에서 배운 Load-Run-Save 패턴, [세션 10.3: 영구 메시지 저장소](./03-영구-메시지-저장소.md)에서 배운 다양한 저장소 연동 방법

**학습 목표**:
- `trim_messages`의 전략(strategy), 토큰 카운터, 시스템 메시지 보존 옵션을 이해하고 활용할 수 있다
- 슬라이딩 윈도우 패턴으로 최근 N개 대화만 유지하는 체인을 구성할 수 있다
- LLM 기반 대화 요약으로 장기 맥락을 압축하는 패턴을 구현할 수 있다
- 상황에 맞는 최적의 메모리 전략을 선택하고 조합할 수 있다

## 왜 알아야 할까?

실제 서비스에서 돌아가는 챗봇을 떠올려 보세요. 고객 상담 봇이 하루 종일 수백 턴의 대화를 이어간다면 어떻게 될까요? GPT-4o의 컨텍스트 윈도우가 128K 토큰이라 해도, 대화 히스토리를 그대로 쌓으면 결국 한계에 도달합니다. 그리고 한계에 도달하기 전에도, 불필요하게 긴 히스토리는 **응답 속도 저하**, **비용 증가**, **"바늘 찾기(needle-in-a-haystack)" 성능 저하**를 일으킵니다.

앞서 [세션 10.3: 영구 메시지 저장소](./03-영구-메시지-저장소.md)에서 대화를 안전하게 *저장*하는 방법은 배웠죠. 하지만 저장하는 것과 LLM에 *전달*하는 것은 완전히 다른 문제입니다. 모든 히스토리를 저장소에 보존하되, LLM에는 꼭 필요한 만큼만 전달하는 전략이 필요합니다. 이것이 바로 메모리 최적화의 핵심이에요.

[세션 9.3: 대화형 RAG](09-ragretrieval-augmented-generation-구축/03-대화형-rag.md)에서 RAG 파이프라인 안에서 히스토리를 간단히 관리하는 방법을 맛보기로 다뤘는데요, 거기서는 검색된 문서와 히스토리가 경쟁하는 RAG 특유의 상황에 초점을 맞췄습니다. 이번 섹션에서는 RAG 여부와 무관하게 **모든 대화형 LLM 애플리케이션**에 적용할 수 있는 범용 전략을 깊이 있게 다룹니다. LangChain이 공식 제공하는 `trim_messages` 유틸리티의 모든 파라미터를 파헤치고, 실전에서 가장 많이 쓰이는 하이브리드 패턴까지 완성해 보겠습니다.

## 핵심 개념

### 개념 1: trim_messages — 정밀한 토큰 가위

> 💡 **비유**: 여행 가방에 짐을 쌀 때를 생각해 보세요. 가방 용량(토큰 한도)은 정해져 있고, 가장 최근에 입은 옷(최근 대화)을 넣되, 여권(시스템 메시지)은 반드시 챙기고, 필요하면 옷을 반으로 접어서(partial split) 넣을 수도 있습니다. `trim_messages`가 바로 이 똑똑한 짐 싸기 도우미입니다.

Ch9.3에서는 `tiktoken`을 직접 사용하거나 리스트 슬라이싱으로 히스토리를 수동 관리했죠. `langchain_core`에서 제공하는 `trim_messages`는 이런 수동 작업을 **선언적 API 하나로 대체**하는 유틸리티 함수입니다. LangChain 0.2에서 도입되어, 기존의 `ConversationBufferWindowMemory`나 `ConversationTokenBufferMemory`를 대체하는 **현대적 접근 방식**이죠.

```python
from langchain_core.messages import (
    SystemMessage,
    HumanMessage,
    AIMessage,
    trim_messages,
)
from langchain_openai import ChatOpenAI

# 대화 히스토리 예시
messages = [
    SystemMessage(content="당신은 친절한 한국어 AI 어시스턴트입니다."),
    HumanMessage(content="안녕하세요! 파이썬에 대해 알려주세요."),
    AIMessage(content="안녕하세요! 파이썬은 1991년 귀도 반 로섬이 만든 프로그래밍 언어입니다..."),
    HumanMessage(content="파이썬의 장점은 뭔가요?"),
    AIMessage(content="파이썬의 주요 장점은 간결한 문법, 풍부한 라이브러리, 활발한 커뮤니티입니다..."),
    HumanMessage(content="LangChain은 어떤 건가요?"),
    AIMessage(content="LangChain은 LLM 기반 애플리케이션 개발 프레임워크입니다..."),
    HumanMessage(content="LCEL이 뭔지 설명해주세요."),
]

# 최근 대화만 유지하되, 시스템 메시지는 보존
trimmed = trim_messages(
    messages,
    max_tokens=200,                        # 최대 200토큰
    token_counter=ChatOpenAI(model="gpt-4o"),  # 실제 모델로 토큰 계산
    strategy="last",                       # 최근 메시지부터 보존
    include_system=True,                   # 시스템 메시지는 항상 유지
    start_on="human",                      # Human 메시지부터 시작하도록 보장
    allow_partial=False,                   # 메시지를 중간에 자르지 않음
)

for msg in trimmed:
    print(f"[{msg.type}] {msg.content[:50]}...")
# [system] 당신은 친절한 한국어 AI 어시스턴트입니다....
# [human] LCEL이 뭔지 설명해주세요....
```

**핵심 파라미터를 하나씩 살펴볼까요?**

| 파라미터 | 타입 | 기본값 | 설명 |
|----------|------|--------|------|
| `max_tokens` | `int` | 필수 | 트리밍 후 메시지의 최대 토큰 수 |
| `token_counter` | `Callable \| LLM \| "approximate"` | 필수 | 토큰 수를 세는 방법 |
| `strategy` | `"first" \| "last"` | `"last"` | 앞부터 유지 vs 뒤부터 유지 |
| `include_system` | `bool` | `False` | 인덱스 0의 SystemMessage 보존 여부 |
| `start_on` | `str \| type` | `None` | 트리밍 결과가 이 타입으로 시작하도록 보장 |
| `allow_partial` | `bool` | `False` | 메시지를 부분적으로 잘라도 되는지 여부 |

`token_counter`에는 세 가지를 전달할 수 있습니다:

```python
# 1. LLM 인스턴스 — 가장 정확한 토큰 계산
trimmed = trim_messages(messages, max_tokens=100,
    token_counter=ChatOpenAI(model="gpt-4o"), strategy="last")

# 2. "approximate" 문자열 — API 호출 없이 근사 계산 (빠르고 무료)
trimmed = trim_messages(messages, max_tokens=100,
    token_counter="approximate", strategy="last")

# 3. 커스텀 함수 — 완전한 제어
def count_tokens(msgs: list) -> int:
    """단순하게 문자 수 / 4로 토큰 수 추정"""
    return sum(len(m.content) // 4 for m in msgs)

trimmed = trim_messages(messages, max_tokens=100,
    token_counter=count_tokens, strategy="last")
```

> 🔥 **실무 팁**: 프로덕션에서는 `"approximate"`를 기본으로 사용하고, 정밀 제어가 필요한 경우에만 LLM 인스턴스를 전달하세요. `"approximate"` 모드는 API 호출 없이 토큰을 추정하므로 비용과 지연이 발생하지 않습니다. Ch9.3에서 `tiktoken`을 직접 사용하는 패턴을 봤는데, `trim_messages`가 이 작업을 내부적으로 처리해주므로 직접 토큰을 세는 코드를 작성할 필요가 없습니다.

### 개념 2: LCEL 체인에 trim_messages 통합하기

> 💡 **비유**: 수도 파이프에 정수 필터를 끼우는 것과 같아요. 물(메시지)이 흘러가는 파이프(체인)의 중간에 필터(trim_messages)를 장착하면, 불순물(오래된 대화)이 자동으로 걸러져서 깨끗한 물(최적화된 컨텍스트)만 목적지(LLM)에 도달합니다.

`trim_messages`는 Runnable이기도 해서, LCEL 파이프 연산자(`|`)와 바로 조합할 수 있습니다. [세션 10.2](./02-runnablewithmessagehistory.md)에서 배운 `RunnableWithMessageHistory`의 Load-Run-Save 패턴과 함께 사용하면 강력한 조합이 됩니다. 여기서는 `RunnableWithMessageHistory`의 기본 사용법은 이미 익혔다고 가정하고, **trim_messages를 체인에 끼워 넣는 통합 패턴**에 집중하겠습니다.

```python
from langchain_core.messages import trim_messages, SystemMessage
from langchain_core.prompts import ChatPromptTemplate, MessagesPlaceholder
from langchain_core.chat_history import InMemoryChatMessageHistory
from langchain_core.runnables.history import RunnableWithMessageHistory
from langchain_openai import ChatOpenAI

# 1. 모델과 트리머 설정
model = ChatOpenAI(model="gpt-4o", temperature=0.7)
trimmer = trim_messages(
    max_tokens=1000,
    strategy="last",
    token_counter="approximate",
    include_system=True,
    start_on="human",
)

# 2. 프롬프트 템플릿
prompt = ChatPromptTemplate.from_messages([
    ("system", "당신은 친절한 한국어 AI 어시스턴트입니다."),
    MessagesPlaceholder(variable_name="history"),
    ("human", "{input}"),
])

# 3. LCEL 체인: 트리머를 히스토리 앞에 배치
chain = prompt | trimmer | model

# 4. 세션 관리 (10.2에서 배운 패턴)
store = {}

def get_session_history(session_id: str):
    if session_id not in store:
        store[session_id] = InMemoryChatMessageHistory()
    return store[session_id]

# 5. RunnableWithMessageHistory로 감싸기
chain_with_history = RunnableWithMessageHistory(
    chain,
    get_session_history,
    input_messages_key="input",
    history_messages_key="history",
)

# 대화가 아무리 길어져도 1000토큰 이내로 트리밍됨
config = {"configurable": {"session_id": "user-123"}}
response = chain_with_history.invoke(
    {"input": "LangChain 메모리에 대해 알려주세요"},
    config=config,
)
print(response.content)
```

이 패턴의 핵심은 **저장소에는 전체 히스토리가 보존**되지만, **LLM에는 트리밍된 히스토리만 전달**된다는 것입니다. Ch9.3에서 수동으로 `messages[-(max_turns * 2):]` 슬라이싱을 하거나 `tiktoken`으로 직접 토큰을 세며 원본 히스토리를 변경했던 것과 달리, 이 접근법은 저장소의 원본 데이터를 전혀 건드리지 않습니다. 사용자가 이전 대화를 다시 볼 수 있으면서도, LLM은 토큰 한도 내에서 효율적으로 동작하죠.

### 개념 3: 슬라이딩 윈도우 — 최근 N턴만 유지

> 💡 **비유**: 엘리베이터에 정원이 있는 것과 같습니다. 새 사람(메시지)이 타면 가장 먼저 탄 사람이 내려야 하죠. 항상 최근 N명만 탑승할 수 있는 구조입니다.

Ch9.3에서 리스트 슬라이싱으로 간단한 윈도우를 구현해 봤었죠. 여기서는 `trim_messages`의 `token_counter=len` 트릭을 활용해서 같은 결과를 **선언적이고 재사용 가능한 Runnable**로 만드는 방법을 다룹니다. 이렇게 하면 LCEL 파이프라인 어디든 자유롭게 끼워 넣을 수 있습니다.

```python
from langchain_core.messages import (
    trim_messages,
    SystemMessage,
    HumanMessage,
    AIMessage,
)

# 슬라이딩 윈도우: 최근 k개 메시지만 유지
def sliding_window_trimmer(k: int = 10):
    """최근 k개 메시지만 유지하는 트리머 생성.
    
    Args:
        k: 유지할 최근 메시지 수 (시스템 메시지 제외)
    """
    return trim_messages(
        max_tokens=k,
        token_counter=len,            # len을 사용하면 메시지 "개수"로 카운트
        strategy="last",
        include_system=True,           # 시스템 메시지는 항상 보존
        start_on="human",             # 항상 Human 메시지부터 시작
    )

# 20개의 대화 턴이 쌓인 상황 시뮬레이션
messages = [SystemMessage(content="당신은 AI 어시스턴트입니다.")]
for i in range(10):
    messages.append(HumanMessage(content=f"질문 {i+1}번입니다."))
    messages.append(AIMessage(content=f"답변 {i+1}번입니다."))

print(f"트리밍 전: {len(messages)}개 메시지")  # 21개

# 최근 6개 메시지만 유지 (시스템 메시지 + 최근 3턴)
trimmer = sliding_window_trimmer(k=6)
trimmed = trimmer.invoke(messages)

print(f"트리밍 후: {len(trimmed)}개 메시지")   # 7개 (시스템 1 + 최근 6)
for msg in trimmed:
    print(f"  [{msg.type}] {msg.content}")
# [system] 당신은 AI 어시스턴트입니다.
# [human] 질문 8번입니다.
# [ai] 답변 8번입니다.
# [human] 질문 9번입니다.
# [ai] 답변 9번입니다.
# [human] 질문 10번입니다.
# [ai] 답변 10번입니다.
```

슬라이딩 윈도우의 장단점은 명확합니다:

| 장점 | 단점 |
|------|------|
| 구현이 매우 단순 | 오래된 맥락을 완전히 잃음 |
| 예측 가능한 토큰 사용량 | "5번째 질문에서 뭐라고 했었지?" 에 답할 수 없음 |
| 연산 비용 없음 (LLM 호출 불필요) | k 값 선택이 어려움 |

### 개념 4: 대화 요약(Summarization) — 압축하되 기억하기

> 💡 **비유**: 회의록을 생각해 보세요. 3시간짜리 회의 전체를 녹취록으로 보관하는 것(전체 히스토리)과, 핵심 결정 사항만 정리한 요약본(summary)을 만드는 것의 차이입니다. 요약본이 있으면 새 참석자도 빠르게 맥락을 파악할 수 있죠.

Ch9.3에서 `await llm.ainvoke(summary_prompt)` 한 줄로 빠르게 요약하는 패턴을 봤었는데요, 프로덕션에서는 더 정교한 설계가 필요합니다. **기존 요약과 새 대화를 점진적으로 병합하는 누적 요약(progressive summarization)** 패턴, 별도의 저렴한 모델을 요약 전용으로 쓰는 비용 최적화, 그리고 요약 품질을 높이는 프롬프트 설계까지 다뤄보겠습니다.

LangChain 0.3에서 기존의 `ConversationSummaryMemory`는 deprecated 되었지만, 요약 *패턴* 자체는 여전히 강력합니다. 현대적인 LCEL 방식으로 직접 구현하는 방법을 알아보겠습니다.

```python
from langchain_core.messages import (
    SystemMessage,
    HumanMessage,
    AIMessage,
    BaseMessage,
)
from langchain_core.prompts import ChatPromptTemplate
from langchain_openai import ChatOpenAI

# 요약 전용 모델 (비용 절감을 위해 작은 모델 사용)
summarizer_llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)

# 요약 프롬프트 — 핵심 정보 보존에 특화
SUMMARY_PROMPT = ChatPromptTemplate.from_messages([
    ("system",
     "당신은 대화 요약 전문가입니다. "
     "기존 요약과 새로운 대화 내용을 받아, "
     "핵심 정보를 보존하면서 간결하게 통합 요약하세요. "
     "사용자의 이름, 선호도, 중요한 결정 사항은 반드시 포함하세요."),
    ("human",
     "기존 요약:\n{existing_summary}\n\n"
     "새로운 대화:\n{new_lines}\n\n"
     "위 내용을 통합한 업데이트된 요약을 작성하세요."),
])

# 요약 체인
summary_chain = SUMMARY_PROMPT | summarizer_llm


def summarize_messages(
    messages: list[BaseMessage],
    existing_summary: str = "",
    keep_recent: int = 4,
) -> tuple[str, list[BaseMessage]]:
    """오래된 메시지를 요약하고 최근 메시지만 유지.
    
    Args:
        messages: 전체 메시지 리스트
        existing_summary: 이전에 만들어둔 요약
        keep_recent: 요약하지 않고 유지할 최근 메시지 수
    
    Returns:
        (업데이트된 요약, 유지할 최근 메시지 리스트)
    """
    if len(messages) <= keep_recent:
        return existing_summary, messages
    
    # 요약할 메시지와 유지할 메시지 분리
    to_summarize = messages[:-keep_recent]
    to_keep = messages[-keep_recent:]
    
    # 요약할 대화를 텍스트로 변환
    new_lines = "\n".join(
        f"{msg.type}: {msg.content}" for msg in to_summarize
    )
    
    # LLM으로 요약 생성
    result = summary_chain.invoke({
        "existing_summary": existing_summary or "없음",
        "new_lines": new_lines,
    })
    
    updated_summary = result.content
    return updated_summary, to_keep
```

이 요약 함수를 `RunnableWithMessageHistory`와 조합하면 완전한 요약 기반 메모리 시스템을 만들 수 있습니다:

```python
from langchain_core.chat_history import InMemoryChatMessageHistory
from langchain_core.prompts import ChatPromptTemplate, MessagesPlaceholder
from langchain_core.runnables.history import RunnableWithMessageHistory

# 대화 모델
chat_model = ChatOpenAI(model="gpt-4o", temperature=0.7)

# 요약을 포함한 프롬프트 — 요약은 시스템 메시지에 주입
def create_prompt_with_summary(summary: str = ""):
    system_content = "당신은 친절한 AI 어시스턴트입니다."
    if summary:
        system_content += f"\n\n[이전 대화 요약]\n{summary}"
    
    return ChatPromptTemplate.from_messages([
        ("system", system_content),
        MessagesPlaceholder(variable_name="history"),
        ("human", "{input}"),
    ])

# 세션별 요약과 히스토리를 관리하는 클래스
class SummarizingChatManager:
    """요약 기반 메모리를 관리하는 대화 매니저."""
    
    def __init__(
        self,
        model: ChatOpenAI,
        max_messages: int = 10,
        keep_recent: int = 4,
    ):
        self.model = model
        self.max_messages = max_messages      # 요약 트리거 임계값
        self.keep_recent = keep_recent        # 요약 시 유지할 최근 메시지 수
        self.histories: dict[str, InMemoryChatMessageHistory] = {}
        self.summaries: dict[str, str] = {}   # 세션별 요약 저장
    
    def get_session_history(self, session_id: str):
        if session_id not in self.histories:
            self.histories[session_id] = InMemoryChatMessageHistory()
            self.summaries[session_id] = ""
        return self.histories[session_id]
    
    def chat(self, session_id: str, user_input: str) -> str:
        history = self.get_session_history(session_id)
        
        # 메시지가 임계값을 넘으면 요약 실행
        if len(history.messages) >= self.max_messages:
            summary, recent = summarize_messages(
                history.messages,
                self.summaries.get(session_id, ""),
                self.keep_recent,
            )
            self.summaries[session_id] = summary
            history.clear()
            for msg in recent:
                history.add_message(msg)
        
        # 요약을 포함한 프롬프트 생성
        prompt = create_prompt_with_summary(
            self.summaries.get(session_id, "")
        )
        chain = prompt | self.model
        
        # RunnableWithMessageHistory로 감싸기
        chain_with_history = RunnableWithMessageHistory(
            chain,
            self.get_session_history,
            input_messages_key="input",
            history_messages_key="history",
        )
        
        config = {"configurable": {"session_id": session_id}}
        response = chain_with_history.invoke(
            {"input": user_input},
            config=config,
        )
        return response.content

# 사용 예시
manager = SummarizingChatManager(
    model=ChatOpenAI(model="gpt-4o"),
    max_messages=10,   # 10개 메시지가 쌓이면 요약
    keep_recent=4,     # 최근 4개는 원본 유지
)

# 대화가 길어져도 안정적으로 동작
response = manager.chat("session-1", "안녕하세요! 저는 김민수입니다.")
print(response)
```

### 개념 5: 하이브리드 전략 — 요약 + 슬라이딩 윈도우

> 💡 **비유**: 도서관의 서가 정리와 비슷합니다. 최근에 빌려간 책(최근 대화)은 반납대에 그대로 두고, 오래된 책(지난 대화)은 분류 카드(요약)만 남기고 서고에 넣어두는 거죠. 누군가 그 책에 대해 물으면 카드를 보고 빠르게 찾을 수 있습니다.

가장 실전에서 많이 쓰이는 전략은 요약과 슬라이딩 윈도우를 결합하는 것입니다. 이것은 기존의 `ConversationSummaryBufferMemory`가 하던 일을 현대적 LCEL 방식으로 재현한 패턴이에요.

```python
from langchain_core.messages import (
    trim_messages,
    SystemMessage,
    HumanMessage,
    AIMessage,
    BaseMessage,
)
from langchain_openai import ChatOpenAI

def create_hybrid_memory_manager(
    token_limit: int = 2000,
    summary_threshold: int = 1500,
):
    """하이브리드 메모리 매니저: 토큰 트리밍 + 자동 요약.
    
    Args:
        token_limit: LLM에 전달할 최대 토큰 수
        summary_threshold: 이 토큰 수를 넘으면 요약 트리거
    """
    trimmer = trim_messages(
        max_tokens=token_limit,
        strategy="last",
        token_counter="approximate",
        include_system=True,
        start_on="human",
    )
    
    summarizer = ChatOpenAI(model="gpt-4o-mini", temperature=0)
    
    def process_messages(
        messages: list[BaseMessage],
        existing_summary: str = "",
    ) -> list[BaseMessage]:
        """메시지를 최적화하여 반환."""
        # 1단계: 토큰 수 확인
        trimmed = trimmer.invoke(messages)
        
        # 2단계: 트리밍으로 메시지가 많이 잘렸으면 요약도 함께 제공
        if len(trimmed) < len(messages):
            # 잘린 부분을 요약
            dropped = messages[:len(messages) - len(trimmed)]
            new_lines = "\n".join(
                f"{m.type}: {m.content}" for m in dropped
                if not isinstance(m, SystemMessage)
            )
            
            if new_lines:
                summary_response = summarizer.invoke(
                    f"기존 요약: {existing_summary or '없음'}\n\n"
                    f"새로운 대화:\n{new_lines}\n\n"
                    f"통합 요약을 작성하세요."
                )
                updated_summary = summary_response.content
                
                # 시스템 메시지에 요약 추가
                if trimmed and isinstance(trimmed[0], SystemMessage):
                    trimmed[0] = SystemMessage(
                        content=trimmed[0].content
                        + f"\n\n[이전 대화 요약]\n{updated_summary}"
                    )
        
        return trimmed
    
    return process_messages
```

"""메모리 최적화 전략 비교 실습.

세 가지 트리밍 전략의 동작을 시각적으로 비교합니다.

from langchain_core.messages import (
    trim_messages,
    SystemMessage,

    AIMessage,
    BaseMessage,

# ── 1. 테스트용 대화 히스토리 생성 ──
def create_sample_conversation(n_turns: int = 8) -> list[BaseMessage]:
    """n_turns 턴의 샘플 대화를 생성합니다."""
    messages = [
        SystemMessage(content="당신은 요리 전문 AI 셰프입니다.")
    ]
    topics = [
        ("김치찌개 레시피 알려주세요", "김치찌개는 김치, 돼지고기, 두부가 필요합니다..."),
        ("양념 비율이 어떻게 되나요?", "고춧가루 2큰술, 간장 1큰술, 다진 마늘 1큰술..."),
        ("채식으로 바꿀 수 있나요?", "돼지고기 대신 버섯을 사용하면 됩니다..."),
        ("된장찌개도 알려주세요", "된장찌개는 된장 2큰술, 호박, 감자, 두부..."),
        ("찌개 끓이는 시간은요?", "중불에서 20분, 약불에서 10분 정도..."),
        ("냉장 보관은 며칠이나?", "밀폐 용기에 넣으면 3-4일 보관 가능합니다..."),
        ("전자레인지로 데워도 되나요?", "네, 2-3분 정도 데우면 됩니다..."),
        ("제 이름은 민수인데, 기억해주세요", "네, 민수님! 기억하겠습니다..."),
    ]
    for q, a in topics[:n_turns]:
        messages.append(HumanMessage(content=q))
        messages.append(AIMessage(content=a))
    return messages


# ── 2. 전략 1: 토큰 기반 트리밍 ──
def strategy_token_trim(messages: list[BaseMessage]) -> list[BaseMessage]:
    """토큰 기반으로 최근 메시지만 유지."""
    trimmer = trim_messages(
        max_tokens=200,                # 최대 200토큰
        strategy="last",               # 최근 메시지 우선
        token_counter="approximate",   # 근사 토큰 계산
        include_system=True,           # 시스템 메시지 보존
        start_on="human",             # Human부터 시작
    )
    return trimmer.invoke(messages)


# ── 3. 전략 2: 메시지 개수 기반 슬라이딩 윈도우 ──
def strategy_sliding_window(
    messages: list[BaseMessage],
    k: int = 6,
) -> list[BaseMessage]:
    """최근 k개 메시지만 유지 (슬라이딩 윈도우)."""
    trimmer = trim_messages(
        max_tokens=k,          # k개 메시지
        token_counter=len,     # len = 메시지 개수로 카운트
        strategy="last",
        include_system=True,
        start_on="human",
    )
    return trimmer.invoke(messages)


# ── 4. 전략 3: 수동 요약 시뮬레이션 ──
def strategy_summary_simulation(
    messages: list[BaseMessage],
    keep_recent: int = 4,
) -> list[BaseMessage]:
    """오래된 메시지를 요약 텍스트로 대체 (LLM 없이 시뮬레이션)."""
    if len(messages) <= keep_recent + 1:  # +1은 시스템 메시지
        return messages
    
    system_msg = messages[0] if isinstance(messages[0], SystemMessage) else None
    recent = messages[-keep_recent:]
    
    # 잘린 메시지들의 요약을 시뮬레이션
    dropped = messages[1:-keep_recent] if system_msg else messages[:-keep_recent]
    summary_topics = []
    for msg in dropped:
        if isinstance(msg, HumanMessage):
            summary_topics.append(msg.content[:20])
    
    summary_text = f"[이전 대화 요약: {', '.join(summary_topics)} 등에 대해 논의함]"
    
    result = []
    if system_msg:
        result.append(SystemMessage(
            content=system_msg.content + "\n\n" + summary_text
        ))
    result.extend(recent)
    return result


# ── 5. 비교 실행 ──
def compare_strategies():
    """세 가지 전략의 결과를 비교합니다."""
    messages = create_sample_conversation(8)
    
    print(f"{'='*60}")
    print(f"원본 대화: {len(messages)}개 메시지")
    print(f"{'='*60}\n")
    
    strategies = [
        ("토큰 트리밍 (200토큰)", strategy_token_trim(messages)),
        ("슬라이딩 윈도우 (k=6)", strategy_sliding_window(messages, k=6)),
        ("요약 시뮬레이션 (keep=4)", strategy_summary_simulation(messages, keep_recent=4)),
    ]
    
    for name, result in strategies:
        print(f"── {name} → {len(result)}개 메시지 ──")
        for msg in result:
            content = msg.content[:60]
            if len(msg.content) > 60:
                content += "..."
            print(f"  [{msg.type:6s}] {content}")
        print()

compare_strategies()

## 더 깊이 알아보기

### 메모리 관리의 역사: 인지 심리학에서 LLM까지

메모리 최적화 전략은 사실 인지 심리학에서 오래전부터 연구되어 온 주제입니다. 1956년 조지 밀러(George Miller)가 발표한 유명한 논문 *"The Magical Number Seven, Plus or Minus Two"*에서 인간의 단기 기억 용량이 약 7±2개 항목으로 제한된다는 것을 밝혔죠. 이것이 바로 "슬라이딩 윈도우"의 인지적 근거입니다.

LangChain의 메모리 시스템도 이런 인지 모델에서 영감을 받았습니다. 초기 LangChain(0.0.x)에서는 `ConversationBufferWindowMemory`(슬라이딩 윈도우), `ConversationSummaryMemory`(요약), `ConversationSummaryBufferMemory`(하이브리드)라는 세 가지 전용 클래스를 제공했습니다. 하지만 Harrison Chase와 LangChain 팀은 이런 특수 목적 클래스 대신 **조합 가능한 작은 도구들**이 더 유연하다는 것을 깨달았어요.

그래서 LangChain 0.2에서 `trim_messages` 유틸리티가 도입되었고, 0.3에서는 기존 메모리 클래스들이 deprecated 되면서 `trim_messages` + LCEL 조합이 공식 권장 방식이 되었습니다. 이는 소프트웨어 설계에서 "상속보다 조합(composition over inheritance)"이라는 원칙이 LLM 프레임워크에도 적용된 좋은 사례죠.

### 토큰 카운팅의 미묘한 세계

"approximate" 토큰 카운터는 내부적으로 텍스트의 바이트 수를 기반으로 추정합니다. 영어의 경우 대략 1토큰 ≈ 4문자이지만, 한국어는 1토큰 ≈ 1.5~2문자에 가깝습니다. 이 차이 때문에 한국어 애플리케이션에서는 `"approximate"` 모드가 토큰 수를 과소 추정할 수 있습니다. 정밀한 제어가 필요하면 실제 LLM 인스턴스를 `token_counter`로 전달하는 것이 안전합니다.

## 흔한 오해와 팁

> ⚠️ **흔한 오해**: "trim_messages를 쓰면 원본 메시지도 삭제된다" — 아닙니다! `trim_messages`는 **새로운 리스트를 반환**할 뿐, 원본 메시지 리스트를 수정하지 않습니다. 저장소의 전체 히스토리는 그대로 보존되고, LLM에 전달되는 복사본만 트리밍됩니다.

> 💡 **알고 계셨나요?**: LangChain 0.3부터 `ConversationSummaryMemory`, `ConversationBufferWindowMemory`, `ConversationSummaryBufferMemory` 등 기존 메모리 클래스는 모두 deprecated 되었습니다. 공식 마이그레이션 가이드에서는 `trim_messages` + `RunnableWithMessageHistory` 조합 또는 LangGraph persistence를 권장합니다. 하지만 `RunnableWithMessageHistory`와 `BaseChatMessageHistory`는 앞으로도 계속 지원될 예정이에요.

> 🔥 **실무 팁**: 토큰 트리밍과 요약을 조합할 때, 요약용 LLM은 **반드시 메인 대화 모델보다 저렴한 모델**을 사용하세요. 예를 들어 대화에 `gpt-4o`를 쓴다면 요약에는 `gpt-4o-mini`를 쓰는 식입니다. 요약은 창의성이 필요 없으므로 `temperature=0`으로 설정하는 것도 좋은 관행이에요.

> 🔥 **실무 팁**: `include_system=True`와 `start_on="human"`은 거의 항상 함께 사용해야 합니다. 시스템 메시지가 빠지면 LLM이 역할을 잊고, AI 메시지로 시작하면 대화 흐름이 깨집니다. 이 두 옵션을 "기본값"처럼 늘 세팅해두세요.

## 핵심 정리

| 개념 | 설명 |
|------|------|
| `trim_messages` | `langchain_core`의 토큰/개수 기반 메시지 트리밍 유틸리티. `strategy`, `include_system`, `allow_partial` 등 세밀한 제어 가능 |
| `strategy="last"` | 최근 메시지부터 보존하는 전략. 대부분의 대화 시나리오에 적합 |
| `token_counter` | `len`(개수), `"approximate"`(근사), LLM 인스턴스(정확) 중 선택 |
| 슬라이딩 윈도우 | `token_counter=len`으로 최근 k개 메시지만 유지. 단순하지만 오래된 맥락 소실 |
| 대화 요약 | LLM으로 오래된 대화를 압축. 맥락 보존은 좋지만 추가 LLM 호출 비용 발생 |
| 하이브리드 전략 | 트리밍 + 요약 결합. 최근 대화는 원본 유지, 오래된 대화는 요약으로 대체 |
| `include_system=True` | SystemMessage를 트리밍에서 제외하여 항상 보존 |
| `start_on="human"` | 트리밍 결과가 HumanMessage로 시작하도록 보장 |
