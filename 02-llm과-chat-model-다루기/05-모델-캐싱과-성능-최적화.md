# 모델 캐싱과 성능 최적화

> LLM 응답을 캐싱하여 비용을 절감하고 응답 속도를 극적으로 높이는 전략을 배웁니다

## 개요

이 섹션에서는 LangChain의 캐싱 시스템을 활용하여 동일한 프롬프트에 대한 반복 API 호출을 제거하고, 응답 시간과 비용을 최적화하는 방법을 학습합니다. InMemoryCache부터 SQLiteCache, RedisCache까지 상황별 최적 캐시를 선택하고 설계하는 능력을 기릅니다.

**선수 지식**: [Session 2.1: ChatOpenAI 심화](./01-chatopenai-심화.md)에서 배운 토큰 추적과 비용 관리, [Session 2.4: 모델 폴백과 안정성](./04-모델-폴백과-안정성.md)에서 배운 with_fallbacks/with_retry 패턴

**학습 목표**:
- `set_llm_cache`를 사용하여 전역/개별 캐시를 설정할 수 있다
- InMemoryCache, SQLiteCache, RedisCache의 특성을 이해하고 상황에 맞게 선택할 수 있다
- 캐시 전략을 설계하여 프로덕션 환경에서 비용과 응답 속도를 최적화할 수 있다
- 응답 시간을 측정하고 캐시 효과를 정량적으로 분석할 수 있다

## 왜 알아야 할까?

LLM API 호출은 비쌉니다. GPT-4o 기준 입력 1M 토큰에 약 $2.50, 출력 1M 토큰에 약 $10이 듭니다. 그런데 실제 서비스를 운영하다 보면, 같은 질문이 놀랍게도 자주 반복됩니다. "배송 정책이 뭐예요?", "환불 어떻게 해요?" 같은 질문은 하루에도 수백, 수천 번 들어올 수 있죠.

캐싱 없이 이런 반복 호출을 모두 API로 처리하면? 매달 수백만 원의 비용이 그대로 빠져나갑니다. 반면 캐싱을 적용하면 동일 프롬프트에 대해 **즉시 응답**(수 밀리초)하면서 API 비용은 0원입니다. [Session 2.1](./01-chatopenai-심화.md)에서 `get_openai_callback`으로 토큰 비용을 추적하는 방법을 배웠는데, 캐싱은 그 비용 자체를 줄여주는 근본적인 해결책이에요.

## 핵심 개념

### 개념 1: LangChain 캐싱의 동작 원리

> 💡 **비유**: 도서관에서 책을 찾는 상황을 떠올려 보세요. 처음 빌려 읽은 책의 핵심 내용을 노트에 적어두면, 다음에 같은 내용이 필요할 때 다시 도서관까지 갈 필요 없이 노트만 펼치면 되죠. LLM 캐싱도 똑같습니다. 한 번 받은 응답을 저장해두고, 동일한 질문이 오면 저장된 응답을 즉시 돌려줍니다.

LangChain의 캐싱은 **프롬프트 텍스트 + 모델 설정**(llm_string)을 키로 사용합니다. 동일한 프롬프트를 동일한 모델 설정으로 호출하면 캐시에서 결과를 반환하고, 그렇지 않으면 실제 API를 호출한 뒤 결과를 캐시에 저장합니다.

```python
# 캐시 설정의 기본 구조
from langchain_core.globals import set_llm_cache
from langchain_core.caches import InMemoryCache

# 전역 캐시 설정 — 모든 LLM 호출에 적용
set_llm_cache(InMemoryCache())
```

캐시 키는 내부적으로 `(prompt, llm_string)` 튜플로 구성됩니다. `llm_string`에는 모델명, temperature, max_tokens 등 모델 파라미터가 포함되므로, 같은 프롬프트라도 파라미터가 다르면 별도의 캐시 엔트리가 생성됩니다.

### 개념 2: InMemoryCache — 가장 빠른 캐시

> 💡 **비유**: 포스트잇에 메모를 적어 모니터에 붙여두는 것과 같습니다. 바로 눈앞에 있어서 엄청 빠르지만, 컴퓨터를 끄면(프로세스가 종료되면) 사라집니다.

`InMemoryCache`는 파이썬 딕셔너리 기반의 메모리 내 캐시입니다. 가장 빠르지만 프로세스가 종료되면 데이터가 사라지는 휘발성 캐시입니다.

```python
import time
from langchain_core.globals import set_llm_cache
from langchain_core.caches import InMemoryCache
from langchain_openai import ChatOpenAI

# InMemoryCache 설정
set_llm_cache(InMemoryCache())

llm = ChatOpenAI(model="gpt-4o", temperature=0)

# 첫 번째 호출 — API 실제 호출 (느림)
start = time.time()
response1 = llm.invoke("대한민국의 수도는 어디인가요?")
first_call = time.time() - start
print(f"첫 번째 호출: {first_call:.3f}초")
print(f"응답: {response1.content[:50]}...")

# 두 번째 호출 — 캐시에서 반환 (매우 빠름)
start = time.time()
response2 = llm.invoke("대한민국의 수도는 어디인가요?")
second_call = time.time() - start
print(f"두 번째 호출: {second_call:.3f}초")
print(f"속도 향상: {first_call / second_call:.0f}배 빨라짐!")

# 출력 예시:
# 첫 번째 호출: 1.234초
# 응답: 대한민국의 수도는 서울입니다...
# 두 번째 호출: 0.001초
# 속도 향상: 1234배 빨라짐!
```

`InMemoryCache`는 `maxsize` 파라미터로 최대 저장 항목 수를 제한할 수 있습니다. 메모리 부족을 방지하는 데 유용하죠.

```python
# 최대 100개 항목만 캐시 (초과 시 가장 오래된 항목 제거)
set_llm_cache(InMemoryCache(maxsize=100))
```

### 개념 3: SQLiteCache — 영구 보존 캐시

> 💡 **비유**: 노트에 메모하는 대신 책장에 파일을 정리해 보관하는 것과 같습니다. 집을 나갔다 돌아와도(프로세스를 재시작해도) 파일은 그대로 있죠. 찾는 데 포스트잇보다 살짝 느리지만, 안전하게 보존됩니다.

`SQLiteCache`는 로컬 SQLite 데이터베이스 파일에 캐시를 저장합니다. 프로세스가 종료되어도 캐시가 유지되므로, 개발 환경이나 단일 서버 환경에서 매우 유용합니다.

```python
from langchain_community.cache import SQLiteCache
from langchain_core.globals import set_llm_cache

# SQLite 파일 기반 영구 캐시 설정
set_llm_cache(SQLiteCache(database_path=".langchain_cache.db"))

# 이제 프로그램을 종료했다가 다시 실행해도
# 이전에 캐시된 응답이 그대로 남아 있습니다!
```

SQLiteCache의 장점은 **설정이 극도로 간단**하다는 것입니다. 별도의 서버나 인프라 없이 파일 하나로 영구 캐시를 구현할 수 있습니다. 다만 동시에 여러 프로세스가 같은 DB 파일에 접근하면 문제가 생길 수 있으므로, 멀티 프로세스 환경에서는 RedisCache를 고려해야 합니다.

### 개념 4: RedisCache — 프로덕션 분산 캐시

> 💡 **비유**: 회사의 공유 서버에 문서를 올려두는 것과 같습니다. 어떤 팀원이 어떤 컴퓨터에서 접속하든 같은 문서에 접근할 수 있고, 문서에 만료 기한(TTL)을 붙여서 자동으로 정리할 수도 있습니다.

프로덕션 환경에서는 여러 서버 인스턴스가 동시에 실행됩니다. 이때 각 서버마다 별도의 InMemoryCache를 쓰면 캐시 적중률이 떨어지죠. RedisCache는 중앙 Redis 서버를 통해 모든 인스턴스가 캐시를 공유합니다.

```python
from langchain_redis import RedisCache
from langchain_core.globals import set_llm_cache

# Redis 캐시 설정 (TTL 1시간)
redis_cache = RedisCache(
    redis_url="redis://localhost:6379",  # Redis 서버 주소
    ttl=3600,                             # 캐시 유효기간: 3600초(1시간)
    prefix="myapp"                        # 키 접두사 (네임스페이스 분리)
)
set_llm_cache(redis_cache)
```

RedisCache의 핵심 파라미터를 정리하면 다음과 같습니다:

| 파라미터 | 기본값 | 설명 |
|---------|--------|------|
| `redis_url` | `"redis://localhost:6379"` | Redis 서버 접속 URL |
| `ttl` | `None` | 캐시 만료 시간(초). None이면 영구 보존 |
| `prefix` | `"redis"` | Redis 키 접두사. 여러 앱이 같은 Redis를 공유할 때 분리용 |
| `redis` | `None` | 기존 Redis 클라이언트 인스턴스 직접 전달 |

```python
# 기존 Redis 클라이언트를 재사용하는 경우
import redis

existing_client = redis.Redis(host="cache.myapp.com", port=6379, db=2)
redis_cache = RedisCache(redis=existing_client, ttl=1800)
set_llm_cache(redis_cache)
```

### 개념 5: 캐시 전략 설계 — 언제 어떤 캐시를 쓸까?

캐시 선택은 환경과 요구사항에 따라 달라집니다. 핵심 판단 기준은 세 가지입니다: **영속성**, **공유 범위**, **성능 요구사항**.

```
┌─────────────────────────────────────────────────────┐
│                캐시 선택 가이드                        │
├─────────────────────────────────────────────────────┤
│  개발/프로토타입   →  InMemoryCache                    │
│  단일 서버/로컬    →  SQLiteCache                      │
│  멀티 서버/프로덕션 →  RedisCache                      │
└─────────────────────────────────────────────────────┘
```

| 캐시 타입 | 영속성 | 공유 | 속도 | 설정 난이도 | 적합한 환경 |
|-----------|--------|------|------|------------|------------|
| InMemoryCache | ❌ 휘발성 | ❌ 단일 프로세스 | ⚡ 가장 빠름 | ⭐ 매우 쉬움 | 개발, 테스트, 노트북 |
| SQLiteCache | ✅ 영구 | ❌ 단일 서버 | 🔵 빠름 | ⭐ 쉬움 | 개발, 소규모 서비스 |
| RedisCache | ✅ 영구 | ✅ 멀티 서버 | 🔵 빠름 | ⭐⭐ 보통 | 프로덕션, 대규모 서비스 |

실무에서는 환경별로 캐시를 다르게 설정하는 패턴이 흔합니다:

```python
import os
from langchain_core.globals import set_llm_cache
from langchain_core.caches import InMemoryCache

def setup_cache(env: str = "development") -> None:
    """환경별 최적 캐시를 설정합니다.

    Args:
        env: 실행 환경 ("development", "staging", "production")
    """
    if env == "production":
        from langchain_redis import RedisCache
        cache = RedisCache(
            redis_url=os.getenv("REDIS_URL", "redis://localhost:6379"),
            ttl=3600,
            prefix="prod_llm"
        )
    elif env == "staging":
        from langchain_community.cache import SQLiteCache
        cache = SQLiteCache(database_path=".langchain_cache.db")
    else:
        cache = InMemoryCache()

    set_llm_cache(cache)
    print(f"[{env}] 캐시 설정 완료: {type(cache).__name__}")

# 환경 변수로 자동 선택
setup_cache(os.getenv("APP_ENV", "development"))
```

### 개념 6: 응답 시간 측정과 캐시 효과 분석

캐시가 정말 효과가 있는지 확인하려면, 숫자로 증명해야 합니다. 응답 시간 측정과 캐시 적중률 추적은 캐싱 전략의 필수 요소입니다.

```python
import time
from dataclasses import dataclass, field
from typing import Any

@dataclass
class CacheMetrics:
    """캐시 성능 지표를 추적하는 클래스."""
    total_calls: int = 0
    cache_hits: int = 0
    total_time: float = 0.0
    times: list[float] = field(default_factory=list)

    @property
    def hit_rate(self) -> float:
        """캐시 적중률을 반환합니다."""
        return self.cache_hits / self.total_calls if self.total_calls > 0 else 0.0

    @property
    def avg_time(self) -> float:
        """평균 응답 시간을 반환합니다."""
        return self.total_time / self.total_calls if self.total_calls > 0 else 0.0

    def record(self, elapsed: float, is_cache_hit: bool) -> None:
        """호출 결과를 기록합니다."""
        self.total_calls += 1
        self.total_time += elapsed
        self.times.append(elapsed)
        if is_cache_hit:
            self.cache_hits += 1

    def report(self) -> str:
        """성능 리포트를 반환합니다."""
        return (
            f"총 호출: {self.total_calls}회\n"
            f"캐시 적중: {self.cache_hits}회 ({self.hit_rate:.1%})\n"
            f"평균 응답 시간: {self.avg_time:.3f}초\n"
            f"총 소요 시간: {self.total_time:.3f}초"
        )
```

## 실습: 직접 해보기

세 가지 캐시 타입의 성능을 직접 비교하는 완전한 실습 코드입니다.

```python
"""
LangChain 캐시 비교 실습
- InMemoryCache vs SQLiteCache 성능 비교
- 캐시 적중 시 응답 시간 측정
- 비용 절감 효과 확인
"""
import os
import time
from dotenv import load_dotenv
from langchain_core.globals import set_llm_cache
from langchain_core.caches import InMemoryCache
from langchain_community.cache import SQLiteCache
from langchain_openai import ChatOpenAI
from langchain_community.callbacks import get_openai_callback

load_dotenv()

# 테스트할 프롬프트 목록 (일부 중복 포함)
PROMPTS = [
    "Python의 리스트 컴프리헨션을 설명해주세요.",
    "REST API와 GraphQL의 차이점은?",
    "Python의 리스트 컴프리헨션을 설명해주세요.",  # 중복!
    "도커 컨테이너와 가상 머신의 차이점은?",
    "REST API와 GraphQL의 차이점은?",               # 중복!
    "Python의 리스트 컴프리헨션을 설명해주세요.",  # 중복!
]


def benchmark_cache(cache_name: str, cache_instance) -> dict:
    """특정 캐시의 성능을 벤치마크합니다.

    Args:
        cache_name: 캐시 이름 (출력용)
        cache_instance: 캐시 인스턴스

    Returns:
        성능 측정 결과 딕셔너리
    """
    set_llm_cache(cache_instance)
    llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)

    total_time = 0.0
    call_times = []

    # 전체 API 비용 추적
    with get_openai_callback() as cb:
        for i, prompt in enumerate(PROMPTS):
            start = time.time()
            response = llm.invoke(prompt)
            elapsed = time.time() - start
            total_time += elapsed
            call_times.append(elapsed)

            # 캐시 적중 여부 판단 (0.1초 미만이면 캐시 적중으로 추정)
            hit = "✅ 캐시" if elapsed < 0.1 else "🌐 API"
            print(f"  [{hit}] {elapsed:.3f}초 | {prompt[:30]}...")

    # 캐시 없이 모든 호출이 API였을 경우의 예상 비용과 비교
    unique_prompts = len(set(PROMPTS))
    total_prompts = len(PROMPTS)
    saved_calls = total_prompts - unique_prompts

    print(f"\n  📊 {cache_name} 결과:")
    print(f"  총 호출: {total_prompts}회 (고유: {unique_prompts}, 캐시 적중: {saved_calls})")
    print(f"  총 소요 시간: {total_time:.3f}초")
    print(f"  실제 API 비용: ${cb.total_cost:.6f}")
    print(f"  절약한 API 호출: {saved_calls}회")
    print()

    return {
        "name": cache_name,
        "total_time": total_time,
        "api_cost": cb.total_cost,
        "saved_calls": saved_calls,
    }


def main():
    """캐시 비교 벤치마크를 실행합니다."""
    print("=" * 60)
    print("🔬 LangChain 캐시 성능 비교 벤치마크")
    print("=" * 60)

    results = []

    # 1. InMemoryCache 테스트
    print("\n📦 [1] InMemoryCache 테스트")
    print("-" * 40)
    results.append(benchmark_cache("InMemoryCache", InMemoryCache()))

    # 2. SQLiteCache 테스트
    db_path = ".benchmark_cache.db"
    # 이전 테스트 DB가 있으면 삭제하여 공정한 비교
    if os.path.exists(db_path):
        os.remove(db_path)

    print("📦 [2] SQLiteCache 테스트")
    print("-" * 40)
    results.append(benchmark_cache("SQLiteCache", SQLiteCache(database_path=db_path)))

    # 결과 비교
    print("=" * 60)
    print("📋 최종 비교")
    print("=" * 60)
    print(f"{'캐시':<20} {'시간':>10} {'비용':>12} {'절약 호출':>10}")
    print("-" * 55)
    for r in results:
        print(
            f"{r['name']:<20} {r['total_time']:>8.3f}초 "
            f"${r['api_cost']:>10.6f} {r['saved_calls']:>8}회"
        )

    # 정리
    if os.path.exists(db_path):
        os.remove(db_path)

    print("\n✅ 벤치마크 완료!")


if __name__ == "__main__":
    main()
```

> 🔥 **실무 팁**: 실습에서 `elapsed < 0.1`로 캐시 적중을 추정했는데, 실제 프로덕션에서는 LangChain의 콜백(Callback) 시스템이나 LangSmith를 사용하여 정확한 캐시 적중률을 추적하는 것이 좋습니다. 이 내용은 [Ch16: 콜백과 관찰 가능성](16-콜백과-관찰-가능성/01-콜백-시스템-이해.md)에서 자세히 다룹니다.

## 더 깊이 알아보기

### 캐싱의 역사: 메모이제이션에서 분산 캐시까지

캐싱(Caching)이라는 개념은 컴퓨터 과학 초기부터 존재했습니다. 1967년, IBM의 엔지니어 **라일 존슨(Lyle Johnson)**이 System/360 Model 85에 최초의 하드웨어 캐시 메모리를 설계했을 때, "cache"라는 용어가 공식적으로 사용되기 시작했습니다. 프랑스어 "cacher"(숨기다)에서 유래한 이 단어는, CPU가 느린 메인 메모리 대신 빠른 소형 메모리에 자주 쓰는 데이터를 "숨겨두는" 개념을 잘 표현했죠.

소프트웨어 영역에서는 **메모이제이션(Memoization)**이라는 이름으로 같은 아이디어가 발전했습니다. 1968년 도널드 미치(Donald Michie)가 이 용어를 처음 사용했는데, 함수의 입력-출력 쌍을 테이블에 저장해두고 같은 입력이 들어오면 저장된 출력을 반환하는 기법입니다. LangChain의 캐싱도 본질적으로는 이 메모이제이션의 확장이에요. 프롬프트(입력)와 응답(출력) 쌍을 저장한다는 점에서요.

분산 캐시의 시대는 2003년 **Memcached**의 등장으로 열렸고, 2009년에는 **Redis**가 등장하며 더 풍부한 데이터 구조와 영속성을 지원하기 시작했습니다. Redis의 창시자 살바토레 산필리포(Salvatore Sanfilippo)는 처음에 자신의 웹 분석 스타트업의 확장성 문제를 해결하기 위해 Redis를 만들었는데, 이것이 오늘날 LLM 캐싱의 기반 기술이 된 셈이죠.

### 시맨틱 캐시: 미래의 캐싱

일반 캐시는 프롬프트가 **정확히 동일**해야만 적중합니다. "서울 날씨 알려줘"와 "서울 기온 알려줘"는 의미는 비슷하지만 다른 캐시 키로 취급되죠. **시맨틱 캐시(Semantic Cache)**는 임베딩을 활용해 의미적으로 유사한 프롬프트에도 캐시를 적중시킵니다. `langchain_redis` 패키지의 `RedisSemanticCache`가 이를 지원하는데, 이 개념은 [Ch7: 임베딩과 벡터 스토어](07-임베딩과-벡터-스토어/01-텍스트-임베딩-이해.md)에서 임베딩을 배운 후에 더 깊이 이해할 수 있습니다.

## 흔한 오해와 팁

> ⚠️ **흔한 오해**: "캐시를 쓰면 항상 좋다"라고 생각하기 쉽지만, 그렇지 않습니다. `temperature`가 0보다 큰 모델에서는 매번 다른 응답이 필요할 수 있습니다. 창의적 글쓰기, 브레인스토밍 같은 용도에서 캐시를 쓰면 항상 같은 결과만 나와서 사용자 경험이 나빠질 수 있어요. 캐시는 **정형화된 답변**이 기대되는 FAQ, 정보 검색, 코드 생성 등에 적합합니다.

> 💡 **알고 계셨나요?**: LangChain의 캐시는 `temperature`, `max_tokens` 등 모델 파라미터도 캐시 키에 포함시킵니다. 따라서 같은 프롬프트라도 `temperature=0`과 `temperature=0.7`로 호출하면 서로 다른 캐시 엔트리가 됩니다. 이 덕분에 같은 질문이라도 용도에 따라 다른 캐시 전략을 안전하게 사용할 수 있죠.

> 🔥 **실무 팁**: 프로덕션에서 RedisCache를 사용할 때는 반드시 **TTL을 설정**하세요. TTL 없이 운영하면 Redis 메모리가 무한히 증가하다가 서버가 다운될 수 있습니다. FAQ 응답은 24시간(`ttl=86400`), 실시간 데이터 관련 응답은 1시간(`ttl=3600`) 정도가 적당합니다. 또한 `prefix`를 활용하여 서비스별로 캐시를 분리하면 관리가 훨씬 수월해집니다.

> ⚠️ **흔한 오해**: "SQLiteCache가 InMemoryCache보다 항상 느리다"고 단정 짓는 분들이 있는데, 현대 SSD 환경에서 SQLite의 읽기 성능은 매우 뛰어납니다. 작은 데이터셋에서는 두 캐시의 성능 차이가 수 밀리초 미만일 수 있어요. SQLiteCache의 진짜 단점은 속도가 아니라 **동시 쓰기 제한**(single-writer)입니다.

## 핵심 정리

| 개념 | 설명 |
|------|------|
| `set_llm_cache()` | 전역 LLM 캐시를 설정하는 함수. `langchain_core.globals`에서 임포트 |
| `InMemoryCache` | 메모리 기반 캐시. 가장 빠르지만 프로세스 종료 시 소멸. `maxsize`로 크기 제한 가능 |
| `SQLiteCache` | SQLite 파일 기반 영구 캐시. 단일 서버에 적합. `database_path`로 DB 경로 지정 |
| `RedisCache` | Redis 기반 분산 캐시. 멀티 서버 프로덕션 환경에 적합. `ttl`, `prefix` 설정 가능 |
| 캐시 키 | `(prompt, llm_string)` 조합. 같은 프롬프트 + 같은 모델 설정 = 캐시 적중 |
| TTL | Time-to-Live. 캐시 엔트리의 자동 만료 시간(초). 메모리 관리에 필수 |
| 시맨틱 캐시 | 임베딩 기반으로 의미가 유사한 프롬프트도 캐시 적중시키는 고급 캐시 |
