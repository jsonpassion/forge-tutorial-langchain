# 체인 구성과 바인딩

> LCEL 체인의 동작을 런타임에 유연하게 제어하는 bind, with_config, with_retry, with_listeners, configurable_fields 패턴을 마스터합니다.

## 개요

지금까지 우리는 LCEL로 체인을 **구성하고 실행하는 방법**을 집중적으로 배웠습니다. 파이프 연산자로 컴포넌트를 연결하고, 병렬 실행으로 성능을 높이고, 조건 분기로 흐름을 제어하는 것까지요. 하지만 현실의 프로덕션 환경에서는 한 가지 질문이 더 남습니다 — "같은 체인인데, **상황에 따라 다르게 동작**하게 할 수 없을까?"

이번 섹션에서는 체인을 한 번 정의해두고, **호출 시점에 동적으로 설정을 바꾸는** 강력한 메서드들을 다룹니다.

**선수 지식**: [5.1 LCEL 기초와 파이프 연산자](./01-lcel-기초와-파이프-연산자.md)의 Runnable 프로토콜과 invoke/stream 인터페이스, [5.2 핵심 Runnable 컴포넌트](./02-핵심-runnable-컴포넌트.md)의 RunnableLambda, [5.4 병렬 실행과 성능 최적화](./04-병렬-실행과-성능-최적화.md)의 비동기 패턴

**학습 목표**:
- `bind()`로 Runnable에 기본 인자를 고정하는 방법을 이해한다
- `with_config()`와 `configurable_fields()`로 런타임 설정을 동적 변경할 수 있다
- `with_retry()`와 `with_fallbacks()`로 프로덕션 수준의 안정성을 확보한다
- `with_listeners()`로 체인 실행 이벤트를 관찰하고 모니터링한다

## 왜 알아야 할까?

여러분이 레스토랑 주방장이라고 상상해보세요. 기본 레시피(체인)는 하나인데, 손님마다 "덜 맵게", "소금 적게", "채식으로" 같은 요청이 들어옵니다. 매번 새 레시피를 작성하는 건 비효율적이겠죠? 대신 기본 레시피에 **"이 부분은 이렇게 바꿔라"**라는 메모를 붙이는 게 훨씬 합리적입니다.

LCEL의 바인딩과 설정 메서드가 정확히 이 역할을 합니다:

- **개발 중**: 모델의 `temperature`를 0으로 낮춰 디버깅하다가, 프로덕션에서는 0.7로 올리고 싶을 때
- **A/B 테스트**: 같은 체인에서 GPT-4o와 Claude를 번갈아 테스트할 때
- **안정성**: API 호출이 실패하면 자동으로 재시도하거나 대체 모델로 전환할 때
- **모니터링**: 체인의 시작/종료 시간, 에러 발생을 실시간으로 추적할 때

이 모든 것을 **체인 코드를 수정하지 않고** 런타임에 처리할 수 있다는 것이 핵심입니다.

## 핵심 개념

### 개념 1: bind() — 기본 인자 고정하기

> 💡 **비유**: `bind()`는 리모컨의 **매크로 버튼**과 같습니다. "TV 켜기 → 채널 11 → 볼륨 15"를 매번 누르는 대신, 매크로 버튼 하나에 이 동작을 미리 저장해두는 거죠. Runnable의 `bind()`도 마찬가지로, 매번 전달해야 할 인자를 미리 고정해놓습니다.

체인 안에서 특정 Runnable에 **항상 동일한 인자**를 전달해야 할 때가 있습니다. 예를 들어, LLM에 `stop` 시퀀스를 지정하거나, `temperature`를 고정하고 싶을 때요. 이때 `bind()`를 사용하면 해당 인자가 invoke 시점에 자동으로 전달됩니다.

```python
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser
from langchain_openai import ChatOpenAI

# 기본 모델 생성
model = ChatOpenAI(model="gpt-4o", temperature=0.7)

# stop 시퀀스를 bind로 고정
# "DONE"이라는 단어가 나오면 생성을 멈춤
model_with_stop = model.bind(stop=["DONE"])

prompt = ChatPromptTemplate.from_template(
    "1부터 10까지 숫자를 나열하세요. 5 다음에 DONE을 출력하세요.\n"
)

# bind 없는 체인 vs bind 있는 체인
chain_normal = prompt | model | StrOutputParser()
chain_with_stop = prompt | model_with_stop | StrOutputParser()

# bind된 체인은 "DONE"에서 멈춤
result = chain_with_stop.invoke({})
print(result)  # 1, 2, 3, 4, 5 (DONE 이후는 생성되지 않음)
```

`bind()`의 핵심은 **체인 구성 시점에 인자를 고정**한다는 점입니다. 체인을 파이프로 연결할 때 중간 Runnable의 출력에 포함되지 않는 인자를 전달해야 하는 상황에서 특히 유용합니다.

```python
from langchain_core.pydantic_v1 import BaseModel, Field

# 구조화된 출력을 위한 함수 스키마 정의
class WeatherQuery(BaseModel):
    """날씨 조회 요청"""
    city: str = Field(description="도시 이름")
    unit: str = Field(description="온도 단위", default="celsius")

# bind_tools로 도구를 모델에 고정
model_with_tools = model.bind_tools([WeatherQuery])

prompt = ChatPromptTemplate.from_messages([
    ("system", "사용자의 날씨 요청을 분석하세요."),
    ("human", "{query}")
])

# 도구가 바인딩된 체인
chain = prompt | model_with_tools
result = chain.invoke({"query": "서울 날씨 알려줘"})
print(result.tool_calls)
# [{'name': 'WeatherQuery', 'args': {'city': '서울', 'unit': 'celsius'}}]
```

### 개념 2: with_config() — 런타임 설정 전달

> 💡 **비유**: `with_config()`는 음식 배달 앱의 **"요청사항"** 란과 같습니다. 같은 메뉴를 주문하지만, 배달마다 "젓가락 빼주세요", "문 앞에 놔주세요" 같은 설정을 다르게 넣을 수 있죠.

`bind()`가 체인 **정의 시점**에 인자를 고정하는 것이라면, `with_config()`는 체인 **호출 시점**에 설정을 전달합니다. `tags`, `metadata`, `max_concurrency` 등의 실행 환경 설정을 동적으로 제어할 수 있습니다.

```python
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser
from langchain_openai import ChatOpenAI

model = ChatOpenAI(model="gpt-4o")
prompt = ChatPromptTemplate.from_template("{topic}에 대해 한 문장으로 설명하세요.")
chain = prompt | model | StrOutputParser()

# 방법 1: invoke 시 config 직접 전달
result = chain.invoke(
    {"topic": "양자컴퓨팅"},
    config={
        "tags": ["test", "quantum"],           # 추적용 태그
        "metadata": {"user_id": "user_123"},   # 메타데이터
        "max_concurrency": 5,                  # 동시 실행 제한
    }
)

# 방법 2: with_config로 설정이 고정된 새 체인 생성
tagged_chain = chain.with_config(
    tags=["production"],
    metadata={"environment": "prod"}
)

# 이후 invoke 시 항상 해당 설정이 적용됨
result = tagged_chain.invoke({"topic": "양자컴퓨팅"})
```

`with_config()`와 `bind()`의 차이가 헷갈릴 수 있는데요, 핵심 차이는 이렇습니다:

| 구분 | `bind()` | `with_config()` |
|------|----------|-----------------|
| 대상 | Runnable의 **invoke 인자** (stop, tools 등) | **실행 환경 설정** (tags, metadata 등) |
| 시점 | 체인 정의 시 고정 | 체인 정의 또는 호출 시 |
| 예시 | `model.bind(stop=["END"])` | `chain.with_config(tags=["v2"])` |

### 개념 3: with_retry() — 자동 재시도로 안정성 확보

> 💡 **비유**: 전화를 걸었는데 상대방이 통화 중이면 어떻게 하나요? 바로 포기하지 않고, 잠시 후 다시 걸어봅니다. 두 번째도 안 되면 좀 더 기다렸다가 다시 시도하죠. `with_retry()`가 정확히 이 패턴 — **지수 백오프(Exponential Backoff)**로 자동 재시도합니다.

LLM API 호출은 네트워크 문제, 레이트 리밋, 일시적 서버 오류 등으로 실패할 수 있습니다. `with_retry()`는 이런 **일시적 오류(transient error)**에 대해 자동으로 재시도 로직을 추가합니다.

```python
import time
from langchain_core.runnables import RunnableLambda

# 처음 2번은 실패하고, 3번째에 성공하는 함수
call_count = 0

def flaky_api_call(input: str) -> str:
    """불안정한 외부 API를 시뮬레이션"""
    global call_count
    call_count += 1
    if call_count < 3:
        raise ConnectionError(f"시도 {call_count}: 연결 실패!")
    return f"성공! (시도 {call_count}회 만에)"

# with_retry 적용
runnable = RunnableLambda(flaky_api_call).with_retry(
    retry_if_exception_type=(ConnectionError,),  # 이 예외만 재시도
    wait_exponential_jitter=True,                 # 지수 백오프 + 랜덤 지터
    stop_after_attempt=5,                         # 최대 5번 시도
)

result = runnable.invoke("테스트")
print(result)  # "성공! (시도 3회 만에)"
```

**핵심 파라미터 정리**:

| 파라미터 | 설명 | 기본값 |
|----------|------|--------|
| `retry_if_exception_type` | 재시도할 예외 타입 튜플 | `(Exception,)` |
| `wait_exponential_jitter` | 지수 백오프 + 랜덤 지터 사용 여부 | `False` |
| `stop_after_attempt` | 최대 시도 횟수 | `3` |

> 🔥 **실무 팁**: `with_retry()`의 범위는 **최대한 좁게** 유지하세요. 전체 체인에 retry를 걸면 이미 성공한 단계까지 다시 실행됩니다. 실패 가능성이 높은 **API 호출 부분에만** 개별적으로 적용하는 것이 베스트 프랙티스입니다.

### 개념 4: with_fallbacks() — 대체 체인으로 우아한 실패 처리

> 💡 **비유**: 비행기 엔진이 하나 고장 나도 다른 엔진으로 비행을 계속할 수 있는 것처럼, `with_fallbacks()`는 메인 체인이 실패하면 **예비 체인**으로 자동 전환합니다.

`with_retry()`가 "같은 것을 다시 시도"라면, `with_fallbacks()`는 "다른 것을 시도"입니다. 메인 모델이 다운되면 백업 모델로 전환하거나, 복잡한 체인이 실패하면 단순한 체인으로 폴백하는 패턴이죠.

```python
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser
from langchain_openai import ChatOpenAI

prompt = ChatPromptTemplate.from_template("{topic}에 대해 설명해주세요.")
parser = StrOutputParser()

# 메인 모델 (GPT-4o)
main_model = ChatOpenAI(model="gpt-4o", temperature=0)
# 백업 모델 (GPT-4o-mini — 더 저렴하고 안정적)
backup_model = ChatOpenAI(model="gpt-4o-mini", temperature=0)

# 메인 체인 + 폴백 체인 구성
main_chain = prompt | main_model | parser
backup_chain = prompt | backup_model | parser

# with_fallbacks로 연결
resilient_chain = main_chain.with_fallbacks([backup_chain])

# main_chain이 실패하면 자동으로 backup_chain 실행
result = resilient_chain.invoke({"topic": "LangChain"})
print(result)
```

`with_retry()`와 `with_fallbacks()`를 **조합**하면 더욱 견고한 체인을 만들 수 있습니다:

```python
# 재시도 후에도 실패하면 → 폴백으로 전환
robust_chain = (
    prompt
    | main_model.with_retry(
        stop_after_attempt=3,
        wait_exponential_jitter=True
    )
    | parser
).with_fallbacks([backup_chain])
```

### 개념 5: with_listeners() — 실행 이벤트 관찰

> 💡 **비유**: CCTV 카메라가 매장의 입구(시작), 출구(종료), 비상구(에러)를 감시하듯, `with_listeners()`는 체인 실행의 **시작, 종료, 에러** 이벤트를 실시간으로 감시합니다.

`with_listeners()`는 Runnable의 실행 생명주기에 콜백을 바인딩합니다. [5.4 병렬 실행과 성능 최적화](./04-병렬-실행과-성능-최적화.md)에서 배운 비동기 실행과 결합하면, 실행 시간 측정, 로깅, 모니터링 등을 체인 로직과 **분리하여** 관리할 수 있습니다.

```python
import time
from langchain_core.runnables import RunnableLambda
from langchain_core.tracers.schemas import Run

# 리스너 함수 정의
def on_start(run: Run):
    """체인 실행 시작 시 호출"""
    print(f"🟢 시작: {run.name} | 입력: {run.inputs}")

def on_end(run: Run):
    """체인 실행 완료 시 호출"""
    elapsed = (run.end_time - run.start_time).total_seconds()
    print(f"🔵 완료: {run.name} | 소요: {elapsed:.2f}초")

def on_error(run: Run):
    """체인 실행 에러 시 호출"""
    print(f"🔴 에러: {run.name} | {run.error}")

# 리스너가 부착된 Runnable
def process_text(text: str) -> str:
    """텍스트를 대문자로 변환"""
    time.sleep(0.5)  # 처리 시간 시뮬레이션
    return text.upper()

monitored_runnable = RunnableLambda(process_text).with_listeners(
    on_start=on_start,
    on_end=on_end,
    on_error=on_error,
)

result = monitored_runnable.invoke("hello langchain")
# 🟢 시작: process_text | 입력: hello langchain
# 🔵 완료: process_text | 소요: 0.50초
print(result)  # HELLO LANGCHAIN
```

### 개념 6: configurable_fields()와 configurable_alternatives() — 런타임 동적 교체

> 💡 **비유**: 자동차의 **주행 모드**와 같습니다. "에코 모드"에서는 연비 중심, "스포츠 모드"에서는 출력 중심 — 같은 차인데 설정만 바꿔서 완전히 다른 특성을 낼 수 있죠. `configurable_fields()`는 파라미터 값을 바꾸고, `configurable_alternatives()`는 아예 컴포넌트 자체를 교체합니다.

`configurable_fields()`는 Runnable의 **특정 필드 값**을 런타임에 변경할 수 있게 해줍니다. 체인을 재정의하지 않고도 temperature, model_name 등을 호출 시점에 바꿀 수 있습니다.

```python
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser
from langchain_core.runnables import ConfigurableField
from langchain_openai import ChatOpenAI

# temperature를 configurable로 선언
model = ChatOpenAI(model="gpt-4o", temperature=0).configurable_fields(
    temperature=ConfigurableField(
        id="llm_temperature",           # 설정 시 사용할 키
        name="LLM Temperature",         # 사람이 읽는 이름
        description="생성의 창의성 수준을 조절합니다",
    )
)

prompt = ChatPromptTemplate.from_template("{topic}에 대한 시를 써주세요.")
chain = prompt | model | StrOutputParser()

# 기본값(temperature=0)으로 실행 — 정확하고 일관된 응답
result_precise = chain.invoke({"topic": "봄"})

# temperature=0.9로 변경하여 실행 — 창의적인 응답
result_creative = chain.invoke(
    {"topic": "봄"},
    config={"configurable": {"llm_temperature": 0.9}}
)
```

`configurable_alternatives()`는 더 나아가 **컴포넌트 자체를 통째로 교체**합니다. A/B 테스트나 멀티 프로바이더 전략에 매우 유용합니다.

```python
from langchain_core.runnables import ConfigurableField
from langchain_openai import ChatOpenAI

# 기본 모델 + 대안 모델 설정
model = ChatOpenAI(model="gpt-4o", temperature=0).configurable_alternatives(
    ConfigurableField(id="llm_provider"),
    default_key="openai_gpt4o",
    openai_mini=ChatOpenAI(model="gpt-4o-mini", temperature=0),
    # anthropic=ChatAnthropic(model="claude-sonnet-4-20250514"),  # 다른 프로바이더도 가능
)

prompt = ChatPromptTemplate.from_template("{question}")
chain = prompt | model | StrOutputParser()

# 기본 모델(GPT-4o)로 실행
result_gpt4 = chain.invoke({"question": "LangChain이란?"})

# GPT-4o-mini로 전환하여 실행
result_mini = chain.invoke(
    {"question": "LangChain이란?"},
    config={"configurable": {"llm_provider": "openai_mini"}}
)
```

## 실습: 직접 해보기

모든 개념을 종합하여 **프로덕션 수준의 다국어 번역 체인**을 만들어봅시다. 모델, temperature, 재시도, 폴백, 모니터링을 모두 갖춘 완전한 예제입니다.

```python
"""
LCEL 체인 구성과 바인딩 — 종합 실습
프로덕션 수준의 다국어 번역 체인 구축
"""
import time
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser
from langchain_core.runnables import (
    ConfigurableField,
    RunnableLambda,
    RunnablePassthrough,
)
from langchain_openai import ChatOpenAI
from langchain_core.tracers.schemas import Run

# ─── 1단계: 모니터링 리스너 정의 ───
def log_start(run: Run):
    """실행 시작 로깅"""
    print(f"📝 [{run.name}] 시작 | 태그: {run.tags}")

def log_end(run: Run):
    """실행 완료 로깅"""
    elapsed = (run.end_time - run.start_time).total_seconds()
    print(f"✅ [{run.name}] 완료 | 소요: {elapsed:.2f}초")

def log_error(run: Run):
    """실행 에러 로깅"""
    print(f"❌ [{run.name}] 에러: {run.error}")

# ─── 2단계: 모델 설정 (configurable_fields + configurable_alternatives) ───
# temperature를 런타임에 조절 가능하게 설정
main_model = ChatOpenAI(
    model="gpt-4o",
    temperature=0.3
).configurable_fields(
    temperature=ConfigurableField(
        id="translation_temperature",
        name="번역 Temperature",
        description="번역의 자연스러움 수준 (0: 직역, 1: 의역)",
    )
)

# 백업 모델
backup_model = ChatOpenAI(model="gpt-4o-mini", temperature=0.3)

# ─── 3단계: 프롬프트 + 파서 구성 ───
translate_prompt = ChatPromptTemplate.from_messages([
    ("system",
     "당신은 전문 번역가입니다. {source_lang}를 {target_lang}로 번역하세요.\n"
     "규칙:\n"
     "- 자연스러운 {target_lang} 표현을 사용하세요\n"
     "- 기술 용어는 원문을 괄호 안에 병기하세요\n"
     "- 번역문만 출력하세요"),
    ("human", "{text}")
])
parser = StrOutputParser()

# ─── 4단계: 메인 체인 조립 ───
# 전처리: 입력에 기본값 추가
def add_defaults(input_dict: dict) -> dict:
    """소스 언어가 없으면 자동 감지로 설정"""
    return {
        "source_lang": input_dict.get("source_lang", "자동 감지"),
        "target_lang": input_dict.get("target_lang", "한국어"),
        "text": input_dict["text"],
    }

# 메인 체인: 전처리 → 프롬프트 → 모델(재시도 포함) → 파서
main_chain = (
    RunnableLambda(add_defaults)
    | translate_prompt
    | main_model.with_retry(          # API 실패 시 자동 재시도
        retry_if_exception_type=(ConnectionError, TimeoutError),
        wait_exponential_jitter=True,
        stop_after_attempt=3,
    ).with_listeners(                 # 실행 모니터링
        on_start=log_start,
        on_end=log_end,
        on_error=log_error,
    )
    | parser
)

# 백업 체인
backup_chain = (
    RunnableLambda(add_defaults)
    | translate_prompt
    | backup_model
    | parser
)

# ─── 5단계: 폴백 연결로 최종 체인 완성 ───
translation_chain = main_chain.with_fallbacks([backup_chain])

# ─── 실행 예제 ───
if __name__ == "__main__":
    # 기본 설정으로 번역
    result1 = translation_chain.invoke(
        {"text": "LangChain is a framework for developing applications powered by LLMs."},
        config={"tags": ["translation", "en-ko"]}
    )
    print(f"\n📌 기본 번역:\n{result1}\n")

    # temperature를 올려서 더 자연스러운 의역
    result2 = translation_chain.invoke(
        {
            "text": "The quick brown fox jumps over the lazy dog.",
            "source_lang": "영어",
            "target_lang": "일본어",
        },
        config={
            "configurable": {"translation_temperature": 0.8},
            "tags": ["translation", "en-ja"],
        }
    )
    print(f"📌 의역 번역 (temperature=0.8):\n{result2}\n")
```

**실행 결과** (예시):
```
📝 [ChatOpenAI] 시작 | 태그: ['translation', 'en-ko']
✅ [ChatOpenAI] 완료 | 소요: 1.23초

📌 기본 번역:
LangChain은 LLM(대규모 언어 모델)으로 구동되는 애플리케이션을 개발하기 위한 프레임워크입니다.

📝 [ChatOpenAI] 시작 | 태그: ['translation', 'en-ja']
✅ [ChatOpenAI] 완료 | 소요: 1.45초

📌 의역 번역 (temperature=0.8):
素早い茶色の狐が怠け者の犬を飛び越える。
```

## 더 깊이 알아보기

### LCEL 바인딩 패턴의 탄생 배경

LangChain의 창시자 Harrison Chase는 2022년 10월, 단 9일 만에 LangChain의 첫 버전을 작성했습니다. 하버드 대학에서 통계학과 컴퓨터공학을 복수전공한 그는, Kensho Technologies에서 비정형 텍스트와 구조화된 지식 그래프를 연결하는 작업을 하면서 "LLM 애플리케이션의 반복적인 패턴"을 체감했습니다.

그런데 2023년 여름, LangChain에 대한 부정적 피드백이 쏟아졌습니다. 사용자들이 가장 많이 요구한 것은 **더 세밀한 제어권**이었죠. 기존 방식은 체인의 동작을 바꾸려면 체인 자체를 재정의해야 했거든요.

이 피드백에 대한 직접적인 답이 바로 2023년 3분기에 등장한 **LCEL**이었습니다. `bind()`, `with_config()`, `configurable_fields()` 같은 메서드들은 "체인은 그대로 두되, 동작만 바꾼다"는 **데코레이터 패턴**의 철학을 반영합니다. 실제로 LangChain 소스 코드에서 `bind()`가 반환하는 `RunnableBinding`은 원본 Runnable을 감싸는 래퍼로, 원본의 invoke/stream/batch 인터페이스를 보존하면서 추가 인자를 주입합니다.

Chase는 이를 "조합 가능성(composability)을 첫 번째 원칙으로"라고 표현했는데, 모놀리식 솔루션 대신 작은 모듈을 유연하게 조립하는 설계 철학이 LCEL의 모든 메서드에 녹아 있습니다.

### RunnableBinding의 내부 구조

`bind()`가 반환하는 `RunnableBinding`은 사실 **프록시 패턴**의 구현입니다:

```
┌─────────────────────────────────┐
│       RunnableBinding           │
│  ┌───────────┐ ┌─────────────┐  │
│  │  bound     │ │  kwargs     │  │
│  │ (원본      │ │ (고정된    │  │
│  │  Runnable) │ │  인자들)   │  │
│  └───────────┘ └─────────────┘  │
│                                 │
│  invoke(input) 호출 시:          │
│  → bound.invoke(input, **kwargs)│
└─────────────────────────────────┘
```

`with_config()`, `with_retry()`, `with_listeners()` 모두 내부적으로 `RunnableBinding`을 생성하거나 확장합니다. 이것이 이 메서드들을 **체이닝**할 수 있는 이유입니다:

```python
# 모두 RunnableBinding 기반이므로 자유롭게 조합 가능
chain = (
    model
    .bind(stop=["END"])
    .with_retry(stop_after_attempt=3)
    .with_listeners(on_end=log_end)
    .with_config(tags=["production"])
)
```

## 흔한 오해와 팁

> ⚠️ **흔한 오해**: "`bind()`와 `with_config()`는 같은 것이다"
> 
> 아닙니다! `bind()`는 Runnable의 **invoke 메서드에 전달될 키워드 인자**를 고정합니다 (예: `stop`, `tools`). 반면 `with_config()`는 **실행 환경 설정**(tags, metadata, max_concurrency, configurable 등)을 전달합니다. 둘은 서로 다른 레이어에서 동작합니다.

> 💡 **알고 계셨나요?**: `with_retry()`의 지수 백오프는 내부적으로 Python의 `tenacity` 라이브러리를 사용합니다. `wait_exponential_jitter=True`를 설정하면 기본 대기 시간에 랜덤한 지터(jitter)가 추가되는데, 이는 여러 클라이언트가 동시에 재시도할 때 발생하는 "Thundering Herd" 문제(한꺼번에 몰리는 재시도)를 방지하기 위한 것입니다. AWS의 분산 시스템 설계에서 유래한 패턴이죠.

> 🔥 **실무 팁**: `configurable_fields()`와 `configurable_alternatives()`는 **LangServe와 궁합이 매우 좋습니다**. LangServe로 체인을 REST API로 배포하면, API 호출 시 `config` 파라미터로 모델 전환, temperature 조절 등을 클라이언트가 직접 제어할 수 있습니다. 코드 한 줄 수정 없이 A/B 테스트가 가능해지는 거죠. Ch17에서 LangServe를 본격적으로 다룰 때 이 패턴을 다시 만나게 될 것입니다.

> ⚠️ **흔한 오해**: "`with_retry()`를 전체 체인에 걸면 안전하다"
> 
> 주의하세요! 전체 체인에 retry를 걸면, 이미 성공한 앞 단계의 Runnable까지 처음부터 다시 실행됩니다. LLM API 호출처럼 **비용이 발생하는 단계**가 불필요하게 재실행될 수 있어요. retry는 반드시 **실패 가능성이 높은 개별 컴포넌트**에만 적용하세요.

## 핵심 정리

| 개념 | 설명 | 주요 용도 |
|------|------|-----------|
| `bind()` | Runnable에 기본 invoke 인자를 고정 | stop 시퀀스, tool binding |
| `with_config()` | 런타임 실행 환경 설정 전달 | tags, metadata, max_concurrency |
| `with_retry()` | 실패 시 자동 재시도 (지수 백오프) | API 일시 장애 대응 |
| `with_fallbacks()` | 실패 시 대체 체인으로 전환 | 모델 폴백, 다중 전략 |
| `with_listeners()` | 실행 생명주기 이벤트 관찰 | 모니터링, 로깅, 시간 측정 |
| `configurable_fields()` | 필드 값을 런타임에 동적 변경 | temperature, model 파라미터 |
| `configurable_alternatives()` | 컴포넌트 자체를 런타임에 교체 | A/B 테스트, 멀티 프로바이더 |
| `RunnableBinding` | 위 메서드들의 내부 구현체 (프록시 패턴) | 원본 보존 + 기능 추가 |
