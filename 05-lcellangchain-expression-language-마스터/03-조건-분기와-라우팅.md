# 조건 분기와 라우팅

> LCEL에서 입력에 따라 서로 다른 체인을 선택적으로 실행하는 조건 분기와 동적 라우팅 패턴을 학습합니다.

## 개요

이 섹션에서는 LCEL의 체인이 항상 같은 경로를 따르지 않아도 되는 방법을 배웁니다. 사용자 입력에 따라 서로 다른 처리 경로로 분기하고, LLM이 직접 경로를 결정하는 동적 라우팅까지 구현해보겠습니다.

**선수 지식**: [Session 5.1: LCEL 기초와 파이프 연산자](ch05/session_5_1.md)에서 배운 Runnable 프로토콜과 파이프 연산자, [Session 5.2: 핵심 Runnable 컴포넌트](ch05/session_5_2.md)에서 배운 RunnablePassthrough, RunnableLambda, RunnableParallel

**학습 목표**:
- `RunnableBranch`를 사용하여 조건에 따라 다른 체인을 실행할 수 있다
- `RunnableLambda`를 활용한 커스텀 함수 라우팅을 구현할 수 있다
- LLM 기반 의도 분류와 동적 라우팅 파이프라인을 설계할 수 있다
- 두 가지 라우팅 방식의 차이점과 적절한 사용 시점을 판단할 수 있다

## 왜 알아야 할까?

실제 LLM 애플리케이션을 생각해보세요. 사용자가 코드 질문을 하면 코딩 전문가 프롬프트로, 수학 질문을 하면 수학 전문가 프롬프트로, 일상적인 대화라면 범용 프롬프트로 처리해야 합니다. 챗봇 하나가 모든 질문에 같은 방식으로 답하는 건 비효율적이거든요.

이것은 웹 서버의 URL 라우팅과 본질적으로 같은 문제입니다. Flask나 Django에서 `/api/users`와 `/api/products`를 다른 핸들러로 보내는 것처럼, LCEL에서도 입력의 성격에 따라 다른 체인으로 보내야 할 때가 있습니다. RAG 시스템에서 질문 유형별로 다른 검색 전략을 적용하거나, 에이전트가 도구를 선택하는 것도 결국 라우팅의 한 형태입니다.

조건 분기와 라우팅은 단순한 선형 체인에서 벗어나, **적응적이고 지능적인 파이프라인**을 만드는 핵심 기술입니다.

## 핵심 개념

### 개념 1: RunnableBranch — 선언적 조건 분기

> 💡 **비유**: 공항 입국장의 심사대를 떠올려보세요. 여권 색깔에 따라 "내국인 심사대", "외국인 심사대", "외교관 심사대"로 안내됩니다. 조건(여권 색깔)을 순서대로 확인하고, 첫 번째로 해당되는 곳으로 보내는 거죠. 어디에도 해당하지 않으면? "일반 심사대"로 갑니다. `RunnableBranch`가 정확히 이렇게 동작합니다.

`RunnableBranch`는 `(조건 함수, 실행할 Runnable)` 튜플의 리스트를 받고, 마지막 인자로 기본(default) Runnable을 받습니다. 입력이 들어오면 조건을 **위에서부터 순서대로** 평가하고, **처음으로 `True`가 되는** 조건의 Runnable을 실행합니다. 어떤 조건도 만족하지 않으면 기본 Runnable이 실행됩니다.

```python
from langchain_core.runnables import RunnableBranch

# 기본 구조: (조건, 실행할 Runnable) 튜플들 + 기본 Runnable
branch = RunnableBranch(
    (lambda x: isinstance(x, str), lambda x: x.upper()),   # 문자열이면 대문자로
    (lambda x: isinstance(x, int), lambda x: x * 2),       # 정수면 2배로
    (lambda x: isinstance(x, float), lambda x: x + 0.5),   # 실수면 0.5 더하기
    lambda x: "알 수 없는 타입입니다",                         # 기본: 안내 메시지
)

print(branch.invoke("hello"))   # 출력: HELLO
print(branch.invoke(10))        # 출력: 20
print(branch.invoke(3.14))      # 출력: 3.64
print(branch.invoke(None))      # 출력: 알 수 없는 타입입니다
```

핵심 규칙을 정리하면:
- 조건은 **위에서 아래로** 순서대로 평가됩니다 (첫 번째 `True`에서 멈춤)
- 마지막 인자는 반드시 **기본 Runnable**이어야 합니다 (조건 없이 단독)
- 최소 **2개의 인자**가 필요합니다 (조건-Runnable 쌍 1개 + 기본 1개)

이제 LLM 체인에서 실제로 활용해봅시다:

```python
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser
from langchain_core.runnables import RunnableBranch
from langchain_openai import ChatOpenAI

llm = ChatOpenAI(model="gpt-4o", temperature=0.7)

# 분야별 전문가 체인 정의
code_chain = (
    ChatPromptTemplate.from_template(
        "당신은 시니어 소프트웨어 엔지니어입니다. "
        "다음 코딩 질문에 코드와 함께 답변하세요:\n\n{input}"
    )
    | llm
    | StrOutputParser()
)

math_chain = (
    ChatPromptTemplate.from_template(
        "당신은 수학 교수입니다. "
        "다음 수학 문제를 단계별로 풀어주세요:\n\n{input}"
    )
    | llm
    | StrOutputParser()
)

general_chain = (
    ChatPromptTemplate.from_template(
        "다음 질문에 친절하게 답변하세요:\n\n{input}"
    )
    | llm
    | StrOutputParser()
)

# RunnableBranch로 라우팅
branch = RunnableBranch(
    # 조건: 입력 딕셔너리의 "input" 키에 특정 키워드가 포함되어 있는지 확인
    (lambda x: any(kw in x["input"] for kw in ["코드", "함수", "파이썬", "프로그래밍"]), code_chain),
    (lambda x: any(kw in x["input"] for kw in ["계산", "수학", "방정식", "미분"]), math_chain),
    general_chain,  # 기본 체인
)

# 실행
result = branch.invoke({"input": "파이썬으로 퀵소트를 구현해주세요"})
print(result)  # → 코딩 전문가 답변
```

### 개념 2: RunnableLambda 커스텀 함수 라우팅 — 권장 방식

> 💡 **비유**: 이번에는 레스토랑의 숙련된 접수 직원을 상상해보세요. 손님이 "스테이크 먹고 싶어요"라고 하면 양식 코너로, "초밥 먹고 싶어요"라면 일식 코너로, 결정을 못 하면 추천 코너로 안내합니다. 미리 정해진 규칙표(RunnableBranch)가 아니라, 접수 직원의 **판단 로직(함수)**이 라우팅을 결정하는 거죠. 이게 바로 커스텀 함수 라우팅입니다.

LangChain 공식 문서에서는 `RunnableBranch`보다 **커스텀 함수 라우팅**을 더 권장합니다. 이유는 간단한데요 — 더 유연하고, Python의 자연스러운 제어 흐름을 그대로 활용할 수 있기 때문입니다. 핵심 아이디어는 **Runnable을 반환하는 함수**를 `RunnableLambda`로 감싸는 것입니다.

```python
from langchain_core.runnables import RunnableLambda, RunnablePassthrough

# 라우팅 함수: Runnable을 "반환"합니다 (실행하지 않음!)
def route(info: dict) -> Runnable:
    """입력의 topic 필드를 보고 적절한 체인을 반환합니다."""
    topic = info.get("topic", "")
    if "코드" in topic or "프로그래밍" in topic:
        return code_chain
    elif "수학" in topic or "계산" in topic:
        return math_chain
    else:
        return general_chain

# 전체 파이프라인 구성
full_chain = (
    {"topic": lambda x: x["input"], "input": lambda x: x["input"]}
    | RunnableLambda(route)  # 반환된 Runnable이 자동으로 실행됨
)

result = full_chain.invoke({"input": "파이썬으로 퀵소트를 구현해주세요"})
print(result)
```

여기서 핵심은 `route` 함수가 체인을 **실행하는 것이 아니라 반환**한다는 점입니다. LCEL이 반환된 Runnable을 자동으로 `invoke`해줍니다. 이 패턴이 강력한 이유는 다음과 같습니다:

1. **Python 제어 흐름 활용**: `if/elif/else`, `match/case`, 딕셔너리 매핑 등 원하는 방식으로 라우팅 로직을 작성할 수 있습니다
2. **타입 검증 용이**: 함수 시그니처에 타입 힌트를 달 수 있고, 테스트도 쉽습니다
3. **성능 이점**: `RunnableBranch`는 모든 조건을 순차 평가하면서 콜백 오버헤드가 발생하지만, 커스텀 함수는 단순 분기입니다

> ⚠️ **흔한 오해**: `route` 함수 안에서 `code_chain.invoke(info)` 처럼 직접 실행하면 안 됩니다! 체인을 **반환만** 해야 합니다. LCEL이 반환된 Runnable을 자동으로 실행하기 때문에, 직접 실행하면 이중 실행이 되어 예상치 못한 결과가 발생합니다.

### 개념 3: LLM 기반 동적 라우팅

> 💡 **비유**: 키워드 매칭은 도서관의 분류 번호 시스템과 같습니다 — "500번대는 과학", "800번대는 문학"처럼 미리 정해진 규칙이죠. 하지만 "양자역학이 철학에 미친 영향"이라는 책은 과학일까요, 철학일까요? 이런 애매한 경우에는 **사서(LLM)**에게 물어봐야 합니다. LLM이 입력을 분석하고, 가장 적합한 카테고리를 판단하는 거죠.

키워드 기반 조건 분기는 간단하지만 한계가 있습니다. "머신러닝으로 주식 가격 예측하는 코드 짜줘"는 코드 질문일까요, 수학 질문일까요? 이런 경우 LLM이 직접 의도를 분류하는 **동적 라우팅**이 필요합니다.

```python
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser
from langchain_core.runnables import RunnableLambda, RunnablePassthrough
from langchain_openai import ChatOpenAI

llm = ChatOpenAI(model="gpt-4o", temperature=0)

# 1단계: 의도 분류 체인
classifier_prompt = ChatPromptTemplate.from_template(
    "다음 사용자 질문을 분류하세요. "
    "반드시 'code', 'math', 'general' 중 하나만 출력하세요.\n\n"
    "질문: {input}\n분류:"
)
classifier_chain = classifier_prompt | llm | StrOutputParser()

# 2단계: 분류 결과에 따른 라우팅 함수
def route_by_classification(info: dict):
    """분류 결과를 바탕으로 적절한 체인을 반환합니다."""
    topic = info["topic"].strip().lower()
    if topic == "code":
        return code_chain
    elif topic == "math":
        return math_chain
    else:
        return general_chain

# 3단계: 전체 파이프라인 조립
full_chain = (
    # 원본 입력을 유지하면서 분류 결과를 추가
    RunnablePassthrough.assign(
        topic=lambda x: classifier_chain.invoke(x)
    )
    | RunnableLambda(route_by_classification)
)

# 실행
result = full_chain.invoke({"input": "머신러닝으로 주식 가격 예측하는 코드 짜줘"})
print(result)  # → LLM이 "code"로 분류 → 코딩 전문가 답변
```

이 패턴의 데이터 흐름을 따라가보겠습니다:

| 단계 | 데이터 형태 | 설명 |
|------|------------|------|
| 입력 | `{"input": "머신러닝으로 주식 가격..."}` | 사용자 원본 질문 |
| `assign` 후 | `{"input": "머신러닝으로...", "topic": "code"}` | 분류 결과 추가 |
| `route` 후 | `code_chain` (Runnable 객체) | 라우팅 함수가 체인 반환 |
| 최종 실행 | `"코딩 전문가의 답변..."` | 선택된 체인이 원본 입력으로 실행 |

### 개념 4: 딕셔너리 매핑 라우터 패턴

분기가 많아지면 `if/elif` 체인도 길어지겠죠. 이런 경우 **딕셔너리 매핑** 패턴이 깔끔합니다.

```python
from langchain_core.runnables import RunnableLambda

# 카테고리별 체인을 딕셔너리에 등록
chain_map: dict[str, Runnable] = {
    "code": code_chain,
    "math": math_chain,
    "science": science_chain,
    "history": history_chain,
}

# 기본 체인
default_chain = general_chain

def route_with_map(info: dict):
    """딕셔너리에서 체인을 찾아 반환합니다."""
    topic = info["topic"].strip().lower()
    return chain_map.get(topic, default_chain)

# 파이프라인
full_chain = (
    RunnablePassthrough.assign(
        topic=lambda x: classifier_chain.invoke(x)
    )
    | RunnableLambda(route_with_map)
)
```

이 패턴의 장점은 명확합니다:
- **확장성**: 새 카테고리 추가 시 딕셔너리에 한 줄만 추가하면 됩니다
- **가독성**: 어떤 카테고리가 어떤 체인으로 가는지 한눈에 보입니다
- **유지보수**: 라우팅 로직과 체인 정의가 깔끔하게 분리됩니다

## 실습: 직접 해보기

다국어 고객 지원 챗봇의 라우팅 시스템을 구현해봅시다. 사용자의 질문을 분석해서 **언어 감지**, **카테고리 분류**, **적절한 전문가 체인 라우팅**까지 하나의 파이프라인으로 구성합니다.

```python
"""
다국어 고객 지원 라우팅 시스템
LLM이 언어와 카테고리를 감지하여 적절한 전문가 체인으로 라우팅합니다.
"""
import os
from dotenv import load_dotenv
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser
from langchain_core.runnables import (
    RunnableLambda,
    RunnablePassthrough,
    RunnableParallel,
)
from langchain_openai import ChatOpenAI

# 환경 변수 로드
load_dotenv()

# 모델 설정
MODEL_NAME = "gpt-4o"
llm = ChatOpenAI(model=MODEL_NAME, temperature=0)
creative_llm = ChatOpenAI(model=MODEL_NAME, temperature=0.7)

# ─── 1단계: 의도 분류기 ───
classifier_prompt = ChatPromptTemplate.from_template(
    "사용자 메시지를 분석하세요.\n\n"
    "카테고리: billing(결제), technical(기술), refund(환불), general(일반)\n\n"
    "반드시 카테고리 하나만 영어 소문자로 출력하세요.\n\n"
    "메시지: {input}\n카테고리:"
)
classifier_chain = classifier_prompt | llm | StrOutputParser()

# ─── 2단계: 카테고리별 전문가 체인 ───
billing_chain = (
    ChatPromptTemplate.from_template(
        "당신은 결제 전문 상담사입니다. 결제 관련 정책을 숙지하고 있습니다.\n"
        "다음 고객 문의에 정확하고 친절하게 답변하세요.\n\n"
        "문의: {input}"
    )
    | llm
    | StrOutputParser()
)

technical_chain = (
    ChatPromptTemplate.from_template(
        "당신은 기술 지원 엔지니어입니다. 제품의 기술적 문제를 해결합니다.\n"
        "다음 기술 문의에 단계별로 해결 방법을 안내하세요.\n\n"
        "문의: {input}"
    )
    | llm
    | StrOutputParser()
)

refund_chain = (
    ChatPromptTemplate.from_template(
        "당신은 환불 처리 담당자입니다. 환불 정책에 따라 안내합니다.\n"
        "환불 가능 여부를 확인하고 절차를 안내하세요.\n\n"
        "문의: {input}"
    )
    | llm
    | StrOutputParser()
)

general_chain = (
    ChatPromptTemplate.from_template(
        "당신은 친절한 고객 상담사입니다.\n"
        "다음 문의에 도움이 되는 답변을 해주세요.\n\n"
        "문의: {input}"
    )
    | creative_llm
    | StrOutputParser()
)

# ─── 3단계: 딕셔너리 매핑 라우터 ───
chain_map: dict[str, any] = {
    "billing": billing_chain,
    "technical": technical_chain,
    "refund": refund_chain,
    "general": general_chain,
}

def route_to_expert(info: dict):
    """분류 결과에 따라 전문가 체인을 반환합니다."""
    category = info["category"].strip().lower()
    selected = chain_map.get(category, general_chain)
    print(f"[라우터] 카테고리: {category} → {category}_chain 선택")
    return selected

# ─── 4단계: 전체 파이프라인 조립 ───
support_pipeline = (
    # 원본 입력을 유지하면서 카테고리 분류 결과를 추가
    RunnablePassthrough.assign(
        category=lambda x: classifier_chain.invoke(x)
    )
    # 분류 결과에 따라 적절한 전문가 체인으로 라우팅
    | RunnableLambda(route_to_expert)
)

# ─── 실행 ───
test_queries = [
    "지난달 카드 결제가 두 번 됐어요",
    "앱이 자꾸 튕겨요. iOS 17인데 호환 문제인가요?",
    "제품 불량으로 환불 받고 싶어요",
    "영업시간이 어떻게 되나요?",
]

for query in test_queries:
    print(f"\n{'='*60}")
    print(f"질문: {query}")
    print(f"{'='*60}")
    result = support_pipeline.invoke({"input": query})
    print(f"답변: {result[:200]}...")  # 앞부분만 출력
```

## 더 깊이 알아보기

### RunnableBranch의 탄생과 "함수 라우팅 권장"의 배경

`RunnableBranch`는 LangChain이 레거시 `Chain` 클래스에서 LCEL로 전환하던 2023년 하반기에 도입되었습니다. 당시 `LLMRouterChain`이라는 클래스가 있었는데, 이것은 LCEL 이전의 명령형(imperative) 방식으로 라우팅을 처리했습니다. LCEL의 선언적(declarative) 철학에 맞춰 `RunnableBranch`가 설계된 거죠.

그런데 흥미로운 일이 일어났습니다. LangChain 커뮤니티에서 `RunnableBranch`가 `RunnableLambda` 기반 커스텀 함수보다 **약 0.5~1초 느리다**는 벤치마크 결과가 보고되었습니다. 이유는 `RunnableBranch`가 내부적으로 각 조건 평가마다 콜백 시스템을 거치기 때문이었습니다. 이후 LangChain 공식 문서에서도 "커스텀 함수로 Runnable을 반환하는 방식을 권장한다"는 가이드가 추가되었습니다.

이것은 소프트웨어 설계에서 흔히 볼 수 있는 패턴입니다 — **선언적 추상화가 항상 최선은 아니다**는 것이죠. 때로는 언어 자체의 제어 흐름(`if/else`)이 프레임워크의 추상화보다 더 효율적이고 읽기 쉬울 수 있습니다. Python의 "Explicit is better than implicit" 철학과도 맞닿아 있죠.

### 라우팅 패턴의 기원: Content-Based Router

사실 조건 분기 라우팅은 LLM 세계에서 새로 나온 개념이 아닙니다. 2003년 Gregor Hohpe와 Bobby Woolf가 쓴 *Enterprise Integration Patterns*에서 이미 **Content-Based Router** 패턴을 정의했습니다. "메시지의 내용을 검사하여 적절한 채널로 라우팅하는" 이 패턴은, 20년이 지난 지금 LLM 파이프라인에서 거의 동일한 형태로 되살아난 셈입니다. 차이라면 라우팅 조건을 **규칙 기반**이 아닌 **LLM 기반**으로 판단할 수 있다는 점이겠죠.

## 흔한 오해와 팁

> ⚠️ **흔한 오해**: "RunnableBranch가 LCEL의 공식 라우팅 방법이니까 항상 RunnableBranch를 써야 한다." 사실이 아닙니다! LangChain 공식 문서에서도 **커스텀 함수로 Runnable을 반환하는 방식**을 더 권장합니다. `RunnableBranch`는 여전히 유효하지만, 복잡한 라우팅 로직에서는 Python의 `if/elif/else`나 딕셔너리 매핑이 더 읽기 쉽고 빠릅니다.

> 💡 **알고 계셨나요?**: 라우팅 함수에서 Runnable을 **반환**하면 LCEL이 자동으로 그 Runnable을 실행해줍니다. 이것은 LCEL의 핵심 설계 원칙 중 하나인 "Runnable composition" 덕분인데요, `RunnableLambda`가 함수의 반환값이 Runnable인지 감지하고, 맞다면 그 Runnable을 같은 입력으로 자동 실행합니다. 단순 값을 반환하면 그 값이 그대로 출력됩니다.

> 🔥 **실무 팁**: LLM 기반 의도 분류기를 사용할 때는 `temperature=0`으로 설정하세요. 분류는 창의성이 필요 없는 결정적(deterministic) 작업이므로, temperature를 높이면 같은 입력에 다른 분류가 나올 수 있습니다. 반면 분류 이후 응답을 생성하는 체인은 `temperature=0.7` 정도가 적당합니다. 파이프라인 내에서 **단계별로 다른 temperature**를 사용하는 것이 실무 핵심입니다.

> 🔥 **실무 팁**: 분류기의 출력이 예상 카테고리에 속하지 않는 경우를 반드시 처리하세요. 딕셔너리 매핑에서 `dict.get(key, default_chain)`을 사용하면 LLM이 예상 밖의 카테고리를 출력해도 기본 체인으로 안전하게 폴백됩니다. 프로덕션에서는 이 폴백 횟수를 모니터링하여 분류 프롬프트를 개선하는 피드백 루프를 만드는 것이 좋습니다.

## 핵심 정리

| 개념 | 설명 |
|------|------|
| `RunnableBranch` | `(조건, Runnable)` 튜플 리스트 + 기본 Runnable. 조건을 위에서부터 순서대로 평가하여 첫 번째 True의 Runnable 실행 |
| 커스텀 함수 라우팅 | `RunnableLambda`로 감싼 함수가 Runnable을 **반환**. 공식 권장 방식, 더 유연하고 성능 우수 |
| LLM 기반 동적 라우팅 | LLM 분류기 → 분류 결과 → 라우팅 함수 → 전문가 체인. 키워드 매칭의 한계를 극복 |
| 딕셔너리 매핑 패턴 | `{카테고리: 체인}` 딕셔너리 + `dict.get(key, default)`. 분기가 많을 때 가독성과 확장성 확보 |
| `RunnablePassthrough.assign` | 원본 입력을 유지하면서 분류 결과 등 새 필드를 추가. 라우팅 파이프라인의 핵심 접착제 |
