# 배포와 운영

> LangServe 애플리케이션을 Docker로 컨테이너화하고, 로드 밸런싱, 헬스 체크, 모니터링을 갖춘 프로덕션 환경을 구축합니다.

## 개요

이 섹션에서는 앞서 구축한 LangServe API를 실제 프로덕션 환경에 배포하고 안정적으로 운영하는 방법을 다룹니다. Docker 컨테이너화부터 Nginx 로드 밸런싱, 헬스 체크 엔드포인트, 그리고 LangSmith를 활용한 모니터링까지 — 개발 환경에서 프로덕션으로 가는 전체 여정을 실습합니다.

**선수 지식**: 이전 세션에서 배운 LangServe 기초(17.1), FastAPI 통합(17.2), 인증과 보안(17.3), RemoteRunnable 클라이언트(17.4)
**학습 목표**:
- Docker와 Docker Compose로 LangServe 애플리케이션을 컨테이너화할 수 있다
- 환경 변수와 시크릿을 안전하게 관리할 수 있다
- Nginx를 활용한 로드 밸런싱과 리버스 프록시를 구성할 수 있다
- 헬스 체크, 로깅, LangSmith 트레이싱으로 프로덕션 모니터링 체계를 갖출 수 있다

## 왜 알아야 할까?

로컬에서 `uvicorn main:app`으로 실행하는 것과 실제 사용자에게 서비스하는 것은 완전히 다른 세계입니다. "제 노트북에서는 잘 되는데요?"라는 말은 프로덕션에서 통하지 않죠. 서버가 갑자기 죽으면? 트래픽이 몰리면? API 키가 유출되면?

실제로 2023년 한 스타트업은 LLM 기반 서비스를 Docker 없이 EC2에 직접 배포했다가, 의존성 충돌로 3시간 동안 서비스가 중단된 적이 있습니다. 컨테이너화, 헬스 체크, 로드 밸런싱은 선택이 아닌 **필수**입니다. 이 세션을 마치면, 여러분의 LangServe API를 "실제로 돈을 벌 수 있는" 수준으로 배포할 수 있게 됩니다.

## 핵심 개념

### 개념 1: Docker 컨테이너화 — "이사 가도 똑같은 방"

> 💡 **비유**: Docker 컨테이너는 "완벽하게 포장된 이삿짐 박스"와 같습니다. 가구(코드), 가전(의존성), 배선(설정)까지 전부 들어 있어서, 어느 집(서버)에 가져가도 **박스를 열기만 하면** 똑같이 작동하죠. Dockerfile은 이 박스의 포장 설명서이고, Docker Compose는 여러 박스를 한꺼번에 배치하는 이삿짐센터 지시서입니다.

Docker는 애플리케이션과 그 실행 환경을 하나의 이미지로 패키징합니다. "제 컴퓨터에서는 되는데..."라는 문제를 근본적으로 해결해주죠.

LangServe 프로젝트의 Dockerfile은 다음과 같이 작성합니다:

```python
# Dockerfile
# ──────────────────────────────────────────────
# LangServe 프로덕션 Dockerfile (멀티스테이지 빌드)
# ──────────────────────────────────────────────

# 1단계: 빌더 — 의존성 설치
FROM python:3.11-slim AS builder

WORKDIR /app

# 시스템 의존성 설치 (빌드에만 필요)
RUN apt-get update && apt-get install -y --no-install-recommends \
    build-essential \
    && rm -rf /var/lib/apt/lists/*

# 의존성 파일 복사 후 설치 (캐시 최적화)
COPY requirements.txt .
RUN pip install --no-cache-dir --prefix=/install -r requirements.txt

# 2단계: 런타임 — 최소한의 실행 환경
FROM python:3.11-slim AS runtime

WORKDIR /app

# 빌더에서 설치된 패키지만 복사
COPY --from=builder /install /usr/local

# 애플리케이션 코드 복사
COPY ./app ./app

# 비루트 사용자로 실행 (보안 강화)
RUN useradd --create-home appuser
USER appuser

# 포트 노출
EXPOSE 8000

# 헬스 체크 설정 (30초마다 확인)
HEALTHCHECK --interval=30s --timeout=10s --start-period=5s --retries=3 \
    CMD python -c "import urllib.request; urllib.request.urlopen('http://localhost:8000/health')" || exit 1

# Gunicorn + Uvicorn 워커로 실행
CMD ["gunicorn", "app.server:app", \
     "-w", "4", \
     "-k", "uvicorn.workers.UvicornWorker", \
     "--bind", "0.0.0.0:8000", \
     "--timeout", "120", \
     "--graceful-timeout", "30"]
```

멀티스테이지 빌드를 사용하면 최종 이미지 크기가 크게 줄어듭니다. 빌드 도구(`build-essential` 등)는 런타임 이미지에 포함되지 않으니까요.

> ⚠️ **흔한 오해**: "Uvicorn만 쓰면 되지, Gunicorn은 왜 필요하죠?" — Uvicorn은 단일 프로세스 ASGI 서버입니다. CPU 코어가 4개인 서버에서 Uvicorn 하나만 돌리면 **코어 1개만** 사용하는 셈이에요. Gunicorn은 여러 Uvicorn 워커 프로세스를 관리하는 **프로세스 매니저** 역할을 합니다. 워커 수는 보통 `CPU 코어 수 × 2 + 1`로 설정하지만, LLM 호출처럼 I/O 바운드 작업이 많은 LangServe에서는 코어 수만큼이면 충분합니다.

### 개념 2: 환경 변수와 시크릿 관리 — "금고의 비밀번호는 금고 밖에"

> 💡 **비유**: API 키를 코드에 하드코딩하는 것은 집 열쇠를 현관문에 테이프로 붙여놓는 것과 같습니다. 환경 변수는 **주머니에 넣고 다니는 열쇠**이고, Docker Secrets는 **은행 금고**처럼 암호화된 보관소입니다.

프로덕션에서 환경 변수를 관리하는 핵심 원칙은 **12-Factor App**의 설정 원칙을 따르는 것입니다. 모든 설정은 코드 밖에서 환경 변수로 주입합니다.

```python
# app/config.py
# ──────────────────────────────────────────────
# 환경 변수 기반 설정 관리
# ──────────────────────────────────────────────
from pydantic_settings import BaseSettings
from functools import lru_cache


class Settings(BaseSettings):
    """프로덕션 설정 — 환경 변수에서 자동으로 읽어옵니다."""

    # OpenAI API 설정
    openai_api_key: str  # 필수값 — 없으면 시작 시 에러
    openai_model: str = "gpt-4o"
    openai_temperature: float = 0.7

    # 서버 설정
    app_env: str = "production"  # development | staging | production
    log_level: str = "INFO"
    workers: int = 4

    # LangSmith 트레이싱 (선택)
    langchain_tracing_v2: bool = False
    langchain_api_key: str = ""
    langchain_project: str = "langserve-prod"

    # 보안 설정
    allowed_origins: list[str] = ["https://myapp.com"]
    api_key_header: str = "X-API-Key"

    class Config:
        env_file = ".env"  # 로컬 개발 시 .env 파일 사용
        case_sensitive = False


@lru_cache  # 설정은 한 번만 로드 (싱글턴 패턴)
def get_settings() -> Settings:
    return Settings()
```

Docker Compose에서 환경 변수를 관리하는 방법:

```yaml
# docker-compose.yml
version: "3.9"

services:
  langserve-api:
    build: .
    ports:
      - "8000:8000"
    environment:
      - APP_ENV=production
      - LOG_LEVEL=INFO
      - OPENAI_MODEL=gpt-4o
    env_file:
      - .env.production  # 시크릿은 별도 파일로 분리
    secrets:
      - openai_api_key    # Docker Secrets 사용 (Swarm 모드)
    restart: unless-stopped

# Docker Secrets 정의 (Swarm 모드에서만 동작)
secrets:
  openai_api_key:
    file: ./secrets/openai_api_key.txt
```

> 🔥 **실무 팁**: `.env` 파일은 절대 Git에 커밋하지 마세요! `.gitignore`에 반드시 추가하세요. 대신 `.env.example` 파일에 필요한 환경 변수 목록(값 제외)을 기록해 두면 팀원들이 어떤 설정이 필요한지 파악할 수 있습니다.

### 개념 3: Nginx 로드 밸런싱 — "교통 경찰관"

> 💡 **비유**: Nginx 로드 밸런서는 교차로의 교통 경찰관과 같습니다. 차량(요청)이 몰려오면 한 도로(서버)에만 보내지 않고, 여러 도로로 분산시키죠. 도로가 공사 중(서버 다운)이면 자동으로 다른 도로로 우회시킵니다.

LLM API 호출은 응답 시간이 수 초에서 수십 초까지 걸릴 수 있습니다. 동시 요청이 많아지면 단일 서버로는 감당이 안 되죠. Nginx를 리버스 프록시이자 로드 밸런서로 사용하면, 여러 LangServe 인스턴스에 요청을 분산할 수 있습니다.

```nginx
# nginx/nginx.conf
# ──────────────────────────────────────────────
# Nginx 로드 밸런서 + 리버스 프록시 설정
# ──────────────────────────────────────────────

upstream langserve_backend {
    # 라운드 로빈 (기본) — 요청을 순서대로 분배
    server langserve-api-1:8000;
    server langserve-api-2:8000;
    server langserve-api-3:8000;

    # 헬스 체크 실패 시 자동 제외
    # max_fails=3: 3번 실패하면 제외
    # fail_timeout=30s: 30초 후 다시 시도
    server langserve-api-1:8000 max_fails=3 fail_timeout=30s;
    server langserve-api-2:8000 max_fails=3 fail_timeout=30s;
    server langserve-api-3:8000 max_fails=3 fail_timeout=30s;
}

server {
    listen 80;
    server_name api.myapp.com;

    # 요청 크기 제한 (큰 문서 RAG 용)
    client_max_body_size 10M;

    # 프록시 타임아웃 (LLM 응답 대기)
    proxy_read_timeout 120s;  # LLM 응답은 오래 걸릴 수 있음
    proxy_connect_timeout 10s;
    proxy_send_timeout 30s;

    # SSE 스트리밍 지원 (LangServe stream 엔드포인트)
    location / {
        proxy_pass http://langserve_backend;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        # SSE(Server-Sent Events) 스트리밍을 위한 설정
        proxy_buffering off;           # 버퍼링 비활성화
        proxy_cache off;               # 캐싱 비활성화
        proxy_set_header Connection ''; # keep-alive 연결
        proxy_http_version 1.1;        # HTTP/1.1 필수
    }

    # 헬스 체크 엔드포인트 — 로드 밸런서가 직접 확인
    location /health {
        proxy_pass http://langserve_backend/health;
        proxy_read_timeout 5s;
    }
}
```

> ⚠️ **흔한 오해**: LangServe의 `stream` 엔드포인트를 Nginx 뒤에 배치할 때, `proxy_buffering off`를 설정하지 않으면 **스트리밍이 작동하지 않습니다**. Nginx가 기본적으로 응답을 버퍼에 모았다가 한꺼번에 전달하기 때문이에요. SSE 스트리밍을 사용한다면 반드시 버퍼링을 꺼야 합니다.

### 개념 4: 헬스 체크와 모니터링 — "환자의 바이탈 사인"

> 💡 **비유**: 헬스 체크는 병원의 바이탈 사인 모니터링과 같습니다. 심박수(응답 시간), 혈압(메모리 사용량), 체온(에러율)을 실시간으로 감시하다가 이상이 생기면 즉시 알람이 울리죠. **Liveness 체크**는 "환자가 살아있는가?"이고, **Readiness 체크**는 "환자가 진료를 받을 수 있는 상태인가?"입니다.

프로덕션 LangServe 서버에는 두 가지 레벨의 헬스 체크가 필요합니다:

```python
# app/server.py
# ──────────────────────────────────────────────
# 프로덕션 LangServe 서버 (헬스 체크 + 모니터링)
# ──────────────────────────────────────────────
import os
import time
import logging
from contextlib import asynccontextmanager
from datetime import datetime

from fastapi import FastAPI
from fastapi.middleware.cors import CORSMiddleware
from langchain_openai import ChatOpenAI
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser
from langserve import add_routes

from app.config import get_settings

# 구조화된 로깅 설정
logging.basicConfig(
    level=logging.INFO,
    format="%(asctime)s [%(levelname)s] %(name)s: %(message)s",
)
logger = logging.getLogger("langserve")

# 서버 시작 시각 기록 (업타임 계산용)
START_TIME = time.time()

# 요청 카운터 (간단한 메트릭)
request_metrics = {
    "total_requests": 0,
    "successful_requests": 0,
    "failed_requests": 0,
}


@asynccontextmanager
async def lifespan(app: FastAPI):
    """서버 시작/종료 시 실행되는 라이프사이클 관리자."""
    settings = get_settings()
    logger.info(f"서버 시작 — 환경: {settings.app_env}")
    logger.info(f"모델: {settings.openai_model}, 트레이싱: {settings.langchain_tracing_v2}")
    yield
    logger.info("서버 종료 — 리소스 정리 완료")


app = FastAPI(
    title="LangServe Production API",
    version="1.0.0",
    lifespan=lifespan,
)

settings = get_settings()

# CORS 설정
app.add_middleware(
    CORSMiddleware,
    allow_origins=settings.allowed_origins,
    allow_methods=["GET", "POST"],
    allow_headers=["*"],
)

# ── 헬스 체크 엔드포인트 ──────────────────────

@app.get("/health")
async def health_check():
    """Liveness 체크 — 서버 프로세스가 살아있는지 확인."""
    return {
        "status": "healthy",
        "timestamp": datetime.utcnow().isoformat(),
        "uptime_seconds": round(time.time() - START_TIME, 2),
        "instance_id": os.getenv("INSTANCE_ID", "unknown"),
    }


@app.get("/health/ready")
async def readiness_check():
    """Readiness 체크 — 서비스가 요청을 처리할 준비가 되었는지 확인."""
    checks = {}

    # 1. LLM 연결 확인
    try:
        llm = ChatOpenAI(
            model=settings.openai_model,
            api_key=settings.openai_api_key,
            max_tokens=5,
        )
        await llm.ainvoke("ping")  # 최소한의 요청으로 연결 테스트
        checks["llm"] = "ok"
    except Exception as e:
        checks["llm"] = f"error: {str(e)}"

    # 2. 전체 상태 판단
    all_ok = all(v == "ok" for v in checks.values())
    status_code = 200 if all_ok else 503

    from fastapi.responses import JSONResponse
    return JSONResponse(
        status_code=status_code,
        content={
            "status": "ready" if all_ok else "not_ready",
            "checks": checks,
            "metrics": request_metrics,
        },
    )


# ── LangSmith 트레이싱 설정 ────────────────────
# 환경 변수로 자동 활성화됨:
# LANGCHAIN_TRACING_V2=true
# LANGCHAIN_API_KEY=ls__...
# LANGCHAIN_PROJECT=langserve-prod

# ── 체인 정의 및 라우트 등록 ────────────────────
llm = ChatOpenAI(
    model=settings.openai_model,
    temperature=settings.openai_temperature,
    api_key=settings.openai_api_key,
)

prompt = ChatPromptTemplate.from_messages([
    ("system", "당신은 유능한 AI 어시스턴트입니다."),
    ("human", "{input}"),
])

chain = prompt | llm | StrOutputParser()

# LangServe 라우트 등록 (17.1에서 배운 패턴)
add_routes(app, chain, path="/chat")

logger.info("LangServe 라우트 등록 완료: /chat")
```

### 개념 5: 구조화된 로깅과 LangSmith 트레이싱

프로덕션에서는 `print()`가 아닌 **구조화된 로깅**을 사용해야 합니다. 그리고 LLM 호출 전체를 추적하려면 LangSmith 트레이싱을 활성화합니다.

```python
# app/middleware.py
# ──────────────────────────────────────────────
# 요청 로깅 미들웨어
# ──────────────────────────────────────────────
import time
import logging
import uuid

from starlette.middleware.base import BaseHTTPMiddleware
from starlette.requests import Request

logger = logging.getLogger("langserve.access")


class RequestLoggingMiddleware(BaseHTTPMiddleware):
    """모든 요청의 메서드, 경로, 응답 시간, 상태 코드를 기록합니다."""

    async def dispatch(self, request: Request, call_next):
        # 고유 요청 ID 생성 (추적용)
        request_id = str(uuid.uuid4())[:8]
        start_time = time.time()

        # 요청 시작 로그
        logger.info(
            f"[{request_id}] → {request.method} {request.url.path}"
        )

        try:
            response = await call_next(request)
            duration = round(time.time() - start_time, 3)

            # 응답 완료 로그
            logger.info(
                f"[{request_id}] ← {response.status_code} "
                f"({duration}s)"
            )

            # 응답 헤더에 요청 ID 추가 (디버깅용)
            response.headers["X-Request-ID"] = request_id
            return response

        except Exception as e:
            duration = round(time.time() - start_time, 3)
            logger.error(
                f"[{request_id}] ✗ ERROR ({duration}s): {str(e)}"
            )
            raise
```

LangSmith 트레이싱은 환경 변수만 설정하면 자동으로 활성화됩니다:

```bash
# .env.production — LangSmith 트레이싱 설정
LANGCHAIN_TRACING_V2=true
LANGCHAIN_API_KEY=ls__your_langsmith_api_key
LANGCHAIN_PROJECT=langserve-prod
LANGCHAIN_ENDPOINT=https://api.smith.langchain.com
```

> 💡 **알고 계셨나요?**: LangSmith에서는 트레이싱 레벨을 환경별로 다르게 설정하는 것이 권장됩니다. 개발 환경에서는 모든 요청을 전수 트레이싱(full tracing)하고, 프로덕션에서는 샘플링 트레이싱(예: 10%만 기록)을 적용하면 성능 오버헤드를 줄이면서 충분한 모니터링이 가능합니다. LangSmith 대시보드에서는 P50/P99 레이턴시, 토큰 사용량, 에러율, 비용 등을 실시간으로 추적할 수 있습니다.

```python
# app/server.py
# ──────────────────────────────────────────────
# 프로덕션 LangServe 서버 — 전체 통합 버전
# ──────────────────────────────────────────────
import os
import time
import logging
from contextlib import asynccontextmanager
from datetime import datetime

from fastapi import FastAPI
from fastapi.middleware.cors import CORSMiddleware
from fastapi.responses import JSONResponse
from langchain_openai import ChatOpenAI
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser
from langserve import add_routes

from app.config import get_settings
from app.middleware import RequestLoggingMiddleware

logging.basicConfig(
    level=logging.INFO,
    format="%(asctime)s [%(levelname)s] %(name)s: %(message)s",
)
logger = logging.getLogger("langserve")

START_TIME = time.time()


@asynccontextmanager
async def lifespan(app: FastAPI):
    settings = get_settings()
    logger.info(f"서버 시작 — 환경: {settings.app_env}, 모델: {settings.openai_model}")
    yield
    logger.info("서버 종료")


app = FastAPI(
    title="LangServe Production API",
    version="1.0.0",
    lifespan=lifespan,
)

settings = get_settings()

# 미들웨어 등록
app.add_middleware(RequestLoggingMiddleware)  # 요청 로깅
app.add_middleware(
    CORSMiddleware,
    allow_origins=settings.allowed_origins,
    allow_methods=["GET", "POST"],
    allow_headers=["*"],
)


# ── 헬스 체크 ──────────────────────────────────

@app.get("/health")
async def liveness():
    """Liveness — 프로세스가 살아있는가?"""
    return {
        "status": "healthy",
        "uptime": round(time.time() - START_TIME, 2),
        "instance": os.getenv("INSTANCE_ID", "local"),
    }


@app.get("/health/ready")
async def readiness():
    """Readiness — 요청을 처리할 준비가 되었는가?"""
    errors = {}
    try:
        # LLM 연결 확인 (최소 비용 호출)
        test_llm = ChatOpenAI(
            model=settings.openai_model,
            api_key=settings.openai_api_key,
            max_tokens=5,
        )
        await test_llm.ainvoke("test")
    except Exception as e:
        errors["llm"] = str(e)

    if errors:
        return JSONResponse(status_code=503, content={
            "status": "not_ready", "errors": errors,
        })
    return {"status": "ready"}


# ── 체인 등록 ──────────────────────────────────

llm = ChatOpenAI(
    model=settings.openai_model,
    temperature=settings.openai_temperature,
    api_key=settings.openai_api_key,
)

# 범용 채팅 체인
chat_chain = (
    ChatPromptTemplate.from_messages([
        ("system", "당신은 유능한 AI 어시스턴트입니다. 한국어로 답변하세요."),
        ("human", "{input}"),
    ])
    | llm
    | StrOutputParser()
)

# 요약 체인
summarize_chain = (
    ChatPromptTemplate.from_messages([
        ("system", "주어진 텍스트를 3문장 이내로 요약하세요."),
        ("human", "{text}"),
    ])
    | llm
    | StrOutputParser()
)

# LangServe 라우트 등록
add_routes(app, chat_chain, path="/chat")
add_routes(app, summarize_chain, path="/summarize")

logger.info("라우트 등록 완료: /chat, /summarize")
```

## 더 깊이 알아보기

### Docker의 탄생 — "잘 안 돼요"에서 "어디서든 돼요"로

2013년, dotCloud이라는 PaaS 스타트업의 창업자 Solomon Hykes는 PyCon 라이트닝 토크에서 5분짜리 데모를 했습니다. "The future of Linux containers"라는 제목의 이 발표에서 그는 나중에 Docker가 될 기술을 처음 세상에 공개했죠. 당시 Linux 컨테이너(LXC) 기술은 이미 존재했지만, 사용법이 너무 복잡했습니다. Docker는 이를 **개발자 친화적인 인터페이스**로 감싸면서 혁명을 일으켰습니다. dotCloud은 결국 회사 이름까지 Docker Inc.로 바꾸게 되었고, "It works on my machine"이라는 개발자의 영원한 변명을 역사의 뒤안길로 보내버렸습니다.

### LangServe의 현재와 미래

LangChain 팀은 2024년 하반기부터 새로운 프로젝트에는 **LangGraph Platform**(현재 LangSmith Deployment로 이름 변경)을 권장하고 있습니다. LangServe는 커뮤니티 버그 수정은 받지만 새로운 기능 추가는 중단된 상태입니다. 하지만 LangServe에서 배운 FastAPI 통합, Docker 배포, 인증, 모니터링 패턴은 LangGraph Platform에서도 그대로 활용됩니다. 특히 이번 세션에서 다룬 **Docker 컨테이너화**, **헬스 체크**, **로드 밸런싱** 패턴은 어떤 프레임워크를 쓰든 변하지 않는 프로덕션 인프라의 핵심이죠.

## 흔한 오해와 팁

> ⚠️ **흔한 오해**: "Docker만 쓰면 자동으로 확장된다" — Docker는 애플리케이션을 **패키징**하는 도구이지, 자동 확장(Auto Scaling)을 제공하지 않습니다. 자동 확장이 필요하면 Kubernetes(K8s)나 AWS ECS 같은 오케스트레이션 플랫폼을 함께 사용해야 합니다. Docker Compose의 `deploy.replicas`는 단일 호스트 내에서만 동작하는 한계가 있어요.

> 💡 **알고 계셨나요?**: Nginx의 `upstream` 블록에서 `least_conn` 디렉티브를 사용하면 라운드 로빈 대신 **현재 연결이 가장 적은 서버**로 요청을 보냅니다. LLM 호출처럼 요청 처리 시간이 불균일한 서비스에서는 라운드 로빈보다 `least_conn`이 더 효과적입니다. 한 요청이 30초 걸리는 동안 다른 서버에 새 요청을 보낼 수 있으니까요.

> 🔥 **실무 팁**: 프로덕션 배포 전 반드시 **배포 체크리스트**를 확인하세요: (1) `.env` 파일이 `.gitignore`에 포함? (2) Docker 이미지에 비루트 사용자 설정? (3) 헬스 체크 엔드포인트 동작 확인? (4) Gunicorn 워커 수가 CPU에 맞게 설정? (5) CORS가 프로덕션 도메인만 허용? (6) API 인증이 적용? (7) 로깅과 모니터링이 활성화? — 이 중 하나라도 빠지면 프로덕션 장애의 원인이 될 수 있습니다.

> 🔥 **실무 팁**: Docker 이미지 빌드 시 `.dockerignore` 파일을 반드시 작성하세요. `.env`, `__pycache__`, `.git`, `venv` 등을 제외하면 빌드 속도가 빨라지고, **시크릿이 이미지에 포함되는 사고**를 방지할 수 있습니다.

## 핵심 정리

| 개념 | 설명 |
|------|------|
| Dockerfile 멀티스테이지 빌드 | 빌드 단계와 런타임 단계를 분리하여 이미지 크기를 최소화 |
| Gunicorn + UvicornWorker | Gunicorn이 프로세스 매니저, Uvicorn이 ASGI 워커로 동작하여 멀티코어 활용 |
| 환경 변수 관리 | Pydantic Settings로 타입 안전한 설정, `.env` 파일은 Git에서 제외 |
| Docker Compose | 여러 서비스(API, Nginx)를 선언적으로 정의하고 한 번에 배포 |
| Nginx 로드 밸런싱 | 리버스 프록시로 여러 API 인스턴스에 트래픽 분산, SSE 스트리밍 지원 |
| Liveness 체크 | 서버 프로세스가 살아있는지 확인하는 경량 엔드포인트 (`/health`) |
| Readiness 체크 | LLM 연결 등 의존성이 준비되었는지 확인하는 엔드포인트 (`/health/ready`) |
| LangSmith 트레이싱 | 환경 변수 설정으로 LLM 호출의 레이턴시, 토큰, 비용을 자동 추적 |
| 구조화된 로깅 | 요청 ID, 응답 시간, 상태 코드를 포함한 프로덕션급 로그 |
