# 도구 정의와 바인딩

> LLM에게 "손과 발"을 달아주는 첫 번째 단계 — 커스텀 도구를 정의하고 모델에 연결하는 방법을 배웁니다.

## 개요

이 섹션에서는 LangChain에서 도구(Tool)를 정의하는 세 가지 방법과, 정의한 도구를 LLM에 연결(`bind_tools`)하는 과정을 다룹니다. 도구는 LLM이 외부 세계와 상호작용할 수 있게 해주는 핵심 메커니즘이며, 에이전트(Agent) 구축의 기초가 됩니다.

**선수 지식**: [Ch5: LCEL 마스터]에서 배운 파이프 연산자(`|`)와 Runnable 인터페이스, [Ch4: 출력 파서와 구조화된 출력]에서 다룬 Pydantic 모델 정의
**학습 목표**:
- `@tool` 데코레이터로 간단한 도구를 정의할 수 있다
- `StructuredTool.from_function()`으로 세밀하게 도구를 구성할 수 있다
- Pydantic `args_schema`로 도구 입력 스키마를 명시적으로 정의할 수 있다
- `bind_tools()`로 도구를 LLM에 연결하고, `AIMessage.tool_calls`를 해석할 수 있다

## 왜 알아야 할까?

LLM은 본질적으로 "텍스트를 읽고 텍스트를 생성하는" 존재입니다. 아무리 똑똑해도 현재 날씨를 확인하거나, 데이터베이스를 조회하거나, 이메일을 보내는 건 불가능하죠. 하지만 **도구(Tool)**를 연결하면 이야기가 달라집니다. LLM이 "이 도구를 이런 인자로 호출해주세요"라고 요청하고, 그 결과를 받아서 응답을 완성할 수 있게 되거든요.

실무에서 도구는 곳곳에 쓰입니다:
- **검색 엔진 연동**: 최신 정보가 필요할 때 웹 검색 도구 호출
- **데이터베이스 조회**: 고객 정보, 주문 내역 등을 실시간으로 가져오기
- **외부 API 호출**: 날씨, 환율, 주가 등 실시간 데이터 활용
- **파일 시스템 조작**: 문서 읽기, 리포트 생성 등

이번 섹션에서 도구 정의와 바인딩을 제대로 익혀두면, 이어지는 에이전트(Agent) 챕터에서 LLM이 스스로 판단하여 도구를 선택하고 실행하는 자율적 시스템을 구축할 수 있습니다.

## 핵심 개념

### 개념 1: 도구(Tool)란 무엇인가?

> 💡 **비유**: 도구는 LLM에게 달아주는 **스마트폰 앱**과 같습니다. 스마트폰(LLM) 자체는 범용적이지만, 날씨 앱을 설치하면 날씨를 알 수 있고, 계산기 앱을 설치하면 복잡한 계산을 할 수 있죠. 마찬가지로 LLM에 도구를 "설치"하면, 모델이 그 도구의 기능을 활용할 수 있게 됩니다.

LangChain에서 도구(Tool)는 세 가지 핵심 요소로 구성됩니다:

| 요소 | 설명 | 역할 |
|------|------|------|
| `name` | 도구의 이름 | LLM이 어떤 도구를 호출할지 식별 |
| `description` | 도구의 설명 | LLM이 **언제** 이 도구를 사용할지 판단하는 근거 |
| `args_schema` | 입력 파라미터 스키마 | LLM이 **어떤 인자**를 전달해야 하는지 파악 |

특히 `description`이 매우 중요합니다. LLM은 이 설명을 읽고 "지금 상황에서 이 도구를 쓸지 말지"를 결정하거든요. 설명이 모호하면 LLM이 적절한 타이밍에 도구를 호출하지 못합니다.

### 개념 2: @tool 데코레이터 — 가장 간단한 방법

> 💡 **비유**: `@tool` 데코레이터는 일반 함수에 **이름표를 붙이는 것**과 같습니다. 회사에서 아무리 유능한 직원이라도 이름표와 직무 설명이 없으면 누가 무슨 일을 하는 사람인지 알 수 없죠. `@tool`을 붙이면 함수에 이름, 설명, 입력 스키마가 자동으로 부여됩니다.

`@tool` 데코레이터는 일반 Python 함수를 LangChain 도구로 변환하는 가장 간단한 방법입니다. 함수의 이름이 도구 이름이 되고, 독스트링(docstring)이 도구 설명이 되며, 타입 힌트가 입력 스키마로 자동 변환됩니다.

```python
from langchain_core.tools import tool

@tool
def get_word_count(text: str) -> int:
    """주어진 텍스트의 단어 수를 반환합니다.

    Args:
        text: 단어 수를 셀 텍스트
    """
    return len(text.split())

# 도구의 속성 확인
print(f"이름: {get_word_count.name}")           # 이름: get_word_count
print(f"설명: {get_word_count.description}")     # 설명: 주어진 텍스트의 단어 수를 반환합니다. ...
print(f"스키마: {get_word_count.args_schema.model_json_schema()}")
# 스키마: {'properties': {'text': {'title': 'Text', 'type': 'string'}}, 'required': ['text'], ...}
```

**이름을 직접 지정**하고 싶다면 데코레이터에 인자를 전달할 수 있습니다:

```python
@tool("word_counter", return_direct=True)
def get_word_count(text: str) -> int:
    """주어진 텍스트의 단어 수를 반환합니다."""
    return len(text.split())

print(f"이름: {get_word_count.name}")  # 이름: word_counter
```

> ⚠️ **흔한 오해**: "독스트링은 개발자를 위한 주석이니까 대충 써도 된다"고 생각하기 쉬운데요, 도구에서는 **독스트링이 곧 LLM에게 전달되는 프롬프트**입니다. 독스트링을 명확하고 구체적으로 작성하지 않으면 LLM이 도구를 잘못된 상황에서 호출하거나 아예 사용하지 않을 수 있습니다.

### 개념 3: args_schema로 입력 스키마 명시하기

`@tool` 데코레이터는 타입 힌트에서 스키마를 자동 추론하지만, 더 정교한 제어가 필요할 때는 Pydantic 모델로 `args_schema`를 직접 정의합니다. 이렇게 하면 각 필드에 대한 상세 설명, 기본값, 유효성 검증 규칙 등을 LLM에게 전달할 수 있습니다.

```python
from pydantic import BaseModel, Field
from langchain_core.tools import tool

# Pydantic 모델로 입력 스키마 정의
class WeatherInput(BaseModel):
    """날씨 조회를 위한 입력 스키마"""
    city: str = Field(description="날씨를 조회할 도시 이름 (예: '서울', 'Tokyo')")
    unit: str = Field(
        default="celsius",
        description="온도 단위. 'celsius' 또는 'fahrenheit'"
    )

@tool(args_schema=WeatherInput)
def get_weather(city: str, unit: str = "celsius") -> str:
    """지정한 도시의 현재 날씨를 조회합니다.

    실시간 기상 데이터를 반환하며, 도시 이름은 한국어 또는 영어로 입력할 수 있습니다.
    """
    # 실제로는 외부 API를 호출하지만, 예제에서는 더미 데이터 반환
    weather_data = {
        "서울": {"temp": 22, "condition": "맑음"},
        "부산": {"temp": 25, "condition": "흐림"},
        "Tokyo": {"temp": 28, "condition": "비"},
    }
    data = weather_data.get(city, {"temp": 20, "condition": "정보 없음"})
    temp = data["temp"] if unit == "celsius" else data["temp"] * 9/5 + 32
    unit_symbol = "°C" if unit == "celsius" else "°F"
    return f"{city}: {temp}{unit_symbol}, {data['condition']}"

# 스키마 확인 — Field description이 포함됨
import json
print(json.dumps(get_weather.args_schema.model_json_schema(), indent=2, ensure_ascii=False))
```

출력 결과를 보면 `description` 필드가 스키마에 포함되어 있어서, LLM이 각 파라미터의 용도를 정확히 파악할 수 있습니다.

### 개념 4: StructuredTool — 프로그래밍 방식의 도구 정의

이미 존재하는 함수를 수정하지 않고 도구로 감싸고 싶거나, 동적으로 도구를 생성해야 할 때는 `StructuredTool.from_function()`을 사용합니다.

```python
from langchain_core.tools import StructuredTool
from pydantic import BaseModel, Field

def calculate_bmi(height_cm: float, weight_kg: float) -> str:
    """BMI를 계산합니다."""
    height_m = height_cm / 100
    bmi = weight_kg / (height_m ** 2)
    if bmi < 18.5:
        category = "저체중"
    elif bmi < 25:
        category = "정상"
    elif bmi < 30:
        category = "과체중"
    else:
        category = "비만"
    return f"BMI: {bmi:.1f} ({category})"

# 입력 스키마 정의
class BMIInput(BaseModel):
    height_cm: float = Field(description="키 (cm 단위, 예: 175.5)")
    weight_kg: float = Field(description="몸무게 (kg 단위, 예: 70.0)")

# StructuredTool로 도구 생성
bmi_tool = StructuredTool.from_function(
    func=calculate_bmi,
    name="bmi_calculator",
    description="키(cm)와 몸무게(kg)를 입력받아 BMI 지수와 비만도를 계산합니다",
    args_schema=BMIInput,
    return_direct=False,  # True면 에이전트가 이 결과를 바로 사용자에게 반환
)

# 도구 직접 실행
print(bmi_tool.invoke({"height_cm": 175, "weight_kg": 70}))
# 출력: BMI: 22.9 (정상)
```

`@tool` 데코레이터와 `StructuredTool.from_function()`의 차이를 정리하면:

| 비교 항목 | `@tool` 데코레이터 | `StructuredTool.from_function()` |
|-----------|-------------------|----------------------------------|
| 사용 시점 | 새 함수를 만들 때 | 기존 함수를 감쌀 때 |
| 이름/설명 | 함수명/독스트링에서 자동 추출 | 명시적으로 지정 |
| 유연성 | 간편하지만 제한적 | 모든 속성을 세밀하게 제어 |
| 동적 생성 | 어려움 | 런타임에 동적 생성 가능 |

### 개념 5: bind_tools() — 도구를 모델에 연결하기

> 💡 **비유**: 도구를 정의하는 것은 앱을 개발한 것이고, `bind_tools()`는 그 앱을 스마트폰에 **설치**하는 것입니다. 설치하지 않으면 아무리 좋은 앱도 사용할 수 없죠.

`bind_tools()`는 정의한 도구들을 LLM에 알려주는 메서드입니다. 이 메서드를 호출하면, LLM이 응답을 생성할 때 필요하다고 판단되면 도구 호출을 요청할 수 있게 됩니다.

```python
from langchain_openai import ChatOpenAI
from langchain_core.tools import tool

@tool
def add(a: int, b: int) -> int:
    """두 정수를 더합니다."""
    return a + b

@tool
def multiply(a: int, b: int) -> int:
    """두 정수를 곱합니다."""
    return a * b

# 모델 생성 및 도구 바인딩
llm = ChatOpenAI(model="gpt-4o", temperature=0)
llm_with_tools = llm.bind_tools([add, multiply])

# 도구가 필요한 질문
response = llm_with_tools.invoke("3과 7을 더한 다음, 그 결과에 5를 곱해주세요")

# AIMessage의 tool_calls 확인
print(response.tool_calls)
# [{'name': 'add', 'args': {'a': 3, 'b': 7}, 'id': 'call_abc123', 'type': 'tool_call'}]
```

여기서 핵심은 `bind_tools()` 이후 모델을 호출하면, 응답이 일반 텍스트가 아니라 **도구 호출 요청(`tool_calls`)**이 될 수 있다는 점입니다. `AIMessage.tool_calls`는 다음과 같은 구조를 가집니다:

```python
# tool_calls의 구조
{
    "name": "add",          # 호출할 도구 이름
    "args": {"a": 3, "b": 7},  # 전달할 인자
    "id": "call_abc123",    # 호출 고유 ID (결과 매칭에 사용)
    "type": "tool_call"     # 타입 식별자
}
```

> 🔥 **실무 팁**: `bind_tools()`는 새로운 모델 인스턴스를 반환합니다. 원본 `llm`은 변경되지 않으므로, 도구가 필요한 호출과 일반 호출을 구분하여 사용할 수 있습니다. 이는 LCEL 체인에서 상황에 따라 도구를 동적으로 바인딩하는 패턴에 활용됩니다.

### 개념 6: 도구 호출 결과 처리하기

모델이 도구 호출을 요청하면, 우리가 직접 도구를 실행하고 그 결과를 다시 모델에게 전달해야 합니다. 이때 `ToolMessage`를 사용합니다.

```python
from langchain_core.messages import HumanMessage, ToolMessage

@tool
def search_product(query: str) -> str:
    """상품 데이터베이스에서 상품을 검색합니다."""
    # 더미 데이터
    products = {
        "노트북": "MacBook Pro 16인치 - 3,499,000원",
        "키보드": "HHKB Professional - 330,000원",
    }
    for key, value in products.items():
        if key in query:
            return value
    return "검색 결과가 없습니다."

llm = ChatOpenAI(model="gpt-4o", temperature=0)
llm_with_tools = llm.bind_tools([search_product])

# 1단계: 사용자 메시지로 모델 호출
messages = [HumanMessage(content="노트북 가격 알려줘")]
ai_msg = llm_with_tools.invoke(messages)
messages.append(ai_msg)

# 2단계: tool_calls가 있으면 도구 실행
if ai_msg.tool_calls:
    for tool_call in ai_msg.tool_calls:
        # 도구 이름으로 실제 도구 함수 매핑
        tool_map = {"search_product": search_product}
        selected_tool = tool_map[tool_call["name"]]

        # 도구 실행
        result = selected_tool.invoke(tool_call["args"])

        # ToolMessage로 결과를 대화에 추가
        messages.append(
            ToolMessage(content=str(result), tool_call_id=tool_call["id"])
        )

# 3단계: 도구 결과를 포함한 메시지로 다시 모델 호출
final_response = llm_with_tools.invoke(messages)
print(final_response.content)
# 출력: MacBook Pro 16인치의 가격은 3,499,000원입니다.
```

이 3단계 흐름을 정리하면:

```
사용자 질문 → LLM (도구 호출 요청) → 도구 실행 → LLM (최종 답변)
```

이 패턴을 잘 기억해두세요. 이후 [Ch12: 에이전트 기초]에서 이 과정을 자동화하는 방법을 배우게 됩니다.

## 실습: 직접 해보기

다양한 유형의 도구를 정의하고, 모델에 바인딩한 뒤, 도구 호출 결과까지 처리하는 완전한 예제를 만들어보겠습니다.

```python
"""
실습: 다목적 도우미 봇 — 도구 정의와 바인딩 완전 예제
"""
import json
from datetime import datetime
from pydantic import BaseModel, Field
from langchain_core.tools import tool, StructuredTool
from langchain_core.messages import HumanMessage, ToolMessage
from langchain_openai import ChatOpenAI

# ─────────────────────────────────────────────
# 1. @tool 데코레이터로 간단한 도구 정의
# ─────────────────────────────────────────────
@tool
def get_current_time() -> str:
    """현재 날짜와 시간을 반환합니다. 사용자가 지금 몇 시인지 물어볼 때 사용합니다."""
    now = datetime.now()
    return now.strftime("%Y년 %m월 %d일 %H시 %M분")

# ─────────────────────────────────────────────
# 2. args_schema로 정교한 입력 스키마 정의
# ─────────────────────────────────────────────
class CurrencyConvertInput(BaseModel):
    """환율 변환 입력 스키마"""
    amount: float = Field(description="변환할 금액")
    from_currency: str = Field(description="원래 통화 코드 (예: 'USD', 'KRW', 'JPY')")
    to_currency: str = Field(description="변환할 통화 코드 (예: 'USD', 'KRW', 'JPY')")

@tool(args_schema=CurrencyConvertInput)
def convert_currency(amount: float, from_currency: str, to_currency: str) -> str:
    """통화를 변환합니다. 주요 통화 간의 환율 변환을 지원합니다."""
    # 더미 환율 (실제로는 외부 API 사용)
    rates = {
        ("USD", "KRW"): 1350.0,
        ("KRW", "USD"): 1 / 1350.0,
        ("USD", "JPY"): 155.0,
        ("JPY", "USD"): 1 / 155.0,
        ("KRW", "JPY"): 155.0 / 1350.0,
        ("JPY", "KRW"): 1350.0 / 155.0,
    }
    key = (from_currency.upper(), to_currency.upper())
    if key[0] == key[1]:
        return f"{amount:.2f} {from_currency}"
    rate = rates.get(key)
    if rate is None:
        return f"지원하지 않는 통화 쌍입니다: {from_currency} → {to_currency}"
    converted = amount * rate
    return f"{amount:,.2f} {from_currency} = {converted:,.2f} {to_currency} (환율: {rate:.4f})"

# ─────────────────────────────────────────────
# 3. StructuredTool로 기존 함수 감싸기
# ─────────────────────────────────────────────
def _calculate_tip(bill_amount: float, tip_percent: float, num_people: int) -> str:
    """팁 계산 로직 (기존 함수)"""
    tip = bill_amount * (tip_percent / 100)
    total = bill_amount + tip
    per_person = total / num_people
    return (
        f"음식값: {bill_amount:,.0f}원\n"
        f"팁 ({tip_percent}%): {tip:,.0f}원\n"
        f"총액: {total:,.0f}원\n"
        f"1인당: {per_person:,.0f}원 ({num_people}명)"
    )

class TipInput(BaseModel):
    bill_amount: float = Field(description="음식 총 금액 (원)")
    tip_percent: float = Field(default=15.0, description="팁 비율 (%)")
    num_people: int = Field(default=1, description="인원 수")

tip_calculator = StructuredTool.from_function(
    func=_calculate_tip,
    name="tip_calculator",
    description="음식값, 팁 비율, 인원 수를 받아 1인당 내야 할 금액을 계산합니다",
    args_schema=TipInput,
)

# ─────────────────────────────────────────────
# 4. 모델에 도구 바인딩 및 호출 처리
# ─────────────────────────────────────────────
# 모든 도구를 리스트로 모으기
tools = [get_current_time, convert_currency, tip_calculator]

# 도구 이름 → 도구 객체 매핑 딕셔너리
tool_map = {t.name: t for t in tools}

# 모델에 도구 바인딩
llm = ChatOpenAI(model="gpt-4o", temperature=0)
llm_with_tools = llm.bind_tools(tools)

def ask(question: str) -> str:
    """사용자 질문을 처리하고 도구 호출까지 완료하는 함수"""
    messages = [HumanMessage(content=question)]

    # 1차 호출: 모델이 도구 호출 여부를 판단
    ai_msg = llm_with_tools.invoke(messages)
    messages.append(ai_msg)

    # 도구 호출이 없으면 바로 응답 반환
    if not ai_msg.tool_calls:
        return ai_msg.content

    # 도구 호출 실행
    for tc in ai_msg.tool_calls:
        print(f"  🔧 도구 호출: {tc['name']}({tc['args']})")
        result = tool_map[tc["name"]].invoke(tc["args"])
        print(f"  📋 결과: {result}")
        messages.append(ToolMessage(content=str(result), tool_call_id=tc["id"]))

    # 2차 호출: 도구 결과를 바탕으로 최종 답변 생성
    final = llm_with_tools.invoke(messages)
    return final.content

# ─────────────────────────────────────────────
# 5. 테스트
# ─────────────────────────────────────────────
print("=" * 50)
print("질문 1: 지금 몇 시야?")
print(ask("지금 몇 시야?"))

print("\n" + "=" * 50)
print("질문 2: 100달러를 원화로 바꾸면?")
print(ask("100달러를 한국 원화로 바꾸면 얼마야?"))

print("\n" + "=" * 50)
print("질문 3: 더치페이 계산")
print(ask("5만원짜리 식사에 팁 10%를 주고, 3명이 나누면 1인당 얼마야?"))
```

실행하면 각 질문에 대해 모델이 적절한 도구를 선택하고, 호출 결과를 기반으로 자연스러운 한국어 답변을 생성하는 것을 확인할 수 있습니다.

## 더 깊이 알아보기

### 도구 호출의 역사 — Function Calling에서 Tool Calling으로

도구 호출(Tool Calling)의 역사는 2023년 6월, **OpenAI가 "Function Calling"**이라는 기능을 처음 발표한 것에서 시작됩니다. 이전까지 LLM에게 외부 함수를 실행하게 하려면, 프롬프트에 "다음 JSON 형식으로 함수 호출을 출력하세요"라고 지시하고 출력을 파싱하는 불안정한 방법을 써야 했거든요.

OpenAI의 Function Calling은 모델이 **구조화된 JSON**으로 함수 호출을 출력하도록 학습시킨 혁신적인 접근이었습니다. 이후 2023년 11월, OpenAI는 이를 **"Tool Calling"으로 리브랜딩**하면서 한 번에 여러 함수를 호출하는 "parallel tool calling"을 지원하기 시작했습니다.

문제는 이후 벌어진 **파편화**였습니다. Google(Gemini), Anthropic(Claude), Mistral 등 주요 모델 제공자들이 각자의 도구 호출 인터페이스를 출시했는데, 모두 형식이 달랐습니다. OpenAI는 `function_call` 필드를, Anthropic은 `tool_use` 블록을, Gemini는 또 다른 구조를 사용했죠.

이 파편화를 해결하기 위해 LangChain이 나섰습니다. `bind_tools()`와 `AIMessage.tool_calls`라는 **통합 인터페이스**를 만들어, 어떤 모델을 사용하든 동일한 코드로 도구를 연결하고 결과를 처리할 수 있게 한 것입니다. 덕분에 개발자들은 모델 교체 시 도구 관련 코드를 수정할 필요가 없어졌습니다.

> 💡 **알고 계셨나요?**: OpenAI가 "Function Calling"에서 "Tool Calling"으로 이름을 바꾼 이유가 있습니다. 프로그래밍에서 "function"은 특정 의미를 가지지만, LLM이 호출하는 대상은 함수뿐 아니라 API, 데이터베이스 쿼리, 웹 검색 등 다양한 "도구"이기 때문입니다. 더 포괄적인 개념을 반영하기 위해 "Tool"로 바뀐 것이죠.

## 흔한 오해와 팁

> ⚠️ **흔한 오해**: "`bind_tools()`를 호출하면 모델이 자동으로 도구를 실행한다"고 생각하기 쉽습니다. 실제로는 모델은 **도구 호출을 요청할 뿐**, 실제 실행은 우리 코드에서 직접 해야 합니다. 모델은 "이 도구를 이 인자로 호출해주세요"라는 구조화된 메시지를 반환할 뿐이에요. 이 실행 루프를 자동화한 것이 바로 에이전트(Agent)입니다.

> 🔥 **실무 팁**: 도구의 `description`은 LLM이 도구를 선택하는 가장 중요한 판단 근거입니다. 다음 원칙을 따르세요:
> - **무엇을 하는지** 명확히 서술 (예: "현재 날씨를 조회합니다")
> - **언제 사용해야 하는지** 명시 (예: "사용자가 날씨를 물어볼 때 사용합니다")
> - **제한사항** 포함 (예: "한국 도시만 지원합니다")
> - 너무 짧거나 모호한 설명은 도구가 무시되거나 오용되는 원인이 됩니다.

> 🔥 **실무 팁**: `return_direct=True`를 설정하면 에이전트가 도구 결과를 그대로 사용자에게 반환합니다. 도구의 출력이 이미 최종 답변 형태일 때(예: 이미지 URL 반환, 포맷된 보고서) 유용하지만, 대부분의 경우에는 `False`(기본값)로 두고 LLM이 결과를 요약하거나 가공하게 하는 것이 더 자연스럽습니다.

## 핵심 정리

| 개념 | 설명 |
|------|------|
| Tool의 3요소 | `name`(이름), `description`(설명), `args_schema`(입력 스키마) |
| `@tool` 데코레이터 | 함수를 LangChain 도구로 변환하는 가장 간단한 방법. 함수명 → 이름, 독스트링 → 설명, 타입 힌트 → 스키마 |
| `args_schema` | Pydantic `BaseModel`로 정의하여 각 인자에 상세 설명과 검증 규칙 부여 |
| `StructuredTool.from_function()` | 기존 함수를 수정 없이 도구로 감싸거나, 동적으로 도구를 생성할 때 사용 |
| `bind_tools()` | 도구 리스트를 LLM에 연결. 새 모델 인스턴스를 반환하며, 원본은 불변 |
| `AIMessage.tool_calls` | 모델의 도구 호출 요청. `name`, `args`, `id`를 포함하는 표준 인터페이스 |
| `ToolMessage` | 도구 실행 결과를 모델에게 전달하는 메시지. `tool_call_id`로 요청과 매칭 |
| 3단계 흐름 | 사용자 질문 → LLM(도구 호출 요청) → 도구 실행 + ToolMessage → LLM(최종 답변) |

## 다음 섹션 미리보기

도구를 정의하고 바인딩하는 방법을 배웠으니, 다음 섹션 **"도구 호출과 실행 루프"**에서는 이 3단계 흐름을 더 정교하게 만들어봅니다. 여러 도구를 연속으로 호출하는 멀티 스텝 실행, 에러 처리와 재시도 전략, 그리고 LCEL 체인 안에서 도구 호출을 통합하는 패턴을 다룹니다.
