# 고급 도구 패턴

> LLM이 도구를 더 똑똑하고 안전하게 사용하도록 제어하는 프로덕션 수준의 패턴을 익힙니다.

## 개요

이 섹션에서는 도구 시스템의 **제어권**을 한 단계 끌어올립니다. 앞서 [11.3: 도구 호출 처리](./03-도구-호출-처리.md)에서 자동화 루프를 만들었다면, 이번에는 그 루프가 **실패하지 않도록**, 그리고 실패하더라도 **우아하게 복구하도록** 만드는 방법을 배웁니다.

**선수 지식**:
- [11.1: 도구 정의와 바인딩](./01-도구-정의와-바인딩.md)의 `bind_tools`, `tool_calls`, `ToolMessage` 개념
- [11.3: 도구 호출 처리](./03-도구-호출-처리.md)의 도구 실행 자동화 루프(`run_tool_loop`)
- LCEL 파이프 연산자(`|`)와 `Runnable` 인터페이스 기본 이해

**학습 목표**:
- `tool_choice` 파라미터로 도구 선택 전략을 커스터마이징할 수 있다
- `ToolException`과 `handle_tool_error`로 도구 에러를 우아하게 처리할 수 있다
- `with_retry()`와 `asyncio.wait_for()`로 재시도와 타임아웃을 구현할 수 있다
- 도구 결과를 검증하고 자기 수정(self-correcting) 패턴을 적용할 수 있다

## 왜 알아야 할까?

프로토타입에서는 도구가 항상 성공한다고 가정해도 괜찮습니다. 하지만 프로덕션에서는 상황이 완전히 다르죠.

외부 API는 언제든 타임아웃이 발생하고, 데이터베이스 연결이 끊기며, LLM이 엉뚱한 도구를 선택하거나 잘못된 인자를 보낼 수 있습니다. **2023년 LangChain 커뮤니티 설문에서 프로덕션 에이전트의 가장 큰 고충 1위는 "도구 실행 실패 처리"였습니다.**

이 섹션에서 배우는 패턴은 단순히 "에러를 잡는 것"이 아닙니다. LLM에게 **"이 도구를 써야 해"**, **"실패하면 이렇게 해"**, **"결과가 이상하면 다시 해봐"**라고 지시하는 **정책 계층(policy layer)**을 구축하는 것입니다. Netflix의 Chaos Engineering처럼, 실패를 전제로 설계하는 사고방식이 에이전트 개발에서도 핵심이 됩니다.

## 핵심 개념

### 개념 1: tool_choice — 도구 선택 제어

> 💡 **비유**: 레스토랑에서 주문한다고 생각해보세요. "아무거나 추천해주세요"(auto)도 있고, "반드시 파스타 중에서 골라주세요"(any), "카르보나라로 해주세요"(특정 도구 지정)도 있죠. `tool_choice`는 LLM에게 메뉴 선택 규칙을 정해주는 것과 같습니다.

`bind_tools()`의 `tool_choice` 파라미터를 사용하면 LLM의 도구 선택 행동을 세밀하게 제어할 수 있습니다.

| tool_choice 값 | 의미 | 언제 사용? |
|---|---|---|
| `"auto"` (기본값) | 모델이 자유롭게 판단 — 도구를 쓸 수도, 안 쓸 수도 | 일반적인 대화형 에이전트 |
| `"any"` 또는 `"required"` | 반드시 도구 1개 이상 호출 | 도구 결과가 반드시 필요한 파이프라인 |
| `"도구이름"` (문자열) | 특정 도구를 강제 호출 | 특정 단계에서 반드시 실행해야 하는 도구 |
| `"none"` | 도구를 호출하지 않음 | 도구 없이 텍스트 답변만 원할 때 |

```python
from langchain_openai import ChatOpenAI
from langchain_core.tools import tool

# --- 도구 정의 ---
@tool
def search_web(query: str) -> str:
    """웹에서 최신 정보를 검색합니다."""
    return f"'{query}' 검색 결과: LangChain 0.3이 2024년에 출시되었습니다."

@tool
def calculate(expression: str) -> str:
    """수학 표현식을 계산합니다."""
    return str(eval(expression))  # 실습용 — 프로덕션에서는 안전한 파서 사용

tools = [search_web, calculate]
llm = ChatOpenAI(model="gpt-4o", temperature=0)

# 1) auto — 모델이 자유 판단
llm_auto = llm.bind_tools(tools, tool_choice="auto")

# 2) any — 반드시 도구 사용 강제
llm_any = llm.bind_tools(tools, tool_choice="any")

# 3) 특정 도구 강제 — 항상 search_web만 호출
llm_forced = llm.bind_tools(tools, tool_choice="search_web")
```

**동적 도구 선택**도 가능합니다. 대화의 맥락에 따라 도구를 바꿔 끼우는 패턴이죠:

```python
def get_tools_for_step(step: str) -> list:
    """단계별로 사용 가능한 도구를 동적으로 결정합니다."""
    if step == "research":
        return [search_web]  # 리서치 단계: 검색만 허용
    elif step == "analysis":
        return [calculate]   # 분석 단계: 계산만 허용
    else:
        return [search_web, calculate]  # 전체 허용

# 리서치 단계에서는 검색 도구만 바인딩
step = "research"
llm_step = llm.bind_tools(
    get_tools_for_step(step),
    tool_choice="any"  # 반드시 도구 사용
)

result = llm_step.invoke("LangChain 최신 버전은?")
print(result.tool_calls)
# [{'name': 'search_web', 'args': {'query': 'LangChain 최신 버전'}, ...}]
```

> ⚠️ **흔한 오해**: `tool_choice="any"`를 쓰면 모든 도구가 동시에 호출된다고 생각하는 분이 있는데요, 사실은 "최소 1개 이상" 호출하라는 의미입니다. 어떤 도구를 선택할지는 여전히 모델이 판단합니다.

### 개념 2: ToolException과 에러 핸들링

> 💡 **비유**: 자동차 에어백을 떠올려보세요. 사고(에러)가 나면 에어백(에러 핸들러)이 터져서 운전자(에이전트)를 보호합니다. 에어백 없이 달리는 건 프로토타입이고, 에어백을 장착하는 게 프로덕션입니다.

LangChain은 `ToolException`이라는 전용 예외 클래스를 제공합니다. 일반 `Exception`과 달리, 이 예외는 에이전트를 **중단시키지 않고** 에러 정보를 LLM에게 전달해서 **스스로 복구**할 기회를 줍니다.

```python
from langchain_core.tools import tool, ToolException

@tool
def get_weather(city: str) -> str:
    """도시의 현재 날씨를 조회합니다. 한국 도시만 지원합니다."""
    supported_cities = {"서울", "부산", "대전", "광주", "제주"}

    if city not in supported_cities:
        # ToolException을 발생시키면 에이전트가 이를 관찰(observation)으로 받음
        raise ToolException(
            f"'{city}'은(는) 지원하지 않는 도시입니다. "
            f"지원 도시: {', '.join(supported_cities)}"
        )
    return f"{city}: 맑음, 22°C"

# handle_tool_error를 True로 설정하면 ToolException을 자동 포착
get_weather.handle_tool_error = True
```

`handle_tool_error`에는 세 가지 설정이 가능합니다:

```python
# 1) True — 예외 메시지를 그대로 LLM에게 전달
get_weather.handle_tool_error = True

# 2) 문자열 — 고정된 에러 메시지 전달
get_weather.handle_tool_error = "날씨 조회에 실패했습니다. 다른 도시를 시도해주세요."

# 3) 커스텀 함수 — 에러를 가공해서 전달
def custom_error_handler(error: ToolException) -> str:
    """에러 정보를 구조화하여 LLM에게 전달합니다."""
    return (
        f"[도구 실행 오류]\n"
        f"오류 내용: {error.args[0]}\n"
        f"권장 조치: 입력값을 수정하거나 다른 도구를 사용하세요."
    )

get_weather.handle_tool_error = custom_error_handler
```

**LangGraph의 ToolNode를 사용하는 경우**에는 노드 레벨에서 에러를 처리할 수도 있습니다:

```python
from langgraph.prebuilt import ToolNode

# ToolNode 레벨에서 에러 처리 설정
tool_node = ToolNode(
    tools=[get_weather, search_web],
    handle_tool_errors=True  # 모든 도구의 에러를 포착
)

# 커스텀 핸들러 함수도 가능
def node_error_handler(error: Exception) -> str:
    return f"도구 실행 중 문제가 발생했습니다: {str(error)[:200]}"

tool_node = ToolNode(
    tools=[get_weather, search_web],
    handle_tool_errors=node_error_handler
)
```

### 개념 3: 재시도(Retry)와 타임아웃(Timeout)

> 💡 **비유**: 전화를 걸었는데 안 받으면 어떻게 하시나요? 보통 몇 초 기다렸다가(타임아웃) 다시 전화합니다(재시도). 그런데 10번이나 계속 걸진 않죠 — 최대 시도 횟수를 정해두고, 안 되면 문자를 보내는(폴백) 식입니다.

#### with_retry() — LCEL 재시도 패턴

LangChain의 모든 `Runnable`은 `.with_retry()` 메서드를 지원합니다. 외부 API 호출처럼 일시적 실패가 발생할 수 있는 컴포넌트에 재시도 로직을 붙이는 가장 간결한 방법입니다.

```python
import httpx
from langchain_core.tools import tool
from langchain_core.runnables import RunnableLambda

@tool
def fetch_stock_price(ticker: str) -> str:
    """주식 시세를 조회합니다. 외부 API를 호출하므로 실패할 수 있습니다."""
    response = httpx.get(
        f"https://api.example.com/stock/{ticker}",
        timeout=5.0
    )
    response.raise_for_status()
    data = response.json()
    return f"{ticker}: ${data['price']}"

# 도구를 RunnableLambda로 감싸서 with_retry 적용
stock_tool_with_retry = RunnableLambda(
    fetch_stock_price.invoke
).with_retry(
    retry_if_exception_type=(httpx.TimeoutException, httpx.HTTPStatusError),
    stop_after_attempt=3,            # 최대 3회 시도
    wait_exponential_jitter=True     # 지수 백오프 + 지터
)

# 체인에서 사용 — 실패 시 자동으로 최대 3회 재시도
# stock_tool_with_retry.invoke({"ticker": "AAPL"})
```

> 🔥 **실무 팁**: `with_retry()`의 범위는 최소화하세요. 전체 체인에 재시도를 걸면 프롬프트 → 모델 → 파서가 모두 재실행됩니다. 실패 가능성이 있는 **특정 컴포넌트만** 감싸는 것이 LangChain 공식 권장 패턴입니다.

#### asyncio 기반 타임아웃

외부 서비스가 응답하지 않을 때 무한 대기하면 전체 에이전트가 멈춥니다. `asyncio.wait_for()`를 활용하면 도구 단위로 타임아웃을 적용할 수 있습니다.

```python
import asyncio
from langchain_core.tools import tool

@tool
async def slow_api_call(query: str) -> str:
    """느린 외부 API를 호출합니다."""
    await asyncio.sleep(10)  # 시뮬레이션: 10초 소요
    return f"결과: {query}"

async def execute_with_timeout(
    tool_func,
    tool_input: dict,
    timeout_seconds: float = 5.0
) -> str:
    """도구를 타임아웃 제한과 함께 실행합니다."""
    try:
        result = await asyncio.wait_for(
            tool_func.ainvoke(tool_input),
            timeout=timeout_seconds
        )
        return result
    except asyncio.TimeoutError:
        return (
            f"[타임아웃] {tool_func.name} 도구가 "
            f"{timeout_seconds}초 내에 응답하지 않았습니다."
        )

# 사용 예
# result = await execute_with_timeout(slow_api_call, {"query": "test"}, timeout_seconds=3.0)
# 출력: [타임아웃] slow_api_call 도구가 3.0초 내에 응답하지 않았습니다.
```

#### 재시도 + 타임아웃 통합 패턴

실전에서는 재시도와 타임아웃을 **함께** 사용하는 경우가 대부분입니다:

```python
import asyncio
import logging
from typing import Any

logger = logging.getLogger(__name__)

async def resilient_tool_execute(
    tool_func,
    tool_input: dict,
    max_retries: int = 3,
    timeout_seconds: float = 10.0,
    backoff_factor: float = 2.0
) -> dict[str, Any]:
    """재시도 + 타임아웃 + 지수 백오프를 통합한 도구 실행기."""
    last_error = None

    for attempt in range(1, max_retries + 1):
        try:
            result = await asyncio.wait_for(
                tool_func.ainvoke(tool_input),
                timeout=timeout_seconds
            )
            return {"status": "success", "result": result, "attempts": attempt}

        except asyncio.TimeoutError:
            last_error = f"타임아웃 ({timeout_seconds}초)"
            logger.warning(
                f"[{tool_func.name}] 시도 {attempt}/{max_retries}: {last_error}"
            )
        except Exception as e:
            last_error = str(e)
            logger.warning(
                f"[{tool_func.name}] 시도 {attempt}/{max_retries}: {last_error}"
            )

        if attempt < max_retries:
            wait_time = backoff_factor ** (attempt - 1)  # 1초 → 2초 → 4초
            logger.info(f"  {wait_time}초 후 재시도...")
            await asyncio.sleep(wait_time)

    return {
        "status": "failed",
        "error": f"{max_retries}회 시도 실패. 마지막 오류: {last_error}",
        "attempts": max_retries
    }
```

### 개념 4: 도구 결과 검증과 자기 수정

> 💡 **비유**: 편의점에서 거스름돈을 받으면 금액이 맞는지 한 번 확인하죠? 도구 결과 검증도 마찬가지입니다. 도구가 "성공"이라고 했지만, 실제 내용이 비어 있거나 기대한 형식이 아닐 수 있거든요. 결과를 **신뢰하기 전에 검증**하는 습관이 프로덕션의 핵심입니다.

도구가 에러 없이 실행되어도 결과가 유효하지 않을 수 있습니다. 예를 들어 검색 도구가 빈 결과를 반환하거나, 계산 도구가 비현실적인 값을 출력할 수 있죠. 이를 위한 **결과 검증 + 자기 수정** 패턴을 구현합니다.

```python
from pydantic import BaseModel, field_validator
from typing import Any

class ToolResultValidator:
    """도구 결과를 검증하고, 실패 시 수정 지시를 생성합니다."""

    def __init__(self):
        self.validators: dict[str, callable] = {}

    def register(self, tool_name: str, validator_func: callable):
        """특정 도구에 대한 검증 함수를 등록합니다."""
        self.validators[tool_name] = validator_func

    def validate(self, tool_name: str, result: Any) -> dict:
        """도구 결과를 검증합니다."""
        if tool_name not in self.validators:
            return {"valid": True, "result": result}

        try:
            self.validators[tool_name](result)
            return {"valid": True, "result": result}
        except ValueError as e:
            return {
                "valid": False,
                "result": result,
                "error": str(e),
                "suggestion": f"'{tool_name}' 결과가 유효하지 않습니다: {e}. "
                              f"다른 입력으로 다시 시도하세요."
            }


# --- 검증 함수 정의 ---
def validate_search_result(result: str):
    """검색 결과가 비어 있거나 너무 짧은지 확인합니다."""
    if not result or len(result.strip()) < 10:
        raise ValueError("검색 결과가 비어 있거나 너무 짧습니다")
    if "결과 없음" in result or "not found" in result.lower():
        raise ValueError("유의미한 검색 결과가 없습니다")

def validate_calculation(result: str):
    """계산 결과가 숫자인지 확인합니다."""
    try:
        float(result.replace(",", ""))
    except ValueError:
        raise ValueError(f"계산 결과가 숫자가 아닙니다: '{result}'")


# 검증기 초기화 및 등록
validator = ToolResultValidator()
validator.register("search_web", validate_search_result)
validator.register("calculate", validate_calculation)

# 사용 예
result = validator.validate("search_web", "")
print(result)
# {'valid': False, 'result': '', 'error': '검색 결과가 비어 있거나 너무 짧습니다',
#  'suggestion': "'search_web' 결과가 유효하지 않습니다: ..."}
```

**자기 수정 루프(Self-correcting Loop)**를 자동화 루프와 결합하면, 잘못된 도구 결과를 LLM에게 피드백하여 재시도하게 만들 수 있습니다:

```python
from langchain_core.messages import AIMessage, HumanMessage, ToolMessage

async def self_correcting_tool_loop(
    llm_with_tools,
    messages: list,
    tool_map: dict,
    validator: ToolResultValidator,
    max_iterations: int = 5,
    max_validation_retries: int = 2
) -> list:
    """도구 결과 검증 + 자기 수정이 포함된 도구 실행 루프."""
    for iteration in range(max_iterations):
        response: AIMessage = await llm_with_tools.ainvoke(messages)
        messages.append(response)

        # 도구 호출이 없으면 최종 응답 — 루프 종료
        if not response.tool_calls:
            break

        for tool_call in response.tool_calls:
            tool_name = tool_call["name"]
            tool_args = tool_call["args"]
            tool_id = tool_call["id"]

            # 도구 실행
            try:
                raw_result = await tool_map[tool_name].ainvoke(tool_args)
            except Exception as e:
                raw_result = f"[실행 오류] {e}"

            # 결과 검증
            validation = validator.validate(tool_name, raw_result)

            if validation["valid"]:
                content = str(raw_result)
            else:
                # 검증 실패 — 수정 지시를 ToolMessage에 포함
                content = (
                    f"[검증 실패] {validation['error']}\n"
                    f"원본 결과: {raw_result}\n"
                    f"권장: {validation['suggestion']}"
                )

            messages.append(ToolMessage(
                content=content,
                tool_call_id=tool_id
            ))

    return messages
```

## 실습: 직접 해보기

아래 코드는 지금까지 배운 네 가지 패턴을 모두 통합한 **안전한 리서치 어시스턴트**입니다. 복사해서 실행해보세요.

```python
"""
고급 도구 패턴 실습: 안전한 리서치 어시스턴트
- tool_choice로 도구 선택 제어
- ToolException으로 에러 핸들링
- 타임아웃과 재시도
- 결과 검증
"""
import asyncio
import logging
from typing import Any

from langchain_openai import ChatOpenAI
from langchain_core.tools import tool, ToolException
from langchain_core.messages import AIMessage, HumanMessage, ToolMessage

logging.basicConfig(level=logging.INFO)
logger = logging.getLogger(__name__)

# ============================================================
# 1단계: 에러 핸들링이 적용된 도구 정의
# ============================================================

SUPPORTED_CITIES = {"서울", "부산", "대전", "광주", "제주", "인천"}
FAKE_DB = {
    "서울": {"인구": "약 950만", "면적": "605km²"},
    "부산": {"인구": "약 330만", "면적": "770km²"},
}

@tool
def city_info(city: str) -> str:
    """한국 도시의 기본 정보를 조회합니다."""
    if city not in SUPPORTED_CITIES:
        raise ToolException(
            f"'{city}'은(는) 지원하지 않는 도시입니다. "
            f"사용 가능: {', '.join(sorted(SUPPORTED_CITIES))}"
        )
    info = FAKE_DB.get(city, {"정보": "준비 중"})
    return f"{city} 정보: {info}"

# ToolException을 LLM에게 전달하는 커스텀 핸들러
def handle_city_error(error: ToolException) -> str:
    return f"[오류] {error.args[0]}\n다른 도시를 시도하거나, 지원 도시 목록을 확인하세요."

city_info.handle_tool_error = handle_city_error


@tool
def safe_calculate(expression: str) -> str:
    """안전하게 수학 계산을 수행합니다. 사칙연산만 지원합니다."""
    import re
    # 허용된 문자만 통과 (숫자, 연산자, 괄호, 공백, 소수점)
    if not re.match(r'^[\d\s\+\-\*\/\(\)\.]+$', expression):
        raise ToolException(
            f"안전하지 않은 표현식입니다: '{expression}'. "
            f"숫자와 사칙연산(+, -, *, /)만 사용하세요."
        )
    try:
        result = eval(expression)  # 검증 후 실행
        return str(result)
    except ZeroDivisionError:
        raise ToolException("0으로 나눌 수 없습니다.")
    except Exception as e:
        raise ToolException(f"계산 오류: {e}")

safe_calculate.handle_tool_error = True


# ============================================================
# 2단계: 결과 검증기 설정
# ============================================================

class ResultValidator:
    """도구 결과 검증기."""
    def __init__(self):
        self._rules: dict[str, callable] = {}

    def add_rule(self, tool_name: str, check: callable):
        self._rules[tool_name] = check

    def check(self, tool_name: str, result: str) -> dict[str, Any]:
        if tool_name not in self._rules:
            return {"ok": True, "result": result}
        try:
            self._rules[tool_name](result)
            return {"ok": True, "result": result}
        except ValueError as e:
            return {"ok": False, "result": result, "reason": str(e)}

# 검증 규칙 등록
rv = ResultValidator()
rv.add_rule("city_info", lambda r: (_ for _ in ()).throw(ValueError("결과 없음")) if len(r) < 5 else None)
rv.add_rule("safe_calculate", lambda r: float(r.replace(",", "")))  # 숫자인지 확인


# ============================================================
# 3단계: 타임아웃 + 재시도 실행기
# ============================================================

async def run_tool_safe(
    tool_func, args: dict,
    timeout: float = 5.0, retries: int = 2
) -> str:
    """타임아웃과 재시도가 적용된 안전한 도구 실행."""
    for attempt in range(1, retries + 1):
        try:
            result = await asyncio.wait_for(
                tool_func.ainvoke(args),
                timeout=timeout
            )
            return result
        except asyncio.TimeoutError:
            logger.warning(f"[{tool_func.name}] 시도 {attempt}: 타임아웃 ({timeout}초)")
        except Exception as e:
            logger.warning(f"[{tool_func.name}] 시도 {attempt}: {e}")
            return str(e)  # ToolException은 handle_tool_error가 처리

        if attempt < retries:
            await asyncio.sleep(2 ** (attempt - 1))  # 지수 백오프

    return f"[실패] {tool_func.name}: {retries}회 시도 후에도 응답 없음"


# ============================================================
# 4단계: 통합 — 자기 수정 도구 루프
# ============================================================

async def advanced_tool_loop(
    query: str,
    tool_choice: str = "auto",
    max_iterations: int = 5
):
    """고급 패턴이 모두 적용된 도구 실행 루프."""
    tools = [city_info, safe_calculate]
    tool_map = {t.name: t for t in tools}

    llm = ChatOpenAI(model="gpt-4o", temperature=0)
    llm_with_tools = llm.bind_tools(tools, tool_choice=tool_choice)

    messages = [HumanMessage(content=query)]

    for i in range(max_iterations):
        logger.info(f"\n--- 반복 {i + 1} ---")
        response: AIMessage = await llm_with_tools.ainvoke(messages)
        messages.append(response)

        # 도구 호출이 없으면 최종 답변
        if not response.tool_calls:
            logger.info(f"최종 답변: {response.content[:100]}...")
            return response.content

        # 각 도구 호출 처리
        for tc in response.tool_calls:
            logger.info(f"도구 호출: {tc['name']}({tc['args']})")

            # 안전한 실행 (타임아웃 + 재시도)
            result = await run_tool_safe(
                tool_map[tc["name"]], tc["args"],
                timeout=10.0, retries=2
            )

            # 결과 검증
            validation = rv.check(tc["name"], result)
            if validation["ok"]:
                content = result
                logger.info(f"  ✓ 검증 통과")
            else:
                content = (
                    f"[검증 실패] {validation['reason']}\n"
                    f"원본: {result}\n다른 입력으로 재시도하세요."
                )
                logger.warning(f"  ✗ 검증 실패: {validation['reason']}")

            messages.append(ToolMessage(
                content=content,
                tool_call_id=tc["id"]
            ))

        # 마지막 반복 이후에는 도구 호출 없이 최종 응답 유도
        if i == max_iterations - 2:
            llm_with_tools = llm.bind_tools(tools, tool_choice="auto")

    return "최대 반복 횟수에 도달했습니다."


# ============================================================
# 실행
# ============================================================

async def main():
    # 시나리오 1: 정상 케이스
    print("=== 시나리오 1: 서울 정보 조회 ===")
    answer = await advanced_tool_loop("서울의 인구와 면적을 알려주세요.")
    print(f"답변: {answer}\n")

    # 시나리오 2: 에러 핸들링 — 지원하지 않는 도시
    print("=== 시나리오 2: 지원하지 않는 도시 ===")
    answer = await advanced_tool_loop("도쿄의 인구를 알려주세요.")
    print(f"답변: {answer}\n")

    # 시나리오 3: 도구 강제 선택
    print("=== 시나리오 3: 계산 도구 강제 ===")
    answer = await advanced_tool_loop(
        "서울 인구가 950만이면 부산의 3배는?",
        tool_choice="safe_calculate"
    )
    print(f"답변: {answer}\n")

# asyncio.run(main())
# 실행 결과 (시나리오 1 예시):
# === 시나리오 1: 서울 정보 조회 ===
# --- 반복 1 ---
# 도구 호출: city_info({'city': '서울'})
#   ✓ 검증 통과
# --- 반복 2 ---
# 최종 답변: 서울의 인구는 약 950만 명이며, 면적은 605km²입니다.
```

## 더 깊이 알아보기

### Circuit Breaker 패턴의 기원

도구 에러 핸들링에서 자주 언급되는 **서킷 브레이커(Circuit Breaker)** 패턴은 원래 전기 공학에서 온 개념입니다. 과전류가 흐르면 자동으로 회로를 차단하는 차단기(circuit breaker)에서 이름을 딴 거죠.

소프트웨어에서 이 패턴을 처음 체계화한 사람은 **마이클 나이가드(Michael Nygard)**로, 2007년 그의 저서 *Release It!*에서 소개했습니다. 외부 서비스 호출이 반복 실패할 때, 일정 횟수 이상 실패하면 아예 호출을 **차단(Open)** 상태로 만들고, 일정 시간 후 **반개방(Half-Open)** 상태에서 한 번 시도해본 뒤 성공하면 **닫힘(Closed)** 상태로 복귀하는 3단계 상태 머신입니다.

이 패턴은 Netflix의 **Hystrix** 라이브러리(2012)를 통해 마이크로서비스 세계에서 크게 유명해졌고, 지금은 LLM 에이전트의 도구 호출에서도 동일한 원리가 적용됩니다. LangChain의 `with_retry()`는 가장 단순한 형태의 회복력 패턴이지만, 프로덕션 에이전트에서는 서킷 브레이커를 구현하여 반복 실패하는 도구를 임시로 비활성화하는 전략도 사용합니다.

### "Let it crash" 철학

흥미롭게도, Erlang/OTP의 "Let it crash" 철학과 LangChain의 `handle_tool_error` 접근법은 일맥상통합니다. 1986년 에릭슨에서 개발된 Erlang은 통신 시스템의 99.9999999% 가용성을 위해 "프로세스가 죽게 놔두되, 감독자(supervisor)가 즉시 재시작한다"는 철학을 채택했습니다. LLM 에이전트에서도 도구 실패를 두려워하지 말고, 대신 **에러 정보를 LLM에게 전달해서 스스로 복구하게 하는** 것이 같은 원리입니다.

## 흔한 오해와 팁

> ⚠️ **흔한 오해**: "도구에서 `try/except`로 모든 에러를 잡아서 빈 문자열을 반환하면 되겠지"라고 생각하기 쉽습니다. 하지만 이렇게 하면 LLM은 도구가 성공했다고 판단하고 잘못된 정보로 답변을 생성합니다. **에러는 숨기지 말고, 구조화된 형태로 LLM에게 전달**하는 것이 핵심입니다. `ToolException`이 바로 이 목적으로 설계되었습니다.

> 💡 **알고 계셨나요?**: LangChain의 `tool_choice` 파라미터는 OpenAI의 Function Calling API에서 유래했습니다. OpenAI가 2023년 6월에 Function Calling을 발표하면서 `function_call` 파라미터를 도입했고, 이후 `tool_choice`로 이름이 바뀌었는데요. LangChain은 이를 **프로바이더 독립적으로 추상화**하여, OpenAI든 Anthropic이든 동일한 `tool_choice="any"` 문법으로 사용할 수 있게 만들었습니다.

> 🔥 **실무 팁**: 타임아웃 값은 도구마다 다르게 설정하세요. 데이터베이스 조회는 2-3초면 충분하지만, 웹 검색은 10초 이상 필요할 수 있습니다. 도구별 타임아웃을 **설정 객체로 관리**하면 운영 중에도 쉽게 튜닝할 수 있습니다:
> ```python
> TOOL_TIMEOUTS = {
>     "db_query": 3.0,
>     "web_search": 15.0,
>     "calculate": 2.0,
>     "default": 10.0
> }
> timeout = TOOL_TIMEOUTS.get(tool_name, TOOL_TIMEOUTS["default"])
> ```

## 핵심 정리

| 개념 | 설명 |
|------|------|
| `tool_choice` | `bind_tools()`의 파라미터로, `"auto"`, `"any"`, `"none"`, 또는 특정 도구명을 지정하여 LLM의 도구 선택을 제어 |
| `ToolException` | 도구 에러 전용 예외 클래스. 에이전트를 중단시키지 않고 에러 정보를 LLM에게 관찰(observation)으로 전달 |
| `handle_tool_error` | `True`, 문자열, 또는 커스텀 함수를 설정하여 `ToolException` 발생 시 LLM에게 전달할 메시지를 제어 |
| `with_retry()` | `Runnable`의 메서드로, 특정 예외 발생 시 지수 백오프와 함께 자동 재시도 |
| 타임아웃 | `asyncio.wait_for()`로 도구 실행에 시간 제한을 적용하여 무한 대기 방지 |
| 결과 검증 | 도구 실행 성공 후에도 결과의 유효성을 별도 검증하고, 실패 시 LLM에게 피드백 |
| 자기 수정 루프 | 검증 실패 시 에러 정보를 `ToolMessage`로 전달하여 LLM이 다른 입력으로 재시도하게 유도 |
| 서킷 브레이커 | 반복 실패하는 도구를 임시 비활성화하는 회복력 패턴 (프로덕션 고급 패턴) |
