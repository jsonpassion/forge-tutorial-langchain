# 확장성과 성능

> LangChain 애플리케이션이 수천 명의 동시 사용자를 감당하게 만드는 비동기 처리, 동시성 제어, 데이터베이스 최적화, 수평 확장 전략을 배웁니다.

## 왜 알아야 할까?

개발 환경에서 잘 작동하던 LangChain 애플리케이션을 프로덕션에 배포하면, 첫 번째로 맞닥뜨리는 벽이 바로 **성능**입니다. 로컬에서 1명이 쓸 때는 2초면 충분했던 RAG 파이프라인이, 100명이 동시에 요청하면 20초가 걸리거나 아예 타임아웃으로 실패하죠. 왜 그럴까요?

LLM API 호출은 본질적으로 **I/O 바운드** 작업입니다. 네트워크를 통해 외부 API에 요청을 보내고 응답을 기다리는 시간이 대부분이거든요. 이때 동기(synchronous) 방식으로 처리하면, 하나의 요청이 완료될 때까지 서버는 그냥 *멍하니* 기다리고 있는 셈입니다. 마치 식당에서 웨이터가 한 테이블 주문을 주방에 넣고, 음식이 나올 때까지 다른 테이블은 아예 쳐다보지도 않는 것과 같죠.

실제 사례를 보면 그 차이가 극명합니다. 어떤 팀은 동기 방식에서 비동기로 전환한 것만으로 **응답 시간이 30초에서 3초 미만으로** 줄고, **동시 처리 사용자 수가 50배** 늘었다고 보고합니다. 비동기 처리, 수평 확장, 데이터베이스 최적화는 단순한 "있으면 좋은 것"이 아니라, 프로덕션의 **필수 조건**입니다.

하지만 이 모든 것을 한꺼번에 도입할 필요는 없습니다. 가장 간단한 비동기 전환부터 시작하여, 트래픽이 늘어남에 따라 단계적으로 확장 전략을 적용하는 것이 실전에서의 올바른 접근법입니다. 이 세션은 그 **점진적 확장 경로**를 따라갑니다.

## 핵심 개념

### 개념 1: 비동기 처리와 스트리밍 — 웨이터의 멀티태스킹

> 💡 **비유**: 식당을 생각해보세요. 동기 방식 웨이터는 A 테이블 주문을 받아 주방에 넣고, 음식이 나올 때까지 부동 자세로 서 있습니다. 비동기 방식 웨이터는 A 주문을 넣자마자 B 테이블로 달려가 주문을 받고, C 테이블에 물을 따르면서 동시에 여러 테이블을 서빙합니다. 주방(LLM API)의 조리 속도는 똑같지만, 웨이터(서버)가 *대기 시간*을 활용하느냐 낭비하느냐가 핵심입니다.

LangChain의 모든 Runnable 컴포넌트는 동기와 비동기 인터페이스를 모두 제공합니다:

| 동기 메서드 | 비동기 메서드 | 용도 |
|-------------|---------------|------|
| `invoke()` | `ainvoke()` | 단일 입력 처리 |
| `batch()` | `abatch()` | 다수 입력 병렬 처리 |
| `stream()` | `astream()` | 토큰 단위 스트리밍 |
| — | `astream_events()` | 체인 전체 이벤트 스트리밍 |

#### 기본 비동기 처리

```python
import asyncio
from langchain_openai import ChatOpenAI
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser

# 비동기 체인 구성
prompt = ChatPromptTemplate.from_template("'{topic}'에 대해 한 문장으로 설명해주세요.")
model = ChatOpenAI(model="gpt-4o-mini", temperature=0.7)
chain = prompt | model | StrOutputParser()

async def process_single():
    """단일 비동기 호출"""
    result = await chain.ainvoke({"topic": "양자 컴퓨팅"})
    print(result)

async def process_batch():
    """배치 비동기 호출 — 여러 입력을 동시에 처리"""
    topics = [
        {"topic": "양자 컴퓨팅"},
        {"topic": "블록체인"},
        {"topic": "유전자 편집"},
        {"topic": "핵융합"},
        {"topic": "뇌-컴퓨터 인터페이스"},
    ]
    # max_concurrency로 동시 실행 수 제한
    results = await chain.abatch(
        topics,
        config={"max_concurrency": 3}  # 최대 3개 동시 처리
    )
    for topic, result in zip(topics, results):
        print(f"[{topic['topic']}] {result[:50]}...")

# 실행
asyncio.run(process_batch())
# 출력: 5개 주제가 3개씩 병렬 처리되어 순차 대비 약 2배 빠르게 완료
```

`max_concurrency`는 동시 실행 수를 제한하는 안전장치입니다. API 레이트 리밋에 걸리지 않도록 적절한 값을 설정하는 것이 중요합니다.

#### 스트리밍으로 체감 속도 향상

LLM 응답은 길 때 수 초가 걸릴 수 있는데요, 전체 응답을 기다리지 않고 토큰이 생성되는 대로 실시간으로 전달하면 사용자 체감 속도가 크게 향상됩니다:

```python
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
from langchain_openai import ChatOpenAI
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser

app = FastAPI()
chain = (
    ChatPromptTemplate.from_template("{question}에 대해 자세히 설명해주세요.")
    | ChatOpenAI(model="gpt-4o", temperature=0.7, streaming=True)
    | StrOutputParser()
)

@app.post("/stream")
async def stream_response(question: str):
    """SSE(Server-Sent Events)로 토큰 스트리밍"""
    async def generate():
        async for chunk in chain.astream({"question": question}):
            # SSE 형식으로 전송
            yield f"data: {chunk}\n\n"
        yield "data: [DONE]\n\n"

    return StreamingResponse(
        generate(),
        media_type="text/event-stream"
    )
```

> 🔥 **실무 팁**: `astream`과 `astream_events`의 차이를 이해하세요. `astream`은 최종 출력만 스트리밍하고, `astream_events`는 체인 내부의 모든 컴포넌트(프롬프트 구성, 모델 호출, 파서 실행 등)의 이벤트를 상세히 스트리밍합니다. 디버깅이나 로깅에는 `astream_events`가, 사용자 대면 UI에는 `astream`이 적합합니다.

#### 레이트 리밋 제어

```python
from langchain_core.rate_limiters import InMemoryRateLimiter

# 토큰 버킷 기반 레이트 리미터: 초당 10개 요청 허용
rate_limiter = InMemoryRateLimiter(
    requests_per_second=10,       # 초당 허용 요청 수
    check_every_n_seconds=0.1,    # 체크 간격
    max_bucket_size=20            # 버스트 허용 한도
)

# 모델에 레이트 리미터 적용
model = ChatOpenAI(
    model="gpt-4o-mini",
    rate_limiter=rate_limiter
)
```

`InMemoryRateLimiter`는 토큰 버킷(Token Bucket) 알고리즘을 사용합니다. 여기서 "토큰"은 LLM 토큰이 아니라, 요청 허용량을 추적하는 단위입니다. 버킷에 토큰이 있으면 요청이 즉시 처리되고, 없으면 토큰이 채워질 때까지 대기하죠.

### 개념 2: 동시성 제어에서 작업 큐로 — 단계적 확장

> 💡 **비유**: 작은 카페에서 바리스타 한 명이 여러 주문을 능숙하게 번갈아 처리하는 것(비동기 동시성)은, 손님이 적을 때는 충분합니다. 하지만 줄이 건물 밖까지 이어지면? 바리스타를 더 고용하고(수평 확장), 주문 접수 대장(작업 큐)에 적어서 분업해야 합니다. 첫 번째 단계는 한 명의 바리스타를 더 효율적으로 만드는 것이고, 두 번째 단계는 바리스타를 늘리는 것입니다.

#### 1단계: asyncio 기반 동시성 제어 (단일 서버)

수평 확장 전에, 먼저 **하나의 서버 안에서** 동시 요청을 안전하게 관리하는 패턴부터 익혀봅시다. `asyncio.Semaphore`를 사용하면 동시 실행 수를 제한하여 서버 과부하를 방지할 수 있습니다:

```python
import asyncio
from langchain_openai import ChatOpenAI
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser

# 세마포어: 동시에 최대 10개 요청만 LLM에 전달
llm_semaphore = asyncio.Semaphore(10)

chain = (
    ChatPromptTemplate.from_template("다음 질문에 답해주세요: {question}")
    | ChatOpenAI(model="gpt-4o-mini")
    | StrOutputParser()
)

async def safe_invoke(question: str) -> str:
    """세마포어로 동시 LLM 호출 수를 제한"""
    async with llm_semaphore:
        return await chain.ainvoke({"question": question})

async def handle_burst(questions: list[str]) -> list[str]:
    """50개 요청이 동시에 와도 10개씩만 LLM에 전달"""
    tasks = [safe_invoke(q) for q in questions]
    return await asyncio.gather(*tasks)
```

이 패턴만으로도 단일 서버에서 수십~수백 명의 동시 요청을 안정적으로 처리할 수 있습니다. 하지만 한계가 있습니다:

- **단일 장애점**: 서버가 죽으면 모든 처리가 중단됩니다
- **CPU 바운드 한계**: 임베딩 생성, 문서 전처리 등 CPU를 많이 쓰는 작업은 비동기만으로 빨라지지 않습니다
- **재시도 어려움**: 서버가 재시작되면 처리 중이던 작업이 모두 유실됩니다

이런 한계를 넘으려면 **작업 큐**가 필요합니다.

#### 2단계: 작업 큐란 무엇인가?

작업 큐(Task Queue)는 "할 일 목록"과 비슷합니다. 웹 서버가 요청을 받으면 직접 처리하는 대신, **할 일 목록(큐)**에 적어두고 즉시 "접수 완료"라고 응답합니다. 그러면 별도의 **워커(worker)** 프로세스들이 큐에서 할 일을 하나씩 꺼내 처리하죠.

```
[클라이언트] → [웹 서버: "접수 완료!"] → [메시지 큐(Redis)] → [워커 1] → [LLM API]
                                                              → [워커 2] → [LLM API]
                                                              → [워커 3] → [LLM API]
```

이 구조의 핵심 장점은:
- **웹 서버와 처리를 분리**: 웹 서버는 요청 접수만 하므로 빠르게 응답
- **독립적 확장**: 워커만 늘리면 처리량 증가
- **작업 보존**: 워커가 죽어도 큐에 남아 있는 작업은 다른 워커가 처리
- **재시도 내장**: 실패한 작업을 자동으로 재시도

**Celery**는 Python에서 가장 널리 쓰이는 작업 큐 라이브러리입니다. **Redis**를 메시지 브로커(큐 역할)로 사용하는 조합이 가장 일반적이죠.

#### 3단계: Celery + Redis 작업 큐 구현

```python
# celery_app.py — Celery 워커 설정
from celery import Celery

# Redis를 메시지 브로커와 결과 백엔드로 사용
celery_app = Celery(
    "langchain_worker",
    broker="redis://localhost:6379/0",       # 작업 큐
    backend="redis://localhost:6379/1",      # 결과 저장소
)

# 워커 설정
celery_app.conf.update(
    task_serializer="json",
    result_serializer="json",
    accept_content=["json"],
    task_acks_late=True,           # 작업 완료 후 ACK (안정성 ↑)
    worker_prefetch_multiplier=1,  # 워커당 한 번에 1개 작업만 가져옴
    task_time_limit=300,           # 작업 타임아웃: 5분
    task_soft_time_limit=240,      # 소프트 타임아웃: 4분 (정리 시간 확보)
)
```

```python
# tasks.py — LangChain 작업 정의
from celery_app import celery_app
from langchain_openai import ChatOpenAI
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser

@celery_app.task(bind=True, max_retries=3, default_retry_delay=60)
def run_rag_query(self, question: str, user_id: str) -> dict:
    """RAG 쿼리를 Celery 작업으로 실행"""
    try:
        chain = (
            ChatPromptTemplate.from_template(
                "다음 질문에 답해주세요: {question}"
            )
            | ChatOpenAI(model="gpt-4o-mini")
            | StrOutputParser()
        )
        # Celery 워커 내에서는 동기 invoke 사용
        result = chain.invoke({"question": question})
        return {"status": "success", "answer": result, "user_id": user_id}

    except Exception as exc:
        # 재시도 로직: 지수 백오프
        raise self.retry(exc=exc, countdown=60 * (2 ** self.request.retries))
```

```python
# api.py — FastAPI에서 Celery 작업 호출
from fastapi import FastAPI
from tasks import run_rag_query

app = FastAPI()

@app.post("/query")
async def submit_query(question: str, user_id: str):
    """비동기로 작업을 큐에 넣고 즉시 응답"""
    task = run_rag_query.delay(question, user_id)
    return {"task_id": task.id, "status": "queued"}

@app.get("/result/{task_id}")
async def get_result(task_id: str):
    """작업 결과 조회 (폴링 방식)"""
    from celery.result import AsyncResult
    result = AsyncResult(task_id)

    if result.ready():
        return {"status": "completed", "data": result.get()}
    elif result.failed():
        return {"status": "failed", "error": str(result.result)}
    else:
        return {"status": "processing"}
```

수평 확장은 워커 수를 늘리는 것만큼 간단합니다:

```bash
# 워커 1대 시작
celery -A celery_app worker --loglevel=info --concurrency=4

# 트래픽 증가 시, 다른 서버에서 워커 추가 시작
celery -A celery_app worker --loglevel=info --concurrency=8 --hostname=worker2@%h
```

> **언제 어떤 전략을 쓸까?** 비동기(`asyncio`)와 작업 큐(Celery)는 보완 관계입니다. 동시 사용자 수백 명 이하에 I/O 바운드 작업 위주라면 `asyncio`만으로 충분합니다. 하지만 CPU 바운드 작업(임베딩 생성, 문서 전처리)이 섞여 있거나, 수천 명 이상의 트래픽을 감당해야 하거나, 작업 실패 시 자동 재시도가 필요하다면 Celery 작업 큐를 도입하세요.

### 개념 3: 데이터베이스 최적화 — 도서관의 색인 시스템

> 💡 **비유**: 도서관에 100만 권의 책이 있는데 색인(인덱스)이 없다면, 원하는 책을 찾으려면 모든 서가를 하나하나 뒤져야 합니다. 하지만 잘 정리된 카탈로그(인덱스)가 있다면 몇 초면 찾을 수 있죠. 데이터베이스 최적화도 마찬가지입니다. 벡터 스토어의 인덱스 설정, 커넥션 풀링, 쿼리 최적화가 RAG 시스템의 응답 속도를 좌우합니다.

RAG 시스템에서 벡터 검색은 전체 파이프라인의 병목이 되기 쉽습니다. 특히 문서가 수십만~수백만 건으로 늘어나면 최적화 없이는 검색 속도가 급격히 느려지거든요. 간단한 캐싱부터 시작해서, 커넥션 풀링, 인덱스 최적화까지 단계별로 살펴보겠습니다.

#### 1단계: 인메모리 캐싱으로 반복 검색 최소화

가장 먼저 적용할 수 있는 최적화는 **캐싱**입니다. 같은 질문에 대한 벡터 검색을 반복하지 않도록, [20.2: 비용 최적화](ch20/session2.md)에서 배운 캐시 전략을 벡터 검색에도 적용합니다:

```python
import hashlib
import json
import redis

class VectorSearchCache:
    """벡터 검색 결과를 캐싱하는 래퍼"""

    def __init__(self, vectorstore, redis_client: redis.Redis, ttl: int = 3600):
        self.vectorstore = vectorstore
        self.redis = redis_client
        self.ttl = ttl  # 캐시 만료 시간(초)

    def _cache_key(self, query: str, k: int) -> str:
        """쿼리와 검색 수로 캐시 키 생성"""
        raw = f"{query}:{k}"
        return f"vsearch:{hashlib.sha256(raw.encode()).hexdigest()}"

    async def asimilarity_search(self, query: str, k: int = 4) -> list:
        """캐시 우선 벡터 검색"""
        key = self._cache_key(query, k)

        # 1. 캐시 히트 확인
        cached = self.redis.get(key)
        if cached:
            return json.loads(cached)

        # 2. 캐시 미스 → 실제 벡터 검색
        docs = await self.vectorstore.asimilarity_search(query, k=k)
        serialized = [
            {"content": doc.page_content, "metadata": doc.metadata}
            for doc in docs
        ]

        # 3. 결과 캐싱
        self.redis.setex(key, self.ttl, json.dumps(serialized, ensure_ascii=False))
        return docs
```

캐시만으로도 반복 쿼리의 응답 시간이 수백 ms에서 수 ms로 줄어듭니다. 하지만 캐시 미스인 경우의 검색 속도를 높이려면 DB 수준 최적화가 필요합니다.

#### 2단계: 커넥션 풀링 — DB 연결 비용 절감

매 요청마다 DB 연결을 새로 생성하고 끊는 것은 큰 비용입니다 — TCP 핸드셰이크, 인증, SSL 협상 등이 매번 반복되거든요. **커넥션 풀**을 사용하면 미리 만들어둔 연결을 빌려 쓰고 반납하는 방식으로, 이 비용을 제거할 수 있습니다:

```python
from sqlalchemy import create_engine
from sqlalchemy.pool import QueuePool
from langchain_postgres import PGVector
from langchain_openai import OpenAIEmbeddings

# 커넥션 풀 설정으로 DB 연결 오버헤드 최소화
CONNECTION_STRING = "postgresql+psycopg://user:pass@localhost:5432/vectordb"

engine = create_engine(
    CONNECTION_STRING,
    poolclass=QueuePool,
    pool_size=20,             # 기본 연결 수
    max_overflow=10,          # 추가 허용 연결 수 (최대 30)
    pool_timeout=30,          # 연결 대기 타임아웃(초)
    pool_recycle=3600,        # 1시간마다 연결 재생성 (stale 방지)
    pool_pre_ping=True,       # 사용 전 연결 유효성 검사
)

# PGVector에 커넥션 풀이 적용된 엔진 전달
vectorstore = PGVector(
    embeddings=OpenAIEmbeddings(model="text-embedding-3-small"),
    collection_name="documents",
    connection=engine,
    use_jsonb=True,           # JSONB 메타데이터 — 필터링 성능 ↑
)
```

#### 3단계: 벡터 인덱스 최적화 — pgvector와 HNSW

데이터가 수십만 건을 넘어가면, 벡터 검색의 속도가 급격히 느려집니다. 기본적으로 벡터 검색은 모든 벡터와의 거리를 계산하는 **전수 검색(brute-force)**이기 때문이죠. 이를 해결하는 것이 **인덱스**입니다.

**pgvector**는 PostgreSQL에 벡터 검색 기능을 추가하는 확장입니다. 별도의 벡터 DB를 운영하지 않아도, 이미 사용 중인 PostgreSQL에서 벡터 검색을 수행할 수 있는 장점이 있습니다. pgvector가 지원하는 인덱스 중 **HNSW(Hierarchical Navigable Small World)**는 현재 가장 널리 쓰이는 근사 최근접 이웃(ANN) 알고리즘입니다.

HNSW의 핵심 아이디어는 간단합니다. 벡터들을 **계층적 그래프**로 연결해두면, 몇 번의 "점프"만으로 가장 가까운 이웃을 찾을 수 있다는 것이죠. 정확한 전수 검색 대비 약간의 정확도를 포기하는 대신, 검색 속도를 수십~수백 배 향상시킵니다:

```python
from sqlalchemy import text

with engine.connect() as conn:
    # HNSW 인덱스 생성 — 근사 최근접 이웃 검색
    conn.execute(text("""
        CREATE INDEX IF NOT EXISTS idx_embedding_hnsw
        ON langchain_pg_embedding
        USING hnsw (embedding vector_cosine_ops)
        WITH (
            m = 16,              -- 노드당 연결 수 (높을수록 정확도 ↑, 메모리 ↑)
            ef_construction = 64 -- 인덱스 빌드 시 탐색 범위 (높을수록 빌드 느림, 정확도 ↑)
        );
    """))

    # 메타데이터 필터링용 인덱스
    conn.execute(text("""
        CREATE INDEX IF NOT EXISTS idx_metadata_gin
        ON langchain_pg_embedding
        USING gin (cmetadata jsonb_path_ops);
    """))

    conn.commit()
    print("인덱스 생성 완료")
```

> **HNSW 파라미터 가이드**: `m`은 그래프에서 각 노드의 연결 수입니다. 16이 일반적인 기본값이며, 정확도가 더 필요하면 32로 높이세요(메모리 사용량 증가). `ef_construction`은 인덱스를 만들 때의 탐색 범위로, 64~128이 적정 범위입니다. 인덱스 빌드는 한 번만 하므로 좀 느려도 괜찮지만, `m`은 검색 때마다 영향을 주므로 신중하게 설정하세요.

### 개념 4: 수평 확장 아키텍처 — Kubernetes와 오토스케일링

> 💡 **비유**: 비동기 처리가 한 명의 웨이터를 더 효율적으로 만드는 것이라면, 수평 확장은 웨이터를 더 고용하는 것입니다. 그리고 오토스케일링은 손님이 많은 점심시간에 자동으로 아르바이트생을 투입하고, 한가한 오후에는 귀가시키는 **자동 인력 관리** 시스템이죠.

단일 서버의 비동기 처리와 Celery 워커로도 상당한 트래픽을 감당할 수 있지만, 서비스가 성장하면 결국 **여러 서버에 걸친 수평 확장**이 필요합니다. Kubernetes(K8s)는 이 수평 확장을 자동화하는 컨테이너 오케스트레이션 플랫폼입니다.

> 📌 Docker 컨테이너 빌드와 기본 배포 설정은 [17.5: 프로덕션 배포](ch17/session5.md)를 참조하세요. 여기서는 이미 컨테이너화된 애플리케이션을 **수평 확장**하는 데 집중합니다.

#### HPA(Horizontal Pod Autoscaler) — 트래픽 기반 자동 확장

Kubernetes의 핵심 기능 중 하나는 **트래픽에 따라 자동으로 파드(서버 인스턴스)를 늘리고 줄이는** 오토스케일링입니다:

```yaml
# k8s/hpa.yaml — 수평 파드 오토스케일러
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: langchain-api-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: langchain-api
  minReplicas: 2     # 최소 2개 파드 유지 (가용성 보장)
  maxReplicas: 20    # 최대 20개까지 확장
  metrics:
    # CPU 사용률 기반 스케일링
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70   # CPU 70% 초과 시 스케일 아웃
    # 메모리 사용률 기반 스케일링
    - type: Resource
      resource:
        name: memory
        target:
          type: Utilization
          averageUtilization: 80   # 메모리 80% 초과 시 스케일 아웃
  behavior:
    scaleUp:
      stabilizationWindowSeconds: 60   # 1분 관찰 후 스케일 업
      policies:
        - type: Pods
          value: 4                      # 한 번에 최대 4개 파드 추가
          periodSeconds: 60
    scaleDown:
      stabilizationWindowSeconds: 300  # 5분 관찰 후 스케일 다운 (급격한 축소 방지)
      policies:
        - type: Pods
          value: 2                      # 한 번에 최대 2개 파드 제거
          periodSeconds: 120
```

이 설정의 동작을 풀어보면:
- 평소에는 2개의 파드가 트래픽을 처리
- CPU가 70%를 넘으면 60초 관찰 후 최대 4개 파드를 추가로 생성
- 트래픽이 줄어들면 5분 관찰 후 천천히 파드를 제거 (급격한 축소로 인한 서비스 불안정 방지)
- 어떤 상황에서도 최소 2개, 최대 20개 파드 유지

## 실습: 직접 해보기

다음은 지금까지 배운 비동기 처리, 동시성 제어, 캐싱을 통합한 **프로덕션 수준의 비동기 RAG 서버**입니다. 이전 세션에서 배운 보안 계층과 비용 최적화도 함께 녹여냈습니다:

```python
# production_rag_server.py
# 비동기 처리 + 커넥션 풀링 + 레이트 리밋 + 캐싱을 결합한 프로덕션 RAG 서버

import asyncio
import hashlib
import json
import time
import logging
from typing import AsyncGenerator

from fastapi import FastAPI, HTTPException
from fastapi.responses import StreamingResponse
from pydantic import BaseModel, Field
from contextlib import asynccontextmanager

from langchain_openai import ChatOpenAI, OpenAIEmbeddings
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser
from langchain_core.rate_limiters import InMemoryRateLimiter
from langchain_core.runnables import RunnableConfig

import redis.asyncio as aioredis  # 비동기 Redis 클라이언트

# ─── 로깅 설정 ───
logging.basicConfig(level=logging.INFO)
logger = logging.getLogger("rag_server")

# ─── 설정 상수 ───
MODEL_NAME = "gpt-4o-mini"
EMBEDDING_MODEL = "text-embedding-3-small"
CACHE_TTL = 3600  # 캐시 만료: 1시간
MAX_CONCURRENT_REQUESTS = 50  # 최대 동시 요청 수
RATE_LIMIT_RPS = 20  # 초당 허용 요청 수


# ─── 요청/응답 모델 ───
class QueryRequest(BaseModel):
    question: str = Field(..., min_length=1, max_length=2000)
    stream: bool = Field(default=False, description="스트리밍 응답 여부")

class QueryResponse(BaseModel):
    answer: str
    cached: bool
    processing_time: float


# ─── 글로벌 리소스 ───
# 세마포어로 동시 요청 수 제한
semaphore = asyncio.Semaphore(MAX_CONCURRENT_REQUESTS)

# 레이트 리미터
rate_limiter = InMemoryRateLimiter(
    requests_per_second=RATE_LIMIT_RPS,
    check_every_n_seconds=0.05,
    max_bucket_size=RATE_LIMIT_RPS * 2,
)


@asynccontextmanager
async def lifespan(app: FastAPI):
    """앱 시작/종료 시 리소스 관리"""
    # 시작: Redis 연결, 모델 초기화
    app.state.redis = aioredis.from_url(
        "redis://localhost:6379/0",
        decode_responses=True,
        max_connections=20,  # Redis 커넥션 풀
    )
    app.state.model = ChatOpenAI(
        model=MODEL_NAME,
        temperature=0.7,
        rate_limiter=rate_limiter,
        streaming=True,
    )
    app.state.chain = (
        ChatPromptTemplate.from_template(
            "다음 질문에 정확하고 도움이 되는 답변을 해주세요.\n\n"
            "질문: {question}"
        )
        | app.state.model
        | StrOutputParser()
    )
    logger.info("서버 초기화 완료")

    yield  # 앱 실행

    # 종료: 리소스 정리
    await app.state.redis.close()
    logger.info("서버 종료, 리소스 정리 완료")


app = FastAPI(title="Production RAG Server", lifespan=lifespan)


# ─── 캐시 유틸리티 ───
def make_cache_key(question: str) -> str:
    """질문으로부터 캐시 키 생성"""
    return f"rag:answer:{hashlib.sha256(question.strip().lower().encode()).hexdigest()}"


# ─── 핵심 엔드포인트 ───
@app.post("/query", response_model=QueryResponse)
async def query(req: QueryRequest):
    """동기식 응답 엔드포인트 (캐시 → LLM 순서)"""
    start = time.time()

    # 1. 캐시 확인
    cache_key = make_cache_key(req.question)
    cached_answer = await app.state.redis.get(cache_key)

    if cached_answer:
        logger.info(f"캐시 히트: {cache_key[:20]}...")
        return QueryResponse(
            answer=cached_answer,
            cached=True,
            processing_time=time.time() - start,
        )

    # 2. 세마포어로 동시 요청 제한
    async with semaphore:
        logger.info(f"LLM 호출 시작: {req.question[:30]}...")

        # 3. 비동기 LLM 호출
        answer = await app.state.chain.ainvoke({"question": req.question})

        # 4. 결과 캐싱
        await app.state.redis.setex(cache_key, CACHE_TTL, answer)

    return QueryResponse(
        answer=answer,
        cached=False,
        processing_time=time.time() - start,
    )


@app.post("/query/stream")
async def query_stream(req: QueryRequest):
    """SSE 스트리밍 응답 엔드포인트"""

    async def generate() -> AsyncGenerator[str, None]:
        chunks = []
        async with semaphore:
            async for chunk in app.state.chain.astream(
                {"question": req.question}
            ):
                chunks.append(chunk)
                yield f"data: {json.dumps({'chunk': chunk}, ensure_ascii=False)}\n\n"

        # 스트리밍 완료 후 전체 응답 캐싱
        full_answer = "".join(chunks)
        cache_key = make_cache_key(req.question)
        await app.state.redis.setex(cache_key, CACHE_TTL, full_answer)

        yield f"data: {json.dumps({'done': True}, ensure_ascii=False)}\n\n"

    return StreamingResponse(generate(), media_type="text/event-stream")


@app.post("/batch")
async def batch_query(questions: list[str]):
    """배치 처리 엔드포인트 — 여러 질문을 동시에 처리"""
    if len(questions) > 20:
        raise HTTPException(status_code=400, detail="최대 20개 질문까지 가능합니다")

    start = time.time()

    # 캐시 히트/미스 분류
    cached_results = {}
    uncached_questions = []

    for q in questions:
        key = make_cache_key(q)
        cached = await app.state.redis.get(key)
        if cached:
            cached_results[q] = cached
        else:
            uncached_questions.append(q)

    # 캐시 미스 질문들만 abatch로 병렬 처리
    if uncached_questions:
        inputs = [{"question": q} for q in uncached_questions]
        async with semaphore:
            results = await app.state.chain.abatch(
                inputs,
                config=RunnableConfig(max_concurrency=5),
            )

        # 결과 캐싱
        for q, result in zip(uncached_questions, results):
            cached_results[q] = result
            key = make_cache_key(q)
            await app.state.redis.setex(key, CACHE_TTL, result)

    return {
        "results": [
            {"question": q, "answer": cached_results[q]}
            for q in questions
        ],
        "cached_count": len(questions) - len(uncached_questions),
        "processing_time": time.time() - start,
    }


@app.get("/health")
async def health():
    """헬스 체크 — 로드 밸런서와 Kubernetes 프로브용"""
    redis_ok = await app.state.redis.ping()
    return {
        "status": "healthy" if redis_ok else "degraded",
        "active_requests": MAX_CONCURRENT_REQUESTS - semaphore._value,
        "max_concurrent": MAX_CONCURRENT_REQUESTS,
    }
```

## 더 깊이 알아보기

### C10K 문제와 비동기 혁명

비동기 프로그래밍이 웹 서버의 표준이 된 데에는 흥미로운 역사가 있습니다. 1999년, Dan Kegel이라는 엔지니어가 ["The C10K Problem"](http://www.kegel.com/c10k.html)이라는 글을 발표했는데요, "어떻게 하면 하나의 서버에서 동시에 10,000개의 클라이언트 연결을 처리할 수 있을까?"라는 질문이었습니다. 당시 대부분의 웹 서버는 연결 하나당 스레드(또는 프로세스)를 할당하는 방식이었기 때문에, 10,000개 연결은 10,000개 스레드를 의미했고 이는 메모리와 컨텍스트 스위칭 비용으로 인해 현실적으로 불가능했습니다.

이 문제를 해결하기 위해 **이벤트 루프** 패턴이 등장했습니다. 하나의 스레드가 I/O 이벤트를 감시하다가, 준비된 작업만 처리하는 방식이죠. 이 아이디어가 2009년 Node.js로 대중화되었고, Python에서는 2015년 Python 3.5의 `async`/`await` 문법으로 본격 도입되었습니다. LangChain이 모든 Runnable에 비동기 인터페이스를 제공하는 것도 이 역사적 흐름의 연장선입니다.

### Celery의 탄생 — Ask Solem의 "분산 당근"

Celery는 2009년 노르웨이 개발자 Ask Solem이 만든 분산 작업 큐입니다. "Celery(셀러리)"라는 이름은 사실 특별한 의미 없이 "발음하기 좋고 기억하기 쉬운 채소 이름"으로 골랐다고 합니다. 초기에는 Django 전용이었지만, 곧 프레임워크 독립적으로 발전했죠. 재미있는 점은 Celery가 처음 해결하려 했던 문제가 "이메일 전송을 비동기로 하고 싶다"는 아주 단순한 요구였다는 것입니다. 그 작은 시작이 지금은 수백만 개의 프로덕션 시스템에서 사용되는 핵심 인프라가 되었습니다.

### HNSW — 작은 세상 네트워크

HNSW(Hierarchical Navigable Small World)는 사회학의 "여섯 다리 건너면 모든 사람을 알 수 있다(Six Degrees of Separation)"에서 영감을 받은 알고리즘입니다. 2018년 Yury Malkov와 Dmitry Yashunin이 발표한 이 알고리즘은, 벡터 공간에서도 "작은 세상 네트워크"를 구성하면 몇 번의 점프만으로 가장 가까운 이웃을 찾을 수 있다는 아이디어에 기반합니다. 현재 pgvector, Pinecone, FAISS 등 거의 모든 벡터 데이터베이스의 핵심 인덱스 알고리즘으로 사용됩니다.

### 엣지 배포와 CDN — 글로벌 응답 지연 최소화

서비스가 글로벌 사용자를 대상으로 한다면, **엣지 배포**를 통해 응답 지연을 더 줄일 수 있습니다. LLM API 자체는 중앙 서버에서 처리되지만, 전후 처리(프롬프트 구성, 캐시 조회, 입력 검증, 결과 포맷팅)는 사용자에게 가까운 엣지 서버에서 수행할 수 있거든요. 특히 캐시 히트인 경우 LLM 호출 없이 즉시 응답할 수 있어 글로벌 사용자의 체감 속도가 크게 향상됩니다.

```
[글로벌 사용자]
      ↓
[CDN / 엣지 캐시] ← 정적 응답, 캐시 히트
      ↓ (캐시 미스)
[엣지 함수] ← 입력 검증, 프롬프트 구성, 결과 포맷팅
      ↓
[중앙 API 서버] ← LLM 호출, RAG 검색
      ↓
[LLM Provider API]
```

**Cloudflare Workers**는 전 세계 330개 이상 위치에서 코드를 실행할 수 있는 서버리스 플랫폼으로, 2025년부터 Python과 FastAPI를 공식 지원합니다. 엣지에서 입력 검증과 KV Store 기반 캐시 조회를 수행하고, 캐시 미스인 경우에만 중앙 서버로 요청을 프록시하는 구조를 구성할 수 있습니다. 이 주제에 관심이 있다면 참고 자료의 Cloudflare Workers 가이드를 참조하세요.

## 흔한 오해와 팁

> ⚠️ **흔한 오해**: "async를 쓰면 무조건 빨라진다"는 잘못된 생각입니다. async는 **I/O 대기 시간**을 활용하는 것이지, CPU 연산 자체를 빠르게 하지 않습니다. 만약 체인 내에 CPU 집약적인 작업(큰 문서 파싱, 대규모 텍스트 전처리 등)이 있다면 async로 감싸도 속도 향상이 없습니다. 이런 경우는 `concurrent.futures.ProcessPoolExecutor`로 별도 프로세스에서 실행하거나, Celery 워커로 분리해야 합니다.

> ⚠️ **흔한 오해**: "비동기 처리를 하면 수평 확장이 필요 없다"고 생각하기 쉽습니다. 하지만 비동기는 **하나의 서버** 내에서 I/O 대기 시간을 효율적으로 쓰는 것이고, 수평 확장은 **여러 서버**로 부하를 분산하는 것입니다. 두 전략은 보완 관계이지 대체 관계가 아닙니다. 단계별로 적용하세요: asyncio 동시성 → Celery 작업 큐 → Kubernetes 오토스케일링.

> 💡 **알고 계셨나요?**: LangChain의 LCEL 체인에서 `ainvoke`를 호출하면, 체인 내의 각 컴포넌트가 네이티브 async를 지원하지 않는 경우 자동으로 `ThreadPoolExecutor`에서 동기 메서드를 실행합니다. 이 때문에 "모든 컴포넌트가 async를 지원하지 않아도 ainvoke가 동작한다"는 편리함은 있지만, 완전한 네이티브 async 대비 스레드 전환 오버헤드가 발생합니다. 최적의 성능을 위해서는 `langchain_openai`처럼 네이티브 async를 지원하는 패키지를 사용하세요.

> 🔥 **실무 팁**: `abatch` 대신 `abatch_as_completed`를 사용하면, 모든 결과가 완료될 때까지 기다리지 않고 완료된 순서대로 결과를 받을 수 있습니다. 사용자에게 "첫 번째 결과를 먼저 보여주고 나머지는 순차적으로 표시"하는 UX를 구현할 때 유용합니다. 레이트 리미터와 결합하면 버스트 트래픽도 부드럽게 처리할 수 있죠.

> 🔥 **실무 팁**: Kubernetes HPA의 `stabilizationWindowSeconds`를 너무 짧게 설정하면 "플래핑(flapping)" — 파드가 반복적으로 생성되고 삭제되는 현상 — 이 발생합니다. 스케일 업은 빠르게(60초), 스케일 다운은 느리게(300초)로 비대칭 설정하는 것이 안정적입니다.

## 핵심 정리

| 개념 | 설명 |
|------|------|
| `ainvoke` / `abatch` | 비동기 호출로 I/O 대기 시간에 다른 요청을 처리하여 동시 처리량 극대화 |
| `astream` / `astream_events` | 토큰 단위 스트리밍으로 사용자 체감 응답 시간 단축 |
| `InMemoryRateLimiter` | 토큰 버킷 알고리즘 기반 레이트 리밋으로 API 과부하 방지 |
| `asyncio.Semaphore` | 단일 서버 내 동시 요청 수 상한을 두어 서버 과부하 방지 |
| Celery + Redis | 작업 큐 기반 수평 확장, 웹 서버와 LLM 처리를 분리하고 재시도 자동화 |
| 커넥션 풀링 | 미리 생성한 DB 연결을 재사용하여 연결 생성 비용 제거 |
| HNSW 인덱스 | pgvector의 근사 최근접 이웃 검색으로 벡터 검색 속도 수십 배 향상 |
| Kubernetes HPA | 트래픽에 따라 파드를 자동으로 늘리고 줄이는 오토스케일링 |
| `max_concurrency` | `RunnableConfig`에서 동시 실행 수 제한, 리소스 보호 |
