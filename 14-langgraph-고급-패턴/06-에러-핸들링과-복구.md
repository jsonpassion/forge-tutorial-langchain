# 에러 핸들링과 복구

> LangGraph 그래프 실행 중 발생하는 에러를 우아하게 처리하고, 자동 재시도와 폴백으로 복원력 있는 AI 워크플로우를 구축합니다.

## 개요

이 섹션에서는 LangGraph 애플리케이션에서 에러가 발생했을 때 시스템이 멈추지 않고 스스로 복구하는 방법을 학습합니다. 노드 레벨의 `RetryPolicy`부터 그래프 수준의 폴백 라우팅, 그리고 체크포인트 기반 복구까지 — 프로덕션 환경에서 반드시 갖춰야 할 에러 핸들링 전략을 체계적으로 다룹니다.

**선수 지식**:
- [Session 14.1: Human-in-the-Loop 패턴](ch14/session_14_1.md)에서 배운 `interrupt()`와 `Command(resume=...)`
- [Session 14.2: 체크포인팅과 상태 영속성](ch14/session_14_2.md)에서 배운 체크포인터, `get_state()`, 타임 트래블
- [Session 14.4: 병렬 실행과 맵-리듀스](ch14/session_14_4.md)에서 배운 `Send` API와 병렬 노드 실행
- [Session 14.5: 스트리밍과 실시간 업데이트](ch14/session_14_5.md)에서 배운 스트림 모드

**학습 목표**:
- `RetryPolicy`를 활용하여 노드 레벨에서 자동 재시도를 구성할 수 있다
- 그래프 상태에 에러를 기록하고 조건부 엣지로 폴백 경로를 설계할 수 있다
- 타임아웃과 재귀 제한으로 무한 루프와 데드락을 방지할 수 있다
- 체크포인트 기반 복구로 실패 지점에서 그래프 실행을 재개할 수 있다

## 왜 알아야 할까?

프로덕션 환경에서 AI 워크플로우가 실패하는 건 "만약"의 문제가 아니라 "언제"의 문제입니다. LLM API가 일시적으로 응답하지 않을 수 있고, 외부 도구가 타임아웃될 수 있으며, 예상치 못한 입력이 파싱 에러를 일으킬 수 있죠.

실제로 LLM 기반 애플리케이션에서 흔히 발생하는 에러들을 보면:
- **API 속도 제한(Rate Limit)**: OpenAI, Anthropic 등의 API가 429 에러를 반환
- **네트워크 일시 장애**: 외부 서비스 연결이 잠깐 끊김
- **모델 출력 파싱 실패**: LLM이 예상과 다른 형식으로 응답
- **도구 실행 에러**: 외부 API, 데이터베이스 쿼리 실패

이런 상황에서 전체 워크플로우가 중단되면 사용자 경험은 최악이 됩니다. 반면, 적절한 에러 핸들링이 갖춰져 있다면 사용자는 장애를 인지조차 못할 수 있거든요. LangGraph는 노드, 그래프, 애플리케이션 세 가지 레벨에서 에러를 처리할 수 있는 구조를 제공합니다.

## 핵심 개념

### 개념 1: RetryPolicy — 노드 레벨 자동 재시도

> 💡 **비유**: 전화 통화를 생각해보세요. 상대방이 통화 중이면 바로 포기하지 않고, 잠시 후 다시 거죠. 두 번째에도 안 되면 조금 더 기다렸다가 한 번 더 시도합니다. 매번 대기 시간이 조금씩 늘어나는 게 바로 **지수 백오프(Exponential Backoff)** 입니다. LangGraph의 `RetryPolicy`가 정확히 이 방식으로 동작합니다.

LangGraph는 `RetryPolicy`를 통해 노드 실행이 실패했을 때 자동으로 재시도하는 메커니즘을 제공합니다. `add_node()` 시 `retry` 파라미터로 정책을 지정하면, 해당 노드에서 예외가 발생할 때 지수 백오프 방식으로 재시도합니다.

```python
from langgraph.graph import StateGraph, START, END
from langgraph.types import RetryPolicy
from typing import TypedDict

# RetryPolicy의 주요 파라미터
policy = RetryPolicy(
    initial_interval=0.5,   # 첫 재시도까지 대기 시간(초)
    backoff_factor=2.0,     # 대기 시간 증가 배수 (0.5 → 1.0 → 2.0 → ...)
    max_interval=10.0,      # 최대 대기 시간(초)
    max_attempts=3,         # 최대 시도 횟수 (첫 시도 포함)
    jitter=True,            # 무작위 지터(jitter) 추가 여부
)

class State(TypedDict):
    query: str
    result: str

def call_api(state: State) -> dict:
    """외부 API를 호출하는 노드 — 일시적 실패가 가능"""
    import httpx
    response = httpx.get(f"https://api.example.com/search?q={state['query']}")
    response.raise_for_status()  # 5xx 에러 시 예외 발생
    return {"result": response.text}

# 노드에 RetryPolicy 적용
builder = StateGraph(State)
builder.add_node("call_api", call_api, retry=policy)
builder.add_edge(START, "call_api")
builder.add_edge("call_api", END)
graph = builder.compile()
```

`retry_on` 파라미터를 사용하면 특정 예외 타입에만 재시도를 적용할 수 있습니다:

```python
import httpx

# 특정 예외에만 재시도 적용
builder.add_node(
    "call_api",
    call_api,
    retry=RetryPolicy(
        max_attempts=3,
        retry_on=httpx.HTTPStatusError,  # HTTP 에러에만 재시도
    ),
)

# 여러 예외 타입을 함수로 지정
def should_retry(exc: Exception) -> bool:
    """재시도할 예외인지 판단하는 커스텀 함수"""
    if isinstance(exc, httpx.HTTPStatusError):
        return exc.response.status_code >= 500  # 5xx만 재시도
    if isinstance(exc, TimeoutError):
        return True  # 타임아웃은 항상 재시도
    return False

builder.add_node(
    "call_api",
    call_api,
    retry=RetryPolicy(max_attempts=5, retry_on=should_retry),
)
```

> ⚠️ **흔한 오해**: `max_attempts=3`은 "3번 재시도"가 아니라 "첫 시도 포함 총 3번 시도"입니다. 즉, 재시도는 최대 2번만 일어납니다. 실제 재시도 횟수를 기준으로 생각하면 `max_attempts = 원하는_재시도_횟수 + 1`로 설정해야 합니다.

### 개념 2: 상태 기반 에러 추적

> 💡 **비유**: 병원 응급실에서 환자의 상태를 차트에 계속 기록하는 것과 같습니다. "어디가 아프고, 어떤 치료를 시도했고, 결과가 어땠는지" 모두 기록해야 다음 의사가 이어서 치료할 수 있죠. 그래프 상태에 에러 정보를 기록하면, 후속 노드가 "무엇이 실패했는지" 알고 적절한 대응을 할 수 있습니다.

`RetryPolicy`는 일시적 장애에 효과적이지만, 모든 에러를 재시도로 해결할 수는 없습니다. 비즈니스 로직 에러나 데이터 검증 실패 같은 경우에는 그래프 상태에 에러를 기록하고, 조건부 엣지로 적절한 처리 경로로 분기해야 합니다.

```python
from typing import TypedDict, Annotated, Optional
from dataclasses import dataclass, field
import operator

@dataclass
class ErrorInfo:
    """에러 정보를 구조화하여 저장"""
    node: str           # 에러가 발생한 노드 이름
    error_type: str     # 예외 클래스 이름
    message: str        # 에러 메시지
    retryable: bool     # 재시도 가능 여부

class State(TypedDict):
    query: str
    result: Optional[str]
    errors: Annotated[list[ErrorInfo], operator.add]  # 에러 누적 리듀서

def search_node(state: State) -> dict:
    """검색 노드 — 에러 발생 시 상태에 기록"""
    try:
        # 외부 API 호출 시뮬레이션
        result = perform_search(state["query"])
        return {"result": result}
    except TimeoutError as e:
        return {
            "result": None,
            "errors": [ErrorInfo(
                node="search",
                error_type="TimeoutError",
                message=str(e),
                retryable=True,
            )],
        }
    except ValueError as e:
        return {
            "result": None,
            "errors": [ErrorInfo(
                node="search",
                error_type="ValueError",
                message=str(e),
                retryable=False,
            )],
        }
```

핵심은 노드가 예외를 던지지 않고, **에러 정보를 상태에 기록한 뒤 정상적으로 반환**한다는 점입니다. 이렇게 하면 그래프 실행이 중단되지 않고, 후속 노드나 조건부 엣지에서 에러에 맞는 처리를 할 수 있습니다.

### 개념 3: 그래프 수준 폴백 라우팅

> 💡 **비유**: 내비게이션을 떠올려보세요. 고속도로가 사고로 막히면 "우회 경로"를 안내해줍니다. 그래프 수준 폴백도 마찬가지입니다. 메인 경로(노드)가 실패하면 자동으로 대체 경로(폴백 노드)로 전환됩니다.

상태에 기록된 에러 정보를 기반으로 조건부 엣지(conditional edge)를 설계하면, 에러 유형에 따라 다른 처리 경로로 분기할 수 있습니다:

```python
from langgraph.graph import StateGraph, START, END

def route_after_search(state: State) -> str:
    """검색 결과에 따라 다음 경로 결정"""
    # 에러가 없으면 정상 진행
    if state.get("result") is not None:
        return "process_result"
    
    # 최근 에러 확인
    errors = state.get("errors", [])
    if not errors:
        return "process_result"
    
    latest_error = errors[-1]
    
    # 에러 유형별 분기
    if latest_error.retryable:
        return "fallback_search"   # 대체 검색 엔진 시도
    else:
        return "error_response"    # 사용자에게 에러 안내

def fallback_search(state: State) -> dict:
    """폴백 검색 — 대체 검색 엔진 사용"""
    try:
        result = perform_backup_search(state["query"])
        return {"result": result}
    except Exception as e:
        return {
            "result": None,
            "errors": [ErrorInfo(
                node="fallback_search",
                error_type=type(e).__name__,
                message=f"폴백도 실패: {e}",
                retryable=False,
            )],
        }

def error_response(state: State) -> dict:
    """에러 응답 생성 — 사용자에게 적절한 안내 제공"""
    errors = state.get("errors", [])
    error_summary = "; ".join(e.message for e in errors)
    return {
        "result": f"죄송합니다. 요청을 처리하는 중 문제가 발생했습니다: {error_summary}"
    }

# 그래프 구성 — 폴백 라우팅
builder = StateGraph(State)
builder.add_node("search", search_node)
builder.add_node("fallback_search", fallback_search)
builder.add_node("error_response", error_response)
builder.add_node("process_result", process_result)

builder.add_edge(START, "search")
builder.add_conditional_edges("search", route_after_search)
builder.add_edge("fallback_search", "process_result")
builder.add_edge("error_response", END)
builder.add_edge("process_result", END)

graph = builder.compile()
```

이 패턴의 핵심 구조를 정리하면:

| 레벨 | 전략 | 용도 |
|------|------|------|
| **노드** | `try/except` + 상태 기록 | 에러를 잡고 구조화된 정보 저장 |
| **엣지** | 조건부 라우팅 | 에러 유형별 분기 |
| **그래프** | 폴백 노드 | 대체 경로로 처리 |

### 개념 4: 타임아웃과 재귀 제한

> 💡 **비유**: 미로에 들어갈 때 "30분 안에 못 나오면 입구로 돌아간다"는 규칙을 정해두는 것과 같습니다. 타임아웃과 재귀 제한은 그래프가 무한히 빙빙 돌거나 끝없이 기다리는 것을 막아주는 안전장치입니다.

LangGraph에서 그래프 실행이 끝나지 않는 두 가지 주요 원인이 있습니다:

1. **무한 루프**: 조건부 엣지가 특정 노드를 계속 반복 방문
2. **응답 없는 외부 호출**: LLM API나 외부 서비스가 응답하지 않음

**재귀 제한(Recursion Limit)**으로 무한 루프를 방지합니다:

```python
from langgraph.errors import GraphRecursionError

# 실행 시 recursion_limit 설정
config = {"recursion_limit": 25}  # 최대 25 스텝(기본값: 25)

try:
    result = graph.invoke({"query": "LangGraph 에러 핸들링"}, config=config)
except GraphRecursionError:
    print("그래프가 최대 실행 단계를 초과했습니다. 로직을 점검하세요.")
```

**노드 내부 타임아웃**으로 장기 대기를 방지합니다:

```python
import asyncio
from typing import TypedDict

class State(TypedDict):
    query: str
    result: str

async def call_llm_with_timeout(state: State) -> dict:
    """LLM 호출에 타임아웃 적용"""
    try:
        # 30초 타임아웃 설정
        result = await asyncio.wait_for(
            slow_llm_call(state["query"]),
            timeout=30.0,
        )
        return {"result": result}
    except asyncio.TimeoutError:
        return {
            "result": "응답 시간을 초과했습니다. 잠시 후 다시 시도해주세요.",
            "errors": [ErrorInfo(
                node="call_llm",
                error_type="TimeoutError",
                message="LLM 호출이 30초를 초과했습니다",
                retryable=True,
            )],
        }
```

> 🔥 **실무 팁**: `recursion_limit`은 "스텝 수"를 기준으로 합니다. 복잡한 에이전트에서 노드 4개를 매 루프마다 거친다면, `recursion_limit=25`는 약 6회 루프 후 중단됩니다. 에이전트의 루프 구조를 고려하여 적절한 값을 설정하세요.

### 개념 5: 체크포인트 기반 복구

앞서 [Session 14.2: 체크포인팅과 상태 영속성](ch14/session_14_2.md)에서 배운 체크포인터가 에러 복구에서 빛을 발합니다. 그래프 실행 중 에러가 발생하더라도 마지막 성공 체크포인트가 저장되어 있으므로, 실패 원인을 해결한 뒤 해당 지점부터 실행을 재개할 수 있습니다.

```python
from langgraph.checkpoint.memory import MemorySaver
from langgraph.graph import StateGraph, START, END
from langgraph.types import RetryPolicy

checkpointer = MemorySaver()

class State(TypedDict):
    query: str
    search_result: str
    analysis: str

def search(state: State) -> dict:
    return {"search_result": f"'{state['query']}'에 대한 검색 결과"}

def analyze(state: State) -> dict:
    # 이 노드에서 에러가 발생한다고 가정
    raise ConnectionError("분석 서비스에 연결할 수 없습니다")

builder = StateGraph(State)
builder.add_node("search", search)
builder.add_node("analyze", analyze, retry=RetryPolicy(max_attempts=2))
builder.add_edge(START, "search")
builder.add_edge("search", "analyze")
builder.add_edge("analyze", END)

graph = builder.compile(checkpointer=checkpointer)

config = {"configurable": {"thread_id": "recovery-demo"}}

# 첫 실행 — analyze 노드에서 실패
try:
    result = graph.invoke({"query": "LangGraph"}, config)
except ConnectionError:
    print("에러 발생! 체크포인트에서 상태를 확인합니다.")

# 마지막 성공 상태 확인
snapshot = graph.get_state(config)
print(f"다음 실행 예정 노드: {snapshot.next}")
# 출력: ('analyze',) — search는 이미 완료, analyze부터 재개 가능

# 문제 해결 후 update_state로 수정하거나 그대로 재실행
# (실제로는 analyze 노드의 에러 원인을 수정한 뒤)
# result = graph.invoke(None, config)  # None을 전달하면 체크포인트에서 재개
```

이 패턴은 특히 오래 걸리는 파이프라인에서 위력적입니다. 10개 노드 중 8번째에서 실패했다면, 처음부터 다시 시작하지 않고 8번째부터 재개할 수 있거든요.

## 실습: 직접 해보기

모든 에러 핸들링 패턴을 결합한 완전한 예제를 만들어봅시다. 사용자 질문을 받아 검색하고 분석하는 워크플로우에 다단계 에러 처리를 적용합니다.

```python
"""LangGraph 에러 핸들링 종합 실습
  
노드 레벨 RetryPolicy, 상태 기반 에러 추적,
그래프 수준 폴백 라우팅을 결합한 복원력 있는 워크플로우
"""
from __future__ import annotations

import operator
import random
from dataclasses import dataclass
from typing import Annotated, Optional, TypedDict

from langgraph.checkpoint.memory import MemorySaver
from langgraph.graph import StateGraph, START, END
from langgraph.types import RetryPolicy


# ── 에러 정보 모델 ──────────────────────────────
@dataclass
class ErrorInfo:
    node: str
    error_type: str
    message: str
    retryable: bool


# ── 상태 정의 ──────────────────────────────────
class State(TypedDict):
    query: str
    search_result: Optional[str]
    analysis: Optional[str]
    final_answer: Optional[str]
    errors: Annotated[list[ErrorInfo], operator.add]  # 에러 누적


# ── 노드 함수들 ────────────────────────────────
def search(state: State) -> dict:
    """메인 검색 노드 — 30% 확률로 일시적 실패 시뮬레이션"""
    print(f"  [search] '{state['query']}' 검색 중...")
    
    if random.random() < 0.3:
        # RetryPolicy가 자동으로 재시도
        raise ConnectionError("검색 서비스 일시 장애")
    
    return {"search_result": f"'{state['query']}'에 대한 검색 결과 3건"}


def fallback_search(state: State) -> dict:
    """폴백 검색 — 메인 검색 실패 시 대체 경로"""
    print(f"  [fallback_search] 대체 검색 엔진으로 '{state['query']}' 검색 중...")
    # 폴백은 안정적이라고 가정
    return {"search_result": f"'{state['query']}'에 대한 폴백 검색 결과 1건"}


def analyze(state: State) -> dict:
    """분석 노드 — try/except로 에러를 상태에 기록"""
    print(f"  [analyze] 검색 결과 분석 중...")
    
    try:
        if state.get("search_result") is None:
            raise ValueError("분석할 검색 결과가 없습니다")
        
        # 20% 확률로 파싱 실패 시뮬레이션
        if random.random() < 0.2:
            raise ValueError("LLM 응답 파싱 실패: 예상과 다른 형식")
        
        analysis = f"분석 완료: {state['search_result']}의 핵심 내용 추출"
        return {"analysis": analysis}
    
    except ValueError as e:
        print(f"  [analyze] 에러 발생: {e}")
        return {
            "analysis": None,
            "errors": [ErrorInfo(
                node="analyze",
                error_type="ValueError",
                message=str(e),
                retryable=False,
            )],
        }


def generate_answer(state: State) -> dict:
    """최종 답변 생성"""
    print(f"  [generate_answer] 답변 생성 중...")
    
    if state.get("analysis"):
        return {"final_answer": f"답변: {state['analysis']}을 바탕으로 정리한 내용입니다."}
    else:
        return {"final_answer": f"검색 결과({state.get('search_result', '없음')})를 기반으로 간략히 답변드립니다."}


def error_response(state: State) -> dict:
    """에러 응답 — 모든 시도가 실패했을 때"""
    errors = state.get("errors", [])
    summary = ", ".join(f"{e.node}: {e.message}" for e in errors[-3:])  # 최근 3개
    print(f"  [error_response] 에러 응답 생성: {summary}")
    return {"final_answer": f"죄송합니다. 요청 처리 중 문제가 발생했습니다. ({summary})"}


# ── 라우팅 함수들 ───────────────────────────────
def route_after_search(state: State) -> str:
    """검색 결과 유무에 따라 분기"""
    if state.get("search_result"):
        return "analyze"
    # search 노드가 RetryPolicy 소진 후에도 실패하면
    # 예외가 상위로 전파되므로 이 경우는 폴백 분기
    return "fallback_search"


def route_after_analyze(state: State) -> str:
    """분석 결과에 따라 분기"""
    errors = state.get("errors", [])
    recent_errors = [e for e in errors if e.node == "analyze"]
    
    if state.get("analysis"):
        return "generate_answer"  # 분석 성공 → 답변 생성
    elif len(recent_errors) < 2:
        return "generate_answer"  # 분석 실패해도 검색 결과로 답변 시도
    else:
        return "error_response"   # 반복 실패 → 에러 응답


# ── 그래프 구성 ─────────────────────────────────
builder = StateGraph(State)

# 노드 등록 — search에만 RetryPolicy 적용
builder.add_node(
    "search", search,
    retry=RetryPolicy(
        max_attempts=3,
        initial_interval=0.5,
        backoff_factor=2.0,
        jitter=True,
    ),
)
builder.add_node("fallback_search", fallback_search)
builder.add_node("analyze", analyze)
builder.add_node("generate_answer", generate_answer)
builder.add_node("error_response", error_response)

# 엣지 구성 — 조건부 폴백 라우팅
builder.add_edge(START, "search")
builder.add_conditional_edges("search", route_after_search)
builder.add_edge("fallback_search", "analyze")
builder.add_conditional_edges("analyze", route_after_analyze)
builder.add_edge("generate_answer", END)
builder.add_edge("error_response", END)

# 체크포인터 포함 컴파일
checkpointer = MemorySaver()
graph = builder.compile(checkpointer=checkpointer)


# ── 실행 ────────────────────────────────────────
if __name__ == "__main__":
    config = {
        "configurable": {"thread_id": "error-handling-demo"},
        "recursion_limit": 15,  # 무한 루프 방지
    }
    
    print("=== 에러 핸들링 종합 실습 ===\n")
    
    try:
        result = graph.invoke(
            {"query": "LangGraph 에러 핸들링 패턴", "errors": []},
            config,
        )
        print(f"\n최종 답변: {result['final_answer']}")
        
        # 에러 이력 확인
        if result.get("errors"):
            print(f"\n발생한 에러 이력:")
            for err in result["errors"]:
                print(f"  - [{err.node}] {err.error_type}: {err.message}")
    
    except Exception as e:
        print(f"\n치명적 에러: {e}")
        
        # 체크포인트에서 상태 확인
        snapshot = graph.get_state(config)
        print(f"마지막 완료 노드 이후 대기 중인 노드: {snapshot.next}")
        print(f"저장된 상태: {snapshot.values}")
```

실행하면 랜덤 요소에 따라 다양한 경로로 진행됩니다:

### 서킷 브레이커 — 전기 공학에서 소프트웨어로

에러 핸들링에서 빼놓을 수 없는 패턴이 **서킷 브레이커(Circuit Breaker)** 입니다. 이 이름은 실제 전기 회로의 차단기에서 유래했는데요, 과전류가 흐르면 자동으로 회로를 끊어 화재를 방지하듯, 소프트웨어에서도 특정 서비스가 반복적으로 실패하면 아예 호출을 차단하는 패턴입니다.

이 패턴을 소프트웨어 세계에 본격적으로 소개한 사람은 **Michael Nygard**입니다. 그는 2007년 출간한 저서 *Release It!* 에서 프로덕션 환경의 분산 시스템이 얼마나 취약한지 생생하게 묘사하며, 서킷 브레이커를 핵심 안정성 패턴으로 제안했죠.

그 후 2011~2012년, **Netflix**의 API 엔지니어링 팀이 **Hystrix** 라이브러리를 개발하면서 이 패턴이 업계 전반에 폭발적으로 퍼졌습니다. Netflix는 수백 개의 마이크로서비스가 서로 호출하는 구조였기에, 하나의 서비스 장애가 연쇄적으로 전체 시스템을 마비시키는 "연쇄 실패(Cascading Failure)"가 큰 문제였거든요. Hystrix 도입 후 Netflix는 하루에 수백억 건의 서비스 호출을 안정적으로 처리하게 되었습니다.

LangGraph에는 서킷 브레이커가 내장되어 있지 않지만, 상태 기반 에러 추적과 조건부 라우팅을 결합하면 유사한 패턴을 구현할 수 있습니다:

```python
from dataclasses import dataclass, field
from time import time

@dataclass
class CircuitBreaker:
    """간단한 서킷 브레이커 구현"""
    failure_threshold: int = 3     # 이 횟수만큼 연속 실패하면 회로 차단
    recovery_timeout: float = 60.0 # 차단 후 이 시간(초) 경과 시 재시도 허용
    _failure_count: int = field(default=0, init=False)
    _last_failure_time: float = field(default=0.0, init=False)
    _state: str = field(default="closed", init=False)  # closed, open, half_open

    def can_execute(self) -> bool:
        if self._state == "closed":
            return True
        if self._state == "open":
            # recovery_timeout 경과 시 half_open으로 전환
            if time() - self._last_failure_time > self.recovery_timeout:
                self._state = "half_open"
                return True
            return False
        return True  # half_open 상태에서는 한 번 시도 허용

    def record_success(self) -> None:
        self._failure_count = 0
        self._state = "closed"

    def record_failure(self) -> None:
        self._failure_count += 1
        self._last_failure_time = time()
        if self._failure_count >= self.failure_threshold:
            self._state = "open"
```

### 지수 백오프의 역사

`RetryPolicy`의 핵심인 **지수 백오프(Exponential Backoff)** 알고리즘은 사실 1970년대 이더넷(Ethernet) 프로토콜에서 유래했습니다. 이더넷의 공동 발명자인 **Bob Metcalfe**와 **David Boggs**가 네트워크에서 여러 장치가 동시에 데이터를 보내 충돌이 발생했을 때, 각 장치가 무작위로 점점 더 긴 시간을 기다린 후 재전송하는 방식을 고안한 것이죠. 이 아이디어가 50년이 지난 지금도 클라우드 API 재시도 전략의 기본이 된 셈입니다.

## 흔한 오해와 팁

> ⚠️ **흔한 오해**: "RetryPolicy를 설정하면 모든 예외에 재시도한다." 사실 `RetryPolicy`의 기본 `retry_on` 함수는 `ValueError`, `TypeError`, `ArithmeticError` 같은 **프로그래밍 에러**에는 재시도하지 않습니다. 코드 버그를 재시도로 해결할 수는 없으니까요. 기본적으로 네트워크 관련 에러나 일시적 서비스 장애에만 재시도가 적용됩니다. 특정 예외만 재시도하고 싶다면 `retry_on` 파라미터를 명시적으로 지정하세요.

> 💡 **알고 계셨나요?**: LangGraph에서 `RetryPolicy`가 모든 시도를 소진하고도 실패하면, 마지막 예외가 그래프 실행자에게 전파됩니다. 이때 체크포인터가 설정되어 있다면, **실패 직전의 상태가 체크포인트에 저장**됩니다. 이를 통해 `get_state()`로 상태를 확인하고, 문제를 수정한 뒤 `invoke(None, config)`로 해당 지점부터 재개할 수 있습니다.

> 🔥 **실무 팁**: 에러 핸들링 전략을 설계할 때 "3단계 방어선"을 기억하세요. **1단계**: `RetryPolicy`로 일시적 실패 자동 복구. **2단계**: `try/except` + 상태 기록 + 조건부 라우팅으로 폴백 처리. **3단계**: 체크포인터 기반 수동 복구. 대부분의 에러는 1단계에서 해결되고, 1~2% 정도만 2단계로 넘어가며, 3단계까지 가는 경우는 극히 드물어야 정상입니다. 만약 2~3단계 도달 빈도가 높다면, 근본 원인을 해결해야 합니다.

> 🔥 **실무 팁**: `jitter=True`는 단순히 "랜덤 대기"가 아닙니다. 수백 개의 요청이 동시에 실패한 후 동시에 재시도하면 서버에 "**재시도 폭풍(Retry Storm)**"이 발생합니다. 지터는 각 재시도 시점을 분산시켜 이 문제를 방지합니다. 프로덕션에서는 항상 `jitter=True`를 권장합니다.

## 핵심 정리

| 개념 | 설명 |
|------|------|
| `RetryPolicy` | 노드에 자동 재시도를 적용하는 정책. `max_attempts`, `backoff_factor`, `retry_on` 등을 설정 |
| `retry_on` | 재시도할 예외 타입을 지정. 기본값은 프로그래밍 에러를 제외한 대부분의 예외 |
| 지수 백오프 | 재시도 간격을 `initial_interval × backoff_factor^n`으로 점진적 증가 |
| 지터(Jitter) | 재시도 시점에 랜덤 편차를 추가하여 재시도 폭풍 방지 |
| 상태 기반 에러 추적 | `ErrorInfo`를 상태에 기록하여 후속 노드가 에러 맥락을 인지 |
| 폴백 라우팅 | 조건부 엣지로 에러 유형에 따라 대체 경로로 분기 |
| `recursion_limit` | 그래프 최대 실행 스텝 수 제한으로 무한 루프 방지 |
| 체크포인트 복구 | 실패 직전 상태에서 재개하여 처음부터 재실행 방지 |
| 서킷 브레이커 | 반복 실패 시 호출 자체를 차단하여 연쇄 실패 방지 |
