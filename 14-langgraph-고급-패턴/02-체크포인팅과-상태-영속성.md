# 체크포인팅과 상태 영속성

> LangGraph의 체크포인터를 활용하여 그래프 상태를 영구 저장하고, 스레드 기반 다중 대화를 관리하는 방법을 학습합니다.

## 개요

이 섹션에서는 [세션 14.1: Human-in-the-Loop 패턴](./01-human-in-the-loop-패턴.md)에서 간단히 사용했던 `InMemorySaver`를 넘어, 프로덕션 환경에서 사용할 수 있는 다양한 체크포인터(Checkpointer)를 깊이 있게 다룹니다. `SqliteSaver`, `PostgresSaver`를 활용한 영속적 상태 관리, `thread_id` 기반 다중 대화 관리, 그리고 체크포인트 조회와 타임 트래블(Time Travel)까지 실습합니다.

**선수 지식**: 세션 14.1에서 배운 `interrupt()`, `Command(resume=...)`, `InMemorySaver()` 사용법
**학습 목표**:
- MemorySaver, SqliteSaver, PostgresSaver의 차이와 적합한 사용 시나리오를 구분할 수 있다
- `thread_id`를 활용하여 독립적인 다중 대화를 관리할 수 있다
- `get_state()`, `get_state_history()`로 체크포인트를 조회하고 타임 트래블할 수 있다
- 프로덕션 환경에 적합한 체크포인터를 선택하고 설정할 수 있다

## 왜 알아야 할까?

여러분이 카카오톡 같은 메신저 앱을 만든다고 상상해보세요. 사용자가 앱을 껐다가 다시 켜면 이전 대화가 모두 사라진다면 어떨까요? 아무도 그런 앱을 쓰지 않겠죠.

AI 에이전트도 마찬가지입니다. 세션 14.1에서 `InMemorySaver`를 사용했는데, 이 체크포인터는 파이썬 프로세스가 종료되면 **모든 상태가 사라집니다**. 프로덕션 환경에서는 서버가 재시작되더라도 사용자의 대화 맥락이 유지되어야 하고, 수천 명의 사용자가 각각 독립적인 대화를 진행할 수 있어야 합니다.

체크포인팅은 단순한 "저장/불러오기"가 아닙니다. LangGraph의 체크포인팅 시스템은 그래프 실행의 **매 스텝마다** 자동으로 스냅샷을 찍어서, Human-in-the-Loop, 장애 복구(Fault Tolerance), 타임 트래블 디버깅까지 가능하게 합니다. 이것이 LangGraph를 단순한 워크플로우 엔진이 아닌 **프로덕션급 에이전트 프레임워크**로 만들어주는 핵심 기능이죠.

## 핵심 개념

### 개념 1: 체크포인터의 작동 원리 — "자동 저장 게임"

> 💡 **비유**: RPG 게임의 자동 저장 시스템을 떠올려보세요. 던전의 각 방을 클리어할 때마다 게임이 자동으로 저장되어서, 게임 오버가 되어도 마지막 세이브 포인트에서 다시 시작할 수 있죠. LangGraph의 체크포인터는 그래프의 **각 슈퍼스텝(super-step)**이 끝날 때마다 자동으로 상태를 저장하는 "자동 세이브 시스템"입니다.

LangGraph에서 체크포인터를 사용하면, 그래프가 실행될 때 매 슈퍼스텝(노드 실행 단위)마다 상태가 자동으로 기록됩니다. 이 기록은 **스레드(Thread)**라는 단위로 관리되는데요, 하나의 스레드는 하나의 대화 세션이라고 생각하면 됩니다.

모든 체크포인터는 `BaseCheckpointSaver` 인터페이스를 구현하며, 핵심 메서드는 다음과 같습니다:

| 메서드 | 역할 |
|--------|------|
| `.put()` | 체크포인트 저장 |
| `.put_writes()` | 중간 쓰기(pending writes) 저장 |
| `.get_tuple()` | 특정 체크포인트 조회 |
| `.list()` | 체크포인트 목록 조회 |

```python
from langgraph.checkpoint.memory import MemorySaver

# 가장 간단한 체크포인터 — 메모리에만 저장
checkpointer = MemorySaver()

# 그래프 컴파일 시 체크포인터 연결
graph = builder.compile(checkpointer=checkpointer)

# 실행 시 thread_id 지정 — 이게 "세이브 슬롯" 역할
config = {"configurable": {"thread_id": "user-123"}}
result = graph.invoke({"messages": [("user", "안녕하세요")]}, config)
```

### 개념 2: 세 가지 체크포인터 비교 — "노트 → 수첩 → 서류함"

> 💡 **비유**: 메모를 남기는 세 가지 방법을 생각해보세요. **포스트잇(MemorySaver)**은 빠르고 편하지만 바람에 날아가면 끝입니다. **노트북(SqliteSaver)**은 파일로 보관되어 안전하지만 혼자만 볼 수 있죠. **회사 서류함(PostgresSaver)**은 여러 사람이 동시에 접근할 수 있고 백업도 되지만, 설치하고 관리해야 합니다.

#### MemorySaver (InMemorySaver)

프로세스 메모리에 저장됩니다. 개발/테스트 단계에서 빠르게 프로토타이핑할 때 사용하세요.

```python
from langgraph.checkpoint.memory import MemorySaver

# 별도 설치 불필요 — langgraph에 기본 포함
memory = MemorySaver()
graph = builder.compile(checkpointer=memory)
```

**장점**: 설정 없이 바로 사용, 빠른 속도
**단점**: 프로세스 종료 시 데이터 소멸, 다중 서버 불가

#### SqliteSaver

SQLite 파일에 저장됩니다. 로컬 개발이나 단일 서버 환경에 적합합니다.

```python
# pip install langgraph-checkpoint-sqlite
from langgraph.checkpoint.sqlite import SqliteSaver

# 파일 기반 — 프로세스 재시작해도 유지
with SqliteSaver.from_conn_string("checkpoints.db") as memory:
    graph = builder.compile(checkpointer=memory)
    # 그래프 사용...
```

비동기 환경에서는 `AsyncSqliteSaver`를 사용합니다:

```python
# pip install langgraph-checkpoint-sqlite aiosqlite
from langgraph.checkpoint.sqlite.aio import AsyncSqliteSaver

async with AsyncSqliteSaver.from_conn_string("checkpoints.db") as memory:
    graph = builder.compile(checkpointer=memory)
    result = await graph.ainvoke(input_data, config)
```

**장점**: 파일 기반 영속성, 설정 간편
**단점**: 동시 쓰기 성능 제한, 단일 서버만 지원

#### PostgresSaver

PostgreSQL 데이터베이스에 저장됩니다. 프로덕션 환경의 표준 선택입니다.

```python
# pip install langgraph-checkpoint-postgres psycopg[binary] psycopg-pool
from langgraph.checkpoint.postgres import PostgresSaver

DB_URI = "postgresql://user:password@localhost:5432/langgraph_db"

with PostgresSaver.from_conn_string(DB_URI) as memory:
    # 최초 1회 — 필요한 테이블 생성
    memory.setup()
    graph = builder.compile(checkpointer=memory)
```

비동기 + 커넥션 풀 설정 (프로덕션 권장):

```python
from langgraph.checkpoint.postgres.aio import AsyncPostgresSaver
from psycopg_pool import AsyncConnectionPool

async def create_graph():
    pool = AsyncConnectionPool(
        conninfo="postgresql://user:password@localhost:5432/langgraph_db",
        max_size=20,  # 동시 연결 수 제한
    )
    checkpointer = AsyncPostgresSaver(pool)
    await checkpointer.setup()  # 테이블 생성
    graph = builder.compile(checkpointer=checkpointer)
    return graph, pool
```

**장점**: 다중 서버 지원, 높은 동시성, 백업/복제 가능
**단점**: PostgreSQL 서버 필요, 설정 복잡

### 개념 3: thread_id 기반 다중 대화 관리 — "호텔 객실 번호"

> 💡 **비유**: 호텔을 생각해보세요. 각 객실(thread)에는 고유한 번호(thread_id)가 있고, 투숙객(사용자)마다 독립적인 공간이 제공됩니다. 101호 손님이 방을 어지럽혀도 102호에는 영향이 없죠. 마찬가지로 `thread_id`가 다르면 완전히 독립된 대화가 됩니다.

`thread_id`는 체크포인터가 상태를 저장하고 불러올 때 사용하는 **기본 키(Primary Key)**입니다. 같은 `thread_id`로 그래프를 호출하면 이전 대화가 이어지고, 다른 `thread_id`를 사용하면 완전히 새로운 대화가 시작됩니다.

```python
from langchain_openai import ChatOpenAI
from langgraph.graph import StateGraph, MessagesState, START, END
from langgraph.checkpoint.memory import MemorySaver

# 간단한 챗봇 그래프 구성
llm = ChatOpenAI(model="gpt-4o")

def chatbot(state: MessagesState):
    return {"messages": [llm.invoke(state["messages"])]}

builder = StateGraph(MessagesState)
builder.add_node("chatbot", chatbot)
builder.add_edge(START, "chatbot")
builder.add_edge("chatbot", END)

memory = MemorySaver()
graph = builder.compile(checkpointer=memory)

# === 사용자 A의 대화 (thread_id: "user-a") ===
config_a = {"configurable": {"thread_id": "user-a"}}
graph.invoke({"messages": [("user", "제 이름은 민수입니다")]}, config_a)
# 이후 같은 thread_id로 호출하면 이전 대화가 이어짐
result_a = graph.invoke({"messages": [("user", "제 이름이 뭐였죠?")]}, config_a)
print(result_a["messages"][-1].content)
# 출력: "민수님이시죠!" (이전 대화 맥락 기억)

# === 사용자 B의 대화 (thread_id: "user-b") ===
config_b = {"configurable": {"thread_id": "user-b"}}
result_b = graph.invoke({"messages": [("user", "제 이름이 뭐였죠?")]}, config_b)
print(result_b["messages"][-1].content)
# 출력: "죄송합니다, 이름을 알려주신 적이 없는데요." (독립된 대화)
```

### 개념 4: 체크포인트 조회와 타임 트래블 — "대화의 되감기 버튼"

> 💡 **비유**: 구글 독스의 "버전 기록" 기능을 떠올려보세요. 문서의 모든 수정 이력이 저장되어 있어서, 과거 어느 시점으로든 돌아갈 수 있죠. LangGraph의 타임 트래블도 마찬가지입니다. 그래프 실행의 모든 스텝이 기록되어 있어서, 특정 시점의 상태를 조회하거나 그 시점부터 다시 실행할 수 있습니다.

LangGraph는 그래프 인스턴스에서 직접 상태를 조회하는 메서드를 제공합니다:

```python
# 현재 상태 조회 — 가장 최근 체크포인트
state = graph.get_state(config_a)
print(state.values)       # 현재 상태 값
print(state.next)          # 다음에 실행될 노드
print(state.config)        # 체크포인트 설정 정보

# 상태 이력 조회 — 모든 체크포인트를 역순으로 반환
for state in graph.get_state_history(config_a):
    print(f"체크포인트: {state.config['configurable']['checkpoint_id']}")
    print(f"다음 노드: {state.next}")
    print(f"메시지 수: {len(state.values.get('messages', []))}")
    print("---")
```

특정 체크포인트로 돌아가서 다시 실행하는 **타임 트래블**도 가능합니다:

```python
# 1. 상태 이력에서 원하는 체크포인트 찾기
all_states = list(graph.get_state_history(config_a))
target_state = all_states[2]  # 3번째 이전 체크포인트

# 2. 해당 체크포인트의 config 사용하여 재실행
# — 새로운 포크(fork)가 생성되어 원본 이력은 보존됨
result = graph.invoke(
    {"messages": [("user", "다른 질문을 해볼게요")]},
    target_state.config  # 과거 체크포인트의 config 사용
)
```

`update_state()`를 사용하면 특정 체크포인트의 상태를 직접 수정할 수도 있습니다:

```python
# 상태를 직접 수정 — 디버깅이나 관리자 개입 시 유용
graph.update_state(
    config_a,
    {"messages": [("system", "관리자 메모: 이 사용자는 VIP입니다")]},
)
```

## 실습: 직접 해보기

아래는 `SqliteSaver`를 활용하여 프로세스 재시작 후에도 대화가 유지되는 챗봇을 만드는 완전한 예제입니다. 타임 트래블 기능까지 포함되어 있습니다.

```python
"""
체크포인팅 실습: SqliteSaver를 활용한 영속적 대화 관리
pip install langgraph langgraph-checkpoint-sqlite langchain-openai python-dotenv
"""
import os
from dotenv import load_dotenv
from typing import Annotated
from typing_extensions import TypedDict

from langchain_openai import ChatOpenAI
from langchain_core.messages import HumanMessage, AIMessage
from langgraph.graph import StateGraph, MessagesState, START, END
from langgraph.checkpoint.sqlite import SqliteSaver

load_dotenv()

# === 1단계: 그래프 정의 ===
MODEL_NAME = "gpt-4o"
DB_PATH = "chat_checkpoints.db"  # SQLite 파일 경로

llm = ChatOpenAI(model=MODEL_NAME, temperature=0.7)

def chatbot_node(state: MessagesState) -> dict:
    """사용자 메시지에 응답하는 챗봇 노드"""
    response = llm.invoke(state["messages"])
    return {"messages": [response]}

# 그래프 빌더 구성
builder = StateGraph(MessagesState)
builder.add_node("chatbot", chatbot_node)
builder.add_edge(START, "chatbot")
builder.add_edge("chatbot", END)


def main():
    # === 2단계: SqliteSaver로 컴파일 ===
    # context manager를 사용하여 DB 연결 자동 관리
    with SqliteSaver.from_conn_string(DB_PATH) as checkpointer:
        graph = checkpointer_graph = builder.compile(checkpointer=checkpointer)

        # 스레드 설정 — 사용자별 고유 ID
        user_thread = {"configurable": {"thread_id": "demo-user-001"}}

        # === 3단계: 대화 진행 ===
        print("=== 첫 번째 대화 ===")
        result1 = graph.invoke(
            {"messages": [HumanMessage(content="안녕하세요! 저는 파이썬 개발자 지수입니다.")]},
            user_thread,
        )
        print(f"AI: {result1['messages'][-1].content}\n")

        result2 = graph.invoke(
            {"messages": [HumanMessage(content="LangGraph의 체크포인팅에 대해 알려주세요.")]},
            user_thread,
        )
        print(f"AI: {result2['messages'][-1].content}\n")

        result3 = graph.invoke(
            {"messages": [HumanMessage(content="제 이름을 기억하시나요?")]},
            user_thread,
        )
        print(f"AI: {result3['messages'][-1].content}\n")

        # === 4단계: 상태 조회 ===
        print("=== 현재 상태 조회 ===")
        current_state = graph.get_state(user_thread)
        print(f"메시지 수: {len(current_state.values['messages'])}")
        print(f"다음 노드: {current_state.next}")  # () — 실행 완료 상태
        print()

        # === 5단계: 상태 이력 (타임 트래블) ===
        print("=== 체크포인트 이력 ===")
        history = list(graph.get_state_history(user_thread))
        for i, state in enumerate(history):
            checkpoint_id = state.config["configurable"]["checkpoint_id"]
            msg_count = len(state.values.get("messages", []))
            next_nodes = state.next
            print(f"[{i}] checkpoint: {checkpoint_id[:12]}... | 메시지: {msg_count}개 | next: {next_nodes}")
        print()

        # === 6단계: 과거 시점으로 돌아가기 ===
        if len(history) >= 4:
            # 첫 번째 대화 직후 시점으로 돌아감
            past_config = history[-3].config  # 첫 응답 이후 체크포인트
            print("=== 타임 트래블: 첫 대화 직후로 돌아가기 ===")

            # 과거 시점에서 다른 질문으로 분기
            fork_result = graph.invoke(
                {"messages": [HumanMessage(content="LangChain 대신 LlamaIndex에 대해 알려주세요.")]},
                past_config,  # 과거 체크포인트의 config 사용
            )
            print(f"AI (분기된 대화): {fork_result['messages'][-1].content}\n")

            # 원래 대화는 그대로 유지됨
            original_state = graph.get_state(user_thread)
            print(f"원래 대화 메시지 수: {len(original_state.values['messages'])}")
            print("→ 타임 트래블은 원본을 수정하지 않고 새 포크를 생성합니다!")


if __name__ == "__main__":
    main()
    # chat_checkpoints.db 파일이 생성됨
    # 프로그램을 다시 실행하면 이전 대화가 유지됩니다!
```

## 더 깊이 알아보기

### 체크포인팅의 탄생 배경 — 이벤트 소싱에서 AI 에이전트로

LangGraph의 체크포인팅 시스템은 사실 소프트웨어 공학에서 오래된 개념인 **이벤트 소싱(Event Sourcing)**에서 영감을 받았습니다. 이벤트 소싱은 2005년 마틴 파울러(Martin Fowler)가 체계화한 패턴으로, 시스템의 상태를 직접 저장하는 대신 **상태를 변경한 이벤트의 시퀀스**를 저장하는 방식입니다. 은행 계좌의 잔액을 직접 저장하는 대신, "입금 1000원", "출금 500원" 같은 거래 내역을 저장하면 어느 시점의 잔액이든 재구성할 수 있죠.

LangGraph 팀은 이 아이디어를 AI 에이전트에 적용했습니다. 2024년 초 LangGraph v0.2 출시와 함께 체크포인터 시스템이 대폭 리팩터링되었는데, 핵심 변화는 체크포인터를 별도 패키지(`langgraph-checkpoint`, `langgraph-checkpoint-sqlite`, `langgraph-checkpoint-postgres`)로 분리한 것이었습니다. 이를 통해 Redis, MongoDB, Couchbase 등 다양한 백엔드를 커뮤니티가 자유롭게 구현할 수 있게 되었죠.

이 설계 결정의 배경에는 Harrison Chase(LangChain CEO)의 철학이 있습니다. 그는 "AI 에이전트는 본질적으로 비결정적(non-deterministic)이기 때문에, 상태의 완전한 감사 추적(audit trail)이 필수"라고 강조했는데요, 체크포인팅이 바로 그 감사 추적을 가능하게 하는 메커니즘입니다.

### 슈퍼스텝(Super-step)이란?

체크포인트가 저장되는 단위인 "슈퍼스텝"은 병렬 컴퓨팅에서 빌려온 개념입니다. 1990년 레슬리 발리언트(Leslie Valiant)가 제안한 BSP(Bulk Synchronous Parallel) 모델에서 유래했는데, 하나의 슈퍼스텝은 "모든 병렬 노드가 실행을 마치고 동기화되는 하나의 단위"를 의미합니다. LangGraph에서도 동일하게, 하나의 슈퍼스텝이 끝날 때마다 체크포인트가 자동으로 생성됩니다.

## 흔한 오해와 팁

> ⚠️ **흔한 오해**: "MemorySaver도 충분히 쓸만하다"
> 개발/테스트 환경에서는 맞는 말이지만, 프로덕션에서는 절대 사용하면 안 됩니다. 서버가 재시작되거나 프로세스가 죽으면 모든 대화 상태가 사라집니다. 게다가 메모리 사용량이 대화가 쌓일수록 계속 증가하므로, 장시간 운영 시 OOM(Out of Memory)이 발생할 수 있습니다.

> 💡 **알고 계셨나요?**: 체크포인터의 `list()` 메서드에 필터를 적용하면 특정 조건의 체크포인트만 조회할 수 있습니다. 예를 들어 `before` 파라미터로 특정 시점 이전의 체크포인트만 가져오거나, `limit`로 개수를 제한할 수 있죠. 이 기능은 대화 이력이 많아졌을 때 효율적인 조회에 필수적입니다.

> 🔥 **실무 팁**: PostgresSaver를 프로덕션에서 사용할 때는 반드시 **커넥션 풀(Connection Pool)**을 설정하세요. 매 요청마다 DB 연결을 새로 만들면 성능이 급격히 떨어집니다. `psycopg_pool.AsyncConnectionPool`의 `max_size`는 동시 사용자 수를 고려하여 설정하되, 보통 **20~50** 정도가 적당합니다. 또한 `setup()` 메서드는 최초 1회만 호출하면 됩니다 — 매 요청마다 호출하면 불필요한 오버헤드가 생깁니다.

> 🔥 **실무 팁**: `thread_id`는 사용자 ID와 대화 세션을 조합하여 만드는 것이 좋습니다. 예: `f"user-{user_id}-chat-{chat_id}"`. 이렇게 하면 한 사용자가 여러 대화를 동시에 진행할 수 있고, 대화별로 독립적인 상태를 유지할 수 있습니다.

## 핵심 정리

| 개념 | 설명 |
|------|------|
| 체크포인터(Checkpointer) | 그래프 실행 상태를 매 슈퍼스텝마다 자동 저장하는 영속성 계층 |
| MemorySaver | 메모리 기반, 개발/테스트용 — 프로세스 종료 시 소멸 |
| SqliteSaver | SQLite 파일 기반, 로컬/단일 서버용 — 재시작 후에도 유지 |
| PostgresSaver | PostgreSQL 기반, 프로덕션용 — 다중 서버, 높은 동시성 지원 |
| thread_id | 대화 세션을 식별하는 고유 키 — 호텔 객실 번호와 같은 역할 |
| get_state() | 현재(최신) 체크포인트의 상태 조회 |
| get_state_history() | 모든 체크포인트 이력을 역순으로 조회 |
| 타임 트래블(Time Travel) | 과거 체크포인트로 돌아가 새로운 경로로 실행 — 원본 보존 |
| update_state() | 특정 체크포인트의 상태를 직접 수정 |
| BaseCheckpointSaver | 모든 체크포인터가 구현하는 기본 인터페이스 |

## 다음 섹션 미리보기

이번 섹션에서 체크포인팅을 통해 상태를 "저장하고 불러오는" 방법을 배웠다면, 다음 섹션에서는 그래프를 **모듈 단위로 쪼개는** 방법을 다룹니다. **서브그래프(Subgraph)**를 활용하면 복잡한 워크플로우를 독립적인 하위 그래프로 분리하여, 재사용성과 유지보수성을 크게 높일 수 있습니다. 체크포인팅이 서브그래프 내부에서 어떻게 작동하는지도 함께 살펴볼 예정입니다.
