# 서브그래프와 모듈화

> LangGraph의 서브그래프를 활용하여 복잡한 워크플로우를 독립적이고 재사용 가능한 모듈로 분리하는 방법을 학습합니다.

## 개요

이 섹션에서는 LangGraph의 **서브그래프(Subgraph)** 개념을 배웁니다. 하나의 거대한 그래프 대신, 독립적인 소규모 그래프들을 조합하여 복잡한 에이전트 시스템을 구축하는 방법을 다룹니다.

**선수 지식**:
- [세션 13장] LangGraph 기초 — StateGraph, 노드, 엣지의 기본 개념
- [세션 14.1: Human-in-the-Loop 패턴](14.1) — interrupt()와 Command를 활용한 그래프 실행 제어
- [세션 14.2: 체크포인팅과 상태 영속성](14.2) — MemorySaver, SqliteSaver와 thread_id 기반 상태 관리

**학습 목표**:
- 서브그래프를 정의하고 부모 그래프에 통합할 수 있다
- 부모-자식 그래프 간 상태(State) 전달과 변환 패턴을 구현할 수 있다
- 재사용 가능한 그래프 컴포넌트를 설계할 수 있다
- 계층적 에이전트 구조(Hierarchical Agent Architecture)를 구축할 수 있다

## 왜 알아야 할까?

실제 프로덕션에서 AI 에이전트를 만들다 보면, 하나의 그래프가 점점 커지는 상황을 피할 수 없습니다. 고객 문의 처리 에이전트 하나만 해도 "문서 검색", "주문 조회", "환불 처리", "에스컬레이션" 같은 여러 하위 작업이 필요하죠. 이걸 전부 하나의 StateGraph에 넣으면 어떻게 될까요?

노드가 수십 개로 불어나고, 엣지 조건이 복잡해지며, 한 부분을 수정하면 다른 부분이 깨지기 쉬워집니다. 마치 모든 기능을 하나의 함수에 넣는 것과 같은 문제인데요.

서브그래프는 이 문제를 해결합니다. 각 하위 작업을 **독립적인 그래프**로 만들고, 부모 그래프에서 이들을 **노드처럼 조합**하는 방식이죠. 이렇게 하면:

- **팀 분업**: 서로 다른 팀이 독립적으로 서브그래프를 개발할 수 있습니다
- **테스트 용이성**: 각 서브그래프를 개별적으로 테스트할 수 있습니다
- **재사용**: 한번 만든 서브그래프를 여러 프로젝트에서 재사용할 수 있습니다
- **관심사 분리**: 각 서브그래프가 자신만의 상태(State)와 로직을 캡슐화합니다

## 핵심 개념

### 개념 1: 서브그래프란 무엇인가?

> 💡 **비유**: 레고 블록을 떠올려 보세요. 작은 레고 블록들로 자동차 모듈, 날개 모듈, 조종석 모듈을 각각 만든 다음, 이 모듈들을 조합해서 하나의 비행기를 완성합니다. 각 모듈은 독립적으로 만들 수 있고, 다른 작품에서도 재사용할 수 있죠. 서브그래프가 바로 이 "모듈"에 해당합니다.

서브그래프(Subgraph)는 **다른 그래프의 노드로 사용되는 그래프**입니다. 일반 함수가 노드가 되듯이, 컴파일된 그래프도 노드가 될 수 있습니다. 서브그래프는 자신만의 로직, 상태, 노드, 엣지를 가지며, 독립적으로 컴파일되고 실행됩니다.

```python
from typing_extensions import TypedDict
from langgraph.graph.state import StateGraph, START, END

# 서브그래프 정의 — 독립적인 상태와 로직을 가진 작은 그래프
class AnalysisState(TypedDict):
    text: str          # 분석할 텍스트
    sentiment: str     # 감성 분석 결과

def analyze_sentiment(state: AnalysisState) -> dict:
    """텍스트의 감성을 분석하는 노드"""
    text = state["text"]
    # 실제로는 LLM을 호출하겠지만, 여기선 간단한 규칙 기반
    if any(word in text for word in ["좋아", "훌륭", "최고"]):
        return {"sentiment": "긍정"}
    elif any(word in text for word in ["나빠", "별로", "최악"]):
        return {"sentiment": "부정"}
    return {"sentiment": "중립"}

# 서브그래프 빌더 생성 및 컴파일
analysis_builder = StateGraph(AnalysisState)
analysis_builder.add_node("analyze", analyze_sentiment)
analysis_builder.add_edge(START, "analyze")
analysis_builder.add_edge("analyze", END)

# compile()으로 실행 가능한 그래프를 만든다
analysis_subgraph = analysis_builder.compile()

# 서브그래프 단독 테스트 — 독립적으로 실행 가능!
result = analysis_subgraph.invoke({"text": "이 제품 정말 좋아요!"})
print(result)  # {'text': '이 제품 정말 좋아요!', 'sentiment': '긍정'}
```

핵심은 `compile()`입니다. 서브그래프를 `compile()`하면 `CompiledStateGraph` 객체가 되고, 이것을 부모 그래프의 `add_node()`에 직접 전달할 수 있습니다.

### 개념 2: 부모 그래프에 서브그래프 통합하기

> 💡 **비유**: 회사의 조직도를 생각해 보세요. CEO(부모 그래프)가 마케팅팀, 개발팀, 영업팀(서브그래프)에 업무를 지시합니다. 각 팀은 자기 팀만의 내부 프로세스가 있지만, CEO는 그 세부사항을 알 필요 없이 결과만 받으면 되죠.

서브그래프를 부모 그래프에 통합하는 방법은 **두 가지 시나리오**에 따라 달라집니다.

#### 시나리오 A: 부모와 자식이 상태 키를 공유하는 경우

부모 그래프와 서브그래프의 TypedDict에 같은 이름의 키가 있으면, 컴파일된 서브그래프를 `add_node()`에 바로 전달할 수 있습니다.

```python
from typing import Annotated
from typing_extensions import TypedDict
from langgraph.graph.state import StateGraph, START, END
from operator import add

# ── 서브그래프: 메시지를 처리하는 그래프 ──
class SharedState(TypedDict):
    messages: Annotated[list[str], add]  # 리듀서로 리스트 누적

def process_message(state: SharedState) -> dict:
    """메시지를 가공하는 서브그래프 노드"""
    last_msg = state["messages"][-1]
    return {"messages": [f"[처리됨] {last_msg}"]}

sub_builder = StateGraph(SharedState)
sub_builder.add_node("process", process_message)
sub_builder.add_edge(START, "process")
sub_builder.add_edge("process", END)
sub_graph = sub_builder.compile()

# ── 부모 그래프: 서브그래프를 노드로 사용 ──
def greeting(state: SharedState) -> dict:
    return {"messages": ["안녕하세요! 무엇을 도와드릴까요?"]}

parent_builder = StateGraph(SharedState)
parent_builder.add_node("greet", greeting)
parent_builder.add_node("processor", sub_graph)  # 컴파일된 서브그래프를 직접 전달!
parent_builder.add_edge(START, "greet")
parent_builder.add_edge("greet", "processor")
parent_builder.add_edge("processor", END)

parent_graph = parent_builder.compile()

# 실행
result = parent_graph.invoke({"messages": ["주문 상태 알려주세요"]})
print(result["messages"])
# ['주문 상태 알려주세요', '안녕하세요! 무엇을 도와드릴까요?',
#  '[처리됨] 안녕하세요! 무엇을 도와드릴까요?']
```

#### 시나리오 B: 부모와 자식의 상태 스키마가 다른 경우

실무에서 더 흔한 패턴입니다. 서브그래프가 부모와 완전히 다른 상태 스키마를 가질 때, **래퍼 함수(wrapper function)**로 상태를 변환합니다.

```python
from typing_extensions import TypedDict
from langgraph.graph.state import StateGraph, START, END

# ── 서브그래프: 자체 스키마를 가진 번역 그래프 ──
class TranslationState(TypedDict):
    source_text: str       # 원본 텍스트
    target_language: str   # 대상 언어
    translated: str        # 번역 결과

def translate(state: TranslationState) -> dict:
    """번역 수행 (실제로는 LLM 호출)"""
    text = state["source_text"]
    lang = state["target_language"]
    # 간단한 시뮬레이션
    return {"translated": f"[{lang}로 번역됨] {text}"}

trans_builder = StateGraph(TranslationState)
trans_builder.add_node("translate", translate)
trans_builder.add_edge(START, "translate")
trans_builder.add_edge("translate", END)
translation_subgraph = trans_builder.compile()

# ── 부모 그래프: 다른 스키마를 사용 ──
class ParentState(TypedDict):
    user_query: str       # 사용자 질의
    response: str         # 최종 응답

def call_translation(state: ParentState) -> dict:
    """래퍼 함수: 부모 상태 → 서브그래프 상태 변환"""
    # 1) 부모 상태에서 서브그래프 입력으로 변환
    sub_input = {
        "source_text": state["user_query"],
        "target_language": "영어",
        "translated": "",
    }
    # 2) 서브그래프 호출
    sub_output = translation_subgraph.invoke(sub_input)
    # 3) 서브그래프 출력을 부모 상태 형식으로 변환
    return {"response": sub_output["translated"]}

parent_builder = StateGraph(ParentState)
parent_builder.add_node("translate", call_translation)  # 래퍼 함수를 노드로!
parent_builder.add_edge(START, "translate")
parent_builder.add_edge("translate", END)

parent_graph = parent_builder.compile()

result = parent_graph.invoke({
    "user_query": "오늘 날씨가 좋습니다",
    "response": "",
})
print(result["response"])
# [영어로 번역됨] 오늘 날씨가 좋습니다
```

> ⚠️ **흔한 오해**: 부모와 서브그래프의 상태 키가 하나도 겹치지 않는데 컴파일된 서브그래프를 `add_node()`에 바로 전달하면, LangGraph가 에러를 발생시킵니다. 이 경우 반드시 래퍼 함수를 사용해야 합니다.

### 개념 3: 체크포인터와 서브그래프의 관계

앞서 [세션 14.2: 체크포인팅과 상태 영속성](14.2)에서 체크포인터의 중요성을 배웠는데요, 서브그래프에서 체크포인터가 어떻게 동작하는지는 꼭 이해해야 할 부분입니다.

> 💡 **비유**: 대학 시험을 생각해 보세요. 학교(부모 그래프)에서 시험지를 관리하는 시스템(체크포인터)을 하나 운영합니다. 각 학과(서브그래프)가 별도의 시험지 관리 시스템을 만들 필요 없이, 학교의 중앙 시스템을 함께 쓰면 되죠. LangGraph에서도 마찬가지입니다.

**핵심 규칙**: 체크포인터는 **부모 그래프를 컴파일할 때만** 전달합니다. LangGraph가 자동으로 모든 서브그래프에 체크포인터를 전파(propagate)합니다.

```python
from langgraph.checkpoint.memory import MemorySaver

# ❌ 잘못된 방법 — 서브그래프에 체크포인터를 직접 전달
# sub_graph = sub_builder.compile(checkpointer=MemorySaver())  # 무시됨!

# ✅ 올바른 방법 — 부모 그래프에만 체크포인터 전달
checkpointer = MemorySaver()
parent_graph = parent_builder.compile(checkpointer=checkpointer)

# 서브그래프에도 체크포인터가 자동 전파됨
# 이제 서브그래프 내부 노드의 실행 상태도 저장됩니다
result = parent_graph.invoke(
    {"messages": ["첫 번째 메시지"]},
    config={"configurable": {"thread_id": "user-123"}},
)
```

### 개념 4: 서브그래프 스트리밍

부모 그래프를 `stream()`으로 실행할 때, 서브그래프 내부에서 발생하는 이벤트도 볼 수 있습니다. 이를 위해 `subgraphs=True` 파라미터를 사용합니다.

```python
# 서브그래프 이벤트까지 포함하여 스트리밍
for namespace, chunk in parent_graph.stream(
    {"messages": ["분석해 주세요"]},
    stream_mode="updates",
    subgraphs=True,
):
    # namespace: 어떤 그래프(부모/자식)에서 온 이벤트인지 식별
    # chunk: 해당 노드의 상태 업데이트
    print(f"[{namespace}] {chunk}")

# 출력 예시:
# [()] {'greet': {'messages': ['안녕하세요! ...']}}
# [('processor:abc123',)] {'process': {'messages': ['[처리됨] ...']}}
```

`namespace`는 튜플로, 빈 튜플 `()`은 부모 그래프를, `('서브그래프이름:id',)` 형태는 서브그래프를 나타냅니다. 이 네임스페이스 격리(namespace isolation) 덕분에 부모와 서브그래프의 이벤트를 명확히 구분할 수 있습니다.

### 개념 5: 계층적 에이전트 구조

> 💡 **비유**: 대기업의 조직도를 떠올려 보세요. CEO 아래에 CTO, CMO, CFO가 있고, CTO 아래에는 프론트엔드팀, 백엔드팀, 인프라팀이 있습니다. 각 팀장(중간 관리자)은 자기 팀의 업무를 관장하고, 필요하면 팀원에게 세부 업무를 위임하죠. 이런 계층 구조를 그래프로 그대로 옮긴 것이 계층적 에이전트 구조입니다.

계층적 에이전트 구조(Hierarchical Agent Architecture)는 서브그래프를 여러 겹으로 중첩하는 패턴입니다. 최상위 **슈퍼바이저(supervisor)** 그래프가 업무를 분류하고, 각 전문 서브그래프(팀)에 작업을 위임합니다.

```python
from typing import Literal
from typing_extensions import TypedDict
from langgraph.graph.state import StateGraph, START, END

# ── 1단계: 전문가 서브그래프들 ──
class ExpertState(TypedDict):
    query: str
    answer: str

def tech_expert(state: ExpertState) -> dict:
    """기술 지원 전문가"""
    return {"answer": f"기술 답변: {state['query']}에 대한 해결 방법은..."}

def billing_expert(state: ExpertState) -> dict:
    """결제 전문가"""
    return {"answer": f"결제 답변: {state['query']}에 대한 처리 결과는..."}

# 기술 지원 서브그래프
tech_builder = StateGraph(ExpertState)
tech_builder.add_node("expert", tech_expert)
tech_builder.add_edge(START, "expert")
tech_builder.add_edge("expert", END)
tech_subgraph = tech_builder.compile()

# 결제 서브그래프
billing_builder = StateGraph(ExpertState)
billing_builder.add_node("expert", billing_expert)
billing_builder.add_edge(START, "expert")
billing_builder.add_edge("expert", END)
billing_subgraph = billing_builder.compile()

# ── 2단계: 슈퍼바이저 (라우터 역할) ──
class SupervisorState(TypedDict):
    query: str
    category: str
    answer: str

def classify(state: SupervisorState) -> dict:
    """질문을 분류하는 노드 (실제로는 LLM 활용)"""
    query = state["query"].lower()
    if any(w in query for w in ["오류", "버그", "설치", "에러"]):
        return {"category": "tech"}
    return {"category": "billing"}

def route_to_expert(state: SupervisorState) -> Literal["tech_team", "billing_team"]:
    """분류 결과에 따라 적절한 팀으로 라우팅"""
    return "tech_team" if state["category"] == "tech" else "billing_team"

def call_tech(state: SupervisorState) -> dict:
    """기술팀 서브그래프 호출 래퍼"""
    result = tech_subgraph.invoke({"query": state["query"], "answer": ""})
    return {"answer": result["answer"]}

def call_billing(state: SupervisorState) -> dict:
    """결제팀 서브그래프 호출 래퍼"""
    result = billing_subgraph.invoke({"query": state["query"], "answer": ""})
    return {"answer": result["answer"]}

# 슈퍼바이저 그래프 조립
supervisor_builder = StateGraph(SupervisorState)
supervisor_builder.add_node("classify", classify)
supervisor_builder.add_node("tech_team", call_tech)
supervisor_builder.add_node("billing_team", call_billing)

supervisor_builder.add_edge(START, "classify")
supervisor_builder.add_conditional_edges("classify", route_to_expert)
supervisor_builder.add_edge("tech_team", END)
supervisor_builder.add_edge("billing_team", END)

supervisor = supervisor_builder.compile()

# 실행
result = supervisor.invoke({
    "query": "프로그램 설치 중 오류가 발생합니다",
    "category": "",
    "answer": "",
})
print(result["answer"])
# 기술 답변: 프로그램 설치 중 오류가 발생합니다에 대한 해결 방법은...
```

## 실습: 직접 해보기

실제 프로덕션에서 쓸 법한 **다단계 문서 처리 파이프라인**을 서브그래프로 구축해 봅시다. 문서를 받아서 요약, 키워드 추출, 감성 분석을 각각 독립적인 서브그래프로 처리한 뒤, 결과를 통합하는 구조입니다.

```python
"""
다단계 문서 처리 파이프라인
- 요약 서브그래프: 문서 요약 담당
- 키워드 서브그래프: 핵심 키워드 추출
- 감성 서브그래프: 감성 분석
- 오케스트레이터: 3개 서브그래프를 통합 관리
"""

from typing import Annotated
from typing_extensions import TypedDict
from langgraph.graph.state import StateGraph, START, END
from langgraph.checkpoint.memory import MemorySaver

# ========================================
# 1. 요약 서브그래프
# ========================================
class SummaryState(TypedDict):
    text: str
    summary: str

def summarize(state: SummaryState) -> dict:
    """텍스트를 요약하는 노드 (실제로는 LLM 호출)"""
    text = state["text"]
    # 간단한 시뮬레이션: 앞 50자 + "..."
    short = text[:50] + "..." if len(text) > 50 else text
    return {"summary": f"요약: {short}"}

summary_builder = StateGraph(SummaryState)
summary_builder.add_node("summarize", summarize)
summary_builder.add_edge(START, "summarize")
summary_builder.add_edge("summarize", END)
summary_subgraph = summary_builder.compile()

# ========================================
# 2. 키워드 추출 서브그래프
# ========================================
class KeywordState(TypedDict):
    text: str
    keywords: list[str]

def extract_keywords(state: KeywordState) -> dict:
    """핵심 키워드를 추출하는 노드"""
    text = state["text"]
    # 간단한 시뮬레이션: 긴 단어 추출
    words = text.replace(",", "").replace(".", "").split()
    keywords = [w for w in words if len(w) >= 3][:5]
    return {"keywords": keywords}

keyword_builder = StateGraph(KeywordState)
keyword_builder.add_node("extract", extract_keywords)
keyword_builder.add_edge(START, "extract")
keyword_builder.add_edge("extract", END)
keyword_subgraph = keyword_builder.compile()

# ========================================
# 3. 감성 분석 서브그래프
# ========================================
class SentimentState(TypedDict):
    text: str
    sentiment: str
    confidence: float

def analyze_sentiment(state: SentimentState) -> dict:
    """감성을 분석하는 노드"""
    text = state["text"]
    positive_words = ["좋", "훌륭", "만족", "추천", "최고", "감사"]
    negative_words = ["나쁘", "불만", "실망", "별로", "최악", "화남"]

    pos_count = sum(1 for w in positive_words if w in text)
    neg_count = sum(1 for w in negative_words if w in text)
    total = pos_count + neg_count

    if total == 0:
        return {"sentiment": "중립", "confidence": 0.5}
    elif pos_count > neg_count:
        return {"sentiment": "긍정", "confidence": pos_count / total}
    else:
        return {"sentiment": "부정", "confidence": neg_count / total}

sentiment_builder = StateGraph(SentimentState)
sentiment_builder.add_node("analyze", analyze_sentiment)
sentiment_builder.add_edge(START, "analyze")
sentiment_builder.add_edge("analyze", END)
sentiment_subgraph = sentiment_builder.compile()

# ========================================
# 4. 오케스트레이터 (부모 그래프)
# ========================================
class PipelineState(TypedDict):
    document: str           # 원본 문서
    summary: str            # 요약 결과
    keywords: list[str]     # 키워드 목록
    sentiment: str          # 감성 분석 결과
    confidence: float       # 감성 신뢰도
    final_report: str       # 최종 통합 보고서

def call_summary(state: PipelineState) -> dict:
    """요약 서브그래프 호출 래퍼"""
    result = summary_subgraph.invoke({
        "text": state["document"],
        "summary": "",
    })
    return {"summary": result["summary"]}

def call_keywords(state: PipelineState) -> dict:
    """키워드 추출 서브그래프 호출 래퍼"""
    result = keyword_subgraph.invoke({
        "text": state["document"],
        "keywords": [],
    })
    return {"keywords": result["keywords"]}

def call_sentiment(state: PipelineState) -> dict:
    """감성 분석 서브그래프 호출 래퍼"""
    result = sentiment_subgraph.invoke({
        "text": state["document"],
        "sentiment": "",
        "confidence": 0.0,
    })
    return {
        "sentiment": result["sentiment"],
        "confidence": result["confidence"],
    }

def generate_report(state: PipelineState) -> dict:
    """모든 분석 결과를 통합하여 최종 보고서 생성"""
    report = f"""
=== 문서 분석 보고서 ===
📝 요약: {state['summary']}
🔑 키워드: {', '.join(state['keywords'])}
💬 감성: {state['sentiment']} (신뢰도: {state['confidence']:.0%})
========================
""".strip()
    return {"final_report": report}

# 오케스트레이터 그래프 조립
pipeline_builder = StateGraph(PipelineState)

# 노드 등록 — 래퍼 함수를 사용하여 상태 변환
pipeline_builder.add_node("summarize", call_summary)
pipeline_builder.add_node("extract_keywords", call_keywords)
pipeline_builder.add_node("analyze_sentiment", call_sentiment)
pipeline_builder.add_node("report", generate_report)

# 엣지 연결 — 순차 실행
pipeline_builder.add_edge(START, "summarize")
pipeline_builder.add_edge("summarize", "extract_keywords")
pipeline_builder.add_edge("extract_keywords", "analyze_sentiment")
pipeline_builder.add_edge("analyze_sentiment", "report")
pipeline_builder.add_edge("report", END)

# 체크포인터는 부모 그래프에만 전달 — 서브그래프에 자동 전파
checkpointer = MemorySaver()
pipeline = pipeline_builder.compile(checkpointer=checkpointer)

# ========================================
# 5. 실행!
# ========================================
test_document = """
LangChain은 대규모 언어 모델 기반 애플리케이션을 만들기 위한 훌륭한 프레임워크입니다.
특히 RAG 구축이나 에이전트 개발에서 생산성이 매우 높아 만족스럽습니다.
다양한 벡터 스토어와의 통합도 추천할 만하며, 커뮤니티 지원도 최고입니다.
"""

result = pipeline.invoke(
    {
        "document": test_document,
        "summary": "",
        "keywords": [],
        "sentiment": "",
        "confidence": 0.0,
        "final_report": "",
    },
    config={"configurable": {"thread_id": "doc-analysis-001"}},
)

print(result["final_report"])
# 출력:
# === 문서 분석 보고서 ===
# 📝 요약: 요약: LangChain은 대규모 언어 모델 기반 애플리케이션을 만들기 위한 훌륭한 프레임워크입니...
# 🔑 키워드: LangChain은, 대규모, 언어, 모델, 기반
# 💬 감성: 긍정 (신뢰도: 100%)
# ========================

# 스트리밍으로 서브그래프 내부 이벤트 관찰
print("\n--- 스트리밍 출력 ---")
for namespace, chunk in pipeline.stream(
    {
        "document": "이 서비스는 별로입니다. 불만족스러워요.",
        "summary": "",
        "keywords": [],
        "sentiment": "",
        "confidence": 0.0,
        "final_report": "",
    },
    config={"configurable": {"thread_id": "doc-analysis-002"}},
    stream_mode="updates",
    subgraphs=True,
):
    node_name = list(chunk.keys())[0] if isinstance(chunk, dict) else "unknown"
    print(f"  네임스페이스: {namespace} | 노드: {node_name}")
```

## 더 깊이 알아보기

### 그래프 이론에서 서브그래프까지

서브그래프라는 개념은 사실 **그래프 이론(Graph Theory)**에서 온 용어입니다. 1736년 레온하르트 오일러(Leonhard Euler)가 쾨니히스베르크 다리 문제를 풀며 시작된 그래프 이론에서, 서브그래프는 "원래 그래프의 정점과 간선의 부분집합으로 이루어진 그래프"를 뜻합니다. LangGraph가 이 이름을 차용한 건 자연스러운 선택이었죠.

LangGraph의 창시자 해리슨 체이스(Harrison Chase)는 2024년 초 LangGraph를 발표하면서 이런 말을 했습니다:

> "에이전트에 대한 모든 논문을 보면, 에이전트를 그래프로 설명하고 시각화합니다. 그렇다면 왜 그래프로 쉽게 만들 수 있는 방법이 없을까요?"

LangGraph는 구글의 **Pregel** 시스템과 **Apache Beam**에서 영감을 받았고, 공개 인터페이스는 Python의 NetworkX 라이브러리에서 영감을 받았습니다. Pregel은 구글이 웹 페이지 랭킹(PageRank) 같은 대규모 그래프 연산을 처리하기 위해 만든 시스템인데, "전체 그래프를 작은 단위로 분할하여 병렬 처리한다"는 핵심 아이디어가 LangGraph의 서브그래프 개념과 일맥상통합니다.

### 마이크로서비스와의 유사성

서브그래프 패턴은 소프트웨어 아키텍처의 **마이크로서비스(Microservice)** 패턴과 놀라울 정도로 유사합니다. 모놀리식 애플리케이션을 독립적인 서비스들로 분리하듯, 거대한 에이전트 그래프를 독립적인 서브그래프들로 분리하는 거죠. 각 서비스/서브그래프는 자신만의 "상태"를 가지고, 잘 정의된 인터페이스(상태 스키마)를 통해 소통합니다.

## 흔한 오해와 팁

> ⚠️ **흔한 오해**: "서브그래프에도 체크포인터를 별도로 전달해야 한다"고 생각하는 분이 많습니다. 실제로는 **부모 그래프에만 체크포인터를 전달**하면 LangGraph가 자동으로 모든 서브그래프에 전파합니다. 서브그래프의 `compile()`에 체크포인터를 전달해도 무시됩니다.

> 💡 **알고 계셨나요?**: LangGraph의 서브그래프 스트리밍에서 반환되는 `namespace` 튜플의 깊이는 서브그래프의 중첩 수준을 나타냅니다. `()`는 부모, `('sub1:id',)`는 1단계 서브그래프, `('sub1:id', 'sub2:id')`는 2단계 중첩 서브그래프입니다. 이 구조 덕분에 아무리 깊이 중첩된 서브그래프라도 어디서 발생한 이벤트인지 추적할 수 있습니다.

> 🔥 **실무 팁**: 서브그래프를 설계할 때 가장 중요한 원칙은 **"인터페이스 우선 설계"**입니다. 서브그래프의 입력/출력 상태 스키마(TypedDict)를 먼저 정의하고, 내부 로직은 나중에 구현하세요. 이렇게 하면 다른 팀이 부모 그래프를 먼저 만들고, 서브그래프 구현은 각 담당 팀에서 독립적으로 진행할 수 있습니다. 마치 API 스펙을 먼저 정하고 프론트/백엔드를 분리 개발하는 것과 같은 원리입니다.

> 🔥 **실무 팁**: 서브그래프가 3단계 이상 중첩되면 디버깅이 매우 어려워집니다. 대부분의 경우 2단계(부모 → 자식)면 충분합니다. 꼭 3단계 이상이 필요하다면, `subgraphs=True` 스트리밍과 LangSmith 트레이싱을 반드시 함께 사용하세요.

## 핵심 정리

| 개념 | 설명 |
|------|------|
| 서브그래프(Subgraph) | 다른 그래프의 노드로 사용되는 독립적인 컴파일된 그래프 |
| 공유 스키마 통합 | 부모/자식이 같은 상태 키를 가지면 `add_node()`에 컴파일된 그래프 직접 전달 |
| 상태 변환 (래퍼 함수) | 스키마가 다를 때 래퍼 함수로 부모 상태 ↔ 서브그래프 상태 변환 |
| 체크포인터 전파 | 부모 그래프에만 체크포인터 전달, 서브그래프에 자동 전파 |
| 네임스페이스 격리 | 스트리밍 시 `subgraphs=True`로 서브그래프 이벤트를 네임스페이스로 구분 |
| 계층적 에이전트 구조 | 슈퍼바이저 → 전문가 서브그래프로 업무를 분류·위임하는 아키텍처 |
| 인터페이스 우선 설계 | 상태 스키마(TypedDict)를 먼저 정의하고, 내부 구현은 독립적으로 진행 |
