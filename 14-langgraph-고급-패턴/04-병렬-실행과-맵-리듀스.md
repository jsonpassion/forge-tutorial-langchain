# 병렬 실행과 맵-리듀스

> LangGraph의 Send API를 활용하여 런타임에 동적으로 병렬 노드를 생성하고, 맵-리듀스 패턴으로 결과를 집계하는 방법을 학습합니다.

## 개요

이 섹션에서는 LangGraph의 가장 강력한 제어 흐름 기능 중 하나인 **Send API**를 배웁니다. 앞서 [서브그래프와 모듈화](ch14_03)에서 복잡한 워크플로우를 모듈 단위로 분리하는 법을 익혔다면, 이번에는 **런타임에 작업의 수가 결정되는** 동적 병렬 실행을 다룹니다. Send API와 리듀서(Reducer)를 조합하면 고전적인 맵-리듀스 패턴을 LangGraph 그래프 위에서 깔끔하게 구현할 수 있습니다.

**선수 지식**: 
- [서브그래프와 모듈화](ch14_03)에서 배운 서브그래프 구성과 상태 전달
- [체크포인팅과 상태 영속성](ch14_02)에서 배운 체크포인터와 슈퍼스텝 개념
- [LCEL 마스터](ch05)에서 배운 `Runnable` 인터페이스와 파이프 연산자

**학습 목표**:
- `Send` 클래스를 사용하여 동적으로 병렬 노드를 생성할 수 있다
- `Annotated`와 `operator.add` 리듀서로 병렬 결과를 안전하게 집계할 수 있다
- 맵-리듀스 패턴을 실전 워크플로우에 적용할 수 있다
- `max_concurrency` 설정으로 병렬 실행을 제어할 수 있다

## 왜 알아야 할까?

실무에서 LLM 애플리케이션을 만들다 보면, **처리할 작업의 수를 미리 알 수 없는** 상황이 빈번하게 발생합니다. 몇 가지 예를 들어볼까요?

- 사용자가 업로드한 문서가 3개일 수도, 30개일 수도 있는 **다중 문서 분석**
- 검색 결과 5건을 동시에 요약해야 하는 **병렬 요약**
- 하나의 에세이를 여러 관점(문법, 논리, 창의성)에서 동시에 평가하는 **병렬 평가**
- 여러 도시의 날씨를 한꺼번에 조회하는 **배치 API 호출**

이런 경우 순차 실행은 시간이 N배로 늘어납니다. 문서 10개를 각각 30초씩 분석하면 5분이지만, 병렬로 처리하면 30초면 끝나거든요. LangGraph의 Send API는 이런 동적 병렬 실행을 **그래프의 일급 시민(first-class citizen)**으로 지원합니다. 별도의 스레딩 코드나 복잡한 동기화 로직 없이, 선언적으로 "이 데이터들을 병렬로 처리해줘"라고 표현할 수 있죠.

## 핵심 개념

### 개념 1: 정적 병렬 vs 동적 병렬

> 💡 **비유**: 레스토랑에서 주문을 받는 두 가지 방식을 생각해보세요. **정적 병렬**은 항상 3명의 셰프가 전채·메인·디저트를 동시에 만드는 코스 요리입니다 — 메뉴가 정해져 있죠. 반면 **동적 병렬**은 뷔페 케이터링입니다. 손님이 50명일 수도, 200명일 수도 있어서, 주문이 들어온 뒤에야 몇 명의 셰프가 필요한지 결정됩니다.

LangGraph에서 정적 병렬은 이미 익숙한 패턴입니다. 그래프를 설계할 때 여러 노드가 동일한 선행 노드에서 분기되도록 엣지를 그으면, LangGraph는 이들을 **같은 슈퍼스텝(super-step)**에서 동시에 실행합니다.

```python
# 정적 병렬: 그래프 정의 시점에 분기가 결정됨
builder.add_edge("start", "analyze_grammar")
builder.add_edge("start", "analyze_logic")
builder.add_edge("start", "analyze_creativity")
```

하지만 이 방식은 **분기 수가 고정**되어 있습니다. 처리할 항목이 3개든 100개든 유연하게 대응할 수 없죠. 여기서 **Send API**가 등장합니다.

### 개념 2: Send API — 동적 팬아웃(Fan-out)

> 💡 **비유**: Send는 복사기와 같습니다. 원본 문서(원래 상태)를 넣으면 필요한 만큼 복사본을 만들어서, 각 복사본을 다른 사람(노드)에게 동시에 전달합니다. 복사본마다 "3페이지를 번역해주세요", "5페이지를 요약해주세요"처럼 **서로 다른 지시사항**을 붙일 수도 있습니다.

`Send`는 LangGraph의 조건부 엣지(conditional edge)에서 반환하는 객체입니다. 각 `Send` 객체는 **어떤 노드를**, **어떤 상태로** 호출할지를 지정합니다.

```python
from langgraph.types import Send

# Send(노드_이름, 해당_노드에_전달할_상태)
Send("process_item", {"item": "첫 번째 항목", "index": 0})
```

핵심은 **조건부 엣지 함수에서 `Send` 객체의 리스트를 반환**하면, LangGraph가 이들을 모두 병렬로 실행한다는 것입니다.

```python
from typing import Literal
from langgraph.types import Send

def fan_out(state: OverallState) -> list[Send]:
    """런타임에 항목 수만큼 동적으로 Send 생성"""
    return [
        Send("process_item", {"item": item})
        for item in state["items"]
    ]

# 조건부 엣지로 등록
builder.add_conditional_edges("collect_items", fan_out, ["process_item"])
```

여기서 `state["items"]`가 3개면 3개의 `process_item` 노드가, 100개면 100개가 동시에 실행됩니다. **런타임에 항목 수가 결정**되는 거죠.

### 개념 3: 리듀서(Reducer) — 팬인(Fan-in)과 결과 집계

> 💡 **비유**: 학교에서 모둠 과제를 할 때, 각 조원이 맡은 파트를 완성하면 한 사람이 그걸 모아서 하나의 보고서로 합치잖아요? 리듀서가 바로 그 "합치는 사람" 역할입니다. 각 병렬 노드가 반환한 결과물들을 하나의 상태로 통합합니다.

병렬 노드들이 **같은 상태 키**에 값을 쓰면 문제가 생깁니다. 마지막에 완료된 노드의 결과만 남고 나머지는 덮어쓰이거든요. 이를 방지하려면 **리듀서 함수**를 지정해야 합니다.

```python
import operator
from typing import Annotated, TypedDict

class OverallState(TypedDict):
    items: list[str]                                    # 처리할 항목 목록
    results: Annotated[list[str], operator.add]         # 리듀서: 리스트 합치기
```

`Annotated[list[str], operator.add]`는 "이 키에 새 값이 들어오면 기존 값에 **append(합치기)**해라"라는 의미입니다. 병렬 노드 A가 `["결과1"]`을, 노드 B가 `["결과2"]`를 반환하면, 최종 상태는 `["결과1", "결과2"]`가 됩니다.

자주 사용되는 리듀서 패턴을 표로 정리해볼까요?

| 리듀서 | 용도 | 예시 |
|--------|------|------|
| `operator.add` | 리스트 합치기 | `[1,2] + [3] → [1,2,3]` |
| 커스텀 함수 | 딕셔너리 병합 | `{a:1} ∪ {b:2} → {a:1, b:2}` |
| `add_messages` | 메시지 리스트 관리 | LangChain 메시지 누적 |

### 개념 4: 맵-리듀스 패턴 완성하기

> 💡 **비유**: 맵-리듀스는 **피자 만들기** 과정과 같습니다. 주문서(입력)를 받으면, "토핑 준비", "도우 밀기", "소스 만들기"를 **동시에**(Map) 진행한 뒤, 모든 재료가 준비되면 하나의 피자로 **조립**(Reduce)합니다. 주문이 10판이면 10개의 라인이 동시에 돌아가는 거죠.

맵-리듀스 패턴은 세 단계로 구성됩니다:

1. **분배(Distribute)**: 입력 데이터를 개별 작업 단위로 분할
2. **맵(Map)**: 각 작업을 병렬로 처리 — `Send`가 담당
3. **리듀스(Reduce)**: 결과를 집계 — `Annotated` 리듀서가 담당

전체 구조를 코드로 보겠습니다:

```python
import operator
from typing import Annotated, TypedDict
from langgraph.graph import StateGraph, START, END
from langgraph.types import Send

# ── 1. 상태 정의 ──
class OverallState(TypedDict):
    """메인 그래프의 전체 상태"""
    topics: list[str]                                      # 처리할 주제 목록
    summaries: Annotated[list[str], operator.add]          # 리듀서로 결과 집계
    final_report: str                                      # 최종 통합 보고서


class TopicState(TypedDict):
    """개별 맵 노드가 받는 상태 (메인 상태와 다를 수 있음!)"""
    topic: str


# ── 2. 노드 함수 정의 ──
def collect_topics(state: OverallState) -> dict:
    """주제 목록을 준비하는 시작 노드"""
    return {"topics": state["topics"]}


def summarize_topic(state: TopicState) -> dict:
    """개별 주제를 요약하는 맵 노드 (병렬 실행됨)"""
    topic = state["topic"]
    # 실제로는 여기서 LLM을 호출합니다
    summary = f"[{topic}] 에 대한 상세 분석 결과입니다."
    return {"summaries": [summary]}  # 리스트로 감싸서 반환!


def compile_report(state: OverallState) -> dict:
    """모든 요약을 통합하는 리듀스 노드"""
    all_summaries = "\n".join(state["summaries"])
    return {"final_report": f"통합 보고서:\n{all_summaries}"}


# ── 3. 팬아웃 라우터 ──
def fan_out_topics(state: OverallState) -> list[Send]:
    """주제 수만큼 동적으로 Send 객체 생성"""
    return [
        Send("summarize_topic", {"topic": topic})
        for topic in state["topics"]
    ]


# ── 4. 그래프 조립 ──
builder = StateGraph(OverallState)
builder.add_node("collect_topics", collect_topics)
builder.add_node("summarize_topic", summarize_topic)
builder.add_node("compile_report", compile_report)

builder.add_edge(START, "collect_topics")
builder.add_conditional_edges("collect_topics", fan_out_topics, ["summarize_topic"])
builder.add_edge("summarize_topic", "compile_report")
builder.add_edge("compile_report", END)

graph = builder.compile()
```

여기서 주목할 점이 있습니다. `summarize_topic` 노드는 `OverallState`가 아닌 **별도의 `TopicState`**를 받습니다. Send로 전달되는 상태는 메인 그래프의 상태와 **다른 스키마**를 가질 수 있거든요. 하지만 반환값의 키(`summaries`)는 반드시 **메인 그래프의 상태 키**와 일치해야 합니다.

### 개념 5: 동시성 제어 — max_concurrency

> 💡 **비유**: 고속도로의 톨게이트를 생각해보세요. 차가 100대가 와도 톨게이트가 5개뿐이면 동시에 5대만 통과할 수 있습니다. `max_concurrency`는 이 톨게이트 수를 설정하는 것입니다.

Send로 100개의 병렬 작업을 생성했다고 해서 실제로 100개가 동시에 실행되는 것은 아닙니다. API 호출 제한(rate limit)이나 메모리 제약이 있을 수 있거든요. `max_concurrency`로 동시 실행 수를 제한할 수 있습니다.

```python
# 최대 5개의 노드만 동시 실행
result = graph.invoke(
    {"topics": topics_list},
    config={"max_concurrency": 5}
)
```

이 설정이 없으면 LangGraph는 가능한 한 많은 노드를 동시에 실행합니다. 프로덕션 환경에서는 **반드시** `max_concurrency`를 설정하여 외부 API의 rate limit을 존중하는 것이 좋습니다.

## 실습: 직접 해보기

여러 주제에 대한 기사를 동시에 생성하고, 결과를 하나의 뉴스레터로 합치는 **병렬 기사 생성기**를 만들어봅시다.

```python
"""
병렬 기사 생성기 — LangGraph Send API + 맵-리듀스 패턴 실습
"""
import operator
from typing import Annotated, TypedDict

from dotenv import load_dotenv
from langchain_openai import ChatOpenAI
from langchain_core.prompts import ChatPromptTemplate
from langgraph.graph import StateGraph, START, END
from langgraph.types import Send

load_dotenv()

# ── 설정 ──
MODEL_NAME = "gpt-4o"
TEMPERATURE = 0.7


# ── 상태 정의 ──
class NewsletterState(TypedDict):
    """전체 뉴스레터 상태"""
    topics: list[str]                                       # 기사로 작성할 주제 목록
    articles: Annotated[list[dict], operator.add]           # 리듀서로 기사 모음
    newsletter: str                                         # 최종 뉴스레터


class ArticleState(TypedDict):
    """개별 기사 작성 노드의 상태"""
    topic: str                                              # 작성할 주제


# ── LLM 체인 ──
llm = ChatOpenAI(model=MODEL_NAME, temperature=TEMPERATURE)

article_prompt = ChatPromptTemplate.from_messages([
    ("system", "당신은 기술 뉴스 기자입니다. 주어진 주제로 200자 이내의 짧은 기사를 작성하세요."),
    ("human", "주제: {topic}")
])
article_chain = article_prompt | llm

newsletter_prompt = ChatPromptTemplate.from_messages([
    ("system", "당신은 뉴스레터 편집자입니다. 여러 기사를 하나의 뉴스레터로 편집하세요."),
    ("human", "기사 목록:\n{articles_text}")
])
newsletter_chain = newsletter_prompt | llm


# ── 노드 함수 ──
def generate_article(state: ArticleState) -> dict:
    """개별 기사를 생성하는 맵 노드 (병렬 실행됨)"""
    topic = state["topic"]
    response = article_chain.invoke({"topic": topic})
    # 리스트로 감싸서 반환 — operator.add 리듀서가 합침
    return {"articles": [{"topic": topic, "content": response.content}]}


def compile_newsletter(state: NewsletterState) -> dict:
    """모든 기사를 하나의 뉴스레터로 통합하는 리듀스 노드"""
    articles_text = "\n\n---\n\n".join(
        f"## {a['topic']}\n{a['content']}" for a in state["articles"]
    )
    response = newsletter_chain.invoke({"articles_text": articles_text})
    return {"newsletter": response.content}


# ── 팬아웃 라우터 ──
def route_to_articles(state: NewsletterState) -> list[Send]:
    """주제 수만큼 동적으로 기사 작성 노드를 생성"""
    return [
        Send("generate_article", {"topic": topic})
        for topic in state["topics"]
    ]


# ── 그래프 조립 ──
builder = StateGraph(NewsletterState)
builder.add_node("generate_article", generate_article)
builder.add_node("compile_newsletter", compile_newsletter)

# START → (팬아웃) → generate_article(들) → compile_newsletter → END
builder.add_conditional_edges(START, route_to_articles, ["generate_article"])
builder.add_edge("generate_article", "compile_newsletter")
builder.add_edge("compile_newsletter", END)

graph = builder.compile()


# ── 실행 ──
if __name__ == "__main__":
    result = graph.invoke(
        {
            "topics": [
                "LangGraph의 Send API",
                "RAG 파이프라인 최적화",
                "멀티 에이전트 시스템의 미래",
            ]
        },
        # 동시에 최대 2개 노드만 실행 (API rate limit 고려)
        config={"max_concurrency": 2},
    )

    print("=" * 60)
    print("📰 생성된 뉴스레터")
    print("=" * 60)
    print(result["newsletter"])
    print("\n--- 개별 기사 수:", len(result["articles"]))
```

실행하면 3개의 주제에 대해 기사가 **병렬로 생성**되고(단, `max_concurrency=2`이므로 동시에 2개까지), 완료되면 하나의 뉴스레터로 편집됩니다.

```
============================================================
📰 생성된 뉴스레터
============================================================
# 이번 주 AI 기술 뉴스레터

## LangGraph의 Send API
LangGraph가 Send API를 통해 동적 병렬 실행을 지원합니다...

## RAG 파이프라인 최적화  
검색 증강 생성(RAG) 파이프라인의 성능 최적화 전략이...

## 멀티 에이전트 시스템의 미래
여러 AI 에이전트가 협력하는 멀티 에이전트 시스템이...

--- 개별 기사 수: 3
```

## 더 깊이 알아보기

### 맵-리듀스의 탄생: 구글의 빅데이터 혁명

맵-리듀스라는 이름이 낯익게 느껴지시나요? 이 패턴의 기원은 2004년 구글의 **제프리 딘(Jeffrey Dean)**과 **산제이 게마와트(Sanjay Ghemawat)**가 발표한 논문 *"MapReduce: Simplified Data Processing on Large Clusters"*로 거슬러 올라갑니다.

당시 구글은 매일 수십 테라바이트의 웹 데이터를 처리해야 했는데, 개별 프로그래머가 분산 처리 로직을 직접 작성하는 것은 너무 복잡했습니다. 딘과 게마와트는 함수형 프로그래밍의 `map`과 `reduce` 개념에서 영감을 받아, 개발자가 **"각 데이터를 어떻게 처리할지(Map)"**와 **"결과를 어떻게 합칠지(Reduce)"**만 정의하면, 분산·병렬·내결함성은 프레임워크가 알아서 처리하는 모델을 만들었습니다.

이 논문은 이후 하둡(Hadoop) 에코시스템의 탄생으로 이어졌고, 빅데이터 시대의 서막을 열었습니다. 흥미로운 점은, 20년이 지난 지금 **같은 패턴이 LLM 워크플로우**에서 부활했다는 것입니다. 대규모 언어 모델의 API 호출은 본질적으로 "느린 I/O 작업"이므로, 맵-리듀스의 병렬 처리 철학이 완벽하게 들어맞거든요.

### 슈퍼스텝과 Pregel 모델

LangGraph의 내부 실행 엔진 이름이 **Pregel**인 것도 흥미롭습니다. 이 이름은 구글이 2010년에 발표한 대규모 그래프 처리 시스템 "Pregel"에서 따온 것인데요, 이는 다시 유명한 **쾨니히스베르크(Königsberg)의 다리 문제**를 풀었던 프레겔(Pregel) 강의 이름에서 유래했습니다. LangGraph의 **슈퍼스텝(super-step)** 개념 — 한 슈퍼스텝 안에서 여러 노드가 동시에 실행되고, 모두 완료된 뒤 다음 슈퍼스텝으로 넘어가는 방식 — 역시 이 Pregel 모델에서 비롯된 것입니다.

Send로 생성된 병렬 노드들은 모두 **같은 슈퍼스텝** 안에서 실행되며, 전부 완료된 후에야 다음 노드(리듀스 단계)가 시작됩니다. 이것이 맵-리듀스의 **동기 배리어(synchronization barrier)** 역할을 합니다.

### Send의 상태 격리

Send가 전달하는 상태는 메인 그래프의 상태와 **완전히 다른 스키마**를 가질 수 있다는 점이 중요합니다. 앞서 [서브그래프와 모듈화](ch14_03)에서 배운 것처럼 부모-자식 간 상태 변환이 필요할 수 있는데, Send는 이를 더 단순하게 만들어줍니다. Send 객체의 두 번째 인자로 전달하는 딕셔너리가 곧 해당 노드의 입력 상태가 되니까요.

```python
# 메인 상태: {"topics": [...], "results": [...]}
# Send로 전달하는 상태: {"topic": "단일 주제"} — 완전히 다른 스키마!
Send("process", {"topic": "AI 에이전트", "priority": "high"})
```

## 흔한 오해와 팁

> ⚠️ **흔한 오해**: "Send로 보낸 노드의 반환값은 Send에서 정의한 상태(예: `TopicState`)에 기록된다"고 생각하기 쉽습니다. 하지만 실제로 병렬 노드의 **반환값은 메인 그래프의 `OverallState`에 합산**됩니다. 그래서 반환하는 딕셔너리의 키는 반드시 메인 상태의 키와 일치해야 하고, 해당 키에는 리듀서가 설정되어 있어야 합니다.

> 💡 **알고 계셨나요?**: `Send` 클래스는 `langgraph.types`에서 임포트합니다. 일부 예전 문서나 튜토리얼에서 `from langgraph.graph import Send`로 안내하는 경우가 있는데, 이는 최신 버전에서 `ImportError`를 유발합니다. 반드시 `from langgraph.types import Send`를 사용하세요.

> 🔥 **실무 팁**: 병렬 노드에서 **LLM API를 호출**할 때는 `max_concurrency`를 반드시 설정하세요. OpenAI API의 경우 분당 요청 수(RPM) 제한이 있어서, Send로 50개 노드를 동시에 실행하면 `429 Too Many Requests` 에러가 발생할 수 있습니다. 보통 `max_concurrency=3~5` 정도로 시작해서 rate limit에 맞춰 조절하는 것이 좋습니다.

> 🔥 **실무 팁**: 병렬 노드의 결과를 집계할 때 `operator.add` 대신 **커스텀 리듀서**를 쓰면 더 정교한 집계가 가능합니다. 예를 들어 중복 제거, 점수순 정렬, 에러 결과 필터링 등을 리듀서 안에서 처리할 수 있습니다.
>
> ```python
> def merge_unique(existing: list, new: list) -> list:
>     """중복 없이 결과를 합치는 커스텀 리듀서"""
>     seen = {item["id"] for item in existing}
>     return existing + [item for item in new if item["id"] not in seen]
>
> class State(TypedDict):
>     results: Annotated[list[dict], merge_unique]
> ```

## 핵심 정리

| 개념 | 설명 |
|------|------|
| `Send(node, state)` | 조건부 엣지에서 반환하여 동적으로 병렬 노드를 생성하는 클래스 |
| 팬아웃(Fan-out) | 하나의 노드에서 여러 병렬 노드로 작업을 분배하는 패턴 |
| 팬인(Fan-in) | 병렬 노드의 결과를 하나의 상태로 통합하는 패턴 |
| `Annotated[list, operator.add]` | 리스트 합치기 리듀서 — 병렬 결과를 안전하게 집계 |
| `max_concurrency` | 동시 실행 노드 수를 제한하는 설정 |
| 맵-리듀스 | 분배 → 병렬 처리(Map) → 결과 집계(Reduce)의 3단계 패턴 |
| 슈퍼스텝 | Send로 생성된 병렬 노드들이 모두 같은 슈퍼스텝에서 실행됨 |
| 상태 격리 | Send로 전달되는 상태는 메인 그래프와 다른 스키마 가능 |
