# 텍스트 임베딩 이해

> 텍스트를 숫자의 언어로 번역하는 법 — 컴퓨터가 "의미"를 이해하는 첫걸음

## 개요

이 섹션에서는 텍스트 임베딩이 무엇인지, 왜 필요한지, 그리고 LangChain에서 어떻게 사용하는지를 배웁니다. 텍스트를 벡터(숫자 배열)로 변환하면 컴퓨터가 "의미적 유사도"를 계산할 수 있게 되는데요, 이것이 RAG, 시맨틱 검색, 추천 시스템 등 현대 AI 애플리케이션의 핵심 기반 기술입니다.

**선수 지식**: 앞서 Ch6 "문서 로더와 텍스트 분할"에서 배운 문서 처리 파이프라인의 개념. 텍스트를 청크로 분할하는 이유를 이해하고 있어야 합니다.
**학습 목표**:
- 텍스트 임베딩의 개념과 벡터 공간을 직관적으로 설명할 수 있다
- 코사인 유사도와 유클리드 거리의 차이를 이해하고 적절히 선택할 수 있다
- LangChain의 `OpenAIEmbeddings`를 사용하여 텍스트를 벡터로 변환할 수 있다
- 임베딩 차원(dimensions)이 성능과 비용에 미치는 영향을 설명할 수 있다

## 왜 알아야 할까?

컴퓨터는 본질적으로 숫자만 이해합니다. "강아지"와 "puppy"가 비슷한 뜻이라는 걸 사람은 바로 알지만, 컴퓨터에게 이 두 문자열은 완전히 다른 바이트 시퀀스일 뿐이죠. 그렇다면 컴퓨터에게 "의미의 유사함"을 어떻게 가르칠 수 있을까요?

바로 **임베딩(Embedding)**이 그 답입니다. 텍스트를 고차원 벡터 공간의 좌표로 변환하면, 의미가 비슷한 텍스트끼리 가까이 모이게 됩니다. 이것이 가능해지면:

- **시맨틱 검색**: 키워드가 아닌 "의미"로 문서를 찾을 수 있습니다
- **RAG**: 질문과 관련된 문서를 정확히 검색하여 LLM에 전달할 수 있습니다
- **클러스터링/분류**: 비슷한 문서끼리 자동으로 그룹핑할 수 있습니다
- **추천 시스템**: 사용자가 좋아할 만한 콘텐츠를 찾을 수 있습니다

Ch9에서 본격적으로 구축할 RAG 시스템의 핵심 부품이 바로 이 임베딩이거든요. 이번 챕터를 제대로 이해하면, 벡터 스토어와 검색기(Retriever)를 다루는 Ch7~Ch8이 훨씬 수월해질 겁니다.

## 핵심 개념

### 개념 1: 임베딩이란 무엇인가?

> 💡 **비유**: 도서관의 분류 번호를 떠올려 보세요. "파이썬 프로그래밍" 책과 "자바 프로그래밍" 책은 같은 서가(005.13)에 꽂혀 있지만, "프랑스 요리" 책은 전혀 다른 서가(641.5)에 있죠. 도서관 분류 번호가 책의 "주제"를 숫자로 표현하듯, 임베딩은 텍스트의 "의미"를 숫자 벡터로 표현합니다. 다만 분류 번호는 사람이 만든 규칙이고, 임베딩은 AI가 수백만 개의 텍스트를 학습해서 스스로 만들어낸다는 점이 다릅니다.

텍스트 임베딩(Text Embedding)은 텍스트(단어, 문장, 문서)를 **고차원 벡터 공간의 숫자 배열**로 변환한 것입니다. 예를 들어, "오늘 날씨가 좋다"라는 문장은 `[0.023, -0.118, 0.456, ..., 0.089]`처럼 수백~수천 개의 숫자로 표현됩니다.

핵심은 이겁니다: **의미가 비슷한 텍스트는 비슷한 벡터를 갖습니다.** "오늘 날씨가 좋다"와 "today the weather is nice"는 전혀 다른 글자로 이루어져 있지만, 임베딩 공간에서는 매우 가까운 위치에 놓이게 되죠.

```python
# 임베딩의 직관적 이해
# 텍스트가 벡터(숫자 리스트)로 변환되는 과정

text = "오늘 날씨가 좋다"
# 임베딩 모델을 거치면...
embedding = [0.023, -0.118, 0.456, 0.891, -0.234, ...]  # 1536차원 벡터

# 비슷한 의미의 텍스트 → 비슷한 벡터
# "날씨가 화창하다" → [0.025, -0.112, 0.448, 0.887, -0.229, ...]  (매우 유사!)
# "주식이 폭락했다" → [-0.891, 0.567, -0.123, 0.045, 0.678, ...]  (매우 다름!)
```

### 개념 2: 벡터 공간과 의미의 기하학

> 💡 **비유**: 서울 지도에서 강남역과 역삼역은 가깝고, 강남역과 부산역은 멀죠. 임베딩 공간도 마찬가지입니다. "고양이", "강아지", "햄스터"는 한 동네에 모여 있고, "미적분", "선형대수", "확률"은 다른 동네에 모여 있습니다. 심지어 "왕 - 남자 + 여자 = 여왕"처럼 벡터 연산으로 의미 관계를 표현할 수도 있어요!

벡터 공간에서 텍스트의 의미는 **방향**과 **위치**로 표현됩니다. 이 공간에서는 놀라운 특성들이 나타나는데요:

| 특성 | 예시 | 설명 |
|------|------|------|
| 유사도 | "행복" ≈ "기쁨" | 비슷한 의미 → 가까운 위치 |
| 관계 보존 | 서울:한국 = 도쿄:일본 | 관계가 벡터 방향으로 보존됨 |
| 클러스터링 | 과일류끼리, 동물류끼리 | 같은 범주가 자연스럽게 모임 |

### 개념 3: 코사인 유사도와 거리 측정

두 임베딩이 얼마나 비슷한지 어떻게 측정할까요? 가장 널리 쓰이는 방법이 **코사인 유사도(Cosine Similarity)**입니다.

> 💡 **비유**: 두 사람이 나침반을 들고 서 있다고 상상해 보세요. 둘 다 같은 방향(북쪽)을 가리키면 "완전히 같다(유사도 1.0)", 직각으로 다른 방향이면 "관계없다(유사도 0)", 정반대 방향이면 "완전히 반대다(유사도 -1)"입니다. 코사인 유사도는 두 벡터가 **어느 방향을 가리키는지**만 봅니다. 벡터의 길이(크기)는 무시하죠.

$$\text{cosine\_similarity}(\mathbf{A}, \mathbf{B}) = \frac{\mathbf{A} \cdot \mathbf{B}}{||\mathbf{A}|| \times ||\mathbf{B}||}$$

- $\mathbf{A} \cdot \mathbf{B}$: 두 벡터의 내적 (각 차원의 곱을 모두 합한 것)
- $||\mathbf{A}||$: 벡터 A의 크기 (L2 노름)
- 결과 범위: -1 ~ 1 (1에 가까울수록 유사)

이게 의미하는 바는, 두 텍스트의 "의미적 방향"이 얼마나 일치하는지를 숫자 하나로 알려준다는 것입니다.

**유클리드 거리(Euclidean Distance)**도 자주 쓰이는데요, 이건 두 점 사이의 직선 거리를 계산합니다:

$$d(\mathbf{A}, \mathbf{B}) = \sqrt{\sum_{i=1}^{n}(A_i - B_i)^2}$$

| 측정 방법 | 무엇을 보는가 | 범위 | LLM 임베딩에서 | 특징 |
|-----------|---------------|------|-----------------|------|
| 코사인 유사도 | 방향(각도) | -1 ~ 1 | 가장 널리 사용 | 문서 길이에 영향 없음 |
| 유클리드 거리 | 직선 거리 | 0 ~ ∞ | 보조적 사용 | 차원 수에 민감 |
| 내적(Dot Product) | 방향 + 크기 | -∞ ~ ∞ | 정규화된 벡터에서 사용 | 코사인 유사도와 동치(정규화 시) |

```python
import numpy as np

def cosine_similarity(vec_a: list[float], vec_b: list[float]) -> float:
    """두 벡터의 코사인 유사도를 계산합니다."""
    a = np.array(vec_a)
    b = np.array(vec_b)
    # 내적을 각 벡터의 크기(노름)의 곱으로 나눔
    return np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b))

def euclidean_distance(vec_a: list[float], vec_b: list[float]) -> float:
    """두 벡터의 유클리드 거리를 계산합니다."""
    a = np.array(vec_a)
    b = np.array(vec_b)
    return np.linalg.norm(a - b)

# 예시: 3차원 벡터로 직관적 이해
cat = [1.0, 0.9, 0.1]     # "고양이" 임베딩 (가상)
dog = [0.9, 1.0, 0.15]    # "강아지" 임베딩 (가상)
car = [0.1, 0.15, 1.0]    # "자동차" 임베딩 (가상)

print(f"고양이 ↔ 강아지 코사인 유사도: {cosine_similarity(cat, dog):.4f}")  # ~0.99 (매우 유사)
print(f"고양이 ↔ 자동차 코사인 유사도: {cosine_similarity(cat, car):.4f}")  # ~0.30 (매우 다름)
print(f"고양이 ↔ 강아지 유클리드 거리: {euclidean_distance(cat, dog):.4f}")  # ~0.16 (가까움)
print(f"고양이 ↔ 자동차 유클리드 거리: {euclidean_distance(cat, car):.4f}")  # ~1.35 (멀음)
```

### 개념 4: LangChain의 OpenAIEmbeddings

LangChain은 다양한 임베딩 모델을 통합 인터페이스로 제공합니다. 그중 가장 널리 쓰이는 것이 OpenAI의 임베딩 모델이죠.

```python
from langchain_openai import OpenAIEmbeddings
from dotenv import load_dotenv

# .env 파일에서 OPENAI_API_KEY 로드
load_dotenv()

# 임베딩 모델 초기화
embeddings = OpenAIEmbeddings(
    model="text-embedding-3-small"  # 기본 모델 (1536차원)
)

# 단일 텍스트 임베딩 (쿼리용)
query_vector = embeddings.embed_query("LangChain이란 무엇인가요?")
print(f"벡터 차원: {len(query_vector)}")   # 출력: 벡터 차원: 1536
print(f"처음 5개 값: {query_vector[:5]}")  # 출력: [0.023, -0.118, ...]

# 여러 문서를 한 번에 임베딩 (배치 처리)
documents = [
    "LangChain은 LLM 기반 애플리케이션 프레임워크입니다",
    "벡터 스토어는 임베딩을 저장하는 데이터베이스입니다",
    "RAG는 검색 증강 생성 기법입니다",
]
doc_vectors = embeddings.embed_documents(documents)
print(f"문서 수: {len(doc_vectors)}")         # 출력: 문서 수: 3
print(f"각 벡터 차원: {len(doc_vectors[0])}")  # 출력: 각 벡터 차원: 1536
```

여기서 중요한 포인트가 있습니다. LangChain의 `Embeddings` 인터페이스는 두 가지 메서드를 제공하는데요:

| 메서드 | 입력 | 출력 | 용도 |
|--------|------|------|------|
| `embed_query(text)` | 단일 문자열 | 단일 벡터 | 검색 쿼리 임베딩 |
| `embed_documents(texts)` | 문자열 리스트 | 벡터 리스트 | 문서 배치 임베딩 |

왜 굳이 두 개로 나눴을까요? 일부 임베딩 모델은 검색 쿼리와 문서에 서로 다른 전략을 사용하기 때문입니다. OpenAI 모델에서는 둘의 차이가 없지만, 다른 모델(예: Instructor 기반 모델)에서는 쿼리에 "query: " 접두사를, 문서에 "passage: " 접두사를 붙이는 식으로 다르게 처리하거든요.

### 개념 5: 임베딩 차원과 성능 — 마트료시카 임베딩

OpenAI의 `text-embedding-3` 시리즈는 **Matryoshka Representation Learning** 기법으로 학습되었습니다.

> 💡 **비유**: 러시아 인형 마트료시카를 아시죠? 큰 인형 안에 작은 인형이, 그 안에 더 작은 인형이 들어있잖아요. 마트료시카 임베딩도 마찬가지입니다. 3072차원 벡터의 앞쪽 256개 차원만 잘라내도 의미 있는 임베딩이 됩니다. 앞쪽 차원일수록 더 중요한 정보를 담고 있거든요!

| 모델 | 기본 차원 | 축소 가능 범위 | MTEB 점수 | 가격 (100만 토큰) |
|------|-----------|----------------|-----------|-------------------|
| text-embedding-3-small | 1536 | 256 ~ 1536 | 62.3% | $0.02 |
| text-embedding-3-large | 3072 | 256 ~ 3072 | 64.6% | $0.13 |
| text-embedding-ada-002 (레거시) | 1536 | 축소 불가 | 61.0% | $0.10 |

놀랍게도, `text-embedding-3-large`를 256차원으로 축소해도 `text-embedding-ada-002`의 1536차원보다 성능이 좋습니다! 즉, 저장 공간과 검색 속도를 대폭 절약하면서도 품질은 유지할 수 있다는 뜻이죠.

```python
from langchain_openai import OpenAIEmbeddings

# 기본 차원 (3072)
embeddings_full = OpenAIEmbeddings(
    model="text-embedding-3-large"
)

# 차원 축소 (256) — 저장 공간 12배 절약!
embeddings_small = OpenAIEmbeddings(
    model="text-embedding-3-large",
    dimensions=256  # dimensions 파라미터로 차원 제어
)

text = "LangChain으로 RAG 시스템을 구축합니다"

vec_full = embeddings_full.embed_query(text)
vec_small = embeddings_small.embed_query(text)

print(f"전체 차원: {len(vec_full)}")   # 출력: 전체 차원: 3072
print(f"축소 차원: {len(vec_small)}")  # 출력: 축소 차원: 256
```

## 실습: 직접 해보기

실제로 텍스트를 임베딩하고, 유사도를 계산해서 시맨틱 검색을 직접 구현해 봅시다.

```python
"""
텍스트 임베딩 실습: 시맨틱 유사도 검색 구현
필요 패키지: pip install langchain-openai python-dotenv numpy
"""
import numpy as np
from dotenv import load_dotenv
from langchain_openai import OpenAIEmbeddings

# 환경 변수 로드
load_dotenv()

# 임베딩 모델 초기화
embeddings = OpenAIEmbeddings(model="text-embedding-3-small")

# ── 1단계: 문서 임베딩 ──────────────────────────────
# 검색 대상 문서들
documents = [
    "파이썬은 간결하고 읽기 쉬운 프로그래밍 언어입니다",
    "자바스크립트는 웹 브라우저에서 동작하는 스크립트 언어입니다",
    "머신러닝은 데이터에서 패턴을 학습하는 인공지능의 한 분야입니다",
    "딥러닝은 신경망을 사용한 머신러닝의 하위 분야입니다",
    "LangChain은 LLM 애플리케이션을 구축하기 위한 프레임워크입니다",
    "벡터 데이터베이스는 임베딩 벡터를 효율적으로 저장하고 검색합니다",
    "김치찌개는 한국의 대표적인 찌개 요리입니다",
    "축구는 전 세계에서 가장 인기 있는 스포츠입니다",
]

# 모든 문서를 벡터로 변환
doc_vectors = embeddings.embed_documents(documents)
print(f"✅ {len(documents)}개 문서 임베딩 완료 (각 {len(doc_vectors[0])}차원)")


# ── 2단계: 코사인 유사도 함수 ──────────────────────────
def cosine_similarity(a: list[float], b: list[float]) -> float:
    """두 벡터 간의 코사인 유사도를 계산합니다."""
    a_arr, b_arr = np.array(a), np.array(b)
    return float(np.dot(a_arr, b_arr) / (np.linalg.norm(a_arr) * np.linalg.norm(b_arr)))


# ── 3단계: 시맨틱 검색 ──────────────────────────────
def semantic_search(
    query: str,
    documents: list[str],
    doc_vectors: list[list[float]],
    top_k: int = 3,
) -> list[tuple[str, float]]:
    """쿼리와 가장 유사한 문서를 찾아 반환합니다."""
    # 쿼리를 벡터로 변환
    query_vector = embeddings.embed_query(query)

    # 모든 문서와의 유사도 계산
    similarities = [
        (doc, cosine_similarity(query_vector, doc_vec))
        for doc, doc_vec in zip(documents, doc_vectors)
    ]

    # 유사도 내림차순 정렬 후 상위 k개 반환
    similarities.sort(key=lambda x: x[1], reverse=True)
    return similarities[:top_k]


# ── 4단계: 검색 테스트 ──────────────────────────────
queries = [
    "인공지능과 학습 알고리즘",
    "웹 개발 언어",
    "한국 음식",
]

for query in queries:
    print(f"\n🔍 쿼리: '{query}'")
    results = semantic_search(query, documents, doc_vectors)
    for rank, (doc, score) in enumerate(results, 1):
        print(f"  {rank}. [{score:.4f}] {doc}")

# ── 5단계: 문서 간 유사도 히트맵 ──────────────────────
print("\n📊 문서 간 유사도 매트릭스 (상위 5개 문서):")
print(f"{'':>4}", end="")
for i in range(5):
    print(f"  문서{i+1}", end="")
print()

for i in range(5):
    print(f"문서{i+1}", end="")
    for j in range(5):
        sim = cosine_similarity(doc_vectors[i], doc_vectors[j])
        print(f"  {sim:.3f}", end="")
    print()
```

## 더 깊이 알아보기

### Word2Vec — 단어에 좌표를 부여한 혁명

텍스트 임베딩의 역사는 2013년으로 거슬러 올라갑니다. 체코 출신의 연구원 **토마스 미콜로프(Tomáš Mikolov)**가 구글에서 **Word2Vec**을 발표하면서 NLP의 판도가 완전히 바뀌었거든요.

재미있는 건 이 논문의 탄생 과정입니다. 사실 미콜로프는 구글에 입사하기 전, 2012년 체코 브르노 공과대학에서 박사 학위를 받을 때 이미 핵심 아이디어를 갖고 있었습니다. 그런데 딥러닝의 대부 **요슈아 벤지오(Yoshua Bengio)**가 미콜로프에게 단어 유추(word analogy) 실험에 대한 논문을 써보라고 제안했고, 미콜로프는 2012년 크리스마스 연휴를 캘리포니아에서 논문 작성에 바쳤습니다.

그렇게 탄생한 논문 *"Efficient Estimation of Word Representations in Vector Space"*는 ICLR 2013에 제출되었지만 — 놀랍게도 **리뷰어들에 의해 거절**당했습니다! 미콜로프는 포기하지 않고 확장 논문을 작성하여 NeurIPS에 발표했는데, 오늘날 이 논문의 인용 수는 아마 ICLR 2013에 채택된 모든 논문의 인용 수를 합친 것보다 많을 것입니다.

Word2Vec이 보여준 "king - man + woman = queen" 같은 벡터 연산은 전 세계 연구자들을 흥분시켰습니다. 단어가 단순한 기호가 아니라 **의미를 가진 좌표**가 될 수 있다는 것을 처음 대규모로 증명했으니까요. 이후 GloVe(2014), FastText(2016), ELMo(2018), BERT(2018)를 거쳐, 오늘날 OpenAI의 `text-embedding-3` 시리즈까지 이어지는 긴 여정의 출발점이었습니다.

### 마트료시카 표현 학습의 원리

2022년 발표된 *"Matryoshka Representation Learning"* 논문은 임베딩의 효율성에 혁신을 가져왔습니다. 핵심 아이디어는 간단합니다: 학습 과정에서 벡터의 앞쪽 차원이 뒤쪽 차원보다 더 중요한 정보를 담도록 강제하는 것이죠. 마치 마트료시카 인형처럼 큰 벡터 안에 작은 벡터가, 그 안에 더 작은 벡터가 유효한 표현으로 존재하게 됩니다.

OpenAI는 2024년 1월 이 기법을 적용한 `text-embedding-3-small`과 `text-embedding-3-large`를 발표했고, 이를 통해 개발자는 성능과 비용의 트레이드오프를 유연하게 조절할 수 있게 되었습니다.

## 흔한 오해와 팁

> ⚠️ **흔한 오해**: "임베딩 차원이 높을수록 무조건 좋다" — 아닙니다! 차원이 높으면 더 풍부한 표현이 가능하지만, 저장 공간, 검색 속도, 메모리 사용량이 모두 증가합니다. `text-embedding-3-large`를 256차원으로 축소해도 레거시 `ada-002` 1536차원보다 성능이 좋을 수 있습니다. 실무에서는 벤치마크보다 **자신의 데이터로 직접 테스트**하는 것이 가장 정확합니다.

> ⚠️ **흔한 오해**: "코사인 유사도가 0.9면 90% 유사하다" — 코사인 유사도는 퍼센트가 아닙니다! 임베딩 모델마다 유사도 분포가 다릅니다. 어떤 모델은 관련 없는 문서도 0.7 이상을 주고, 어떤 모델은 매우 유사한 문서도 0.6 정도를 줍니다. 절대값보다 **상대적 순위**가 더 중요합니다.

> 💡 **알고 계셨나요?**: OpenAI의 `text-embedding-3` 시리즈는 ℓ₂ 정규화(normalized)된 벡터를 반환합니다. 즉, 모든 벡터의 크기가 1입니다. 이 경우 코사인 유사도와 내적(dot product)이 수학적으로 동일해지므로, 내적만으로도 유사도를 계산할 수 있어 성능상 이점이 있습니다.

> 🔥 **실무 팁**: `embed_documents()`는 내부적으로 배치 처리를 수행합니다. 문서 1000개를 임베딩할 때 `embed_query()`를 1000번 호출하지 말고, 반드시 `embed_documents()`를 사용하세요. API 호출 횟수가 크게 줄어들어 비용과 시간을 절약할 수 있습니다. LangChain의 OpenAIEmbeddings는 기본적으로 한 번에 최대 2048개의 텍스트를 배치로 처리합니다.

> 🔥 **실무 팁**: API 키를 코드에 절대 하드코딩하지 마세요! 반드시 `.env` 파일과 `python-dotenv`를 사용하고, `.gitignore`에 `.env`를 추가하세요. 이것은 보안의 기본 중의 기본입니다.

## 핵심 정리

| 개념 | 설명 |
|------|------|
| 텍스트 임베딩 | 텍스트를 고차원 벡터(숫자 배열)로 변환하여 의미를 수치적으로 표현하는 기법 |
| 벡터 공간 | 임베딩이 존재하는 다차원 좌표계. 의미가 유사한 텍스트끼리 가까이 위치함 |
| 코사인 유사도 | 두 벡터의 방향(각도) 기반 유사도 측정. -1~1 범위, 텍스트 임베딩에서 가장 널리 사용 |
| 유클리드 거리 | 두 벡터 사이의 직선 거리. 코사인 유사도의 보조 지표로 활용 |
| `embed_query()` | 단일 텍스트를 벡터로 변환. 검색 쿼리에 사용 |
| `embed_documents()` | 여러 텍스트를 배치로 벡터 변환. 문서 인덱싱에 사용 |
| `dimensions` 파라미터 | text-embedding-3 시리즈에서 출력 차원을 조절. 마트료시카 학습 덕분에 축소해도 품질 유지 |
| text-embedding-3-small | 1536차원 기본, 가성비 최고의 범용 임베딩 모델 |
| text-embedding-3-large | 3072차원 기본, 최고 성능의 임베딩 모델. 차원 축소 시 더 큰 이점 |
