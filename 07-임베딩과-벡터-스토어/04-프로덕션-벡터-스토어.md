# 프로덕션 벡터 스토어

> 클라우드 벡터 데이터베이스(Pinecone, Weaviate, Qdrant)를 LangChain과 연동하여 프로덕션 환경에서 안정적으로 벡터 검색을 운영하는 방법을 배웁니다.

## 개요

이전 세션에서 FAISS와 Chroma를 사용해 로컬 환경에서 벡터 스토어를 구축해봤는데요. 실제 서비스에서는 수백만~수십억 개의 벡터를 안정적으로 저장하고 검색해야 합니다. 이번 세션에서는 대표적인 클라우드 벡터 데이터베이스 세 가지 — **Pinecone**, **Weaviate**, **Qdrant** — 를 LangChain과 연동하는 방법을 다룹니다.

**선수 지식**: [7.1 텍스트 임베딩 이해](./01-텍스트-임베딩-이해.md)에서 배운 임베딩 개념, [7.3 벡터 스토어 구축](./03-벡터-스토어-구축---faiss와-chroma.md)에서 다룬 `from_documents`, `similarity_search` 패턴
**학습 목표**:
- Pinecone Serverless 인덱스를 생성하고 LangChain에서 벡터 검색을 수행할 수 있다
- Weaviate Cloud에 연결하여 하이브리드 검색을 구현할 수 있다
- Qdrant를 로컬(Docker)과 클라우드 환경에서 모두 활용할 수 있다
- 프로젝트 요구사항에 맞는 클라우드 벡터 DB를 선택할 수 있다

## 왜 알아야 할까?

앞서 [7.3 벡터 스토어 구축](./03-벡터-스토어-구축---faiss와-chroma.md)에서 FAISS와 Chroma로 벡터 스토어를 구축해봤죠. 로컬 개발과 프로토타이핑에는 충분했지만, 실제 프로덕션 환경에서는 다른 차원의 요구사항이 생깁니다.

서비스에 100만 명의 사용자가 동시에 질문을 던진다고 상상해보세요. FAISS 파일 하나로는 감당할 수 없겠죠? 프로덕션에서는 **고가용성**(서버가 죽어도 데이터가 사라지지 않아야), **수평 확장**(사용자가 늘어나면 서버를 추가할 수 있어야), **실시간 업데이트**(새 문서가 들어오면 바로 검색 가능해야), **접근 제어**(API 키 관리, 팀별 권한) 같은 요구사항이 필수입니다.

클라우드 벡터 데이터베이스는 이런 문제를 해결해주는 **"관리형 인프라"**입니다. 여러분은 인덱스 설계와 검색 로직에만 집중하면 되고, 서버 관리·백업·스케일링은 클라우드가 알아서 처리해주거든요. LangChain의 통합 인터페이스 덕분에, 어떤 벡터 DB를 쓰든 `similarity_search()` 한 줄이면 검색이 가능합니다.

## 핵심 개념

### 개념 1: Pinecone — 완전 관리형 벡터 데이터베이스

> 💡 **비유**: Pinecone은 벡터 세계의 **AWS S3** 같은 존재입니다. S3에 파일을 올리면 알아서 저장·복제·백업해주듯, Pinecone에 벡터를 올리면 인덱싱·검색·스케일링을 모두 알아서 처리해줍니다. 서버를 직접 관리할 필요가 전혀 없죠.

Pinecone은 **완전 관리형(Fully Managed)** 벡터 데이터베이스로, 설치나 인프라 구성 없이 API 키 하나로 바로 사용할 수 있습니다. 2024년에 출시된 **Serverless** 아키텍처 덕분에, 사용한 만큼만 비용을 내는 모델로 전환되었습니다.

**설치 및 기본 설정:**

```python
# 필요한 패키지 설치
# pip install langchain-pinecone langchain-openai pinecone

import os
from dotenv import load_dotenv

load_dotenv()

# 환경 변수 설정 (.env 파일에 저장)
# PINECONE_API_KEY=your-pinecone-api-key
# OPENAI_API_KEY=your-openai-api-key
```

**Pinecone 인덱스 생성 및 LangChain 연동:**

```python
from pinecone import Pinecone, ServerlessSpec
from langchain_pinecone import PineconeVectorStore
from langchain_openai import OpenAIEmbeddings
from langchain_core.documents import Document

# Pinecone 클라이언트 초기화
pc = Pinecone(api_key=os.environ["PINECONE_API_KEY"])

INDEX_NAME = "langchain-demo"

# Serverless 인덱스 생성 (이미 존재하면 건너뜀)
if INDEX_NAME not in pc.list_indexes().names():
    pc.create_index(
        name=INDEX_NAME,
        dimension=1536,          # text-embedding-3-small 차원
        metric="cosine",         # 코사인 유사도 사용
        spec=ServerlessSpec(
            cloud="aws",         # AWS 클라우드
            region="us-east-1"   # 리전 선택
        )
    )

# 임베딩 모델 준비
embeddings = OpenAIEmbeddings(model="text-embedding-3-small")

# 문서 준비
documents = [
    Document(page_content="LangChain은 LLM 애플리케이션 개발 프레임워크입니다.", 
             metadata={"source": "docs", "topic": "langchain"}),
    Document(page_content="RAG는 검색 증강 생성으로, 외부 지식을 활용합니다.", 
             metadata={"source": "docs", "topic": "rag"}),
    Document(page_content="벡터 임베딩은 텍스트의 의미를 수치로 표현합니다.", 
             metadata={"source": "tutorial", "topic": "embedding"}),
]

# 문서를 Pinecone에 업로드하고 벡터 스토어 생성
vector_store = PineconeVectorStore.from_documents(
    documents=documents,
    embedding=embeddings,
    index_name=INDEX_NAME,
    namespace="my-namespace"     # 네임스페이스로 데이터 분리
)

# 유사도 검색
results = vector_store.similarity_search(
    "LLM 앱 개발에 좋은 프레임워크는?",
    k=2
)

for doc in results:
    print(f"[{doc.metadata['topic']}] {doc.page_content}")
# 출력:
# [langchain] LangChain은 LLM 애플리케이션 개발 프레임워크입니다.
# [rag] RAG는 검색 증강 생성으로, 외부 지식을 활용합니다.
```

**기존 인덱스에 연결:**

이미 데이터가 있는 Pinecone 인덱스에 연결할 때는 `from_existing_index`를 사용합니다.

```python
# 기존 인덱스에 연결 (데이터 재업로드 없이)
vector_store = PineconeVectorStore.from_existing_index(
    index_name=INDEX_NAME,
    embedding=embeddings,
    namespace="my-namespace"
)

# 메타데이터 필터링과 함께 검색
results = vector_store.similarity_search(
    "임베딩이란?",
    k=3,
    filter={"topic": "embedding"}  # topic이 "embedding"인 것만 검색
)
```

> ⚠️ **흔한 오해**: Pinecone의 `dimension` 파라미터는 인덱스 생성 시 한 번만 설정할 수 있고, 이후에는 변경이 불가능합니다. 임베딩 모델을 바꾸면 차원이 달라질 수 있으므로, 인덱스를 새로 만들어야 합니다. `text-embedding-3-small`은 1536, `text-embedding-3-large`는 3072 차원이에요.

### 개념 2: Weaviate — 시맨틱 검색 + 하이브리드 검색

> 💡 **비유**: Weaviate는 벡터 세계의 **스위스 아미 나이프**입니다. 벡터 검색만 하는 것이 아니라, 키워드 검색과 벡터 검색을 동시에 할 수 있는 **하이브리드 검색**을 기본으로 제공합니다. 마치 도서관에서 목차(키워드)와 내용(의미)을 동시에 검색하는 것과 같죠.

Weaviate는 오픈소스 벡터 데이터베이스로, **Weaviate Cloud**를 통해 관리형 서비스를 사용하거나, Docker로 직접 호스팅할 수 있습니다. 가장 큰 특징은 **하이브리드 검색**(벡터 + 키워드)을 기본 지원한다는 점이에요.

**설치 및 연결:**

```python
# 필요한 패키지 설치
# pip install langchain-weaviate weaviate-client langchain-openai

import weaviate
from weaviate.classes.init import Auth
from langchain_weaviate import WeaviateVectorStore
from langchain_openai import OpenAIEmbeddings
from langchain_core.documents import Document

# Weaviate Cloud에 연결
weaviate_client = weaviate.connect_to_weaviate_cloud(
    cluster_url=os.environ["WEAVIATE_URL"],       # Weaviate Cloud 클러스터 URL
    auth_credentials=Auth.api_key(
        os.environ["WEAVIATE_API_KEY"]             # Weaviate API 키
    ),
    headers={
        "X-OpenAI-Api-Key": os.environ["OPENAI_API_KEY"]  # 임베딩용
    }
)

# 또는 로컬 Docker 인스턴스에 연결
# docker run -d -p 8080:8080 -p 50051:50051 semitechnologies/weaviate
# weaviate_client = weaviate.connect_to_local()
```

**벡터 스토어 생성 및 검색:**

```python
embeddings = OpenAIEmbeddings(model="text-embedding-3-small")

documents = [
    Document(page_content="Python은 데이터 과학에 널리 쓰이는 프로그래밍 언어입니다.",
             metadata={"category": "programming"}),
    Document(page_content="TensorFlow와 PyTorch는 대표적인 딥러닝 프레임워크입니다.",
             metadata={"category": "ml"}),
    Document(page_content="FastAPI는 Python으로 REST API를 빠르게 만들 수 있는 웹 프레임워크입니다.",
             metadata={"category": "web"}),
]

# 벡터 스토어 생성
vector_store = WeaviateVectorStore.from_documents(
    documents=documents,
    embedding=embeddings,
    client=weaviate_client,
)

# 기본 유사도 검색 (Weaviate는 하이브리드 검색이 기본)
results = vector_store.similarity_search(
    "머신러닝에 쓰이는 도구",
    k=2
)

for doc in results:
    print(f"[{doc.metadata['category']}] {doc.page_content}")
# 출력:
# [ml] TensorFlow와 PyTorch는 대표적인 딥러닝 프레임워크입니다.
# [programming] Python은 데이터 과학에 널리 쓰이는 프로그래밍 언어입니다.

# 사용 후 반드시 연결 종료
weaviate_client.close()
```

Weaviate의 `similarity_search`는 내부적으로 하이브리드 검색을 사용합니다. `alpha` 파라미터로 벡터 검색과 키워드 검색의 가중치를 조절할 수 있는데요, `alpha=1`이면 순수 벡터 검색, `alpha=0`이면 순수 키워드 검색이 됩니다.

> 🔥 **실무 팁**: Weaviate 클라이언트는 **컨텍스트 매니저**로 사용하는 것이 안전합니다. `with weaviate.connect_to_weaviate_cloud(...) as client:` 패턴을 쓰면 연결 종료를 깜빡할 일이 없어요.

### 개념 3: Qdrant — 고성능 Rust 기반 벡터 엔진

> 💡 **비유**: Qdrant는 벡터 세계의 **포뮬러1 레이싱카**입니다. Rust로 만들어져서 극한의 성능을 뽑아내면서도, **메타데이터 필터링**이 매우 강력합니다. "빨간색이면서 가격이 1만 원 이하인 제품 중 의미적으로 유사한 것"처럼 복잡한 조건을 빠르게 처리할 수 있죠.

Qdrant(쿼드란트)는 Rust로 작성된 고성능 벡터 검색 엔진으로, **로컬 모드**(메모리/디스크), **Docker 셀프 호스팅**, **Qdrant Cloud** 세 가지 방식으로 사용할 수 있습니다. 특히 **고급 필터링**과 **하이브리드 검색**(밀집 + 희소 벡터)을 네이티브로 지원합니다.

**설치 및 기본 사용:**

```python
# 필요한 패키지 설치
# pip install langchain-qdrant langchain-openai

from langchain_qdrant import QdrantVectorStore
from langchain_openai import OpenAIEmbeddings
from langchain_core.documents import Document

embeddings = OpenAIEmbeddings(model="text-embedding-3-small")

documents = [
    Document(page_content="LCEL은 파이프 연산자로 체인을 구성하는 LangChain의 핵심 문법입니다.",
             metadata={"chapter": 5, "difficulty": "중급"}),
    Document(page_content="에이전트는 LLM이 도구를 자율적으로 사용하는 시스템입니다.",
             metadata={"chapter": 12, "difficulty": "고급"}),
    Document(page_content="프롬프트 템플릿으로 재사용 가능한 프롬프트를 만들 수 있습니다.",
             metadata={"chapter": 3, "difficulty": "초급"}),
]
```

**Qdrant Cloud 연결:**

```python
# Qdrant Cloud에 연결하여 벡터 스토어 생성
vector_store = QdrantVectorStore.from_documents(
    documents=documents,
    embedding=embeddings,
    url=os.environ["QDRANT_URL"],         # Qdrant Cloud URL
    api_key=os.environ["QDRANT_API_KEY"], # Qdrant API 키
    collection_name="langchain-course",
)

# 유사도 검색
results = vector_store.similarity_search("체인 구성 방법", k=2)

for doc in results:
    print(f"[Ch.{doc.metadata['chapter']}] {doc.page_content}")
# 출력:
# [Ch.5] LCEL은 파이프 연산자로 체인을 구성하는 LangChain의 핵심 문법입니다.
# [Ch.3] 프롬프트 템플릿으로 재사용 가능한 프롬프트를 만들 수 있습니다.
```

**로컬 Docker 모드:**

```python
# Docker로 Qdrant 실행:
# docker run -d -p 6333:6333 -p 6334:6334 qdrant/qdrant

# 로컬 Qdrant에 연결
vector_store = QdrantVectorStore.from_documents(
    documents=documents,
    embedding=embeddings,
    url="http://localhost:6333",       # 로컬 Docker 주소
    collection_name="langchain-local",
)
```

**인메모리 모드 (테스트/프로토타이핑용):**

```python
# 인메모리 모드 — 설치 없이 바로 사용 (데이터는 프로세스 종료 시 소멸)
vector_store = QdrantVectorStore.from_documents(
    documents=documents,
    embedding=embeddings,
    location=":memory:",              # 메모리 모드
    collection_name="test-collection",
)
```

Qdrant의 가장 큰 장점은 **배포 방식의 유연성**이에요. 개발할 때는 인메모리 모드로 빠르게 테스트하고, 스테이징에서는 Docker로 확인한 뒤, 프로덕션에서는 Qdrant Cloud로 전환하는 흐름이 자연스럽습니다. 코드 변경은 URL과 API 키 부분만 바꾸면 되거든요.

### 개념 4: 클라우드 벡터 DB 비교와 선택 기준

세 가지 벡터 DB의 특징을 한눈에 비교해봅시다.

| 항목 | Pinecone | Weaviate | Qdrant |
|------|----------|----------|--------|
| **배포 방식** | 완전 관리형만 | 클라우드 / Docker / 임베디드 | 클라우드 / Docker / 인메모리 |
| **작성 언어** | — (비공개) | Go | Rust |
| **오픈소스** | ❌ | ✅ | ✅ |
| **하이브리드 검색** | 별도 설정 필요 | 기본 지원 | 기본 지원 |
| **메타데이터 필터링** | 기본 지원 | 기본 지원 | 고급 필터 지원 |
| **무료 티어** | Starter 플랜 (제한적) | 무료 샌드박스 | 1GB RAM 무료 |
| **강점** | 가장 쉬운 시작, 엔터프라이즈 | 하이브리드 검색, 지식 그래프 | 성능, 필터링, 배포 유연성 |
| **LangChain 패키지** | `langchain-pinecone` | `langchain-weaviate` | `langchain-qdrant` |

**어떤 걸 선택해야 할까요?**

- **"인프라는 전혀 신경 쓰고 싶지 않다"** → **Pinecone**: API 키 하나면 끝. 가장 빠르게 프로덕션에 올릴 수 있습니다.
- **"키워드 + 벡터 검색을 동시에 써야 한다"** → **Weaviate**: 하이브리드 검색이 기본 내장되어 있습니다.
- **"셀프 호스팅으로 비용을 줄이고 싶다"** → **Qdrant**: 오픈소스에 Docker 한 줄이면 실행 가능하고, 성능이 뛰어납니다.
- **"데이터 주권이 중요하다 (해외 반출 불가)"** → **Weaviate** 또는 **Qdrant**: 셀프 호스팅이 가능합니다.

### 개념 5: 인덱스 관리와 업데이트 전략

프로덕션에서 벡터 스토어를 운영할 때 가장 중요한 것은 **데이터를 어떻게 추가·수정·삭제할 것인가**입니다.

> 💡 **비유**: 벡터 스토어 관리는 **도서관 사서의 일**과 비슷합니다. 새 책이 들어오면 분류해서 서가에 꽂고(추가), 개정판이 나오면 옛 판을 교체하고(업데이트), 절판된 책은 서가에서 빼야(삭제) 합니다. 이 작업을 매끄럽게 하려면 체계적인 전략이 필요하죠.

**증분 업데이트 패턴:**

```python
from langchain_core.documents import Document

# 새 문서 추가 — add_documents로 기존 인덱스에 문서를 추가
new_docs = [
    Document(
        page_content="LangGraph는 상태 기반 그래프로 에이전트를 구축합니다.",
        metadata={"source": "ch13", "version": "2.0", "updated_at": "2026-03-01"}
    )
]

# Pinecone, Weaviate, Qdrant 모두 동일한 인터페이스
vector_store.add_documents(new_docs)
```

**문서 ID 기반 관리:**

문서를 업데이트하거나 삭제하려면 **고유 ID**가 필수입니다.

```python
import hashlib

def generate_doc_id(content: str, source: str) -> str:
    """문서 내용과 출처로 고유 ID 생성"""
    raw = f"{source}:{content}"
    return hashlib.sha256(raw.encode()).hexdigest()[:16]

# ID를 지정하여 문서 추가
docs_with_ids = [
    Document(page_content="LCEL 체인은 | 연산자로 구성합니다.",
             metadata={"source": "ch05_s01"})
]

ids = [generate_doc_id(doc.page_content, doc.metadata["source"]) 
       for doc in docs_with_ids]

# ID를 명시적으로 지정
vector_store.add_documents(docs_with_ids, ids=ids)

# 같은 ID로 다시 추가하면 기존 문서를 덮어씁니다 (upsert)
updated_docs = [
    Document(page_content="LCEL 체인은 | 연산자로 구성하며, RunnableSequence를 생성합니다.",
             metadata={"source": "ch05_s01", "version": "2.0"})
]

vector_store.add_documents(updated_docs, ids=ids)
```

**네임스페이스/컬렉션 전략:**

대규모 시스템에서는 데이터를 논리적으로 분리하는 전략이 중요합니다.

```python
# Pinecone: 네임스페이스로 데이터 분리
store_v1 = PineconeVectorStore.from_existing_index(
    index_name="my-index",
    embedding=embeddings,
    namespace="production-v1"  # 현재 운영 버전
)

store_v2 = PineconeVectorStore.from_existing_index(
    index_name="my-index",
    embedding=embeddings,
    namespace="production-v2"  # 새 버전 준비
)

# Qdrant: 컬렉션으로 데이터 분리
store_v1 = QdrantVectorStore.from_existing_collection(
    embedding=embeddings,
    url=os.environ["QDRANT_URL"],
    collection_name="docs-v1",  # 현재 운영 버전
)
```

이렇게 버전별 네임스페이스를 분리하면, 새 데이터를 준비한 뒤 한 번에 전환하는 **블루-그린 배포**가 가능합니다.

## 실습: 직접 해보기

아래 코드는 **Qdrant 인메모리 모드**를 사용하여 별도 인프라 설치 없이 바로 실행할 수 있습니다. 문서를 로드하고, 벡터 스토어를 구축한 뒤, 다양한 검색을 수행하는 전체 워크플로우입니다.

```python
"""
프로덕션 벡터 스토어 실습
- Qdrant 인메모리 모드로 벡터 스토어 구축 및 검색
- 실행: pip install langchain-qdrant langchain-openai python-dotenv
"""

import os
from dotenv import load_dotenv
from langchain_core.documents import Document
from langchain_openai import OpenAIEmbeddings
from langchain_qdrant import QdrantVectorStore

load_dotenv()

# ── 1단계: 임베딩 모델 준비 ──
embeddings = OpenAIEmbeddings(model="text-embedding-3-small")

# ── 2단계: 샘플 문서 준비 (LangChain 강의 내용) ──
documents = [
    Document(
        page_content="LangChain은 LLM 기반 애플리케이션 개발을 위한 프레임워크입니다. "
                     "모델 연동, 데이터 처리, 에이전트 구축을 지원합니다.",
        metadata={"chapter": 1, "topic": "langchain", "difficulty": "초급"}
    ),
    Document(
        page_content="LCEL은 파이프 연산자(|)를 사용하여 컴포넌트를 조합하는 선언적 문법입니다. "
                     "RunnableSequence, RunnableParallel 등을 지원합니다.",
        metadata={"chapter": 5, "topic": "lcel", "difficulty": "중급"}
    ),
    Document(
        page_content="RAG는 외부 지식 소스에서 관련 정보를 검색하여 "
                     "LLM 응답의 정확성과 최신성을 향상시키는 기법입니다.",
        metadata={"chapter": 9, "topic": "rag", "difficulty": "중급"}
    ),
    Document(
        page_content="에이전트는 LLM이 도구를 선택적으로 사용하여 "
                     "복잡한 작업을 자율적으로 수행하는 시스템입니다.",
        metadata={"chapter": 12, "topic": "agent", "difficulty": "고급"}
    ),
    Document(
        page_content="LangGraph는 상태 기반 그래프를 사용하여 "
                     "복잡한 에이전트 워크플로우를 구축하는 라이브러리입니다.",
        metadata={"chapter": 13, "topic": "langgraph", "difficulty": "고급"}
    ),
    Document(
        page_content="벡터 임베딩은 텍스트의 의미를 고차원 벡터로 변환하여 "
                     "유사도 검색을 가능하게 합니다.",
        metadata={"chapter": 7, "topic": "embedding", "difficulty": "중급"}
    ),
]

# ── 3단계: Qdrant 인메모리 벡터 스토어 구축 ──
vector_store = QdrantVectorStore.from_documents(
    documents=documents,
    embedding=embeddings,
    location=":memory:",                    # 인메모리 모드 (설치 불필요)
    collection_name="langchain-course",
)

print("=== 벡터 스토어 구축 완료 ===\n")

# ── 4단계: 기본 유사도 검색 ──
print("--- 기본 유사도 검색 ---")
query = "LLM 앱을 만들려면 어떤 프레임워크를 써야 할까?"
results = vector_store.similarity_search(query, k=3)

for i, doc in enumerate(results, 1):
    print(f"{i}. [Ch.{doc.metadata['chapter']}] {doc.page_content[:50]}...")
# 출력:
# 1. [Ch.1] LangChain은 LLM 기반 애플리케이션 개발을 위한 프레임워크입니다...
# 2. [Ch.5] LCEL은 파이프 연산자(|)를 사용하여 컴포넌트를 조합하는 선언적...
# 3. [Ch.9] RAG는 외부 지식 소스에서 관련 정보를 검색하여 LLM 응답의 정확...

# ── 5단계: 점수와 함께 검색 ──
print("\n--- 유사도 점수 포함 검색 ---")
results_with_scores = vector_store.similarity_search_with_score(query, k=3)

for doc, score in results_with_scores:
    print(f"  점수: {score:.4f} | [Ch.{doc.metadata['chapter']}] {doc.page_content[:40]}...")
# 출력 (점수가 높을수록 유사):
# 점수: 0.8234 | [Ch.1] LangChain은 LLM 기반 애플리케이션 개발을 ...
# 점수: 0.7891 | [Ch.5] LCEL은 파이프 연산자(|)를 사용하여 컴포넌트...
# 점수: 0.7456 | [Ch.9] RAG는 외부 지식 소스에서 관련 정보를 검색하...

# ── 6단계: 증분 업데이트 ──
print("\n--- 증분 업데이트 ---")
new_doc = Document(
    page_content="LangServe는 LangChain 체인을 REST API로 배포하는 라이브러리입니다.",
    metadata={"chapter": 17, "topic": "langserve", "difficulty": "중급"}
)

vector_store.add_documents([new_doc])
print("새 문서 1건 추가 완료")

# 추가된 문서가 검색에 반영되는지 확인
results = vector_store.similarity_search("API 배포 방법", k=1)
print(f"검색 결과: {results[0].page_content[:50]}...")
# 출력: 검색 결과: LangServe는 LangChain 체인을 REST API로 배포하는...

# ── 7단계: Retriever 변환 (RAG 파이프라인 연결 준비) ──
print("\n--- Retriever 변환 ---")
retriever = vector_store.as_retriever(
    search_type="similarity",     # 유사도 검색
    search_kwargs={"k": 2}        # 상위 2개 반환
)

# Retriever로 검색 (invoke 사용)
docs = retriever.invoke("자율적으로 작업하는 AI 시스템")
for doc in docs:
    print(f"  [Ch.{doc.metadata['chapter']}] {doc.page_content[:50]}...")
# 출력:
# [Ch.12] 에이전트는 LLM이 도구를 선택적으로 사용하여 복잡한 작업을...
# [Ch.13] LangGraph는 상태 기반 그래프를 사용하여 복잡한 에이전트 워크...

print("\n=== 실습 완료 ===")
```

## 더 깊이 알아보기

### 벡터 데이터베이스의 탄생 — "검색"의 패러다임 전환

벡터 데이터베이스라는 분야는 사실 갑자기 등장한 것이 아닙니다. 그 뿌리는 2000년대 초반 **ANN(Approximate Nearest Neighbor)** 검색 알고리즘 연구에 있습니다. 하지만 이것이 독립적인 데이터베이스 카테고리로 자리잡은 것은 2019년 이후입니다.

**Pinecone** 창업자 **에도 리버티(Edo Liberty)**는 AWS에서 Amazon SageMaker를 이끌던 시절, 기업들이 ML 모델을 학습시키는 것보다 **대규모 벡터 데이터를 실시간으로 다루는 것**을 훨씬 더 어려워한다는 사실을 발견했습니다. 당연히 누군가 이 문제를 해결하는 제품을 만들었을 거라고 생각했지만, 놀랍게도 그런 제품은 존재하지 않았죠. 그래서 2019년에 Pinecone을 창업하며, "벡터 데이터베이스"라는 새로운 카테고리를 개척했습니다.

**Weaviate**의 이야기는 더 흥미롭습니다. 공동 창업자 **밥 판 라위트(Bob van Luijt)**는 원래 네덜란드에서 **재즈와 전자 작곡**을 전공한 음악가였습니다. 버클리 음대까지 다녔죠! 2015년 소프트웨어 컨설팅 회사를 운영하던 그는 초기 언어 모델이 생성하는 벡터 임베딩에 매료되어, 2016년에 의미 기반 데이터 검색 시스템인 Weaviate를 오픈소스로 시작했습니다. AI가 유행하기 한참 전에 시작한 셈이죠.

**Qdrant**는 2021년 베를린에서 **안드레이 바스네초프(Andrey Vasnetsov)**와 **안드레 자야르니(André Zayarni)**가 공동 창업했습니다. 안드레이는 러시아 바우만 공대에서 정보기술을 전공한 뒤, 틴코프 은행에서 ML 엔지니어로 일하며 벡터 검색의 필요성을 절감했습니다. GitHub에 첫 버전을 올렸는데, 예상보다 훨씬 많은 개발자들이 관심을 보여 정식 회사로 전환하게 되었죠. **Rust**로 작성한 덕에 성능 면에서 독보적인 위치를 차지하고 있습니다.

> 💡 **알고 계셨나요?**: "Qdrant"라는 이름은 "Quadrant(사분면)"에서 따왔습니다. 벡터 공간의 좌표 체계를 떠올리게 하는 이름이죠. 한편 "Weaviate"는 "Weave(엮다)"에서 유래했는데, 데이터를 의미적으로 엮어서 검색한다는 철학을 담고 있습니다.

## 흔한 오해와 팁

> ⚠️ **흔한 오해**: "클라우드 벡터 DB는 비싸서 스타트업이 쓰기 어렵다"는 오해가 있습니다. 실제로 Pinecone은 Serverless 모델 도입으로 사용량이 적을 때는 월 몇 달러 수준이고, Qdrant와 Weaviate는 무료 티어를 제공합니다. 초기 단계에서는 로컬 FAISS/Chroma로 시작하고, 사용자가 늘면 클라우드로 전환하는 것이 일반적인 패턴입니다.

> 💡 **알고 계셨나요?**: LangChain의 벡터 스토어 인터페이스(`VectorStore` 추상 클래스)는 모든 벡터 DB에 대해 `similarity_search()`, `add_documents()`, `as_retriever()` 같은 동일한 메서드를 제공합니다. 덕분에 Chroma에서 Pinecone으로 전환할 때 비즈니스 로직 코드는 거의 바꿀 필요가 없습니다. 인터페이스만 바꾸면 되죠 — 바로 이것이 LangChain을 사용하는 큰 이점 중 하나입니다.

> 🔥 **실무 팁**: 프로덕션에서 벡터 스토어를 운영할 때 가장 중요한 것은 **임베딩 모델과 인덱스의 라이프사이클 관리**입니다. 임베딩 모델을 업그레이드하면 기존 벡터와 새 벡터의 차원이나 의미 공간이 달라지므로, **전체 재인덱싱**이 필요합니다. 네임스페이스나 컬렉션을 버전별로 분리하여, 새 임베딩으로 전체 데이터를 재처리한 뒤 한 번에 전환하는 블루-그린 전략을 사용하세요.

> 🔥 **실무 팁**: 대량의 문서를 업로드할 때는 **배치 처리**가 필수입니다. `add_documents()`에 수만 건을 한 번에 넣으면 타임아웃이 발생할 수 있으므로, 500~1000건씩 나누어 처리하세요. Pinecone의 경우 한 번에 최대 1000건(2MB)까지 upsert가 가능합니다.

## 핵심 정리

| 개념 | 설명 |
|------|------|
| Pinecone | 완전 관리형 벡터 DB. API 키만으로 즉시 사용 가능. Serverless로 비용 효율적 |
| Weaviate | 오픈소스 벡터 DB. 하이브리드 검색(벡터+키워드) 기본 지원. 클라우드/셀프호스팅 |
| Qdrant | Rust 기반 고성능 벡터 DB. 강력한 필터링. 인메모리/Docker/클라우드 모두 지원 |
| `PineconeVectorStore` | `langchain-pinecone` 패키지의 벡터 스토어 클래스 |
| `WeaviateVectorStore` | `langchain-weaviate` 패키지의 벡터 스토어 클래스 |
| `QdrantVectorStore` | `langchain-qdrant` 패키지의 벡터 스토어 클래스 |
| `from_documents()` | 문서 리스트로 벡터 스토어를 생성하는 공통 클래스 메서드 |
| `add_documents()` | 기존 벡터 스토어에 새 문서를 추가하는 증분 업데이트 메서드 |
| 네임스페이스/컬렉션 | 인덱스 내에서 데이터를 논리적으로 분리하는 단위 |
| 블루-그린 배포 | 새 버전을 별도 공간에 준비한 뒤 한 번에 전환하는 운영 전략 |
