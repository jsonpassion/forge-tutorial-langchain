# 벡터 검색 최적화

> 벡터 스토어의 검색 품질과 효율을 극대화하는 MMR, 메타데이터 필터링, 임베딩 캐싱 기법을 마스터하고, 하이브리드 검색의 개념을 이해합니다.

## 개요

이 섹션에서는 벡터 스토어를 "잘 만드는 것"을 넘어 "잘 검색하는 것"에 초점을 맞춥니다. 앞서 [7.3: 벡터 스토어 구축 - FAISS와 Chroma](ch07/session_03.md)와 [7.4: 프로덕션 벡터 스토어](ch07/session_04.md)에서 벡터 스토어를 구축하고 프로덕션 환경에 배포하는 방법을 배웠다면, 이번에는 그 위에서 **검색 품질**과 **운영 효율**을 끌어올리는 고급 기법들을 다룹니다.

**선수 지식**: 벡터 스토어 기본 구축(FAISS, Chroma), `similarity_search`, `as_retriever`, 코사인 유사도(cosine similarity) 개념
**학습 목표**:
- MMR(Maximal Marginal Relevance) 검색으로 관련성과 다양성을 동시에 확보할 수 있다
- 메타데이터 필터링으로 검색 범위를 정밀하게 제한할 수 있다
- 하이브리드 검색(BM25 + 벡터)의 개념과 필요성을 이해할 수 있다
- `CacheBackedEmbeddings`로 임베딩 비용과 지연 시간을 줄일 수 있다

## 왜 알아야 할까?

벡터 스토어를 구축한 뒤 `similarity_search`만 호출하면 끝일까요? 실전에서는 그렇지 않습니다.

"인공지능의 역사"를 검색했더니 비슷비슷한 문단 5개가 나온 적 있나요? 내용이 거의 겹치는데 다섯 개나 가져오면 LLM의 소중한 컨텍스트 윈도우만 낭비하는 셈이죠. **MMR 검색**은 이 문제를 해결합니다.

또한 "2024년 이후 발표된 논문만 검색하고 싶다"거나 "특정 카테고리 문서만 보고 싶다"면 어떨까요? **메타데이터 필터링**이 필요합니다. 그리고 같은 문서를 반복해서 임베딩할 때마다 API 비용이 발생한다면? **임베딩 캐싱**으로 비용을 절감할 수 있습니다.

이 기법들은 RAG 시스템의 품질을 "쓸 만한 수준"에서 "프로덕션 수준"으로 끌어올리는 핵심 도구입니다. [9장: RAG 구축](ch09/session_01.md)에서 본격적으로 RAG 파이프라인을 만들기 전에, 검색 품질의 기반을 다져두겠습니다.

## 핵심 개념

### 개념 1: MMR(Maximal Marginal Relevance) 검색

> 💡 **비유**: 뷔페에서 음식을 고른다고 생각해보세요. "가장 맛있는 것만" 고르면 치킨만 다섯 접시를 가져올 수 있습니다. 하지만 "맛있으면서도 다양한 종류"를 고르면 치킨, 샐러드, 스테이크, 디저트 — 훨씬 만족스러운 한 끼가 되죠. MMR은 바로 이 "맛있으면서도 다양한" 전략입니다.

일반적인 `similarity_search`는 쿼리와 **가장 유사한** 문서 k개를 반환합니다. 문제는 상위 문서들이 서로 거의 같은 내용일 수 있다는 거죠. MMR은 **관련성(Relevance)**과 **다양성(Diversity)**을 동시에 최적화합니다.

MMR의 수식은 다음과 같습니다:

$$MMR = \arg\max_{D_i \in R \setminus S} \left[ \lambda \cdot Sim_1(D_i, Q) - (1-\lambda) \cdot \max_{D_j \in S} Sim_2(D_i, D_j) \right]$$

- $D_i$: 후보 문서, $Q$: 쿼리, $S$: 이미 선택된 문서 집합
- $\lambda$: 관련성과 다양성 사이의 균형 파라미터 (0~1)
- $Sim_1$: 쿼리와 문서 간 유사도, $Sim_2$: 문서 간 유사도

이게 의미하는 바는 — 쿼리와 관련성이 높으면서도($\lambda$ 항), 이미 선택된 문서들과는 가능한 한 다른 문서($1-\lambda$ 항)를 우선 선택하겠다는 뜻입니다.

LangChain에서는 `max_marginal_relevance_search` 메서드로 간단하게 사용할 수 있습니다:

```python
from langchain_openai import OpenAIEmbeddings
from langchain_community.vectorstores import FAISS
from langchain_core.documents import Document

# 임베딩 모델 설정
embeddings = OpenAIEmbeddings(model="text-embedding-3-small")

# 예시 문서 준비 (AI 관련 문서들)
documents = [
    Document(page_content="인공지능은 인간의 지능을 모방하는 컴퓨터 시스템이다.", metadata={"topic": "AI 개론"}),
    Document(page_content="인공지능 기술은 인간 지능의 다양한 측면을 컴퓨터로 구현한다.", metadata={"topic": "AI 개론"}),
    Document(page_content="머신러닝은 데이터로부터 패턴을 학습하는 AI의 하위 분야이다.", metadata={"topic": "ML"}),
    Document(page_content="딥러닝은 다층 신경망을 사용하여 복잡한 패턴을 학습한다.", metadata={"topic": "DL"}),
    Document(page_content="자연어 처리(NLP)는 컴퓨터가 인간 언어를 이해하게 한다.", metadata={"topic": "NLP"}),
    Document(page_content="강화학습은 환경과의 상호작용을 통해 최적 행동을 학습한다.", metadata={"topic": "RL"}),
]

# 벡터 스토어 생성
vector_store = FAISS.from_documents(documents, embeddings)

# --- 일반 유사도 검색 ---
print("=== 일반 유사도 검색 ===")
results = vector_store.similarity_search("인공지능이란 무엇인가?", k=3)
for doc in results:
    print(f"  [{doc.metadata['topic']}] {doc.page_content[:50]}...")

# --- MMR 검색 ---
print("\n=== MMR 검색 (다양성 강조) ===")
mmr_results = vector_store.max_marginal_relevance_search(
    "인공지능이란 무엇인가?",
    k=3,           # 최종 반환할 문서 수
    fetch_k=20,    # MMR 알고리즘에 전달할 후보 문서 수
    lambda_mult=0.5  # 0.0=최대 다양성, 1.0=최대 관련성
)
for doc in mmr_results:
    print(f"  [{doc.metadata['topic']}] {doc.page_content[:50]}...")
```

핵심 파라미터를 정리하면:

| 파라미터 | 기본값 | 설명 |
|----------|--------|------|
| `k` | 4 | 최종 반환할 문서 수 |
| `fetch_k` | 20 | MMR 재정렬에 사용할 후보 문서 수 |
| `lambda_mult` | 0.5 | 0.0(최대 다양성) ~ 1.0(최대 관련성) |

> 🔥 **실무 팁**: `fetch_k`는 `k`의 3~5배 정도로 설정하는 것이 좋습니다. 너무 작으면 다양성을 확보할 후보가 부족하고, 너무 크면 연산 비용이 늘어납니다.

Retriever 인터페이스에서도 MMR을 사용할 수 있습니다:

```python
# as_retriever에서 MMR 사용하기
retriever = vector_store.as_retriever(
    search_type="mmr",          # "similarity" 대신 "mmr" 지정
    search_kwargs={
        "k": 4,
        "fetch_k": 20,
        "lambda_mult": 0.5
    }
)

# LCEL 체인에서 바로 사용 가능
docs = retriever.invoke("인공지능의 핵심 분야는?")
```

#### MMR과 유사도 검색의 결과 차이 분석

MMR의 효과를 직관적으로 이해하기 위해, 동일한 쿼리에 대해 `lambda_mult` 값을 변화시키며 결과를 비교해봅시다:

```python
# lambda_mult에 따른 MMR 결과 변화 비교
print("=== lambda_mult에 따른 결과 비교 ===\n")

for lam in [1.0, 0.7, 0.5, 0.0]:
    results = vector_store.max_marginal_relevance_search(
        "인공지능이란 무엇인가?",
        k=3, fetch_k=20, lambda_mult=lam
    )
    topics = [doc.metadata['topic'] for doc in results]
    print(f"  lambda_mult={lam}: {topics}")
    # lambda_mult=1.0 → ["AI 개론", "AI 개론", "ML"]  (유사도 검색과 동일)
    # lambda_mult=0.5 → ["AI 개론", "ML", "NLP"]       (다양성 확보)
    # lambda_mult=0.0 → ["AI 개론", "RL", "DL"]        (최대 다양성)
```

이처럼 `lambda_mult`를 낮출수록 이미 선택된 문서와 **다른** 주제의 문서가 선택됩니다. RAG 시스템에서는 `0.5~0.7` 범위가 관련성과 다양성의 균형점으로 적합합니다.

### 개념 2: 메타데이터 필터링

> 💡 **비유**: 도서관에서 책을 찾을 때, 모든 서가를 뒤지는 대신 "3층 과학 섹션, 2020년 이후 출판"으로 범위를 좁히면 훨씬 빠르고 정확하게 원하는 책을 찾을 수 있죠. 메타데이터 필터링은 벡터 검색에 이 "범위 좁히기"를 더하는 것입니다.

[7.3: 벡터 스토어 구축](ch07/session_03.md)에서 `metadata` 딕셔너리를 문서에 첨부하는 방법을 배웠습니다. 이제 그 메타데이터를 **검색 시 필터 조건**으로 활용해봅시다. 메타데이터 필터링은 벡터 유사도 계산 전(또는 후)에 조건에 맞지 않는 문서를 걸러내어, 검색의 정밀도를 크게 높여줍니다.

```python
from langchain_openai import OpenAIEmbeddings
from langchain_community.vectorstores import Chroma
from langchain_core.documents import Document

embeddings = OpenAIEmbeddings(model="text-embedding-3-small")

# 메타데이터가 풍부한 문서들
documents = [
    Document(
        page_content="LangChain은 LLM 애플리케이션 개발 프레임워크이다.",
        metadata={"category": "framework", "year": 2023, "language": "python"}
    ),
    Document(
        page_content="FastAPI는 고성능 Python 웹 프레임워크이다.",
        metadata={"category": "framework", "year": 2019, "language": "python"}
    ),
    Document(
        page_content="React는 사용자 인터페이스 구축을 위한 JavaScript 라이브러리이다.",
        metadata={"category": "framework", "year": 2013, "language": "javascript"}
    ),
    Document(
        page_content="GPT-4는 OpenAI가 개발한 대규모 언어 모델이다.",
        metadata={"category": "model", "year": 2023, "language": "python"}
    ),
    Document(
        page_content="BERT는 양방향 인코더 기반의 사전학습 언어 모델이다.",
        metadata={"category": "model", "year": 2018, "language": "python"}
    ),
]

# Chroma 벡터 스토어 생성
vector_store = Chroma.from_documents(documents, embeddings)

# --- 필터 없이 검색 ---
print("=== 필터 없이 검색 ===")
results = vector_store.similarity_search("Python 프레임워크", k=3)
for doc in results:
    print(f"  [{doc.metadata['category']}/{doc.metadata['year']}] {doc.page_content[:40]}...")

# --- 메타데이터 필터 적용 ---
print("\n=== 2020년 이후 Python 프레임워크만 검색 ===")
filtered_results = vector_store.similarity_search(
    "Python 프레임워크",
    k=3,
    filter={
        "$and": [
            {"category": {"$eq": "framework"}},
            {"year": {"$gte": 2020}},        # 2020년 이후
            {"language": {"$eq": "python"}}
        ]
    }
)
for doc in filtered_results:
    print(f"  [{doc.metadata['category']}/{doc.metadata['year']}] {doc.page_content[:40]}...")
```

Chroma의 필터 문법은 MongoDB 스타일 연산자를 사용합니다:

| 연산자 | 설명 | 예시 |
|--------|------|------|
| `$eq` | 같음 | `{"year": {"$eq": 2023}}` |
| `$ne` | 같지 않음 | `{"category": {"$ne": "model"}}` |
| `$gt` / `$gte` | 초과 / 이상 | `{"year": {"$gte": 2020}}` |
| `$lt` / `$lte` | 미만 / 이하 | `{"year": {"$lt": 2023}}` |
| `$in` | 목록 중 하나 | `{"language": {"$in": ["python", "rust"]}}` |
| `$and` / `$or` | 논리 조합 | `{"$and": [...]}` |

FAISS에서는 벡터 스토어 자체에 메타데이터 필터가 내장되어 있지 않지만, Retriever 레벨에서 필터를 적용할 수 있습니다:

```python
# FAISS에서 메타데이터 기반 필터링 (직접 구현)
from langchain_core.documents import Document

def filtered_search(vector_store, query, metadata_filter, k=4):
    """FAISS에서 메타데이터 필터링을 적용한 검색"""
    # 넉넉히 후보를 가져온 뒤 필터링
    candidates = vector_store.similarity_search(query, k=k * 5)
    filtered = [
        doc for doc in candidates
        if all(doc.metadata.get(key) == value for key, value in metadata_filter.items())
    ]
    return filtered[:k]

# 사용 예시
results = filtered_search(
    vector_store, 
    "Python 프레임워크", 
    metadata_filter={"category": "framework"},
    k=3
)
```

> ⚠️ **흔한 오해**: "메타데이터 필터링은 벡터 검색 후에 결과를 걸러내는 것이다"라고 생각하기 쉽지만, Chroma나 Pinecone 같은 벡터 DB는 **인덱스 레벨에서 필터링을 먼저 수행**합니다(pre-filtering). 이 방식이 훨씬 효율적이고, "필터 후 결과가 k개 미만"인 문제도 줄여줍니다.

#### MMR과 메타데이터 필터링의 결합

MMR과 메타데이터 필터링은 독립적인 기법이므로 함께 사용하면 시너지가 큽니다. 메타데이터로 **검색 범위를 좁히고**, MMR로 그 범위 내에서 **다양한 결과를 확보**하는 전략입니다:

```python
# Chroma에서 메타데이터 필터 + MMR 결합
filtered_mmr_results = vector_store.max_marginal_relevance_search(
    "Python 개발 도구",
    k=3,
    fetch_k=10,
    lambda_mult=0.5,
    filter={"language": {"$eq": "python"}}  # Python 관련 문서만 대상으로 MMR 적용
)
for doc in filtered_mmr_results:
    print(f"  [{doc.metadata['category']}/{doc.metadata['year']}] {doc.page_content[:40]}...")

# Retriever 인터페이스에서도 동일하게 사용 가능
filtered_mmr_retriever = vector_store.as_retriever(
    search_type="mmr",
    search_kwargs={
        "k": 4,
        "fetch_k": 20,
        "lambda_mult": 0.5,
        "filter": {
            "$and": [
                {"year": {"$gte": 2023}},
                {"language": {"$eq": "python"}}
            ]
        }
    }
)

# LCEL 체인에 바로 연결
docs = filtered_mmr_retriever.invoke("LLM 애플리케이션 개발")
```

이 패턴은 실무에서 매우 자주 사용됩니다. 예를 들어 다국어 문서 컬렉션에서 특정 언어의 문서만 대상으로 다양한 검색 결과를 얻거나, 특정 기간의 문서만 대상으로 중복 없는 결과를 확보할 수 있습니다.

### 개념 3: 하이브리드 검색 — 키워드 + 벡터의 결합

> 💡 **비유**: 온라인 쇼핑몰에서 "나이키 운동화"를 검색한다고 생각해보세요. **키워드 검색**은 "나이키"가 정확히 적힌 상품을 찾고, **벡터(의미) 검색**은 "Nike"나 "스포츠 신발"처럼 의미가 비슷한 상품도 찾아줍니다. 둘을 합치면? "나이키"가 정확히 들어간 상품도 놓치지 않으면서, 관련된 상품도 함께 볼 수 있는 최고의 검색 경험이 됩니다.

벡터 검색은 **의미적 유사성**에 강하지만, 특정 키워드가 반드시 포함된 문서를 보장하지 않습니다. 반대로 키워드 검색(BM25)은 **정확한 용어 매칭**에 강하지만 동의어나 의미 변형을 잡지 못합니다. 하이브리드 검색은 두 방식의 장점을 결합합니다.

LangChain에서는 `BM25Retriever`와 `EnsembleRetriever`를 조합하여 하이브리드 검색을 구현할 수 있습니다:

```python
# 필요 패키지 설치: pip install rank-bm25
from langchain_community.retrievers import BM25Retriever
from langchain.retrievers import EnsembleRetriever

# BM25 키워드 검색기 생성
bm25_retriever = BM25Retriever.from_documents(documents)
bm25_retriever.k = 3

# 벡터 검색기와 앙상블
vector_retriever = vector_store.as_retriever(search_kwargs={"k": 3})
ensemble_retriever = EnsembleRetriever(
    retrievers=[bm25_retriever, vector_retriever],
    weights=[0.4, 0.6]  # BM25 40% + 벡터 60% 가중치
)

results = ensemble_retriever.invoke("FAISS 벡터 검색")
```

`EnsembleRetriever`는 내부적으로 **Reciprocal Rank Fusion(RRF)** 알고리즘을 사용하여 각 검색기의 결과를 재정렬합니다. RRF는 각 문서의 순위를 기반으로 점수를 계산하여 합산하는데, 여러 검색기에서 모두 높은 순위로 등장한 문서가 최종적으로 상위에 오게 됩니다.

> 📌 **포워드 레퍼런스**: 하이브리드 검색의 BM25, EnsembleRetriever, RRF 알고리즘은 [8.2: 키워드 및 앙상블 검색](ch08/session_02.md)에서 본격적으로 다룹니다. BM25의 수식과 파라미터 튜닝, `weights` 최적화 전략, RRF의 수학적 원리, 한국어 토크나이징 처리 등 심화 내용은 해당 세션을 참고하세요. 이번 세션에서는 벡터 검색 최적화의 관점에서 하이브리드 검색이 **왜 필요한지**를 이해하는 데 집중하겠습니다.

### 개념 4: 임베딩 캐싱

> 💡 **비유**: 매번 외국어 단어를 번역기에 넣어 번역하는 대신, 한 번 번역한 단어를 단어장에 적어두고 다시 찾아보는 것과 같습니다. 같은 단어를 다시 번역할 필요 없이, 단어장에서 바로 꺼내 쓰면 되니까요.

임베딩 API를 호출할 때마다 비용이 발생하고 네트워크 지연이 생깁니다. 같은 텍스트를 여러 번 임베딩하는 상황 — 예를 들어 벡터 스토어를 다시 구축하거나, 개발 중 반복 테스트하거나, 동일 문서를 여러 인덱스에 넣는 경우 — 에는 **캐싱**으로 큰 효과를 볼 수 있습니다.

LangChain의 `CacheBackedEmbeddings`는 임베딩 결과를 키-값 저장소에 캐싱합니다. 텍스트를 해싱하여 키로 사용하고, 캐시 히트 시 임베딩 API를 호출하지 않고 바로 결과를 반환합니다.

```python
from langchain.embeddings import CacheBackedEmbeddings
from langchain_openai import OpenAIEmbeddings
from langchain.storage import LocalFileStore, InMemoryByteStore
from langchain_community.vectorstores import FAISS
from langchain_core.documents import Document
import time

# 기본 임베딩 모델
underlying_embeddings = OpenAIEmbeddings(model="text-embedding-3-small")

# --- 방법 1: 메모리 캐시 (세션 동안만 유지) ---
memory_store = InMemoryByteStore()
cached_embeddings_memory = CacheBackedEmbeddings.from_bytes_store(
    underlying_embeddings,
    memory_store,
    namespace=underlying_embeddings.model  # 모델별 캐시 분리
)

# --- 방법 2: 파일 시스템 캐시 (영구 저장) ---
file_store = LocalFileStore("./embedding_cache/")
cached_embeddings_file = CacheBackedEmbeddings.from_bytes_store(
    underlying_embeddings,
    file_store,
    namespace=underlying_embeddings.model
)

# 캐싱 효과 측정
documents = [
    Document(page_content=f"테스트 문서 {i}번: LangChain으로 RAG 시스템 구축하기")
    for i in range(50)
]

# 첫 번째 실행 — 캐시 미스, API 호출 발생
start = time.time()
vector_store_1 = FAISS.from_documents(documents, cached_embeddings_file)
first_run = time.time() - start
print(f"첫 번째 실행 (캐시 미스): {first_run:.2f}초")

# 두 번째 실행 — 캐시 히트, API 호출 없음
start = time.time()
vector_store_2 = FAISS.from_documents(documents, cached_embeddings_file)
second_run = time.time() - start
print(f"두 번째 실행 (캐시 히트): {second_run:.2f}초")
print(f"속도 향상: {first_run / second_run:.1f}배")
```

`namespace` 파라미터가 중요한 이유는, 같은 텍스트라도 **다른 임베딩 모델**로 생성하면 전혀 다른 벡터가 나오기 때문입니다. 모델 이름을 네임스페이스로 사용하면 모델 간 캐시 충돌을 방지할 수 있습니다.

| 캐시 저장소 | 특징 | 적합한 상황 |
|-------------|------|-------------|
| `InMemoryByteStore` | 메모리에 저장, 빠름, 프로세스 종료 시 소멸 | 개발/테스트, 일회성 작업 |
| `LocalFileStore` | 파일 시스템에 영구 저장 | 로컬 개발, 반복 빌드 |
| `RedisStore` | Redis에 저장, 분산 환경 지원 | 프로덕션, 멀티 서버 |

## 더 깊이 알아보기

### MMR의 탄생 — Carbonell과 Goldstein의 1998년 논문

MMR(Maximal Marginal Relevance)은 1998년 카네기멜론대학교의 **Jaime Carbonell**과 **Jade Goldstein**이 ACM SIGIR 학회에서 발표한 논문 *"The Use of MMR, Diversity-Based Reranking for Reordering Documents and Producing Summaries"*에서 처음 소개되었습니다.

당시 정보 검색 시스템의 최대 고민은 "상위 결과가 너무 비슷하다"는 것이었습니다. 검색 엔진이 관련성만 기준으로 문서를 정렬하니, 1위부터 10위까지 거의 같은 내용인 경우가 허다했죠. Carbonell과 Goldstein은 이를 해결하기 위해 **한계 관련성(marginal relevance)** 개념을 도입했습니다 — "이미 본 것과 얼마나 다른가"를 관련성 계산에 포함시킨 것이죠.

놀라운 점은 이 논문이 원래 **문서 요약(summarization)**을 위해 설계되었다는 것입니다. 긴 문서에서 핵심 문장을 추출할 때, 비슷한 문장이 반복 선택되는 문제를 해결하려 했습니다. 그런데 이 아이디어가 정보 검색, 추천 시스템, 그리고 지금의 RAG 시스템에까지 광범위하게 적용되고 있으니, 26년 전의 통찰이 여전히 유효한 셈입니다.

## 흔한 오해와 팁

> ⚠️ **흔한 오해**: "MMR의 `lambda_mult=0`이면 다양성이 최대니까, 항상 0에 가깝게 설정하면 좋다"라고 생각할 수 있습니다. 하지만 `lambda_mult`가 너무 낮으면 쿼리와 관련 없는 문서까지 포함될 수 있습니다. 대부분의 경우 `0.5~0.7` 사이가 적절하며, RAG 시스템에서는 `0.5`가 좋은 시작점입니다.

> ⚠️ **흔한 오해**: "메타데이터 필터를 많이 걸수록 검색이 정확해진다"라고 생각할 수 있지만, 필터 조건이 너무 엄격하면 후보 문서가 극소수로 줄어들어 오히려 관련성 높은 문서를 놓칠 수 있습니다. 필터는 **확실한 조건**(연도, 카테고리 등)에만 적용하고, 의미적 판단은 벡터 유사도에 맡기세요.

> 🔥 **실무 팁**: `CacheBackedEmbeddings`의 `LocalFileStore` 캐시가 계속 쌓이면 디스크를 차지할 수 있습니다. 임베딩 모델을 업그레이드했을 때는 반드시 이전 모델의 캐시를 삭제하세요. `namespace`에 모델 이름을 넣어두면 모델별로 캐시 폴더를 분리할 수 있어 관리가 수월합니다.

> 🔥 **실무 팁**: MMR의 `fetch_k`를 늘리면 다양성은 확보되지만 검색 지연이 증가합니다. 프로덕션 환경에서는 `fetch_k`를 `k`의 3~5배로 설정하되, 응답 시간 요구사항에 맞게 조정하세요. Chroma에서 `fetch_k=50` 이상은 성능 저하가 눈에 띄기 시작합니다.

| `CacheBackedEmbeddings` | 임베딩 결과를 캐싱하여 비용과 지연 절감 |
| `LocalFileStore` | 파일 시스템 기반 영구 캐시. 반복 빌드에 적합 |
| `InMemoryByteStore` | 메모리 기반 휘발성 캐시. 개발/테스트에 적합 |

## 다음 섹션 미리보기

이번 챕터에서 텍스트 임베딩의 원리부터 벡터 스토어 구축, 프로덕션 배포, 그리고 검색 최적화까지 벡터 검색의 전체 파이프라인을 마스터했습니다. 다음 [8장: 검색기(Retriever) 심화](ch08/session_01.md)에서는 벡터 스토어 기반 검색기를 넘어서, Self-Query Retriever, Contextual Compression, Parent Document Retriever 등 **고급 검색 전략**을 배우게 됩니다. 특히 [8.2: 키워드 및 앙상블 검색](ch08/session_02.md)에서는 이번 세션에서 간략히 소개한 BM25, EnsembleRetriever, RRF를 본격적으로 다루며, 한국어 토크나이징 최적화와 가중치 튜닝 전략까지 심도 있게 학습합니다. 이번 세션에서 익힌 MMR과 메타데이터 필터링 기법이 8장의 검색기들과 결합되면, RAG 시스템의 검색 품질을 한 단계 더 끌어올릴 수 있습니다.
