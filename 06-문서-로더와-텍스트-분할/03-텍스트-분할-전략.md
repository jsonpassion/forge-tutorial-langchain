# 텍스트 분할 전략

> 로드된 문서를 LLM이 소화할 수 있는 적절한 크기의 청크로 나누는 기술을 마스터합니다.

## 개요

이 섹션에서는 앞서 [6.1: 문서 로더 기초](06-문서-로더와-텍스트-분할/01-문서-로더-기초.md)와 [6.2: 다양한 소스 로더](06-문서-로더와-텍스트-분할/02-다양한-소스-로더.md)에서 로드한 Document 객체들을 RAG 시스템에 적합한 크기로 분할하는 방법을 배웁니다. 텍스트 분할은 RAG 파이프라인의 검색 품질을 좌우하는 핵심 단계입니다.

**선수 지식**: Document 객체 구조(page_content, metadata), 기본 로더 사용법(TextLoader, PyPDFLoader)
**학습 목표**:
- RecursiveCharacterTextSplitter와 CharacterTextSplitter의 차이를 이해하고 적절히 선택할 수 있다
- chunk_size와 chunk_overlap 파라미터를 목적에 맞게 최적화할 수 있다
- 분할 결과의 품질을 정량적으로 평가할 수 있다
- 토큰 기반 분할과 문자 기반 분할의 차이를 설명할 수 있다

## 왜 알아야 할까?

여러분이 도서관 사서라고 상상해 보세요. 500쪽짜리 두꺼운 백과사전을 통째로 들고 다니면서 질문에 답할 순 없겠죠? 대신 관련 페이지만 찾아서 펼쳐 보여줄 겁니다. RAG 시스템에서 텍스트 분할이 바로 이 역할을 합니다.

LLM에는 한 번에 처리할 수 있는 **컨텍스트 윈도우**(Context Window) 제한이 있습니다. 그리고 설령 긴 문서를 한 번에 넣을 수 있더라도, 너무 긴 텍스트에서는 정보를 정확하게 찾아내지 못하는 **"건초더미에서 바늘 찾기"(Lost in the Middle)** 현상이 발생합니다.

텍스트 분할이 잘못되면 어떤 일이 벌어질까요?

- **너무 크게 자르면**: 검색 정밀도가 떨어지고, 불필요한 정보가 컨텍스트를 차지합니다
- **너무 작게 자르면**: 문맥이 끊겨 의미를 잃고, LLM이 충분한 정보를 얻지 못합니다
- **엉뚱한 곳에서 자르면**: 문장 중간이 잘려 의미가 왜곡됩니다

실제로 RAG 시스템의 성능 차이 중 상당 부분이 **청킹 전략**(Chunking Strategy) 선택에서 비롯됩니다. 같은 임베딩 모델, 같은 벡터 스토어를 써도 분할 전략에 따라 검색 정확도가 크게 달라지거든요.

## 핵심 개념

### 개념 1: 텍스트 분할의 기본 원리

> 💡 **비유**: 텍스트 분할은 긴 롤케이크를 자르는 것과 비슷합니다. 칼로 일정 간격(chunk_size)마다 자르되, 크림이 끊기지 않도록 살짝 겹치게(chunk_overlap) 자릅니다. 그리고 어디를 자를지가 중요합니다 — 딸기 위를 자르면 안 되겠죠? 문단 경계나 문장 경계처럼 자연스러운 지점에서 자르는 게 핵심입니다.

LangChain의 모든 텍스트 스플리터는 두 가지 핵심 파라미터로 동작합니다:

- **`chunk_size`**: 각 청크의 최대 크기 (문자 수 또는 토큰 수)
- **`chunk_overlap`**: 인접한 청크 사이에 겹치는 부분의 크기

```python
# 텍스트 분할의 기본 개념 시각화
from langchain_text_splitters import RecursiveCharacterTextSplitter

text = "ABCDEFGHIJKLMNOPQRSTUVWXYZ"

# chunk_size=10, chunk_overlap=3인 경우
splitter = RecursiveCharacterTextSplitter(
    chunk_size=10,       # 각 청크 최대 10자
    chunk_overlap=3,     # 3자씩 겹침
    separators=[""],     # 문자 단위 분할 (예시용)
    length_function=len, # 길이 측정 함수
)

chunks = splitter.split_text(text)
for i, chunk in enumerate(chunks):
    print(f"청크 {i}: '{chunk}' (길이: {len(chunk)})")

# 출력:
# 청크 0: 'ABCDEFGHIJ' (길이: 10)
# 청크 1: 'HIJKLMNOPQ' (길이: 10)
# 청크 2: 'OPQRSTUVWX' (길이: 10)
# 청크 3: 'VWXYZ' (길이: 5)
```

overlap이 있으니 `HIJ`, `OPQ`, `VWX` 부분이 인접 청크 양쪽에 모두 포함됩니다. 이렇게 겹치는 부분이 있어야 문맥이 끊기는 것을 방지할 수 있죠.

### 개념 2: CharacterTextSplitter — 단일 구분자 분할

> 💡 **비유**: CharacterTextSplitter는 종이를 **점선 하나**를 따라 자르는 것과 같습니다. 딱 하나의 구분자(예: 줄바꿈)만 보고 자릅니다.

CharacterTextSplitter는 가장 단순한 분할기입니다. **하나의 구분자**(separator)를 기준으로 텍스트를 나눕니다.

```python
from langchain_text_splitters import CharacterTextSplitter

text = """첫 번째 문단입니다.
이 문단은 LangChain에 대한 설명입니다.

두 번째 문단입니다.
텍스트 분할의 중요성을 다룹니다.

세 번째 문단입니다.
RAG 시스템에서의 활용을 설명합니다."""

# 기본 구분자는 "\n\n" (빈 줄)
splitter = CharacterTextSplitter(
    separator="\n\n",    # 빈 줄 기준으로 분할
    chunk_size=80,       # 최대 80자
    chunk_overlap=0,     # 겹침 없음
)

chunks = splitter.split_text(text)
for i, chunk in enumerate(chunks):
    print(f"--- 청크 {i} (길이: {len(chunk)}) ---")
    print(chunk)
    print()

# 출력:
# --- 청크 0 (길이: 42) ---
# 첫 번째 문단입니다.
# 이 문단은 LangChain에 대한 설명입니다.
#
# --- 청크 1 (길이: 41) ---
# 두 번째 문단입니다.
# 텍스트 분할의 중요성을 다룹니다.
#
# --- 청크 2 (길이: 41) ---
# 세 번째 문단입니다.
# RAG 시스템에서의 활용을 설명합니다.
```

CharacterTextSplitter의 중요한 특성이 하나 있습니다. 구분자로 나눈 개별 조각이 chunk_size보다 크면 **더 이상 쪼개지 않습니다**. 즉, chunk_size를 지정해도 실제 청크가 그보다 클 수 있어요. 이것이 RecursiveCharacterTextSplitter와의 핵심 차이점입니다.

> ⚠️ **흔한 오해**: CharacterTextSplitter에 chunk_size를 설정하면 모든 청크가 그 크기 이하가 될 거라고 생각하기 쉽습니다. 하지만 실제로는 구분자로 나눈 조각이 chunk_size보다 크면 그대로 반환됩니다. 엄격한 크기 제어가 필요하다면 RecursiveCharacterTextSplitter를 사용하세요.

### 개념 3: RecursiveCharacterTextSplitter — 재귀적 분할의 힘

> 💡 **비유**: RecursiveCharacterTextSplitter는 **러시안 마트료시카 인형**처럼 작동합니다. 먼저 가장 큰 경계(문단)로 나눠 보고, 아직 크면 그다음 경계(줄바꿈)로, 그래도 크면 더 작은 경계(공백)로, 최종적으로는 글자 단위까지 내려갑니다. 항상 **가장 의미 있는 경계**부터 시도하는 거죠.

RecursiveCharacterTextSplitter는 LangChain에서 **가장 권장되는** 텍스트 분할기입니다. 여러 구분자를 **우선순위 순서대로** 시도하며, 모든 청크가 chunk_size 이하가 되도록 보장합니다.

기본 구분자 목록(우선순위 순):
1. `"\n\n"` — 문단 경계 (빈 줄)
2. `"\n"` — 줄바꿈
3. `" "` — 공백 (단어 경계)
4. `""` — 문자 단위 (최후의 수단)

```python
from langchain_text_splitters import RecursiveCharacterTextSplitter

long_text = """인공지능의 역사는 1950년대로 거슬러 올라갑니다. 앨런 튜링은 "기계가 생각할 수 있는가?"라는 질문을 던졌고, 이것이 인공지능 연구의 시발점이 되었습니다.

1956년 다트머스 회의에서 "인공지능"이라는 용어가 공식적으로 탄생했습니다. 존 매카시, 마빈 민스키, 클로드 섀넌 등이 참석한 이 회의는 AI 연구의 공식적인 출발점으로 여겨집니다.

이후 수십 년간 AI는 여러 차례의 붐과 겨울을 겪었습니다. 전문가 시스템의 부흥, 신경망의 재발견, 그리고 딥러닝의 혁명을 거쳐 오늘날의 대규모 언어 모델(LLM)에 이르렀습니다. 특히 2017년 트랜스포머 아키텍처의 등장은 자연어 처리 분야에 혁명을 가져왔으며, GPT, BERT, LLaMA 등 현재 우리가 사용하는 모델들의 기반이 되었습니다."""

# RecursiveCharacterTextSplitter 사용
splitter = RecursiveCharacterTextSplitter(
    chunk_size=150,      # 각 청크 최대 150자
    chunk_overlap=30,    # 30자 겹침
    # separators는 기본값 사용: ["\n\n", "\n", " ", ""]
)

chunks = splitter.split_text(long_text)
for i, chunk in enumerate(chunks):
    print(f"--- 청크 {i} (길이: {len(chunk)}) ---")
    print(chunk)
    print()
```

**분할 과정을 추적해 볼까요?**

1단계: `"\n\n"`(빈 줄)으로 나눠서 3개의 문단을 얻습니다.
2단계: 각 문단이 150자 이하인지 확인합니다. 초과하는 문단이 있으면…
3단계: `"\n"`(줄바꿈)으로 다시 나눕니다. 아직 큰 조각이 있으면…
4단계: `" "`(공백)으로 나누고, 인접 조각들을 chunk_size 이내로 병합합니다.

이 과정이 **재귀적(recursive)**으로 일어나기 때문에 이름에 "Recursive"가 붙었습니다.

### 개념 4: Document 객체 직접 분할하기

실제 파이프라인에서는 문자열이 아니라 Document 객체를 분할하는 경우가 훨씬 많습니다. `split_documents()` 메서드를 사용하면 **메타데이터가 자동으로 보존**됩니다.

```python
from langchain_text_splitters import RecursiveCharacterTextSplitter
from langchain_core.documents import Document

# 문서 로더로 얻은 Document 객체들 (시뮬레이션)
documents = [
    Document(
        page_content="LangChain은 LLM 기반 애플리케이션 개발을 위한 프레임워크입니다. "
                     "모델 연동, 데이터 처리, 에이전트 구축 등을 지원합니다. "
                     "LCEL을 사용하면 선언적으로 체인을 구성할 수 있습니다.",
        metadata={"source": "langchain_intro.pdf", "page": 1}
    ),
    Document(
        page_content="RAG는 외부 지식을 활용하여 LLM 응답의 정확성을 높이는 기법입니다. "
                     "문서를 로드하고, 분할하고, 임베딩하여 벡터 스토어에 저장합니다. "
                     "질문이 들어오면 관련 문서를 검색하여 LLM에 함께 전달합니다.",
        metadata={"source": "rag_guide.pdf", "page": 5}
    ),
]

splitter = RecursiveCharacterTextSplitter(
    chunk_size=80,
    chunk_overlap=20,
)

# split_documents(): Document → Document (메타데이터 보존!)
split_docs = splitter.split_documents(documents)

for doc in split_docs:
    print(f"[{doc.metadata['source']} p.{doc.metadata['page']}]")
    print(f"  내용: {doc.page_content[:60]}...")
    print()

# 출력 예시:
# [langchain_intro.pdf p.1]
#   내용: LangChain은 LLM 기반 애플리케이션 개발을 위한 프레임워크입니다. 모델 연동, 데이...
#
# [langchain_intro.pdf p.1]
#   내용: 모델 연동, 데이터 처리, 에이전트 구축 등을 지원합니다. LCEL을 사용하면 선언적으로...
#
# [rag_guide.pdf p.5]
#   내용: RAG는 외부 지식을 활용하여 LLM 응답의 정확성을 높이는 기법입니다. 문서를 로드하고,...
# ...
```

`split_text()`는 문자열 리스트를 반환하고, `split_documents()`는 Document 리스트를 반환합니다. RAG 파이프라인에서는 메타데이터가 검색 결과의 출처 추적에 필수이므로 `split_documents()`를 사용하는 것이 좋습니다.

### 개념 5: chunk_size와 chunk_overlap 최적화

chunk_size와 chunk_overlap을 어떻게 정해야 할까요? 정답은 "상황에 따라 다르다"이지만, 실전에서 검증된 가이드라인이 있습니다.

**chunk_size 가이드라인:**

| 용도 | 권장 chunk_size | 이유 |
|------|----------------|------|
| 사실 기반 Q&A (팩트 검색) | 256~512 토큰 | 정밀한 검색에 유리 |
| 분석·요약 질문 | 1,024+ 토큰 | 넓은 맥락이 필요 |
| 코드 문서 | 100~200 토큰 | 함수/클래스 단위 분리 |
| 논문·연구 보고서 | 512~1,024 토큰 | 논증 흐름 보존 |

**chunk_overlap 가이드라인:**
- 일반적으로 chunk_size의 **10~20%** 를 권장합니다
- 예: chunk_size=1000이면 chunk_overlap=100~200
- 너무 크면 중복 저장으로 비용 증가, 너무 작으면 문맥 단절

```python
from langchain_text_splitters import RecursiveCharacterTextSplitter

sample_text = """머신러닝 모델의 학습 과정은 크게 세 단계로 나눌 수 있습니다.

첫째, 데이터 전처리 단계에서는 원시 데이터를 모델이 이해할 수 있는 형태로 변환합니다. 결측값 처리, 정규화, 인코딩 등의 작업이 포함됩니다.

둘째, 모델 학습 단계에서는 전처리된 데이터를 사용하여 모델의 파라미터를 최적화합니다. 손실 함수를 최소화하는 방향으로 경사 하강법을 반복합니다.

셋째, 평가 단계에서는 학습된 모델의 성능을 검증 데이터로 측정합니다. 정확도, 정밀도, 재현율, F1 스코어 등 다양한 지표를 활용합니다."""

# 서로 다른 설정 비교
configs = [
    {"chunk_size": 100, "chunk_overlap": 0,  "label": "작은 청크, 겹침 없음"},
    {"chunk_size": 100, "chunk_overlap": 20, "label": "작은 청크, 겹침 있음"},
    {"chunk_size": 200, "chunk_overlap": 40, "label": "중간 청크, 겹침 있음"},
]

for config in configs:
    splitter = RecursiveCharacterTextSplitter(
        chunk_size=config["chunk_size"],
        chunk_overlap=config["chunk_overlap"],
    )
    chunks = splitter.split_text(sample_text)
    print(f"\n=== {config['label']} ===")
    print(f"  chunk_size={config['chunk_size']}, "
          f"chunk_overlap={config['chunk_overlap']}")
    print(f"  생성된 청크 수: {len(chunks)}")
    for i, c in enumerate(chunks):
        print(f"  청크 {i}: {len(c)}자")
```

### 개념 6: 토큰 기반 분할 — from_tiktoken_encoder

LLM은 문자가 아니라 **토큰(Token)** 단위로 텍스트를 처리합니다. 한국어의 경우 한 글자가 2~3개 토큰으로 인코딩되는 경우가 많아, 문자 수 기반 분할과 실제 토큰 수 사이에 괴리가 생길 수 있습니다.

> 💡 **비유**: 문자 기반 분할은 줄자로 재는 것이고, 토큰 기반 분할은 저울로 재는 것과 같습니다. LLM이라는 요리사에게 재료를 건넬 때는 줄자(글자 수)보다 저울(토큰 수) 기준이 더 정확하겠죠?

```python
from langchain_text_splitters import RecursiveCharacterTextSplitter

# from_tiktoken_encoder: 토큰 수 기준으로 길이를 측정하는 스플리터 생성
splitter = RecursiveCharacterTextSplitter.from_tiktoken_encoder(
    model_name="gpt-4o",    # 사용할 모델의 토크나이저
    chunk_size=100,          # 최대 100 토큰
    chunk_overlap=20,        # 20 토큰 겹침
)

text = ("LangChain은 대규모 언어 모델(LLM)을 활용한 애플리케이션 개발을 "
        "위한 오픈소스 프레임워크입니다. " * 10)

chunks = splitter.split_text(text)
for i, chunk in enumerate(chunks):
    print(f"청크 {i}: {len(chunk)}자 | 내용 미리보기: {chunk[:50]}...")
```

`from_tiktoken_encoder()`는 내부적으로 tiktoken 라이브러리를 사용하여 **분할 기준은 문자 구분자**를 쓰되, **길이 측정만 토큰 수**로 합니다. 이렇게 하면 한국어처럼 멀티바이트 문자가 청크 도중에 깨지는 문제를 방지하면서도, 토큰 수 기준의 정확한 크기 제어가 가능합니다.

### 개념 7: 분할 품질 평가하기

분할 결과가 좋은지 어떻게 알 수 있을까요? 몇 가지 정량적 지표로 평가할 수 있습니다:

```python
from langchain_text_splitters import RecursiveCharacterTextSplitter
import statistics

def evaluate_chunks(chunks: list[str]) -> dict:
    """분할 결과의 품질을 평가하는 함수"""
    lengths = [len(c) for c in chunks]

    return {
        "총_청크_수": len(chunks),
        "평균_길이": round(statistics.mean(lengths), 1),
        "길이_표준편차": round(statistics.stdev(lengths), 1) if len(lengths) > 1 else 0,
        "최소_길이": min(lengths),
        "최대_길이": max(lengths),
        "너무_짧은_청크(50자_미만)": sum(1 for l in lengths if l < 50),
    }

# 테스트용 텍스트
with open("sample_document.txt", "r", encoding="utf-8") as f:
    text = f.read()  # 실제 문서 파일을 읽어옵니다

# 여러 설정으로 분할 후 비교
for chunk_size in [200, 500, 1000]:
    splitter = RecursiveCharacterTextSplitter(
        chunk_size=chunk_size,
        chunk_overlap=int(chunk_size * 0.15),  # 15% 겹침
    )
    chunks = splitter.split_text(text)
    stats = evaluate_chunks(chunks)
    print(f"\nchunk_size={chunk_size}:")
    for key, value in stats.items():
        print(f"  {key}: {value}")
```

**좋은 분할의 특징**:
- 길이의 표준편차가 낮다 (균일한 크기)
- 너무 짧은 청크(50자 미만)가 거의 없다
- 각 청크가 독립적으로 읽혔을 때 의미가 통한다

## 실습: 직접 해보기

실제 텍스트를 다양한 전략으로 분할하고, 결과를 비교·평가하는 완전한 실습입니다.

```python
"""
Session 6.3 실습: 텍스트 분할 전략 비교
필요 패키지: pip install langchain-text-splitters tiktoken
"""
from langchain_text_splitters import (
    CharacterTextSplitter,
    RecursiveCharacterTextSplitter,
)
from langchain_core.documents import Document
import statistics

# ── 1. 실습용 텍스트 준비 ──────────────────────────────
sample_text = """# 트랜스포머 아키텍처 개요

트랜스포머(Transformer)는 2017년 구글 연구팀이 "Attention Is All You Need" 논문에서 제안한 딥러닝 아키텍처입니다. 기존의 순환 신경망(RNN)이나 합성곱 신경망(CNN) 기반 시퀀스 모델의 한계를 극복하기 위해 설계되었습니다.

## 핵심 메커니즘: 셀프 어텐션

셀프 어텐션(Self-Attention)은 입력 시퀀스의 각 위치가 다른 모든 위치와의 관계를 동시에 계산하는 메커니즘입니다. 이를 통해 장거리 의존성(Long-range Dependency)을 효과적으로 포착할 수 있습니다.

수학적으로 어텐션 함수는 다음과 같이 정의됩니다:
Attention(Q, K, V) = softmax(QK^T / sqrt(d_k))V

여기서 Q(Query), K(Key), V(Value)는 입력에서 선형 변환으로 얻어지며, d_k는 Key의 차원입니다.

## 멀티헤드 어텐션

멀티헤드 어텐션(Multi-Head Attention)은 셀프 어텐션을 여러 번 병렬로 수행하는 기법입니다. 각 "헤드"가 서로 다른 표현 부분 공간에서 정보를 추출하므로, 다양한 관점에서 입력을 이해할 수 있습니다.

## 인코더-디코더 구조

트랜스포머는 인코더(Encoder)와 디코더(Decoder)로 구성됩니다. 인코더는 입력 시퀀스를 고차원 표현으로 변환하고, 디코더는 이 표현을 바탕으로 출력 시퀀스를 생성합니다.

BERT는 인코더만 사용하는 대표적 모델이고, GPT는 디코더만 사용합니다. T5나 BART는 인코더-디코더 둘 다 사용합니다.

## 위치 인코딩

트랜스포머는 순환 구조가 없으므로, 토큰의 위치 정보를 별도로 제공해야 합니다. 이를 위해 사인/코사인 함수 기반의 위치 인코딩(Positional Encoding)을 사용합니다. 최근에는 RoPE(Rotary Position Embedding) 같은 상대적 위치 인코딩 기법이 널리 사용됩니다.

## 현대적 변형

트랜스포머 이후 다양한 변형이 등장했습니다. Flash Attention은 메모리 효율을 극적으로 개선했고, Mixture of Experts(MoE)는 파라미터 효율성을 높였습니다. LLaMA, Mistral 등 오픈소스 모델들이 이러한 기술을 적극 활용하고 있습니다."""

# ── 2. CharacterTextSplitter vs RecursiveCharacterTextSplitter ───
print("=" * 60)
print("비교 1: CharacterTextSplitter vs RecursiveCharacterTextSplitter")
print("=" * 60)

# CharacterTextSplitter: 단일 구분자
char_splitter = CharacterTextSplitter(
    separator="\n\n",    # 빈 줄 기준으로만 분할
    chunk_size=200,
    chunk_overlap=0,
)
char_chunks = char_splitter.split_text(sample_text)

# RecursiveCharacterTextSplitter: 재귀적 분할
recursive_splitter = RecursiveCharacterTextSplitter(
    chunk_size=200,
    chunk_overlap=30,
)
recursive_chunks = recursive_splitter.split_text(sample_text)

print(f"\nCharacterTextSplitter: {len(char_chunks)}개 청크")
for i, c in enumerate(char_chunks):
    print(f"  청크 {i}: {len(c)}자 {'⚠️ >200자!' if len(c) > 200 else '✓'}")

print(f"\nRecursiveCharacterTextSplitter: {len(recursive_chunks)}개 청크")
for i, c in enumerate(recursive_chunks):
    print(f"  청크 {i}: {len(c)}자 {'⚠️ >200자!' if len(c) > 200 else '✓'}")

# ── 3. chunk_size / chunk_overlap 실험 ──────────────────
print("\n" + "=" * 60)
print("비교 2: chunk_size / chunk_overlap 조합 비교")
print("=" * 60)

configs = [
    (100, 0),    # 작은 청크, 겹침 없음
    (200, 30),   # 중간 청크, 15% 겹침
    (500, 75),   # 큰 청크, 15% 겹침
    (500, 200),  # 큰 청크, 40% 겹침 (과도한 겹침)
]

for chunk_size, chunk_overlap in configs:
    splitter = RecursiveCharacterTextSplitter(
        chunk_size=chunk_size,
        chunk_overlap=chunk_overlap,
    )
    chunks = splitter.split_text(sample_text)
    lengths = [len(c) for c in chunks]
    avg_len = round(statistics.mean(lengths), 1)
    std_len = round(statistics.stdev(lengths), 1) if len(lengths) > 1 else 0

    print(f"\n  size={chunk_size}, overlap={chunk_overlap}")
    print(f"    청크 수: {len(chunks)}, 평균 길이: {avg_len}, "
          f"표준편차: {std_len}, 최소: {min(lengths)}, 최대: {max(lengths)}")

# ── 4. Document 객체 분할 (메타데이터 보존 확인) ──────────
print("\n" + "=" * 60)
print("비교 3: Document 객체 분할과 메타데이터 보존")
print("=" * 60)

docs = [
    Document(
        page_content=sample_text,
        metadata={"source": "transformer_guide.md", "chapter": 1}
    )
]

splitter = RecursiveCharacterTextSplitter(
    chunk_size=300,
    chunk_overlap=50,
)

split_docs = splitter.split_documents(docs)
print(f"\n원본 문서 수: {len(docs)} → 분할 후: {len(split_docs)}개")
for i, doc in enumerate(split_docs):
    print(f"\n  청크 {i}:")
    print(f"    메타데이터: {doc.metadata}")
    # 첫 50자만 미리보기
    print(f"    내용: {doc.page_content[:50]}...")

# ── 5. 커스텀 구분자 활용 ─────────────────────────────
print("\n" + "=" * 60)
print("비교 4: 마크다운 문서를 위한 커스텀 구분자")
print("=" * 60)

# 마크다운 헤더를 우선 구분자로 설정
md_splitter = RecursiveCharacterTextSplitter(
    separators=[
        "\n## ",    # h2 헤더 경계 우선
        "\n### ",   # h3 헤더 경계
        "\n\n",     # 빈 줄 (문단)
        "\n",       # 줄바꿈
        " ",        # 공백
        "",         # 최후의 수단
    ],
    chunk_size=300,
    chunk_overlap=30,
)

md_chunks = md_splitter.split_text(sample_text)
print(f"\n마크다운 최적화 분할: {len(md_chunks)}개 청크")
for i, c in enumerate(md_chunks):
    # 첫 줄만 표시
    first_line = c.strip().split("\n")[0]
    print(f"  청크 {i} ({len(c)}자): {first_line}")
```

## 더 깊이 알아보기

### 텍스트 분할의 탄생 배경

텍스트를 작은 조각으로 나누는 아이디어는 사실 정보 검색(Information Retrieval) 분야에서 오래전부터 사용되어 왔습니다. 1990년대 웹 검색 엔진들도 웹 페이지를 "패시지(passage)" 단위로 분할하여 인덱싱했거든요.

하지만 LLM 시대의 텍스트 분할이 특별해진 이유는 **토큰 제한**과 **임베딩 품질** 때문입니다. 2020년 Facebook AI Research(현 Meta AI)의 RAG 논문이 발표되면서, "검색된 문서 조각을 LLM에 주입한다"는 패러다임이 확립되었고, 이때 문서를 얼마나 잘 나누느냐가 전체 시스템 성능을 좌우한다는 사실이 부각되었습니다.

### Greg Kamradt의 "5 Levels of Text Splitting"

2023년, RAG 전문가 Greg Kamradt는 텍스트 분할 전략을 5단계로 정리한 영상과 [ChunkViz](https://chunkviz.up.railway.app/) 시각화 도구를 공개하여 커뮤니티에 큰 반향을 일으켰습니다. 그가 제시한 5단계는 다음과 같습니다:

1. **Level 1 — 고정 길이 분할**: 단순히 N자마다 자르기
2. **Level 2 — 재귀적 분할**: RecursiveCharacterTextSplitter (구분자 계층)
3. **Level 3 — 문서 구조 기반**: Markdown, HTML, 코드 등 구조 인식
4. **Level 4 — 시맨틱 분할**: 임베딩 유사도로 의미 경계 탐지
5. **Level 5 — 에이전트 기반**: LLM이 직접 분할 지점을 판단

이번 섹션에서 배운 CharacterTextSplitter는 Level 1에, RecursiveCharacterTextSplitter는 Level 2에 해당합니다. Level 3 이상의 고급 분할 전략은 다음 세션들에서 다룹니다.

### 최적 chunk_size를 둘러싼 논쟁

최적의 chunk_size에 대해서는 연구자들 사이에서도 합의가 없습니다. 2024~2025년 발표된 여러 벤치마크 연구를 종합하면:

- **LlamaIndex 벤치마크**: 질문 유형별 최적 크기가 다르며, 사실 기반 질문에는 256~512 토큰이, 분석적 질문에는 1,024+ 토큰이 유리
- **NVIDIA 연구(2024)**: 7가지 청킹 전략을 5개 데이터셋에서 비교한 결과, 페이지 단위 분할이 0.648 정확도로 가장 높은 성능
- **Multi-Dataset Analysis(2025)**: 소규모 팩트 기반 데이터셋에서는 64~128 토큰, 넓은 맥락이 필요한 데이터셋에서는 512~1,024 토큰이 최적

결론적으로 "만능 chunk_size는 없다"는 것이 학계의 컨센서스입니다. 자신의 데이터와 질문 유형에 맞게 실험하고 평가하는 과정이 필수적이에요.

## 흔한 오해와 팁

> ⚠️ **흔한 오해**: "chunk_size를 크게 잡으면 더 많은 정보가 들어가니까 항상 좋다"고 생각하기 쉽습니다. 하지만 큰 청크는 임베딩의 의미가 희석되어 검색 정밀도가 오히려 떨어집니다. 또한 LLM 컨텍스트 윈도우를 불필요하게 많이 차지하여 비용도 증가합니다.

> 💡 **알고 계셨나요?**: LangChain의 RecursiveCharacterTextSplitter는 Harrison Chase가 LangChain 초기 버전(2022년 말)부터 포함시킨 핵심 유틸리티입니다. "대부분의 경우 이것만으로 충분하다"는 공식 문서의 권고처럼, 실무에서도 가장 널리 쓰이는 분할기입니다. 심지어 LangChain 공식 블로그에서도 "start with RecursiveCharacterTextSplitter"라고 명시적으로 추천하고 있습니다.

> 🔥 **실무 팁**: chunk_overlap은 공짜가 아닙니다! overlap이 크면 동일한 내용이 여러 청크에 중복 저장되어 벡터 스토어 용량이 늘어나고, 검색 시 중복 결과가 반환될 수 있습니다. 일반적으로 chunk_size의 **10~20%** 가 적정선입니다. 그리고 실전에서는 여러 설정으로 분할 후 `evaluate_chunks()` 같은 함수로 통계를 비교한 뒤 결정하세요.

> 🔥 **실무 팁**: 한국어 텍스트를 다룰 때는 `from_tiktoken_encoder()`를 적극 활용하세요. 한국어는 영어보다 토큰 효율이 낮아서(같은 의미를 표현하는 데 더 많은 토큰 필요), 문자 수 기준 chunk_size로는 실제 토큰 소비량을 정확히 예측할 수 없습니다.

## 핵심 정리

| 개념 | 설명 |
|------|------|
| CharacterTextSplitter | 단일 구분자로 분할. 간단하지만 chunk_size 초과 가능 |
| RecursiveCharacterTextSplitter | 여러 구분자를 우선순위대로 재귀 적용. **가장 권장되는** 범용 스플리터 |
| chunk_size | 각 청크의 최대 크기. 용도에 따라 256~1,024 토큰 범위에서 조정 |
| chunk_overlap | 인접 청크 간 겹침. 보통 chunk_size의 10~20% |
| split_text() | 문자열 → 문자열 리스트 반환 |
| split_documents() | Document 리스트 → Document 리스트 반환 (메타데이터 보존) |
| from_tiktoken_encoder() | 토큰 수 기준으로 길이를 측정하는 스플리터 생성. 한국어에 특히 유용 |
| 분할 품질 평가 | 청크 수, 평균 길이, 표준편차, 최소/최대 길이로 정량 평가 |
