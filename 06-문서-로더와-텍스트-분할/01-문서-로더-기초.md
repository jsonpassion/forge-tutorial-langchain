# 문서 로더 기초

> LangChain의 Document 객체 구조를 이해하고, TextLoader·PyPDFLoader·CSVLoader로 다양한 파일을 LLM이 처리할 수 있는 형태로 변환하는 방법을 배웁니다.

## 개요

이 섹션에서는 LangChain의 데이터 입수(ingestion) 파이프라인의 첫 번째 단계인 **문서 로딩**을 학습합니다. 외부 데이터를 LLM이 이해할 수 있는 형태로 변환하는 것은 RAG 시스템 구축의 출발점이죠.

**선수 지식**: [Ch5: LCEL 마스터](ch05/index.md)에서 배운 파이프 연산자(`|`)와 Runnable 인터페이스 개념
**학습 목표**:
- LangChain의 `Document` 객체 구조(`page_content`, `metadata`)를 이해한다
- `TextLoader`, `PyPDFLoader`, `CSVLoader`로 텍스트·PDF·CSV 파일을 로드할 수 있다
- `load()`와 `lazy_load()`의 차이를 이해하고 상황에 맞게 선택할 수 있다
- `BaseLoader` 인터페이스를 통해 모든 로더의 공통 패턴을 파악한다

## 왜 알아야 할까?

LLM은 놀라울 정도로 똑똑하지만, 한 가지 큰 약점이 있습니다 — **학습 데이터에 없는 최신 정보나 내부 문서는 알지 못한다는 것**이죠. 여러분 회사의 내부 매뉴얼, 최신 논문, 고객 데이터를 LLM이 활용하게 하려면 어떻게 해야 할까요?

바로 여기서 **문서 로더**가 등장합니다. 텍스트 파일, PDF, CSV, 웹 페이지 등 다양한 형식의 데이터를 LangChain이 이해하는 표준 형식으로 변환해주는 거죠. 이 변환이 제대로 되지 않으면, 이후의 텍스트 분할, 임베딩, 벡터 검색 — RAG 파이프라인 전체가 무너집니다.

문서 로더는 RAG의 **첫 번째 관문**이자, 데이터 품질을 결정짓는 핵심 단계입니다. 실무에서 "RAG 성능이 안 나온다"는 문제의 상당수가 사실 이 로딩 단계에서 데이터가 제대로 추출되지 않아 발생하거든요.

## 핵심 개념

### 개념 1: Document 객체 — LangChain의 표준 데이터 단위

> 💡 **비유**: Document 객체는 **도서관의 인덱스 카드**와 같습니다. 카드 앞면에는 책의 내용 요약(page_content)이, 뒷면에는 저자·출판년도·분류번호 같은 부가 정보(metadata)가 적혀 있죠. LangChain도 이와 똑같은 방식으로 모든 데이터를 정리합니다.

LangChain에서 모든 문서 데이터는 `Document` 객체로 표현됩니다. 이 객체는 `langchain_core.documents`에 정의되어 있고, 딱 **두 가지 핵심 속성**을 가집니다:

| 속성 | 타입 | 설명 |
|------|------|------|
| `page_content` | `str` | 실제 텍스트 내용 |
| `metadata` | `dict` | 출처, 페이지 번호 등 부가 정보 |

```python
from langchain_core.documents import Document

# Document 객체 직접 생성
doc = Document(
    page_content="LangChain은 LLM 애플리케이션 개발 프레임워크입니다.",
    metadata={"source": "intro.txt", "author": "Harrison Chase", "page": 0}
)

print(doc.page_content)  # LangChain은 LLM 애플리케이션 개발 프레임워크입니다.
print(doc.metadata)       # {'source': 'intro.txt', 'author': 'Harrison Chase', 'page': 0}
```

왜 텍스트만 저장하지 않고 metadata를 함께 두는 걸까요? RAG 시스템에서 검색 결과를 반환할 때 **"이 정보가 어디서 왔는지"** 알려주는 출처 추적(provenance)이 필수적이기 때문입니다. 나중에 사용자에게 "이 답변은 `report_2024.pdf`의 3페이지를 참고했습니다"라고 말할 수 있어야 하죠.

### 개념 2: BaseLoader — 모든 로더의 공통 인터페이스

> 💡 **비유**: USB 충전기가 스마트폰, 태블릿, 이어폰 — 기기가 달라도 **같은 포트 규격**을 쓰는 것처럼, LangChain의 200개 이상의 문서 로더도 모두 `BaseLoader`라는 **같은 인터페이스 규격**을 따릅니다.

`BaseLoader`는 `langchain_core.document_loaders`에 정의된 추상 클래스로, 모든 로더가 구현해야 하는 두 가지 핵심 메서드를 정의합니다:

| 메서드 | 반환 타입 | 설명 |
|--------|-----------|------|
| `load()` | `List[Document]` | 모든 문서를 한 번에 메모리에 로드 |
| `lazy_load()` | `Iterator[Document]` | 문서를 하나씩 순차적으로 반환 (제너레이터) |

```python
# load() — 한 번에 전부 가져오기 (작은 파일에 적합)
docs = loader.load()
print(f"총 {len(docs)}개 문서 로드")

# lazy_load() — 하나씩 스트리밍 (대용량 파일에 적합)
for doc in loader.lazy_load():
    process(doc)  # 한 번에 하나씩 처리, 메모리 절약
```

사실 `load()`의 내부 구현은 `list(self.lazy_load())`입니다. 즉, `lazy_load()`가 진짜 핵심이고 `load()`는 편의 메서드인 셈이죠. 수천 개의 문서를 처리해야 하는 프로덕션 환경에서는 반드시 `lazy_load()`를 사용해야 합니다 — 그래야 GB 단위의 데이터도 메모리 걱정 없이 처리할 수 있거든요.

### 개념 3: TextLoader — 텍스트 파일 로딩

> 💡 **비유**: TextLoader는 **메모장(Notepad)** 같은 존재입니다. 가장 단순하지만, 순수 텍스트를 읽는 데는 이것만큼 확실한 게 없죠.

`TextLoader`는 `.txt` 파일을 하나의 `Document` 객체로 변환하는 가장 기본적인 로더입니다.

```python
from langchain_community.document_loaders import TextLoader

# 기본 사용법
loader = TextLoader("./data/sample.txt")
docs = loader.load()

print(docs[0].page_content[:100])   # 텍스트 내용 (처음 100자)
print(docs[0].metadata)              # {'source': './data/sample.txt'}
```

#### 인코딩 처리 — 한국어 파일의 함정

한국어 텍스트 파일은 UTF-8이 아닌 EUC-KR이나 CP949로 인코딩된 경우가 흔합니다. 이때 아무 설정 없이 로드하면 `UnicodeDecodeError`가 발생하죠.

```python
# 방법 1: 인코딩 직접 지정
loader = TextLoader("./data/korean_doc.txt", encoding="utf-8")

# 방법 2: 인코딩 자동 감지 (chardet 라이브러리 필요)
loader = TextLoader("./data/unknown_encoding.txt", autodetect_encoding=True)

# 방법 3: 인코딩 지정 + 자동 감지 폴백
loader = TextLoader(
    "./data/mixed_doc.txt",
    encoding="utf-8",
    autodetect_encoding=True  # utf-8 실패 시 자동 감지 시도
)

docs = loader.load()
```

`autodetect_encoding=True`로 설정하면, 지정된 인코딩으로 읽기에 실패했을 때 자동으로 파일의 실제 인코딩을 감지하여 다시 시도합니다. 한국어 문서를 다룰 때 특히 유용한 옵션이에요.

### 개념 4: PyPDFLoader — PDF 파일 로딩

> 💡 **비유**: PDF는 **밀봉된 편지 봉투**와 같습니다. 눈으로 보기엔 깔끔하지만, 텍스트를 추출하려면 봉투를 뜯는 과정이 필요하죠. PyPDFLoader가 바로 이 "봉투를 뜯는 도구"입니다.

`PyPDFLoader`는 PDF 파일을 **페이지 단위**로 분할하여 각 페이지를 별도의 `Document` 객체로 만듭니다. 내부적으로 `pypdf` 라이브러리를 사용합니다.

```python
from langchain_community.document_loaders import PyPDFLoader

# 기본 사용법 — 페이지별로 Document 생성
loader = PyPDFLoader("./data/report.pdf")
pages = loader.load()

print(f"총 {len(pages)}페이지")          # 총 15페이지
print(pages[0].page_content[:200])       # 첫 페이지 텍스트 (처음 200자)
print(pages[0].metadata)
# {'source': './data/report.pdf', 'page': 0}
```

주목할 점은 metadata에 `page` 키가 자동으로 포함된다는 것입니다. 이 덕분에 나중에 RAG 시스템에서 "몇 페이지에 있는 정보입니다"라고 출처를 정확하게 알려줄 수 있죠.

#### 고급 옵션

```python
# 암호가 걸린 PDF 열기
loader = PyPDFLoader("./data/protected.pdf", password="my_secret")

# 이미지 추출 활성화
loader = PyPDFLoader("./data/with_images.pdf", extract_images=True)

# 전체 PDF를 하나의 Document로 로드 (단일 모드)
loader = PyPDFLoader("./data/report.pdf", mode="single")
single_doc = loader.load()  # Document 1개 반환
```

> ⚠️ **흔한 오해**: PyPDFLoader는 **모든 PDF에서 완벽하게 텍스트를 추출하지는 않습니다**. 스캔된 이미지 PDF, 복잡한 레이아웃(다단 편집), 표가 많은 문서에서는 텍스트가 뒤섞이거나 누락될 수 있습니다. 이런 경우 `PyMuPDFLoader`나 `UnstructuredPDFLoader` 같은 대안을 고려하세요.

### 개념 5: CSVLoader — CSV 파일 로딩

> 💡 **비유**: CSV 파일이 **엑셀 스프레드시트**라면, CSVLoader는 각 행(row)을 개별 **포스트잇**으로 옮겨 적는 작업자입니다. 각 포스트잇에는 그 행의 내용이 적히고, 뒷면에는 "몇 번째 행인지" 같은 메모가 붙죠.

`CSVLoader`는 CSV 파일의 **각 행(row)을 하나의 Document**로 변환합니다. 기본적으로 모든 열의 값을 `key: value` 형태로 합쳐서 `page_content`에 넣습니다.

```python
from langchain_community.document_loaders import CSVLoader

# 기본 사용법 — 각 행이 하나의 Document
loader = CSVLoader("./data/products.csv")
docs = loader.load()

print(docs[0].page_content)
# name: 무선 키보드
# price: 45000
# category: 전자기기

print(docs[0].metadata)
# {'source': './data/products.csv', 'row': 0}
```

#### source_column — 출처를 특정 열로 지정

기본적으로 metadata의 `source`는 파일 경로입니다. 하지만 `source_column`을 지정하면 특정 열의 값을 출처로 사용할 수 있습니다.

```python
# source를 'product_url' 열 값으로 지정
loader = CSVLoader(
    "./data/products.csv",
    source_column="product_url"  # metadata의 source가 URL로 변경
)
docs = loader.load()

print(docs[0].metadata)
# {'source': 'https://shop.example.com/keyboard-001', 'row': 0}
```

#### metadata_columns — 메타데이터 열 분리

특정 열을 `page_content`가 아닌 `metadata`로 분리하고 싶을 때 사용합니다.

```python
# category와 brand를 메타데이터로 분리
loader = CSVLoader(
    "./data/products.csv",
    metadata_columns=["category", "brand"]  # 이 열들은 metadata로 이동
)
docs = loader.load()

print(docs[0].page_content)
# name: 무선 키보드
# price: 45000

print(docs[0].metadata)
# {'source': './data/products.csv', 'row': 0, 'category': '전자기기', 'brand': 'TechBrand'}
```

#### csv_args — CSV 파싱 옵션 커스터마이징

탭 구분자나 특수 인용 문자를 쓰는 파일이라면, Python의 `csv.DictReader`에 전달할 인수를 직접 지정할 수 있습니다.

```python
# 탭으로 구분된 TSV 파일 로딩
loader = CSVLoader(
    "./data/products.tsv",
    csv_args={
        "delimiter": "\t",       # 탭 구분자
        "quotechar": '"',        # 인용 문자
        "fieldnames": ["name", "price", "category"]  # 헤더가 없는 경우
    }
)
```

> from langchain_community.document_loaders import PyPDFLoader
> pdf_loader = PyPDFLoader("./sample_data/your_file.pdf")
> pdf_docs = pdf_loader.load()
> for page in pdf_docs[:3]:  # 처음 3페이지만
>     print(f"[Page {page.metadata['page']}] {page.page_content[:100]}...")
> ```

## 더 깊이 알아보기

### LangChain의 탄생과 Document 추상화의 배경

LangChain은 2022년 10월, **Harrison Chase**가 머신러닝 스타트업 Robust Intelligence에서 근무하던 시절 개인 GitHub 계정(`hwchase17`)에서 시작한 **800줄짜리 Python 패키지**였습니다. Chase는 당시 미팅이나 밋업에서 사람들이 LLM으로 실험적인 것들을 만드는 걸 보면서, 몇 가지 **공통 패턴**을 발견했다고 합니다. 그 패턴들을 코드로 정리한 것이 LangChain의 시작이었죠.

초기 LangChain에서 가장 중요했던 통찰 중 하나가 바로 "**외부 데이터를 LLM에 연결하려면 표준 데이터 형식이 필요하다**"는 것이었습니다. 텍스트 파일, PDF, 웹 페이지 — 소스는 다양하지만, LLM에게 전달되는 형식은 결국 "텍스트 + 메타정보"로 통일될 수 있거든요. 이 아이디어가 `Document(page_content, metadata)` 추상화로 구체화되었고, 이후 모든 LangChain 로더의 기반이 되었습니다.

불과 6개월 뒤인 2023년 4월, LangChain은 Sequoia Capital으로부터 2,000만 달러 이상의 투자를 유치하며 기업가치 2억 달러 이상으로 평가받았습니다. 이때 이미 50개 이상의 문서 로더를 지원했는데, 현재는 **200개 이상**으로 늘어났죠. 하나의 간단한 추상화가 이렇게 거대한 생태계의 토대가 된 셈입니다.

### load()와 lazy_load() — 왜 두 개가 필요할까?

초기 LangChain에서는 `load()` 메서드만 존재했습니다. 하지만 사용자들이 수천 개의 문서, 수 GB의 PDF를 처리하기 시작하면서 메모리 문제가 빈번해졌죠. 이에 Python의 **제너레이터 패턴**을 활용한 `lazy_load()`가 도입되었고, 현재는 오히려 `lazy_load()`가 주 메서드, `load()`가 편의 메서드라는 설계 철학으로 바뀌었습니다. 실제로 `BaseLoader` 소스코드를 보면, `load()`의 구현은 단순히 `return list(self.lazy_load())`입니다.

## 흔한 오해와 팁

> ⚠️ **흔한 오해**: "`Document`는 반드시 로더를 통해서만 만들 수 있다" — 아닙니다! `Document(page_content="...", metadata={...})`로 직접 생성할 수 있습니다. REST API 응답, 데이터베이스 쿼리 결과 등 로더가 지원하지 않는 소스에서도 수동으로 Document를 만들어 LangChain 파이프라인에 넣을 수 있어요.

> 💡 **알고 계셨나요?**: LangChain의 `Document` 클래스는 내부적으로 Pydantic의 `BaseModel`을 상속합니다. 덕분에 자동 직렬화(JSON 변환), 타입 검증, 스키마 생성이 모두 가능하죠. `doc.model_dump()`로 딕셔너리로 변환하거나, `Document.model_validate(dict_data)`로 딕셔너리에서 복원할 수 있습니다.

> 🔥 **실무 팁**: `CSVLoader`로 대용량 CSV(수만 행 이상)를 로드하면 각 행이 하나의 Document가 되어 Document 수가 폭발합니다. 이런 경우 pandas로 먼저 데이터를 전처리하고, 관련 행을 그룹핑한 뒤 수동으로 `Document` 객체를 만드는 게 훨씬 효율적입니다.

> 🔥 **실무 팁**: PDF에서 텍스트 품질이 나쁘다면 로더를 바꿔보세요. `PyPDFLoader`(기본) → `PyMuPDFLoader`(빠르고 레이아웃 보존) → `UnstructuredPDFLoader`(복잡한 문서 구조 처리)  순으로 시도하는 것이 실무에서의 일반적인 접근법입니다.

## 핵심 정리

| 개념 | 설명 |
|------|------|
| `Document` | `page_content`(텍스트) + `metadata`(딕셔너리)로 구성된 LangChain의 표준 데이터 단위 |
| `BaseLoader` | 모든 로더가 구현하는 공통 인터페이스 — `load()`, `lazy_load()` 제공 |
| `load()` | 모든 문서를 한 번에 `List[Document]`로 반환 (소량 데이터에 적합) |
| `lazy_load()` | 문서를 제너레이터로 하나씩 반환 (대용량 데이터, 프로덕션에 권장) |
| `TextLoader` | 텍스트 파일 로더 — `encoding`, `autodetect_encoding` 파라미터 지원 |
| `PyPDFLoader` | PDF 로더 — 페이지별 Document 생성, `page` 메타데이터 자동 포함 |
| `CSVLoader` | CSV 로더 — 행별 Document 생성, `source_column`, `metadata_columns` 지원 |
| `metadata` | 출처 추적, 필터링, 검색 범위 제한 등에 활용되는 부가 정보 |
