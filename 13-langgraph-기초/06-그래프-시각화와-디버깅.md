# 그래프 시각화와 디버깅

> LangGraph 그래프를 눈으로 보고, 실행 과정을 추적하고, 체크포인트로 상태를 저장/복원하며, Human-in-the-Loop 패턴으로 사람의 개입을 구현하는 방법을 배웁니다.

## 개요

지금까지 우리는 LangGraph의 핵심 구성 요소를 하나씩 배워왔습니다. StateGraph로 상태를 정의하고, 노드와 엣지로 워크플로우를 조립하고, 조건부 분기와 도구 호출까지 구현했죠. 이제 마지막 퍼즐 조각을 맞출 차례입니다—**만든 그래프를 시각적으로 확인하고, 실행 과정을 실시간으로 추적하며, 체크포인트로 상태를 저장/복원하고, Human-in-the-Loop 패턴으로 사람의 승인을 받는 방법**입니다.

**선수 지식**: [13.1 LangGraph 소개와 핵심 개념](../13-1)에서 배운 StateGraph/노드/엣지 기본 구조, [13.2 첫 번째 LangGraph 에이전트](../13-2)에서 다룬 compile/invoke/stream, [13.3 조건부 엣지와 라우팅](../13-3)의 add_conditional_edges, [13.4 도구 호출 에이전트](../13-4)의 ToolNode/tools_condition, [13.5 상태 관리 심화](../13-5)의 리듀서 패턴

**학습 목표**:
- `get_graph().draw_mermaid()`로 그래프 구조를 시각화할 수 있다
- `stream_mode`의 다섯 가지 모드(`values`, `updates`, `debug`, `messages`, `custom`)를 이해하고 디버깅에 활용한다
- 노드 단위 테스트와 플로우 경로 테스트를 작성하여 그래프의 정확성을 검증한다
- `MemorySaver` 체크포인터로 그래프 실행 상태를 저장/복원할 수 있다
- `interrupt_before`/`interrupt_after`를 활용한 Human-in-the-Loop 패턴을 구현할 수 있다
- LangGraph Studio의 시각적 디버깅 기능을 이해한다

## 왜 알아야 할까?

"코드는 작성하는 시간보다 디버깅하는 시간이 두 배다"라는 유명한 격언이 있습니다. LangGraph 에이전트는 특히 그렇습니다. 노드가 10개, 조건부 엣지가 5개인 그래프를 상상해 보세요. 머릿속으로 모든 실행 경로를 추적하는 건 거의 불가능하죠.

실무에서 자주 마주치는 상황들을 떠올려 보세요:
- "이 에이전트가 왜 무한 루프에 빠졌지?"
- "도구를 호출한 다음 어떤 노드로 갔는지 모르겠어"
- "코드 한 줄 고쳤는데 다른 경로가 깨졌어"
- "에이전트가 위험한 작업을 실행하기 전에 사람이 확인해야 하는데..."

시각화는 **지도**를 주고, 스트리밍 디버깅은 **블랙박스 레코더**를 주며, 테스트는 **안전망**을 제공합니다. 그리고 체크포인트와 Human-in-the-Loop는 **비상 브레이크**를 제공합니다. 이 네 가지를 갖추면 복잡한 에이전트도 자신 있게 개발하고 유지보수할 수 있습니다.

## 핵심 개념

### 개념 1: 그래프 시각화 — 에이전트의 설계도 그리기

> 💡 **비유**: 건축가가 건물을 짓기 전에 설계도를 그리듯, 에이전트 개발자도 그래프의 구조를 눈으로 확인해야 합니다. `get_graph().draw_mermaid()`는 여러분의 에이전트를 위한 청사진을 자동으로 생성해 주는 도구입니다.

LangGraph는 컴파일된 그래프 객체에서 `.get_graph()` 메서드를 호출하면 내부 구조를 나타내는 `Graph` 객체를 반환합니다. 이 객체에서 다양한 시각화 메서드를 사용할 수 있습니다.

**세 가지 시각화 방법**:

| 방법 | 설명 | 추가 의존성 |
|------|------|-------------|
| `draw_mermaid()` | Mermaid 다이어그램 소스 코드 반환 | 없음 |
| `draw_mermaid_png()` | PNG 이미지 바이트 반환 | 없음 (Mermaid.ink API 사용) |
| `draw_ascii()` | ASCII 아트로 그래프 표시 | 없음 |

```python
from typing import TypedDict, Annotated, Literal
from langgraph.graph import StateGraph, START, END
import operator

# 간단한 그래프 정의
class State(TypedDict):
    query: str
    category: str
    response: str

def classify(state: State) -> dict:
    """질문을 분류하는 노드"""
    return {"category": "technical"}

def handle_technical(state: State) -> dict:
    """기술 질문 처리 노드"""
    return {"response": "기술 답변입니다."}

def handle_general(state: State) -> dict:
    """일반 질문 처리 노드"""
    return {"response": "일반 답변입니다."}

def route(state: State) -> Literal["handle_technical", "handle_general"]:
    """분류 결과에 따라 라우팅"""
    if state["category"] == "technical":
        return "handle_technical"
    return "handle_general"

# 그래프 조립
builder = StateGraph(State)
builder.add_node("classify", classify)
builder.add_node("handle_technical", handle_technical)
builder.add_node("handle_general", handle_general)

builder.add_edge(START, "classify")
builder.add_conditional_edges("classify", route)
builder.add_edge("handle_technical", END)
builder.add_edge("handle_general", END)

graph = builder.compile()

# 방법 1: Mermaid 소스 코드 출력
print(graph.get_graph().draw_mermaid())
# 출력:
# %%{init: {'flowchart': {'curve': 'linear'}}}%%
# graph TD;
#     __start__([<p>__start__</p>]):::first
#     classify(classify)
#     handle_technical(handle_technical)
#     handle_general(handle_general)
#     __end__([<p>__end__</p>]):::last
#     __start__ --> classify;
#     classify -.-> handle_technical;
#     classify -.-> handle_general;
#     handle_technical --> __end__;
#     handle_general --> __end__;

# 방법 2: PNG 이미지로 저장
png_bytes = graph.get_graph().draw_mermaid_png()
with open("graph_diagram.png", "wb") as f:
    f.write(png_bytes)

# 방법 3: Jupyter 노트북에서 바로 표시
# from IPython.display import Image, display
# display(Image(graph.get_graph().draw_mermaid_png()))

# 방법 4: ASCII 아트로 터미널에서 확인
graph.get_graph().print_ascii()
# 출력 예:
#        +-----------+
#        | __start__ |
#        +-----------+
#              *
#              *
#              *
#         +----------+
#         | classify |
#         +----------+
#          **        **
#        **            **
#       *                *
# +-----------+  +-----------+
# |handle_tech|  |handle_gen |
# +-----------+  +-----------+
#          **        **
#            **    **
#              *  *
#         +---------+
#         | __end__ |
#         +---------+
```

> 🔥 **실무 팁**: `draw_mermaid()`로 얻은 소스 코드를 [Mermaid Live Editor](https://mermaid.live)에 붙여넣으면 브라우저에서 인터랙티브하게 확인하고 SVG/PNG로 내보낼 수 있습니다. CI/CD 파이프라인에서 `draw_mermaid_png()`를 호출해 PR마다 최신 그래프 다이어그램을 자동 생성하는 팀도 많습니다.

### 개념 2: 스트리밍 모드 — 실행 과정을 실시간으로 들여다보기

> 💡 **비유**: 자동차 계기판을 생각해 보세요. 속도계(values 모드)는 현재 전체 상태를 보여주고, 연비 변화(updates 모드)는 무엇이 바뀌었는지만 알려주며, OBD-II 진단 포트(debug 모드)는 엔진 내부의 모든 센서 데이터를 쏟아냅니다. 라디오 자막(messages 모드)은 실시간으로 흘러가는 텍스트고, 네비게이션 알림(custom 모드)은 여러분이 직접 설정한 맞춤 알림이죠.

LangGraph의 `stream()` 메서드는 `stream_mode` 파라미터로 다섯 가지 모드를 제공합니다. 각각의 용도가 다르기 때문에 상황에 맞게 선택해야 합니다.

```python
from typing import TypedDict, Annotated
from langgraph.graph import StateGraph, START, END
import operator

class DebugState(TypedDict):
    input: str
    steps: Annotated[list[str], operator.add]
    result: str

def step_a(state: DebugState) -> dict:
    return {"steps": ["A 실행"], "result": "A 완료"}

def step_b(state: DebugState) -> dict:
    return {"steps": ["B 실행"], "result": "B 완료"}

builder = StateGraph(DebugState)
builder.add_node("step_a", step_a)
builder.add_node("step_b", step_b)
builder.add_edge(START, "step_a")
builder.add_edge("step_a", "step_b")
builder.add_edge("step_b", END)
graph = builder.compile()

initial_input = {"input": "테스트", "steps": []}

# ── 1) values 모드: 매 노드 실행 후 전체 상태 반환 ──
print("=== values 모드 ===")
for chunk in graph.stream(initial_input, stream_mode="values"):
    print(f"전체 상태: {chunk}")
# 출력:
# 전체 상태: {'input': '테스트', 'steps': []}
# 전체 상태: {'input': '테스트', 'steps': ['A 실행'], 'result': 'A 완료'}
# 전체 상태: {'input': '테스트', 'steps': ['A 실행', 'B 실행'], 'result': 'B 완료'}

# ── 2) updates 모드: 각 노드가 반환한 변경분만 표시 ──
print("\n=== updates 모드 ===")
for chunk in graph.stream(initial_input, stream_mode="updates"):
    print(f"변경분: {chunk}")
# 출력:
# 변경분: {'step_a': {'steps': ['A 실행'], 'result': 'A 완료'}}
# 변경분: {'step_b': {'steps': ['B 실행'], 'result': 'B 완료'}}

# ── 3) debug 모드: 노드 진입/종료, 전후 상태, 타임스탬프 등 상세 추적 ──
print("\n=== debug 모드 ===")
for event in graph.stream(initial_input, stream_mode="debug"):
    event_type = event.get("type", "unknown")
    if event_type == "task":
        print(f"[TASK] 노드 '{event['payload']['name']}' 시작")
    elif event_type == "task_result":
        print(f"[RESULT] 노드 '{event['payload']['name']}' → "
              f"결과: {event['payload']['result']}")
# 출력 (간략화):
# [TASK] 노드 'step_a' 시작
# [RESULT] 노드 'step_a' → 결과: [{'steps': ['A 실행'], 'result': 'A 완료'}]
# [TASK] 노드 'step_b' 시작
# [RESULT] 노드 'step_b' → 결과: [{'steps': ['B 실행'], 'result': 'B 완료'}]
```

**다섯 가지 스트리밍 모드 비교**:

| 모드 | 반환 내용 | 주 용도 |
|------|-----------|---------|
| `values` | 매 노드 후 **전체 상태** | 상태 변화 추적, 대시보드 |
| `updates` | 각 노드의 **변경분(delta)** | 경량 모니터링, 로깅 |
| `debug` | 진입/종료, 전후 상태, 타임스탬프 | **개발 중 디버깅** |
| `messages` | LLM 토큰 단위 스트리밍 | 채팅 UI 실시간 출력 |
| `custom` | 사용자 정의 데이터 | 진행률 표시, 커스텀 이벤트 |

**여러 모드를 동시에 사용**할 수도 있습니다:

```python
# updates와 debug를 동시에 스트리밍
for mode, chunk in graph.stream(
    initial_input,
    stream_mode=["updates", "debug"]  # 리스트로 여러 모드 전달
):
    print(f"[{mode}] {chunk}")
# 출력: 각 이벤트마다 (모드명, 데이터) 튜플이 반환됩니다
```

### 개념 3: 단계별 실행 추적 — 에이전트의 블랙박스 열기

> 💡 **비유**: 비행기의 블랙박스(Flight Data Recorder)는 비행 중 모든 데이터를 기록하여 사고 원인을 분석할 수 있게 해줍니다. LangGraph의 `debug` 모드와 `astream_events()`는 에이전트의 블랙박스 역할을 합니다.

`debug` 모드로 스트리밍하면 각 노드 실행의 전체 라이프사이클을 추적할 수 있습니다. 이벤트 타입별로 구조화된 데이터가 나오기 때문에 로그를 파싱하기도 쉽습니다.

```python
import json

def debug_trace(graph, inputs: dict) -> None:
    """그래프 실행을 debug 모드로 추적하고 보기 좋게 출력하는 유틸리티"""
    print("=" * 60)
    print("🔍 실행 추적 시작")
    print("=" * 60)

    for event in graph.stream(inputs, stream_mode="debug"):
        event_type = event.get("type", "")
        payload = event.get("payload", {})

        if event_type == "checkpoint":
            # 체크포인트: 현재 상태 스냅샷
            print(f"\n📸 [체크포인트]")

        elif event_type == "task":
            # 노드 실행 시작
            node_name = payload.get("name", "?")
            print(f"\n▶️  [노드 시작] {node_name}")
            # 입력 상태 확인
            if "input" in payload:
                print(f"   입력: {json.dumps(payload['input'], ensure_ascii=False, indent=2)[:200]}")

        elif event_type == "task_result":
            # 노드 실행 완료
            node_name = payload.get("name", "?")
            result = payload.get("result", [])
            print(f"✅ [노드 완료] {node_name}")
            for r in result:
                print(f"   출력: {json.dumps(r, ensure_ascii=False, indent=2)[:200]}")

    print("\n" + "=" * 60)
    print("🏁 실행 추적 완료")
    print("=" * 60)

# 사용 예
# debug_trace(graph, {"input": "테스트", "steps": []})
```

비동기 환경에서는 `astream_events()`를 사용하면 더 세밀한 이벤트를 받을 수 있습니다:

import asyncio

async def trace_with_events(graph, inputs: dict) -> None:
    """astream_events를 사용한 상세 이벤트 추적"""
    async for event in graph.astream_events(inputs, version="v2"):
        kind = event["event"]
        name = event.get("name", "")

        if kind == "on_chain_start" and name:
            print(f"⛓️  체인 시작: {name}")
        elif kind == "on_chain_end" and name:
            print(f"⛓️  체인 종료: {name}")
        elif kind == "on_chat_model_stream":
            # LLM 토큰 스트리밍 (채팅 모델 사용 시)

                print(content, end="", flush=True)
        elif kind == "on_tool_start":
            print(f"\n🔧 도구 호출: {name}")
        elif kind == "on_tool_end":
            print(f"🔧 도구 완료: {name}")

### 개념 4: 그래프 테스트 전략 — 에이전트의 안전망

> 💡 **비유**: 등산할 때 로프가 안전망 역할을 하듯, 테스트는 에이전트 코드의 안전망입니다. 노드 하나를 수정했을 때 전체 그래프가 여전히 올바르게 동작하는지 자동으로 확인해 줍니다. 로프 없이 암벽을 오르는 건 무모한 거죠.

LangGraph 그래프 테스트는 **두 가지 레벨**로 나눕니다:

**레벨 1: 노드 단위 테스트(Node-level Test)**
각 노드 함수는 그냥 파이썬 함수입니다. 입력 상태를 넣고 출력을 검증하면 됩니다.

```python
import pytest

# ── 노드 함수 정의 (테스트 대상) ──
def classify_query(state: dict) -> dict:
    """쿼리를 분류하는 노드"""
    query = state["query"].lower()
    if any(kw in query for kw in ["에러", "버그", "오류", "설치"]):
        return {"category": "technical"}
    return {"category": "general"}

def generate_response(state: dict) -> dict:
    """카테고리에 따라 응답을 생성하는 노드"""
    if state["category"] == "technical":
        return {"response": "기술 지원팀에 연결합니다."}
    return {"response": "일반 안내를 드리겠습니다."}

# ── 레벨 1: 노드 단위 테스트 ──
class TestNodes:
    def test_classify_technical(self):
        """기술 키워드가 포함된 쿼리를 technical로 분류"""
        state = {"query": "설치 에러가 발생했습니다"}
        result = classify_query(state)
        assert result["category"] == "technical"

    def test_classify_general(self):
        """일반 쿼리를 general로 분류"""
        state = {"query": "영업 시간이 어떻게 되나요?"}
        result = classify_query(state)
        assert result["category"] == "general"

    def test_generate_technical_response(self):
        """기술 카테고리에 대한 올바른 응답"""
        state = {"category": "technical"}
        result = generate_response(state)
        assert "기술 지원" in result["response"]

    def test_generate_general_response(self):
        """일반 카테고리에 대한 올바른 응답"""
        state = {"category": "general"}
        result = generate_response(state)
        assert "일반 안내" in result["response"]
```

**레벨 2: 플로우 경로 테스트(Flow-path Test)**
전체 그래프를 컴파일하고 실행하여 올바른 경로로 흐르는지 확인합니다.

```python
from langgraph.graph import StateGraph, START, END
from typing import TypedDict, Literal

class SupportState(TypedDict):
    query: str
    category: str
    response: str

def build_support_graph() -> StateGraph:
    """테스트마다 새 그래프를 생성하는 팩토리 함수"""
    def route(state: SupportState) -> Literal["technical", "general"]:
        return state["category"]

    builder = StateGraph(SupportState)
    builder.add_node("classify", classify_query)
    builder.add_node("technical", generate_response)
    builder.add_node("general", generate_response)

    builder.add_edge(START, "classify")
    builder.add_conditional_edges("classify", route)
    builder.add_edge("technical", END)
    builder.add_edge("general", END)

    return builder.compile()

# ── 레벨 2: 플로우 경로 테스트 ──
class TestFlowPaths:
    def test_technical_path(self):
        """기술 질문이 technical 노드를 거치는지 확인"""
        graph = build_support_graph()
        # updates 모드로 어떤 노드가 실행되었는지 추적
        visited_nodes = []
        for chunk in graph.stream(
            {"query": "설치 에러입니다"},
            stream_mode="updates"
        ):
            visited_nodes.extend(chunk.keys())

        assert "classify" in visited_nodes
        assert "technical" in visited_nodes
        assert "general" not in visited_nodes

    def test_general_path(self):
        """일반 질문이 general 노드를 거치는지 확인"""
        graph = build_support_graph()
        visited_nodes = []
        for chunk in graph.stream(
            {"query": "영업 시간 알려주세요"},
            stream_mode="updates"
        ):
            visited_nodes.extend(chunk.keys())

        assert "classify" in visited_nodes
        assert "general" in visited_nodes
        assert "technical" not in visited_nodes

    def test_final_state(self):
        """최종 상태가 올바른지 확인"""
        graph = build_support_graph()
        result = graph.invoke({"query": "버그 리포트"})
        assert result["category"] == "technical"
        assert "기술 지원" in result["response"]
```

> ⚠️ **흔한 오해**: "LLM을 사용하는 노드는 테스트할 수 없다"고 생각하는 분이 많은데, 사실 그렇지 않습니다. LLM 호출 부분을 모킹(mocking)하면 결정론적 테스트가 가능합니다. 노드 함수 내부의 LLM을 파라미터로 주입받도록 설계하면 테스트 시 가짜 LLM을 넣을 수 있죠.

### 개념 5: 체크포인트와 Human-in-the-Loop — 에이전트에 비상 브레이크 달기

> 💡 **비유**: 공장의 조립 라인을 떠올려 보세요. 각 공정 사이에 **품질 검사 스테이션**이 있어서, 불량이 발견되면 라인을 멈추고 사람이 확인합니다. 확인이 끝나면 다시 라인이 돌아가죠. LangGraph의 체크포인트와 `interrupt_before`/`interrupt_after`가 바로 이 품질 검사 스테이션 역할을 합니다.

[13.4 도구 호출 에이전트](../13-4)와 [13.5 상태 관리 심화](../13-5)에서 예고했던 내용이 바로 이것입니다. 에이전트가 자율적으로 작업을 수행할 때, 특정 시점에서 **사람의 승인**을 받거나 **중간 상태를 저장**해두는 것은 안전한 에이전트 운용의 핵심이거든요.

#### 체크포인터(Checkpointer) — 상태의 스냅샷

체크포인터는 그래프 실행의 각 단계에서 **상태를 자동으로 저장**합니다. 게임의 세이브 포인트와 같은 개념이죠. LangGraph는 `MemorySaver`(인메모리), `SqliteSaver`(SQLite), `PostgresSaver`(PostgreSQL) 등 다양한 체크포인터를 제공합니다.

```python
from langgraph.graph import StateGraph, START, END
from langgraph.checkpoint.memory import MemorySaver
from typing import TypedDict, Annotated
import operator

class OrderState(TypedDict):
    order_id: str
    items: list[str]
    total: int
    status: str
    log: Annotated[list[str], operator.add]

def validate_order(state: OrderState) -> dict:
    """주문 유효성 검사"""
    return {
        "status": "validated",
        "log": ["주문 검증 완료"]
    }

def calculate_total(state: OrderState) -> dict:
    """총액 계산"""
    total = len(state["items"]) * 15000  # 개당 15,000원
    return {
        "total": total,
        "status": "calculated",
        "log": [f"총액 계산: {total:,}원"]
    }

def process_payment(state: OrderState) -> dict:
    """결제 처리 — 이 단계 전에 사람의 확인이 필요!"""
    return {
        "status": "paid",
        "log": [f"결제 완료: {state['total']:,}원"]
    }

# 체크포인터와 함께 그래프 컴파일
checkpointer = MemorySaver()

builder = StateGraph(OrderState)
builder.add_node("validate", validate_order)
builder.add_node("calculate", calculate_total)
builder.add_node("payment", process_payment)

builder.add_edge(START, "validate")
builder.add_edge("validate", "calculate")
builder.add_edge("calculate", "payment")
builder.add_edge("payment", END)

# ✅ 체크포인터를 연결하면 매 노드 실행 후 상태가 자동 저장됩니다
graph = builder.compile(checkpointer=checkpointer)

# thread_id로 실행 세션을 구분합니다
config = {"configurable": {"thread_id": "order-001"}}
result = graph.invoke(
    {"order_id": "ORD-001", "items": ["노트북", "마우스"], "log": []},
    config=config
)
print(f"결과: {result['status']}")
# 출력: 결과: paid

# 저장된 상태를 언제든 조회할 수 있습니다
saved_state = graph.get_state(config)
print(f"저장된 로그: {saved_state.values['log']}")
# 출력: 저장된 로그: ['주문 검증 완료', '총액 계산: 30,000원', '결제 완료: 30,000원']

# 이전 체크포인트 히스토리도 탐색 가능합니다
for state in graph.get_state_history(config):
    print(f"  상태: {state.values.get('status', '초기')} "
          f"(다음 노드: {state.next})")
# 출력:
#   상태: paid (다음 노드: ())
#   상태: calculated (다음 노드: ('payment',))
#   상태: validated (다음 노드: ('calculate',))
#   상태: 초기 (다음 노드: ('validate',))
```

#### Human-in-the-Loop — 사람의 승인 받기

`interrupt_before` 또는 `interrupt_after`를 설정하면, 해당 노드 실행 전/후에 그래프가 **자동으로 멈춥니다**. 체크포인터가 있어야만 동작하는데, 멈춘 시점의 상태를 저장해두어야 나중에 이어서 실행할 수 있기 때문이죠.

```python
# 결제 전에 사람의 승인을 받는 Human-in-the-Loop 패턴
checkpointer = MemorySaver()

hitl_graph = builder.compile(
    checkpointer=checkpointer,
    interrupt_before=["payment"]  # payment 노드 실행 전에 멈춤!
)

config = {"configurable": {"thread_id": "order-hitl-001"}}

# 1단계: 실행 → payment 전에서 멈춤
print("🚀 주문 처리 시작...")
for chunk in hitl_graph.stream(
    {"order_id": "ORD-002", "items": ["키보드", "모니터", "헤드셋"], "log": []},
    config=config,
    stream_mode="updates"
):
    for node, update in chunk.items():
        print(f"  ✅ {node} 완료")
# 출력:
#   ✅ validate 완료
#   ✅ calculate 완료
# (payment 전에서 멈춤 — 결제가 실행되지 않았습니다!)

# 2단계: 멈춘 상태 확인 — 사람이 검토
state = hitl_graph.get_state(config)
print(f"\n⏸️  일시 정지됨")
print(f"  다음 노드: {state.next}")          # ('payment',)
print(f"  주문 항목: {state.values['items']}")
print(f"  총액: {state.values['total']:,}원")
# 출력:
#   다음 노드: ('payment',)
#   주문 항목: ['키보드', '모니터', '헤드셋']
#   총액: 45,000원

# 3단계: (선택) 상태를 수정할 수도 있습니다
# 예: 사람이 할인을 적용하기로 결정
hitl_graph.update_state(
    config,
    {"total": 40000, "log": ["관리자 할인 적용: 45,000 → 40,000원"]}
)

# 4단계: 승인 후 실행 재개 — None을 전달하면 현재 상태에서 이어서 실행
print("\n▶️  승인 완료, 실행 재개...")
for chunk in hitl_graph.stream(None, config=config, stream_mode="updates"):
    for node, update in chunk.items():
        print(f"  ✅ {node} 완료")
# 출력:
#   ✅ payment 완료

final = hitl_graph.get_state(config)
print(f"\n📋 최종 결과:")
print(f"  상태: {final.values['status']}")
print(f"  총액: {final.values['total']:,}원")
print(f"  전체 로그: {final.values['log']}")
# 출력:
#   상태: paid
#   총액: 40,000원
#   전체 로그: ['주문 검증 완료', '총액 계산: 45,000원', '관리자 할인 적용: ...', '결제 완료: 40,000원']
```

> ⚠️ **흔한 오해**: "체크포인터 없이도 `interrupt_before`를 쓸 수 있다"고 생각하기 쉬운데, 실제로는 **체크포인터가 필수**입니다. 인터럽트는 현재 상태를 저장한 뒤 실행을 멈추는 것이기 때문에, 저장할 곳(체크포인터)이 없으면 `ValueError`가 발생합니다.

**체크포인터 종류와 선택 기준**:

| 체크포인터 | 용도 | 설치 |
|-----------|------|------|
| `MemorySaver` | 개발/테스트, 프로토타이핑 | `langgraph` 내장 |
| `SqliteSaver` | 단일 서버, 경량 프로덕션 | `pip install langgraph-checkpoint-sqlite` |
| `PostgresSaver` | 프로덕션, 멀티 서버 | `pip install langgraph-checkpoint-postgres` |

### 개념 6: LangGraph Studio — 시각적 디버깅 IDE

> 💡 **비유**: 코드 디버깅에 VS Code의 브레이크포인트가 있다면, 에이전트 디버깅에는 LangGraph Studio가 있습니다. 그래프 실행을 한 노드씩 멈추고, 상태를 수정하고, 다시 실행할 수 있는 전용 IDE입니다.

LangGraph Studio는 LangChain 팀이 만든 **세계 최초의 에이전트 IDE**입니다. 2024년 8월에 처음 공개되었으며, 에이전트의 시각화, 인터랙션, 디버깅을 위한 전용 도구입니다.

**핵심 기능**:

1. **실시간 그래프 시각화**: 코드에서 정의한 그래프가 자동으로 시각적 다이어그램으로 표시되며, 코드를 수정하면 다이어그램이 즉시 업데이트됩니다.

2. **단계별 실행(Step-through)**: 앞서 배운 `interrupt_before`와 `interrupt_after`를 GUI에서 클릭 한 번으로 설정할 수 있습니다. 각 노드에서 실행을 일시 정지하고, 해당 시점의 상태를 확인할 수 있죠.

3. **시간 여행(Time Travel)**: 체크포인터에 저장된 이전 단계로 돌아가서 상태를 변경한 후 다시 실행할 수 있습니다. "만약 이 노드에서 다른 결과가 나왔다면?"을 쉽게 실험할 수 있죠.

4. **상태 편집(State Editing)**: `update_state()`를 GUI에서 직접 수행할 수 있습니다. 실행 중인 그래프의 상태를 시각적으로 수정하여 특정 조건을 만들어 테스트하기에 유용합니다.

## 실습: 직접 해보기

이제 지금까지 배운 모든 것을 종합하여, 시각화 가능하고, 디버깅 가능하며, 테스트 가능하고, **사람의 승인까지 받을 수 있는** 에이전트를 만들어 보겠습니다.

```python
"""
종합 실습: 시각화/디버깅/테스트/Human-in-the-Loop이 가능한 고객 지원 에이전트
"""
from typing import TypedDict, Annotated, Literal
from langgraph.graph import StateGraph, START, END
from langgraph.checkpoint.memory import MemorySaver
import operator
import json

# ── 1단계: 상태 정의 ──
class CustomerState(TypedDict):
    query: str                                       # 고객 질문
    category: str                                    # 분류 결과
    priority: str                                    # 우선순위
    response: str                                    # 응답
    approved: bool                                   # 사람 승인 여부
    trace: Annotated[list[str], operator.add]         # 실행 추적 로그

# ── 2단계: 노드 함수 정의 ──
def classify_node(state: CustomerState) -> dict:
    """질문을 카테고리로 분류"""
    query = state["query"].lower()
    if any(kw in query for kw in ["에러", "버그", "오류", "작동", "안됨"]):
        category = "technical"
    elif any(kw in query for kw in ["환불", "결제", "가격", "구독"]):
        category = "billing"
    else:
        category = "general"
    return {
        "category": category,
        "trace": [f"classify → {category}"]
    }

def prioritize_node(state: CustomerState) -> dict:
    """카테고리에 따라 우선순위 결정"""
    priority_map = {
        "technical": "high",
        "billing": "medium",
        "general": "low"
    }
    priority = priority_map.get(state["category"], "low")
    return {
        "priority": priority,
        "trace": [f"prioritize → {priority}"]
    }

def handle_technical(state: CustomerState) -> dict:
    """기술 문제 처리"""
    return {
        "response": f"[기술지원] '{state['query']}'에 대한 기술 지원을 시작합니다. "
                    f"우선순위: {state['priority']}",
        "trace": ["handle → technical"]
    }

def handle_billing(state: CustomerState) -> dict:
    """결제 문제 처리"""
    return {
        "response": f"[결제지원] '{state['query']}'에 대해 결제팀이 확인합니다.",
        "trace": ["handle → billing"]
    }

def handle_general(state: CustomerState) -> dict:
    """일반 문의 처리"""
    return {
        "response": f"[일반안내] '{state['query']}'에 대해 안내드립니다.",
        "trace": ["handle → general"]
    }

# ── 3단계: 라우팅 함수 ──
def route_by_category(state: CustomerState) -> Literal[
    "handle_technical", "handle_billing", "handle_general"
]:
    """카테고리에 따라 적절한 처리 노드로 라우팅"""
    route_map = {
        "technical": "handle_technical",
        "billing": "handle_billing",
        "general": "handle_general"
    }
    return route_map.get(state["category"], "handle_general")

# ── 4단계: 그래프 조립 ──
def build_customer_graph(checkpointer=None, interrupt_before=None):
    """고객 지원 그래프 팩토리"""
    builder = StateGraph(CustomerState)

    # 노드 등록
    builder.add_node("classify", classify_node)
    builder.add_node("prioritize", prioritize_node)
    builder.add_node("handle_technical", handle_technical)
    builder.add_node("handle_billing", handle_billing)
    builder.add_node("handle_general", handle_general)

    # 엣지 연결
    builder.add_edge(START, "classify")
    builder.add_edge("classify", "prioritize")
    builder.add_conditional_edges("prioritize", route_by_category)
    builder.add_edge("handle_technical", END)
    builder.add_edge("handle_billing", END)
    builder.add_edge("handle_general", END)

    return builder.compile(
        checkpointer=checkpointer,
        interrupt_before=interrupt_before or []
    )

# ── 5단계: 시각화 ──
graph = build_customer_graph()

# Mermaid 다이어그램 출력
print("📊 그래프 구조:")
print(graph.get_graph().draw_mermaid())
print()

# PNG 파일로 저장 (선택)
# png_data = graph.get_graph().draw_mermaid_png()
# with open("customer_support_graph.png", "wb") as f:
#     f.write(png_data)

# ── 6단계: 실행 & 디버깅 ──
test_queries = [
    "로그인이 안됩니다. 에러가 계속 나요",
    "구독 환불 요청합니다",
    "서비스 이용 방법을 알고 싶어요"
]

print("🔍 실행 추적 (updates 모드):")
print("-" * 50)

for query in test_queries:
    print(f"\n질문: \"{query}\"")

    # updates 모드로 각 노드의 변경분 추적
    visited = []
    for chunk in graph.stream(
        {"query": query, "trace": []},
        stream_mode="updates"
    ):
        for node_name, update in chunk.items():
            visited.append(node_name)
            if "trace" in update:
                print(f"  → {update['trace'][-1]}")

    # 최종 결과
    result = graph.invoke({"query": query, "trace": []})
    print(f"  📝 응답: {result['response']}")
    print(f"  📍 경로: {' → '.join(visited)}")

# ── 7단계: 디버그 모드 상세 추적 ──
print("\n\n🐛 디버그 모드 상세 추적:")
print("=" * 50)

for event in graph.stream(
    {"query": "결제 에러 발생", "trace": []},
    stream_mode="debug"
):
    event_type = event.get("type", "")
    payload = event.get("payload", {})

    if event_type == "task":
        node = payload.get("name", "?")
        print(f"\n▶️  [{node}] 실행 시작")
    elif event_type == "task_result":
        node = payload.get("name", "?")
        results = payload.get("result", [])
        print(f"✅ [{node}] 완료")
        for r in results:
            if "trace" in r:
                print(f"   추적: {r['trace']}")
            if "response" in r:
                print(f"   응답: {r['response']}")

# ── 8단계: Human-in-the-Loop — 고우선순위 응답 전 사람 승인 ──
print("\n\n⏸️  Human-in-the-Loop (결제/기술 응답 전 승인):")
print("=" * 50)

checkpointer = MemorySaver()
hitl_graph = build_customer_graph(
    checkpointer=checkpointer,
    # 실제 응답을 보내기 전에 사람이 확인하도록 설정
    interrupt_before=["handle_technical", "handle_billing"]
)
config = {"configurable": {"thread_id": "hitl-demo-1"}}

# 실행 → 분류/우선순위 결정 후, 처리 노드 전에서 멈춤
print("\n🚀 '버그 리포트' 처리 시작...")
for chunk in hitl_graph.stream(
    {"query": "버그 리포트", "trace": []},
    config=config,
    stream_mode="updates"
):
    for node, update in chunk.items():
        print(f"  ✅ {node} 완료: {update.get('trace', [''][-1])}")

# 현재 상태 확인 — 사람이 검토
state = hitl_graph.get_state(config)
print(f"\n⏸️  일시 정지됨!")
print(f"  카테고리: {state.values.get('category')}")
print(f"  우선순위: {state.values.get('priority')}")
print(f"  다음 실행 노드: {state.next}")

# 사람이 확인 후 승인 → 실행 재개
print(f"\n  👤 관리자: 카테고리와 우선순위 확인 완료, 승인합니다.")
print("\n▶️  실행 재개...")
for chunk in hitl_graph.stream(None, config=config, stream_mode="updates"):
    for node, update in chunk.items():
        print(f"  ✅ {node} 완료")
        if "response" in update:
            print(f"     응답: {update['response']}")

# 체크포인트 히스토리 확인
print("\n📜 실행 히스토리:")
for saved in hitl_graph.get_state_history(config):
    status = saved.values.get("status", saved.values.get("category", "시작"))
    print(f"  • {status} → 다음: {saved.next or '완료'}")

final_state = hitl_graph.get_state(config)
print(f"\n📋 최종 응답: {final_state.values.get('response')}")
print(f"📍 전체 추적: {final_state.values.get('trace')}")
```

## 더 깊이 알아보기

### Mermaid의 탄생 — 코드로 다이어그램을 그리다

LangGraph의 시각화가 **Mermaid**라는 도구를 기반으로 한다는 건 이미 알고 계시죠. Mermaid는 2014년 스웨덴 개발자 **Knut Sveidqvist**가 만든 오픈소스 프로젝트입니다. 그는 기술 문서에 다이어그램을 넣을 때마다 별도의 도구를 열어 그림을 그리고, 수정할 때마다 다시 그리는 번거로움에 지쳤거든요.

"마크다운으로 텍스트를 쓰듯, 코드로 다이어그램을 그릴 수 있으면 좋겠다"는 발상에서 Mermaid가 탄생했습니다. 현재 GitHub가 마크다운에서 Mermaid를 네이티브로 지원할 정도로 성장했고, LangGraph 팀도 이 도구를 선택해 그래프 시각화의 기본 포맷으로 사용하고 있습니다.

### Human-in-the-Loop의 역사 — 사람과 기계의 협업

Human-in-the-Loop(HITL)라는 개념은 AI 시대에 새로 생긴 게 아닙니다. 1950~60년대 초기 컴퓨터 시스템에서부터 "사람이 최종 판단을 내리는" 패턴이 존재했는데요. 특히 1960년대 NASA의 우주 프로그램에서 **반자동 유도 시스템**이 대표적입니다. 아폴로 우주선의 컴퓨터가 경로를 계산하되, 최종 점화 명령은 반드시 우주비행사가 승인해야 했죠.

LangGraph가 이 패턴을 도입한 건 2024년인데, AI 에이전트가 점점 자율적으로 행동하게 되면서 "위험한 작업 전에 사람이 확인한다"는 원칙이 더욱 중요해졌기 때문입니다. `interrupt_before`라는 간단한 파라미터 하나로 수십 년 된 안전 원칙을 구현할 수 있다니, 놀랍지 않나요?

### 에이전트 IDE의 시대 — LangGraph Studio

2024년 8월, LangChain 팀은 LangGraph Studio를 "**세계 최초의 에이전트 IDE**"로 발표했습니다. 전통적인 소프트웨어 개발에는 VS Code나 IntelliJ 같은 강력한 IDE가 있었지만, 에이전트 개발에는 그런 도구가 없었거든요. LangGraph Studio는 그래프 모드(Graph Mode)와 채팅 모드(Chat Mode)를 제공하여, 개발자가 에이전트의 내부를 들여다보면서 동시에 최종 사용자 관점에서도 테스트할 수 있도록 설계되었습니다.

이후 **LangGraph Studio Web** 버전도 출시되어 브라우저에서 바로 사용할 수 있게 되었으며, LangSmith와 통합되어 트레이싱, 평가, 프롬프트 관리까지 하나의 워크플로우로 연결됩니다.

## 흔한 오해와 팁

> ⚠️ **흔한 오해**: "`debug` 모드를 프로덕션에서 써도 괜찮다"고 생각하는 분이 있습니다. `debug` 모드는 노드 진입/종료, 전후 상태, 타임스탬프 등 **엄청난 양의 데이터**를 생성합니다. 프로덕션 환경에서는 `updates`나 `values` 모드를 사용하고, `debug`는 개발/스테이징 환경에서만 활성화하세요.

> 💡 **알고 계셨나요?**: `draw_mermaid_png()`는 기본적으로 **Mermaid.ink API**라는 외부 서비스를 호출합니다. 보안이 중요한 환경에서는 그래프 구조가 외부로 전송될 수 있으므로, `draw_mermaid()`로 소스 코드만 얻어서 로컬에서 렌더링하거나, `pyppeteer` 패키지를 설치하여 로컬 렌더링을 사용할 수 있습니다: `pip install pyppeteer`.

> ⚠️ **흔한 오해**: "`MemorySaver`를 프로덕션에서 써도 된다"고 생각할 수 있는데, `MemorySaver`는 **인메모리** 저장이라 프로세스가 종료되면 모든 체크포인트가 사라집니다. 프로덕션에서는 반드시 `SqliteSaver`나 `PostgresSaver`를 사용하세요. `MemorySaver`는 개발과 테스트 용도로만 적합합니다.

> 🔥 **실무 팁**: 그래프 테스트에서 가장 효과적인 패턴은 **팩토리 함수**입니다. `build_graph()` 같은 함수로 그래프를 생성하면, 테스트마다 독립적인 그래프 인스턴스를 사용할 수 있어 테스트 간 상태 오염을 방지합니다. 또한 `stream_mode="updates"`로 실행하면 어떤 노드가 실행되었는지 추적할 수 있어, 플로우 경로 검증이 간단해집니다.

> 🔥 **실무 팁**: Human-in-the-Loop에서 `interrupt_before`와 `interrupt_after`를 어떻게 선택할까요? 일반적으로 **위험한 작업 전에 승인**을 받으려면 `interrupt_before`를, **결과를 확인한 후 다음으로 넘어갈지 결정**하려면 `interrupt_after`를 사용합니다. 예를 들어, 이메일 발송 노드 전에는 `interrupt_before`로 내용을 확인하고, 검색 결과 노드 후에는 `interrupt_after`로 결과가 충분한지 판단하는 식이죠.

## 핵심 정리

| 개념 | 설명 |
|------|------|
| `get_graph().draw_mermaid()` | 컴파일된 그래프를 Mermaid 다이어그램 소스 코드로 반환 |
| `draw_mermaid_png()` | 그래프를 PNG 이미지 바이트로 반환 (Mermaid.ink API 사용) |
| `print_ascii()` | 그래프를 ASCII 아트로 터미널에 출력 |
| `stream_mode="values"` | 매 노드 실행 후 **전체 상태**를 스트리밍 |
| `stream_mode="updates"` | 각 노드가 반환한 **변경분(delta)**만 스트리밍 |
| `stream_mode="debug"` | 노드 진입/종료, 전후 상태 등 **상세 실행 추적** |
| `stream_mode="messages"` | LLM **토큰 단위** 실시간 스트리밍 |
| `stream_mode="custom"` | 사용자 정의 이벤트 스트리밍 |
| `astream_events()` | 비동기 환경에서 세밀한 이벤트 추적 |
| 노드 단위 테스트 | 노드 함수를 독립적으로 입력/출력 검증 |
| 플로우 경로 테스트 | 전체 그래프 실행 후 올바른 노드 경로 검증 |
| `MemorySaver` | 인메모리 체크포인터, 개발/테스트용 상태 저장 |
| `SqliteSaver` / `PostgresSaver` | 프로덕션용 영속 체크포인터 |
| `interrupt_before/after` | 특정 노드 전후에 실행을 일시 정지 |
| `get_state()` / `get_state_history()` | 저장된 상태 조회 및 히스토리 탐색 |
| `update_state()` | 인터럽트 중 상태를 수동으로 수정 |
| Human-in-the-Loop | 에이전트 실행 중 사람의 승인/수정을 받는 패턴 |
| LangGraph Studio | 시각적 디버깅을 위한 전용 에이전트 IDE |
