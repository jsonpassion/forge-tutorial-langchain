# 조건부 엣지와 라우팅

> LangGraph의 `add_conditional_edges`를 활용하여 상태에 따라 동적으로 실행 흐름을 분기하는 방법을 배웁니다.

## 개요

이 섹션에서는 LangGraph의 가장 강력한 기능 중 하나인 **조건부 엣지(Conditional Edge)**를 학습합니다. 앞서 [13.2: 첫 번째 LangGraph 에이전트](./02-첫-번째-langgraph-에이전트.md)에서 `add_edge`로 노드를 순차적으로 연결했다면, 이번에는 **런타임 상태에 따라 다음 실행 경로를 동적으로 결정**하는 방법을 배웁니다.

**선수 지식**: 
- Session 13.1에서 배운 StateGraph, 노드, 엣지의 기본 개념
- Session 13.2에서 배운 TypedDict 상태 스키마, `add_node`, `add_edge`, `compile`, `invoke`

**학습 목표**:
- `add_conditional_edges`의 시그니처와 매개변수를 이해한다
- 라우팅 함수를 작성하여 상태 기반 분기를 구현할 수 있다
- `should_continue` 패턴으로 에이전트 루프를 구축할 수 있다
- START 노드에서의 조건부 진입점을 설정할 수 있다

## 왜 알아야 할까?

현실 세계의 워크플로우는 거의 항상 **"만약 ~라면"**이라는 조건을 포함합니다. 고객 문의가 들어왔을 때 결제 관련이면 결제팀으로, 기술 문제면 기술팀으로 보내야 하죠. AI 에이전트도 마찬가지입니다. LLM이 도구를 호출하기로 결정했으면 도구 실행 노드로, 최종 답변을 생성했으면 종료로 보내야 합니다.

`add_edge`만으로는 이런 동적 분기가 불가능합니다. A → B → C처럼 정해진 순서로만 흐를 수 있거든요. **조건부 엣지**는 이 한계를 깨고, 그래프에 **"지능"**을 부여합니다. 실제로 LangGraph 기반 에이전트의 거의 모든 실전 패턴 — ReAct 루프, 멀티 에이전트 라우팅, 사람 승인(Human-in-the-Loop) — 이 조건부 엣지 위에 세워져 있습니다.

## 핵심 개념

### 개념 1: 조건부 엣지란?

> 💡 **비유**: 일반 엣지(`add_edge`)가 고속도로의 **직선 구간**이라면, 조건부 엣지(`add_conditional_edges`)는 **나들목(IC)**과 같습니다. 나들목에 도착하면 내비게이션(라우팅 함수)이 현재 목적지(상태)를 확인하고, "서울 방면으로 가세요" 또는 "부산 방면으로 가세요"라고 안내하죠. 어디로 갈지는 **달리고 있는 그 순간**에 결정됩니다.

조건부 엣지는 **소스 노드**가 실행을 마친 후, **라우팅 함수**가 현재 상태를 검사하여 다음에 실행할 노드를 동적으로 결정하는 메커니즘입니다. 일반 엣지와의 핵심 차이를 비교해 보겠습니다.

| 구분 | 일반 엣지 (`add_edge`) | 조건부 엣지 (`add_conditional_edges`) |
|------|------------------------|---------------------------------------|
| 다음 노드 결정 시점 | 그래프 정의 시 (정적) | 실행 시 (동적) |
| 분기 가능 여부 | 단일 경로만 가능 | 여러 경로로 분기 가능 |
| 추가 필요 요소 | 없음 | 라우팅 함수 필요 |
| 사용 예 | 전처리 → 메인 로직 | LLM 판단에 따른 도구 호출 / 종료 |

### 개념 2: `add_conditional_edges` API

`add_conditional_edges` 메서드의 시그니처를 살펴보겠습니다.

```python
graph.add_conditional_edges(
    source: str,           # 소스 노드 이름 (또는 START)
    path: Callable,        # 라우팅 함수
    path_map: dict | list | None = None  # (선택) 반환값 → 노드 매핑
)
```

세 번째 매개변수 `path_map`은 세 가지 형태를 받을 수 있습니다:

**1. 딕셔너리 매핑** — 라우팅 함수의 반환값을 다른 노드 이름으로 매핑할 때:

```python
# 라우팅 함수가 True/False를 반환하면, 각각 다른 노드로 매핑
graph.add_conditional_edges(
    "classifier",          # 소스 노드
    classify_intent,       # 라우팅 함수 (True 또는 False 반환)
    {True: "handle_positive", False: "handle_negative"}  # 매핑
)
```

**2. 리스트** — 라우팅 함수가 직접 노드 이름을 반환할 때:

```python
# 라우팅 함수가 "tool_node" 또는 END를 직접 반환
graph.add_conditional_edges(
    "llm_call",
    should_continue,
    ["tool_node", END]     # 가능한 목적지 목록
)
```

**3. 생략(None)** — 라우팅 함수의 반환값이 곧 노드 이름일 때:

```python
# 라우팅 함수가 반환하는 문자열이 그대로 노드 이름으로 사용됨
graph.add_conditional_edges(
    "router",
    route_by_category      # "billing", "technical", "general" 등을 반환
)
```

> ⚠️ **흔한 오해**: `path_map`을 딕셔너리로 줄 때, 라우팅 함수의 반환값이 딕셔너리의 **키**와 일치해야 합니다. **값(노드 이름)**이 아닌 **키**를 반환해야 한다는 점을 혼동하기 쉬워요. 예를 들어 `{True: "node_a"}`라면, 라우팅 함수는 `"node_a"`가 아니라 `True`를 반환해야 합니다.

### 개념 3: 라우팅 함수 작성법

> 💡 **비유**: 라우팅 함수는 병원의 **접수 데스크 직원**과 같습니다. 환자(상태)가 접수 데스크에 도착하면, 직원이 증상을 확인하고 "내과로 가세요", "외과로 가세요", "응급실로 가세요"라고 안내하죠. 직원은 환자의 상태만 읽을 뿐, 직접 치료하지는 않습니다.

라우팅 함수는 몇 가지 규칙을 따라야 합니다:

1. **상태를 입력으로 받는다** — 그래프의 현재 상태(TypedDict)를 매개변수로 받습니다
2. **문자열(또는 Hashable 값)을 반환한다** — 다음 노드를 가리키는 값을 반환합니다
3. **상태를 변경하지 않는다** — 라우팅 함수는 읽기 전용이며, 상태를 수정해서는 안 됩니다

```python
from typing import Literal
from langgraph.graph import END

def route_by_category(state: dict) -> Literal["billing", "technical", "general"]:
    """사용자 문의 카테고리에 따라 라우팅"""
    category = state.get("category", "general")  # 상태에서 카테고리 읽기
    
    if category == "billing":
        return "billing"       # 결제 처리 노드로
    elif category == "technical":
        return "technical"     # 기술 지원 노드로
    else:
        return "general"       # 일반 응대 노드로
```

`typing.Literal` 타입 힌트를 사용하면 반환 가능한 값을 명시할 수 있어서, IDE 자동완성과 LangGraph의 그래프 시각화에도 도움이 됩니다.

### 개념 4: `should_continue` 패턴 — 에이전트 루프의 핵심

> 💡 **비유**: 이 패턴은 **요리사의 간보기**와 같습니다. 요리사가 요리를 한 단계 진행한 후 맛을 봅니다. "좀 더 필요하다"면 양념을 추가하고(도구 호출), "완성됐다"면 접시에 담습니다(END). 이 "요리 → 간보기 → 계속/완료" 사이클이 바로 에이전트 루프입니다.

LangGraph 에이전트에서 가장 많이 사용되는 패턴이 바로 `should_continue`입니다. LLM이 도구를 호출했는지 확인하여 루프를 계속할지 종료할지를 결정하죠.

```python
from typing import Literal
from langgraph.graph import END, MessagesState

def should_continue(state: MessagesState) -> Literal["tool_node", "__end__"]:
    """LLM이 도구를 호출했으면 도구 노드로, 아니면 종료"""
    messages = state["messages"]
    last_message = messages[-1]  # 가장 최근 메시지 확인
    
    # LLM이 도구 호출을 요청했는가?
    if last_message.tool_calls:
        return "tool_node"  # 도구 실행 노드로 라우팅
    
    # 도구 호출이 없으면 최종 응답 — 종료
    return END
```

이 패턴으로 구성되는 전형적인 에이전트 루프 구조는 다음과 같습니다:

```
[START] → [LLM 호출] → (should_continue 판단)
                              ├─ tool_calls 있음 → [도구 실행] → [LLM 호출] (루프)
                              └─ tool_calls 없음 → [END]
```

이 루프는 LLM이 더 이상 도구를 호출하지 않을 때까지 계속 돌아갑니다. 이것이 바로 [Session 12에서 배운](12-에이전트agent-기초/01-에이전트-개념과-react-패턴.md) ReAct 패턴의 LangGraph 구현입니다.

### 개념 5: START에서의 조건부 진입

`add_conditional_edges`는 일반 노드뿐 아니라, 가상의 **START** 노드에서도 사용할 수 있습니다. 이를 통해 입력 상태에 따라 **그래프의 첫 번째 실행 노드를 동적으로 결정**할 수 있죠.

```python
from langgraph.graph import START

# START에서 조건부 분기 — 입력에 따라 다른 경로로 시작
graph.add_conditional_edges(
    START,                    # 가상 시작 노드
    route_by_input_type,      # 입력 타입을 판단하는 라우팅 함수
    {
        "text": "process_text",
        "image": "process_image",
        "audio": "process_audio"
    }
)
```

이 패턴은 하나의 그래프가 **다양한 유형의 입력**을 처리해야 할 때 특히 유용합니다.

## 실습: 직접 해보기

고객 지원 라우터를 만들어 봅시다. 사용자 문의를 분석하여 적절한 전문 노드로 라우팅하고, 복잡한 문의는 에스컬레이션하는 워크플로우입니다.

```python
from typing import TypedDict, Literal, Annotated
from langgraph.graph import StateGraph, START, END
import operator

# ──────────────────────────────────────────────
# 1단계: 상태 스키마 정의
# ──────────────────────────────────────────────
class SupportState(TypedDict):
    query: str                          # 사용자 문의 내용
    category: str                       # 분류된 카테고리
    response: str                       # 최종 응답
    escalated: bool                     # 에스컬레이션 여부
    log: Annotated[list[str], operator.add]  # 처리 로그 (리듀서로 누적)


# ──────────────────────────────────────────────
# 2단계: 노드 함수 정의
# ──────────────────────────────────────────────
def classify_query(state: SupportState) -> dict:
    """사용자 문의를 카테고리로 분류하는 노드"""
    query = state["query"].lower()
    
    # 간단한 키워드 기반 분류 (실전에서는 LLM 사용)
    if any(word in query for word in ["결제", "환불", "가격", "구독"]):
        category = "billing"
    elif any(word in query for word in ["오류", "버그", "안됨", "에러", "설치"]):
        category = "technical"
    elif any(word in query for word in ["화남", "최악", "소송", "신고"]):
        category = "escalation"
    else:
        category = "general"
    
    return {
        "category": category,
        "log": [f"[분류] 카테고리: {category}"]
    }


def handle_billing(state: SupportState) -> dict:
    """결제 관련 문의 처리 노드"""
    return {
        "response": f"결제팀입니다. '{state['query']}'에 대해 확인하겠습니다. 영업일 기준 1-2일 내 처리됩니다.",
        "escalated": False,
        "log": ["[결제팀] 문의 처리 완료"]
    }


def handle_technical(state: SupportState) -> dict:
    """기술 지원 문의 처리 노드"""
    return {
        "response": f"기술팀입니다. '{state['query']}'에 대해 조사하겠습니다. 로그를 확인 중입니다.",
        "escalated": False,
        "log": ["[기술팀] 문의 처리 완료"]
    }


def handle_general(state: SupportState) -> dict:
    """일반 문의 처리 노드"""
    return {
        "response": f"안녕하세요! '{state['query']}'에 대한 답변입니다. 추가 문의 사항이 있으시면 말씀해 주세요.",
        "escalated": False,
        "log": ["[일반] 문의 처리 완료"]
    }


def handle_escalation(state: SupportState) -> dict:
    """에스컬레이션 처리 노드"""
    return {
        "response": f"고객님의 불편에 진심으로 사과드립니다. 담당 매니저가 직접 연락드리겠습니다.",
        "escalated": True,
        "log": ["[에스컬레이션] 매니저에게 전달됨"]
    }


def finalize(state: SupportState) -> dict:
    """최종 응답 정리 노드"""
    suffix = " (에스컬레이션됨 — 매니저 검토 예정)" if state.get("escalated") else ""
    return {
        "response": state["response"] + suffix,
        "log": ["[완료] 최종 응답 생성"]
    }


# ──────────────────────────────────────────────
# 3단계: 라우팅 함수 정의
# ──────────────────────────────────────────────
def route_by_category(state: SupportState) -> Literal["billing", "technical", "general", "escalation"]:
    """카테고리에 따라 적절한 처리 노드로 라우팅"""
    return state["category"]  # 카테고리 값이 곧 노드 이름


# ──────────────────────────────────────────────
# 4단계: 그래프 조립
# ──────────────────────────────────────────────
graph = StateGraph(SupportState)

# 노드 등록
graph.add_node("classify", classify_query)
graph.add_node("billing", handle_billing)
graph.add_node("technical", handle_technical)
graph.add_node("general", handle_general)
graph.add_node("escalation", handle_escalation)
graph.add_node("finalize", finalize)

# 엣지 연결
graph.add_edge(START, "classify")          # 시작 → 분류

# 핵심: 조건부 엣지로 동적 분기!
graph.add_conditional_edges(
    "classify",                             # 소스 노드
    route_by_category,                      # 라우팅 함수
    {                                       # 반환값 → 노드 매핑
        "billing": "billing",
        "technical": "technical",
        "general": "general",
        "escalation": "escalation"
    }
)

# 각 처리 노드 → 최종 정리 → 종료
graph.add_edge("billing", "finalize")
graph.add_edge("technical", "finalize")
graph.add_edge("general", "finalize")
graph.add_edge("escalation", "finalize")
graph.add_edge("finalize", END)

# 그래프 컴파일
app = graph.compile()


# ──────────────────────────────────────────────
# 5단계: 실행 및 결과 확인
# ──────────────────────────────────────────────
test_queries = [
    "결제가 두 번 됐어요, 환불해 주세요",
    "앱 설치 중에 오류가 발생합니다",
    "영업 시간이 어떻게 되나요?",
    "정말 최악이에요, 소송 걸겠습니다",
]

for query in test_queries:
    result = app.invoke({
        "query": query,
        "category": "",
        "response": "",
        "escalated": False,
        "log": []
    })
    
    print(f"문의: {query}")
    print(f"카테고리: {result['category']}")
    print(f"응답: {result['response']}")
    print(f"처리 로그: {' → '.join(result['log'])}")
    print("-" * 60)
```

**실행 결과:**

```
문의: 결제가 두 번 됐어요, 환불해 주세요
카테고리: billing
응답: 결제팀입니다. '결제가 두 번 됐어요, 환불해 주세요'에 대해 확인하겠습니다. 영업일 기준 1-2일 내 처리됩니다.
처리 로그: [분류] 카테고리: billing → [결제팀] 문의 처리 완료 → [완료] 최종 응답 생성
------------------------------------------------------------
문의: 앱 설치 중에 오류가 발생합니다
카테고리: technical
응답: 기술팀입니다. '앱 설치 중에 오류가 발생합니다'에 대해 조사하겠습니다. 로그를 확인 중입니다.
처리 로그: [분류] 카테고리: technical → [기술팀] 문의 처리 완료 → [완료] 최종 응답 생성
------------------------------------------------------------
문의: 영업 시간이 어떻게 되나요?
카테고리: general
응답: 안녕하세요! '영업 시간이 어떻게 되나요?'에 대한 답변입니다. 추가 문의 사항이 있으시면 말씀해 주세요.
처리 로그: [분류] 카테고리: general → [일반] 문의 처리 완료 → [완료] 최종 응답 생성
------------------------------------------------------------
문의: 정말 최악이에요, 소송 걸겠습니다
카테고리: escalation
응답: 고객님의 불편에 진심으로 사과드립니다. 담당 매니저가 직접 연락드리겠습니다. (에스컬레이션됨 — 매니저 검토 예정)
처리 로그: [분류] 카테고리: escalation → [에스컬레이션] 매니저에게 전달됨 → [완료] 최종 응답 생성
------------------------------------------------------------
```

이 실습에서 핵심은 **`add_conditional_edges`** 한 줄로 4개의 서로 다른 경로가 만들어졌다는 것입니다. 각 문의는 `classify` 노드를 거친 후, `route_by_category` 함수의 판단에 따라 적절한 전문 노드로 동적 분기됩니다.

이제 `should_continue` 패턴을 활용한 에이전트 루프도 구현해 봅시다:

```python
from typing import TypedDict, Literal, Annotated
from langgraph.graph import StateGraph, START, END
import operator
import random

# ──────────────────────────────────────────────
# 상태 정의
# ──────────────────────────────────────────────
class ResearchState(TypedDict):
    topic: str                                     # 조사 주제
    findings: Annotated[list[str], operator.add]   # 조사 결과 누적
    iteration: int                                 # 현재 반복 횟수
    max_iterations: int                            # 최대 반복 횟수
    summary: str                                   # 최종 요약


# ──────────────────────────────────────────────
# 노드 함수
# ──────────────────────────────────────────────
def research(state: ResearchState) -> dict:
    """조사를 수행하는 노드 (시뮬레이션)"""
    iteration = state["iteration"] + 1
    # 실전에서는 LLM이 검색 도구를 호출
    finding = f"[반복 {iteration}] '{state['topic']}'에 대한 새로운 발견: 자료 #{random.randint(100, 999)}"
    return {
        "findings": [finding],
        "iteration": iteration
    }


def summarize(state: ResearchState) -> dict:
    """수집된 자료를 요약하는 노드"""
    all_findings = "\n".join(state["findings"])
    return {
        "summary": f"총 {state['iteration']}회 조사 완료.\n수집된 자료:\n{all_findings}"
    }


# ──────────────────────────────────────────────
# 라우팅 함수: should_continue 패턴
# ──────────────────────────────────────────────
def should_continue_research(state: ResearchState) -> Literal["research", "__end__"]:
    """최대 반복 횟수에 도달했는지 확인하여 계속/종료 결정"""
    if state["iteration"] < state["max_iterations"]:
        return "research"  # 아직 더 조사 필요 → 루프 계속
    return END             # 충분히 조사함 → 요약 단계로


# ──────────────────────────────────────────────
# 그래프 조립
# ──────────────────────────────────────────────
graph = StateGraph(ResearchState)

graph.add_node("research", research)
graph.add_node("summarize", summarize)

graph.add_edge(START, "research")       # 시작 → 조사

# 조건부 엣지: 조사 후 계속할지 판단
graph.add_conditional_edges(
    "research",
    should_continue_research,
    {"research": "research", END: END}  # 루프 or 종료
)

# "research" → END 경로에 summarize를 끼워넣기 위해 수정
# 실제로는 should_continue가 "summarize"를 반환하도록 변경
def should_continue_v2(state: ResearchState) -> Literal["research", "summarize"]:
    """개선된 버전: 종료 대신 요약 노드로 라우팅"""
    if state["iteration"] < state["max_iterations"]:
        return "research"
    return "summarize"

# 그래프 재조립 (개선 버전)
graph2 = StateGraph(ResearchState)
graph2.add_node("research", research)
graph2.add_node("summarize", summarize)

graph2.add_edge(START, "research")
graph2.add_conditional_edges(
    "research",
    should_continue_v2,
    ["research", "summarize"]   # 리스트 형태의 path_map
)
graph2.add_edge("summarize", END)

app2 = graph2.compile()

# 실행
result = app2.invoke({
    "topic": "LangGraph 조건부 엣지",
    "findings": [],
    "iteration": 0,
    "max_iterations": 3,
    "summary": ""
})

print(result["summary"])
# 출력 예시:
# 총 3회 조사 완료.
# 수집된 자료:
# [반복 1] 'LangGraph 조건부 엣지'에 대한 새로운 발견: 자료 #472
# [반복 2] 'LangGraph 조건부 엣지'에 대한 새로운 발견: 자료 #815
# [반복 3] 'LangGraph 조건부 엣지'에 대한 새로운 발견: 자료 #263
```

두 번째 실습의 핵심은 **`research` → `research`** 자기 자신으로의 루프입니다. 조건부 엣지 덕분에 반복 횟수가 충족될 때까지 같은 노드를 재실행하고, 조건이 만족되면 `summarize`로 빠져나갑니다. 이것이 에이전트 루프의 원형입니다.

## 더 깊이 알아보기

### 상태 머신에서 LangGraph까지

조건부 엣지의 뿌리는 컴퓨터 과학의 고전인 **유한 상태 머신(Finite State Machine, FSM)**에 있습니다. 1943년 워런 매컬러(Warren McCulloch)와 월터 피츠(Walter Pitts)가 신경망 논문에서 처음 형식화한 이 개념은, 이후 앨런 튜링의 연구와 합쳐져 현대 컴퓨터 과학의 기초가 되었죠.

LangGraph의 이름 자체가 **"Lang(언어모델) + Graph(그래프)"**인 이유가 있습니다. LangChain 팀의 Harrison Chase는 2024년 초, 기존 `AgentExecutor`가 복잡한 에이전트 워크플로우를 표현하기에 너무 경직되어 있다는 커뮤니티 피드백을 받았습니다. 특히 **"조건에 따라 다른 경로를 타야 한다"**는 요구가 압도적이었죠. 이에 그래프 이론의 **방향 그래프(Directed Graph)**와 상태 머신의 **전이 함수(Transition Function)** 개념을 결합하여 LangGraph를 설계했습니다.

흥미로운 점은, `add_conditional_edges`의 설계가 React(UI 프레임워크)의 **조건부 렌더링** 패턴에서 영감을 받았다는 것입니다. "현재 상태를 보고 다음에 무엇을 보여줄지(실행할지) 결정한다"는 핵심 아이디어가 동일하죠. 다만 React는 UI 컴포넌트를 선택하고, LangGraph는 실행 노드를 선택한다는 차이가 있을 뿐입니다.

### path_map: 딕셔너리 vs 리스트 vs 생략

`path_map` 매개변수의 세 가지 형태는 각기 다른 사용 시나리오를 위해 설계되었습니다:

- **딕셔너리**: 라우팅 함수의 반환값과 노드 이름이 다를 때. 예를 들어 `True`/`False`를 반환하되 노드 이름은 `"approved"`/`"rejected"`일 때 유용합니다
- **리스트**: 반환값이 곧 노드 이름이지만, 가능한 목적지를 명시적으로 선언하고 싶을 때. LangGraph가 그래프를 검증하고 시각화하는 데 도움이 됩니다
- **생략**: 반환값이 곧 노드 이름이고, 별도의 선언이 필요 없을 때. 가장 간결하지만, 그래프 시각화 시 모든 경로가 표시되지 않을 수 있습니다

## 흔한 오해와 팁

> ⚠️ **흔한 오해**: "조건부 엣지의 라우팅 함수에서 상태를 수정해도 된다"고 생각하기 쉽습니다. 하지만 라우팅 함수는 **순수 함수(pure function)**여야 합니다. 상태를 읽기만 하고, 다음 노드를 가리키는 값만 반환해야 합니다. 상태 변경은 반드시 노드 함수에서만 이뤄져야 합니다. 라우팅 함수에서 상태를 수정하면, 예측 불가능한 동작이 발생할 수 있습니다.

> 💡 **알고 계셨나요?**: `add_conditional_edges`의 라우팅 함수가 **리스트**를 반환하면, 해당 노드들이 **병렬로 실행**됩니다! 예를 들어 `["search", "analyze"]`를 반환하면 두 노드가 동시에 실행되는 "팬아웃(fan-out)" 패턴이 됩니다. 이를 활용하면 독립적인 작업을 병렬화하여 전체 실행 시간을 크게 줄일 수 있습니다.

> 🔥 **실무 팁**: 라우팅 함수에는 반드시 `typing.Literal` 타입 힌트를 사용하세요. `def route(state) -> Literal["a", "b", "c"]`처럼요. 이렇게 하면 세 가지 이점이 있습니다: (1) IDE에서 오타를 잡아줍니다, (2) `path_map`을 생략해도 LangGraph가 가능한 경로를 추론합니다, (3) `graph.get_graph().draw_mermaid()`로 시각화할 때 모든 분기가 정확히 표시됩니다.

> 🔥 **실무 팁**: 디버깅 시 `stream` 모드를 활용하면 조건부 엣지의 라우팅 결과를 실시간으로 추적할 수 있습니다. `for event in app.stream(input):`으로 실행하면 각 노드의 실행 순서와 상태 변화를 단계별로 확인할 수 있어, 라우팅이 의도대로 작동하는지 검증하기 좋습니다.

## 핵심 정리

| 개념 | 설명 |
|------|------|
| `add_conditional_edges` | 소스 노드의 실행 후 라우팅 함수를 통해 다음 노드를 동적으로 결정하는 메서드 |
| 라우팅 함수 | 상태를 입력받아 다음 노드를 가리키는 값을 반환하는 순수 함수 |
| `path_map` | 라우팅 함수의 반환값을 노드 이름으로 매핑하는 딕셔너리, 리스트, 또는 None |
| `should_continue` 패턴 | LLM의 도구 호출 여부를 확인하여 루프를 계속/종료하는 에이전트의 핵심 패턴 |
| START 조건부 진입 | `add_conditional_edges(START, ...)`으로 입력에 따라 첫 실행 노드를 동적 결정 |
| `Literal` 타입 힌트 | 라우팅 함수의 반환 타입을 명시하여 검증·시각화·자동완성을 돕는 타입 어노테이션 |
| 에이전트 루프 | 조건부 엣지로 자기 자신을 가리켜 LLM → 도구 → LLM 사이클을 반복하는 구조 |

## 다음 섹션 미리보기

지금까지 조건부 엣지로 그래프의 **분기(branching)**를 배웠습니다. 하지만 실제 에이전트에서는 LLM이 웹 검색, 계산기, 데이터베이스 조회 같은 **도구(Tool)**를 호출해야 하죠. 다음 섹션에서는 LangGraph에 **도구 노드(Tool Node)**를 통합하고, `should_continue` 패턴과 결합하여 **실제 LLM 기반의 ReAct 에이전트**를 LangGraph로 구현합니다. 조건부 엣지가 진짜 힘을 발휘하는 순간을 만나게 될 것입니다.
