## 왜 알아야 할까?

### 지금까지의 상태 관리, 돌아보기

잠깐, 지금까지 우리가 만든 에이전트들의 상태를 떠올려 봅시다:

```python
# Session 13.2에서 만든 첫 에이전트의 상태
class SimpleState(TypedDict):
    messages: Annotated[list, add_messages]
    name: str

# Session 13.4에서 쓴 도구 호출 에이전트의 상태
class MessagesState(TypedDict):
    messages: Annotated[list[AnyMessage], add_messages]
```

사실 지금까지는 `messages` 하나면 충분했습니다. 사용자 메시지가 들어오고, LLM이 응답하고, 도구를 호출하고, 그 결과가 다시 메시지로 돌아오는 — 모든 게 `messages` 리스트 하나로 해결됐죠. `add_messages` 리듀서가 알아서 메시지를 쌓아줬고, 우리는 별 걱정 없이 `MessagesState`만 상속하면 됐습니다.

그런데 **실제 프로덕션 에이전트를 만들다 보면**, 이렇게 단순한 상태로는 부족한 순간이 찾아옵니다. 예를 들어볼까요?

- 병렬로 실행되는 3개의 노드가 **동시에 같은 상태 키를 업데이트**하면 어떻게 될까요?
- 대화가 100턴 넘어가면서 **토큰 비용이 폭발**할 때, 오래된 메시지를 어떻게 정리할까요?
- 에이전트가 **검색 횟수, 에러 카운터, 중간 결과물**을 동시에 추적해야 한다면요?

### 기본 패턴의 한계를 직접 느껴봅시다

Session 13.4의 도구 호출 에이전트를 떠올려 보세요. 에이전트가 도구를 3번 호출했다고 합시다. "도구를 몇 번 호출했지?"라는 단순한 질문에 답하려면 어떻게 해야 할까요? 메시지 리스트를 직접 순회하면서 `ToolMessage`를 세야 합니다. 상태에 `tool_call_count` 같은 필드가 있었다면 훨씬 간단했을 텐데요.

```python
# ❌ 메시지만으로는 불편한 경우
def check_tool_usage(state: MessagesState) -> dict:
    # 메시지를 전부 뒤져서 도구 호출 횟수를 세야 함
    tool_calls = sum(
        1 for m in state["messages"]
        if hasattr(m, "tool_calls") and m.tool_calls
    )
    if tool_calls >= 5:
        return {"messages": [AIMessage(content="도구 호출 한도에 도달했습니다.")]}
    return {"messages": []}

# ✅ 상태 필드로 추적하면 깔끔
class BetterState(MessagesState):
    tool_call_count: Annotated[int, operator.add]  # 자동으로 누적!

def check_tool_usage(state: BetterState) -> dict:
    if state["tool_call_count"] >= 5:  # 한 줄로 확인
        return {"messages": [AIMessage(content="도구 호출 한도에 도달했습니다.")]}
    return {"messages": []}
```

차이가 느껴지시나요? `operator.add` 리듀서 하나만 붙였을 뿐인데, 매번 메시지를 순회할 필요 없이 **상태를 한 번 읽으면 끝**입니다. 이런 편리함이 리듀서의 진짜 가치죠.

이 모든 질문의 답이 바로 **상태 관리 심화**에 있습니다. 리듀서를 제대로 이해하면, 노드 간 데이터 충돌 없이 복잡한 워크플로우를 안전하게 구축할 수 있고, 메시지 관리 전략으로 토큰 비용을 제어할 수 있으며, 커스텀 상태 스키마로 어떤 복잡한 비즈니스 로직도 표현할 수 있게 됩니다.

## 핵심 개념

### 개념 0: 기본 상태 패턴 리뷰 — 어디까지 왔나?

본격적인 심화에 들어가기 전에, 지금까지 배운 상태 패턴을 한 곳에 모아 정리해 봅시다. 이미 아는 내용이지만, **정확히 어떤 메커니즘이 동작하고 있었는지** 의식하면서 보면 앞으로의 내용이 훨씬 잘 들어옵니다.

```python
from typing import TypedDict, Annotated
from langchain_core.messages import AnyMessage
from langgraph.graph.message import add_messages
import operator

# 패턴 1: 단순 필드 (리듀서 없음) — Session 13.2
class BasicState(TypedDict):
    name: str           # 노드가 반환하면 → 기존 값을 덮어씀
    messages: list      # 리듀서 없이 리스트를 쓰면? → 역시 덮어씀! (위험!)

# 패턴 2: Annotated 리듀서 — Session 13.2
class CountingState(TypedDict):
    count: Annotated[int, operator.add]  # 노드가 반환한 값을 기존 값에 더함
    messages: Annotated[list[AnyMessage], add_messages]  # 메시지를 ID 기반으로 관리

# 패턴 3: MessagesState 상속 — Session 13.4
from langgraph.graph import MessagesState

class ToolAgentState(MessagesState):
    # messages 필드를 자동으로 상속받음 (add_messages 리듀서 포함)
    pass
```

여기서 핵심 질문: **리듀서가 없는 리스트 필드**에 두 노드가 동시에 값을 반환하면 어떻게 될까요?

```python
# ❌ 위험한 상황: 리듀서 없는 리스트
class UnsafeState(TypedDict):
    results: list  # 리듀서 없음!

# 노드 A가 {"results": ["결과A"]}를 반환
# 노드 B가 {"results": ["결과B"]}를 반환
# → 최종 results는? "결과A"? "결과B"? 보장 없음!

# ✅ 안전한 방법: 리듀서 추가
class SafeState(TypedDict):
    results: Annotated[list, operator.add]  # 두 결과가 합쳐짐: ["결과A", "결과B"]
```

이제 감이 오시죠? Session 13.2-13.4에서는 `messages` 필드 하나에만 `add_messages` 리듀서를 썼고, 나머지 필드는 단순 덮어쓰기로 충분했습니다. 하지만 **에이전트가 복잡해질수록 더 많은 필드에 더 정교한 리듀서가 필요**해집니다. 지금부터 그 세계로 들어가 봅시다.

### 개념 1: 리듀서 함수의 동작 원리

> 💡 **비유**: 리듀서는 **은행 통장의 거래 기록** 같은 것입니다. 통장 잔액(상태)을 업데이트할 때, 이전 잔액을 지우고 새 값을 쓰는 게 아니라 "입금 +50,000원"처럼 **기존 값에 새 값을 합치는 규칙**을 정하는 거죠. 리듀서 없이 상태를 업데이트하면? 이전 잔액이 통째로 사라지고 새 금액만 남습니다.

LangGraph에서 `Annotated`는 단순한 타입 힌트가 아닙니다. **프레임워크에게 "이 필드를 업데이트할 때 이 규칙을 따르라"고 지시하는 선언**이거든요. 리듀서 함수의 시그니처는 항상 동일합니다:

```
(현재_값, 새로운_값) → 병합된_결과
```

리듀서가 **없으면** 새 값이 기존 값을 덮어쓰고, 리듀서가 **있으면** 두 값을 합치는 규칙대로 병합합니다.

```python
from typing import TypedDict, Annotated
import operator

# ❌ 리듀서 없음: 덮어쓰기
class OverwriteState(TypedDict):
    count: int  # 노드가 {"count": 5}를 반환하면 → 무조건 5로 교체

# ✅ 리듀서 있음: 누적
class AccumulateState(TypedDict):
    count: Annotated[int, operator.add]  # 노드가 {"count": 5}를 반환하면 → 기존값 + 5
```

`operator.add`는 Python 표준 라이브러리의 함수로, `operator.add(a, b)`는 `a + b`와 동일합니다. LangGraph는 내부적으로 이 함수를 `(현재_count, 새로운_count)` 형태로 호출하죠.

```python
from langgraph.graph import StateGraph, START, END

class CounterState(TypedDict):
    count: Annotated[int, operator.add]
    label: str  # 리듀서 없음 → 덮어쓰기

def step_one(state: CounterState) -> dict:
    print(f"step_one 진입: count={state['count']}")
    return {"count": 3, "label": "첫 번째"}  # count: 0+3=3, label: "첫 번째"

def step_two(state: CounterState) -> dict:
    print(f"step_two 진입: count={state['count']}")
    return {"count": 7, "label": "두 번째"}  # count: 3+7=10, label: "두 번째"(덮어쓰기)

graph = StateGraph(CounterState)
graph.add_node("step_one", step_one)
graph.add_node("step_two", step_two)
graph.add_edge(START, "step_one")
graph.add_edge("step_one", "step_two")
graph.add_edge("step_two", END)

app = graph.compile()
result = app.invoke({"count": 0, "label": ""})
print(f"최종 결과: count={result['count']}, label={result['label']}")
# 출력:
# step_one 진입: count=0
# step_two 진입: count=3
# 최종 결과: count=10, label=두 번째
```

`count`는 `operator.add` 리듀서 덕분에 0 → 3 → 10으로 **누적**되고, `label`은 리듀서가 없으므로 "첫 번째" → "두 번째"로 **덮어써진** 것을 확인할 수 있습니다. Session 13.2에서 처음 본 패턴이지만, 이제는 **왜** 이렇게 동작하는지 원리까지 이해한 셈이죠.

### 개념 2: 내장 리듀서와 커스텀 리듀서

> 💡 **비유**: 리듀서를 고를 때는 **조리법을 고르는 것**과 비슷합니다. 숫자를 합치고 싶으면 "더하기 레시피"(`operator.add`), 리스트를 이어붙이고 싶으면 "연결 레시피"(`operator.add` for lists), 메시지를 똑똑하게 관리하고 싶으면 "대화 관리 레시피"(`add_messages`)를 선택하는 거죠. 기존 레시피로 부족하면? 직접 레시피를 만들면 됩니다!

자주 사용하는 리듀서 패턴을 정리해 보겠습니다:

```python
import operator
from typing import TypedDict, Annotated
from langchain_core.messages import AnyMessage
from langgraph.graph.message import add_messages

class ComprehensiveState(TypedDict):
    # 1. 숫자 누적: operator.add
    retry_count: Annotated[int, operator.add]

    # 2. 리스트 연결: operator.add (리스트에도 동작)
    visited_urls: Annotated[list[str], operator.add]

    # 3. 메시지 관리: add_messages (ID 기반 중복 제거 + 삭제 지원)
    messages: Annotated[list[AnyMessage], add_messages]

    # 4. 리듀서 없음: 항상 최신 값으로 덮어쓰기
    current_step: str
```

리스트에 `operator.add`를 쓰면 두 리스트를 단순 연결(`[1,2] + [3,4] = [1,2,3,4]`)합니다. 하지만 메시지 리스트에는 **`add_messages`를 쓰는 게 훨씬 낫습니다**. 왜냐하면 `add_messages`는 메시지 ID를 기반으로 중복을 방지하고, 같은 ID의 메시지가 오면 덮어쓰며, `RemoveMessage`로 특정 메시지를 삭제할 수 있거든요.

이제 **커스텀 리듀서**를 만들어 봅시다. 앞서 본 내장 리듀서만으로는 해결 안 되는 상황이 있습니다. 예를 들어, 여러 노드가 딕셔너리를 반환하는데 **기존 키는 유지하면서 새 키만 추가**하고 싶다면요? `operator.add`는 딕셔너리에 동작하지 않고, 리듀서 없이 쓰면 덮어써 버립니다. 이럴 때 커스텀 리듀서가 필요합니다:

```python
def merge_dicts(left: dict, right: dict) -> dict:
    """두 딕셔너리를 깊은 병합(deep merge)하는 커스텀 리듀서"""
    merged = left.copy()
    for key, value in right.items():
        if key in merged and isinstance(merged[key], dict) and isinstance(value, dict):
            merged[key] = merge_dicts(merged[key], value)  # 재귀적 병합
        else:
            merged[key] = value  # 새 값으로 업데이트
    return merged

class AgentState(TypedDict):
    messages: Annotated[list[AnyMessage], add_messages]
    metadata: Annotated[dict, merge_dicts]  # 커스텀 리듀서 적용
    error_count: Annotated[int, operator.add]
```

```python
# 커스텀 리듀서 동작 확인
result = merge_dicts(
    {"user": {"name": "홍길동"}, "session": "abc"},       # 기존 상태
    {"user": {"age": 30}, "topic": "LangGraph"}            # 새 업데이트
)
print(result)
# 출력: {'user': {'name': '홍길동', 'age': 30}, 'session': 'abc', 'topic': 'LangGraph'}
# → 기존 dict에 새 키가 추가되고, 중첩 dict도 병합됨
```

> ⚠️ **흔한 오해**: `operator.add`는 숫자에만 쓸 수 있다고 생각하기 쉽지만, Python의 `+` 연산자를 지원하는 모든 타입(리스트, 문자열, 튜플 등)에 동작합니다. 다만 정수에 리스트를 더하는 식의 타입 불일치가 발생하면 런타임 에러가 나니 주의하세요.

### 개념 3: add_messages 심층 분석과 RemoveMessage

> 💡 **비유**: `add_messages`를 **채팅방의 관리자**라고 생각해 보세요. 새 메시지가 오면 대화 기록에 추가하고, 같은 ID의 메시지가 다시 오면 "아, 수정된 메시지구나" 하고 교체하며, 특별한 삭제 요청이 오면 해당 메시지를 기록에서 지워줍니다. 단순한 리스트 연결(`operator.add`)은 이런 똑똑한 관리를 못 하죠.

Session 13.4에서 `add_messages`를 사용하긴 했지만, 당시에는 "메시지가 알아서 쌓인다" 정도로만 이해하고 넘어갔습니다. 이제 그 내부 동작을 정확히 알아봅시다. `add_messages`는 단순 리스트 연결이 아니라 세 가지 핵심 동작을 수행합니다:

1. **추가(Append)**: 새 메시지의 ID가 기존 리스트에 없으면 끝에 추가
2. **교체(Replace)**: 새 메시지의 ID가 기존 리스트에 있으면 해당 메시지를 교체
3. **삭제(Remove)**: `RemoveMessage`가 오면 해당 ID의 메시지를 제거

```python
from langchain_core.messages import (
    HumanMessage, AIMessage, SystemMessage,
    RemoveMessage
)
from langgraph.graph.message import add_messages

# 1. 기본 추가
existing = [HumanMessage(content="안녕하세요", id="msg-1")]
new = [AIMessage(content="반갑습니다!", id="msg-2")]
result = add_messages(existing, new)
print(len(result))  # 2 — msg-1, msg-2

# 2. ID 기반 교체 (같은 ID로 보내면 교체)
updated = [HumanMessage(content="안녕하세요! (수정됨)", id="msg-1")]
result = add_messages(result, updated)
print(result[0].content)  # "안녕하세요! (수정됨)" — msg-1이 교체됨

# 3. RemoveMessage로 삭제
delete = [RemoveMessage(id="msg-1")]
result = add_messages(result, delete)
print(len(result))  # 1 — msg-1이 삭제되고 msg-2만 남음
```

실제 에이전트에서 **메시지 트리밍**이 필요한 상황은 매우 흔합니다. 대화가 길어지면 토큰 사용량이 급증하거든요. 그래프 내 노드에서 오래된 메시지를 정리하는 패턴을 봅시다:

```python
from langgraph.graph import StateGraph, START, END, MessagesState

def trim_messages_node(state: MessagesState) -> dict:
    """오래된 메시지를 정리하는 노드 (시스템 메시지는 보존)"""
    messages = state["messages"]

    # 시스템 메시지와 나머지를 분리
    system_msgs = [m for m in messages if isinstance(m, SystemMessage)]
    other_msgs = [m for m in messages if not isinstance(m, SystemMessage)]

    # 최근 10개만 유지, 나머지는 삭제
    if len(other_msgs) > 10:
        to_remove = other_msgs[:-10]
        return {
            "messages": [RemoveMessage(id=m.id) for m in to_remove]
        }
    return {"messages": []}  # 삭제할 것 없음
```

> 🔥 **실무 팁**: `RemoveMessage`를 사용하려면 메시지에 반드시 `id`가 있어야 합니다. LangChain의 LLM 호출은 자동으로 ID를 생성하지만, 수동으로 메시지를 만들 때는 `HumanMessage(content="...", id="unique-id")`처럼 명시적으로 ID를 지정하는 습관을 들이세요.

### 개념 4: MessagesState와 확장 패턴

[13.4: 도구 호출 에이전트](ch13/session4.md)에서 `MessagesState`를 사용했었죠? `MessagesState`는 LangGraph가 미리 정의해 둔 편의 클래스인데, 내부적으로는 이런 구조입니다:

```python
# MessagesState의 실제 정의 (langgraph 내부)
class MessagesState(TypedDict):
    messages: Annotated[list[AnyMessage], add_messages]
```

단 하나의 필드 `messages`만 있고, `add_messages` 리듀서가 적용되어 있습니다. Session 13.4에서는 이것만으로도 도구 호출 에이전트를 만들기에 충분했지만, 실무에서는 메시지 외에도 추가 필드가 필요한 경우가 대부분입니다. 예를 들어, "왜 알아야 할까?" 섹션에서 본 `tool_call_count` 같은 필드가 필요하죠. 이럴 때 **MessagesState를 상속하여 확장**하는 패턴을 씁니다:

```python
from langgraph.graph import MessagesState

class RAGAgentState(MessagesState):
    """RAG 에이전트를 위한 확장 상태"""
    # messages는 MessagesState에서 상속 (add_messages 리듀서 포함)
    query: str                                       # 현재 검색 쿼리 (덮어쓰기)
    retrieved_docs: Annotated[list[str], operator.add]  # 검색된 문서 (누적)
    search_count: Annotated[int, operator.add]          # 검색 횟수 (누적)
    confidence: float                                    # 신뢰도 점수 (덮어쓰기)
```

이렇게 하면 `messages`에 대한 `add_messages` 리듀서를 다시 정의할 필요 없이, 필요한 필드만 추가하면 됩니다. Session 13.4의 `MessagesState`에서 한 단계 진화한 셈이죠.

```python
from langgraph.graph import StateGraph, START, END

def search_node(state: RAGAgentState) -> dict:
    """검색 노드 — 문서를 찾아서 상태에 추가"""
    query = state.get("query", "LangGraph란?")
    # 실제로는 벡터 스토어에서 검색
    found_docs = [f"'{query}'에 대한 문서 #{state.get('search_count', 0) + 1}"]
    return {
        "retrieved_docs": found_docs,  # 기존 리스트에 추가 (operator.add)
        "search_count": 1,              # 기존 값에 +1 (operator.add)
    }

def evaluate_node(state: RAGAgentState) -> dict:
    """평가 노드 — 검색 결과의 품질을 판단"""
    doc_count = len(state.get("retrieved_docs", []))
    confidence = min(doc_count * 0.3, 1.0)
    return {"confidence": confidence}  # 덮어쓰기

graph = StateGraph(RAGAgentState)
graph.add_node("search", search_node)
graph.add_node("evaluate", evaluate_node)
graph.add_edge(START, "search")
graph.add_edge("search", "evaluate")
graph.add_edge("evaluate", END)

app = graph.compile()
result = app.invoke({
    "messages": [],
    "query": "LangGraph 상태 관리",
    "retrieved_docs": [],
    "search_count": 0,
    "confidence": 0.0,
})
print(f"검색 횟수: {result['search_count']}")       # 1
print(f"문서 수: {len(result['retrieved_docs'])}")   # 1
print(f"신뢰도: {result['confidence']}")             # 0.3
```

### 개념 5: TypedDict vs Pydantic — 상태 스키마 선택 기준

> 💡 **비유**: TypedDict는 **연필로 쓴 메모지**입니다. 가볍고 빠르게 수정할 수 있지만, 누군가 잘못된 값을 써도 그냥 넘어갑니다. Pydantic은 **인쇄된 공식 양식**이에요. 작성할 때 형식을 검증하고, 잘못된 값이 들어오면 즉시 거부합니다. 내부 워크플로우에는 메모지(TypedDict)로 충분하지만, 외부 입력을 받는 곳에서는 공식 양식(Pydantic)이 안전하죠.

LangGraph는 TypedDict, Pydantic `BaseModel`, dataclass를 모두 상태 스키마로 지원합니다. 각각의 특징을 비교해 봅시다:

```python
# 방법 1: TypedDict (가장 흔한 패턴)
from typing import TypedDict, Annotated
import operator

class TypedDictState(TypedDict):
    messages: Annotated[list, add_messages]
    count: Annotated[int, operator.add]
    status: str

# 방법 2: Pydantic BaseModel (런타임 검증이 필요할 때)
from pydantic import BaseModel, Field

class PydanticState(BaseModel):
    messages: Annotated[list, add_messages] = Field(default_factory=list)
    count: Annotated[int, operator.add] = 0
    status: str = "idle"
    confidence: float = Field(default=0.0, ge=0.0, le=1.0)  # 0~1 범위 강제
```

**TypedDict가 LangGraph에서 권장되는 이유**:

1. **부분 업데이트가 자연스러움**: 노드가 `{"count": 1}`만 반환해도 나머지 필드는 그대로 유지
2. **성능 오버헤드 없음**: 런타임 검증이 없어서 빠름
3. **LangGraph 내부와 일관성**: LangGraph의 상태 병합 메커니즘이 dict 기반으로 최적화됨

**Pydantic을 쓰면 좋은 경우**:

1. 외부 API 입력/출력 검증이 필요할 때
2. `confidence: float = Field(ge=0.0, le=1.0)` 같은 **값 제약**이 필요할 때
3. 복잡한 중첩 스키마에서 **자동 직렬화/역직렬화**가 필요할 때

> ⚠️ **흔한 오해**: "Pydantic이 더 안전하니까 항상 Pydantic을 쓰자"라고 생각하기 쉽습니다. 하지만 LangGraph에서 상태 업데이트는 매우 빈번하게 일어나고, Pydantic은 매번 전체 모델을 재구성해야 하므로 성능 저하가 발생할 수 있습니다. **내부 상태는 TypedDict, 외부 경계는 Pydantic**이 실무에서 가장 많이 쓰이는 조합이에요.

## 실습: 직접 해보기

이제 배운 내용을 종합하여, **리서치 에이전트**의 상태를 설계하고 실행해 봅시다. 이 에이전트는 여러 단계를 거치며 정보를 수집하고, 검색 횟수를 추적하며, 최종 보고서를 생성합니다. Session 13.3의 `should_continue` 패턴과 Session 13.4의 `MessagesState` 상속을 결합하되, 이번에는 **커스텀 리듀서와 리듀서 팩토리**까지 동원합니다.

```python
"""LangGraph 상태 관리 심화 실습: 리서치 에이전트"""

import operator
from typing import TypedDict, Annotated, Literal
from langchain_core.messages import (
    HumanMessage, AIMessage, SystemMessage, RemoveMessage
)
from langgraph.graph import StateGraph, START, END, MessagesState
from langgraph.graph.message import add_messages


# --- 1. 커스텀 리듀서 정의 ---

def merge_dicts(left: dict, right: dict) -> dict:
    """딕셔너리를 깊은 병합하는 커스텀 리듀서"""
    merged = left.copy()
    for key, value in right.items():
        if key in merged and isinstance(merged[key], dict) and isinstance(value, dict):
            merged[key] = merge_dicts(merged[key], value)
        else:
            merged[key] = value
    return merged


def keep_latest_n(n: int):
    """최근 N개만 유지하는 리듀서 팩토리"""
    def reducer(left: list, right: list) -> list:
        combined = left + right
        return combined[-n:]  # 최근 N개만 유지
    return reducer


# --- 2. 상태 스키마 정의 ---

class ResearchState(MessagesState):
    """리서치 에이전트 상태: 다양한 리듀서 패턴 시연"""
    # MessagesState 상속 → messages: Annotated[list, add_messages] 포함

    query: str                                               # 검색 쿼리 (덮어쓰기)
    sources: Annotated[list[str], operator.add]              # 수집된 출처 (누적)
    findings: Annotated[dict, merge_dicts]                   # 조사 결과 (딕셔너리 병합)
    search_count: Annotated[int, operator.add]               # 검색 횟수 (누적)
    recent_actions: Annotated[list[str], keep_latest_n(5)]   # 최근 5개 행동만 유지
    report: str                                              # 최종 보고서 (덮어쓰기)
    phase: str                                               # 현재 단계 (덮어쓰기)


# --- 3. 노드 함수 정의 ---

def plan_node(state: ResearchState) -> dict:
    """연구 계획을 수립하는 노드"""
    query = state["query"]
    return {
        "messages": [AIMessage(content=f"'{query}'에 대한 연구를 시작합니다.")],
        "phase": "planning",
        "recent_actions": ["연구 계획 수립"],
        "findings": {"plan": {"topic": query, "steps": 3}},
    }


def search_node(state: ResearchState) -> dict:
    """정보를 검색하는 노드"""
    query = state["query"]
    count = state.get("search_count", 0)

    # 시뮬레이션: 실제로는 웹 검색이나 벡터 DB 쿼리
    new_source = f"https://example.com/result-{count + 1}"
    finding_key = f"search_{count + 1}"

    return {
        "messages": [AIMessage(content=f"검색 #{count + 1}: '{query}' 결과를 찾았습니다.")],
        "sources": [new_source],                        # 기존 리스트에 추가
        "findings": {finding_key: {"relevance": 0.85}}, # 기존 dict에 병합
        "search_count": 1,                               # 기존 값에 +1
        "recent_actions": [f"검색 #{count + 1} 수행"],
        "phase": "searching",
    }


def analyze_node(state: ResearchState) -> dict:
    """수집된 정보를 분석하는 노드"""
    source_count = len(state.get("sources", []))
    search_count = state.get("search_count", 0)

    return {
        "messages": [AIMessage(
            content=f"총 {search_count}회 검색, {source_count}개 출처 분석 완료."
        )],
        "findings": {"analysis": {"total_sources": source_count, "quality": "good"}},
        "recent_actions": ["분석 수행"],
        "phase": "analyzing",
    }


def trim_node(state: ResearchState) -> dict:
    """오래된 메시지를 정리하는 노드"""
    messages = state["messages"]
    # AI 메시지 중 가장 오래된 것들을 제거 (최근 3개만 유지)
    ai_msgs = [m for m in messages if isinstance(m, AIMessage)]
    if len(ai_msgs) > 3:
        to_remove = ai_msgs[:-3]
        return {
            "messages": [RemoveMessage(id=m.id) for m in to_remove],
            "recent_actions": [f"{len(to_remove)}개 메시지 정리"],
        }
    return {"recent_actions": ["메시지 정리 불필요"]}


def report_node(state: ResearchState) -> dict:
    """최종 보고서를 생성하는 노드"""
    findings = state.get("findings", {})
    sources = state.get("sources", [])

    report = f"""
## 연구 보고서: {state['query']}
- 검색 횟수: {state.get('search_count', 0)}회
- 수집 출처: {len(sources)}개
- 조사 결과 키: {list(findings.keys())}
- 출처 목록: {', '.join(sources)}
""".strip()

    return {
        "messages": [AIMessage(content="연구 보고서가 완성되었습니다.")],
        "report": report,
        "recent_actions": ["보고서 생성"],
        "phase": "completed",
    }


def should_continue_searching(state: ResearchState) -> Literal["search", "analyze"]:
    """검색을 계속할지 분석으로 넘어갈지 결정"""
    if state.get("search_count", 0) < 3:
        return "search"
    return "analyze"


# --- 4. 그래프 조립 ---

graph = StateGraph(ResearchState)

graph.add_node("plan", plan_node)
graph.add_node("search", search_node)
graph.add_node("analyze", analyze_node)
graph.add_node("trim", trim_node)
graph.add_node("report", report_node)

graph.add_edge(START, "plan")
graph.add_edge("plan", "search")
graph.add_conditional_edges("search", should_continue_searching)
graph.add_edge("analyze", "trim")
graph.add_edge("trim", "report")
graph.add_edge("report", END)

app = graph.compile()


# --- 5. 실행 및 상태 변화 추적 ---

initial_state = {
    "messages": [HumanMessage(content="LangGraph 상태 관리에 대해 조사해주세요.", id="user-1")],
    "query": "LangGraph 상태 관리",
    "sources": [],
    "findings": {},
    "search_count": 0,
    "recent_actions": [],
    "report": "",
    "phase": "init",
}

# 스트리밍으로 각 노드의 상태 변화를 관찰
print("=" * 60)
print("리서치 에이전트 실행 시작")
print("=" * 60)

for event in app.stream(initial_state):
    for node_name, node_output in event.items():
        print(f"\n📍 [{node_name}] 노드 실행 완료")
        if "search_count" in node_output:
            print(f"   검색 횟수: +{node_output['search_count']}")
        if "sources" in node_output:
            print(f"   새 출처: {node_output.get('sources', [])}")
        if "findings" in node_output:
            print(f"   새 발견: {list(node_output['findings'].keys())}")
        if "recent_actions" in node_output:
            print(f"   행동: {node_output['recent_actions']}")
        if "phase" in node_output:
            print(f"   단계: {node_output['phase']}")

# 최종 상태 확인
final = app.invoke(initial_state)
print("\n" + "=" * 60)
print("최종 상태 요약")
print("=" * 60)
print(f"총 검색 횟수: {final['search_count']}")
print(f"수집된 출처: {final['sources']}")
print(f"조사 결과 키: {list(final['findings'].keys())}")
print(f"최근 행동 (최대 5개): {final['recent_actions']}")
print(f"남은 메시지 수: {len(final['messages'])}")
print(f"\n{final['report']}")
```

실행하면, 다음과 같은 상태 변화를 관찰할 수 있습니다:

- `search_count`: 0 → 1 → 2 → 3 (`operator.add`로 누적)
- `sources`: [] → [url1] → [url1, url2] → [url1, url2, url3] (리스트 누적)
- `findings`: {} → {plan: ...} → {plan: ..., search_1: ...} → ... (딕셔너리 병합)
- `recent_actions`: 항상 최근 5개만 유지 (`keep_latest_n(5)` 커스텀 리듀서)
- `messages`: trim_node에서 오래된 AI 메시지가 `RemoveMessage`로 삭제됨

## 더 깊이 알아보기

### 리듀서의 기원: Redux에서 LangGraph까지

"리듀서(Reducer)"라는 개념은 LangGraph가 처음 만든 게 아닙니다. 이 용어는 2015년 Dan Abramov가 만든 **Redux**에서 유래했는데요, Redux는 JavaScript 프론트엔드의 상태 관리 라이브러리로, `(previousState, action) => newState`라는 패턴을 "reducer"라고 불렀습니다. 이 이름은 JavaScript의 `Array.reduce()` 메서드에서 따온 건데, 여러 값을 하나로 "줄여나간다(reduce)"는 의미죠.

LangGraph 팀은 이 검증된 패턴을 에이전트 워크플로우에 차용했습니다. Harrison Chase(LangChain 창시자)가 2024년 초 LangGraph를 설계할 때, "에이전트도 결국 상태 머신이고, 상태를 안전하게 업데이트하는 방법이 필요하다"는 문제의식에서 Redux의 리듀서 패턴을 도입한 거예요. 다만 Redux와 달리, LangGraph는 Python의 `Annotated` 타입 힌트를 활용하여 리듀서를 상태 스키마에 직접 선언하는 방식을 택했습니다. 덕분에 별도의 리듀서 파일 없이, **상태 정의만 보면 업데이트 규칙을 한눈에 알 수 있는** 깔끔한 구조가 됐죠.

### Python Annotated 타입의 탄생

`Annotated` 타입은 Python 3.9에서 PEP 593으로 추가되었습니다. 원래는 타입 힌트에 추가 메타데이터를 붙이기 위한 범용 메커니즘이었는데, LangGraph가 이를 "리듀서 함수를 타입에 부착한다"는 용도로 창의적으로 활용한 겁니다. `Annotated[int, operator.add]`에서 `int`는 타입이고, `operator.add`는 LangGraph가 읽는 메타데이터인 셈이죠.

## 흔한 오해와 팁

> ⚠️ **흔한 오해**: "리듀서를 안 붙이면 상태가 업데이트 안 된다"고 오해하는 분이 많습니다. 사실 리듀서가 없는 필드도 정상적으로 업데이트됩니다 — 다만 **덮어쓰기** 방식으로 동작할 뿐이에요. 리듀서는 "덮어쓰기 대신 이 규칙으로 합쳐라"라고 지시하는 선택적 옵션입니다.

> 💡 **알고 계셨나요?**: `add_messages` 리듀서는 단순 리스트 연결이 아니라, 메시지의 `id` 필드를 기반으로 **업서트(upsert)** 로직을 수행합니다. 같은 ID의 메시지가 이미 있으면 교체하고, 없으면 추가하죠. 이 덕분에 LLM 응답을 스트리밍할 때 같은 ID로 부분 응답을 보내면 자연스럽게 "마지막 버전"만 남게 됩니다.

> 🔥 **실무 팁**: 병렬 노드(fan-out → fan-in)에서 같은 상태 키를 업데이트하려면 **반드시 리듀서를 정의**해야 합니다. 리듀서 없이 두 노드가 동시에 같은 키를 반환하면, 어느 값이 살아남을지 보장되지 않습니다. `operator.add`든 커스텀 리듀서든, 병렬 업데이트가 예상되는 필드에는 항상 리듀서를 붙이세요.

> 🔥 **실무 팁**: 메시지 트리밍 시 `SystemMessage`는 **절대 삭제하지 마세요**. 시스템 메시지는 에이전트의 기본 지시사항을 담고 있어서, 삭제하면 에이전트의 행동이 완전히 달라질 수 있습니다. 트리밍 로직에서 항상 시스템 메시지를 필터링하고 보호하세요.

## 핵심 정리

| 개념 | 설명 |
|------|------|
| `Annotated[타입, 리듀서]` | 상태 필드에 업데이트 규칙(리듀서)을 선언하는 패턴 |
| `operator.add` | 숫자 덧셈, 리스트 연결 등 `+` 연산자 기반 내장 리듀서 |
| `add_messages` | 메시지 ID 기반 추가/교체/삭제를 지원하는 LangGraph 내장 리듀서 |
| `RemoveMessage` | 특정 ID의 메시지를 상태에서 삭제하는 특수 메시지 타입 |
| 커스텀 리듀서 | `(left, right) → merged` 시그니처의 함수로 직접 정의 |
| 리듀서 팩토리 | `keep_latest_n(5)`처럼 클로저로 파라미터화된 리듀서 생성 |
| `MessagesState` | `messages: Annotated[list, add_messages]`를 미리 정의한 편의 클래스 |
| MessagesState 확장 | `class MyState(MessagesState):`로 상속하여 필드 추가 |
| TypedDict vs Pydantic | 내부 상태는 TypedDict(가볍고 빠름), 외부 경계는 Pydantic(검증) |
| 리듀서 없는 필드 | 새 값이 기존 값을 완전히 덮어쓰기(overwrite) |
