# 테스트와 배포

> 18장에서 구축한 지능형 문서 QA 시스템에 테스트를 작성하고, Streamlit 앱에 맞는 배포 설정과 CI/CD 파이프라인으로 프로덕션에 배포합니다.

## 개요

이 섹션에서는 18.1~18.5에서 완성한 문서 QA 시스템을 **프로덕션 수준**으로 끌어올립니다. 단위 테스트와 통합 테스트로 품질을 보증하고, Streamlit 앱에 특화된 Docker 설정을 추가하며, GitHub Actions CI/CD로 자동화된 배포 파이프라인을 구성합니다.

**선수 지식**: [18.1 프로젝트 설계와 아키텍처](session_18_1.md)에서 정의한 4-계층 구조, [18.2 문서 수집과 인덱싱](session_18_2.md)의 `IngestionPipeline`, [18.3 검색과 생성](session_18_3.md)의 `QAPipeline`, [18.4 대화 관리와 메모리](session_18_4.md)의 `ConversationalQA`, [18.5 Streamlit UI](session_18_5.md)의 `app.py`, [17.5 Docker 컨테이너화와 배포](session_17_5.md)의 멀티스테이지 빌드 기본 구조

**학습 목표**:
- pytest와 FakeListChatModel을 활용하여 LangChain 컴포넌트의 단위 테스트를 작성할 수 있다
- 통합 테스트로 RAG 파이프라인 전체 흐름을 검증할 수 있다
- Streamlit 앱의 Docker 배포에서 기본 FastAPI 배포와 다른 설정 포인트를 이해할 수 있다
- GitHub Actions CI/CD 파이프라인을 구성하여 테스트-빌드-배포를 자동화할 수 있다

## 왜 알아야 할까?

"코드가 돌아간다"와 "프로덕션에서 안정적으로 운영된다" 사이에는 거대한 간극이 있습니다. 여러분이 18.5까지 만든 문서 QA 시스템은 로컬에서는 훌륭하게 동작하지만, 이런 질문들에 답할 수 있나요?

- `IngestionPipeline`의 청크 분할 로직을 수정했는데, **기존 기능이 깨지지 않았다고 확신**할 수 있나요?
- 팀원이 코드를 변경하면 **자동으로 검증**되나요?
- 서버를 교체하더라도 **동일한 환경**에서 실행할 수 있나요?

테스트는 "야간에 편히 잠들 수 있게 해주는 장치"이고, Docker는 "내 컴퓨터에서는 되는데…"를 영원히 없애주는 기술이며, CI/CD는 "배포 전에 테스트 돌렸어?"라는 질문을 불필요하게 만듭니다. LLM 애플리케이션은 비결정적(non-deterministic) 특성 때문에 전통적인 소프트웨어보다 테스트가 더 까다롭고, 그래서 **더더욱** 체계적인 테스트 전략이 필요합니다.

## 핵심 개념

### 개념 1: 단위 테스트 — LangChain 컴포넌트를 고립시켜 검증하기

> 💡 **비유**: 자동차를 조립하기 전에 엔진, 브레이크, 핸들을 각각 따로 테스트하는 것과 같습니다. 완성차를 도로에 올리기 전에 부품 하나하나가 제대로 동작하는지 확인하는 거죠. 엔진을 테스트할 때 실제 도로가 필요 없듯이, LangChain 컴포넌트를 테스트할 때도 실제 OpenAI API가 필요하지 않습니다.

LLM 애플리케이션의 단위 테스트에서 가장 큰 과제는 **비용과 비결정성**입니다. 매 테스트마다 실제 API를 호출하면 비용이 발생하고, 같은 입력에도 매번 다른 출력이 나올 수 있거든요. LangChain은 이 문제를 해결하기 위해 `FakeListChatModel`을 제공합니다. 미리 정해둔 응답을 순서대로 반환하는 가짜 모델이죠.

```python
# tests/test_pipeline.py
import pytest
from unittest.mock import MagicMock, patch
from langchain_community.chat_models.fake import FakeListChatModel
from langchain_core.documents import Document

from src.chains.qa_pipeline import QAPipeline, format_docs


class TestFormatDocs:
    """format_docs 함수의 단위 테스트"""

    def test_single_document(self):
        """단일 문서를 올바르게 포맷하는지 확인"""
        docs = [Document(page_content="LangChain은 프레임워크입니다.")]
        result = format_docs(docs)
        assert result == "LangChain은 프레임워크입니다."

    def test_multiple_documents(self):
        """여러 문서를 줄바꿈으로 구분하는지 확인"""
        docs = [
            Document(page_content="첫 번째 문서"),
            Document(page_content="두 번째 문서"),
        ]
        result = format_docs(docs)
        assert "첫 번째 문서" in result
        assert "두 번째 문서" in result

    def test_empty_documents(self):
        """빈 문서 리스트를 처리하는지 확인"""
        result = format_docs([])
        assert result == ""


class TestQAPipelineUnit:
    """QAPipeline의 단위 테스트 — FakeListChatModel 활용"""

    @pytest.fixture
    def fake_llm(self):
        """실제 API 대신 사용할 가짜 LLM"""
        return FakeListChatModel(
            responses=[
                '{"answer": "LangChain은 LLM 프레임워크입니다.", '
                '"citations": [{"source": "docs.md", "quote": "LangChain is..."}]}'
            ]
        )

    @pytest.fixture
    def mock_retriever(self):
        """가짜 검색기 — 항상 고정된 문서를 반환"""
        retriever = MagicMock()
        retriever.invoke.return_value = [
            Document(
                page_content="LangChain is an open-source framework.",
                metadata={"source": "docs.md", "page": 1},
            )
        ]
        return retriever

    def test_pipeline_returns_answer(self, fake_llm, mock_retriever):
        """파이프라인이 답변을 반환하는지 확인"""
        pipeline = QAPipeline(llm=fake_llm, retriever=mock_retriever)
        result = pipeline.ask("LangChain이 뭔가요?")
        assert "answer" in result
        assert len(result["answer"]) > 0

    def test_pipeline_includes_citations(self, fake_llm, mock_retriever):
        """답변에 소스 인용이 포함되는지 확인"""
        pipeline = QAPipeline(llm=fake_llm, retriever=mock_retriever)
        result = pipeline.ask("LangChain이 뭔가요?")
        assert "citations" in result
        assert len(result["citations"]) > 0
```

핵심은 `FakeListChatModel`입니다. `responses` 리스트에 넣어둔 문자열을 순서대로 반환하기 때문에, **실제 API 호출 없이** 파이프라인 로직만 정확하게 검증할 수 있습니다.

### 개념 2: 통합 테스트 — 파이프라인 전체 흐름 검증

> 💡 **비유**: 부품 검사를 통과한 자동차를 이제 실제 도로에서 시운전하는 단계입니다. 엔진, 변속기, 바퀴가 **함께** 잘 동작하는지 확인하죠. 단위 테스트가 "부품이 정상인가?"라면, 통합 테스트는 "부품들이 조합되었을 때 차가 달리는가?"를 묻습니다.

통합 테스트는 `IngestionPipeline` → `Retriever` → `QAPipeline` → `ConversationalQA`로 이어지는 전체 파이프라인을 검증합니다. 여기서도 LLM은 `FakeListChatModel`로 대체하되, 벡터 스토어는 실제 FAISS를 사용해서 임베딩-검색-생성 흐름이 끊기지 않는지 확인합니다.

```python
# tests/test_integration.py
import pytest
import tempfile
from pathlib import Path

from langchain_community.chat_models.fake import FakeListChatModel
from langchain_community.embeddings import FakeEmbeddings
from langchain_community.vectorstores import FAISS
from langchain_core.documents import Document

from src.chains.ingestion import IngestionPipeline
from src.chains.qa_pipeline import QAPipeline


@pytest.fixture(scope="module")
def sample_docs():
    """테스트용 샘플 문서"""
    return [
        Document(
            page_content="LangChain은 LLM 애플리케이션 개발 프레임워크입니다. "
            "LCEL을 사용하여 체인을 선언적으로 구성할 수 있습니다.",
            metadata={"source": "intro.md", "page": 1},
        ),
        Document(
            page_content="RAG는 외부 문서를 검색하여 LLM 응답의 정확성을 높이는 기법입니다. "
            "벡터 스토어와 리트리버를 결합하여 구현합니다.",
            metadata={"source": "rag_guide.md", "page": 1},
        ),
    ]


@pytest.fixture(scope="module")
def vector_store(sample_docs):
    """FakeEmbeddings를 사용한 테스트용 벡터 스토어"""
    embeddings = FakeEmbeddings(size=384)
    return FAISS.from_documents(sample_docs, embeddings)


class TestEndToEndPipeline:
    """인덱싱 → 검색 → 응답 전체 흐름 통합 테스트"""

    def test_retrieval_returns_relevant_docs(self, vector_store):
        """벡터 스토어에서 관련 문서를 검색하는지 확인"""
        retriever = vector_store.as_retriever(search_kwargs={"k": 1})
        docs = retriever.invoke("RAG란 무엇인가요?")

        assert len(docs) > 0
        # 검색된 문서가 실제로 존재하는 문서인지 확인
        assert any("RAG" in doc.page_content or "검색" in doc.page_content
                    for doc in docs)

    def test_full_qa_pipeline(self, vector_store):
        """검색 → LLM 응답 전체 파이프라인 동작 확인"""
        fake_llm = FakeListChatModel(
            responses=["RAG는 Retrieval-Augmented Generation의 약자로, "
                       "외부 문서를 검색하여 답변의 정확성을 높이는 기법입니다."]
        )
        retriever = vector_store.as_retriever(search_kwargs={"k": 2})

        pipeline = QAPipeline(llm=fake_llm, retriever=retriever)
        result = pipeline.ask("RAG가 뭔가요?")

        assert result is not None
        assert len(str(result)) > 0

    def test_index_persistence(self, vector_store):
        """벡터 인덱스 저장/로드가 정상 동작하는지 확인"""
        with tempfile.TemporaryDirectory() as tmpdir:
            index_path = Path(tmpdir) / "test_index"

            # 저장
            vector_store.save_local(str(index_path))
            assert index_path.exists()

            # 로드
            loaded_store = FAISS.load_local(
                str(index_path),
                FakeEmbeddings(size=384),
                allow_dangerous_deserialization=True,
            )
            docs = loaded_store.as_retriever().invoke("LangChain")
            assert len(docs) > 0
```

> ⚠️ **흔한 오해**: "통합 테스트는 실제 API 키가 있어야 한다"고 생각하기 쉽지만, `FakeEmbeddings`와 `FakeListChatModel`을 조합하면 **API 키 없이도** 전체 파이프라인을 검증할 수 있습니다. 실제 API를 사용하는 테스트는 별도의 `@pytest.mark.integration` 마커로 분리하세요.

### 개념 3: Streamlit 앱의 Docker 배포 — FastAPI와 무엇이 다를까?

> 💡 **비유**: 같은 이사 박스(Docker)를 쓰더라도, 냉장고를 넣을 때와 피아노를 넣을 때는 포장 방법이 다르죠. 멀티스테이지 빌드, `.dockerignore`, 비root 사용자 같은 **기본 구조**는 [17.5 Docker 컨테이너화와 배포](session_17_5.md)에서 배운 것과 동일합니다. 여기서는 Streamlit이라는 "피아노"를 박스에 넣을 때 달라지는 포인트만 집중해서 살펴봅니다.

[17.5](session_17_5.md)에서 LangServe/FastAPI 앱을 Docker로 패키징하는 전체 과정을 다루었죠. 멀티스테이지 빌드, `.dockerignore`, HEALTHCHECK, 비root 사용자 설정 등의 기본기는 그대로 적용됩니다. Streamlit 앱을 배포할 때는 세 가지 핵심 차이점만 추가로 신경 쓰면 됩니다.

**차이점 1: 서버 설정 — Gunicorn 대신 Streamlit 내장 서버**

FastAPI는 Gunicorn + Uvicorn 워커로 실행하지만, Streamlit은 자체 웹소켓 서버를 내장하고 있어서 별도의 WSGI/ASGI 서버가 필요 없습니다. 대신 `.streamlit/config.toml`로 서버 동작을 제어합니다.

```toml
# .streamlit/config.toml — Docker 환경용 설정
[server]
headless = true          # 브라우저 자동 열기 비활성화 (서버 환경 필수)
port = 8501
enableCORS = false       # 리버스 프록시 뒤에서 실행 시 비활성화
enableXsrfProtection = true

[browser]
gatherUsageStats = false  # 텔레메트리 비활성화
```

**차이점 2: ENTRYPOINT — streamlit run**

```dockerfile
# FastAPI (17.5에서 배운 방식)
ENTRYPOINT ["gunicorn", "app:app", "-k", "uvicorn.workers.UvicornWorker", ...]

# Streamlit (이번 프로젝트)
ENTRYPOINT ["streamlit", "run", "app.py", \
            "--server.port=8501", \
            "--server.address=0.0.0.0"]
```

**차이점 3: 헬스체크 엔드포인트**

Streamlit은 `/_stcore/health`라는 전용 헬스체크 경로를 제공합니다. FastAPI에서 직접 `/health` 라우트를 만들었던 것과 다르죠.

```dockerfile
# FastAPI 헬스체크 (17.5)
HEALTHCHECK CMD curl -f http://localhost:8000/health || exit 1

# Streamlit 헬스체크 (내장 엔드포인트 활용)
HEALTHCHECK --interval=30s --timeout=10s --retries=3 \
    CMD curl -f http://localhost:8501/_stcore/health || exit 1
```

이 세 가지 차이점만 [17.5](session_17_5.md)의 기본 Dockerfile 구조에 적용하면, Streamlit 앱의 Docker 이미지가 완성됩니다. 빌드와 실행 명령어는 다음과 같습니다:

```bash
# 이미지 빌드
docker build -t doc-qa-system:latest .

# 컨테이너 실행 (.env 파일로 API 키 전달)
docker run -d \
    --name doc-qa \
    -p 8501:8501 \
    --env-file .env \
    -v $(pwd)/data:/app/data \
    doc-qa-system:latest
```

> 🔥 **실무 팁**: Streamlit은 파일 변경을 감지하면 앱을 자동 리로드합니다. Docker 컨테이너에서는 이 기능이 불필요하므로 `--server.fileWatcherType=none` 플래그를 ENTRYPOINT에 추가하면 불필요한 파일 시스템 감시를 없애 성능이 개선됩니다.

### 개념 4: CI/CD 파이프라인 — 자동화된 품질 보증 게이트

> 💡 **비유**: 공장 컨베이어 벨트를 떠올려보세요. 제품(코드)이 벨트 위를 이동하면서 자동으로 품질 검사(테스트)를 받고, 합격하면 포장(Docker 빌드)되어 출하(배포)됩니다. 불량품이 나오면 벨트가 멈추고 경고등이 켜지죠. CI/CD가 바로 이 자동화된 컨베이어 벨트입니다.

GitHub Actions를 사용하면 코드를 푸시할 때마다 테스트 → 린팅 → Docker 빌드 → 배포까지 자동으로 실행됩니다.

```yaml
# .github/workflows/ci-cd.yml
name: CI/CD Pipeline

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

env:
  PYTHON_VERSION: "3.11"
  DOCKER_IMAGE: doc-qa-system

jobs:
  # ============================
  # Job 1: 린팅 & 타입 체크
  # ============================
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Set up Python
        uses: actions/setup-python@v5
        with:
          python-version: ${{ env.PYTHON_VERSION }}

      - name: Install linters
        run: pip install ruff mypy

      - name: Run Ruff (린터 + 포매터)
        run: ruff check src/ tests/

      - name: Run mypy (타입 체크)
        run: mypy src/ --ignore-missing-imports

  # ============================
  # Job 2: 테스트
  # ============================
  test:
    runs-on: ubuntu-latest
    needs: lint  # 린팅 통과 후 실행

    steps:
      - uses: actions/checkout@v4

      - name: Set up Python
        uses: actions/setup-python@v5
        with:
          python-version: ${{ env.PYTHON_VERSION }}
          cache: "pip"  # pip 캐싱으로 빌드 속도 향상

      - name: Install dependencies
        run: |
          pip install --upgrade pip
          pip install -r requirements.txt
          pip install -r requirements-dev.txt

      - name: Run unit tests
        run: pytest tests/test_pipeline.py -v --tb=short

      - name: Run integration tests
        run: pytest tests/test_integration.py -v --tb=short

      - name: Generate coverage report
        run: |
          pytest tests/ --cov=src --cov-report=xml --cov-report=term
          echo "## Test Coverage" >> $GITHUB_STEP_SUMMARY
          pytest tests/ --cov=src --cov-report=term | tail -20 >> $GITHUB_STEP_SUMMARY

  # ============================
  # Job 3: Docker 빌드 & 푸시
  # ============================
  docker:
    runs-on: ubuntu-latest
    needs: test  # 테스트 통과 후 실행
    if: github.ref == 'refs/heads/main'  # main 브랜치만

    steps:
      - uses: actions/checkout@v4

      - name: Set up Docker Buildx
        uses: docker/setup-buildx-action@v3

      - name: Login to Container Registry
        uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - name: Build and push
        uses: docker/build-push-action@v5
        with:
          context: .
          push: true
          tags: |
            ghcr.io/${{ github.repository }}/${{ env.DOCKER_IMAGE }}:latest
            ghcr.io/${{ github.repository }}/${{ env.DOCKER_IMAGE }}:${{ github.sha }}
          cache-from: type=gha
          cache-to: type=gha,mode=max
```

이 파이프라인은 세 가지 Job이 순차적으로 실행됩니다:

1. **lint** — Ruff로 코드 스타일 검사, mypy로 타입 체크
2. **test** — 단위 테스트 → 통합 테스트 → 커버리지 리포트
3. **docker** — main 브랜치에서만 Docker 이미지를 빌드하고 GitHub Container Registry에 푸시

### 개념 5: LangSmith 모니터링 — 프로덕션 관찰 가능성

> 💡 **비유**: 병원에서 환자에게 붙이는 심전도 모니터와 같습니다. 환자(앱)가 살아있는지만 확인하는 게 아니라, 심박수(응답 시간), 혈압(토큰 사용량), 체온(에러율)까지 실시간으로 추적하죠. 문제가 생기기 **전에** 이상 징후를 포착할 수 있습니다.

LangSmith는 LangChain 애플리케이션의 트레이싱, 디버깅, 평가를 위한 관찰 가능성 플랫폼입니다. 환경 변수 하나만 설정하면 모든 체인 호출이 자동으로 기록됩니다.

```python
# src/monitoring.py
"""LangSmith 모니터링 설정 모듈"""

import os
import logging
from functools import wraps
from typing import Callable, Any
from datetime import datetime

logger = logging.getLogger(__name__)


def setup_langsmith(project_name: str = "doc-qa-system") -> None:
    """LangSmith 트레이싱을 활성화합니다.

    Args:
        project_name: LangSmith 프로젝트 이름
    """
    os.environ["LANGSMITH_TRACING"] = "true"
    os.environ["LANGSMITH_PROJECT"] = project_name

    api_key = os.environ.get("LANGSMITH_API_KEY")
    if not api_key:
        logger.warning(
            "LANGSMITH_API_KEY가 설정되지 않았습니다. "
            "트레이싱이 비활성화됩니다."
        )
        os.environ["LANGSMITH_TRACING"] = "false"
        return

    logger.info(f"LangSmith 트레이싱 활성화: project={project_name}")


def track_latency(func: Callable) -> Callable:
    """함수 실행 시간을 측정하는 데코레이터

    Args:
        func: 측정 대상 함수
    """
    @wraps(func)
    def wrapper(*args: Any, **kwargs: Any) -> Any:
        start = datetime.now()
        try:
            result = func(*args, **kwargs)
            elapsed = (datetime.now() - start).total_seconds()
            logger.info(f"{func.__name__} 완료: {elapsed:.2f}초")
            return result
        except Exception as e:
            elapsed = (datetime.now() - start).total_seconds()
            logger.error(f"{func.__name__} 실패 ({elapsed:.2f}초): {e}")
            raise
    return wrapper
```

```python
# app.py에서 LangSmith 연동 (18.5에서 만든 앱에 추가)

from src.monitoring import setup_langsmith, track_latency

# 앱 시작 시 LangSmith 활성화
```

```python
# tests/test_benchmark.py
"""검색 파이프라인 성능 벤치마크"""

import time
import pytest
from langchain_community.embeddings import FakeEmbeddings
from langchain_community.vectorstores import FAISS
from langchain_core.documents import Document


@pytest.fixture(scope="module")
def large_vector_store():
    """벤치마크용 대규모 벡터 스토어 (1000개 문서)"""
    docs = [
        Document(
            page_content=f"문서 #{i}: 이 문서는 테스트 벤치마크를 위한 "
            f"샘플 콘텐츠입니다. 토픽은 {['Python', 'Docker', 'LangChain', 'RAG', 'CI/CD'][i % 5]}입니다.",
            metadata={"source": f"doc_{i}.md", "page": 1, "topic_id": i % 5},
        )
        for i in range(1000)
    ]
    embeddings = FakeEmbeddings(size=384)
    return FAISS.from_documents(docs, embeddings)


class TestRetrievalBenchmark:
    """검색 성능 벤치마크"""

    def test_single_query_latency(self, large_vector_store):
        """단일 쿼리 응답 시간 측정 — 1초 이내 기대"""
        retriever = large_vector_store.as_retriever(search_kwargs={"k": 5})

        start = time.perf_counter()
        docs = retriever.invoke("Python 프로그래밍")
        elapsed = time.perf_counter() - start

        assert len(docs) == 5
        assert elapsed < 1.0, f"쿼리 시간 {elapsed:.3f}초 — 1초 초과!"
        print(f"\n[벤치마크] 단일 쿼리: {elapsed*1000:.1f}ms, 반환 문서: {len(docs)}개")

    def test_batch_query_throughput(self, large_vector_store):
        """배치 쿼리 처리량 측정"""
        retriever = large_vector_store.as_retriever(search_kwargs={"k": 3})
        queries = [
            "Python 기초", "Docker 배포", "LangChain 체인",
            "RAG 시스템", "CI/CD 파이프라인",
        ]

        start = time.perf_counter()
        results = [retriever.invoke(q) for q in queries]
        elapsed = time.perf_counter() - start

        assert all(len(r) > 0 for r in results)
        qps = len(queries) / elapsed  # Queries Per Second
        print(f"\n[벤치마크] 배치 {len(queries)}개 쿼리: {elapsed*1000:.1f}ms "
              f"(QPS: {qps:.1f})")

    def test_index_build_time(self):
        """인덱스 빌드 시간 측정 — 500개 문서 기준"""
        docs = [
            Document(
                page_content=f"빌드 벤치마크 문서 #{i}",
                metadata={"source": f"bench_{i}.md"},
            )
            for i in range(500)
        ]
        embeddings = FakeEmbeddings(size=384)

        start = time.perf_counter()
        store = FAISS.from_documents(docs, embeddings)

        assert store is not None
```

## 더 깊이 알아보기

### LLM 테스팅의 새로운 패러다임

전통적인 소프트웨어 테스팅은 "같은 입력 → 같은 출력"을 전제하지만, LLM은 본질적으로 비결정적입니다. 이 때문에 2023년 이후 LLM 테스팅의 새로운 패러다임이 형성되고 있습니다.

정확한 문자열 일치 대신 **의미적 유사도**, **구조 검증**, **속성 기반 테스트**를 활용하죠. 예를 들어 "답변에 '서울'이라는 단어가 정확히 포함되어야 한다" 대신 "답변이 한국의 수도에 대한 질문에 맞는 내용인가?"를 검증하는 방식입니다. LangSmith의 pytest 플러그인은 이런 흐름을 반영하여, 전통적인 assert문과 LLM 기반 평가를 하나의 테스트 프레임워크에서 결합할 수 있게 해줍니다.

이 접근법은 세 가지 레벨로 구성됩니다:

1. **레벨 1 — 구조 검증**: 출력 형식이 올바른가? (JSON 파싱 가능? 필수 필드 존재?)
2. **레벨 2 — 속성 검증**: 답변이 특정 조건을 만족하는가? (길이, 언어, 키워드 포함 여부)
3. **레벨 3 — 의미적 검증**: LLM 심사관(judge)이 답변 품질을 평가 (relevance, faithfulness)

이번 세션에서 배운 `FakeListChatModel` 기반 테스트는 레벨 1에 해당하고, 프로덕션에서는 LangSmith Evaluation을 통해 레벨 2~3까지 확장하는 것이 권장됩니다.

### CI/CD의 기원 — Martin Fowler의 한마디

"Continuous Integration"이라는 용어를 대중화한 사람은 Martin Fowler입니다. 2000년에 발표한 글에서 그는 "팀원들이 하루에 최소 한 번은 코드를 통합해야 한다"고 주장했는데, 당시만 해도 이건 꽤 급진적인 주장이었습니다. 그 전에는 몇 주, 심지어 몇 달 동안 따로 작업하다가 한꺼번에 합치는 게 일반적이었거든요. 이 "빅뱅 통합"은 머지 충돌의 악몽을 만들어냈고, Fowler의 CI 원칙은 이 문제에 대한 해법이었습니다. 2011년 Travis CI가 오픈소스 프로젝트에 무료 CI 서비스를 제공하면서 CI/CD가 보편화되었고, 2019년 GitHub Actions가 출시되면서 "코드 저장소와 CI/CD가 한 지붕 아래" 있는 시대가 열렸습니다.

## 흔한 오해와 팁

> ⚠️ **흔한 오해**: "FakeListChatModel로 테스트하면 실제 LLM과 동작이 다를 수 있으니 의미가 없다"고 생각하는 분들이 있습니다. 하지만 단위 테스트의 목적은 LLM의 품질을 검증하는 게 아니라, **LLM을 둘러싼 우리 코드의 로직**(프롬프트 구성, 출력 파싱, 에러 처리 등)이 올바른지 확인하는 것입니다. LLM 품질은 LangSmith Evaluation으로 별도 검증하세요.

> 💡 **알고 계셨나요?**: Streamlit의 `/_stcore/health` 엔드포인트는 단순한 HTTP 200 응답이 아니라 내부 서버 상태를 점검한 뒤 `{"status":"ok"}`를 반환합니다. 반면 FastAPI에서는 개발자가 직접 `/health` 라우트를 만들어야 하죠. 프레임워크마다 배포 시 챙겨야 할 포인트가 다르기 때문에, Docker화할 때는 항상 해당 프레임워크의 **배포 가이드를 먼저 확인**하는 습관을 기르세요.

> 🔥 **실무 팁**: GitHub Actions에서 `pip install`이 매번 느리다면, `actions/setup-python`의 `cache: "pip"` 옵션을 활용하세요. `requirements.txt`가 변경되지 않으면 의존성을 캐시에서 가져와서 CI 시간이 60~70% 단축됩니다. Docker 빌드도 `cache-from: type=gha`로 GitHub Actions 캐시를 활용할 수 있습니다.

> 🔥 **실무 팁**: LangSmith 트레이싱은 환경 변수 `LANGSMITH_TRACING=true`를 설정하는 것만으로 활성화됩니다. LangChain을 사용하고 있다면 **코드 변경 없이** 모든 체인 호출이 자동 기록됩니다. 디버깅이 필요할 때 "이 질문에 어떤 문서가 검색됐고, 프롬프트가 어떻게 구성됐는지"를 LangSmith 대시보드에서 바로 확인할 수 있어서, 프로덕션 문제를 빠르게 추적할 수 있습니다.
