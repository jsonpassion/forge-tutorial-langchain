# 문서 수집과 인덱싱 파이프라인

> Ch6~7에서 배운 문서 로더·텍스트 분할·벡터 스토어의 기초를 넘어, 프로덕션 수준의 인덱싱 파이프라인을 설계하고 구현합니다.

## 개요

이 섹션에서는 지능형 문서 QA 시스템의 **Data Ingestion Layer**를 본격적으로 구현합니다. [Ch6: 문서 로더와 텍스트 분할](ch06/)에서 PyPDFLoader, RecursiveCharacterTextSplitter 같은 개별 도구의 사용법을 익혔고, [Ch7: 임베딩과 벡터 스토어](ch07/)에서 FAISS의 기본 동작을 배웠죠. 이번 세션에서는 그 지식을 토대로 **실전에서 바로 쓸 수 있는 파이프라인**을 만듭니다.

초점은 "어떻게 쓰는가"가 아니라 **"프로덕션에서 어떻게 안정적으로 운영하는가"**입니다. 수백~수천 개 문서를 처리할 때 발생하는 에러 핸들링, 메타데이터 설계 전략, 변경분만 처리하는 증분 인덱싱, 임베딩 API 비용 최적화 같은 실전 과제를 다룹니다.

**선수 지식**: [Session 18.1: 프로젝트 설계와 아키텍처](ch18/session_18_1.md)에서 설계한 4-계층 아키텍처와 팩토리 패턴, [Ch6](ch06/)의 문서 로더·텍스트 분할 기초, [Ch7](ch07/)의 임베딩·FAISS 기본 사용법

**학습 목표**:
- 에러 복구와 폴백(fallback)을 갖춘 견고한 멀티포맷 로더를 구현할 수 있다
- 문서 특성에 따라 청킹 전략을 선택하고, 성능을 벤치마크할 수 있다
- 출처 추적과 필터링 검색을 가능하게 하는 메타데이터 스키마를 설계할 수 있다
- 파일 변경 감지 기반의 **증분 인덱싱**으로 비용과 시간을 절약하는 파이프라인을 완성할 수 있다

## 왜 알아야 할까?

여러분이 회사에서 "사내 문서 검색 시스템을 만들어주세요"라는 요청을 받았다고 상상해 보세요. 폴더를 열어보니 제안서는 PDF, 회의록은 DOCX, 사내 위키는 HTML — 형식이 제각각입니다. Ch6에서 배운 대로 로더를 하나씩 붙이면 되겠지만, 현실은 그렇게 단순하지 않거든요.

PDF 중 하나가 깨진 파일이면? 매일 수십 개씩 새 문서가 올라오면 매번 전체를 다시 인덱싱해야 하나요? 메타데이터 없이 청크만 쌓아두면 "이 답변의 출처가 어디인가요?"라는 질문에 어떻게 답하죠?

바로 이런 **프로덕션 갭(production gap)**을 메우는 것이 이번 세션의 목표입니다. 아무리 뛰어난 LLM을 쓰더라도 데이터가 제대로 준비되지 않으면 "쓰레기가 들어가면 쓰레기가 나온다(Garbage In, Garbage Out)"는 격언이 그대로 적용됩니다. 실제로 RAG 시스템 성능의 **60% 이상이 데이터 전처리 품질**에 의해 좌우된다는 것이 업계의 공통된 경험이에요.

## 핵심 개념

### 개념 1: 견고한 멀티포맷 문서 로더 — 깨진 파일에도 멈추지 않는 파이프라인

> 💡 **비유**: Ch6에서 만든 문서 로더가 **맑은 날의 운전**이라면, 프로덕션 로더는 **폭우 속 고속도로 운전**입니다. 와이퍼(에러 핸들링), 안개등(로깅), 예비 타이어(폴백 로더)가 모두 갖춰져야 사고 없이 목적지에 도착할 수 있죠.

[Ch6](ch06/)에서 `PyPDFLoader`, `Docx2txtLoader`, `BSHTMLLoader`의 기본 사용법은 이미 배웠습니다. 여기서는 이 로더들을 **프로덕션 환경에서 안정적으로 운영**하는 데 필요한 패턴에 집중합니다.

**프로덕션에서 마주치는 현실적 문제들:**

| 문제 | 빈도 | 대응 전략 |
|------|------|-----------|
| 깨진/암호화된 PDF | 높음 | 폴백 로더 + 에러 격리 |
| 인코딩 오류 (EUC-KR, CP949) | 중간 | 다중 인코딩 시도 |
| 빈 문서 / 이미지 전용 PDF | 중간 | 빈 결과 감지 + 스킵 |
| 파일 잠금 (다른 프로세스 사용 중) | 낮음 | 재시도 + 타임아웃 |

핵심은 **하나의 파일 실패가 전체 파이프라인을 멈추지 않도록** 격리하는 것입니다. Ch6에서는 단일 파일 로딩만 다뤘지만, 수백 개 파일을 처리할 때는 에러 격리(error isolation)가 필수입니다.

```python
from pathlib import Path
from typing import List
from langchain_core.documents import Document
from langchain_community.document_loaders import (
    PyPDFLoader,
    Docx2txtLoader,
    BSHTMLLoader,
    UnstructuredPDFLoader,  # 폴백용
)
import logging

logger = logging.getLogger(__name__)

# 확장자 → (기본 로더, 폴백 로더) 매핑
LOADER_MAP: dict[str, tuple[type, type | None]] = {
    ".pdf": (PyPDFLoader, UnstructuredPDFLoader),  # 폴백 포함
    ".docx": (Docx2txtLoader, None),
    ".doc": (Docx2txtLoader, None),
    ".html": (BSHTMLLoader, None),
    ".htm": (BSHTMLLoader, None),
}


def load_document_safe(file_path: Path) -> list[Document]:
    """폴백 전략이 포함된 견고한 문서 로더.

    기본 로더가 실패하면 폴백 로더를 시도하고,
    그마저 실패하면 빈 리스트를 반환합니다 (파이프라인 중단 방지).
    """
    ext = file_path.suffix.lower()
    if ext not in LOADER_MAP:
        logger.warning(f"지원하지 않는 형식 건너뜀: {file_path.name}")
        return []

    primary_cls, fallback_cls = LOADER_MAP[ext]

    # 1차 시도: 기본 로더
    try:
        docs = primary_cls(str(file_path)).load()
        # 빈 결과 감지 (이미지 전용 PDF 등)
        if not docs or all(not d.page_content.strip() for d in docs):
            logger.warning(f"빈 콘텐츠 감지: {file_path.name}")
            if fallback_cls:
                raise ValueError("빈 결과 — 폴백 시도")
            return []
        return docs
    except Exception as e:
        logger.warning(f"기본 로더 실패 ({file_path.name}): {e}")

    # 2차 시도: 폴백 로더
    if fallback_cls:
        try:
            docs = fallback_cls(str(file_path)).load()
            logger.info(f"폴백 로더 성공: {file_path.name}")
            return docs
        except Exception as e:
            logger.error(f"폴백 로더도 실패 ({file_path.name}): {e}")

    return []  # 모든 시도 실패 — 파이프라인은 계속 진행
```

> 🔥 **실무 팁**: `PyPDFLoader`는 가볍고 빠르지만 테이블이나 복잡한 레이아웃이 포함된 PDF에서는 텍스트가 뒤죽박죽 되는 경우가 많습니다. `UnstructuredPDFLoader`를 폴백으로 두면 이런 PDF도 상당 부분 구제할 수 있어요. 다만 `unstructured` 라이브러리의 의존성이 무겁기 때문에, 기본 로더가 성공하면 굳이 호출하지 않는 **lazy fallback** 전략이 실용적입니다.

**대용량 디렉토리의 효율적 처리 — lazy_load와 배치 패턴**

Ch6에서 `load()`로 문서를 한 번에 메모리에 올리는 방법을 배웠는데요, 수 GB 규모의 문서 컬렉션에서는 메모리 부족(OOM)이 발생할 수 있습니다. `lazy_load()`는 제너레이터로 하나씩 처리하므로 메모리 사용량이 일정하게 유지됩니다.

```python
from langchain_community.document_loaders import DirectoryLoader, PyPDFLoader

# lazy_load로 메모리 효율적 처리
loader = DirectoryLoader(
    "documents/",
    glob="**/*.pdf",
    loader_cls=PyPDFLoader,
    show_progress=True,     # tqdm 진행률 표시
    use_multithreading=True, # I/O 바운드이므로 멀티스레드 효과적
    max_concurrency=4,       # 동시 로딩 스레드 수
)

# lazy_load → 배치 단위로 처리하여 메모리 제어
batch: list[Document] = []
BATCH_SIZE = 100

for doc in loader.lazy_load():
    batch.append(doc)
    if len(batch) >= BATCH_SIZE:
        process_batch(batch)  # 분할 → 인덱싱
        batch.clear()         # 메모리 해제

if batch:  # 나머지 처리
    process_batch(batch)
```

### 개념 2: 청킹 전략의 실전 선택 — "어떤 칼을 쓸까?"

> 💡 **비유**: 텍스트 분할은 **피자 자르기**와 같습니다. `RecursiveCharacterTextSplitter`는 규격대로 자르는 피자 커터이고, `SemanticChunker`는 토핑 경계를 보면서 자르는 셰프입니다. 그런데 실전에서는 "이 피자에는 어떤 칼이 맞는지" 판단하는 능력이 더 중요하죠.

[Ch6](ch06/)에서 `RecursiveCharacterTextSplitter`의 동작 원리(구분자 계층 `\n\n` → `\n` → `. ` → ` `)와 `SemanticChunker`의 임베딩 기반 분할을 배웠습니다. 여기서는 **어떤 상황에서 어떤 분할기를 선택해야 하는지**, 그리고 **최적의 chunk_size를 어떻게 찾는지**에 집중합니다.

**문서 유형별 청킹 전략 의사결정 트리:**

```
문서 유형은?
├─ 정형 문서 (계약서, 정책) → RecursiveCharacterTextSplitter
│  └─ chunk_size: 800~1200 (조항 단위 보존)
├─ 비정형 문서 (회의록, 보고서) → SemanticChunker
│  └─ breakpoint_threshold: "percentile" (주제 전환 감지)
├─ 코드/마크다운 → Language-specific Splitter
│  └─ 각 언어의 구문 경계 활용
└─ 짧은 문서 (<500자) → 분할하지 않음
   └─ 문서 전체를 단일 청크로 유지
```

**chunk_size 최적화 — 실험 기반 접근**

적절한 chunk_size는 문서 특성과 질의 유형에 따라 달라지기 때문에, 실무에서는 여러 설정으로 검색 품질을 비교하는 **벤치마크**를 실행합니다.

```python
from langchain_text_splitters import RecursiveCharacterTextSplitter
from langchain_experimental.text_splitter import SemanticChunker
from langchain_openai import OpenAIEmbeddings
import time

def benchmark_chunking(
    docs: list[Document],
    configs: list[dict],
) -> list[dict]:
    """여러 청킹 설정의 성능을 비교합니다.

    Args:
        docs: 테스트 문서 리스트
        configs: [{"name": "...", "chunk_size": N, "chunk_overlap": M}, ...]

    Returns:
        벤치마크 결과 리스트
    """
    results = []

    for cfg in configs:
        splitter = RecursiveCharacterTextSplitter(
            chunk_size=cfg["chunk_size"],
            chunk_overlap=cfg["chunk_overlap"],
        )

        start = time.time()
        chunks = splitter.split_documents(docs)
        elapsed = time.time() - start

        # 청크 크기 분포 분석
        sizes = [len(c.page_content) for c in chunks]
        results.append({
            "name": cfg["name"],
            "total_chunks": len(chunks),
            "avg_size": sum(sizes) / len(sizes),
            "min_size": min(sizes),
            "max_size": max(sizes),
            "elapsed_sec": round(elapsed, 3),
        })

    return results


# 실행 예시
configs = [
    {"name": "small",  "chunk_size": 500,  "chunk_overlap": 50},
    {"name": "medium", "chunk_size": 1000, "chunk_overlap": 200},
    {"name": "large",  "chunk_size": 1500, "chunk_overlap": 300},
]

# results = benchmark_chunking(all_docs, configs)
# 결과를 비교하여 검색 품질이 가장 높은 설정을 선택
```

> ⚠️ **흔한 오해**: "SemanticChunker가 항상 더 좋은 것 아닌가요?" — 아닙니다! SemanticChunker는 **모든 문장을 임베딩**해야 하므로 처리 시간과 API 비용이 크게 증가합니다. 1,000개 문서를 처리하면 임베딩 API 호출만 수만 건에 달할 수 있어요. 시맨틱 청킹은 고품질이 필요한 소수의 핵심 문서에 선별적으로 적용하고, 대부분의 문서에는 `RecursiveCharacterTextSplitter`를 쓰는 것이 현실적입니다.

**하이브리드 전략 — 문서 특성에 따른 자동 선택**

```python
from langchain_text_splitters import RecursiveCharacterTextSplitter
from langchain_experimental.text_splitter import SemanticChunker
from langchain_openai import OpenAIEmbeddings

def create_adaptive_splitter(
    doc: Document,
    embeddings: OpenAIEmbeddings,
) -> RecursiveCharacterTextSplitter | SemanticChunker:
    """문서 특성에 따라 최적의 분할기를 자동 선택합니다.

    - 짧은 문서: 분할하지 않음 (None 반환 대신 큰 chunk_size)
    - 정형 문서 (정책, 계약): RecursiveCharacterTextSplitter
    - 비정형/긴 문서: SemanticChunker
    """
    content_length = len(doc.page_content)
    category = doc.metadata.get("category", "general")

    # 500자 미만: 분할 불필요
    if content_length < 500:
        return RecursiveCharacterTextSplitter(chunk_size=content_length + 1)

    # 정형 문서: 규칙 기반 분할이 더 예측 가능
    if category in ("policy", "contract", "manual"):
        return RecursiveCharacterTextSplitter(
            chunk_size=1000,
            chunk_overlap=200,
            separators=["\n\n", "\n", ". ", " ", ""],
        )

    # 비정형/긴 문서: 의미 기반 분할
    if content_length > 3000:
        return SemanticChunker(
            embeddings=embeddings,
            breakpoint_threshold_type="percentile",
        )

    # 기본값
    return RecursiveCharacterTextSplitter(
        chunk_size=1000,
        chunk_overlap=200,
    )
```

### 개념 3: 메타데이터 스키마 설계 — 검색 품질의 숨은 열쇠

> 💡 **비유**: 메타데이터는 책의 **목차와 색인**과 같습니다. 책 내용을 100조각으로 찢어놓았다고 상상해보세요. 각 조각에 "3장, 15페이지, 소제목: 데이터 전처리"라는 스티커가 붙어 있다면 나중에 원본을 추적하고 정확한 출처를 인용할 수 있습니다. 스티커 없이는? 그냥 종이 조각일 뿐입니다.

Ch6에서 `Document` 객체의 `metadata` 필드를 배웠지만, "어떤 메타데이터를 어떤 구조로 넣어야 하는가"는 다루지 않았죠. 메타데이터 스키마 설계는 QA 시스템의 **출처 인용 품질**과 **필터링 검색 성능**을 직접적으로 좌우합니다.

**3계층 메타데이터 스키마:**

```
Level 1: 파일 메타데이터 (로딩 시점)
├─ file_name, file_type, file_size, file_path
├─ category (policy, manual, report, ...)
└─ last_modified (변경 감지용)

Level 2: 문서 메타데이터 (파서가 추출)
├─ title (HTML title, PDF title 속성)
├─ page (PDF 페이지 번호)
├─ author (가능한 경우)
└─ created_date (문서 자체의 생성일)

Level 3: 청크 메타데이터 (분할 후 부여)
├─ chunk_index, total_chunks
├─ content_hash (중복 감지 + 변경 추적)
├─ char_count (청크 크기)
└─ indexed_at (인덱싱 시점)
```

```python
from langchain_core.documents import Document
from datetime import datetime
from pathlib import Path
import hashlib
import os

def enrich_metadata(
    docs: list[Document],
    file_path: str | Path,
    category: str = "general",
) -> list[Document]:
    """3계층 메타데이터 스키마를 적용합니다.

    Args:
        docs: 원본 Document 리스트
        file_path: 원본 파일 경로
        category: 문서 분류 (예: "policy", "manual", "report")

    Returns:
        메타데이터가 보강된 Document 리스트
    """
    path = Path(file_path)
    stat = path.stat()

    for i, doc in enumerate(docs):
        # Level 1: 파일 메타데이터
        doc.metadata.update({
            "file_name": path.name,
            "file_type": path.suffix.lower(),
            "file_size": stat.st_size,
            "file_path": str(path.resolve()),
            "category": category,
            "last_modified": datetime.fromtimestamp(
                stat.st_mtime
            ).isoformat(),
        })

        # Level 2: 문서 메타데이터 (이미 로더가 넣어준 값 보존)
        # page, title 등은 로더가 자동으로 추가하므로 덮어쓰지 않음

        # Level 3: 청크 메타데이터
        doc.metadata.update({
            "chunk_index": i,
            "total_chunks": len(docs),
            "content_hash": hashlib.sha256(
                doc.page_content.encode()
            ).hexdigest()[:16],  # 앞 16자만 사용
            "char_count": len(doc.page_content),
            "indexed_at": datetime.now().isoformat(),
        })

    return docs
```

**메타데이터 기반 필터링 검색 — 범위를 좁혀 정확도 높이기**

메타데이터가 풍부하면 "인사 정책 문서에서만 검색", "최근 6개월 내 문서에서만 검색" 같은 **범위 한정 검색**이 가능해집니다. 이것은 단순 유사도 검색만으로는 달성하기 어려운, 프로덕션 QA 시스템의 핵심 차별점입니다.

```python
# 카테고리 필터링 — "정책 문서에서만 검색"
results = vectorstore.similarity_search(
    "연차 휴가 정책",
    k=5,
    filter={"category": "policy"},
)

# 복합 필터 — "2024년 이후 보고서에서 검색"
# (FAISS는 단순 필터만 지원, 복합 필터는 Chroma/Qdrant 등 권장)
results = vectorstore.similarity_search(
    "매출 분석",
    k=5,
    filter={"category": "report"},
)

# 출처 인용에 메타데이터 활용
for doc in results:
    source = doc.metadata["file_name"]
    page = doc.metadata.get("page", "N/A")
    print(f"📄 출처: {source}, 페이지: {page}")
```

> 🔥 **실무 팁**: `content_hash`에 MD5 대신 SHA-256을 사용하는 것을 권장합니다. MD5는 충돌(collision) 가능성이 있어 보안 맥락에서 더 이상 권장되지 않거든요. 해시의 목적이 중복 감지라면 SHA-256의 앞 16자만 사용해도 충분하고, 성능 차이는 무시할 수 있는 수준입니다.

### 개념 4: 증분 인덱싱 — 변경된 문서만 똑똑하게 처리하기

> 💡 **비유**: 증분 인덱싱은 **도서관 사서의 일과**와 같습니다. 매일 아침 전체 서가를 다시 정리하는 사서는 없죠. 어제 반납된 책, 오늘 새로 들어온 책만 처리하면 됩니다. `last_modified` 타임스탬프가 "반납일 스탬프" 역할을 합니다.

Ch7에서 `FAISS.from_documents()`와 `merge_from()`의 기본 사용법을 배웠습니다. 하지만 실전에서는 "어떤 문서가 새로 추가되었고, 어떤 문서가 수정되었는지"를 감지해서 **변경분만 인덱싱**하는 것이 핵심입니다. 전체 재인덱싱은 시간과 임베딩 API 비용을 낭비하니까요.

**변경 감지 전략:**

```python
import json
from pathlib import Path
from datetime import datetime

class IndexManifest:
    """인덱싱된 파일의 상태를 추적하는 매니페스트.

    각 파일의 마지막 수정 시간과 크기를 기록하여
    변경/추가/삭제된 파일을 감지합니다.
    """

    def __init__(self, manifest_path: str = "index_manifest.json"):
        self.path = Path(manifest_path)
        self.entries: dict[str, dict] = {}
        if self.path.exists():
            self.entries = json.loads(self.path.read_text())

    def get_changed_files(
        self, file_paths: list[Path]
    ) -> tuple[list[Path], list[Path], list[str]]:
        """새로 추가/수정/삭제된 파일을 분류합니다.

        Returns:
            (new_files, modified_files, deleted_keys)
        """
        current_keys = set()
        new_files: list[Path] = []
        modified_files: list[Path] = []

        for fp in file_paths:
            key = str(fp.resolve())
            current_keys.add(key)
            stat = fp.stat()

            if key not in self.entries:
                new_files.append(fp)
            elif stat.st_mtime > self.entries[key]["mtime"]:
                modified_files.append(fp)
            # else: 변경 없음 — 건너뜀

        # 삭제된 파일 감지
        deleted = [
            k for k in self.entries if k not in current_keys
        ]

        return new_files, modified_files, deleted

    def update(self, file_path: Path) -> None:
        """파일을 인덱싱 완료로 기록합니다."""
        stat = file_path.stat()
        self.entries[str(file_path.resolve())] = {
            "mtime": stat.st_mtime,
            "size": stat.st_size,
            "indexed_at": datetime.now().isoformat(),
        }

    def remove(self, key: str) -> None:
        """삭제된 파일을 매니페스트에서 제거합니다."""
        self.entries.pop(key, None)

    def save(self) -> None:
        """매니페스트를 디스크에 저장합니다."""
        self.path.write_text(
            json.dumps(self.entries, indent=2, ensure_ascii=False)
        )
```

**배치 임베딩으로 API 비용 최적화**

FAISS에 문서를 추가할 때 `from_documents()`는 내부적으로 임베딩 API를 호출합니다. 대량의 청크를 한 번에 넘기면 LangChain이 자동으로 배치 처리하지만, `merge_from()`으로 소량씩 병합하면 배치 효율이 떨어질 수 있습니다. 변경분을 모아서 한 번에 인덱싱하는 것이 비용 효율적입니다.

```python
from langchain_community.vectorstores import FAISS
from langchain_openai import OpenAIEmbeddings

def build_or_update_index(
    new_chunks: list[Document],
    index_dir: str,
    embeddings: OpenAIEmbeddings,
) -> FAISS:
    """기존 인덱스가 있으면 병합, 없으면 새로 생성합니다.

    새 청크를 한 번에 임베딩하여 API 호출 효율을 최적화합니다.
    """
    index_path = Path(index_dir)

    if not new_chunks:
        # 변경 없음 — 기존 인덱스 그대로 반환
        return FAISS.load_local(
            index_dir, embeddings,
            allow_dangerous_deserialization=True,
        )

    # 새 청크를 한 번에 임베딩 (배치 최적화)
    new_index = FAISS.from_documents(new_chunks, embeddings)

    if index_path.exists() and (index_path / "index.faiss").exists():
        # 기존 인덱스에 병합
        existing = FAISS.load_local(
            index_dir, embeddings,
            allow_dangerous_deserialization=True,
        )
        existing.merge_from(new_index)
        existing.save_local(index_dir)
        logger.info(
            f"✓ 인덱스 병합 완료: +{len(new_chunks)}개 청크"
        )
        return existing
    else:
        # 새 인덱스 생성
        new_index.save_local(index_dir)
        logger.info(
            f"✓ 새 인덱스 생성: {len(new_chunks)}개 청크"
        )
        return new_index
```

> 💡 **알고 계셨나요?**: `FAISS.load_local()`에 `allow_dangerous_deserialization=True`를 반드시 넘겨야 하는 이유가 있습니다. FAISS 인덱스를 저장할 때 Python의 `pickle`을 사용하는데, pickle 파일은 악의적인 코드를 포함할 수 있거든요. 이 파라미터는 "이 파일이 안전하다는 것을 내가 보증합니다"라는 명시적 동의입니다. 외부에서 받은 인덱스 파일은 반드시 출처를 확인하세요!

## 실습: 직접 해보기

지금까지 다룬 **에러 복구, 메타데이터 보강, 변경 감지, 증분 인덱싱**을 모두 통합한 프로덕션 수준의 `IngestionPipeline`입니다. 복사-붙여넣기로 바로 실행할 수 있는 완전한 코드입니다.

```python
"""
프로덕션 수준 문서 수집·인덱싱 파이프라인

Ch6-7에서 배운 기본 도구들을 결합하여, 에러 복구·증분 인덱싱·
메타데이터 보강을 갖춘 견고한 파이프라인을 구현합니다.

필요 패키지:
  pip install langchain langchain-community langchain-openai
  pip install langchain-text-splitters langchain-experimental
  pip install faiss-cpu pypdf docx2txt beautifulsoup4 lxml
  pip install python-dotenv
"""
import os
import json
import logging
import hashlib
from pathlib import Path
from datetime import datetime
from typing import List, Optional
from dataclasses import dataclass, field

from dotenv import load_dotenv
from langchain_core.documents import Document
from langchain_community.document_loaders import (
    PyPDFLoader,
    Docx2txtLoader,
    BSHTMLLoader,
)
from langchain_text_splitters import RecursiveCharacterTextSplitter
from langchain_community.vectorstores import FAISS
from langchain_openai import OpenAIEmbeddings

# 환경 변수 로드 (.env 파일에 OPENAI_API_KEY 필요)
load_dotenv()

# 로깅 설정
logging.basicConfig(level=logging.INFO)
logger = logging.getLogger(__name__)

# ── 상수 정의 ──────────────────────────────────────
MODEL_NAME = "text-embedding-3-small"
CHUNK_SIZE = 1000
CHUNK_OVERLAP = 200
INDEX_DIR = "faiss_index"
MANIFEST_FILE = "index_manifest.json"

# 확장자 → 로더 클래스 매핑
LOADER_MAP: dict[str, type] = {
    ".pdf": PyPDFLoader,
    ".docx": Docx2txtLoader,
    ".doc": Docx2txtLoader,
    ".html": BSHTMLLoader,
    ".htm": BSHTMLLoader,
}


@dataclass
class IngestionStats:
    """파이프라인 실행 통계를 추적합니다."""
    files_processed: int = 0
    files_skipped: int = 0      # 변경 없어 건너뛴 파일
    files_failed: int = 0
    total_documents: int = 0
    total_chunks: int = 0
    duplicates_removed: int = 0
    failed_files: list[str] = field(default_factory=list)


class IndexManifest:
    """인덱싱된 파일 상태를 추적하여 증분 인덱싱을 지원합니다."""

    def __init__(self, path: str = MANIFEST_FILE):
        self.path = Path(path)
        self.entries: dict[str, dict] = {}
        if self.path.exists():
            self.entries = json.loads(self.path.read_text())

    def needs_update(self, file_path: Path) -> bool:
        """파일이 새로 추가되었거나 수정되었는지 확인합니다."""
        key = str(file_path.resolve())
        if key not in self.entries:
            return True  # 새 파일
        return file_path.stat().st_mtime > self.entries[key]["mtime"]

    def mark_indexed(self, file_path: Path) -> None:
        """파일을 인덱싱 완료로 기록합니다."""
        stat = file_path.stat()
        self.entries[str(file_path.resolve())] = {
            "mtime": stat.st_mtime,
            "size": stat.st_size,
            "indexed_at": datetime.now().isoformat(),
        }

    def save(self) -> None:
        self.path.write_text(
            json.dumps(self.entries, indent=2, ensure_ascii=False)
        )


class IngestionPipeline:
    """프로덕션 수준의 문서 수집·인덱싱 파이프라인.

    주요 특징:
    - 에러 격리: 개별 파일 실패가 전체 파이프라인을 중단하지 않음
    - 증분 인덱싱: 변경된 파일만 재처리하여 시간·비용 절약
    - 메타데이터 보강: 3계층 스키마로 출처 추적 및 필터링 검색 지원
    - 중복 제거: SHA-256 해시 기반 동일 청크 감지

    사용법:
        pipeline = IngestionPipeline()
        stats = pipeline.run("./documents")
        print(stats)
    """

    def __init__(
        self,
        chunk_size: int = CHUNK_SIZE,
        chunk_overlap: int = CHUNK_OVERLAP,
        index_dir: str = INDEX_DIR,
        manifest_file: str = MANIFEST_FILE,
    ):
        self.embeddings = OpenAIEmbeddings(model=MODEL_NAME)
        self.text_splitter = RecursiveCharacterTextSplitter(
            chunk_size=chunk_size,
            chunk_overlap=chunk_overlap,
            separators=["\n\n", "\n", ". ", " ", ""],
        )
        self.index_dir = index_dir
        self.manifest = IndexManifest(manifest_file)
        self.vectorstore: Optional[FAISS] = None

    # ── 1단계: 문서 로딩 (에러 격리 포함) ──────────
    def load_document(self, file_path: Path) -> list[Document]:
        """단일 파일을 로드하고 3계층 메타데이터를 부여합니다.

        실패 시 빈 리스트를 반환하여 파이프라인 중단을 방지합니다.
        """
        ext = file_path.suffix.lower()
        if ext not in LOADER_MAP:
            return []

        try:
            loader = LOADER_MAP[ext](str(file_path))
            docs = loader.load()

            # 빈 결과 감지 (이미지 전용 PDF 등)
            if not docs or all(
                not d.page_content.strip() for d in docs
            ):
                logger.warning(f"빈 콘텐츠: {file_path.name}")
                return []

            # Level 1 + Level 2 메타데이터 보강
            stat = file_path.stat()
            for doc in docs:
                doc.metadata.update({
                    "file_name": file_path.name,
                    "file_type": ext,
                    "file_size": stat.st_size,
                    "file_path": str(file_path.resolve()),
                    "category": self._infer_category(file_path),
                    "last_modified": datetime.fromtimestamp(
                        stat.st_mtime
                    ).isoformat(),
                    "indexed_at": datetime.now().isoformat(),
                })

            logger.info(
                f"✓ 로드: {file_path.name} ({len(docs)}개 문서)"
            )
            return docs

        except Exception as e:
            logger.warning(f"✗ 로드 실패: {file_path.name} — {e}")
            return []

    def _infer_category(self, file_path: Path) -> str:
        """디렉토리 구조에서 문서 카테고리를 추론합니다.

        예: documents/policies/vacation.pdf → "policies"
        """
        parts = file_path.parent.parts
        if parts:
            return parts[-1]  # 가장 가까운 상위 디렉토리 이름
        return "general"

    def load_directory(
        self,
        dir_path: str | Path,
        incremental: bool = True,
    ) -> tuple[list[Document], IngestionStats]:
        """디렉토리 내 문서를 로드합니다.

        Args:
            dir_path: 문서 디렉토리 경로
            incremental: True이면 변경된 파일만 처리
        """
        dir_path = Path(dir_path)
        all_docs: list[Document] = []
        stats = IngestionStats()

        supported_files = sorted(
            f for f in dir_path.rglob("*")
            if f.is_file() and f.suffix.lower() in LOADER_MAP
        )

        logger.info(f"발견된 문서: {len(supported_files)}개")

        for file_path in supported_files:
            # 증분 인덱싱: 변경 없는 파일은 건너뜀
            if incremental and not self.manifest.needs_update(file_path):
                stats.files_skipped += 1
                continue

            docs = self.load_document(file_path)
            if docs:
                all_docs.extend(docs)
                stats.files_processed += 1
                stats.total_documents += len(docs)
                self.manifest.mark_indexed(file_path)
            else:
                stats.files_failed += 1
                stats.failed_files.append(str(file_path))

        if stats.files_skipped > 0:
            logger.info(
                f"증분 모드: {stats.files_skipped}개 파일 건너뜀 "
                f"(변경 없음)"
            )

        return all_docs, stats

    # ── 2단계: 텍스트 분할 ──────────────────────────
    def split_documents(self, docs: list[Document]) -> list[Document]:
        """문서를 청크로 분할하고 Level 3 메타데이터를 부여합니다."""
        chunks = self.text_splitter.split_documents(docs)

        for i, chunk in enumerate(chunks):
            chunk.metadata["chunk_index"] = i
            chunk.metadata["char_count"] = len(chunk.page_content)
            chunk.metadata["content_hash"] = hashlib.sha256(
                chunk.page_content.encode()
            ).hexdigest()[:16]

        logger.info(
            f"✓ 분할: {len(docs)}개 문서 → {len(chunks)}개 청크"
        )
        return chunks

    # ── 3단계: 중복 제거 ────────────────────────────
    def deduplicate(self, chunks: list[Document]) -> list[Document]:
        """SHA-256 해시 기반으로 중복 청크를 제거합니다."""
        seen: set[str] = set()
        unique: list[Document] = []

        for chunk in chunks:
            h = chunk.metadata["content_hash"]
            if h not in seen:
                seen.add(h)
                unique.append(chunk)

        removed = len(chunks) - len(unique)
        if removed > 0:
            logger.info(f"✓ 중복 제거: {removed}개 청크 제거됨")
        return unique, removed

    # ── 4단계: 벡터 인덱싱 (증분 병합) ──────────────
    def build_or_update_index(
        self,
        chunks: list[Document],
    ) -> FAISS:
        """기존 인덱스가 있으면 병합, 없으면 새로 생성합니다."""
        index_path = Path(self.index_dir)

        if not chunks:
            if index_path.exists():
                self.vectorstore = FAISS.load_local(
                    self.index_dir,
                    self.embeddings,
                    allow_dangerous_deserialization=True,
                )
            return self.vectorstore

        logger.info(f"벡터 인덱싱: {len(chunks)}개 청크...")

        # 새 청크를 한 번에 임베딩 (배치 효율 극대화)
        new_index = FAISS.from_documents(chunks, self.embeddings)

        if index_path.exists() and (index_path / "index.faiss").exists():
            self.vectorstore = FAISS.load_local(
                self.index_dir,
                self.embeddings,
                allow_dangerous_deserialization=True,
            )
            self.vectorstore.merge_from(new_index)
            logger.info("✓ 기존 인덱스에 병합 완료")
        else:
            self.vectorstore = new_index
            logger.info("✓ 새 인덱스 생성 완료")

        self.vectorstore.save_local(self.index_dir)
        return self.vectorstore

    # ── 전체 파이프라인 실행 ────────────────────────
    def run(
        self,
        source_dir: str | Path,
        incremental: bool = True,
    ) -> IngestionStats:
        """전체 파이프라인을 실행합니다.

        로드(변경분) → 분할 → 중복 제거 → 인덱싱(병합)

        Args:
            source_dir: 문서 디렉토리 경로
            incremental: True이면 변경된 파일만 처리

        Returns:
            실행 통계
        """
        logger.info(
            f"=== 인덱싱 파이프라인 시작 "
            f"({'증분' if incremental else '전체'} 모드) ==="
        )

        # 1. 문서 로딩 (변경 감지 포함)
        docs, stats = self.load_directory(source_dir, incremental)
        if not docs:
            logger.info("처리할 새/변경 문서가 없습니다.")
            self.manifest.save()
            return stats

        # 2. 텍스트 분할
        chunks = self.split_documents(docs)

        # 3. 중복 제거
        unique_chunks, removed = self.deduplicate(chunks)
        stats.total_chunks = len(unique_chunks)
        stats.duplicates_removed = removed

        # 4. 벡터 인덱싱 (증분 병합)
        self.build_or_update_index(unique_chunks)

        # 5. 매니페스트 저장
        self.manifest.save()

        logger.info("=== 파이프라인 완료 ===")
        logger.info(
            f"처리: {stats.files_processed}개 / "
            f"건너뜀: {stats.files_skipped}개 / "
            f"실패: {stats.files_failed}개 / "
            f"청크: {stats.total_chunks}개 "
            f"(중복 제거: {stats.duplicates_removed}개)"
        )
        return stats


# ── 실행 예시 ───────────────────────────────────────
```

## 더 깊이 알아보기

### 시맨틱 청킹의 탄생 — Greg Kamradt의 5단계 이론

텍스트를 어떻게 자를지는 RAG 연구의 핵심 질문 중 하나였습니다. 2023년, AI 엔지니어 **Greg Kamradt**는 "5 Levels of Text Splitting"이라는 프레임워크를 제시하며 이 분야에 큰 영향을 미쳤습니다.

그의 5단계는 다음과 같습니다:
1. **Character Split** — 단순히 N글자마다 자르기
2. **Recursive Character Split** — 문단(`\n\n`), 줄바꿈(`\n`), 공백(` `) 순서로 계층적 분할
3. **Document-specific Split** — 코드, 마크다운, PDF 등 형식에 맞는 분할기 사용
4. **Semantic Chunking** — 임베딩 유사도로 의미 경계를 찾아 분할
5. **Agentic Chunking** — AI 에이전트가 문맥을 이해하고 최적의 분할 결정

Kamradt의 시맨틱 청킹 알고리즘은 놀라울 정도로 직관적이었습니다. 첫 번째 문장의 임베딩을 구하고, 두 번째 문장과 비교하고, 계속 이어나가면서 임베딩 거리가 **급격히 증가하는 지점**을 찾아 "여기서 주제가 바뀌었구나"라고 판단하는 것이죠. 이 알고리즘은 LangChain의 `SemanticChunker`로 공식 채택되어, 우리가 위에서 하이브리드 전략에 활용한 바로 그 도구가 되었습니다.

### FAISS의 탄생 — Facebook에서 태어난 검색 엔진

FAISS(Facebook AI Similarity Search)는 2017년 Meta(당시 Facebook) AI Research 팀이 발표한 라이브러리입니다. 당시 Facebook은 매일 수십억 장의 이미지에서 유사 이미지를 찾아야 했는데, 기존의 검색 방법으로는 속도가 턱없이 부족했습니다.

연구팀의 Jeff Johnson, Matthijs Douze, Hervé Jégou는 GPU 가속과 양자화(Quantization) 기법을 결합하여, **10억 개의 벡터에서 밀리초 단위로 최근접 이웃을 찾는** 라이브러리를 만들었습니다. 놀랍게도 이 라이브러리는 오픈소스로 공개되었고, 지금은 RAG 시스템의 사실상 표준 벡터 검색 엔진이 되었죠.

### 프로덕션 인덱싱의 숨은 함정들

실전에서 인덱싱 파이프라인을 운영하다 보면 교과서에 없는 문제들을 만나게 됩니다.

- **문자 인코딩 지옥**: 한국 기업 환경에서는 EUC-KR, CP949로 작성된 레거시 문서가 많습니다. UTF-8만 기대하면 절반이 깨져버릴 수 있어요. `chardet` 라이브러리로 인코딩을 자동 감지하는 전처리 단계를 추가하는 것이 안전합니다.
- **PDF 테이블의 저주**: PyPDFLoader는 테이블 내용을 줄 단위로 쪼개서 반환하기 때문에, "매출: 100억"이 "매출:" 과 "100억"으로 분리되는 경우가 흔합니다. 테이블이 중요한 문서라면 `UnstructuredPDFLoader`나 `pdfplumber` 기반 커스텀 로더를 고려하세요.
- **임베딩 API 비용 폭탄**: 1,000개 문서 × 평균 50청크 = 50,000번의 임베딩 호출. `text-embedding-3-small` 기준으로도 무시 못 할 비용이 됩니다. 위에서 다룬 증분 인덱싱은 이 비용을 극적으로 줄여줍니다.

## 흔한 오해와 팁

> ⚠️ **흔한 오해**: "chunk_size를 작게 할수록 검색 정확도가 올라간다" — 이는 절반만 맞는 이야기입니다. 청크가 너무 작으면 하나의 개념이 여러 청크에 쪼개져서 오히려 맥락을 잃어버립니다. 반대로 너무 크면 관련 없는 내용이 섞여 노이즈가 됩니다. 실무에서는 위에서 보여드린 벤치마크 함수처럼 여러 설정을 실험하며 최적값을 찾는 것이 일반적입니다.

> 💡 **알고 계셨나요?**: `merge_from()` 이후 인덱스 크기가 계속 커지기만 한다는 점을 주의하세요. 삭제된 문서의 벡터는 여전히 인덱스에 남아 있거든요. FAISS는 개별 벡터 삭제를 지원하지 않기 때문에, 삭제가 누적되면 주기적으로 전체 재빌드(`incremental=False`)를 실행하는 스케줄을 세우는 것이 좋습니다.

> 🔥 **실무 팁**: 파이프라인을 처음 세팅할 때는 `incremental=False`로 전체 인덱싱을 한 번 돌리고, 이후 일상 운영에서는 `incremental=True`로 변경분만 처리하세요. 주말마다 전체 재빌드를 스케줄링하면 삭제된 문서의 찌꺼기도 정리할 수 있습니다.

> 🔥 **실무 팁**: `_infer_category()`처럼 디렉토리 구조에서 카테고리를 자동 추론하는 패턴은 실전에서 매우 유용합니다. 문서 관리 담당자가 `documents/policies/`, `documents/reports/`, `documents/manuals/`처럼 폴더를 정리해두면, 코드 수정 없이 카테고리 필터링 검색이 바로 동작하거든요.
