# 개발 환경 설정

> LangChain 개발을 위한 Python 환경 구축부터 API 키 관리, 프로젝트 구조 설계까지 — 실전 개발의 첫 걸음을 내딛습니다.

## 개요

이 섹션에서는 LangChain 애플리케이션을 개발하기 위한 완전한 환경을 구축합니다. Python 가상환경을 만들고, 필요한 패키지를 설치하고, API 키를 안전하게 관리하는 방법을 배웁니다. 마지막으로 확장 가능한 프로젝트 디렉토리 구조를 설계하여, 이후 챕터에서 만들어갈 모든 코드의 기반을 마련합니다.

**선수 지식**: [Session 1.1: LLM 애플리케이션의 진화와 LangChain](session_1_1.md)에서 배운 LangChain의 설계 철학, [Session 1.2](session_1_2.md)에서 살펴본 langchain-core, langchain-community, 통합 패키지의 분리 구조

**학습 목표**:
- Python 가상환경을 생성하고 활성화할 수 있다
- LangChain 핵심 패키지와 통합 패키지를 올바른 순서로 설치할 수 있다
- `.env` 파일과 `python-dotenv`를 사용하여 API 키를 안전하게 관리할 수 있다
- 확장 가능한 LangChain 프로젝트 디렉토리 구조를 설계할 수 있다

## 왜 알아야 할까?

"코드 한 줄 쓰기 전에 환경부터 잡아라." 소프트웨어 개발에서 가장 흔한 좌절 경험 중 하나는 환경 설정 문제입니다. 패키지 버전 충돌, API 키 노출 사고, 뒤엉킨 프로젝트 구조 — 이런 문제들은 개발 초기에 제대로 잡지 않으면 프로젝트가 커질수록 눈덩이처럼 불어납니다.

특히 LangChain은 `langchain-core`, `langchain-openai`, `langchain-community` 등 여러 패키지로 분리되어 있어서, [Session 1.2](session_1_2.md)에서 배운 아키텍처를 이해한 상태로 올바르게 설치하는 것이 중요합니다. 또한 OpenAI, Anthropic 같은 외부 API 키를 다루기 때문에, 보안 사고를 예방하는 습관을 처음부터 들여야 하죠.

## 핵심 개념

### 개념 1: Python 가상환경 — 프로젝트마다 독립된 작업실

> 💡 **비유**: 가상환경은 **요리사의 개인 주방**과 같습니다. 한 주방에서 한식과 양식을 동시에 준비하면 재료와 도구가 뒤섞이겠죠? 프로젝트마다 별도의 주방(가상환경)을 마련하면, 각 프로젝트에 필요한 재료(패키지)를 독립적으로 관리할 수 있습니다.

Python 가상환경은 프로젝트별로 격리된 Python 인터프리터와 패키지 공간을 제공합니다. LangChain 프로젝트 A에서는 `langchain==0.3.x`를, 프로젝트 B에서는 `langchain==1.0.x`를 사용해야 할 수 있거든요. 가상환경 없이는 이런 상황을 관리할 수 없습니다.

**venv — Python 기본 제공 도구**

Python 3.3부터 내장된 `venv` 모듈은 가장 기본적이면서 안정적인 선택입니다:

```bash
# 가상환경 생성
python -m venv .venv

# 활성화 (macOS/Linux)
source .venv/bin/activate

# 활성화 (Windows)
.venv\Scripts\activate

# 비활성화
deactivate
```

**uv — 차세대 패키지 관리 도구**

최근 Python 생태계에서 빠르게 부상하고 있는 `uv`는 Rust로 작성된 초고속 패키지 매니저입니다. `venv` 대비 **80배 빠른** 가상환경 생성 속도를 자랑하며, 글로벌 캐시를 사용하여 디스크 공간도 절약합니다:

```bash
# uv 설치 (macOS/Linux)
curl -LsSf https://astral.sh/uv/install.sh | sh

# uv로 가상환경 생성
uv venv .venv

# 활성화는 venv와 동일
source .venv/bin/activate
```

> 🔥 **실무 팁**: 가상환경 디렉토리 이름은 `.venv`로 통일하세요. 대부분의 IDE(VS Code, PyCharm)가 `.venv`를 자동 감지하고, `.gitignore`에도 쉽게 추가할 수 있습니다.

### 개념 2: LangChain 패키지 설치 — 필요한 블록만 골라 조립하기

> 💡 **비유**: LangChain 패키지는 **레고 블록 세트**와 같습니다. 거대한 올인원 박스를 사지 않고, 원하는 테마(중세 성, 우주선)에 맞는 블록 세트만 골라 사는 거예요. [Session 1.2](session_1_2.md)에서 배운 모듈형 아키텍처 덕분에, 필요한 패키지만 설치하여 프로젝트를 가볍게 유지할 수 있습니다.

[Session 1.2](session_1_2.md)에서 살펴봤듯이, LangChain은 여러 패키지로 분리되어 있습니다. 각 패키지의 역할을 이해하고 필요한 것만 설치하는 것이 중요합니다:

```bash
# 1. 핵심 패키지 — 거의 모든 프로젝트에 필요
pip install langchain           # 메인 프레임워크 (langchain-core 자동 포함)
pip install langchain-openai    # OpenAI 모델 통합

# 2. 유틸리티 — API 키 관리
pip install python-dotenv       # .env 파일에서 환경변수 로드

# 3. 필요에 따라 추가 설치
pip install langchain-community # 서드파티 통합 (다양한 도구/로더)
pip install langchain-anthropic # Anthropic Claude 모델 통합
pip install langchain-chroma    # Chroma 벡터 스토어 (RAG 구축 시)
```

한 번에 설치하려면:

```bash
pip install langchain langchain-openai python-dotenv
```

설치 확인은 이렇게 합니다:

```python
# 버전 확인
import langchain
print(f"langchain: {langchain.__version__}")  # 예: 1.0.5

import langchain_core
print(f"langchain-core: {langchain_core.__version__}")  # 예: 1.2.16

import langchain_openai
print(f"langchain-openai: {langchain_openai.__version__}")  # 예: 1.1.10
```

> ⚠️ **흔한 오해**: `pip install langchain`만 하면 OpenAI 모델을 바로 쓸 수 있다고 생각하기 쉽습니다. 하지만 `langchain`은 코어 프레임워크만 제공하고, 모델 연동 패키지(`langchain-openai` 등)는 별도 설치해야 합니다. 이것이 바로 [Session 1.2](session_1_2.md)에서 배운 모듈형 아키텍처의 특성이에요.

### 개념 3: API 키 관리 — 비밀번호를 금고에 보관하기

> 💡 **비유**: API 키는 **집 열쇠**와 같습니다. 열쇠를 현관 앞 화분 밑에 두면(코드에 하드코딩하면) 누구나 가져갈 수 있죠. 금고에 보관하고(`.env` 파일) 필요할 때만 꺼내 쓰는(환경변수 로드) 것이 안전합니다. 그리고 금고의 존재를 알리지 않는 것(`.gitignore`)도 중요하고요.

LangChain은 다양한 외부 API(OpenAI, Anthropic, Google 등)를 사용합니다. 이 API들의 키를 안전하게 관리하는 것은 **보안의 기본** 중 기본입니다.

**Step 1: `.env` 파일 생성**

```bash
# 프로젝트 루트에 .env 파일 생성
OPENAI_API_KEY=sk-proj-여기에-실제-키-입력
LANGCHAIN_TRACING_V2=true
LANGCHAIN_API_KEY=lsv2_여기에-실제-키-입력
```

**Step 2: `.gitignore`에 추가**

```gitignore
# 환경변수 파일 — 절대 커밋하지 않기!
.env
.env.local
.env.*.local
```

**Step 3: `python-dotenv`로 로드**

```python
import os
from dotenv import load_dotenv

# .env 파일의 변수를 환경변수로 로드
load_dotenv()

# 환경변수에서 API 키 읽기
openai_api_key = os.getenv("OPENAI_API_KEY")

# 키가 설정되었는지 확인
if not openai_api_key:
    raise ValueError("OPENAI_API_KEY가 설정되지 않았습니다. .env 파일을 확인하세요.")

print("API 키가 정상적으로 로드되었습니다!")
```

LangChain의 통합 패키지들은 환경변수 이름 규칙을 따르면 **자동으로** API 키를 감지합니다:

```python
from langchain_openai import ChatOpenAI

# OPENAI_API_KEY 환경변수가 설정되어 있으면 자동 감지!
# 별도로 api_key 파라미터를 전달할 필요 없음
llm = ChatOpenAI(model="gpt-4o")
```

> 🔥 **실무 팁**: `.env.example` 파일을 함께 만들어 Git에 커밋하세요. 실제 키 값 대신 설명을 넣어두면, 다른 개발자가 어떤 환경변수가 필요한지 바로 알 수 있습니다:
> ```
> OPENAI_API_KEY=여기에_OpenAI_API_키_입력
> LANGCHAIN_API_KEY=여기에_LangSmith_API_키_입력
> ```

### 개념 4: 프로젝트 디렉토리 구조 — 설계도를 먼저 그리기

> 💡 **비유**: 프로젝트 구조는 **도서관의 서가 배치**와 같습니다. 책(코드)이 10권일 때는 아무렇게나 쌓아도 찾을 수 있지만, 1000권이 되면 체계적인 분류 체계 없이는 원하는 책을 찾을 수 없습니다.

LangChain 프로젝트가 커지면 체인, 에이전트, 도구, 프롬프트 등 다양한 컴포넌트가 생깁니다. 처음부터 확장 가능한 구조를 설계해두면, 나중에 리팩토링하느라 고생하지 않아도 됩니다:

```
langchain-project/
├── .env                    # API 키 등 환경변수 (Git 제외)
├── .env.example            # 환경변수 템플릿 (Git 포함)
├── .gitignore              # Git 제외 파일 목록
├── requirements.txt        # 의존성 목록
├── src/                    # 소스 코드
│   ├── __init__.py
│   ├── chains/             # LCEL 체인 정의
│   │   └── __init__.py
│   ├── agents/             # 에이전트 로직
│   │   └── __init__.py
│   ├── tools/              # 커스텀 도구
│   │   └── __init__.py
│   ├── prompts/            # 프롬프트 템플릿
│   │   └── __init__.py
│   └── utils/              # 유틸리티 함수
│       └── __init__.py
├── tests/                  # 테스트 코드
│   └── __init__.py
├── notebooks/              # Jupyter 노트북 (실험용)
└── data/                   # 데이터 파일 (문서, PDF 등)
```

이 구조에서 각 디렉토리의 역할은 명확합니다:

| 디렉토리 | 역할 | 예시 |
|-----------|------|------|
| `src/chains/` | LCEL 체인 정의 | `rag_chain.py`, `summary_chain.py` |
| `src/agents/` | 에이전트 로직 | `research_agent.py` |
| `src/tools/` | 커스텀 도구 | `web_search.py`, `calculator.py` |
| `src/prompts/` | 프롬프트 템플릿 | `qa_prompt.py` |
| `src/utils/` | 공통 유틸리티 | `config.py`, `logger.py` |

**Step 5: 환경 검증 스크립트 작성 및 실행**

```python
# src/utils/verify_setup.py
"""개발 환경 검증 스크립트"""

import os
import sys
from dotenv import load_dotenv

def verify_environment():
    """개발 환경이 올바르게 설정되었는지 검증합니다."""

    print("=" * 50)
    print("LangChain 개발 환경 검증")
    print("=" * 50)

    # 1. Python 버전 확인
    python_version = sys.version.split()[0]
    major, minor = map(int, python_version.split(".")[:2])
    if major >= 3 and minor >= 10:
        print(f"✓ Python 버전: {python_version} (3.10 이상 필요)")
    else:
        print(f"✗ Python 버전: {python_version} — 3.10 이상이 필요합니다!")
        return False

    # 2. 패키지 임포트 확인
    packages = {
        "langchain": "langchain",
        "langchain_core": "langchain-core",
        "langchain_openai": "langchain-openai",
        "dotenv": "python-dotenv",
    }

    for module_name, package_name in packages.items():
        try:
            module = __import__(module_name)
            version = getattr(module, "__version__", "버전 정보 없음")
            print(f"✓ {package_name}: {version}")
        except ImportError:
            print(f"✗ {package_name}가 설치되지 않았습니다. "
                  f"pip install {package_name}")
            return False

    # 3. 환경변수 확인
    load_dotenv()
    api_key = os.getenv("OPENAI_API_KEY")
    if api_key and api_key != "여기에_OpenAI_API_키_입력":
        # API 키의 앞 8자만 보여주고 나머지는 마스킹
        masked_key = api_key[:8] + "..." + api_key[-4:]
        print(f"✓ OPENAI_API_KEY: {masked_key}")
    else:
        print("✗ OPENAI_API_KEY가 설정되지 않았습니다. "
              ".env 파일을 확인하세요.")
        return False

    # 4. 디렉토리 구조 확인
    required_dirs = [
        "src/chains", "src/agents", "src/tools",
        "src/prompts", "src/utils", "tests", "notebooks", "data"
    ]
    for dir_path in required_dirs:
        if os.path.isdir(dir_path):
            print(f"✓ 디렉토리: {dir_path}/")
        else:
            print(f"✗ 디렉토리 없음: {dir_path}/")

    print("=" * 50)
    print("환경 검증 완료! LangChain 개발을 시작할 준비가 되었습니다.")
    print("=" * 50)
    return True


if __name__ == "__main__":
    verify_environment()
```

**Step 6: 첫 번째 LangChain 코드 실행**

```python
# src/chains/hello_langchain.py
"""첫 번째 LangChain 체인 — 환경 설정이 올바른지 확인합니다."""

import os
from dotenv import load_dotenv
from langchain_openai import ChatOpenAI
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser

# .env 파일에서 환경변수 로드
load_dotenv()

# 모델 설정
MODEL_NAME = "gpt-4o"
TEMPERATURE = 0.7

# 컴포넌트 정의
prompt = ChatPromptTemplate.from_template(
    "당신은 친절한 AI 어시스턴트입니다. "
    "다음 주제에 대해 한 문장으로 설명해주세요: {topic}"
)

llm = ChatOpenAI(model=MODEL_NAME, temperature=TEMPERATURE)

output_parser = StrOutputParser()

# LCEL 파이프 연산자로 체인 구성 (Session 1.1에서 배운 패턴!)
chain = prompt | llm | output_parser

# 체인 실행
result = chain.invoke({"topic": "LangChain 프레임워크"})
print(f"응답: {result}")
# 출력 예시: 응답: LangChain은 LLM 기반 애플리케이션 개발을 위한
#           모듈형 프레임워크로, 다양한 AI 모델과 도구를 손쉽게
#           통합할 수 있게 해줍니다.
```

## 더 깊이 알아보기

### Harrison Chase와 LangChain의 탄생

LangChain의 탄생 배경에는 흥미로운 이야기가 있습니다. 창립자 **Harrison Chase**는 Harvard University에서 통계학과 컴퓨터 과학을 전공한 후, 핀테크 스타트업 Kensho와 AI 보안 회사 Robust Intelligence에서 일했습니다.

2022년, Chase는 text-davinci-002 모델을 처음 접했을 때, 회사 내부의 Notion과 Slack 데이터를 LLM과 대화하듯 활용하고 싶다는 아이디어를 떠올렸습니다. 지금은 **RAG(Retrieval-Augmented Generation)**로 불리는 이 패턴을 구현하면서, 개발자들이 LLM 애플리케이션을 만들 때 비슷한 문제에 반복적으로 부딪힌다는 것을 깨달았죠.

그래서 2022년 10월 16일부터 25일까지, 단 **10일 만에** LangChain의 첫 번째 버전을 개인 GitHub(`hwchase17`)에 올렸습니다. 약 800줄짜리 단일 Python 파일로, 프롬프트 템플릿을 감싸는 간단한 래퍼였죠. 이 사이드 프로젝트가 2023년 1월 회사로 정식 설립되었고, 현재는 $1.25B(약 1조 7000억 원) 가치의 기업으로 성장했습니다.

이 이야기에서 배울 점이 있습니다. 오늘 우리가 설정하는 이 개발 환경도, LangChain이 800줄의 코드에서 시작한 것처럼, **작지만 올바른 기반** 위에 구축하는 것이 중요하다는 거예요.

### `venv`에서 `uv`로 — Python 패키지 관리의 진화

Python의 패키지 관리 역사는 꽤 고통스러운 여정이었습니다. `easy_install` → `pip` → `pipenv` → `poetry`를 거쳐, 2024년에 등장한 **uv**는 Rust로 작성되어 기존 도구 대비 10~100배 빠른 속도를 자랑합니다. Astral(ruff를 만든 회사)이 개발한 `uv`는 `pip`, `virtualenv`, `pip-tools`를 하나로 통합하며, 글로벌 캐시를 사용하여 같은 패키지를 여러 프로젝트에서 공유합니다.

이 과정에서 익숙해지면 좋을 도구이지만, 이 강의에서는 모든 환경에서 동작하는 **표준 `venv` + `pip`** 조합을 기본으로 사용합니다.

## 흔한 오해와 팁

> ⚠️ **흔한 오해**: "conda 환경을 쓰고 있으니 `venv`는 필요 없다?" — 맞는 말이지만, conda 환경에서도 `pip install`로 LangChain을 설치할 때 주의가 필요합니다. conda와 pip가 충돌하면 의존성 지옥에 빠질 수 있어요. conda를 쓴다면 `conda install` 가능한 패키지는 conda로, 나머지만 `pip`으로 설치하세요.

> 💡 **알고 계셨나요?**: GitHub에 실수로 API 키를 커밋하면, 해당 커밋을 삭제해도 Git 히스토리에 남아 있습니다. GitGuardian에 따르면, 2023년 한 해 동안 GitHub에 공개된 비밀 정보(API 키, 비밀번호 등)가 **1280만 건**에 달했습니다. `.gitignore` 설정은 선택이 아니라 필수입니다!

> 🔥 **실무 팁**: `os.getenv("OPENAI_API_KEY")`는 키가 없을 때 `None`을 반환하고, `os.environ["OPENAI_API_KEY"]`는 `KeyError`를 발생시킵니다. 개발 초기에는 `os.environ`을 사용해서 누락된 키를 바로 발견하고, 프로덕션에서는 `os.getenv`에 기본값을 제공하는 패턴을 추천합니다:
> ```python
> # 개발 시 — 키 누락을 즉시 발견
> api_key = os.environ["OPENAI_API_KEY"]
>
> # 프로덕션 — 안전한 폴백
> debug_mode = os.getenv("DEBUG", "false").lower() == "true"
> ```

## 핵심 정리

| 개념 | 설명 |
|------|------|
| **가상환경** | `python -m venv .venv`로 프로젝트별 격리된 Python 환경 생성 |
| **핵심 패키지** | `langchain` + 모델 통합 패키지(`langchain-openai` 등) 별도 설치 필요 |
| **python-dotenv** | `.env` 파일의 환경변수를 `load_dotenv()`로 로드하는 라이브러리 |
| **API 키 보안** | `.env`에 저장, `.gitignore`에 추가, 코드에 절대 하드코딩 금지 |
| **환경변수 자동 감지** | LangChain 통합 패키지는 `OPENAI_API_KEY` 등의 표준 환경변수 이름을 자동 인식 |
| **프로젝트 구조** | `src/` 하위에 `chains/`, `agents/`, `tools/`, `prompts/`, `utils/` 분리 |
| **uv** | Rust 기반 차세대 패키지 매니저 — venv 대비 80배 빠른 환경 생성 |
