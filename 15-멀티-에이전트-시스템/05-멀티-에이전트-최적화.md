# 멀티 에이전트 최적화

> 멀티 에이전트 시스템의 비용, 속도, 품질을 동시에 잡고, 장애에 강하고 테스트 가능한 프로덕션 시스템을 만드는 최적화 전략을 마스터합니다.

## 개요

이 섹션에서는 멀티 에이전트 시스템을 프로덕션 환경에서 효율적으로 운영하기 위한 최적화 기법을 학습합니다. 비용 폭주를 막는 모델 믹스 전략, 사용자 경험을 좌우하는 지연 시간 최소화, 에이전트 품질을 정량적으로 평가하는 방법, 시스템 병목을 찾아 해결하는 실전 기법, 그리고 워커 에이전트 장애에 대비한 **에러 복구 전략**과 멀티 에이전트 시스템을 체계적으로 검증하는 **통합 테스트 패턴**까지 다룹니다.

**선수 지식**: [15.1 멀티 에이전트 아키텍처 패턴]에서 배운 감독자/스웜 패턴, [15.2 감독자 기반 멀티 에이전트]의 `create_supervisor` 구현, [15.4 에이전트 간 통신]의 공유 상태와 Command 기반 라우팅, [14.6 에러 핸들링과 복구]의 RetryPolicy 개념

**학습 목표**:
- 에이전트별 모델 믹스(Model Mix)를 적용하여 비용을 최적화할 수 있다
- 비동기 실행과 병렬 처리로 지연 시간을 최소화할 수 있다
- LangSmith를 활용하여 멀티 에이전트 성능을 정량적으로 측정하고 평가할 수 있다
- 시스템 병목을 식별하고 체계적으로 해결할 수 있다
- 워커 에이전트 실패 시 감독자의 폴백 전략을 구현할 수 있다
- 모의 에이전트와 통합 테스트로 멀티 에이전트 시스템을 체계적으로 검증할 수 있다

## 왜 알아야 할까?

멀티 에이전트 시스템을 만드는 것과 **잘 운영하는 것**은 완전히 다른 문제입니다.

앞서 [15.1]~[15.4]에서 감독자 패턴, 전문화된 에이전트 설계, 에이전트 간 통신까지 멀티 에이전트 시스템의 "설계와 구축"을 배웠습니다. 하지만 프로덕션에 배포하면 새로운 도전이 시작됩니다:

- **비용 문제**: 에이전트 5개가 모두 GPT-4o를 쓰면? 하루 1만 건 처리 시 월 수백만 원이 청구될 수 있습니다
- **속도 문제**: 에이전트가 순차적으로 실행되면 응답 시간이 30초 이상으로 늘어납니다
- **품질 문제**: "잘 동작하는 것 같다"는 느낌이 아닌, 정량적 지표가 필요합니다
- **안정성 문제**: 워커 에이전트 하나가 실패하면 전체 파이프라인이 멈춰버립니다. 부분 실패를 감내하고 복구하는 전략 없이는 프로덕션 운영이 불가능합니다
- **검증 문제**: 에이전트 5개가 협력하는 시스템을 어떻게 테스트할까요? 실제 LLM을 매번 호출하면 비용도, 시간도, 결과의 일관성도 문제입니다

실제로 한 이커머스 기업은 모든 쿼리를 GPT-4로 라우팅하다가 월 $5,000의 비용이 발생했지만, 모델 라우팅을 도입한 후 월 $1,200으로 76%를 절감하면서도 95%의 고객 만족도를 유지했습니다. 이처럼 최적화는 "있으면 좋은 것"이 아니라 프로덕션의 **필수 역량**입니다.

## 핵심 개념

### 개념 1: 모델 믹스(Model Mix) — 비용 최적화의 핵심

> 💡 **비유**: 레스토랑 주방을 생각해보세요. 모든 요리에 미슐랭 스타 셰프를 투입할 필요는 없습니다. 샐러드 준비는 보조 셰프가, 메인 디시 플레이팅은 헤드 셰프가 담당하면 됩니다. 멀티 에이전트의 모델 믹스도 똑같은 원리입니다 — 작업 난이도에 맞는 모델을 배정하는 것이죠.

모델 믹스란 각 에이전트의 역할과 복잡도에 따라 **서로 다른 크기의 모델을 배정**하는 전략입니다. `create_react_agent`의 `model` 파라미터에 각각 다른 모델을 전달하면 됩니다.

```python
from langchain_openai import ChatOpenAI
from langchain_anthropic import ChatAnthropic
from langgraph.prebuilt import create_react_agent
from langgraph_supervisor import create_supervisor

# 역할별 모델 배정 — 비용 대비 성능 최적화
# 간단한 분류/라우팅: 소형 모델 (비용 ↓)
classifier_model = ChatOpenAI(model="gpt-4o-mini", temperature=0)

# 검색/요약: 중형 모델 (균형)
researcher_model = ChatAnthropic(model="claude-sonnet-4-20250514")

# 복잡한 추론/최종 답변: 대형 모델 (품질 ↑)
reasoner_model = ChatOpenAI(model="gpt-4o", temperature=0.2)

# 각 에이전트에 다른 모델 적용
classifier_agent = create_react_agent(
    model=classifier_model,  # 소형 모델로 빠르게 분류
    tools=[classify_query_tool],
    name="classifier",
    prompt="사용자 쿼리를 분류합니다. 간결하게 응답하세요.",
)

researcher_agent = create_react_agent(
    model=researcher_model,  # 중형 모델로 검색·요약
    tools=[search_tool, summarize_tool],
    name="researcher",
    prompt="관련 정보를 검색하고 핵심을 요약합니다.",
)

writer_agent = create_react_agent(
    model=reasoner_model,  # 대형 모델로 고품질 출력
    tools=[format_tool],
    name="writer",
    prompt="검색 결과를 바탕으로 완성도 높은 답변을 작성합니다.",
)

# 감독자도 소형 모델로 충분 — 라우팅만 하니까
supervisor = create_supervisor(
    agents=[classifier_agent, researcher_agent, writer_agent],
    model=classifier_model,  # 감독자는 라우팅만 → 소형 모델
    prompt="적절한 워커에게 작업을 위임하세요.",
)
```

**비용 절감 효과 시뮬레이션:**

| 전략 | 모델 구성 | 1만 건/일 예상 비용 |
|------|-----------|-------------------|
| 단일 모델 | 전체 GPT-4o | ~$5,000/월 |
| 모델 믹스 | 감독자·분류: mini, 검색: sonnet, 작성: GPT-4o | ~$1,500/월 |
| 적극적 믹스 | 감독자·분류·검색: mini, 작성만 GPT-4o | ~$900/월 |

### 개념 2: 지연 시간 최소화 — 비동기와 병렬 처리

> 💡 **비유**: 카페에서 커피를 주문했는데 바리스타가 원두 갈기 → 물 끓이기 → 추출을 **하나씩 순서대로** 한다면 느리겠죠? 원두를 가는 동안 물을 끓이면 훨씬 빠릅니다. 멀티 에이전트에서도 독립적인 에이전트를 **동시에 실행**하면 전체 응답 시간이 확 줄어듭니다.

LangGraph에서 지연 시간을 줄이는 세 가지 핵심 기법이 있습니다.

**기법 1: 비동기(async) 실행**

```python
import asyncio
from langgraph.prebuilt import create_react_agent

# 비동기 노드 함수 정의
async def research_node(state):
    """비동기로 실행되어 I/O 대기 시간을 최소화"""
    # await를 사용하면 API 응답을 기다리는 동안
    # 다른 코루틴이 실행될 수 있음
    result = await researcher_agent.ainvoke(state)
    return result

# 그래프 비동기 실행
async def run_pipeline(query: str):
    config = {"configurable": {"thread_id": "opt-1"}}
    # astream으로 점진적 결과 전달 — 체감 지연 시간 감소
    async for chunk in supervisor.compiled.astream(
        {"messages": [{"role": "user", "content": query}]},
        config=config,
    ):
        # 실시간으로 중간 결과를 사용자에게 전달
        for node_name, node_output in chunk.items():
            print(f"[{node_name}] 처리 중...")

# 실행
# asyncio.run(run_pipeline("AI 트렌드 보고서를 작성해주세요"))
```

**기법 2: 병렬 에이전트 실행**

LangGraph에서 하나의 노드에서 여러 노드로 엣지를 연결하면 자동으로 병렬 실행됩니다:

```python
from langgraph.graph import StateGraph, START, END
from typing import TypedDict, Annotated
from langchain_core.messages import AnyMessage
from langgraph.graph.message import add_messages

class ResearchState(TypedDict):
    messages: Annotated[list[AnyMessage], add_messages]
    web_results: str
    db_results: str
    final_report: str

def web_researcher(state: ResearchState) -> dict:
    """웹 검색 에이전트 — 독립적으로 실행 가능"""
    # 웹 검색 수행
    return {"web_results": "웹 검색 결과..."}

def db_researcher(state: ResearchState) -> dict:
    """DB 검색 에이전트 — 웹 검색과 동시 실행"""
    # 데이터베이스 조회 수행
    return {"db_results": "DB 조회 결과..."}

def report_writer(state: ResearchState) -> dict:
    """두 검색 결과를 합쳐 보고서 작성"""
    return {"final_report": f"종합: {state['web_results']} + {state['db_results']}"}

# 병렬 실행 그래프 구성
graph = StateGraph(ResearchState)
graph.add_node("web_researcher", web_researcher)
graph.add_node("db_researcher", db_researcher)
graph.add_node("report_writer", report_writer)

# START에서 두 연구원으로 동시에 분기 → 자동 병렬 실행!
graph.add_edge(START, "web_researcher")
graph.add_edge(START, "db_researcher")

# 두 노드가 모두 완료되면 작성자로
graph.add_edge("web_researcher", "report_writer")
graph.add_edge("db_researcher", "report_writer")
graph.add_edge("report_writer", END)

parallel_graph = graph.compile()
```

이 패턴에서 `web_researcher`와 `db_researcher`는 동시에 실행됩니다. 각각 5초가 걸린다면 순차 실행은 10초이지만, 병렬 실행은 약 5초면 끝납니다.

**기법 3: 동시 실행 수 제한**

```python
# max_concurrency로 동시 실행 노드 수 제한
# API 레이트 리밋이 있을 때 유용
config = {
    "max_concurrency": 3,  # 최대 3개 노드 동시 실행
    "configurable": {"thread_id": "opt-2"},
}
result = parallel_graph.invoke(
    {"messages": [{"role": "user", "content": "분석 요청"}]},
    config=config,
)
```

### 개념 3: LangSmith로 성능 측정과 비용 추적

> 💡 **비유**: 자동차를 튜닝하려면 먼저 계기판을 봐야 합니다. RPM, 연비, 엔진 온도를 모르는데 어떻게 최적화하겠어요? LangSmith는 멀티 에이전트 시스템의 **계기판**입니다 — 각 에이전트가 얼마나 시간을 쓰는지, 토큰을 얼마나 소비하는지, 어디서 병목이 생기는지 보여줍니다.

LangSmith는 멀티 에이전트 시스템의 트레이싱, 비용 추적, 평가를 위한 관찰 가능성(Observability) 플랫폼입니다. 특히 2025년부터 **자동 비용 추적**과 **멀티턴 평가** 기능이 강화되었습니다.

```python
import os

# LangSmith 트레이싱 활성화
os.environ["LANGSMITH_TRACING"] = "true"
os.environ["LANGSMITH_API_KEY"] = "your-api-key"
os.environ["LANGSMITH_PROJECT"] = "multi-agent-optimization"

from langsmith import traceable
from langsmith.run_helpers import get_current_run_tree
import time

@traceable(
    name="multi_agent_pipeline",
    tags=["production", "v2"],  # 태그로 버전/환경 구분
    metadata={"team": "ai-platform"},  # 메타데이터로 비용 귀속
)
def run_multi_agent(query: str) -> dict:
    """트레이싱이 적용된 멀티 에이전트 파이프라인"""
    start = time.time()

    result = supervisor.compiled.invoke(
        {"messages": [{"role": "user", "content": query}]},
        config={
            "configurable": {"thread_id": "traced-1"},
            # 태그로 세분화된 비용 추적 가능
            "tags": ["customer-support"],
            "metadata": {
                "user_id": "user_123",
                "feature": "report_generation",
                # LangSmith가 자동으로 모델별 비용 계산
                "ls_provider": "openai",
                "ls_model_name": "gpt-4o",
            },
        },
    )

    elapsed = time.time() - start
    print(f"총 소요 시간: {elapsed:.2f}초")
    return result
```

**커스텀 비용 추적 — 모델별, 에이전트별 비용 분석:**

```python
from langsmith import traceable

@traceable(name="cost_tracked_agent")
def tracked_agent_call(agent, query: str, model_name: str) -> dict:
    """에이전트별 비용을 추적하는 래퍼 함수"""
    result = agent.invoke(
        {"messages": [{"role": "user", "content": query}]},
    )

    # 토큰 사용량 추출 (모델 응답에서)
    messages = result.get("messages", [])
    total_tokens = 0
    for msg in messages:
        if hasattr(msg, "usage_metadata") and msg.usage_metadata:
            total_tokens += msg.usage_metadata.get("total_tokens", 0)

    print(f"[{model_name}] 토큰 사용: {total_tokens}")
    return result
```

### 개념 4: 병목 식별과 체계적 해결

> 💡 **비유**: 고속도로에서 차가 막히면 어디가 병목인지 찾아야 합니다. 도로 전체를 넓히는 것보다, 좁은 구간 하나를 넓히는 게 훨씬 효과적이죠. 멀티 에이전트도 마찬가지입니다 — 가장 느린 에이전트 하나를 최적화하면 전체 시스템이 빨라집니다.

멀티 에이전트 시스템에서 흔히 발생하는 병목 유형과 해결 전략:

**병목 유형 1: 토큰 과다 사용**

메시지 히스토리가 누적되면서 각 에이전트 호출마다 토큰이 기하급수적으로 증가하는 문제입니다. [15.4 에이전트 간 통신]에서 배운 `output_mode`와 요약 핸드오프를 활용합니다.

```python
from langgraph_supervisor import create_supervisor

# output_mode="last_message"로 메시지 히스토리 축소
supervisor = create_supervisor(
    agents=[researcher_agent, writer_agent],
    model=ChatOpenAI(model="gpt-4o-mini"),
    prompt="작업을 위임하세요.",
    output_mode="last_message",  # 마지막 메시지만 전달 → 토큰 절약
)
```

**병목 유형 2: 불필요한 에이전트 호출**

감독자가 이미 답을 알고 있는데도 워커에게 위임하는 경우입니다. 감독자 프롬프트에 직접 응답 조건을 추가합니다:

```python
OPTIMIZED_SUPERVISOR_PROMPT = """당신은 멀티 에이전트 시스템의 감독자입니다.

## 직접 응답 조건 (워커 호출 불필요):
- 인사말, 간단한 질문 ("안녕하세요", "감사합니다")
- 이전 대화에서 이미 답변한 내용의 반복 질문
- "네/아니오"로 답할 수 있는 단순 확인

## 워커 위임 조건:
- 검색이 필요한 질문 → researcher
- 코드 작성이 필요한 요청 → coder
- 복잡한 분석/보고서 → researcher → writer 순서

불필요한 위임은 비용과 시간을 낭비합니다. 직접 답할 수 있으면 직접 답하세요.
"""
```

**병목 유형 3: 에이전트 루프**

에이전트가 서로에게 작업을 반복적으로 넘기는 무한 루프 상황:

```python
from langgraph.prebuilt import create_react_agent

# recursion_limit으로 무한 루프 방지
config = {
    "recursion_limit": 25,  # 최대 25단계까지만 허용
    "configurable": {"thread_id": "loop-safe"},
}

try:
    result = supervisor.compiled.invoke(
        {"messages": [{"role": "user", "content": query}]},
        config=config,
    )
except Exception as e:
    if "recursion" in str(e).lower():
```

### 개념 5: 멀티 에이전트 에러 복구

> 💡 **비유**: 축구팀에서 공격수가 부상당하면 경기가 끝나는 게 아닙니다. 감독이 교체 선수를 투입하거나, 전술을 바꿔서 경기를 계속 이어가죠. 멀티 에이전트 시스템에서도 워커 에이전트 하나가 실패했다고 전체 파이프라인이 멈춰서는 안 됩니다 — 감독자가 **대안을 마련**하고 시스템을 계속 운영해야 합니다.

단일 에이전트 시스템에서는 에러 핸들링이 비교적 단순합니다. 하지만 멀티 에이전트에서는 **에러 전파**가 문제입니다. 워커 A가 실패하면 그 결과에 의존하는 워커 B, C까지 연쇄적으로 영향을 받거든요. 이를 해결하기 위한 세 가지 전략을 살펴보겠습니다.

**전략 1: 감독자 폴백 — 실패한 워커를 우회하기**

[14.6 에러 핸들링과 복구]에서 배운 `RetryPolicy`를 멀티 에이전트에 적용하되, 재시도 실패 시 감독자가 대체 경로를 선택합니다:

```python
from langgraph.graph import StateGraph, START, END
from langgraph.types import RetryPolicy
from langchain_openai import ChatOpenAI
from langgraph.prebuilt import create_react_agent
from langgraph_supervisor import create_supervisor
from langchain_core.tools import tool
from typing import TypedDict, Annotated
from langchain_core.messages import AnyMessage, AIMessage
from langgraph.graph.message import add_messages

class AgentState(TypedDict):
    messages: Annotated[list[AnyMessage], add_messages]
    failed_agents: list[str]  # 실패한 에이전트 추적

def safe_worker_node(worker_agent, agent_name: str, fallback_message: str):
    """워커 에이전트를 에러 안전하게 감싸는 래퍼"""
    def node_fn(state: AgentState) -> dict:
        try:
            result = worker_agent.invoke(state)
            return result
        except Exception as e:
            # 실패 기록 + 폴백 메시지 반환
            failed = state.get("failed_agents", [])
            failed.append(agent_name)
            return {
                "messages": [AIMessage(
                    content=f"[{agent_name} 장애] {fallback_message}",
                    name=agent_name,
                )],
                "failed_agents": failed,
            }
    return node_fn

# RetryPolicy 적용 — Ch14.6에서 배운 재시도 정책
# 일시적 네트워크 오류나 API 레이트 리밋은 재시도로 해결
retry_policy = RetryPolicy(
    max_attempts=3,              # 최대 3회 재시도
    initial_interval=1.0,        # 첫 재시도까지 1초 대기
    backoff_factor=2.0,          # 대기 시간 2배씩 증가
    retry_on=ConnectionError,    # 네트워크 오류에만 재시도
)

# 그래프에 RetryPolicy 적용
graph = StateGraph(AgentState)
graph.add_node(
    "researcher",
    safe_worker_node(researcher_agent, "researcher", "검색 서비스 일시 장애"),
    retry=retry_policy,  # 노드 단위 재시도 정책
)
graph.add_node(
    "writer",
    safe_worker_node(writer_agent, "writer", "작성 서비스 일시 장애"),
    retry=retry_policy,
)
```

**전략 2: 부분 실패 허용 — 가용한 결과로 최선의 응답 생성**

### 개념 6: 멀티 에이전트 통합 테스트

> 💡 **비유**: 오케스트라 연습을 생각해보세요. 각 악기 연주자가 개별적으로 완벽해도, **합주**했을 때 타이밍이 안 맞으면 엉망이 됩니다. 멀티 에이전트 테스트도 마찬가지입니다 — 개별 에이전트 테스트만으로는 부족하고, 에이전트들이 **함께 작동**할 때 감독자가 올바르게 라우팅하는지, 결과가 제대로 합쳐지는지까지 검증해야 합니다.

[13.6]에서 단일 LangGraph 그래프의 테스트를 다뤘는데요, 멀티 에이전트 시스템은 에이전트 간 상호작용이 추가되므로 테스트 전략이 달라져야 합니다. 핵심은 **실제 LLM 호출 없이도** 시스템 동작을 검증할 수 있는 모의(Mock) 패턴입니다.

**패턴 1: 모의 워커 에이전트 — LLM 호출 없이 테스트하기**

```python
def create_deterministic_supervisor(agents, routing_map: dict):
    """테스트용 결정적 감독자 — 쿼리 키워드로 라우팅 결정
    
    Args:
        agents: 에이전트 노드 리스트
        routing_map: {"키워드": "에이전트이름"} 매핑
    """
    def supervisor_node(state):
        query = state["messages"][-1].content.lower()
        for keyword, agent_name in routing_map.items():
            if keyword in query:
                return {"next": agent_name}
        return {"next": "FINISH"}
    
    return supervisor_node
```

**패턴 2: 감독자 라우팅 검증 — 올바른 워커에게 위임하는지 확인**

감독자가 쿼리에 따라 정확한 워커를 선택하는지 테스트합니다. 이 테스트는 실제 LLM을 호출하지만, 워커는 모의 에이전트로 대체하여 비용을 최소화합니다:

```python
def test_supervisor_routing():
    """감독자가 쿼리 유형에 따라 올바른 워커에게 라우팅하는지 검증"""

    call_tracker = {"faq_agent": 0, "tech_agent": 0}

            "messages": [AIMessage(content="FAQ 응답", name="faq_agent")]
        }

        call_tracker["tech_agent"] += 1
        return {
            "messages": [AIMessage(content="기술 지원 응답", name="tech_agent")]
        }

        agents=[tracked_faq_agent, tracked_tech_agent],
        model=ChatOpenAI(model="gpt-4o-mini", temperature=0),

    call_tracker = {"faq_agent": 0, "tech_agent": 0}  # 리셋
```

```python
- 인사말, 감사 표현
- "네/아니오"로 답할 수 있는 단순 확인

## 라우팅 규칙:
- FAQ, 주문 조회, 일반 문의 → faq_agent
- 기술적 오류, 버그 리포트, 시스템 장애 → tech_agent

## 장애 복구:
- 워커가 "[장애]" 또는 에러 메시지를 반환하면:
  1. 다른 워커로 재위임을 시도하세요
  2. 대안이 없으면 가용한 정보로 직접 답변하되,
     "일부 서비스가 일시적으로 제한됩니다"라고 안내하세요
- 핵심 워커 2개 모두 실패 시, 에스컬레이션을 안내하세요

불필요한 에이전트 호출을 최소화하세요."""

supervisor = create_supervisor(
    agents=[faq_agent, tech_agent],
    model=router_model,  # 감독자도 소형 모델로 비용 절감
    prompt=SUPERVISOR_PROMPT,
    output_mode="last_message",  # 토큰 절약: 마지막 메시지만 전달
)

app = supervisor.compile()

# ============================================================
# 5단계: 성능 측정 래퍼
# ============================================================

class PerformanceTracker:
    """멀티 에이전트 성능을 측정하는 간단한 트래커"""

    def __init__(self):
        self.metrics = []

    def track(self, query: str, result: dict, elapsed: float):
        """단일 실행의 성능 지표를 기록"""
        # 메시지에서 토큰 사용량 추출
        messages = result.get("messages", [])
        total_tokens = 0
        agent_calls = []

        for msg in messages:
            # 어떤 에이전트가 호출되었는지 추적
            if hasattr(msg, "name") and msg.name:
                agent_calls.append(msg.name)
            # 토큰 사용량 집계
            if hasattr(msg, "usage_metadata") and msg.usage_metadata:
                total_tokens += msg.usage_metadata.get("total_tokens", 0)

        metric = {
            "query": query[:50],
            "elapsed_sec": round(elapsed, 2),
            "total_tokens": total_tokens,
            "agent_calls": agent_calls,
            "num_messages": len(messages),
        }
        self.metrics.append(metric)
        return metric

    def summary(self) -> dict:
        """전체 실행의 요약 통계"""
        if not self.metrics:
            return {}
        avg_time = sum(m["elapsed_sec"] for m in self.metrics) / len(self.metrics)
        avg_tokens = sum(m["total_tokens"] for m in self.metrics) / len(self.metrics)
        total_calls = sum(len(m["agent_calls"]) for m in self.metrics)
        return {
            "total_queries": len(self.metrics),
            "avg_latency_sec": round(avg_time, 2),
            "avg_tokens": round(avg_tokens),
            "total_agent_calls": total_calls,
            "avg_calls_per_query": round(total_calls / len(self.metrics), 1),
        }


# ============================================================
# 6단계: 실행 및 성능 비교
# ============================================================

tracker = PerformanceTracker()

# 테스트 쿼리 세트
test_queries = [
    "환불 정책이 어떻게 되나요?",          # FAQ → 소형 모델
    "주문번호 ORD-2024-001 배송 현황",     # FAQ → 소형 모델
    "앱이 자꾸 크래시가 납니다",            # 기술 → 대형 모델
    "안녕하세요",                          # 직접 응답 → 모델 호출 최소
]

print("=" * 60)
print("멀티 에이전트 최적화 성능 테스트")
print("=" * 60)

for query in test_queries:
    start = time.time()
    result = app.invoke(
        {"messages": [{"role": "user", "content": query}]},
        config={
            "recursion_limit": 25,  # 루프 방지
            "configurable": {"thread_id": f"test-{hash(query)}"},
        },
    )
    elapsed = time.time() - start

    metric = tracker.track(query, result, elapsed)

    # 마지막 AI 메시지 출력
    last_ai_msg = None
    for msg in reversed(result["messages"]):
        if hasattr(msg, "content") and msg.type == "ai" and msg.content:
            last_ai_msg = msg.content
            break

    print(f"\n질문: {query}")
    print(f"  응답: {last_ai_msg[:80] if last_ai_msg else 'N/A'}...")
    print(f"  소요: {metric['elapsed_sec']}초 | 토큰: {metric['total_tokens']}")
    print(f"  호출된 에이전트: {metric['agent_calls']}")

# 전체 요약
print("\n" + "=" * 60)
print("성능 요약")
print("=" * 60)
summary = tracker.summary()
for key, value in summary.items():
    print(f"  {key}: {value}")
```

> 🔥 **실무 팁**: 위 `PerformanceTracker`는 학습용 간이 구현입니다. 프로덕션에서는 LangSmith의 자동 트레이싱을 사용하면 에이전트별 비용, 지연 시간, 토큰 사용량이 대시보드에서 자동으로 시각화됩니다. `LANGSMITH_TRACING=true` 환경변수만 설정하면 됩니다.

## 더 깊이 알아보기

### "느린 것은 죽은 것이다" — 구글의 100ms 법칙

2006년 구글의 마리사 메이어(Marissa Mayer)는 유명한 실험 결과를 발표했습니다. 검색 결과 표시를 30개에서 10개로 줄였더니 페이지 로딩이 0.5초 빨라졌고, 트래픽이 25% 증가했다는 것이었습니다. 반대로 0.4초 느려지면 사용자 참여도가 눈에 띄게 떨어졌죠.

이 원칙은 AI 에이전트 시대에도 그대로 적용됩니다. 멀티 에이전트 시스템에서 에이전트가 순차적으로 실행되면 각 에이전트의 지연이 누적되어 사용자 체감 시간이 급격히 늘어납니다. 이것이 `astream`을 통한 스트리밍과 병렬 실행이 중요한 이유입니다.

### 모델 캐스케이드(Model Cascade)의 기원

모델 믹스의 아이디어는 사실 머신러닝의 오래된 기법인 **캐스케이드 분류기(Cascade Classifier)**에서 왔습니다. 2001년 Paul Viola와 Michael Jones가 얼굴 검출을 위해 발표한 이 기법은, 간단한 분류기로 쉬운 케이스를 빠르게 걸러내고 복잡한 분류기는 어려운 케이스에만 사용하는 방식이었습니다. 20년이 지난 지금, LLM 세계에서도 같은 원리가 "소형 모델 → 대형 모델" 라우팅으로 부활한 셈이죠.

### LangSmith의 탄생과 관찰 가능성의 중요성

LangChain 창시자 Harrison Chase는 LangChain을 만들면서 가장 많이 받은 질문이 "디버깅을 어떻게 하느냐"였다고 합니다. 프롬프트가 들어가고 답이 나오는 과정이 블랙박스였기 때문이죠. 이 문제를 해결하기 위해 2023년 LangSmith가 탄생했고, 이제는 멀티 에이전트의 복잡한 실행 흐름까지 중첩 스팬(Nested Span)으로 시각화할 수 있게 되었습니다. 2025년에는 **멀티턴 평가(Multi-turn Evals)**와 **인사이트 에이전트(Insights Agent)** 기능이 추가되어, 에이전트 대화의 의도·결과·궤적을 자동으로 분석할 수 있게 되었습니다.

### "카오스 엔지니어링"에서 배우는 에러 복구

넷플릭스의 **카오스 몽키(Chaos Monkey)**를 아시나요? 2011년 넷플릭스는 프로덕션 서버를 **의도적으로 랜덤하게 종료**시키는 도구를 만들었습니다. 미친 짓 같지만, 이를 통해 시스템의 장애 복구 능력을 극적으로 향상시켰죠. 멀티 에이전트 시스템에도 같은 원리를 적용할 수 있습니다 — 테스트 환경에서 워커 에이전트를 의도적으로 실패시켜보면, 감독자의 폴백 로직이 제대로 작동하는지 사전에 검증할 수 있습니다. 이것이 바로 우리가 모의 에이전트와 에러 주입 테스트를 중요하게 다루는 이유입니다.

## 흔한 오해와 팁

> ⚠️ **흔한 오해**: "대형 모델을 쓰면 항상 더 좋은 결과가 나온다"
> 그렇지 않습니다. 라우팅, FAQ 검색, 단순 분류 같은 작업에서는 소형 모델이 대형 모델과 동등하거나 오히려 더 빠르고 정확합니다. 특히 GPT-4o-mini는 간단한 작업에서 GPT-4o와 거의 같은 성능을 보이면서 비용은 1/30 수준입니다. 모든 에이전트에 대형 모델을 배정하는 것은 "모든 못을 황금 망치로 박는" 것과 같습니다.

> ⚠️ **흔한 오해**: "멀티 에이전트 시스템은 단위 테스트로 충분하다"
> 개별 에이전트가 완벽하게 동작해도 시스템 레벨에서는 문제가 생길 수 있습니다. 감독자가 엉뚱한 워커에게 라우팅하거나, 워커 간 상태 전달에서 정보가 유실되거나, 특정 입력 조합에서만 무한 루프가 발생하는 경우가 흔합니다. **라우팅 테스트**와 **시나리오 기반 통합 테스트**를 반드시 병행하세요.

> 💡 **알고 계셨나요?**: LangSmith의 비용 추적은 OpenAI, Anthropic, Google 등 주요 모델 공급자의 가격 정보를 **내장하고 있어서** 별도 설정 없이도 자동으로 토큰 비용을 계산해줍니다. 커스텀 모델이나 자체 호스팅 모델을 사용하는 경우에만 `ls_provider`와 `ls_model_name`을 메타데이터에 수동 설정하면 됩니다.

> 🔥 **실무 팁**: 멀티 에이전트 시스템에서 가장 흔한 성능 병목은 **메시지 히스토리 폭발**입니다. 감독자가 워커에게 전체 대화 이력을 전달하면, 에이전트 수 × 대화 턴 수만큼 토큰이 기하급수적으로 증가합니다. `output_mode="last_message"`를 기본으로 설정하고, [15.4]에서 배운 요약 핸드오프로 맥락을 압축하세요. 이것만으로도 토큰 사용량이 50~70% 줄어드는 경우가 많습니다.

> 🔥 **실무 팁**: `recursion_limit`을 반드시 설정하세요. 감독자와 워커 사이에 무한 루프가 발생하면 토큰 비용이 순식간에 폭주합니다. 프로덕션에서는 `recursion_limit=25` 정도가 적당하며, 루프가 감지되면 단일 에이전트 폴백을 구현하는 것이 좋습니다.

> 🔥 **실무 팁**: 워커 에이전트의 에러 핸들링은 **"빠르게 실패하고 알리기(Fail Fast & Report)"** 원칙을 따르세요. 워커가 3회 재시도 후에도 실패하면 오래 매달리지 말고, 즉시 감독자에게 실패를 보고하여 폴백 경로를 타게 하는 것이 사용자 경험에 훨씬 좋습니다. [14.6]의 `RetryPolicy`를 노드 단위로 적용하면 재시도 로직을 깔끔하게 분리할 수 있습니다.

## 핵심 정리

| 개념 | 설명 |
|------|------|
| 모델 믹스(Model Mix) | 에이전트 역할별로 다른 크기의 모델을 배정하여 비용 대비 성능을 최적화하는 전략 |
| 비동기 실행(Async) | `ainvoke`/`astream`으로 I/O 대기 시간을 최소화하고, 스트리밍으로 체감 지연을 줄이는 기법 |
| 병렬 처리(Parallel) | 독립적인 에이전트를 동시에 실행하여 전체 소요 시간을 단축하는 패턴 |
| LangSmith 트레이싱 | 에이전트별 토큰 사용량, 비용, 지연 시간을 자동으로 추적하고 시각화하는 관찰 가능성 도구 |
| 메시지 히스토리 관리 | `output_mode`, 요약 핸드오프로 토큰 폭발을 방지하는 기법 |
| `recursion_limit` | 에이전트 루프를 감지하고 방지하는 안전장치로, 비용 폭주를 막는 필수 설정 |
| `max_concurrency` | 동시 실행 노드 수를 제한하여 API 레이트 리밋을 준수하는 설정 |
| 성능 병목 식별 | 토큰 과다, 불필요한 호출, 에이전트 루프 등 세 가지 주요 병목 유형과 해결 전략 |
| 감독자 폴백(Fallback) | 워커 실패 시 감독자가 대체 워커로 재위임하거나 직접 응답하는 에러 복구 전략 |
| RetryPolicy 적용 | [14.6]의 재시도 정책을 노드 단위로 적용하여 일시적 장애를 자동 복구하는 기법 |
| 부분 실패 허용 | 병렬 에이전트 중 일부가 실패해도 가용한 결과로 최선의 응답을 생성하는 패턴 |
| 모의 워커(Mock Agent) | 실제 LLM 없이 고정 응답을 반환하는 테스트용 에이전트로, 비용 없이 시스템 검증 가능 |
| 라우팅 검증 테스트 | 감독자가 쿼리 유형에 따라 올바른 워커를 선택하는지 검증하는 테스트 패턴 |
| 엔드투엔드 시나리오 테스트 | 모의 에이전트로 전체 워크플로우를 재현하여 라우팅·상태 전달·결과 합성을 통합 검증하는 테스트 |
