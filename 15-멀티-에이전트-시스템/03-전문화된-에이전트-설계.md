# 전문화된 에이전트 설계

> 역할별로 최적화된 에이전트를 설계하고, 도구와 프롬프트를 전략적으로 할당하여 멀티 에이전트 시스템의 성능을 극대화합니다.

## 개요

이 섹션에서는 멀티 에이전트 시스템의 핵심인 **전문화된 에이전트**를 설계하는 방법을 배웁니다. [15.1: 멀티 에이전트 아키텍처 패턴](ch15/session1.md)에서 세 가지 아키텍처를, [15.2: 감독자 기반 멀티 에이전트](ch15/session2.md)에서 감독자 패턴의 구현을 익혔다면, 이제는 각 워커 에이전트를 **어떻게 설계해야 최고의 성능을 내는지** 깊이 파고들 차례입니다.

**선수 지식**: 감독자 패턴의 기본 구조(`create_supervisor`, `create_react_agent`), 핸드오프 도구의 동작 원리, LangGraph의 상태 관리 기초

**학습 목표**:
- 역할 기반 에이전트(연구, 코딩, 분석)를 체계적으로 설계할 수 있다
- 에이전트별로 적절한 도구를 선별하여 할당할 수 있다
- 에이전트의 시스템 프롬프트를 최적화하여 정확도를 높일 수 있다
- 능력 기반 라우팅으로 작업을 올바른 에이전트에 전달할 수 있다

## 왜 알아야 할까?

여러분이 회사를 운영한다고 상상해 보세요. 모든 직원이 "아무 일이나 다 하는 제너럴리스트"라면 어떻게 될까요? 마케팅도, 회계도, 개발도 한 사람이 하면 결국 어느 것 하나 제대로 못 하게 됩니다. 실제 조직이 효율적으로 돌아가는 이유는 **전문 부서**가 있기 때문이죠.

멀티 에이전트 시스템도 마찬가지입니다. LangChain 공식 블로그에서도 이 문제를 명확히 지적하고 있는데요:

- 단일 에이전트에 **도구가 너무 많으면** 어떤 도구를 써야 할지 판단이 흐려집니다
- 작업마다 필요한 **전문 지식과 컨텍스트**가 다릅니다
- 에이전트별로 **다른 LLM이나 파라미터**가 최적일 수 있습니다

전문화된 에이전트 설계는 단순히 "역할을 나누는 것"이 아닙니다. **도구 할당**, **프롬프트 엔지니어링**, **라우팅 전략**까지 종합적으로 고려해야 합니다. 이것이 프로덕션 수준의 멀티 에이전트 시스템과 토이 프로젝트를 가르는 핵심 차이거든요.

## 핵심 개념

### 개념 1: 역할 기반 에이전트 설계 원칙

> 💡 **비유**: 병원의 진료 시스템을 떠올려 보세요. 내과, 외과, 안과 의사가 각각 전문 분야가 있고, 자신의 전문 장비(도구)를 갖추고 있습니다. 환자(작업)가 오면 접수처(감독자)에서 증상을 파악하고 적절한 과(에이전트)로 보내죠. 모든 의사가 MRI도 찍고 수술도 하는 게 아니라, 각자의 영역에서 최고의 실력을 발휘합니다.

역할 기반 에이전트 설계의 핵심 원칙은 **단일 책임 원칙(Single Responsibility Principle)**입니다. 소프트웨어 설계에서 널리 알려진 이 원칙이 에이전트 설계에도 그대로 적용됩니다.

**좋은 역할 분리의 기준**:

| 기준 | 좋은 설계 | 나쁜 설계 |
|------|-----------|-----------|
| 책임 범위 | 명확하고 좁음 | 모호하고 넓음 |
| 도구 수 | 2~5개 | 10개 이상 |
| 프롬프트 길이 | 핵심 지시만 | 모든 경우의 수 나열 |
| 역할 겹침 | 최소화 | 여러 에이전트가 동일 작업 |
| 이름 | 역할을 직관적으로 반영 | 일반적인 이름 |

LangChain 공식 가이드에서 권장하는 에이전트 수는 **2~4개**입니다. 너무 많으면 감독자의 라우팅 부담이 커지고, 너무 적으면 전문화의 이점이 사라지거든요.

```python
from langchain_openai import ChatOpenAI
from langgraph.prebuilt import create_react_agent

# ❌ 나쁜 예: 하나의 에이전트가 모든 것을 담당
# all_tools = [search, calculator, code_executor, db_query, email_sender, ...]
# generic_agent = create_react_agent(model, tools=all_tools)

# ✅ 좋은 예: 역할별로 에이전트를 분리
model = ChatOpenAI(model="gpt-4o", temperature=0)

# 연구 전문 에이전트
research_agent = create_react_agent(
    model,
    tools=[web_search, arxiv_search, wikipedia_lookup],
    prompt="당신은 리서치 전문가입니다. 웹 검색과 학술 자료 조사를 담당합니다."
)

# 코딩 전문 에이전트
coding_agent = create_react_agent(
    model,
    tools=[code_executor, file_reader, linter],
    prompt="당신은 Python 시니어 개발자입니다. 코드 작성과 실행을 담당합니다."
)

# 데이터 분석 전문 에이전트
analysis_agent = create_react_agent(
    model,
    tools=[calculator, chart_generator, csv_reader],
    prompt="당신은 데이터 분석가입니다. 수치 분석과 시각화를 담당합니다."
)
```

### 개념 2: 에이전트별 도구 할당 전략

> 💡 **비유**: 요리사에게 도구를 줄 때를 생각해 보세요. 초밥 장인에게는 회칼과 밥틀을, 파티시에에게는 오븐과 거품기를 줍니다. 초밥 장인에게 오븐을 주면 쓸 데가 없고, 파티시에에게 회칼을 주면 혼란만 생기겠죠? 에이전트에게 도구를 할당하는 것도 같은 논리입니다.

도구 할당은 전문화된 에이전트 설계에서 가장 실질적인 부분입니다. 핵심은 **필요한 도구만, 정확히** 주는 것이에요.

**도구 할당의 3가지 원칙**:

1. **최소 권한 원칙**: 에이전트가 맡은 역할에 꼭 필요한 도구만 제공
2. **도구 설명 최적화**: `@tool` 데코레이터의 docstring이 에이전트의 도구 선택을 좌우
3. **겹침 방지**: 동일한 도구를 여러 에이전트에 할당하지 않음 (혼란 방지)

```python
from langchain_core.tools import tool
from pydantic import BaseModel, Field

# ──────────────────────────────────────────
# 연구 에이전트 전용 도구
# ──────────────────────────────────────────
@tool
def web_search(query: str) -> str:
    """웹에서 최신 정보를 검색합니다. 뉴스, 블로그, 공식 문서 등을 찾을 때 사용합니다."""
    # 실제 구현에서는 Tavily, Google 등의 검색 API 사용
    return f"'{query}'에 대한 검색 결과..."

@tool
def arxiv_search(query: str) -> str:
    """학술 논문 데이터베이스(arXiv)에서 관련 연구를 검색합니다."""
    return f"'{query}' 관련 논문 목록..."

# ──────────────────────────────────────────
# 코딩 에이전트 전용 도구
# ──────────────────────────────────────────
class CodeInput(BaseModel):
    """코드 실행 입력 스키마"""
    code: str = Field(description="실행할 Python 코드")
    timeout: int = Field(default=30, description="실행 제한 시간(초)")

@tool(args_schema=CodeInput)
def execute_python(code: str, timeout: int = 30) -> str:
    """Python 코드를 안전한 샌드박스 환경에서 실행합니다.
    데이터 처리, 계산, 알고리즘 테스트 등에 사용합니다."""
    # 실제 구현에서는 샌드박스 환경 사용
    return f"코드 실행 결과: ..."

@tool
def read_file(file_path: str) -> str:
    """지정된 경로의 파일 내용을 읽어옵니다."""
    return f"'{file_path}' 파일 내용..."

# ──────────────────────────────────────────
# 분석 에이전트 전용 도구
# ──────────────────────────────────────────
@tool
def calculate(expression: str) -> str:
    """수학 표현식을 계산합니다. 사칙연산, 통계 함수 등을 지원합니다."""
    try:
        # 주의: 실제 구현에서는 안전한 수식 파서 사용
        result = eval(expression)  # 교육용 예시
        return str(result)
    except Exception as e:
        return f"계산 오류: {e}"

@tool
def generate_chart(data: str, chart_type: str = "bar") -> str:
    """데이터를 기반으로 차트를 생성합니다.
    지원 차트 유형: bar, line, pie, scatter"""
    return f"'{chart_type}' 차트가 생성되었습니다."
```

> ⚠️ **흔한 오해**: "도구를 많이 줄수록 에이전트가 더 똑똑해진다"고 생각하기 쉬운데, 실제로는 정반대입니다. LangChain 벤치마크에 따르면 도구가 많아질수록 에이전트의 도구 선택 정확도가 떨어집니다. **적은 도구 + 명확한 설명**이 핵심이에요.

### 개념 3: 에이전트 프롬프트 최적화

> 💡 **비유**: 신입 사원에게 업무를 맡길 때, "일 잘 해"라고만 하면 뭘 해야 할지 모르겠죠? "고객 문의 이메일에 24시간 이내에 답변하되, 환불 요청은 팀장에게 에스컬레이션하세요"처럼 구체적으로 알려줘야 합니다. 에이전트의 시스템 프롬프트도 똑같습니다.

에이전트의 프롬프트는 단순한 역할 설명이 아닙니다. **행동 지침서**에 가깝죠. 좋은 에이전트 프롬프트에는 다음 요소가 포함되어야 합니다:

1. **역할 정의**: 누구인지 명확히
2. **행동 범위**: 무엇을 하고, 무엇을 하지 않는지
3. **도구 사용 가이드**: 어떤 상황에서 어떤 도구를 쓸지
4. **출력 형식**: 결과를 어떤 포맷으로 반환할지
5. **에스컬레이션 규칙**: 자신이 처리할 수 없는 작업의 처리 방법

```python
# ❌ 나쁜 프롬프트: 너무 모호함
bad_prompt = "당신은 연구 에이전트입니다."

# ✅ 좋은 프롬프트: 구체적이고 행동 지향적
research_prompt = """당신은 기술 리서치 전문가입니다.

## 역할
최신 기술 동향, 논문, 공식 문서를 조사하여 정확한 정보를 제공합니다.

## 행동 규칙
1. 항상 web_search 도구로 최신 정보를 먼저 확인합니다
2. 학술적 근거가 필요하면 arxiv_search를 사용합니다
3. 검색 결과의 출처를 반드시 포함합니다
4. 확인되지 않은 정보는 "확인 필요"로 표시합니다

## 하지 말아야 할 것
- 코드 작성이나 실행 (→ 코딩 에이전트의 역할)
- 수치 계산이나 통계 분석 (→ 분석 에이전트의 역할)
- 추측에 기반한 답변

## 출력 형식
- 조사 결과를 구조화된 형태로 정리
- 각 정보에 출처 URL 포함
- 핵심 발견사항을 요약으로 시작
"""

coding_prompt = """당신은 시니어 Python 개발자입니다.

## 역할
Python 코드를 작성, 실행, 디버깅합니다.

## 행동 규칙
1. 코드를 작성하기 전에 요구사항을 확인합니다
2. execute_python 도구로 반드시 코드를 테스트합니다
3. 에러가 발생하면 원인을 분석하고 수정합니다
4. 타입 힌트와 docstring을 포함합니다

## 하지 말아야 할 것
- 웹 검색이나 자료 조사 (→ 연구 에이전트의 역할)
- 데이터 시각화 (→ 분석 에이전트의 역할)

## 코딩 스타일
- PEP 8 준수
- 함수 단위 모듈화
- 의미 있는 변수명 사용
"""
```

프롬프트의 **"하지 말아야 할 것"** 섹션이 특히 중요합니다. 이것이 에이전트 간의 역할 경계를 명확하게 만들어 주거든요. LangChain 멀티 에이전트 가이드에서는 이를 **"네거티브 프롬프팅"**이라고 부르며 강력히 권장합니다.

### 개념 4: 능력 기반 라우팅

> 💡 **비유**: 콜센터의 ARS 시스템을 생각해 보세요. "결제 문의는 1번, 기술 지원은 2번, 해지는 3번..." 이렇게 고객의 요구를 파악해서 전문 상담원에게 연결하죠. 감독자의 라우팅도 이와 같습니다. 다만, 번호를 누르는 대신 **자연어 분석**으로 적합한 에이전트를 판단합니다.

능력 기반 라우팅은 감독자가 각 에이전트의 **전문 분야를 이해하고** 작업을 적절히 분배하는 메커니즘입니다. 감독자의 프롬프트에서 각 워커의 능력을 명시하는 것이 핵심이에요.

```python
from langgraph_supervisor import create_supervisor

# 감독자 프롬프트: 각 에이전트의 능력을 명확히 정의
supervisor_prompt = """당신은 프로젝트 매니저입니다. 
사용자의 요청을 분석하여 가장 적합한 전문가에게 작업을 위임합니다.

## 팀원별 전문 분야

### research_agent (리서치 전문가)
- 웹 검색, 논문 조사, 기술 동향 파악
- "~에 대해 알아봐", "최신 동향은?", "~가 뭐야?" 유형의 질문
- 사실 확인이 필요한 모든 작업

### coding_agent (코딩 전문가)  
- Python 코드 작성, 실행, 디버깅
- "코드 짜줘", "함수 만들어", "버그 수정해" 유형의 요청
- 알고리즘 구현, 데이터 처리 스크립트

### analysis_agent (분석 전문가)
- 수치 계산, 통계 분석, 데이터 시각화
- "계산해줘", "차트 그려줘", "분석해줘" 유형의 요청
- 데이터 해석과 인사이트 도출

## 라우팅 규칙
1. 요청의 핵심 의도를 파악하여 가장 적합한 에이전트를 선택합니다
2. 복합 작업은 순차적으로 여러 에이전트에 위임합니다
   예: "최신 AI 논문 찾아서 핵심 알고리즘을 구현해줘"
   → research_agent(논문 조사) → coding_agent(구현)
3. 애매한 경우, 먼저 research_agent에게 조사를 요청합니다
"""

# 전문화된 에이전트 생성
research_agent = create_react_agent(
    model,
    tools=[web_search, arxiv_search],
    prompt=research_prompt,
    name="research_agent",
)

coding_agent = create_react_agent(
    model,
    tools=[execute_python, read_file],
    prompt=coding_prompt,
    name="coding_agent",
)

analysis_agent = create_react_agent(
    model,
    tools=[calculate, generate_chart],
    prompt=analysis_prompt,
    name="analysis_agent",
)

# 감독자가 능력 기반으로 라우팅하는 워크플로우 구성
workflow = create_supervisor(
    [research_agent, coding_agent, analysis_agent],
    model=ChatOpenAI(model="gpt-4o"),
    prompt=supervisor_prompt,
)

# 컴파일 및 실행
app = workflow.compile()
```

**복합 작업의 순차 라우팅** 패턴이 특히 강력합니다. "최신 머신러닝 논문의 핵심 알고리즘을 Python으로 구현해줘"라는 요청이 들어오면, 감독자는 먼저 `research_agent`에게 논문 조사를 시키고, 그 결과를 `coding_agent`에게 넘겨 구현을 맡기죠.

### 개념 5: 컨텍스트 엔지니어링 — 에이전트가 보는 정보 제어

> 💡 **비유**: 회의실에서 프로젝트 논의를 할 때, 디자이너에게 서버 로그를 전부 보여줄 필요는 없죠. 디자이너에게는 UI 와이어프레임과 사용자 피드백만 있으면 충분합니다. 각 에이전트에게 **필요한 정보만 전달**하는 것이 컨텍스트 엔지니어링입니다.

LangChain 공식 가이드에서 강조하는 핵심 개념이 바로 **컨텍스트 엔지니어링**입니다. 감독자가 워커에게 작업을 넘길 때, 워커의 컨텍스트 윈도우에 **불필요한 정보가 쌓이지 않도록** 관리해야 합니다.

```python
from langgraph_supervisor import create_supervisor

# output_mode 설정으로 메시지 히스토리 관리
workflow = create_supervisor(
    [research_agent, coding_agent, analysis_agent],
    model=ChatOpenAI(model="gpt-4o"),
    prompt=supervisor_prompt,
    # "last_message": 워커는 마지막 메시지만 봄 (컨텍스트 경량화)
    # "full_history": 워커가 전체 대화 히스토리를 봄
    output_mode="last_message",
)
```

`output_mode="last_message"`를 사용하면 워커 에이전트가 감독자의 라우팅 로직이나 다른 에이전트의 대화 내용을 볼 필요 없이, 자신에게 할당된 작업에만 집중할 수 있습니다. 이는 **응답 품질과 속도를 모두 개선**하는 실전 기법입니다.

## 실습: 직접 해보기

이제 앞서 배운 개념을 종합하여, **기술 블로그 작성 어시스턴트** 멀티 에이전트 시스템을 구현해 보겠습니다. 연구 에이전트가 자료를 조사하고, 코딩 에이전트가 예제 코드를 작성하며, 에디터 에이전트가 최종 글을 다듬는 시스템입니다.

```python
"""
기술 블로그 작성 어시스턴트 — 전문화된 멀티 에이전트 시스템
pip install langchain-openai langgraph langgraph-supervisor python-dotenv
"""

import os
from dotenv import load_dotenv
from langchain_openai import ChatOpenAI
from langchain_core.tools import tool
from langgraph.prebuilt import create_react_agent
from langgraph_supervisor import create_supervisor

load_dotenv()

# ──────────────────────────────────────────
# 1. 도구 정의: 각 에이전트의 전문 도구
# ──────────────────────────────────────────

# 연구 에이전트용 도구
@tool
def search_tech_articles(query: str) -> str:
    """기술 블로그, 공식 문서, 뉴스에서 관련 기술 정보를 검색합니다.
    기술 동향, 프레임워크 비교, 베스트 프랙티스 등을 찾을 때 사용합니다."""
    # 실제 구현에서는 Tavily API, Google Custom Search 등 사용
    return f"""[검색 결과: '{query}']
    1. "LangGraph v0.3 릴리즈 - 멀티 에이전트 지원 강화" (2025.09)
       - StateGraph 개선, 체크포인터 성능 40% 향상
    2. "프로덕션 멀티 에이전트 시스템 설계 가이드" (2025.08)
       - 에이전트 수는 2~4개 권장, 도구 격리 필수
    3. "LangChain vs CrewAI vs AutoGen 비교" (2025.07)
       - LangGraph의 유연성과 디버깅 용이성이 장점"""

@tool
def search_academic_papers(query: str) -> str:
    """arXiv 등 학술 데이터베이스에서 관련 논문을 검색합니다.
    이론적 배경, 알고리즘 원리, 벤치마크 결과가 필요할 때 사용합니다."""
    return f"""[논문 검색: '{query}']
    1. "Scaling Multi-Agent Systems" (arXiv:2025.12345)
       - 에이전트 수 증가에 따른 성능 변화 분석
    2. "Capability-Based Routing in LLM Agents" (arXiv:2025.67890)
       - 능력 기반 라우팅의 정확도 92% 달성"""

# 코딩 에이전트용 도구
@tool
def run_python_code(code: str) -> str:
    """Python 코드를 실행하고 결과를 반환합니다.
    코드 예제 검증, 알고리즘 테스트에 사용합니다."""
    try:
        # 교육용 간이 구현 (프로덕션에서는 샌드박스 사용)
        import io, contextlib
        output = io.StringIO()
        with contextlib.redirect_stdout(output):
            exec(code, {"__builtins__": __builtins__})
        result = output.getvalue()
        return result if result else "코드가 성공적으로 실행되었습니다. (출력 없음)"
    except Exception as e:
        return f"실행 오류: {type(e).__name__}: {e}"

@tool
def validate_code_style(code: str) -> str:
    """Python 코드의 스타일과 품질을 검사합니다.
    PEP 8 준수 여부, 타입 힌트 유무, docstring 존재 여부를 확인합니다."""
    issues = []
    if "def " in code and '"""' not in code and "'''" not in code:
        issues.append("- docstring이 없는 함수가 있습니다")
    if "def " in code and "->" not in code:
        issues.append("- 반환 타입 힌트가 누락되었습니다")
    if not issues:
        return "✅ 코드 스타일 검사 통과!"
    return "⚠️ 개선 필요:\n" + "\n".join(issues)

# 에디터 에이전트용 도구
@tool
def check_grammar(text: str) -> str:
    """기술 글의 문법, 어조, 가독성을 검사합니다.
    맞춤법, 문장 길이, 전문 용어 일관성을 확인합니다."""
    suggestions = []
    if len(text.split("\n")) < 3:
        suggestions.append("- 문단을 더 나눠 가독성을 높이세요")
    if "그리고" in text and text.count("그리고") > 2:
        suggestions.append("- '그리고'가 반복됩니다. 다양한 접속사를 사용하세요")
    if not suggestions:
        return "✅ 문법 및 스타일 검사 통과!"
    return "📝 개선 제안:\n" + "\n".join(suggestions)

@tool
def format_as_markdown(content: str) -> str:
    """콘텐츠를 기술 블로그에 적합한 마크다운 형식으로 변환합니다.
    제목 계층, 코드 블록, 목록 등을 정리합니다."""
    # 간이 포맷팅 (실제 구현에서는 더 정교하게)
    formatted = content.strip()
    if not formatted.startswith("#"):
        formatted = f"# 기술 블로그\n\n{formatted}"
    return formatted

# ──────────────────────────────────────────
# 2. 전문화된 에이전트 프롬프트
# ──────────────────────────────────────────

RESEARCH_PROMPT = """당신은 기술 리서처입니다.

## 역할
기술 블로그 작성에 필요한 자료를 조사합니다.

## 행동 규칙
1. search_tech_articles로 최신 기술 동향을 먼저 파악합니다
2. 이론적 배경이 필요하면 search_academic_papers를 사용합니다
3. 조사 결과에 항상 출처를 포함합니다
4. 핵심 발견사항을 글머리 기호로 정리합니다

## 하지 말아야 할 것
- 코드 작성이나 실행
- 글 편집이나 교정
"""

CODING_PROMPT = """당신은 Python 시니어 개발자입니다.

## 역할
기술 블로그에 들어갈 코드 예제를 작성하고 검증합니다.

## 행동 규칙
1. 독자가 바로 실행할 수 있는 완전한 코드를 작성합니다
2. run_python_code로 반드시 코드를 테스트합니다
3. validate_code_style로 코드 품질을 확인합니다
4. 주석을 풍부하게 추가합니다

## 하지 말아야 할 것
- 자료 조사 (연구 에이전트의 역할)
- 글 편집이나 문법 교정 (에디터 에이전트의 역할)

## 코딩 스타일
- PEP 8 준수, 타입 힌트 포함
- 단계별로 복잡도를 높이는 예제 구성
"""

EDITOR_PROMPT = """당신은 기술 에디터입니다.

## 역할
연구 자료와 코드 예제를 종합하여 읽기 좋은 기술 블로그를 완성합니다.

## 행동 규칙
1. check_grammar으로 문법과 가독성을 확인합니다
2. format_as_markdown으로 마크다운 형식을 정돈합니다
3. 연구 결과 → 개념 설명 → 코드 예제 순서로 구성합니다
4. 도입부에 독자의 관심을 끄는 한 줄을 배치합니다

## 하지 말아야 할 것
- 자료 조사 (연구 에이전트의 역할)
- 새로운 코드 작성 (코딩 에이전트의 역할)
"""

SUPERVISOR_PROMPT = """당신은 기술 블로그 팀의 프로젝트 매니저입니다.

## 팀 구성
- researcher: 자료 조사 전문가 (기술 동향, 논문 검색)
- coder: Python 개발자 (코드 예제 작성, 검증)
- editor: 기술 에디터 (글 편집, 포맷팅)

## 작업 흐름
1. 블로그 주제를 받으면 먼저 researcher에게 자료 조사를 요청합니다
2. 조사 결과를 바탕으로 coder에게 코드 예제 작성을 요청합니다
3. 모든 자료가 준비되면 editor에게 최종 글 작성을 요청합니다

## 라우팅 규칙
- "조사해줘", "찾아봐", "동향" → researcher
- "코드", "구현", "예제" → coder
- "정리해줘", "다듬어줘", "편집" → editor
- 복합 요청은 위 순서대로 순차 실행합니다
"""

# ──────────────────────────────────────────
# 3. 멀티 에이전트 시스템 조립
# ──────────────────────────────────────────

model = ChatOpenAI(model="gpt-4o", temperature=0)

# 전문 에이전트 생성 — 각자 다른 도구와 프롬프트를 가짐
researcher = create_react_agent(
    model,
    tools=[search_tech_articles, search_academic_papers],
    prompt=RESEARCH_PROMPT,

    tools=[run_python_code, validate_code_style],

    tools=[check_grammar, format_as_markdown],

)

workflow = create_supervisor(
    [researcher, coder, editor],
    model=ChatOpenAI(model="gpt-4o"),  # 감독자도 별도 모델 가능

    output_mode="last_message",  # 워커 컨텍스트 경량화
)

# 컴파일
app = workflow.compile()

# ──────────────────────────────────────────
# 4. 실행
# ──────────────────────────────────────────

if __name__ == "__main__":
    # 복합 작업: 조사 → 코딩 → 편집 순으로 자동 라우팅
    result = app.invoke({
        "messages": [
            {
                "role": "user",
                "content": (
                    "LangGraph 멀티 에이전트 시스템에 대한 "

                    "최신 동향 조사, 코드 예제, 깔끔한 편집까지 부탁해."
                )
            }
        ]
    })

    # 최종 결과 출력
    print("=" * 60)
    print("📝 최종 블로그 결과:")

    for msg in result["messages"]:

```

위 코드에서 주목할 포인트를 정리해 볼까요?

1. **도구 격리**: 각 에이전트가 자신의 전문 도구만 보유 — `researcher`는 검색 도구만, `coder`는 코드 실행 도구만
2. **프롬프트 구조**: 역할 → 행동 규칙 → 금지 사항 → 스타일 가이드 순서의 일관된 구조
3. **감독자 라우팅**: 작업 유형에 따른 명시적 라우팅 규칙과 복합 작업의 순차 처리 전략
4. **컨텍스트 경량화**: `output_mode="last_message"`로 워커 간 불필요한 정보 차단

## 더 깊이 알아보기

### "한 가지 일을 잘 하라" — Unix 철학에서 에이전트 설계까지

전문화된 에이전트 설계의 원칙은 사실 소프트웨어 역사에서 오랫동안 검증되어 온 사상에 뿌리를 두고 있습니다. 1978년, 벨 연구소의 더글러스 매킬로이(Douglas McIlroy)가 정리한 **Unix 철학**이 바로 그것이에요.

> *"한 가지 일을 하라, 그리고 그것을 잘 하라(Do one thing and do it well)."*

Unix에서 `grep`은 텍스트 검색만, `sort`는 정렬만, `wc`는 단어 세기만 합니다. 이 작고 전문적인 프로그램들을 **파이프(`|`)로 연결**해서 복잡한 작업을 수행하죠. `cat access.log | grep "ERROR" | sort | uniq -c`처럼요. 어디서 많이 본 패턴 아닌가요? 맞습니다 — LCEL의 파이프 연산자(`|`)로 컴포넌트를 연결하는 것과 놀랍도록 닮아 있어요!

이 철학은 2010년대 마이크로서비스(Microservices) 아키텍처로 다시 부활했습니다. 거대한 모놀리식 시스템을 잘게 쪼개 **각 서비스가 하나의 비즈니스 기능만 담당**하도록 설계하는 패턴이죠. 넷플릭스는 단일 애플리케이션을 700개 이상의 마이크로서비스로 분리해 서비스 안정성을 크게 높였습니다. 각 서비스는 자체 데이터베이스를 갖고(도구 격리), API 계약으로만 소통하며(인터페이스 정의), 독립적으로 배포됩니다(자율성).

멀티 에이전트 시스템의 전문화 설계는 이 두 가지 전통을 정확히 계승합니다:

| Unix 철학 | 마이크로서비스 | 멀티 에이전트 |
|-----------|---------------|--------------|
| 한 가지 일을 잘 하라 | 단일 책임 원칙 | 역할 기반 에이전트 분리 |
| 파이프로 조합 | API Gateway | 감독자 라우팅 |
| stdin/stdout 표준 인터페이스 | REST/gRPC 계약 | 핸드오프 메시지 |
| 작은 프로그램 선호 | 서비스 경량화 | 도구 2~5개 권장 |

기술의 형태는 커맨드라인 → 웹 서비스 → AI 에이전트로 진화했지만, **"작고 전문적인 단위를 잘 연결하면 복잡한 문제를 풀 수 있다"**는 원칙은 50년째 유효합니다. 에이전트를 설계할 때 이 원칙을 의식적으로 적용하면, 유지보수하기 쉽고 확장 가능한 시스템을 만들 수 있어요.

### 왜 "전문가 2~4명"이 최적일까?

LangChain 팀이 내부 벤치마크에서 발견한 흥미로운 패턴이 있습니다. 에이전트 수를 1개에서 8개까지 늘려가며 테스트했을 때, **3~4개**에서 성능이 정점을 찍고 이후 오히려 하락했습니다. 그 이유는 감독자의 라우팅 정확도 때문이에요. 선택지가 많아질수록 LLM의 의사결정 품질이 떨어지는 현상은, 심리학에서 말하는 "선택의 역설(Paradox of Choice)"과 놀랍도록 유사합니다. 배리 슈워츠(Barry Schwartz)가 2004년에 인간의 의사결정에서 발견한 현상이 LLM에서도 나타나는 거죠.

## 흔한 오해와 팁

> ⚠️ **흔한 오해**: "에이전트마다 다른 LLM을 써야 전문화가 된다"고 생각하기 쉽습니다. 하지만 전문화의 핵심은 LLM 선택이 아니라 **도구 할당과 프롬프트 설계**입니다. 동일한 GPT-4o라도 프롬프트와 도구가 다르면 전혀 다른 행동을 합니다. 물론 비용 최적화를 위해 간단한 작업에 경량 모델을 쓰는 것은 별개의 이야기입니다.

> 💡 **알고 계셨나요?**: LangChain 공식 멀티 에이전트 가이드에서는 감독자가 워커에게 작업을 넘길 때 **핸드오프 메시지를 제거**하라고 권장합니다. 감독자가 "이 작업은 coding_agent에게 넘겨야겠다"라고 판단한 내부 로직이 워커의 컨텍스트에 남아 있으면, 워커가 혼란을 겪을 수 있기 때문이에요. 이것이 `output_mode="last_message"`의 존재 이유입니다.

> 🔥 **실무 팁**: 프로덕션에서 에이전트 프롬프트를 최적화할 때, **실패 케이스 로그**를 분석하는 것이 가장 효과적입니다. LangSmith 같은 관찰 가능성 도구로 에이전트가 잘못된 도구를 선택하거나 역할을 벗어난 응답을 한 경우를 추적하고, 그 패턴을 프롬프트의 "하지 말아야 할 것" 섹션에 추가하세요. 이 피드백 루프가 에이전트 품질을 지속적으로 개선하는 핵심 전략입니다.

## 핵심 정리

| 개념 | 설명 |
|------|------|
| 역할 기반 설계 | 단일 책임 원칙에 따라 에이전트를 전문 영역별로 분리 (2~4개 권장) |
| 도구 할당 | 최소 권한 원칙으로 필요한 도구만 제공, `@tool`의 docstring이 도구 선택을 좌우 |
| 프롬프트 최적화 | 역할 + 행동 규칙 + 금지 사항 + 출력 형식의 구조화된 시스템 프롬프트 |
| 능력 기반 라우팅 | 감독자 프롬프트에 각 에이전트의 전문 분야와 라우팅 규칙을 명시 |
| 컨텍스트 엔지니어링 | `output_mode` 등을 활용하여 에이전트가 보는 정보를 최적화 |
| 네거티브 프롬프팅 | "하지 말아야 할 것"을 명시하여 에이전트 간 역할 경계를 강화 |
