# 에이전트 도구 설계

> 에이전트가 올바른 도구를 선택하도록 만드는 핵심은 도구의 설명과 스키마 설계에 있습니다.

## 개요

이 섹션에서는 에이전트가 도구를 정확하게 선택하고 사용할 수 있도록, 도구를 **잘 설계하는 방법**을 배웁니다. 앞서 [12.1: 에이전트 개념과 ReAct 패턴](./01-에이전트-개념과-react-패턴.md)에서 에이전트의 기본 구조를, [12.2: create_react_agent로 에이전트 구축](./02-create-react-agent로-에이전트-구축.md)에서 에이전트 생성 방법을, [12.3: AgentExecutor 설정과 제어](./03-agentexecutor-설정과-제어.md)에서 실행 제어를 배웠는데요, 이번에는 에이전트의 **손과 발**에 해당하는 도구 자체를 어떻게 설계해야 하는지 다룹니다.

**선수 지식**: `@tool` 데코레이터 기본 사용법, ReAct 패턴의 Thought→Action→Observation 루프, AgentExecutor 기본 설정

**학습 목표**:
- 에이전트가 올바르게 도구를 선택하도록 **도구 설명(description)**을 작성할 수 있다
- Pydantic `args_schema`로 **입력 스키마를 최적화**할 수 있다
- 여러 도구를 효과적으로 **조합하는 전략**을 세울 수 있다
- `return_direct`, `handle_tool_error` 등 **도구 옵션**을 상황에 맞게 설정할 수 있다

## 왜 알아야 할까?

에이전트를 만들어 본 분이라면 이런 경험이 있을 겁니다. 분명 검색 도구와 계산 도구를 모두 제공했는데, 에이전트가 **엉뚱한 도구를 선택**하거나, 올바른 도구를 골랐는데 **입력값을 잘못 넣는** 경우요.

이런 문제의 90%는 도구 코드의 로직이 아니라 **도구의 설명과 스키마 설계**에서 비롯됩니다. LLM은 도구의 내부 구현을 볼 수 없거든요. LLM이 보는 것은 오직 **도구 이름**, **설명문**, **입력 파라미터 스키마** 세 가지뿐입니다. 이 세 가지가 명확하지 않으면, 아무리 뛰어난 LLM이라도 도구를 잘못 사용하게 됩니다.

실무에서 에이전트의 정확도를 높이는 가장 빠른 방법은 모델을 바꾸는 것이 아니라, **도구 설명을 다시 쓰는 것**입니다.

## 핵심 개념

### 개념 1: 도구 설명(Description) 작성의 기술

> 💡 **비유**: 식당 메뉴판을 떠올려보세요. "특선 요리 A" 라고만 적혀 있으면 뭘 주문해야 할지 모르죠. 하지만 "매콤한 해물 짬뽕 — 새우, 오징어, 홍합이 들어간 칼칼한 국물 요리. 매운 음식을 좋아하는 분에게 추천"이라고 적혀 있으면 바로 결정할 수 있습니다. 도구 설명은 에이전트에게 보여주는 **메뉴판**입니다.

에이전트의 LLM은 도구를 선택할 때 **도구 이름**과 **설명문(description)**만을 보고 판단합니다. 그래서 설명문을 어떻게 쓰느냐가 에이전트의 도구 선택 정확도를 결정하죠.

좋은 도구 설명에는 세 가지 요소가 있어야 합니다:

1. **무엇을 하는지** (What): 이 도구의 기능
2. **언제 사용하는지** (When): 어떤 상황에서 이 도구를 골라야 하는지
3. **언제 사용하지 말아야 하는지** (When NOT): 비슷한 도구와의 구분

```python
from langchain_core.tools import tool

# ❌ 나쁜 도구 설명 — 너무 모호함
@tool
def search(query: str) -> str:
    """검색을 수행합니다."""
    # ...

# ✅ 좋은 도구 설명 — 명확하고 구체적
@tool
def search_web(query: str) -> str:
    """인터넷에서 최신 정보를 검색합니다.
    
    이 도구는 실시간 웹 검색을 수행하여 최신 뉴스, 현재 날씨, 
    최근 이벤트 등 시간에 민감한 정보를 찾을 때 사용합니다.
    
    사용해야 할 때: 현재 시점의 정보가 필요한 질문
    사용하지 말아야 할 때: 일반 상식이나 수학 계산이 필요한 경우
    """
    # ...
```

도구 이름도 중요합니다. `search`보다 `search_web`이, `calc`보다 `calculate_math`가 LLM이 의도를 파악하기 훨씬 쉽거든요.

```python
# 도구 이름을 명시적으로 지정하는 방법
@tool("search_company_database")
def search_db(query: str) -> str:
    """회사 내부 데이터베이스에서 직원, 부서, 프로젝트 정보를 검색합니다.
    
    SQL 쿼리가 아닌 자연어 질문을 입력으로 받습니다.
    사용해야 할 때: 회사 내부 정보(직원 이름, 부서, 프로젝트 현황)가 필요한 경우
    사용하지 말아야 할 때: 인터넷 검색이나 일반 지식이 필요한 경우
    """
    # ...
```

### 개념 2: Pydantic args_schema로 입력 스키마 최적화

> 💡 **비유**: 병원 접수 양식을 생각해보세요. "증상을 적어주세요"라고만 하면 어떤 사람은 "아파요"만 쓰고, 어떤 사람은 에세이를 씁니다. 하지만 `이름: ___`, `부위: [머리/배/등]`, `증상 시작일: ____/____/____` 처럼 **구조화된 양식**을 주면 모든 사람이 필요한 정보를 정확히 기입하죠. `args_schema`는 에이전트에게 주는 **접수 양식**입니다.

`@tool` 데코레이터를 사용하면 함수의 타입 힌트에서 자동으로 입력 스키마가 생성됩니다. 하지만 복잡한 도구라면 Pydantic 모델로 **명시적 스키마**를 정의하는 것이 훨씬 효과적입니다.

```python
from pydantic import BaseModel, Field
from langchain_core.tools import tool
from typing import Optional, Literal

# 기본 방식 — 타입 힌트만 사용
@tool
def search_products(query: str, category: str, max_price: int) -> str:
    """상품을 검색합니다."""
    # LLM은 category에 뭘 넣어야 할지, max_price 단위가 뭔지 모릅니다
    ...

# 개선된 방식 — Pydantic args_schema 사용
class ProductSearchInput(BaseModel):
    """상품 검색 입력 스키마"""
    query: str = Field(
        description="검색할 상품명 또는 키워드. 예: '무선 이어폰', '겨울 코트'"
    )
    category: Literal["electronics", "fashion", "food", "books"] = Field(
        description="상품 카테고리. 반드시 지정된 값 중 하나를 선택"
    )
    max_price: Optional[int] = Field(
        default=None,
        description="최대 가격(원). 미지정 시 가격 제한 없음. 예: 50000"
    )

@tool(args_schema=ProductSearchInput)
def search_products(query: str, category: str, max_price: Optional[int] = None) -> str:
    """온라인 쇼핑몰에서 상품을 검색합니다.
    
    상품명, 카테고리, 가격 범위를 기반으로 관련 상품 목록을 반환합니다.
    사용해야 할 때: 사용자가 특정 상품을 찾거나 가격 비교를 원할 때
    """
    results = f"'{query}' 검색 결과 (카테고리: {category}"
    if max_price:
        results += f", {max_price:,}원 이하"
    results += "): ..."
    return results
```

핵심 포인트는 `Field`의 `description`입니다. 이 설명이 LLM에게 전달되어 각 파라미터에 **어떤 값을 넣어야 하는지** 안내하거든요. `Literal`로 선택지를 제한하면 LLM이 엉뚱한 값을 넣는 것도 방지할 수 있습니다.

스키마가 에이전트에게 어떻게 보이는지 확인해볼까요?

```python
# 도구의 스키마를 확인하는 방법
print(search_products.name)
# 출력: search_products

print(search_products.description)
# 출력: 온라인 쇼핑몰에서 상품을 검색합니다. ...

print(search_products.args)
# 출력: {'query': {'description': '검색할 상품명 또는 키워드...', 'title': 'Query', 'type': 'string'},
#        'category': {'description': '상품 카테고리...', 'enum': ['electronics', 'fashion', 'food', 'books'], ...},
#        'max_price': {'description': '최대 가격(원)...', 'title': 'Max Price', 'type': 'integer'}}
```

### 개념 3: 도구 옵션 — return_direct와 handle_tool_error

> 💡 **비유**: 회사에서 업무를 요청할 때, 어떤 일은 "결과 나오면 바로 고객에게 보내줘" (`return_direct=True`)이고, 어떤 일은 "결과 나오면 내가 한번 확인하고 보낼게" (`return_direct=False`)죠. 그리고 실수가 생기면 "일단 뭐가 잘못됐는지 알려줘" (`handle_tool_error=True`)라고 하잖아요.

LangChain 도구에는 실행 동작을 제어하는 중요한 옵션들이 있습니다:

```python
from langchain_core.tools import tool, ToolException

# return_direct: 도구 결과를 바로 최종 응답으로 사용
@tool(return_direct=True)
def get_stock_price(symbol: str) -> str:
    """주식 현재가를 조회합니다.
    
    종목 코드(예: AAPL, GOOGL)를 입력하면 현재 주가를 반환합니다.
    결과가 바로 사용자에게 전달되므로 친절한 형식으로 응답합니다.
    """
    # return_direct=True이면 AgentExecutor가 이 결과를 
    # 바로 최종 응답으로 반환하고 루프를 중단합니다
    price = 185.50  # 실제로는 API 호출
    return f"📈 {symbol} 현재가: ${price:.2f} (2024년 기준)"


# handle_tool_error: 도구 실행 중 에러 처리
@tool(handle_tool_error=True)
def divide_numbers(a: float, b: float) -> str:
    """두 수를 나눕니다."""
    if b == 0:
        raise ToolException(
            "0으로 나눌 수 없습니다. 다른 값을 사용해주세요."
        )
    return f"{a} ÷ {b} = {a / b:.4f}"


# handle_tool_error에 커스텀 핸들러 함수 사용
def custom_error_handler(error: ToolException) -> str:
    return f"⚠️ 도구 실행 오류: {error}. 다른 방법을 시도해보세요."

@tool(handle_tool_error=custom_error_handler)
def query_database(sql: str) -> str:
    """데이터베이스 쿼리를 실행합니다."""
    # 에러 발생 시 custom_error_handler의 반환값이 
    # Observation으로 전달되어 에이전트가 재시도할 수 있습니다
    ...
```

`return_direct=True`는 **정보 조회성 도구**에 유용합니다. 날씨, 주가, 환율처럼 추가 추론 없이 결과를 바로 보여줄 수 있는 경우죠. 반면 계산이나 분석처럼 **후속 추론이 필요한 도구**에는 기본값(`False`)을 유지하세요.

`handle_tool_error=True`로 설정하면 도구에서 `ToolException`이 발생했을 때, 에이전트가 중단되지 않고 **에러 메시지를 Observation으로 받아** 다른 전략을 시도할 수 있습니다.

### 개념 4: 도구 조합 전략과 우선순위

> 💡 **비유**: 요리사에게 칼 10자루를 주면 오히려 혼란스러워집니다. 하지만 **과도(채소용)**, **식칼(고기용)**, **빵칼(빵 전용)**처럼 용도가 명확한 3자루를 주면 바로바로 적절한 칼을 골라 쓰죠. 에이전트에게 도구를 제공할 때도 마찬가지입니다 — **적은 수의 역할이 명확한 도구**가 **많은 수의 모호한 도구**보다 낫습니다.

도구 조합에서 가장 중요한 원칙은 **겹치지 않는 역할 분담**입니다:

```python
from langchain_core.tools import tool
from pydantic import BaseModel, Field
from typing import Literal

# ❌ 나쁜 설계: 역할이 겹치는 도구들
@tool
def search(query: str) -> str:
    """정보를 검색합니다."""  # 무슨 정보? 어디서?
    ...

@tool
def find_info(keyword: str) -> str:
    """정보를 찾습니다."""  # search와 뭐가 다르지?
    ...

# ✅ 좋은 설계: 역할이 명확히 구분된 도구들
@tool
def search_web(query: str) -> str:
    """인터넷에서 최신 정보를 실시간 검색합니다.
    
    사용 시점: 오늘 날씨, 최신 뉴스, 현재 주가 등 실시간 정보가 필요할 때
    사용하지 말 것: 회사 내부 데이터나 수학 계산이 필요할 때
    """
    ...

@tool
def search_internal_docs(query: str) -> str:
    """회사 내부 문서에서 정보를 검색합니다.
    
    사용 시점: 사내 규정, 매뉴얼, 제품 스펙 등 내부 문서 정보가 필요할 때
    사용하지 말 것: 외부 웹 정보나 실시간 데이터가 필요할 때
    """
    ...

@tool
def calculate(expression: str) -> str:
    """수학 계산을 수행합니다.
    
    사용 시점: 사칙연산, 비율 계산, 단위 변환 등 수학적 계산이 필요할 때
    사용하지 말 것: 정보 검색이나 텍스트 분석이 필요할 때
    입력 형식: Python 수식 문자열. 예: '(100 * 1.1) + 500'
    """
    ...
```

도구 개수에 대한 가이드라인도 있습니다:

| 도구 개수 | 권장 상황 | 주의사항 |
|-----------|-----------|----------|
| 1~3개 | 특화된 단일 목적 에이전트 | 가장 안정적인 선택 |
| 4~7개 | 범용 비서 에이전트 | 설명문 차별화 필수 |
| 8개 이상 | 고급 멀티태스크 에이전트 | 도구 그룹핑 또는 동적 선택 고려 |

도구가 많아질수록 **동적 도구 선택(Dynamic Tool Selection)** 패턴이 유용해집니다. 상황에 따라 에이전트에게 보여줄 도구 목록을 필터링하는 전략이죠:

```python
from langchain_core.tools import tool
from langchain_openai import ChatOpenAI
from langchain.agents import create_react_agent, AgentExecutor
from langchain_core.prompts import PromptTemplate

# 모든 도구 정의
all_tools = [search_web, search_internal_docs, calculate, 
             get_stock_price, query_database]

def select_tools_by_context(user_query: str) -> list:
    """사용자 질문의 맥락에 따라 관련 도구만 선택합니다."""
    tools = []
    
    # 키워드 기반 도구 필터링 (실무에서는 더 정교한 분류기 사용)
    query_lower = user_query.lower()
    
    if any(kw in query_lower for kw in ["검색", "찾아", "알려줘", "뉴스"]):
        tools.extend([search_web, search_internal_docs])
    if any(kw in query_lower for kw in ["계산", "얼마", "합계", "평균"]):
        tools.append(calculate)
    if any(kw in query_lower for kw in ["주가", "주식", "코스피"]):
        tools.append(get_stock_price)
    if any(kw in query_lower for kw in ["데이터", "쿼리", "DB"]):
        tools.append(query_database)
    
    # 매칭된 도구가 없으면 기본 도구셋 제공
    if not tools:
        tools = [search_web, calculate]
    
    return tools

# 사용 예시
user_input = "최근 삼성전자 주가를 알려주고 전월 대비 수익률을 계산해줘"
selected_tools = select_tools_by_context(user_input)
print(f"선택된 도구: {[t.name for t in selected_tools]}")
# 출력: 선택된 도구: ['calculate', 'get_stock_price']
```

### 개념 5: BaseTool 클래스로 고급 도구 설계

> 💡 **비유**: `@tool` 데코레이터가 전자레인지(간편하지만 기능 제한)라면, `BaseTool` 서브클래스는 전문 오븐(설정할 것은 많지만 세밀한 제어 가능)입니다. 대부분의 경우 전자레인지로 충분하지만, 상태 관리나 복잡한 초기화가 필요하면 오븐이 필요하죠.

`@tool` 데코레이터는 간단한 도구에 적합하지만, **내부 상태 유지**, **초기화 로직**, **비동기 처리** 같은 고급 기능이 필요하면 `BaseTool`을 상속받아 도구 클래스를 만드는 것이 좋습니다.

```python
from langchain_core.tools import BaseTool
from pydantic import BaseModel, Field
from typing import Optional, Type
import httpx

class WeatherInput(BaseModel):
    """날씨 조회 입력 스키마"""
    city: str = Field(description="도시명. 예: '서울', '부산', 'Tokyo'")
    unit: str = Field(
        default="celsius",
        description="온도 단위. 'celsius' 또는 'fahrenheit'"
    )

class WeatherTool(BaseTool):
    """외부 API를 사용하는 날씨 조회 도구"""
    
    name: str = "get_weather"
    description: str = (
        "특정 도시의 현재 날씨 정보를 조회합니다. "
        "도시명을 입력하면 기온, 습도, 날씨 상태를 반환합니다. "
        "사용 시점: 날씨, 기온, 미세먼지 등 기상 정보가 필요할 때. "
        "사용하지 말 것: 과거 기상 데이터 분석이나 예보가 필요할 때."
    )
    args_schema: Type[BaseModel] = WeatherInput
    return_direct: bool = False
    
    # BaseTool만의 장점: 인스턴스 변수로 상태 관리
    api_base_url: str = "https://api.weather.example.com"
    api_key: str = ""
    _request_count: int = 0  # 프라이빗 상태 추적
    
    def _run(self, city: str, unit: str = "celsius") -> str:
        """동기 실행 메서드 (필수 구현)"""
        self._request_count += 1
        # 실제로는 API 호출
        return f"🌤 {city} 현재 날씨: 맑음, 22°C, 습도 45%"
    
    async def _arun(self, city: str, unit: str = "celsius") -> str:
        """비동기 실행 메서드 (선택 구현)"""
        self._request_count += 1
        async with httpx.AsyncClient() as client:
            # 실제 비동기 API 호출
            return f"🌤 {city} 현재 날씨: 맑음, 22°C, 습도 45%"

# BaseTool 도구 사용
weather_tool = WeatherTool(api_key="your-api-key")
result = weather_tool.invoke({"city": "서울"})
print(result)
# 출력: 🌤 서울 현재 날씨: 맑음, 22°C, 습도 45%
```

`@tool`과 `BaseTool` 중 어떤 것을 선택할지는 다음 기준으로 판단하세요:

| 기준 | `@tool` 데코레이터 | `BaseTool` 서브클래스 |
|------|-------------------|--------------------|
| 복잡도 | 단순 함수 | 상태/초기화 필요 |
| 코드량 | 적음 (함수 1개) | 많음 (클래스 전체) |
| 비동기 | `async def`만 붙이면 됨 | `_arun` 별도 구현 |
| 상태 관리 | 클로저나 전역 변수 | 인스턴스 변수로 깔끔하게 |
| 설정 주입 | 함수 인자나 환경 변수 | 생성자 파라미터 |
| 권장 상황 | 80%의 일반적 경우 | 20%의 고급 케이스 |

## 실습: 직접 해보기

이제 배운 내용을 종합해서, **스마트 리서치 에이전트**를 구축해봅시다. 웹 검색, 계산, 요약 기능을 갖춘 에이전트인데, 도구 설계에 특별히 공을 들여볼 거예요.

```python
import os
from dotenv import load_dotenv
from pydantic import BaseModel, Field
from typing import Optional, Literal
from langchain_core.tools import tool, ToolException
from langchain_openai import ChatOpenAI
from langchain.agents import create_react_agent, AgentExecutor
from langchain_core.prompts import PromptTemplate

load_dotenv()

# ===== 1단계: Pydantic 스키마 정의 =====

class WebSearchInput(BaseModel):
    """웹 검색 입력"""
    query: str = Field(
        description="검색 쿼리. 구체적일수록 좋은 결과를 얻습니다. 예: 'Python 3.12 새로운 기능'"
    )
    num_results: int = Field(
        default=3,
        description="반환할 검색 결과 수 (1~5). 기본값 3"
    )

class CalculatorInput(BaseModel):
    """수학 계산 입력"""
    expression: str = Field(
        description="Python 수식 문자열. 예: '(100 * 0.15) + 250', '2 ** 10'"
    )

class TextSummaryInput(BaseModel):
    """텍스트 요약 입력"""
    text: str = Field(
        description="요약할 텍스트. 최소 50자 이상이어야 의미 있는 요약이 가능합니다"
    )
    style: Literal["bullet", "paragraph"] = Field(
        default="bullet",
        description="요약 형식. 'bullet': 핵심 포인트 목록, 'paragraph': 한 문단 요약"
    )

# ===== 2단계: 도구 구현 =====

@tool(args_schema=WebSearchInput)
def search_web(query: str, num_results: int = 3) -> str:
    """인터넷에서 최신 정보를 실시간 검색합니다.
    
    사용해야 할 때: 최신 뉴스, 기술 동향, 인물 정보, 통계 데이터 등 
    실시간으로 변할 수 있는 정보가 필요할 때
    사용하지 말아야 할 때: 수학 계산이나 이미 주어진 텍스트 분석이 필요할 때
    """
    # 실제로는 검색 API 호출 (데모용 시뮬레이션)
    mock_results = {
        "Python": [
            "Python 3.12 릴리스: 성능 15% 향상, 새로운 타입 힌트 문법 추가",
            "Python이 2024 TIOBE 1위 프로그래밍 언어로 선정",
            "FastAPI 0.110 출시, Python 웹 프레임워크 트렌드 분석",
        ],
        "LangChain": [
            "LangChain v0.3 릴리스: Pydantic v2 완전 지원",
            "LangGraph 0.2: 멀티 에이전트 시스템 지원 강화",
            "RAG 패턴 진화: LangChain 기반 프로덕션 사례 분석",
        ],
    }
    
    # 쿼리에서 키워드 매칭
    for keyword, results in mock_results.items():
        if keyword.lower() in query.lower():
            selected = results[:num_results]
            return "\n".join(
                f"{i+1}. {r}" for i, r in enumerate(selected)
            )
    
    return f"'{query}' 검색 결과: 관련 정보를 찾았습니다. (시뮬레이션 데이터)"


@tool(args_schema=CalculatorInput, handle_tool_error=True)
def calculate(expression: str) -> str:
    """수학 계산을 수행합니다. Python 수식을 평가하여 결과를 반환합니다.
    
    사용해야 할 때: 사칙연산, 비율 계산, 거듭제곱, 단위 변환 등 
    수학적 계산이 필요할 때
    사용하지 말아야 할 때: 정보 검색이나 텍스트 처리가 필요할 때
    주의: 보안을 위해 수학 연산만 허용됩니다 (import, exec 등 불가)
    """
    # 보안: 허용된 함수만 사용 가능하도록 제한
    allowed_names = {
        "abs": abs, "round": round, "min": min, "max": max,
        "sum": sum, "pow": pow, "len": len,
    }
    
    # 위험한 키워드 차단
    dangerous = ["import", "exec", "eval", "open", "__", "os.", "sys."]
    if any(d in expression for d in dangerous):
        raise ToolException(
            "보안 제한: 수학 수식만 입력 가능합니다. "
            "예: '(100 * 1.15) + 500'"
        )
    
    try:
        result = eval(expression, {"__builtins__": {}}, allowed_names)
        return f"계산 결과: {expression} = {result}"
    except Exception as e:
        raise ToolException(
            f"계산 오류: '{expression}'을 계산할 수 없습니다. "
            f"올바른 Python 수식을 입력해주세요. (오류: {e})"
        )


@tool(args_schema=TextSummaryInput)
def summarize_text(text: str, style: str = "bullet") -> str:
    """주어진 텍스트를 핵심만 요약합니다.
    
    사용해야 할 때: 긴 텍스트나 검색 결과를 간결하게 정리해야 할 때
    사용하지 말아야 할 때: 새로운 정보 검색이나 계산이 필요할 때
    """
    # 간단한 요약 시뮬레이션 (실제로는 LLM 호출)
    sentences = [s.strip() for s in text.split(".") if s.strip()]
    
    if style == "bullet":
        # 핵심 문장을 불릿 포인트로
        key_points = sentences[:3]
        return "📋 요약:\n" + "\n".join(
            f"• {point}" for point in key_points
        )
    else:
        # 첫 2문장을 하나의 문단으로
        return f"📋 요약: {'. '.join(sentences[:2])}."


# ===== 3단계: 에이전트 구성 =====

# 도구 목록 — 역할이 명확히 구분된 3개
tools = [search_web, calculate, summarize_text]

# 에이전트가 도구 정보를 확인
for t in tools:
    print(f"[{t.name}]")
    print(f"  설명: {t.description[:60]}...")
    print(f"  파라미터: {list(t.args.keys())}")
    print()

# ReAct 프롬프트 — 도구 선택 가이드 포함
prompt = PromptTemplate.from_template("""
Answer the following questions as best you can. You have access to the following tools:

{tools}

IMPORTANT: Choose the most appropriate tool for each step.
- For finding information: use search_web
- For math calculations: use calculate  
- For summarizing long text: use summarize_text

Use the following format:

Question: the input question you must answer
Thought: you should always think about what to do
Action: the action to take, should be one of [{tool_names}]
Action Input: the input to the action
Observation: the result of the action
... (this Thought/Action/Action Input/Observation can repeat N times)
Thought: I now know the final answer
Final Answer: the final answer to the original input question (answer in Korean)

Begin!

Question: {input}
Thought:{agent_scratchpad}
""")

# LLM 및 에이전트 생성
llm = ChatOpenAI(model="gpt-4o", temperature=0)
agent = create_react_agent(llm, tools, prompt)

executor = AgentExecutor(
    agent=agent,
    tools=tools,
    verbose=True,            # 추론 과정 출력
    max_iterations=5,        # 최대 5번 시도
    handle_parsing_errors=True,
    return_intermediate_steps=True,  # 중간 단계 반환
)

# ===== 4단계: 실행 =====
response = executor.invoke({
    "input": "LangChain 최신 동향을 검색하고, 핵심 내용을 요약해줘"
})

print("\n" + "=" * 60)
print("📌 최종 답변:")
print(response["output"])

# 중간 단계 분석 — 도구 선택이 적절했는지 확인
print("\n📊 에이전트 추론 과정:")
for i, (action, observation) in enumerate(response["intermediate_steps"]):
    print(f"  Step {i+1}: {action.tool}({action.tool_input})")
    print(f"  → 결과: {observation[:80]}...")
    print()
```

실행하면 에이전트가 다음과 같은 추론 과정을 거칩니다:

```
Thought: LangChain 최신 동향을 찾기 위해 웹 검색을 먼저 하겠습니다.
Action: search_web
Action Input: {"query": "LangChain 최신 동향", "num_results": 3}
Observation: 1. LangChain v0.3 릴리스: Pydantic v2 완전 지원...

Thought: 검색 결과를 간결하게 정리해야 합니다.
Action: summarize_text
Action Input: {"text": "LangChain v0.3 릴리스...", "style": "bullet"}
Observation: 📋 요약: • LangChain v0.3 릴리스...

Thought: I now know the final answer

```

도구 설명이 명확하기 때문에 에이전트가 `search_web` → `summarize_text` 순서로 올바르게 도구를 선택한 것을 확인할 수 있습니다.

## 더 깊이 알아보기

### 도구 사용의 역사: ReAct에서 Function Calling까지

에이전트가 도구를 사용한다는 아이디어는 2022년 **Shunyu Yao** 등이 발표한 [ReAct 논문](https://arxiv.org/abs/2210.03629)에서 체계화되었습니다. 이 논문의 핵심 통찰은 당시 별도로 발전하던 **추론(Reasoning)**과 **행동(Acting)**을 결합하면 시너지가 생긴다는 것이었는데요, 놀랍게도 이 논문은 ChatGPT가 등장하기 불과 한 달 전인 2022년 10월에 발표되었습니다. ChatGPT의 폭발적 인기와 함께 이 ReAct 패턴도 급속히 주목받게 되었죠.

초기에 도구 사용은 프롬프트 기반이었습니다. "Action: search\nAction Input: LangChain"처럼 **텍스트 파싱**에 의존했기 때문에, 도구 설명을 잘 쓰는 것이 생사를 가를 정도로 중요했습니다. LLM이 잘못된 형식으로 출력하면 파싱 자체가 실패했거든요.

2023년 6월, OpenAI가 **Function Calling** 기능을 도입하면서 상황이 크게 개선되었습니다. LLM이 JSON 스키마에 맞춰 구조화된 도구 호출을 생성하게 되면서, 파싱 오류는 크게 줄었습니다. 하지만 여전히 **어떤 도구를 언제 사용할지** 판단하는 것은 도구 설명에 달려 있어서, 도구 설계의 중요성은 변하지 않았습니다.

LangChain은 이 흐름을 따라 `@tool` 데코레이터 → `StructuredTool` → `BaseTool`로 이어지는 도구 추상화 계층을 발전시켜 왔고, 현재는 Pydantic v2 기반의 엄격한 타입 검증까지 지원합니다.

### 도구 이름의 힘: 이름만 바꿔도 정확도가 달라진다

실제 연구에 따르면, 도구 이름을 `search`에서 `search_latest_news`로 바꾸기만 해도 에이전트의 도구 선택 정확도가 유의미하게 향상됩니다. LLM은 설명문보다 **도구 이름을 먼저** 처리하기 때문이죠. 이름이 직관적이면 설명을 다 읽지 않아도 대략적인 판단이 가능합니다.

## 흔한 오해와 팁

> ⚠️ **흔한 오해**: "도구 설명은 개발자 주석처럼 대충 써도 된다"  
> 절대 그렇지 않습니다! 도구 설명은 개발자가 아니라 **LLM이 읽는 문서**입니다. LLM은 코드 내부를 볼 수 없기 때문에, 설명문이 곧 그 도구의 전부입니다. 설명이 모호하면 LLM은 100% 잘못된 선택을 합니다. "언제 사용하고, 언제 사용하지 말아야 하는지"를 반드시 포함하세요.

> 💡 **알고 계셨나요?**: LangChain의 `@tool` 데코레이터는 함수의 **docstring**을 자동으로 `description`으로 사용합니다. 그래서 docstring을 잘 작성하는 것이 곧 도구 설명을 잘 작성하는 것이에요. 만약 docstring 없이 `@tool`을 사용하면, 함수명만으로 설명이 생성되는데 이는 거의 항상 부족합니다.

> 🔥 **실무 팁**: 에이전트가 도구를 잘못 선택한다면, **verbose=True로 추론 과정을 확인**하세요. `Thought` 부분에서 LLM이 왜 그 도구를 골랐는지 나옵니다. 대부분 도구 설명의 모호성이 원인인데요, 설명에 "사용하지 말아야 할 때(When NOT to use)" 문구를 추가하는 것만으로도 오선택이 크게 줄어듭니다.

> 🔥 **실무 팁**: Pydantic `Field`의 `description`에 **구체적인 예시**를 넣으세요. `"도시명"` 대신 `"도시명. 예: '서울', '부산', 'New York'"` 처럼 작성하면 LLM이 올바른 형식의 입력을 생성할 확률이 훨씬 높아집니다.

## 핵심 정리

| 개념 | 설명 |
|------|------|
| 도구 설명(Description) | LLM이 도구를 선택하는 유일한 근거. "무엇을/언제/언제 안 되는지" 포함 필수 |
| 도구 이름(Name) | 직관적이고 구체적일수록 좋음. `search` → `search_web_latest` |
| `args_schema` | Pydantic 모델로 입력 스키마를 명시적으로 정의. `Field(description=...)`가 핵심 |
| `return_direct` | `True`면 도구 결과를 바로 최종 응답으로 반환. 조회성 도구에 적합 |
| `handle_tool_error` | `True`면 ToolException을 Observation으로 전달하여 에이전트가 재시도 가능 |
| `BaseTool` 클래스 | 상태 관리, 비동기, 복잡한 초기화가 필요한 고급 도구에 사용 |
| 도구 조합 전략 | 역할이 겹치지 않는 소수의 도구가 최적. 8개 이상이면 동적 선택 고려 |
| 동적 도구 선택 | 맥락에 따라 도구 목록을 필터링하여 에이전트의 선택 정확도 향상 |

다음 [12.5: 에이전트 디버깅과 모니터링](./05-에이전트-디버깅과-모니터링.md)에서는 에이전트가 실행되는 과정을 **실시간으로 추적하고 문제를 진단하는 방법**을 배웁니다. 이번 섹션에서 `verbose=True`로 간단히 살펴본 추론 과정을, 콜백 핸들러와 LangSmith를 활용해 **체계적으로 모니터링**하는 방법으로 발전시킵니다. 잘 설계된 도구가 기대와 다르게 동작할 때, 원인을 빠르게 찾아내는 디버깅 기법도 함께 다룹니다.
