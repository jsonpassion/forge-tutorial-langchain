# 안전장치와 사용자 승인

> LangGraph의 `interrupt()`와 체크포인트를 활용하여 위험한 작업 전 사용자 승인을 요구하고, 실행 취소와 감사 로그로 안전한 자동화를 구현합니다.

## 개요

이 섹션에서는 [세션 19.4: 워크플로우 오케스트레이션](ch19/session4.md)에서 완성한 감독자 기반 멀티 에이전트 시스템에 **안전장치**를 추가합니다. 에이전트가 이메일을 보내거나 일정을 생성하는 등 되돌리기 어려운 작업을 수행하기 전에, 사람의 승인을 받도록 하는 메커니즘을 구축하는 거죠.

**선수 지식**:
- [세션 19.2](ch19/session2.md)에서 배운 외부 서비스 도구(`@tool`, `ToolException`)
- [세션 19.3](ch19/session3.md)에서 배운 전문 에이전트(`create_react_agent`)
- [세션 19.4](ch19/session4.md)에서 배운 `Command` 기반 라우팅과 감독자 오케스트레이션
- [세션 13장](ch13/)에서 배운 LangGraph `StateGraph`, 체크포인터 기초

**학습 목표**:
- `interrupt()` 함수로 그래프 실행을 일시 중지하고 `Command(resume=...)`로 재개할 수 있다
- 도구의 위험도에 따라 자동 실행 / 승인 필요 / 차단을 분류하는 정책을 설계할 수 있다
- 체크포인트 기반 시간 여행(Time Travel)으로 실행 취소 기능을 구현할 수 있다
- 에이전트의 도구 호출과 승인 결정을 추적하는 도구 실행 감사 로그를 구축할 수 있다

## 왜 알아야 할까?

잠깐 상상해 보세요. 여러분이 만든 업무 자동화 에이전트가 새벽 3시에 고객 1,000명에게 잘못된 이메일을 보냈다면요? 또는 중요한 데이터베이스 레코드를 삭제해버렸다면요?

AI 에이전트는 강력하지만, **사람의 감독 없이 외부에 영향을 미치는 작업을 수행하면 치명적인 결과**를 초래할 수 있습니다. 실제로 2024년 한 기업에서 자동화 봇이 승인 없이 거래를 실행하여 수백만 달러의 손실이 발생한 사례도 있었죠.

프로덕션 환경에서 에이전트를 운영하려면 세 가지가 필수입니다:

1. **사전 승인**: 위험한 작업 전에 사람이 확인하고 승인
2. **실행 취소**: 실수를 발견했을 때 이전 상태로 되돌리기
3. **감사 추적**: 누가, 언제, 어떤 도구를 호출했고 누가 승인했는지 기록

LangGraph는 이 세 가지를 우아하게 해결하는 `interrupt()` 함수와 체크포인트 시스템을 제공합니다. [세션 19.4](ch19/session4.md)에서 만든 오케스트레이션 그래프에 이 안전장치를 통합하면, **신뢰할 수 있는 프로덕션 수준의 자동화 시스템**이 완성됩니다.

## 핵심 개념

### 개념 1: `interrupt()` — 에이전트에게 "잠깐!" 외치기

> 💡 **비유**: `interrupt()`는 공장의 **비상 정지 버튼**과 같습니다. 컨베이어 벨트(그래프 실행)가 돌아가다가 품질 검사원(사람)이 확인해야 할 제품이 나오면, 벨트가 멈추고 검사원의 신호를 기다립니다. 검사원이 "OK"를 누르면 벨트가 다시 움직이고, "NG"를 누르면 해당 제품은 폐기 라인으로 보내지죠.

LangGraph의 `interrupt()` 함수는 노드 실행 도중 **그래프 전체를 일시 정지**시킵니다. 이때 현재 상태가 체크포인터에 저장되고, 외부(사람 또는 다른 시스템)의 입력을 기다립니다. 입력이 들어오면 `Command(resume=값)`으로 실행을 재개하며, 그 값이 `interrupt()` 호출의 반환값이 됩니다.

```python
from langgraph.types import interrupt, Command
from langgraph.graph import StateGraph, START, END
from langgraph.checkpoint.memory import MemorySaver
from typing import TypedDict, Optional, Literal


class ApprovalState(TypedDict):
    action: str               # 수행할 작업 설명
    status: Optional[str]     # 승인 상태: pending / approved / rejected


def request_approval(state: ApprovalState) -> Command[Literal["execute", "cancel"]]:
    """사용자 승인을 요청하고, 결과에 따라 라우팅합니다."""
    # interrupt()에 전달하는 값은 호출자에게 표시됩니다
    decision = interrupt({
        "question": "이 작업을 승인하시겠습니까?",
        "action": state["action"],
    })

    # decision이 True이면 실행, False이면 취소
    if decision:
        return Command(goto="execute", update={"status": "approved"})
    else:
        return Command(goto="cancel", update={"status": "rejected"})


def execute_action(state: ApprovalState):
    """승인된 작업을 실행합니다."""
    print(f"✅ 작업 실행: {state['action']}")
    return {"status": "completed"}


def cancel_action(state: ApprovalState):
    """거부된 작업을 취소합니다."""
    print(f"❌ 작업 취소: {state['action']}")
    return {"status": "cancelled"}


# 그래프 구성
builder = StateGraph(ApprovalState)
builder.add_node("approve", request_approval)
builder.add_node("execute", execute_action)
builder.add_node("cancel", cancel_action)
builder.add_edge(START, "approve")
builder.add_edge("execute", END)
builder.add_edge("cancel", END)

# 체크포인터 필수! interrupt() 사용의 전제 조건
graph = builder.compile(checkpointer=MemorySaver())

# --- 실행 ---
config = {"configurable": {"thread_id": "approval-001"}}

# 1단계: 그래프 실행 → interrupt에서 일시 정지
result = graph.invoke(
    {"action": "고객 전체에게 프로모션 이메일 발송", "status": "pending"},
    config=config,
)
print(result)
# 출력: {'__interrupt__': [Interrupt(value={'question': '이 작업을 승인하시겠습니까?', 'action': '고객 전체에게 프로모션 이메일 발송'})]}

# 2단계: 사용자가 승인 → Command(resume=True)로 재개
final = graph.invoke(Command(resume=True), config=config)
print(final)
# 출력: {'action': '고객 전체에게 프로모션 이메일 발송', 'status': 'completed'}
```

핵심 포인트를 정리하면:

- `interrupt(payload)`: 그래프를 멈추고, `payload`를 호출자에게 전달합니다
- `Command(resume=value)`: 같은 `thread_id`로 그래프를 재개하며, `value`가 `interrupt()`의 반환값이 됩니다
- 체크포인터(`MemorySaver` 등)가 **반드시** 필요합니다 — 상태를 저장해야 재개가 가능하거든요

### 개념 2: 위험도 기반 승인 정책

> 💡 **비유**: 병원의 **약품 분류 시스템**을 떠올려 보세요. 일반의약품(비타민)은 간호사가 바로 투여하고, 전문의약품(항생제)은 의사 확인 후 투여하며, 마약성 진통제는 2인 이상의 승인이 필요합니다. 약의 **위험도**에 따라 승인 절차가 달라지는 거죠.

모든 도구 호출에 사용자 승인을 요구하면 생산성이 크게 떨어집니다. 반대로 모든 것을 자동 실행하면 위험하죠. 해답은 **위험도에 따라 승인 수준을 차등 적용**하는 것입니다.

```python
from enum import Enum
from dataclasses import dataclass, field
from typing import Callable


class RiskLevel(Enum):
    """도구 위험도 등급"""
    LOW = "low"          # 자동 실행 (읽기 전용 작업)
    MEDIUM = "medium"    # 사용자 승인 필요 (수정 작업)
    HIGH = "high"        # 상세 검토 + 승인 필요 (외부 영향)
    CRITICAL = "critical"  # 이중 승인 또는 차단 (비가역적 작업)


@dataclass
class ToolPolicy:
    """도구별 실행 정책을 정의합니다."""
    tool_name: str
    risk_level: RiskLevel
    requires_reason: bool = False        # 승인 시 사유 입력 필요
    max_auto_executions: int = -1        # 자동 실행 횟수 제한 (-1: 무제한)
    allowed_hours: tuple[int, int] = (0, 24)  # 허용 실행 시간대
    description: str = ""


# 도구별 정책 정의
TOOL_POLICIES: dict[str, ToolPolicy] = {
    # LOW: 자동 실행 — 데이터를 읽기만 하는 작업
    "search_emails": ToolPolicy(
        tool_name="search_emails",
        risk_level=RiskLevel.LOW,
        description="이메일 검색 (읽기 전용)",
    ),
    "query_database": ToolPolicy(
        tool_name="query_database",
        risk_level=RiskLevel.LOW,
        description="DB 조회 (SELECT만 허용)",
    ),
    # MEDIUM: 승인 필요 — 내부 데이터 수정
    "create_calendar_event": ToolPolicy(
        tool_name="create_calendar_event",
        risk_level=RiskLevel.MEDIUM,
        description="캘린더 일정 생성",
    ),
    "update_database": ToolPolicy(
        tool_name="update_database",
        risk_level=RiskLevel.MEDIUM,
        requires_reason=True,
        description="DB 레코드 수정",
    ),
    # HIGH: 상세 검토 — 외부에 영향을 미치는 작업
    "send_email": ToolPolicy(
        tool_name="send_email",
        risk_level=RiskLevel.HIGH,
        requires_reason=True,
        max_auto_executions=0,   # 자동 실행 불가
        description="이메일 발송",
    ),
    # CRITICAL: 차단 또는 이중 승인
    "delete_database_records": ToolPolicy(
        tool_name="delete_database_records",
        risk_level=RiskLevel.CRITICAL,
        requires_reason=True,
        allowed_hours=(9, 18),   # 업무 시간에만 허용
        description="DB 레코드 삭제 (비가역적)",
    ),
}
```

이 정책을 `interrupt()`와 결합하면 위험도에 따라 다르게 동작하는 승인 노드를 만들 수 있습니다:

```python
from datetime import datetime
from langgraph.types import interrupt, Command


def policy_enforcer(state: dict) -> Command:
    """위험도 기반으로 도구 실행 정책을 적용합니다."""
    tool_name = state["pending_tool"]["name"]
    tool_args = state["pending_tool"]["args"]
    policy = TOOL_POLICIES.get(tool_name)

    if policy is None:
        # 정책 미등록 도구 → 기본적으로 차단
        interrupt({"error": f"미등록 도구: {tool_name}", "action": "blocked"})
        return Command(goto="cancel")

    # LOW 위험도: 자동 실행
    if policy.risk_level == RiskLevel.LOW:
        return Command(goto="execute_tool")

    # 시간대 제한 확인
    current_hour = datetime.now().hour
    if not (policy.allowed_hours[0] <= current_hour < policy.allowed_hours[1]):
        return Command(
            goto="cancel",
            update={"error": f"허용 시간대 외 실행 시도: {policy.allowed_hours}"}
        )

    # MEDIUM 이상: 사용자 승인 요청
    approval_request = {
        "tool": tool_name,
        "args": tool_args,
        "risk_level": policy.risk_level.value,
        "description": policy.description,
        "requires_reason": policy.requires_reason,
    }

    # CRITICAL: 경고 메시지 추가
    if policy.risk_level == RiskLevel.CRITICAL:
        approval_request["warning"] = "⚠️ 이 작업은 되돌릴 수 없습니다!"

    # interrupt로 사용자 입력 대기
    response = interrupt(approval_request)

    # 응답 처리
    if isinstance(response, dict):
        approved = response.get("approved", False)
        reason = response.get("reason", "")
    else:
        approved = bool(response)
        reason = ""

    if approved:
        if policy.requires_reason and not reason:
            # 사유 없이 승인 시도 → 재요청
            response = interrupt({
                **approval_request,
                "message": "이 작업은 승인 사유가 필요합니다.",
            })
            approved = response.get("approved", False)
            reason = response.get("reason", "")

        return Command(
            goto="execute_tool",
            update={"approval_reason": reason, "status": "approved"}
        )
    else:
        return Command(goto="cancel", update={"status": "rejected"})
```

### 개념 3: 도구 내부 `interrupt()` — 승인이 필요한 도구 만들기

> 💡 **비유**: ATM에서 고액 출금을 할 때 "정말 100만 원을 인출하시겠습니까?"라는 확인 화면이 뜨는 것처럼, 도구 자체가 위험한 작업 전에 사용자에게 확인을 요청하는 패턴입니다.

[세션 19.2](ch19/session2.md)에서 만든 도구에 직접 `interrupt()`를 넣으면, 해당 도구가 호출될 때마다 자동으로 승인 과정을 거치게 됩니다:

```python
from langchain_core.tools import tool
from langgraph.types import interrupt


@tool
def send_email_safe(to: str, subject: str, body: str) -> str:
    """이메일을 발송합니다. 발송 전 사용자 승인을 요청합니다."""

    # 발송 전 사용자에게 내용 확인 요청
    response = interrupt({
        "action": "send_email",
        "preview": {
            "to": to,
            "subject": subject,
            "body": body[:200] + ("..." if len(body) > 200 else ""),
        },
        "message": "이 이메일을 발송하시겠습니까? 수정이 필요하면 내용을 함께 보내주세요.",
    })

    # 사용자가 내용을 수정했을 수 있음
    if isinstance(response, dict):
        if not response.get("approved", False):
            return "❌ 이메일 발송이 취소되었습니다."
        # 수정된 내용 반영
        to = response.get("to", to)
        subject = response.get("subject", subject)
        body = response.get("body", body)
    elif not response:
        return "❌ 이메일 발송이 취소되었습니다."

    # 실제 발송 로직 (시뮬레이션)
    print(f"📧 이메일 발송: {to} / {subject}")
    return f"✅ '{subject}' 이메일이 {to}에게 발송되었습니다."


@tool
def create_event_safe(
    title: str, date: str, time: str, attendees: list[str]
) -> str:
    """캘린더 일정을 생성합니다. 생성 전 사용자 승인을 요청합니다."""

    response = interrupt({
        "action": "create_event",
        "preview": {
            "title": title,
            "date": date,
            "time": time,
            "attendees": attendees,
        },
        "message": "이 일정을 생성하시겠습니까?",
    })

    if not response or (isinstance(response, dict) and not response.get("approved")):
        return "❌ 일정 생성이 취소되었습니다."

    print(f"📅 일정 생성: {title} ({date} {time})")
    return f"✅ '{title}' 일정이 {date} {time}에 생성되었습니다."
```

### 개념 4: 체크포인트 기반 실행 취소(Undo)

> 💡 **비유**: 워드프로세서의 **Ctrl+Z(실행 취소)** 기능을 떠올려 보세요. 문서를 편집할 때마다 자동으로 스냅샷이 저장되고, 실수를 하면 이전 상태로 돌아갈 수 있죠. LangGraph의 체크포인터도 마찬가지로, 그래프가 한 단계 실행될 때마다 상태 스냅샷을 저장하여 "시간 여행"을 가능하게 합니다.

LangGraph는 각 노드 실행 후 상태를 체크포인트로 저장합니다. `get_state_history()`로 과거 상태를 조회하고, 특정 체크포인트에서 실행을 재개할 수 있습니다:

```python
from langgraph.checkpoint.memory import MemorySaver


def undo_last_action(graph, config: dict) -> dict | None:
    """마지막 작업을 실행 취소하고 이전 상태로 되돌립니다."""

    # 상태 이력 조회
    history = list(graph.get_state_history(config))

    if len(history) < 2:
        print("되돌릴 상태가 없습니다.")
        return None

    # 현재 상태 (history[0])의 바로 이전 상태
    previous_state = history[1]
    checkpoint_id = previous_state.config["configurable"]["checkpoint_id"]

    print(f"🔙 체크포인트 '{checkpoint_id}'로 되돌립니다.")
    print(f"   되돌리기 전 상태: {history[0].values}")
    print(f"   되돌린 후 상태:   {previous_state.values}")

    # 이전 체크포인트에서 실행 재개 (새로운 분기 생성)
    rollback_config = {
        "configurable": {
            "thread_id": config["configurable"]["thread_id"],
            "checkpoint_id": checkpoint_id,
        }
    }

    # None을 입력하면 해당 체크포인트에서 그래프가 이어서 실행
    result = graph.invoke(None, rollback_config)
    return result


def show_execution_history(graph, config: dict):
    """실행 이력을 시각적으로 보여줍니다."""
    history = list(graph.get_state_history(config))

    print(f"\n📜 실행 이력 (총 {len(history)}개 체크포인트)")
    print("=" * 60)

    for i, state in enumerate(history):
        checkpoint_id = state.config["configurable"]["checkpoint_id"]
        node = state.metadata.get("source", "unknown")
        step = state.metadata.get("step", "?")
        marker = "👉 " if i == 0 else "   "

        print(f"{marker}[Step {step}] {node}")
        print(f"      체크포인트: {checkpoint_id[:16]}...")

        # 주요 상태 값 표시
        for key, value in state.values.items():
            if key.startswith("_"):
                continue
            display_value = str(value)[:50]
            print(f"      {key}: {display_value}")
        print()
```

### 개념 5: 도구 실행 감사 로그(Tool Execution Audit Log)

> 💡 **비유**: 은행 창구에서 큰 금액을 송금할 때 창구 직원이 **거래 전표**를 작성하는 것과 같습니다. "언제, 어떤 고객이, 어떤 거래를, 누구의 승인으로 실행했는지"를 기록하죠. 나중에 이상 거래가 발생하면 이 전표를 추적해서 원인을 파악합니다. 에이전트의 도구 실행 감사 로그도 마찬가지입니다.

에이전트가 어떤 도구를 호출했고, 승인/거부 결정이 어떻게 이루어졌으며, 결과는 어땠는지를 체계적으로 기록해야 합니다. 여기서 다루는 감사 로그는 **에이전트의 도구 호출과 승인 결정 흐름**에 특화된 추적 시스템입니다. LLM 호출 자체의 입출력 감사나 규정 준수 프레임워크와는 관심사가 다릅니다:

| 관심사 | 이 세션 (Ch19.5) | [세션 20.1: 프로덕션 보안](ch20/session1.md) |
|--------|-------------------|----------------------------------------------|
| **추적 대상** | 도구 호출 생명주기 (요청→정책→승인→실행→결과) | LLM 호출 입출력, 전체 애플리케이션 보안 |
| **핵심 질문** | "누가 이 도구를 호출했고, 누가 승인했나?" | "LLM에 어떤 데이터가 노출되었나?" |
| **구현 방식** | 워크플로우 노드 내 직접 기록 | `BaseCallbackHandler` 기반 이벤트 훅 |
| **활용 목적** | 도구 실행 추적, 승인 체인 감사 | OWASP LLM Top 10, GDPR/SOC 2 규정 준수 |

```python
from datetime import datetime, timezone
from dataclasses import dataclass, field
from enum import Enum


class ToolAuditAction(Enum):
    """도구 실행 관련 감사 이벤트 유형"""
    TOOL_REQUESTED = "tool_requested"        # 에이전트가 도구 호출 시도
    AUTO_APPROVED = "auto_approved"          # 정책에 의해 자동 승인 (LOW 위험도)
    APPROVAL_REQUESTED = "approval_requested"  # 사용자 승인 요청됨
    USER_APPROVED = "user_approved"          # 사용자가 승인
    USER_DENIED = "user_denied"              # 사용자가 거부
    EXECUTION_SUCCESS = "execution_success"  # 도구 실행 성공
    EXECUTION_FAILURE = "execution_failure"  # 도구 실행 실패
    POLICY_BLOCKED = "policy_blocked"        # 정책에 의해 차단
    ROLLBACK = "rollback"                    # 실행 취소 수행


@dataclass
class ToolAuditEntry:
    """도구 실행 감사 로그의 단일 항목"""
    timestamp: str              # ISO 8601 타임스탬프
    action: str                 # ToolAuditAction 값
    agent_name: str             # 도구를 호출한 에이전트
    tool_name: str = ""         # 호출된 도구 이름
    tool_args: dict = field(default_factory=dict)  # 도구 인자
    risk_level: str = ""        # 위험도 등급
    decision: str = ""          # approved / rejected / auto / blocked
    decided_by: str = ""        # user / policy / system
    reason: str = ""            # 승인/거부 사유
    result_summary: str = ""    # 실행 결과 요약 (전체 결과가 아닌 요약)
    thread_id: str = ""         # LangGraph 스레드 ID
    checkpoint_id: str = ""     # 관련 체크포인트 ID


class ToolAuditLogger:
    """에이전트 도구 호출의 승인/실행을 추적하는 감사 로그 시스템.

    이 클래스는 LangGraph 워크플로우 내에서 도구 실행의 전체 생명주기
    (요청 → 정책 확인 → 승인/거부 → 실행 → 결과)를 기록합니다.
    각 도구 호출에 대해 "어떤 에이전트가 호출했고, 정책 판정이 어떻게
    되었으며, 누가 승인/거부했는지"를 추적하는 것이 핵심입니다.

    참고: LLM 호출 자체의 보안 감사(입출력 해싱, 민감 데이터 탐지,
    OWASP/GDPR/SOC 2 규정 준수)는 별도의 관심사이며,
    [세션 20.1: 프로덕션 보안](ch20/session1.md)에서 BaseCallbackHandler
    기반의 보안 감사 프레임워크로 다룹니다.
    """

    def __init__(self):
        self.entries: list[ToolAuditEntry] = []

    def log(
        self,
        action: ToolAuditAction,
        agent_name: str,
        tool_name: str = "",
        tool_args: dict | None = None,
        risk_level: str = "",
        decision: str = "",
        decided_by: str = "",
        reason: str = "",
        result_summary: str = "",
        thread_id: str = "",
        checkpoint_id: str = "",
    ) -> ToolAuditEntry:
        """도구 실행 감사 항목을 기록합니다."""
        entry = ToolAuditEntry(
            timestamp=datetime.now(timezone.utc).isoformat(),
            action=action.value,
            agent_name=agent_name,
            tool_name=tool_name,
            tool_args=tool_args or {},
            risk_level=risk_level,
            decision=decision,
            decided_by=decided_by,
            reason=reason,
            result_summary=result_summary,
            thread_id=thread_id,
            checkpoint_id=checkpoint_id,
        )
        self.entries.append(entry)
        return entry

    def get_tool_history(
        self,
        tool_name: str | None = None,
        agent_name: str | None = None,
        only_denied: bool = False,
    ) -> list[ToolAuditEntry]:
        """특정 도구 또는 에이전트의 실행 이력을 조회합니다."""
        filtered = self.entries
        if tool_name:
            filtered = [e for e in filtered if e.tool_name == tool_name]
        if agent_name:
            filtered = [e for e in filtered if e.agent_name == agent_name]
        if only_denied:
            filtered = [
                e for e in filtered
                if e.action in (ToolAuditAction.USER_DENIED.value,
                                ToolAuditAction.POLICY_BLOCKED.value)
            ]
        return filtered

    def get_approval_chain(self, thread_id: str) -> list[ToolAuditEntry]:
        """특정 스레드의 승인 체인(요청→판정→결과)을 시간순으로 반환합니다.

        하나의 워크플로우 실행에서 도구들이 어떤 순서로 승인/거부되었는지
        전체 흐름을 파악할 때 유용합니다.
        """
        approval_actions = {
            ToolAuditAction.APPROVAL_REQUESTED.value,
            ToolAuditAction.USER_APPROVED.value,
            ToolAuditAction.USER_DENIED.value,
            ToolAuditAction.AUTO_APPROVED.value,
            ToolAuditAction.POLICY_BLOCKED.value,
        }
        return [
            e for e in self.entries
            if e.thread_id == thread_id and e.action in approval_actions
        ]

    def get_approval_stats(self) -> dict:
        """승인/거부 통계를 반환합니다."""
        total_requests = sum(
            1 for e in self.entries
            if e.action == ToolAuditAction.APPROVAL_REQUESTED.value
        )
        approved = sum(
            1 for e in self.entries
            if e.action == ToolAuditAction.USER_APPROVED.value
        )
        denied = sum(
            1 for e in self.entries
            if e.action == ToolAuditAction.USER_DENIED.value
        )
        auto = sum(
            1 for e in self.entries
            if e.action == ToolAuditAction.AUTO_APPROVED.value
        )
        blocked = sum(
            1 for e in self.entries
            if e.action == ToolAuditAction.POLICY_BLOCKED.value
        )

        return {
            "total_tool_calls": len([
                e for e in self.entries
                if e.action in (ToolAuditAction.TOOL_REQUESTED.value,
                                ToolAuditAction.AUTO_APPROVED.value)
            ]),
            "approval_requests": total_requests,
            "approved": approved,
            "denied": denied,
            "auto_approved": auto,
            "policy_blocked": blocked,
            "deny_rate": f"{denied / total_requests * 100:.1f}%"
                if total_requests > 0 else "N/A",
        }
```

이 감사 로거는 도구 호출의 **전체 생명주기**를 추적하는 데 집중합니다. 도구가 요청된 순간부터 정책 확인, 사용자 승인, 실행 결과까지의 흐름을 일관된 타임라인으로 기록하죠. `get_approval_chain()`으로 하나의 워크플로우에서 발생한 모든 승인 결정을 시간순으로 추적할 수 있어, "왜 이 이메일이 발송되었지?", "이 도구 호출을 누가 승인했지?" 같은 질문에 정확히 답할 수 있습니다.

> 🔥 **실무 팁**: 이 세션의 감사 로그는 **도구 실행의 승인 흐름 추적**에 집중합니다. "어떤 에이전트가 어떤 도구를 호출했고, 누가 승인했으며, 결과가 어땠는가"를 기록하는 거죠. 프로덕션 환경에서 추가로 필요한 **LLM 호출 수준의 보안 감사** — 입출력 해싱, 민감 데이터 탐지, OWASP LLM Top 10 대응, GDPR/SOC 2 규정 준수 체계 — 는 [세션 20.1: 프로덕션 보안](ch20/session1.md)에서 `BaseCallbackHandler` 기반의 보안 감사 프레임워크로 체계적으로 다룹니다. 두 레이어를 함께 적용하면 도구 실행 추적(이 세션)과 LLM 보안 감사(Ch20.1)가 상호 보완됩니다.

### 개념 6: 안전장치 통합 — 승인 + 감사가 결합된 노드

앞서 배운 세 가지(승인, 실행 취소, 감사 로그)를 하나의 통합 노드로 결합합니다:

```python
from langgraph.types import interrupt, Command


# 전역 도구 감사 로거
audit = ToolAuditLogger()


def safe_tool_executor(state: dict) -> Command:
    """안전장치가 통합된 도구 실행 노드.

    정책 확인 → (필요시) 승인 요청 → 실행 → 감사 기록
    """
    tool_call = state["pending_tool"]
    tool_name = tool_call["name"]
    tool_args = tool_call["args"]
    agent_name = state.get("current_agent", "unknown")
    thread_id = state.get("thread_id", "")

    policy = TOOL_POLICIES.get(tool_name)

    # 1. 정책 확인
    if policy is None:
        audit.log(
            ToolAuditAction.POLICY_BLOCKED,
            agent_name=agent_name,
            tool_name=tool_name,
            tool_args=tool_args,
            decision="blocked",
            decided_by="policy",
            result_summary="미등록 도구 차단",
            thread_id=thread_id,
        )
        return Command(goto="cancel", update={"error": f"미등록 도구: {tool_name}"})

    # 2. LOW 위험도: 자동 실행 + 감사 기록
    if policy.risk_level == RiskLevel.LOW:
        audit.log(
            ToolAuditAction.AUTO_APPROVED,
            agent_name=agent_name,
            tool_name=tool_name,
            tool_args=tool_args,
            risk_level="low",
            decision="auto",
            decided_by="policy",
            thread_id=thread_id,
        )
        return Command(goto="execute_tool")

    # 3. MEDIUM 이상: 승인 요청
    audit.log(
        ToolAuditAction.APPROVAL_REQUESTED,
        agent_name=agent_name,
        tool_name=tool_name,
        tool_args=tool_args,
        risk_level=policy.risk_level.value,
        thread_id=thread_id,
    )

    response = interrupt({
        "type": "approval_required",
        "tool": tool_name,
        "args": tool_args,
        "risk_level": policy.risk_level.value,
        "description": policy.description,
        "warning": "⚠️ 비가역적 작업입니다!" if policy.risk_level == RiskLevel.CRITICAL else None,
    })

    approved = response.get("approved", False) if isinstance(response, dict) else bool(response)
    reason = response.get("reason", "") if isinstance(response, dict) else ""

    # 4. 승인/거부 기록
    if approved:
        audit.log(
            ToolAuditAction.USER_APPROVED,
            agent_name=agent_name,
            tool_name=tool_name,
            risk_level=policy.risk_level.value,
            decision="approved",
            decided_by="user",
            reason=reason,
            thread_id=thread_id,
        )
        return Command(goto="execute_tool", update={"status": "approved"})
    else:
        audit.log(
            ToolAuditAction.USER_DENIED,
            agent_name=agent_name,
            tool_name=tool_name,
            risk_level=policy.risk_level.value,
            decision="rejected",
            decided_by="user",
            reason=reason,
            thread_id=thread_id,
        )
        return Command(goto="cancel", update={"status": "rejected"})
```

## 실습: 직접 해보기

지금까지 배운 모든 개념을 통합하여, 완전한 안전장치가 적용된 업무 자동화 에이전트를 구축합니다.

```python
"""
안전장치가 통합된 업무 자동화 에이전트 (완전한 예제)
- 위험도 기반 승인 정책

- 도구 실행 감사 로그
"""

from datetime import datetime, timezone
from enum import Enum

from langchain_core.tools import tool
from langchain_openai import ChatOpenAI
from langgraph.graph import StateGraph, START, END
from langgraph.checkpoint.memory import MemorySaver
from langgraph.types import interrupt, Command


# ============================================================
# 1. 위험도 등급 및 정책
# ============================================================

class RiskLevel(Enum):
    LOW = "low"
    MEDIUM = "medium"
    HIGH = "high"
    CRITICAL = "critical"

    "search_emails": RiskLevel.LOW,

# ============================================================
# 2. 도구 실행 감사 로거
# ============================================================

class ToolAuditLog:
    """에이전트 도구 호출의 승인/실행을 추적하는 감사 로거.

    도구 호출 → 정책 판정 → 승인/거부 → 실행 결과의
    전체 생명주기를 메모리에 기록합니다.

    LLM 호출 수준의 보안 감사(입출력 해싱, 민감 데이터 탐지 등)는
    별도의 관심사이며 Ch20.1에서 다룹니다.
    """

    def __init__(self):
        self.logs: list[dict] = []

    def log(self, action: str, **kwargs):
        """감사 이벤트를 기록합니다."""
        entry = {
            "timestamp": datetime.now(timezone.utc).isoformat(),
            "action": action,
            **kwargs,
        }
        self.logs.append(entry)
        # 콘솔에 간략 출력
        risk = kwargs.get("risk_level", "")
        tool_name = kwargs.get("tool", "")
        print(f"  📋 [{action}] {tool_name} (risk={risk})")

    def summary(self) -> str:
        """감사 로그 요약을 반환합니다."""
        lines = [f"총 {len(self.logs)}건의 도구 감사 기록:"]
        for log in self.logs:
            lines.append(
                f"  - {log['timestamp'][:19]} | {log['action']} | {log.get('tool', '')}"
            )
        return "\n".join(lines)

    def approval_chain(self) -> list[dict]:
        """승인 체인만 추출합니다 — 도구별 승인/거부 흐름을 추적합니다."""
        chain_actions = {
            "auto_approved", "user_approved", "user_approved_with_edits",
            "user_denied", "user_denied_second", "user_double_approved",
            "policy_blocked",
        }
        return [log for log in self.logs if log["action"] in chain_actions]

    def denied_tools(self) -> list[dict]:
        """거부된 도구 호출만 필터링합니다."""
        return [
            log for log in self.logs
            if log["action"] in ("user_denied", "user_denied_second", "policy_blocked")
        ]


audit = ToolAuditLog()

class SafeWorkflowState(TypedDict):
    """안전장치가 통합된 워크플로우 상태"""
    task: str                                          # 사용자 요청

# 5. 그래프 노드
# ============================================================

def planner(state: SafeWorkflowState) -> dict:
    """작업을 분석하고 실행 계획을 생성합니다."""
    task = state["task"]

    # 시뮬레이션: 실제로는 LLM이 계획을 생성
    if "이메일" in task and "보내" in task:
        plan = [
            {"tool": "search_emails", "args": {"query": "관련 내용"}},
            {"tool": "send_email", "args": {
                "to": "team@company.com",
                "subject": "주간 보고서",
                "body": "이번 주 업무 보고서입니다.",
            }},
        ]
    elif "일정" in task:
        plan = [
            {"tool": "create_event", "args": {
                "title": "팀 미팅",
                "date": "2026-03-10",
            }},
        ]
    else:
        plan = [
            {"tool": "search_emails", "args": {"query": task}},
        ]

    print(f"\n📝 실행 계획 ({len(plan)}단계):")
    for i, step in enumerate(plan, 1):
        risk = RISK_POLICIES.get(step["tool"], RiskLevel.LOW).value
        print(f"   {i}. {step['tool']} (위험도: {risk})")

    return {"plan": plan, "current_step": 0, "status": "planning"}


def safety_gate(state: SafeWorkflowState) -> Command:
    """위험도 기반 안전 게이트: 정책에 따라 실행/승인/차단을 결정합니다."""
    plan = state["plan"]
    step_idx = state["current_step"]

    # 모든 단계 완료 확인
    if step_idx >= len(plan):
        return Command(goto="finalize")

    step = plan[step_idx]
    tool_name = step["tool"]
    tool_args = step["args"]
    risk = RISK_POLICIES.get(tool_name, RiskLevel.HIGH)

    print(f"\n🔒 안전 게이트 — {tool_name} (위험도: {risk.value})")

    # LOW: 자동 통과
    if risk == RiskLevel.LOW:
        audit.log("auto_approved", tool=tool_name, risk_level=risk.value)
        return Command(goto="execute")

    # MEDIUM: 간단 승인
    if risk == RiskLevel.MEDIUM:
        decision = interrupt({
            "type": "approval",
            "tool": tool_name,
            "args": tool_args,
            "risk": risk.value,
            "message": f"'{tool_name}' 실행을 승인하시겠습니까?",
        })
        if decision:
            audit.log("user_approved", tool=tool_name, risk_level=risk.value)
            return Command(goto="execute")
        else:
            audit.log("user_denied", tool=tool_name, risk_level=risk.value)
            return Command(
                goto="skip_step",
                update={"results": [f"⏭️ {tool_name}: 사용자가 거부"]},
            )

    # HIGH: 상세 검토 + 승인
    if risk == RiskLevel.HIGH:
        decision = interrupt({
            "type": "detailed_review",
            "tool": tool_name,
            "args": tool_args,
            "risk": risk.value,
            "message": "상세 내용을 확인하고 승인/수정/거부를 선택하세요.",
            "editable_fields": list(tool_args.keys()),
        })

        if isinstance(decision, dict) and decision.get("approved"):
            # 수정된 인자 반영
            updated_args = {k: decision.get(k, v) for k, v in tool_args.items()}
            plan[step_idx]["args"] = updated_args
            audit.log("user_approved_with_edits", tool=tool_name, risk_level=risk.value)
            return Command(goto="execute", update={"plan": plan})
        else:
            audit.log("user_denied", tool=tool_name, risk_level=risk.value)
            return Command(
                goto="skip_step",
                update={"results": [f"⏭️ {tool_name}: 사용자가 거부"]},
            )

    # CRITICAL: 이중 확인
    first = interrupt({
        "type": "critical_warning",
        "tool": tool_name,
        "args": tool_args,
        "risk": "critical",
        "message": "⚠️ 비가역적 작업입니다. 정말 실행하시겠습니까?",
    })
    if not first:
        audit.log("user_denied", tool=tool_name, risk_level="critical")
        return Command(
            goto="skip_step",
            update={"results": [f"🚫 {tool_name}: 사용자가 거부 (CRITICAL)"]},
        )

    second = interrupt({
        "type": "double_confirm",
        "tool": tool_name,
        "message": "최종 확인: 이 작업은 되돌릴 수 없습니다. 계속하시겠습니까?",
    })
    if second:
        audit.log("user_double_approved", tool=tool_name, risk_level="critical")
        return Command(goto="execute")
    else:
        audit.log("user_denied_second", tool=tool_name, risk_level="critical")
        return Command(
            goto="skip_step",
            update={"results": [f"🚫 {tool_name}: 이중 확인에서 거부"]},
        )


def execute_step(state: SafeWorkflowState) -> dict:
    """승인된 도구를 실행합니다."""
    step = state["plan"][state["current_step"]]
    tool_fn = ALL_TOOLS[step["tool"]]

    try:
        result = tool_fn.invoke(step["args"])
        audit.log(
            "execution_success",
            tool=step["tool"],
            result=str(result)[:100],
        )
        print(f"   ✅ 실행 결과: {result}")
        return {
            "results": [result],
            "current_step": state["current_step"] + 1,
        }
    except Exception as e:
        audit.log(
            "execution_failure",
            tool=step["tool"],
            error=str(e),
        )
        return {
            "results": [f"❌ {step['tool']} 실행 실패: {e}"],
            "current_step": state["current_step"] + 1,
        }


def skip_step(state: SafeWorkflowState) -> dict:
    """거부된 단계를 건너뜁니다."""
    return {"current_step": state["current_step"] + 1}


def finalize(state: SafeWorkflowState) -> dict:
    """최종 결과를 정리합니다."""
    print("\n📊 실행 완료!")
    print(f"   처리된 단계: {state['current_step']}/{len(state['plan'])}")
    for r in state["results"]:
        print(f"   - {r}")
    print(f"\n{audit.summary()}")

    # 승인 체인 요약 — 도구별 승인/거부 흐름 추적
    chain = audit.approval_chain()
    if chain:
        print(f"\n🔗 승인 체인 ({len(chain)}건):")
        for c in chain:
            print(f"   - {c.get('tool', '?')}: {c['action']}")

    # 거부된 도구 호출이 있었다면 별도 표시
    denied = audit.denied_tools()
    if denied:
        print(f"\n⚠️ 거부된 도구 호출 {len(denied)}건:")
        for d in denied:
            print(f"   - {d.get('tool', '?')}: {d['action']}")

    return {"status": "completed"}


# ============================================================
# 6. 그래프 구성
# ============================================================

builder = StateGraph(SafeWorkflowState)

builder.add_node("planner", planner)
builder.add_node("safety_gate", safety_gate)
builder.add_node("execute", execute_step)
builder.add_node("skip_step", skip_step)
builder.add_node("finalize", finalize)

builder.add_edge(START, "planner")
builder.add_edge("planner", "safety_gate")
# execute와 skip_step 모두 다음 단계의 safety_gate로 복귀
builder.add_edge("execute", "safety_gate")
builder.add_edge("skip_step", "safety_gate")
builder.add_edge("finalize", END)

# 체크포인터 — 프로덕션에서는 PostgresSaver 또는 SqliteSaver 사용
checkpointer = MemorySaver()
safe_graph = builder.compile(checkpointer=checkpointer)


# ============================================================
# 7. 실행 예시
# ============================================================

def run_safe_workflow():
    """안전 워크플로우를 대화형으로 실행합니다."""
    config = {"configurable": {"thread_id": "safe-workflow-001"}}

    # 첫 실행: 계획 수립 → 첫 번째 안전 게이트
    result = safe_graph.invoke(
        {
            "task": "팀에게 주간 보고 이메일을 보내줘",
            "plan": [],
            "current_step": 0,
            "results": [],
            "status": "started",
        },
        config=config,
    )

    # interrupt가 있으면 사용자 입력을 받아 재개
    while "__interrupt__" in result and result["__interrupt__"]:
        interrupts = result["__interrupt__"]
        for intr in interrupts:
            print(f"\n🔔 승인 요청: {json.dumps(intr.value, ensure_ascii=False, indent=2)}")

        # 시뮬레이션: 실제로는 UI에서 사용자 입력을 받음
        user_input = input("\n승인하시겠습니까? (y/n): ").strip().lower()
        approved = user_input in ("y", "yes")

        # 승인/거부 결과로 재개
        result = safe_graph.invoke(Command(resume=approved), config=config)

    print("\n🎉 워크플로우가 완료되었습니다!")

    # 실행 이력 확인
    history = list(safe_graph.get_state_history(config))
    print(f"\n⏱️ 총 {len(history)}개의 체크포인트가 기록되었습니다.")


# 실행
if __name__ == "__main__":
    run_safe_workflow()
```

> 🔥 **실무 팁**: 위 코드를 실행하면 `search_emails`(LOW)는 자동으로 통과하고, `send_email`(HIGH)에서 사용자 승인을 요청합니다. `y`를 입력하면 이메일이 "발송"되고, `n`을 입력하면 건너뜁니다. 모든 과정이 감사 로그에 기록되며, `audit.approval_chain()`으로 승인 흐름만 따로 추적할 수 있습니다.

## 더 깊이 알아보기

### `interrupt()`의 탄생 이야기

LangGraph의 Human-in-the-Loop 기능은 처음부터 지금의 모습이 아니었습니다. 초기 버전(2024년)에서는 `interrupt_before`와 `interrupt_after`라는 **그래프 컴파일 옵션**으로 구현되었는데, 이는 "특정 노드 전/후에 무조건 멈추기"라는 단순한 방식이었습니다.

하지만 실제 프로덕션에서는 "이 도구는 승인이 필요하지만 저 도구는 자동 실행해야 한다", "위험도에 따라 다르게 처리해야 한다" 같은 복잡한 요구사항이 쏟아졌죠. 이에 LangChain 팀은 2024년 말 `interrupt()` **함수**를 도입했습니다. 노드 내부 어디서든 호출할 수 있고, 조건부로 사용할 수 있으며, 반환값으로 사용자 입력을 받을 수 있는 유연한 설계였습니다.

2025년 4월, LangGraph v0.4에서는 더 큰 진화가 이루어졌습니다. `invoke()`가 `__interrupt__` 키로 중단 정보를 직접 반환하게 되었고, 병렬 노드에서 발생한 여러 `interrupt()`를 한 번에 재개하는 **Resume Map** 패턴도 추가되었습니다. 각 interrupt에 고유 ID가 부여되어, 어떤 interrupt에 어떤 응답을 줄지 정확히 매핑할 수 있게 된 거죠.

이 진화 과정은 "단순한 도구 → 프로덕션 요구 → 유연한 프리미티브"라는 오픈소스 프로젝트의 전형적인 성장 패턴을 보여줍니다.

### 시간 여행(Time Travel)의 원리

LangGraph의 체크포인트 기반 실행 취소는 데이터베이스의 **트랜잭션 로그**와 같은 원리입니다. 각 노드 실행 후 상태 스냅샷을 저장하되, 이전 체크포인트에서 재개하면 **새로운 분기(fork)**를 생성합니다. 원래 실행 이력은 그대로 보존되기 때문에, "실행 취소"라고 하지만 실제로는 **기록을 지우지 않고 새 경로를 만드는** 안전한 방식입니다.

## 흔한 오해와 팁

> ⚠️ **흔한 오해**: "`interrupt()`를 쓰면 모든 도구 호출이 멈춘다"고 생각하기 쉽지만, `interrupt()`는 **호출한 지점에서만** 그래프를 멈춥니다. 조건문과 결합하면 특정 위험도의 도구만 선택적으로 멈출 수 있습니다. 반면 `interrupt_before=["tools"]` 같은 컴파일 옵션은 해당 노드를 거칠 때마다 무조건 멈추므로, 세밀한 제어가 필요하다면 `interrupt()` 함수를 사용하세요.

> 💡 **알고 계셨나요?**: LangGraph v0.4부터는 병렬 노드에서 여러 `interrupt()`가 동시에 발생해도 Resume Map으로 한 번에 처리할 수 있습니다. 각 interrupt에 고유 ID가 부여되어 `Command(resume={id1: value1, id2: value2})` 형태로 응답하면 되죠. 이전에는 병렬 interrupt 처리가 불가능해서 순차 실행으로 우회해야 했습니다.

> 🔥 **실무 팁**: 프로덕션에서는 `MemorySaver` 대신 `PostgresSaver`나 `SqliteSaver`를 사용하세요. `MemorySaver`는 프로세스가 종료되면 모든 체크포인트가 사라집니다. 또한 감사 로그는 반드시 **도구 실행 결과와 별도로** 저장해야 합니다. 도구 실행이 실패해도 "시도했다"는 기록은 남아야 하거든요.

> 🔥 **실무 팁**: 위험도 정책은 **코드가 아닌 설정 파일(YAML/JSON)**로 관리하는 것이 좋습니다. 새로운 도구가 추가되거나 정책이 변경될 때 코드 배포 없이 정책만 업데이트할 수 있거든요. 미등록 도구는 기본적으로 **차단(deny by default)**하는 것이 안전합니다.

## 핵심 정리

| 개념 | 설명 |
|------|------|
| `interrupt(payload)` | 그래프 실행을 일시 정지하고, payload를 호출자에게 전달 |
| `Command(resume=value)` | 정지된 그래프를 재개하며, value가 `interrupt()`의 반환값이 됨 |
| 체크포인터 | `interrupt()` 사용의 필수 전제. 상태를 저장하여 재개 가능하게 함 |
| 위험도 등급 | LOW(자동) → MEDIUM(승인) → HIGH(상세 검토) → CRITICAL(이중 확인) |
| `get_state_history()` | 실행 이력 조회. 시간 여행과 실행 취소의 기반 |
| 시간 여행(Time Travel) | 과거 체크포인트에서 실행을 재개하여 새로운 분기를 생성 |
| 도구 실행 감사 로그 | 에이전트의 도구 호출, 승인/거부, 실행 결과를 추적하는 기록. LLM 수준의 보안 감사는 [Ch20.1](ch20/session1.md) 참조 |
| Resume Map | 병렬 interrupt를 한 번에 처리. `Command(resume={id: value, ...})` |
| deny by default | 정책 미등록 도구는 기본적으로 차단하는 안전 원칙 |
