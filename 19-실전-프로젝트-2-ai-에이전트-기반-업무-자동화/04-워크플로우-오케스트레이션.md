# 워크플로우 오케스트레이션

> 감독자 노드가 전문 에이전트들을 지휘하고, 작업을 위임하고, 결과를 집계하는 오케스트레이션 시스템을 구축합니다.

## 개요

이 섹션에서는 앞서 구현한 전문 에이전트들을 하나의 통합 워크플로우로 엮는 **오케스트레이션(Orchestration)** 기법을 다룹니다. 감독자 노드가 사용자 요청을 분석하여 적절한 에이전트에게 작업을 위임하고, 완료된 결과를 다시 수거하여 최종 보고서를 만드는 전체 흐름을 구현합니다.

**선수 지식**: [19.1 에이전트 아키텍처 설계](./19-1.md)에서 배운 감독자 패턴과 상태 스키마, [19.2 외부 서비스 도구 구현](./19-2.md)의 커스텀 도구들, [19.3 전문 에이전트 구현](./19-3.md)의 `create_react_agent` 기반 전문 에이전트

**학습 목표**:
- `Command`를 활용한 감독자 노드의 라우팅 로직을 구현할 수 있다
- `langgraph-supervisor` 패키지로 오케스트레이션 그래프를 빠르게 구성할 수 있다
- 에이전트 간 핸드오프 메커니즘을 이해하고 커스터마이징할 수 있다
- 여러 에이전트의 결과를 집계하여 최종 보고서를 생성할 수 있다

## 왜 알아야 할까?

전문 에이전트 하나하나가 아무리 뛰어나도, 이들을 **조율하는 지휘자**가 없으면 혼란만 생깁니다. 실제 업무 자동화에서 "주간 보고서를 작성해줘"라는 요청 하나가 들어오면, 이메일 에이전트가 이번 주 수신 메일을 분석하고, 캘린더 에이전트가 일정을 조회하고, 데이터 에이전트가 매출 지표를 가져오고, 보고서 에이전트가 이 모든 결과를 종합해야 합니다.

이 과정에서 "누가 먼저 일할지", "결과를 어떻게 다음 에이전트에게 전달할지", "모든 작업이 끝났는지 어떻게 판단할지"를 결정하는 것이 바로 **워크플로우 오케스트레이션**입니다. 오케스트레이션이 제대로 되어 있지 않으면 에이전트들이 서로 같은 일을 반복하거나, 필요한 정보를 누락하거나, 무한 루프에 빠지게 됩니다.

## 핵심 개념

### 개념 1: 감독자 노드 — 오케스트라의 지휘자

> 💡 **비유**: 오케스트라 지휘자를 떠올려 보세요. 지휘자는 악기를 직접 연주하지 않지만, 악보(사용자 요청)를 읽고 "바이올린, 여기서 시작!", "첼로, 다음 마디에서 합류!" 하며 연주 순서를 결정합니다. 모든 파트가 끝나면 관객에게 완성된 곡을 들려주죠. 감독자 노드가 바로 이 지휘자입니다.

감독자 노드는 사용자 요청을 분석하여 **어떤 에이전트가 어떤 순서로 작업해야 하는지** 결정하는 핵심 컴포넌트입니다. LangGraph에서는 `Command` 객체를 반환하여 다음에 실행할 노드를 지정합니다.

[19.1 에이전트 아키텍처 설계](./19-1.md)에서 `add_conditional_edges`로 라우팅을 구성했었는데요, `Command`를 사용하면 **상태 업데이트와 라우팅을 한 번에** 처리할 수 있어 훨씬 깔끔합니다.

```python
from typing import Literal, TypedDict, Annotated
from langchain_openai import ChatOpenAI
from langchain_core.messages import SystemMessage, HumanMessage
from langgraph.graph import StateGraph, MessagesState, START, END
from langgraph.types import Command
import operator

# 감독자의 라우팅 결정을 구조화된 출력으로 받기
class SupervisorDecision(TypedDict):
    """감독자의 라우팅 결정"""
    next: Literal["email_agent", "calendar_agent", "data_agent", "report_agent", "FINISH"]
    reason: str  # 왜 이 에이전트를 선택했는지 (디버깅용)

# 감독자가 사용할 LLM
model = ChatOpenAI(model="gpt-4o", temperature=0)

# 구조화된 출력을 생성하는 LLM
router_llm = model.with_structured_output(SupervisorDecision)

MEMBERS = ["email_agent", "calendar_agent", "data_agent", "report_agent"]

SUPERVISOR_PROMPT = f"""당신은 업무 자동화 팀의 감독자입니다.
팀원: {MEMBERS}

각 팀원의 역할:
- email_agent: 이메일 검색, 분석, 요약
- calendar_agent: 일정 조회, 생성, 관리
- data_agent: 데이터베이스 쿼리, 매출/지표 분석
- report_agent: 수집된 정보를 바탕으로 보고서 작성

규칙:
1. 사용자 요청을 분석하여 필요한 팀원을 순서대로 호출하세요.
2. 정보 수집 에이전트(email, calendar, data)를 먼저 호출하고, report_agent는 마지막에 호출하세요.
3. 모든 작업이 완료되면 FINISH를 반환하세요.
4. 한 번에 하나의 팀원만 선택하세요.
"""

def supervisor_node(state: MessagesState) -> Command[Literal[*MEMBERS, "__end__"]]:
    """감독자 노드: 다음에 실행할 에이전트를 결정합니다."""
    messages = [
        SystemMessage(content=SUPERVISOR_PROMPT),
        *state["messages"]  # 지금까지의 대화 기록
    ]

    # LLM이 구조화된 출력으로 다음 에이전트를 선택
    decision = router_llm.invoke(messages)

    # FINISH면 종료, 아니면 해당 에이전트로 라우팅
    goto = END if decision["next"] == "FINISH" else decision["next"]

    return Command(
        goto=goto,
        update={"messages": [
            HumanMessage(
                content=f"[감독자 → {decision['next']}] 이유: {decision['reason']}",
                name="supervisor"
            )
        ]}
    )
```

핵심은 `Command`의 두 가지 역할입니다:
- **`goto`**: 다음에 실행할 노드를 지정 (라우팅)
- **`update`**: 상태에 새로운 메시지를 추가 (컨텍스트 전달)

> ⚠️ **흔한 오해**: `Command`의 타입 힌트 `Command[Literal["agent_1", "agent_2", "__end__"]]`는 런타임에 제약을 걸지 않습니다. 이 힌트는 LangGraph가 **그래프 시각화**에 활용할 뿐, 실제로는 어떤 노드명이든 `goto`에 넣을 수 있습니다. 하지만 존재하지 않는 노드명을 넣으면 런타임 에러가 발생하므로, 타입 힌트를 정확히 맞춰두는 것이 좋습니다.

### 개념 2: 작업 위임과 핸드오프 — 바톤 터치

> 💡 **비유**: 400m 릴레이 경주에서 선수들이 바톤을 넘기는 것과 같습니다. 첫 번째 주자가 100m를 달리고 나서 바톤(현재까지의 결과)을 다음 주자에게 넘깁니다. 바톤을 떨어뜨리면(핸드오프 실패) 실격이듯, 에이전트 간 핸드오프에서도 결과가 유실되면 전체 워크플로우가 실패합니다.

에이전트가 작업을 마치면 다시 감독자에게 제어권을 돌려주는 것을 **핸드오프(Handoff)**라고 합니다. 각 전문 에이전트 노드는 작업이 끝나면 `Command(goto="supervisor")`를 반환하여 감독자에게 바톤을 넘깁니다.

```python
from langgraph.prebuilt import create_react_agent

# 19.3에서 구현한 전문 에이전트 (create_react_agent 기반)
# 이 에이전트들은 작업 완료 후 자동으로 결과를 반환합니다.

email_agent = create_react_agent(
    model=model,
    tools=[search_emails_tool, send_email_tool],
    name="email_agent",
    prompt="당신은 이메일 분석 전문가입니다. 요청받은 이메일 관련 작업을 수행하세요."
)

calendar_agent = create_react_agent(
    model=model,
    tools=[calendar_event_tool],
    name="calendar_agent",
    prompt="당신은 일정 관리 전문가입니다. 요청받은 일정 관련 작업을 수행하세요."
)

data_agent = create_react_agent(
    model=model,
    tools=[query_database_tool],
    name="data_agent",
    prompt="당신은 데이터 분석 전문가입니다. 요청받은 데이터 분석 작업을 수행하세요."
)

report_agent = create_react_agent(
    model=model,
    tools=[],  # 도구 없이 LLM만으로 보고서 작성
    name="report_agent",
    prompt="당신은 보고서 작성 전문가입니다. 대화에서 수집된 정보를 바탕으로 보고서를 작성하세요."
)
```

이 에이전트들은 `create_react_agent`로 생성되었기 때문에, **서브그래프**로 동작합니다. 서브그래프가 종료되면 자동으로 부모 그래프(감독자)로 제어권이 돌아옵니다.

수동으로 핸드오프를 구현하는 경우에는 `Command`를 직접 반환합니다:

```python
def custom_agent_node(state: MessagesState) -> Command[Literal["supervisor"]]:
    """커스텀 에이전트: 작업 후 감독자에게 핸드오프"""
    # 에이전트 작업 수행
    result = do_agent_work(state)

    # 감독자에게 결과를 전달하며 핸드오프
    return Command(
        goto="supervisor",
        update={"messages": [
            HumanMessage(
                content=f"[작업 완료] {result}",
                name="custom_agent"
            )
        ]}
    )
```

### 개념 3: `langgraph-supervisor` 패키지 — 빠른 오케스트레이션

> 💡 **비유**: 직접 재료를 사서 처음부터 요리하는 것(수동 그래프 구성)과, 밀키트를 사용하는 것(`langgraph-supervisor`)의 차이입니다. 밀키트도 맛있는 요리를 만들 수 있지만, 레시피를 세밀하게 조절하고 싶다면 직접 요리하는 게 낫죠.

LangChain 팀은 감독자 패턴을 쉽게 구성할 수 있도록 `langgraph-supervisor` 패키지를 제공합니다. `create_supervisor` 함수 하나로 감독자 그래프를 생성할 수 있어, 프로토타이핑에 매우 유용합니다.

```python
# pip install langgraph-supervisor
from langgraph_supervisor import create_supervisor

# 한 줄로 감독자 워크플로우 생성!
workflow = create_supervisor(
    agents=[email_agent, calendar_agent, data_agent, report_agent],
    model=model,
    prompt=SUPERVISOR_PROMPT,
    output_mode="full_history",  # 전체 대화 기록 유지
    supervisor_name="supervisor"
)

# 그래프 컴파일 및 실행
app = workflow.compile()
result = app.invoke({
    "messages": [{"role": "user", "content": "이번 주 업무 현황 보고서를 작성해줘"}]
})
```

`create_supervisor`의 주요 파라미터:

| 파라미터 | 설명 | 기본값 |
|---------|------|--------|
| `agents` | 전문 에이전트 리스트 | (필수) |
| `model` | 감독자 LLM | (필수) |
| `prompt` | 감독자 시스템 프롬프트 | `None` |
| `output_mode` | `"full_history"` 또는 `"last_message"` | `"last_message"` |
| `handoff_tool_prefix` | 핸드오프 도구 이름 접두사 | `"transfer_to_"` |
| `supervisor_name` | 감독자 노드 이름 | `"supervisor"` |

`output_mode`에 따라 에이전트의 응답이 어떻게 전달되는지가 달라집니다:
- **`"full_history"`**: 에이전트의 중간 사고 과정까지 모두 포함 → 풍부한 컨텍스트, 높은 토큰 비용
- **`"last_message"`**: 에이전트의 최종 응답만 전달 → 깔끔한 히스토리, 낮은 토큰 비용

내부적으로 `create_supervisor`는 각 에이전트에 대해 **핸드오프 도구(Handoff Tool)**를 자동 생성합니다. 감독자 LLM이 이 도구를 호출하면 해당 에이전트로 라우팅되는 구조입니다.

```python
# 핸드오프 도구를 커스터마이징할 수도 있습니다
from langgraph_supervisor import create_handoff_tool

workflow = create_supervisor(
    agents=[email_agent, data_agent],
    model=model,
    tools=[
        create_handoff_tool(
            agent_name="email_agent",
            name="이메일_분석_요청",
            description="이메일 관련 작업이 필요할 때 호출합니다."
        ),
        create_handoff_tool(
            agent_name="data_agent",
            name="데이터_분석_요청",
            description="데이터 조회나 분석이 필요할 때 호출합니다."
        ),
    ]
)
```

> 🔥 **실무 팁**: `create_supervisor`는 빠른 프로토타이핑에 좋지만, 프로덕션에서는 **수동 `Command` 기반 감독자**를 권장합니다. LangChain 팀도 "컨텍스트 엔지니어링"을 세밀하게 제어하려면 수동 구현이 낫다고 안내하고 있습니다. 어떤 메시지를 어떤 에이전트에게 전달할지, 불필요한 히스토리를 어디서 잘라낼지 등을 직접 조율해야 비용과 품질을 최적화할 수 있거든요.

### 개념 4: 결과 집계와 최종 보고

> 💡 **비유**: 뉴스 앵커가 각지의 특파원 리포트를 받아서 하나의 뉴스로 엮는 것과 같습니다. "서울에서는 이런 일이 있었고, 부산에서는 저런 상황입니다"처럼, 감독자는 각 에이전트의 결과를 받아 하나의 일관된 보고서로 종합합니다.

여러 에이전트의 결과를 모아서 최종 출력을 만드는 데는 두 가지 패턴이 있습니다.

**패턴 1: 순차 집계 (Sequential Aggregation)**

감독자가 에이전트를 순차적으로 호출하고, 모든 대화 기록이 `messages`에 누적됩니다. 마지막에 `report_agent`가 전체 기록을 보고 보고서를 작성합니다.

```python
# messages 상태에 자연스럽게 누적되는 방식
# email_agent 결과 → calendar_agent 결과 → data_agent 결과 → report_agent가 종합
```

이 방식은 구현이 간단하지만, 메시지가 길어지면 토큰 비용이 급증합니다.

**패턴 2: 구조화된 집계 (Structured Aggregation)**

각 에이전트의 결과를 별도의 상태 필드에 저장하고, 최종 노드에서 집계합니다.

```python
from typing import Annotated
import operator

class OrchestratorState(MessagesState):
    """에이전트 결과를 구조화하여 저장하는 상태"""
    # operator.add 리듀서: 여러 에이전트가 각자 append
    agent_results: Annotated[list[dict], operator.add]
    final_report: str

def wrap_agent_output(agent_name: str):
    """에이전트 출력을 구조화하는 래퍼"""
    def wrapper(state: OrchestratorState) -> dict:
        # 에이전트 실행
        result = agents[agent_name].invoke(state)
        last_message = result["messages"][-1].content

        # 결과를 구조화하여 agent_results에 추가
        return {
            "messages": result["messages"],
            "agent_results": [{
                "agent": agent_name,
                "result": last_message
            }]
        }
    return wrapper

def synthesize_report(state: OrchestratorState) -> dict:
    """모든 에이전트 결과를 종합하여 최종 보고서 생성"""
    # 각 에이전트의 결과를 섹션별로 정리
    sections = []
    for item in state["agent_results"]:
        sections.append(f"### {item['agent']} 보고\n{item['result']}")

    combined = "\n\n".join(sections)

    # LLM으로 자연스러운 보고서 생성
    report = model.invoke([
        SystemMessage(content="다음 데이터를 종합하여 깔끔한 업무 보고서를 작성하세요."),
        HumanMessage(content=combined)
    ])

    return {"final_report": report.content}
```

`Annotated[list, operator.add]` 리듀서를 사용하면, 여러 에이전트가 동시에 또는 순차적으로 `agent_results`에 값을 추가할 수 있습니다. LangGraph가 자동으로 리스트를 병합해 줍니다.

## 실습: 직접 해보기

이제 19.1~19.3에서 만든 구성 요소들을 모두 엮어, 완전한 업무 자동화 오케스트레이션 시스템을 구축해 봅시다.

```python
"""
업무 자동화 워크플로우 오케스트레이션 시스템
- 감독자 노드가 전문 에이전트들을 지휘
- Command 기반 라우팅으로 유연한 핸드오프
- 구조화된 결과 집계와 최종 보고서 생성
"""

from typing import Literal, TypedDict, Annotated
from dotenv import load_dotenv
import operator

from langchain_openai import ChatOpenAI
from langchain_core.messages import SystemMessage, HumanMessage, AIMessage
from langchain_core.tools import tool
from langgraph.graph import StateGraph, MessagesState, START, END
from langgraph.types import Command
from langgraph.prebuilt import create_react_agent

load_dotenv()

# ============================================================
# 1. 상태 스키마 정의
# ============================================================

class WorkflowState(MessagesState):
    """워크플로우 전체 상태"""
    # 각 에이전트의 결과를 구조화하여 저장
    agent_results: Annotated[list[dict], operator.add]
    # 최종 보고서
    final_report: str
    # 감독자가 호출한 에이전트 순서 추적
    call_history: Annotated[list[str], operator.add]


# ============================================================
# 2. 시뮬레이션 도구 (19.2에서 구현한 도구의 간소화 버전)
# ============================================================

@tool
def search_emails(query: str, max_results: int = 5) -> str:
    """이메일을 검색합니다."""
    # 시뮬레이션 데이터
    return (
        f"'{query}' 검색 결과 ({max_results}건):\n"
        "1. [긴급] Q1 매출 보고서 검토 요청 - 김부장 (3/1)\n"
        "2. 프로젝트 알파 진행 상황 공유 - 이과장 (3/2)\n"
        "3. 3월 팀 회식 일정 투표 - 박대리 (3/3)\n"
    )

@tool
def get_calendar_events(date_range: str) -> str:
    """캘린더 일정을 조회합니다."""
    return (
        f"{date_range} 일정:\n"
        "- 3/3 10:00 주간 스탠드업 미팅\n"
        "- 3/4 14:00 프로젝트 알파 리뷰\n"
        "- 3/5 11:00 팀장 1:1 미팅\n"
        "- 3/7 15:00 분기 회고\n"
    )

@tool
def query_database(sql_query: str) -> str:
    """데이터베이스를 쿼리합니다."""
    return (
        f"쿼리 실행 결과:\n"
        "| 구분 | 1월 | 2월 | 3월(잠정) |\n"
        "|------|-----|-----|----------|\n"
        "| 매출 | 1.2억 | 1.5억 | 1.8억 |\n"
        "| 고객수 | 230 | 280 | 310 |\n"
        "| 이탈률 | 5.2% | 4.8% | 4.1% |\n"
    )


# ============================================================
# 3. 전문 에이전트 생성 (19.3 패턴 활용)
# ============================================================

MODEL_NAME = "gpt-4o"
TEMPERATURE = 0

model = ChatOpenAI(model=MODEL_NAME, temperature=TEMPERATURE)

# 이메일 분석 에이전트
email_agent = create_react_agent(
    model=model,
    tools=[search_emails],
    name="email_agent",
    prompt=(
        "당신은 이메일 분석 전문가입니다. "
        "요청받은 이메일 관련 작업을 수행하고 핵심 내용을 요약하세요."
    )
)

# 일정 관리 에이전트
calendar_agent = create_react_agent(
    model=model,
    tools=[get_calendar_events],
    name="calendar_agent",
    prompt=(
        "당신은 일정 관리 전문가입니다. "
        "요청받은 일정을 조회하고 중요 일정을 정리하세요."
    )
)

# 데이터 분석 에이전트
data_agent = create_react_agent(
    model=model,
    tools=[query_database],
    name="data_agent",
    prompt=(
        "당신은 데이터 분석 전문가입니다. "
        "요청받은 데이터를 조회하고 핵심 인사이트를 도출하세요."
    )
)

# 보고서 작성 에이전트 (도구 없이 LLM만 사용)
report_agent = create_react_agent(
    model=model,
    tools=[],
    name="report_agent",
    prompt=(
        "당신은 보고서 작성 전문가입니다. "
        "대화에서 수집된 모든 정보를 바탕으로 체계적인 보고서를 작성하세요. "
        "보고서에는 핵심 요약, 세부 내용, 액션 아이템을 포함하세요."
    )
)


# ============================================================
# 4. 감독자 노드 구현
# ============================================================

MEMBERS = ["email_agent", "calendar_agent", "data_agent", "report_agent"]

class RouterOutput(TypedDict):
    """감독자의 라우팅 결정"""
    next: Literal["email_agent", "calendar_agent", "data_agent", "report_agent", "FINISH"]
    reason: str

# 구조화된 출력을 위한 LLM
router_llm = model.with_structured_output(RouterOutput)

SUPERVISOR_SYSTEM_PROMPT = f"""당신은 업무 자동화 팀의 감독자입니다.

팀원 목록: {MEMBERS}

각 팀원의 역할:
- email_agent: 이메일 검색, 분석, 요약
- calendar_agent: 일정 조회, 정리
- data_agent: 데이터베이스 쿼리, 매출/지표 분석
- report_agent: 수집된 정보를 종합하여 최종 보고서 작성

위임 규칙:
1. 사용자 요청을 분석하여 필요한 정보 수집 에이전트를 먼저 호출하세요.
2. 이미 호출한 에이전트를 불필요하게 다시 호출하지 마세요.
3. 충분한 정보가 모이면 report_agent를 호출하세요.
4. report_agent가 보고서를 완성하면 FINISH를 반환하세요.
5. 한 번에 하나의 팀원만 선택하세요.
"""


def supervisor(state: WorkflowState) -> Command[Literal[*MEMBERS, "__end__"]]:
    """감독자 노드: 다음 에이전트를 결정하고 작업을 위임합니다."""
    messages = [
        SystemMessage(content=SUPERVISOR_SYSTEM_PROMPT),
        *state["messages"]
    ]

    # LLM이 다음 에이전트를 선택
    decision = router_llm.invoke(messages)

    if decision["next"] == "FINISH":
        return Command(goto=END)

    # 라우팅 로그 남기기
    return Command(
        goto=decision["next"],
        update={
            "messages": [
                HumanMessage(
                    content=f"[감독자 위임] → {decision['next']} | 이유: {decision['reason']}",
                    name="supervisor"
                )
            ],
            "call_history": [decision["next"]]
        }
    )


# ============================================================
# 5. 에이전트 래퍼 — 결과를 구조화하여 저장
# ============================================================

def create_agent_wrapper(agent, agent_name: str):
    """에이전트 실행 후 결과를 agent_results에 저장하는 래퍼"""
    def wrapper(state: WorkflowState) -> Command[Literal["supervisor"]]:
        # 에이전트 실행 (서브그래프)
        result = agent.invoke({"messages": state["messages"]})

        # 마지막 메시지가 에이전트의 최종 응답
        last_msg = result["messages"][-1]

        # 결과를 구조화하여 저장 + 감독자에게 핸드오프
        return Command(
            goto="supervisor",
            update={
                "messages": [
                    AIMessage(
                        content=last_msg.content,
                        name=agent_name
                    )
                ],
                "agent_results": [{
                    "agent": agent_name,
                    "summary": last_msg.content
                }]
            }
        )
    return wrapper


# ============================================================
# 6. 그래프 조립
# ============================================================

def build_orchestration_graph() -> StateGraph:
    """오케스트레이션 그래프를 구성합니다."""
    builder = StateGraph(WorkflowState)

    # 노드 등록
    builder.add_node("supervisor", supervisor)
    builder.add_node("email_agent", create_agent_wrapper(email_agent, "email_agent"))
    builder.add_node("calendar_agent", create_agent_wrapper(calendar_agent, "calendar_agent"))
    builder.add_node("data_agent", create_agent_wrapper(data_agent, "data_agent"))
    builder.add_node("report_agent", create_agent_wrapper(report_agent, "report_agent"))

    # 엣지 구성: 시작 → 감독자
    builder.add_edge(START, "supervisor")
    # 에이전트 → 감독자 핸드오프는 Command로 처리되므로
    # add_edge 불필요 (Command가 라우팅을 담당)

    return builder


# ============================================================
# 7. 실행
# ============================================================

if __name__ == "__main__":
    # 그래프 빌드 및 컴파일
    graph = build_orchestration_graph()
    app = graph.compile()

    # 워크플로우 실행
    print("=" * 60)
    print("업무 자동화 워크플로우 시작")
    print("=" * 60)

    result = app.invoke({
        "messages": [
            HumanMessage(content="이번 주 업무 현황 보고서를 작성해줘. "
                                 "이메일, 일정, 매출 데이터를 모두 포함해줘.")
        ],
        "agent_results": [],
        "call_history": [],
        "final_report": ""
    })

    # 결과 출력
    print("\n" + "=" * 60)
    print("워크플로우 완료!")
    print("=" * 60)

    print(f"\n호출 순서: {' → '.join(result.get('call_history', []))}")

    print("\n--- 에이전트별 수집 결과 ---")
    for item in result.get("agent_results", []):
        print(f"\n[{item['agent']}]")
        print(item["summary"][:200] + "..." if len(item["summary"]) > 200 else item["summary"])

    print("\n--- 최종 메시지 ---")
    if result["messages"]:
        print(result["messages"][-1].content)
```

## 더 깊이 알아보기

### 오케스트레이션의 기원: 마이크로서비스에서 AI 에이전트로

"오케스트레이션"이라는 용어는 원래 **마이크로서비스 아키텍처**에서 왔습니다. 2010년대 넷플릭스가 모놀리식 시스템을 수백 개의 마이크로서비스로 분해하면서, 이 서비스들을 조율하는 "오케스트레이터" 패턴이 유행했죠. Netflix Conductor(2016)가 대표적인 오케스트레이션 엔진이었습니다.

AI 에이전트 영역에서 이 패턴을 본격적으로 도입한 것은 LangGraph입니다. 2024년 1월 LangGraph가 처음 공개되었을 때, 핵심 아이디어는 "에이전트를 **그래프의 노드**로 만들고, 상태 전이를 **엣지**로 표현하자"는 것이었습니다. 기존의 LangChain `AgentExecutor`가 단일 에이전트의 도구 호출 루프만 지원했다면, LangGraph는 **여러 에이전트 간의 복잡한 워크플로우**를 선언적으로 구성할 수 있게 해주었습니다.

`Command` 객체는 2024년 말에 도입된 비교적 새로운 프리미티브인데요, 기존의 `add_conditional_edges`보다 직관적으로 라우팅과 상태 업데이트를 결합할 수 있어 멀티 에이전트 아키텍처 구현을 크게 단순화했습니다. LangChain 블로그에서는 이를 "멀티 에이전트 아키텍처를 위한 새로운 도구"라고 소개하기도 했습니다.

### 수동 구현 vs `langgraph-supervisor` — 언제 무엇을 쓸까?

| 기준 | `create_supervisor` | 수동 `Command` |
|------|-------------------|----------------|
| 프로토타이핑 | 빠르고 편리 | 설정 코드가 많음 |
| 컨텍스트 제어 | `output_mode` 2가지만 | 자유롭게 가공 가능 |
| 에이전트 간 직접 통신 | 불가 (허브-앤-스포크만) | `Command(goto=...)` 자유 |
| 계층 구조 | `compile(name=..)`으로 지원 | 서브그래프 직접 구성 |
| 프로덕션 최적화 | 제한적 | 토큰 절약, 메시지 필터링 등 자유 |

## 흔한 오해와 팁

> ⚠️ **흔한 오해**: "감독자도 LLM이니까 항상 올바른 라우팅을 할 것이다"라고 생각하기 쉽지만, 실제로는 그렇지 않습니다. 감독자 LLM이 같은 에이전트를 무한 반복 호출하거나, 필요한 에이전트를 건너뛰는 경우가 종종 발생합니다. `call_history`를 상태에 포함시켜 "이미 호출한 에이전트"를 감독자에게 알려주고, **최대 호출 횟수**를 프롬프트에 명시하는 것이 안전합니다.

> 💡 **알고 계셨나요?**: `create_supervisor`의 내부 구현을 보면, 사실 각 에이전트에 대한 **핸드오프 도구(Handoff Tool)**를 자동으로 만들어 감독자 LLM에게 제공합니다. 감독자가 "email_agent에게 위임하겠다"고 결정하면, 내부적으로 `transfer_to_email_agent`라는 도구를 호출하고, 이 도구가 `Command(goto="email_agent", graph=Command.PARENT)`를 반환하여 라우팅이 이루어지는 것이죠. 결국 "도구 호출 = 라우팅 결정"인 셈입니다.

> 🔥 **실무 팁**: 토큰 비용을 줄이려면 **컨텍스트 윈도우 관리**가 핵심입니다. 에이전트를 호출할 때 전체 `messages`를 그대로 넘기지 말고, 감독자의 위임 메시지와 관련 컨텍스트만 선별하여 전달하세요. 실습 코드의 `create_agent_wrapper`에서 `state["messages"]` 대신 최근 N개 메시지만 필터링하거나, 에이전트별로 관련 메시지만 추려서 넘기면 비용을 크게 절약할 수 있습니다.

> 🔥 **실무 팁**: 그래프를 시각화하면 디버깅이 훨씬 쉬워집니다. `app.get_graph().draw_mermaid_png()`로 그래프 구조를 이미지로 출력할 수 있고, LangSmith를 연동하면 각 노드의 입출력을 실시간으로 추적할 수 있습니다.

## 핵심 정리

| 개념 | 설명 |
|------|------|
| 감독자 노드 | 사용자 요청을 분석하여 전문 에이전트에게 작업을 순차적으로 위임하는 중앙 제어 노드 |
| `Command` | `goto`(라우팅)와 `update`(상태 변경)를 동시에 수행하는 LangGraph 프리미티브 |
| 핸드오프 | 에이전트가 작업 완료 후 `Command(goto="supervisor")`로 제어권을 돌려주는 메커니즘 |
| `create_supervisor` | 감독자 패턴을 한 줄로 구성하는 `langgraph-supervisor` 패키지 함수 |
| `output_mode` | `"full_history"`(전체 기록)와 `"last_message"`(최종 응답만) 중 선택 |
| 구조화된 집계 | `Annotated[list, operator.add]` 리듀서로 에이전트 결과를 구조화하여 수집 |
| `with_structured_output` | 감독자 LLM의 라우팅 결정을 `TypedDict`로 받아 안정적인 분기 처리 |
| 허브-앤-스포크 | 모든 에이전트가 감독자를 중심으로 통신하는 토폴로지 |

## 다음 섹션 미리보기

이제 에이전트들이 감독자의 지휘 아래 잘 협력하게 되었지만, 한 가지 위험이 남아 있습니다. 이메일을 보내거나 일정을 생성하는 등 **되돌릴 수 없는 작업**을 에이전트가 자동으로 실행해도 될까요? 다음 섹션에서는 **Human-in-the-Loop** 패턴을 적용하여, 위험한 도구 호출 전에 사용자의 승인을 요구하는 안전 메커니즘을 구현합니다. LangGraph의 `interrupt`와 체크포인트를 활용해 워크플로우를 일시 중지하고, 사용자 확인 후 재개하는 방법을 배웁니다.
