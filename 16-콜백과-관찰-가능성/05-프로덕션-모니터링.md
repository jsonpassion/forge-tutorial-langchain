# 프로덕션 모니터링

> LLM 애플리케이션의 핵심 메트릭을 추적하고, 대시보드와 알림으로 프로덕션 품질을 지속적으로 보장하는 방법을 배웁니다.

## 개요

이 섹션에서는 LangChain 애플리케이션을 프로덕션에 배포한 뒤, 안정적으로 운영하기 위한 모니터링 전략을 다룹니다. 앞서 [16.1: 콜백 시스템 이해]에서 배운 이벤트 추적, [16.2: 커스텀 콜백 핸들러]의 비용·토큰 추적, [16.3: LangSmith 트레이싱 심화]의 트레이스 분석, [16.4: LLM 애플리케이션 평가]의 자동화 평가를 모두 통합하여 **"운영 중인 시스템을 지속적으로 관찰하고, 문제가 생기기 전에 잡아내는"** 완전한 모니터링 파이프라인을 구축합니다.

**선수 지식**: 세션 16.1~16.4의 콜백, LangSmith 트레이싱, 평가 파이프라인 개념
**학습 목표**:
- LLM 애플리케이션의 4대 핵심 메트릭(지연 시간, 토큰 사용량, 에러율, 비용)을 정의하고 수집할 수 있다
- LangSmith 대시보드를 구성하여 실시간 시스템 상태를 시각화할 수 있다
- 자동화 규칙(Automations)과 알림(Alerts)을 설정하여 이상 상황에 선제적으로 대응할 수 있다
- 콜백 + 트레이싱 + 평가를 통합한 프로덕션 모니터링 파이프라인을 구축할 수 있다

## 왜 알아야 할까?

LLM 애플리케이션은 전통적인 소프트웨어와 근본적으로 다른 특성이 있습니다. API 호출 비용이 누적되고, 모델 응답 품질이 입력에 따라 크게 달라지며, 외부 API에 의존하기 때문에 지연 시간이 예측하기 어렵죠. 개발 환경에서 잘 동작하던 체인이 프로덕션에서 갑자기 느려지거나, 특정 유형의 입력에서 환각(hallucination)을 일으키는 건 흔한 일입니다.

모니터링 없이 프로덕션을 운영하는 건 **속도계 없이 고속도로를 달리는 것**과 같습니다. 지금 얼마나 빠른지, 연료가 얼마나 남았는지, 엔진에 이상이 있는지 전혀 알 수 없는 거죠. 실제로 많은 팀이 "사용자가 불만을 제기한 뒤에야" 문제를 인지하는 **리액티브(reactive)** 방식으로 운영하다가 큰 장애를 겪습니다. 이 세션에서 배울 **프로액티브(proactive)** 모니터링은 문제가 사용자에게 도달하기 전에 탐지하고 대응하는 것을 목표로 합니다.

## 핵심 개념

### 개념 1: LLM 애플리케이션의 4대 골든 시그널

> 💡 **비유**: 자동차 계기판을 떠올려 보세요. 속도계(지연 시간), 연료 게이지(토큰 사용량), 엔진 경고등(에러율), 주행 비용 미터기(비용) — 이 네 가지만 제대로 보면 차량 상태의 80%는 파악할 수 있죠. LLM 모니터링도 마찬가지입니다.

Google SRE 팀이 정립한 **4대 골든 시그널**(Latency, Traffic, Errors, Saturation)은 모든 분산 시스템 모니터링의 기본 원칙인데요, LLM 애플리케이션에서는 이 개념이 다음과 같이 변환됩니다:

| 골든 시그널 | LLM 모니터링 메트릭 | 의미 |
|------------|---------------------|------|
| Latency | 응답 지연 시간 (P50, P95, P99) | 사용자가 체감하는 대기 시간 |
| Traffic | 요청량 + 토큰 사용량 | 시스템 부하와 API 소비량 |
| Errors | 에러율 + 품질 실패율 | 기술적 에러와 품질 저하 |
| Saturation | 비용 + Rate Limit 근접도 | 리소스 한계까지의 여유 |

전통적 웹 서비스와 다른 점은, LLM 애플리케이션에서는 **"기술적으로 성공했지만 품질이 나쁜"** 응답도 실패로 봐야 한다는 겁니다. HTTP 200을 반환했더라도 환각이 포함되었다면 사용자 관점에서는 장애니까요.

```python
from dataclasses import dataclass, field
from datetime import datetime
from typing import Optional


@dataclass
class LLMMetrics:
    """LLM 애플리케이션의 핵심 메트릭을 수집하는 데이터 클래스"""

    # 지연 시간 관련
    latency_seconds: float = 0.0            # 전체 응답 시간(초)
    time_to_first_token: float = 0.0         # 첫 토큰까지의 시간(초)

    # 토큰 사용량
    input_tokens: int = 0                    # 입력 토큰 수
    output_tokens: int = 0                   # 출력 토큰 수
    total_tokens: int = 0                    # 총 토큰 수

    # 에러 관련
    is_error: bool = False                   # 기술적 에러 발생 여부
    error_type: Optional[str] = None         # 에러 유형
    quality_score: Optional[float] = None    # 품질 점수 (0.0~1.0)

    # 비용
    cost_usd: float = 0.0                   # 예상 비용 (USD)

    # 메타데이터
    model_name: str = ""                     # 사용된 모델명
    timestamp: datetime = field(default_factory=datetime.now)
    trace_id: Optional[str] = None           # LangSmith 트레이스 ID

    @property
    def is_quality_failure(self) -> bool:
        """품질 기준 미달 여부 (점수 0.5 미만이면 실패)"""
        if self.quality_score is not None:
            return self.quality_score < 0.5
        return False
```

### 개념 2: LangSmith 대시보드 구성

> 💡 **비유**: 병원 중환자실의 모니터를 생각해 보세요. 심박수, 혈압, 산소포화도가 실시간으로 표시되고, 이상이 감지되면 즉시 알람이 울립니다. LangSmith 대시보드는 LLM 애플리케이션의 "중환자실 모니터"입니다.

LangSmith는 두 가지 유형의 대시보드를 제공합니다:

1. **프리빌트 대시보드**: 프로젝트를 생성하면 자동으로 생성되며, 트레이스 수, 에러율, 토큰 사용량 등 기본 메트릭을 보여줍니다.
2. **커스텀 대시보드**: P50/P99 지연 시간, 모델별 비용 비교, 피드백 점수 추이 등 팀의 필요에 맞게 차트를 구성할 수 있습니다.

커스텀 대시보드에서 추적할 수 있는 핵심 차트들:
- **지연 시간 분포**: P50, P95, P99 백분위수로 일반적인 응답 시간과 극단값을 모두 파악
- **토큰 사용량 추이**: 입력/출력 토큰을 분리하여 비용 추세를 시각화
- **에러율 타임라인**: 시간대별 에러 비율 변화를 추적하여 패턴을 발견
- **피드백 점수 추이**: 사용자 피드백이나 자동 평가 점수의 시계열 변화

```python
from langsmith import Client
from datetime import datetime, timedelta

# LangSmith 클라이언트 초기화
client = Client()

# 지난 24시간의 트레이스 조회
runs = list(client.list_runs(
    project_name="production-rag",       # 프로젝트 이름
    start_time=datetime.now() - timedelta(hours=24),  # 24시간 이내
    run_type="chain",                    # 체인 실행만 필터링
))

# 핵심 메트릭 집계
latencies = []
token_counts = []
error_count = 0

for run in runs:
    # 지연 시간 (밀리초 → 초)
    if run.end_time and run.start_time:
        latency = (run.end_time - run.start_time).total_seconds()
        latencies.append(latency)

    # 토큰 사용량
    if run.total_tokens:
        token_counts.append(run.total_tokens)

    # 에러 집계
    if run.error:
        error_count += 1

# 결과 출력
import statistics

if latencies:
    print(f"총 실행 수: {len(runs)}")
    print(f"P50 지연 시간: {statistics.median(latencies):.2f}초")
    print(f"P95 지연 시간: {sorted(latencies)[int(len(latencies)*0.95)]:.2f}초")
    print(f"평균 토큰 수: {statistics.mean(token_counts):.0f}")
    print(f"에러율: {error_count/len(runs)*100:.1f}%")
```

### 개념 3: 알림(Alerts)과 자동화 규칙(Automations)

> 💡 **비유**: 집에 설치하는 화재 감지기를 떠올려 보세요. 연기(이상 메트릭)를 감지하면 즉시 경보(알림)를 울리고, 자동으로 스프링클러(자동화 액션)가 작동합니다. 사람이 24시간 불꽃을 지켜볼 필요가 없죠.

LangSmith의 알림 시스템은 크게 두 가지로 나뉩니다:

**알림(Alerts)**: 메트릭 기반 경보
- 에러율, 지연 시간, 피드백 점수 등의 메트릭이 임계값을 넘으면 알림 발생
- 집계 윈도우(5분 또는 15분) 설정으로 민감도 조절
- PagerDuty 또는 웹훅(Slack, 커스텀 엔드포인트 등)으로 전달

**자동화 규칙(Automations)**: 트레이스 기반 자동 액션
- **필터**: 어떤 트레이스에 적용할지 (예: 에러가 발생한 트레이스만)
- **샘플링 비율**: 필터를 통과한 트레이스 중 몇 %에 액션을 적용할지
- **액션**: 데이터셋에 추가, 주석 큐에 추가, 온라인 평가 실행, 웹훅 트리거, 데이터 보존 연장

```python
import json
import logging
from datetime import datetime
from langchain_core.callbacks import BaseCallbackHandler

logger = logging.getLogger("production_monitor")


class ProductionAlertHandler(BaseCallbackHandler):
    """프로덕션 환경에서 이상 상황을 실시간으로 감지하는 콜백 핸들러"""

    def __init__(
        self,
        latency_threshold: float = 10.0,   # 지연 시간 임계값 (초)
        error_rate_window: int = 100,       # 에러율 계산 윈도우
        alert_webhook_url: str | None = None,
    ):
        self.latency_threshold = latency_threshold
        self.error_rate_window = error_rate_window
        self.alert_webhook_url = alert_webhook_url

        # 상태 추적
        self._run_starts: dict[str, datetime] = {}
        self._recent_errors: list[bool] = []  # 최근 N개 실행의 에러 여부

    def on_chain_start(self, serialized, inputs, *, run_id, **kwargs):
        """체인 시작 시각 기록"""
        self._run_starts[str(run_id)] = datetime.now()

    def on_chain_end(self, outputs, *, run_id, **kwargs):
        """체인 종료 시 지연 시간 검사"""
        run_id_str = str(run_id)
        if run_id_str in self._run_starts:
            latency = (datetime.now() - self._run_starts[run_id_str]).total_seconds()

            # 지연 시간 임계값 초과 알림
            if latency > self.latency_threshold:
                self._send_alert(
                    level="warning",
                    message=f"높은 지연 시간 감지: {latency:.2f}초 (임계값: {self.latency_threshold}초)",
                    run_id=run_id_str,
                )

            self._recent_errors.append(False)
            self._check_error_rate()
            del self._run_starts[run_id_str]

    def on_chain_error(self, error, *, run_id, **kwargs):
        """체인 에러 발생 시 알림"""
        self._recent_errors.append(True)
        self._send_alert(
            level="error",
            message=f"체인 실행 에러: {type(error).__name__}: {error}",
            run_id=str(run_id),
        )
        self._check_error_rate()

        # 시작 시각 정리
        run_id_str = str(run_id)
        if run_id_str in self._run_starts:
            del self._run_starts[run_id_str]

    def _check_error_rate(self):
        """에러율이 임계값을 초과하는지 확인"""
        # 윈도우 크기 유지
        if len(self._recent_errors) > self.error_rate_window:
            self._recent_errors = self._recent_errors[-self.error_rate_window:]

        if len(self._recent_errors) >= 10:  # 최소 10개 샘플 필요
            error_rate = sum(self._recent_errors) / len(self._recent_errors)
            if error_rate > 0.1:  # 에러율 10% 초과
                self._send_alert(
                    level="critical",
                    message=f"높은 에러율: {error_rate:.1%} (최근 {len(self._recent_errors)}건)",
                )

    def _send_alert(self, level: str, message: str, run_id: str = ""):
        """알림 전송 (로깅 + 웹훅)"""
        alert_data = {
            "level": level,
            "message": message,
            "run_id": run_id,
            "timestamp": datetime.now().isoformat(),
        }
        # 로그 기록
        log_func = getattr(logger, level if level != "critical" else "error")
        log_func(json.dumps(alert_data, ensure_ascii=False))
```

### 개념 4: 성능 회귀 탐지

> 💡 **비유**: 건강검진에서 "작년 대비 혈당이 15% 올랐다"고 알려주면 아직 당뇨가 아니더라도 식습관을 교정할 수 있죠. 성능 회귀 탐지는 LLM 애플리케이션의 "정기 건강검진"입니다.

성능 회귀란 프롬프트 변경, 모델 업데이트, 데이터 변화 등으로 인해 품질이나 성능이 이전보다 나빠지는 현상을 말합니다. 특히 LLM 애플리케이션에서는 눈에 잘 띄지 않는 **점진적 회귀(gradual regression)**가 위험한데요, 한 번에 큰 장애가 나는 게 아니라 매일 조금씩 응답 품질이 떨어지다가 어느 순간 임계점을 넘기 때문입니다.

회귀 탐지의 핵심은 **기준선(baseline)을 설정하고 현재 상태와 비교**하는 것입니다:

```python
from dataclasses import dataclass
from datetime import datetime, timedelta
import statistics
from langsmith import Client


@dataclass
class PerformanceBaseline:
    """성능 기준선 데이터"""
    p50_latency: float      # 중위 지연 시간
    p95_latency: float      # 95th 백분위 지연 시간
    avg_tokens: float        # 평균 토큰 사용량
    error_rate: float        # 에러율
    avg_feedback: float      # 평균 피드백 점수
    sample_count: int        # 샘플 수
    period_start: datetime   # 기준 기간 시작
    period_end: datetime     # 기준 기간 종료


def create_baseline(
    client: Client,
    project_name: str,
    days: int = 7,
) -> PerformanceBaseline:
    """지정 기간의 성능 기준선을 생성합니다.

    Args:
        client: LangSmith 클라이언트
        project_name: 프로젝트 이름
        days: 기준 기간 (일)

    Returns:
        성능 기준선 데이터
    """
    end_time = datetime.now()
    start_time = end_time - timedelta(days=days)

    # 기준 기간의 트레이스 조회
    runs = list(client.list_runs(
        project_name=project_name,
        start_time=start_time,
        end_time=end_time,
        run_type="chain",
        is_root=True,          # 루트 트레이스만
    ))

    # 지연 시간 수집
    latencies = []
    token_counts = []
    error_count = 0

    for run in runs:
        if run.end_time and run.start_time:
            latency = (run.end_time - run.start_time).total_seconds()
            latencies.append(latency)
        if run.total_tokens:
            token_counts.append(run.total_tokens)
        if run.error:
            error_count += 1

    latencies_sorted = sorted(latencies)

    return PerformanceBaseline(
        p50_latency=statistics.median(latencies) if latencies else 0,
        p95_latency=latencies_sorted[int(len(latencies_sorted) * 0.95)] if latencies else 0,
        avg_tokens=statistics.mean(token_counts) if token_counts else 0,
        error_rate=error_count / len(runs) if runs else 0,
        avg_feedback=0.0,  # 피드백 데이터는 별도 조회 필요
        sample_count=len(runs),
        period_start=start_time,
        period_end=end_time,
    )


def detect_regression(
    baseline: PerformanceBaseline,
    current: PerformanceBaseline,
    threshold: float = 0.2,     # 20% 이상 변화를 회귀로 판단
) -> list[dict]:
    """기준선 대비 성능 회귀를 탐지합니다.

    Args:
        baseline: 기준선 메트릭
        current: 현재 메트릭
        threshold: 회귀 판단 임계값 (비율)

    Returns:
        탐지된 회귀 목록
    """
    regressions = []

    # 지연 시간 회귀 검사 (증가가 나쁜 것)
    if baseline.p50_latency > 0:
        latency_change = (current.p50_latency - baseline.p50_latency) / baseline.p50_latency
        if latency_change > threshold:
            regressions.append({
                "metric": "p50_latency",
                "baseline": f"{baseline.p50_latency:.2f}s",
                "current": f"{current.p50_latency:.2f}s",
                "change": f"+{latency_change:.1%}",
                "severity": "high" if latency_change > 0.5 else "medium",
            })

    # 에러율 회귀 검사
    error_rate_diff = current.error_rate - baseline.error_rate
    if error_rate_diff > 0.05:  # 에러율 5%p 이상 증가
        regressions.append({
            "metric": "error_rate",
            "baseline": f"{baseline.error_rate:.1%}",
            "current": f"{current.error_rate:.1%}",
            "change": f"+{error_rate_diff:.1%}p",
            "severity": "critical" if error_rate_diff > 0.1 else "high",
        })

    # 토큰 사용량 회귀 검사 (비용 증가)
    if baseline.avg_tokens > 0:
        token_change = (current.avg_tokens - baseline.avg_tokens) / baseline.avg_tokens
        if token_change > threshold:
            regressions.append({
                "metric": "avg_tokens",
                "baseline": f"{baseline.avg_tokens:.0f}",
                "current": f"{current.avg_tokens:.0f}",
                "change": f"+{token_change:.1%}",
                "severity": "medium",
            })

    return regressions
```

### 개념 5: 자동화 규칙으로 온라인 평가 연결

> 💡 **비유**: 공장의 품질 검사 라인을 생각해 보세요. 컨베이어 벨트 위의 모든 제품을 검사하는 건 비효율적이니, 일부만 샘플링해서 불량 여부를 검사하죠. LangSmith Automations는 프로덕션 트레이스에 대해 동일한 "샘플링 품질 검사"를 수행합니다.

[16.4: LLM 애플리케이션 평가]에서 배운 오프라인 평가(evaluate)를 프로덕션에서 **온라인으로** 실행하는 것이 자동화 규칙의 핵심 가치입니다. LangSmith에서 자동화 규칙을 설정하면 다음 워크플로우가 자동으로 돌아갑니다:

1. 프로덕션에서 새로운 트레이스가 수집됨
2. 필터 조건에 맞는 트레이스가 선별됨 (예: `run_type == "chain"`)
3. 샘플링 비율에 따라 일부가 선택됨 (예: 10%)
4. 선택된 트레이스에 대해 자동 액션 실행:
   - **온라인 평가**: LLM-as-Judge 자동 채점
   - **데이터셋 추가**: 평가용 데이터 자동 축적
   - **주석 큐 전송**: 사람이 검토해야 할 항목 큐잉
   - **웹훅 트리거**: 외부 시스템에 이벤트 전달

```python
# LangSmith 자동화 규칙은 UI에서 설정하는 것이 일반적이지만,
# 프로그래밍 방식으로 온라인 평가를 시뮬레이션할 수 있습니다.

from langsmith import Client
from datetime import datetime, timedelta

client = Client()


def online_quality_check(
    project_name: str,
    sample_rate: float = 0.1,
    quality_threshold: float = 0.7,
    lookback_minutes: int = 15,
):
    """프로덕션 트레이스에 대한 온라인 품질 검사

    Args:
        project_name: LangSmith 프로젝트 이름
        sample_rate: 샘플링 비율 (0.0~1.0)
        quality_threshold: 품질 기준 점수
        lookback_minutes: 검사 대상 기간 (분)
    """
    import random

    # 최근 트레이스 조회
    recent_runs = list(client.list_runs(
        project_name=project_name,
        start_time=datetime.now() - timedelta(minutes=lookback_minutes),
        run_type="chain",
        is_root=True,
    ))

    # 샘플링
    sampled_runs = [r for r in recent_runs if random.random() < sample_rate]
    print(f"전체 {len(recent_runs)}건 중 {len(sampled_runs)}건 샘플링")

    flagged_runs = []
    for run in sampled_runs:
        # 지연 시간 검사
        if run.end_time and run.start_time:
            latency = (run.end_time - run.start_time).total_seconds()
            if latency > 15.0:  # 15초 초과
                flagged_runs.append({
                    "run_id": str(run.id),
                    "reason": "high_latency",
                    "value": f"{latency:.2f}s",
                })

        # 에러 검사
        if run.error:
            flagged_runs.append({
                "run_id": str(run.id),
                "reason": "error",
                "value": run.error[:100],
            })

        # 피드백 기반 품질 검사
        feedbacks = list(client.list_feedback(run_ids=[run.id]))
        for fb in feedbacks:
            if fb.score is not None and fb.score < quality_threshold:
                flagged_runs.append({
                    "run_id": str(run.id),
                    "reason": "low_quality",
                    "value": f"score={fb.score:.2f}",
                })

    # 결과 리포트
    if flagged_runs:
        print(f"\n⚠️ {len(flagged_runs)}건의 이상 트레이스 발견:")
        for item in flagged_runs:
            print(f"  [{item['reason']}] run={item['run_id'][:8]}... value={item['value']}")
    else:
        print("✅ 이상 없음")

    return flagged_runs
```

```python
"""
프로덕션 모니터링 파이프라인 — 콜백 + 트레이싱 + 평가 통합

필요 패키지:
  pip install langchain langchain-openai langsmith python-dotenv

환경 변수 (.env):
  OPENAI_API_KEY=your-key
  LANGSMITH_API_KEY=your-key
  LANGSMITH_TRACING=true
  LANGSMITH_PROJECT=production-monitoring-demo
"""

import json
import logging
import statistics
import time
from collections import deque
from dataclasses import dataclass, field
from datetime import datetime, timedelta
from typing import Any, Optional

from langchain_core.callbacks import BaseCallbackHandler
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser
from langchain_openai import ChatOpenAI
from langsmith import Client

# 로깅 설정
logging.basicConfig(level=logging.INFO, format="%(asctime)s [%(levelname)s] %(message)s")
logger = logging.getLogger("production_monitor")

# ──────────────────────────────────────────────
# 1단계: 통합 모니터링 콜백 핸들러
# ──────────────────────────────────────────────

# 모델별 가격 (USD per 1K tokens, 2025년 기준)
MODEL_PRICING = {
    "gpt-4o": {"input": 0.0025, "output": 0.01},
    "gpt-4o-mini": {"input": 0.00015, "output": 0.0006},
    "gpt-3.5-turbo": {"input": 0.0005, "output": 0.0015},
}


@dataclass
class MonitoringSnapshot:
    """일정 기간의 모니터링 스냅샷"""
    total_requests: int = 0
    error_count: int = 0
    latencies: list = field(default_factory=list)
    total_input_tokens: int = 0
    total_output_tokens: int = 0
    total_cost_usd: float = 0.0
    quality_scores: list = field(default_factory=list)
    timestamp: datetime = field(default_factory=datetime.now)

    @property
    def error_rate(self) -> float:
        return self.error_count / self.total_requests if self.total_requests > 0 else 0.0

    @property
    def p50_latency(self) -> float:
        return statistics.median(self.latencies) if self.latencies else 0.0

    @property
    def p95_latency(self) -> float:
        if not self.latencies:
            return 0.0
        sorted_lat = sorted(self.latencies)
        idx = int(len(sorted_lat) * 0.95)
        return sorted_lat[min(idx, len(sorted_lat) - 1)]

    def summary(self) -> dict:
        """메트릭 요약 딕셔너리 반환"""
        return {
            "total_requests": self.total_requests,
            "error_rate": f"{self.error_rate:.1%}",
            "p50_latency": f"{self.p50_latency:.2f}s",
            "p95_latency": f"{self.p95_latency:.2f}s",
            "total_tokens": self.total_input_tokens + self.total_output_tokens,
            "total_cost": f"${self.total_cost_usd:.4f}",
        }


class ProductionMonitorCallback(BaseCallbackHandler):
    """프로덕션 환경 통합 모니터링 콜백

    세션 16.2의 비용 추적 핸들러를 확장하여
    지연 시간, 에러율, 품질 모니터링을 통합한 버전입니다.
    """

    def __init__(
        self,
        latency_threshold: float = 10.0,
        error_rate_threshold: float = 0.1,
        window_size: int = 100,
    ):
        self.latency_threshold = latency_threshold
        self.error_rate_threshold = error_rate_threshold

        # 실행 추적
        self._run_starts: dict[str, float] = {}
        self._run_models: dict[str, str] = {}

        # 메트릭 수집 (슬라이딩 윈도우)
        self._recent_latencies: deque = deque(maxlen=window_size)
        self._recent_errors: deque = deque(maxlen=window_size)
        self._snapshot = MonitoringSnapshot()

    def on_llm_start(self, serialized, prompts, *, run_id, **kwargs):
        """LLM 호출 시작 시각 기록"""
        run_id_str = str(run_id)
        self._run_starts[run_id_str] = time.time()

        # 모델 이름 추출
        model_name = kwargs.get("invocation_params", {}).get("model_name", "")
        if not model_name:
            model_name = serialized.get("kwargs", {}).get("model_name", "unknown")
        self._run_models[run_id_str] = model_name

    def on_llm_end(self, response, *, run_id, **kwargs):
        """LLM 호출 종료 — 메트릭 수집"""
        run_id_str = str(run_id)
        self._snapshot.total_requests += 1

        # 지연 시간 계산
        if run_id_str in self._run_starts:
            latency = time.time() - self._run_starts[run_id_str]
            self._snapshot.latencies.append(latency)
            self._recent_latencies.append(latency)

            # 임계값 초과 경고
            if latency > self.latency_threshold:
                logger.warning(f"🐢 높은 지연 시간: {latency:.2f}s (run={run_id_str[:8]})")

            del self._run_starts[run_id_str]

        # 토큰 사용량 집계
        if response.llm_output:
            usage = response.llm_output.get("token_usage", {})
            input_tokens = usage.get("prompt_tokens", 0)
            output_tokens = usage.get("completion_tokens", 0)
            self._snapshot.total_input_tokens += input_tokens
            self._snapshot.total_output_tokens += output_tokens

            # 비용 계산
            model_name = self._run_models.get(run_id_str, "")
            pricing = MODEL_PRICING.get(model_name, {})
            if pricing:
                cost = (
                    input_tokens / 1000 * pricing["input"]
                    + output_tokens / 1000 * pricing["output"]
                )
                self._snapshot.total_cost_usd += cost

        # 에러 없음 기록
        self._recent_errors.append(False)
        self._check_error_rate()

        # 모델 정보 정리
        if run_id_str in self._run_models:
            del self._run_models[run_id_str]

    def on_llm_error(self, error, *, run_id, **kwargs):
        """LLM 에러 발생 — 에러 메트릭 기록"""
        self._snapshot.total_requests += 1
        self._snapshot.error_count += 1
        self._recent_errors.append(True)

        logger.error(f"❌ LLM 에러: {type(error).__name__}: {error}")
        self._check_error_rate()

        # 정리
        run_id_str = str(run_id)
        self._run_starts.pop(run_id_str, None)
        self._run_models.pop(run_id_str, None)

    def _check_error_rate(self):
        """슬라이딩 윈도우 기반 에러율 검사"""
        if len(self._recent_errors) >= 10:
            rate = sum(self._recent_errors) / len(self._recent_errors)
            if rate > self.error_rate_threshold:
                logger.critical(
                    f"🚨 에러율 임계값 초과: {rate:.1%} "
                    f"(최근 {len(self._recent_errors)}건, 임계값: {self.error_rate_threshold:.1%})"
                )

    def get_snapshot(self) -> MonitoringSnapshot:
        """현재까지의 모니터링 스냅샷 반환"""
        return self._snapshot


# ──────────────────────────────────────────────
# 2단계: LangSmith 리포트 생성기
# ──────────────────────────────────────────────

def generate_monitoring_report(
    project_name: str = "production-monitoring-demo",
    hours: int = 24,
) -> dict[str, Any]:
    """LangSmith에서 메트릭을 수집하여 모니터링 리포트를 생성합니다.

    Args:
        project_name: LangSmith 프로젝트 이름
        hours: 리포트 대상 기간 (시간)

    Returns:
        모니터링 리포트 딕셔너리
    """
    client = Client()
    start_time = datetime.now() - timedelta(hours=hours)

    # 트레이스 조회
    runs = list(client.list_runs(
        project_name=project_name,
        start_time=start_time,
        run_type="llm",
    ))

    if not runs:
        return {"status": "no_data", "message": "지정 기간에 트레이스가 없습니다."}

    # 메트릭 계산
    latencies = []
    total_input_tokens = 0
    total_output_tokens = 0
    error_count = 0
    model_distribution: dict[str, int] = {}

    for run in runs:
        # 지연 시간
        if run.end_time and run.start_time:
            latency = (run.end_time - run.start_time).total_seconds()
            latencies.append(latency)

        # 토큰
        if run.prompt_tokens:
            total_input_tokens += run.prompt_tokens
        if run.completion_tokens:
            total_output_tokens += run.completion_tokens

        # 에러
        if run.error:
            error_count += 1

        # 모델 분포
        model = run.extra.get("invocation_params", {}).get("model", "unknown") if run.extra else "unknown"
        model_distribution[model] = model_distribution.get(model, 0) + 1

    sorted_latencies = sorted(latencies) if latencies else [0]

    report = {
        "period": f"최근 {hours}시간",
        "total_runs": len(runs),
        "metrics": {
            "latency": {
                "p50": f"{statistics.median(latencies):.2f}s" if latencies else "N/A",
                "p95": f"{sorted_latencies[int(len(sorted_latencies)*0.95)]:.2f}s" if latencies else "N/A",
                "p99": f"{sorted_latencies[int(len(sorted_latencies)*0.99)]:.2f}s" if latencies else "N/A",
            },
            "tokens": {
                "total_input": total_input_tokens,
                "total_output": total_output_tokens,
                "avg_per_request": (total_input_tokens + total_output_tokens) // len(runs),
            },
            "errors": {
                "count": error_count,
                "rate": f"{error_count / len(runs):.1%}",
            },
            "model_distribution": model_distribution,
        },
        "generated_at": datetime.now().isoformat(),
    }

    return report

def main():
    """통합 모니터링 파이프라인 실행 예시"""
    from dotenv import load_dotenv
    load_dotenv()

    # 모니터링 콜백 생성
    monitor = ProductionMonitorCallback(

    llm = ChatOpenAI(

        callbacks=[monitor],         # 생성자 콜백으로 등록

    prompt = ChatPromptTemplate.from_messages([

    chain = prompt | llm | StrOutputParser()

    # 테스트 요청 시뮬레이션
    test_questions = [
        "파이썬의 GIL이란 무엇인가요?",

    ]

    print("=" * 60)
    print("🔍 프로덕션 모니터링 파이프라인 실행")
    print("=" * 60)

    for i, question in enumerate(test_questions, 1):
        print(f"\n[{i}/{len(test_questions)}] 질문: {question}")
        try:
            # config에 메타데이터 태깅 (LangSmith에서 필터링 가능)
            result = chain.invoke(
                {"question": question},
                config={
                    "metadata": {
                        "environment": "production",
                        "version": "v1.2.0",
                        "user_tier": "premium",
                    },
                    "tags": ["monitoring-demo", "v1.2.0"],
                },
            )
            print(f"   응답: {result[:80]}...")
        except Exception as e:
            print(f"   에러: {e}")

    # 모니터링 스냅샷 출력
    snapshot = monitor.get_snapshot()
    print("\n" + "=" * 60)
    print("📊 모니터링 스냅샷")
    print("=" * 60)
    for key, value in snapshot.summary().items():
        print(f"  {key}: {value}")

    # LangSmith 리포트 생성 (LangSmith 연동 시)
    try:
        report = generate_monitoring_report(hours=1)
        if report.get("status") != "no_data":
            print("\n📋 LangSmith 리포트:")
            print(json.dumps(report, indent=2, ensure_ascii=False))
    except Exception as e:
        print(f"\n(LangSmith 리포트 생성 건너뜀: {e})")

    main()
```

### 골든 시그널의 탄생 — Google SRE 이야기

2003년 Google의 소프트웨어 엔지니어 벤 트레이너 슬로스(Ben Treynor Sloss)가 "Site Reliability Engineering"이라는 개념을 만들었습니다. 당시 Google은 급격한 성장으로 서비스 안정성에 큰 어려움을 겪고 있었는데요, 전통적인 시스템 관리자(sysadmin) 접근법으로는 규모 확장이 불가능했습니다.

SRE 팀은 수천 개의 서비스를 모니터링하면서 "정말 중요한 메트릭이 무엇인가?"를 고민했고, 그 결과 **4대 골든 시그널**(Latency, Traffic, Errors, Saturation)이라는 프레임워크를 정립했습니다. 이 내용은 2016년 출간된 *Site Reliability Engineering: How Google Runs Production Systems*에 공개되어 전 세계 엔지니어링 조직의 모니터링 표준이 되었죠.

흥미로운 점은, 이 골든 시그널이 LLM 시대에 새롭게 해석되고 있다는 것입니다. 전통적 웹 서비스에서 "Saturation"은 CPU나 메모리 사용률을 의미했지만, LLM 애플리케이션에서는 **API 호출 비용**과 **Rate Limit 잔여량**이 핵심 "포화도" 지표가 되었습니다. 서버를 직접 운영하지 않고 외부 API에 의존하는 LLM 앱의 특성이 반영된 거죠.

### 관찰 가능성(Observability)의 어원

"관찰 가능성"이라는 용어는 사실 소프트웨어가 아닌 **제어 이론(Control Theory)**에서 왔습니다. 1960년 헝가리 출신 수학자 루돌프 칼만(Rudolf Kálmán)이 정의한 이 개념은 "시스템의 외부 출력만으로 내부 상태를 추론할 수 있는 정도"를 의미했는데요. 이것이 2010년대 후반 소프트웨어 엔지니어링으로 넘어오면서 "메트릭, 로그, 트레이스(MELT)를 통해 시스템 내부를 이해하는 능력"으로 재정의되었습니다. LangSmith는 이 관찰 가능성 개념을 LLM 애플리케이션에 특화시킨 도구라고 할 수 있습니다.

## 흔한 오해와 팁

> ⚠️ **흔한 오해**: "에러율이 0%이면 시스템이 정상이다"
> LLM 애플리케이션에서는 HTTP 200을 반환하면서도 환각, 무관한 응답, 형식 불일치 등 **품질 실패**가 발생할 수 있습니다. 기술적 에러율 외에 반드시 **품질 메트릭**(피드백 점수, 자동 평가 점수)을 함께 모니터링해야 합니다. 에러율 0%에 안심하다가 품질 저하를 놓치는 경우가 매우 많습니다.

> 💡 **알고 계셨나요?**: LangSmith의 자동화 규칙에는 **백필(Backfill)** 기능이 있습니다. 새로운 규칙을 만들 때 "Apply to past runs" 옵션을 활성화하면, 이미 수집된 과거 트레이스에도 규칙이 소급 적용됩니다. 새로운 평가 기준을 도입할 때 과거 데이터까지 한 번에 분석할 수 있어 매우 유용하죠. 단, 백필은 규칙 생성 시에만 가능하고, 이후에는 설정할 수 없다는 점을 주의하세요.

> 🔥 **실무 팁**: 알림 피로(Alert Fatigue)를 방지하세요. 모든 메트릭에 알림을 설정하면 하루에 수십 개의 알림이 쏟아져 정작 중요한 알림을 놓치게 됩니다. **단계별 전략**을 권장합니다: (1) **Critical** — 에러율 10% 초과 → PagerDuty 즉시 호출, (2) **Warning** — P95 지연 시간 2배 증가 → Slack 채널 알림, (3) **Info** — 일일 비용 예산 80% 도달 → 이메일 리포트. 처음에는 임계값을 넉넉하게 잡고, 운영 데이터가 쌓이면 점차 좁혀가는 것이 핵심입니다.

> 🔥 **실무 팁**: 메타데이터 태깅을 활용하세요. `config={"metadata": {"version": "v1.2.0", "user_tier": "premium"}}` 형태로 트레이스에 메타데이터를 붙이면, 나중에 "v1.2.0에서만 에러가 많은가?", "premium 사용자의 지연 시간은?" 같은 질문에 필터링으로 즉시 답할 수 있습니다. 배포 버전, 사용자 등급, A/B 테스트 그룹 등을 태깅하는 것을 습관화하세요.

## 핵심 정리

| 개념 | 설명 |
|------|------|
| 4대 골든 시그널 | 지연 시간(Latency), 토큰/트래픽(Traffic), 에러율(Errors), 비용/포화도(Saturation) |
| LangSmith 대시보드 | 프리빌트(자동 생성) + 커스텀(P50/P99, 비용, 피드백 차트 구성) |
| 알림(Alerts) | 메트릭 임계값 기반 경보, PagerDuty·웹훅으로 전달, 5/15분 집계 윈도우 |
| 자동화 규칙(Automations) | 필터 + 샘플링 + 액션(데이터셋 추가, 주석 큐, 온라인 평가, 웹훅) |
| 성능 회귀 탐지 | 기준선(baseline) 대비 현재 메트릭 비교, 임계값 초과 시 알림 |
| 품질 실패 | HTTP 200이지만 환각·형식 불일치 등 품질 기준 미달인 응답 |
| 메타데이터 태깅 | 버전, 사용자 등급 등을 트레이스에 첨부하여 세분화 분석 지원 |
| 온라인 평가 | 자동화 규칙으로 프로덕션 트레이스 일부를 실시간 평가하는 패턴 |
