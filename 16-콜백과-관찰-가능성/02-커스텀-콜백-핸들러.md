# 커스텀 콜백 핸들러

> LangChain의 콜백 시스템을 활용하여 로깅, 토큰 추적, 비용 계산, 실시간 알림 등 실무에 필요한 커스텀 핸들러를 직접 만들어봅니다.

## 개요

이 섹션에서는 [16.1: 콜백 시스템 이해](./01-콜백-시스템-이해.md)에서 배운 `BaseCallbackHandler`를 상속하여, 실무에서 바로 사용할 수 있는 다양한 커스텀 콜백 핸들러를 작성합니다. 단순히 이벤트를 출력하는 수준을 넘어, **토큰 사용량 집계**, **API 비용 실시간 계산**, **구조화된 로깅**, **외부 서비스 알림** 등 프로덕션 환경에서 꼭 필요한 패턴을 다룹니다.

**선수 지식**: [16.1: 콜백 시스템 이해](./01-콜백-시스템-이해.md)에서 배운 `BaseCallbackHandler`, `AsyncCallbackHandler`, 콜백 이벤트 생명주기, 생성자 콜백 vs 요청 콜백 개념
**학습 목표**:
- `BaseCallbackHandler`를 상속하여 목적별 커스텀 핸들러를 작성할 수 있다
- `on_llm_end`에서 토큰 사용량을 추출하고 누적 집계할 수 있다
- 모델별 단가를 적용하여 실시간 비용을 계산하는 콜백을 구현할 수 있다
- 외부 서비스(Slack, 웹훅)로 실시간 알림을 보내는 비동기 콜백을 작성할 수 있다

## 왜 알아야 할까?

LLM 애플리케이션을 개발할 때, "잘 돌아가는 것"과 "잘 관리되는 것"은 완전히 다른 문제입니다. 실제 프로덕션 환경에서는 이런 질문들이 매일 나오거든요:

- "이번 달 OpenAI API 비용이 얼마나 나왔죠?"
- "왜 갑자기 응답 시간이 느려졌나요?"
- "어떤 사용자의 요청에서 에러가 발생했는지 추적할 수 있나요?"
- "토큰 사용량이 임계치를 넘으면 바로 알림을 받고 싶어요."

이 모든 질문에 답하려면, LLM 호출의 **모든 단계를 관찰하고 기록하는 시스템**이 필요합니다. 커스텀 콜백 핸들러는 바로 이 역할을 하죠. LangSmith 같은 상용 도구를 쓸 수도 있지만, 내부 정책이나 비용 문제로 직접 구축해야 하는 경우가 많습니다. 이때 커스텀 콜백이 빛을 발합니다.

## 핵심 개념

### 개념 1: 구조화된 로깅 콜백

> 💡 **비유**: 비행기의 블랙박스를 떠올려보세요. 블랙박스는 이륙부터 착륙까지 모든 데이터를 정해진 형식으로 빠짐없이 기록합니다. 로깅 콜백 핸들러는 LLM 애플리케이션의 블랙박스와 같습니다. 어떤 프롬프트가 들어왔고, 얼마나 걸렸고, 무엇을 반환했는지 — 모든 것을 구조화된 형태로 기록하죠.

[16.1](./01-콜백-시스템-이해.md)에서 `on_llm_start`, `on_llm_end` 같은 이벤트 메서드를 배웠는데요, 이제 이들을 조합하여 **실무용 로깅 시스템**을 만들어 보겠습니다. 핵심은 단순 `print` 대신 Python의 `logging` 모듈을 사용하고, JSON 형태로 구조화하여 나중에 분석하기 쉽게 만드는 것입니다.

```python
import logging
import json
import time
from typing import Any, Dict, List, Optional
from uuid import UUID

from langchain_core.callbacks import BaseCallbackHandler
from langchain_core.outputs import LLMResult

# JSON 형식 로거 설정
logger = logging.getLogger("langchain_monitor")
logger.setLevel(logging.INFO)

# 파일 핸들러 추가 (JSON Lines 형식)
handler = logging.FileHandler("llm_calls.jsonl")
handler.setFormatter(logging.Formatter("%(message)s"))
logger.addHandler(handler)


class StructuredLoggingHandler(BaseCallbackHandler):
    """LLM 호출을 JSON 형식으로 구조화하여 로깅하는 콜백 핸들러"""

    def __init__(self) -> None:
        self._start_times: Dict[UUID, float] = {}  # run_id별 시작 시간

    def on_llm_start(
        self,
        serialized: Dict[str, Any],
        prompts: List[str],
        *,
        run_id: UUID,
        **kwargs: Any,
    ) -> None:
        # 시작 시간 기록
        self._start_times[run_id] = time.time()
        log_entry = {
            "event": "llm_start",
            "run_id": str(run_id),
            "model": serialized.get("kwargs", {}).get("model_name", "unknown"),
            "prompt_length": sum(len(p) for p in prompts),
            "timestamp": time.strftime("%Y-%m-%dT%H:%M:%S"),
        }
        logger.info(json.dumps(log_entry, ensure_ascii=False))

    def on_llm_end(
        self,
        response: LLMResult,
        *,
        run_id: UUID,
        **kwargs: Any,
    ) -> None:
        # 경과 시간 계산
        elapsed = time.time() - self._start_times.pop(run_id, time.time())
        log_entry = {
            "event": "llm_end",
            "run_id": str(run_id),
            "elapsed_seconds": round(elapsed, 3),
            "output_length": sum(
                len(g.text) for g in response.generations[0]
            ),
            "timestamp": time.strftime("%Y-%m-%dT%H:%M:%S"),
        }
        logger.info(json.dumps(log_entry, ensure_ascii=False))

    def on_llm_error(
        self,
        error: BaseException,
        *,
        run_id: UUID,
        **kwargs: Any,
    ) -> None:
        elapsed = time.time() - self._start_times.pop(run_id, time.time())
        log_entry = {
            "event": "llm_error",
            "run_id": str(run_id),
            "error_type": type(error).__name__,
            "error_message": str(error),
            "elapsed_seconds": round(elapsed, 3),
            "timestamp": time.strftime("%Y-%m-%dT%H:%M:%S"),
        }
        logger.error(json.dumps(log_entry, ensure_ascii=False))
```

`run_id`를 키로 사용하여 시작과 끝을 매칭하는 패턴이 핵심입니다. 이렇게 하면 여러 LLM 호출이 동시에 실행되더라도 각각의 응답 시간을 정확하게 측정할 수 있거든요.

### 개념 2: 토큰 사용량 추적 콜백

> 💡 **비유**: 수도 요금을 생각해보세요. 수도 계량기는 얼마나 많은 물이 흘렀는지 정확히 측정합니다. 토큰 추적 콜백은 LLM의 "수도 계량기"입니다. 입력 토큰(보낸 물), 출력 토큰(받은 물), 총 토큰(전체 사용량)을 낱낱이 기록하죠.

LangChain v0.3에서는 `AIMessage.usage_metadata`를 통해 토큰 사용량에 접근할 수 있고, `UsageMetadataCallbackHandler`라는 내장 핸들러도 제공됩니다. 하지만 커스텀 핸들러를 직접 만들면 **모델별 집계**, **사용자별 추적**, **임계치 알림** 같은 세밀한 제어가 가능해집니다.

```python
from dataclasses import dataclass, field
from langchain_core.callbacks import BaseCallbackHandler
from langchain_core.outputs import LLMResult


@dataclass
class TokenUsageStats:
    """토큰 사용 통계를 저장하는 데이터 클래스"""
    total_input_tokens: int = 0
    total_output_tokens: int = 0
    total_tokens: int = 0
    call_count: int = 0
    per_model: Dict[str, Dict[str, int]] = field(default_factory=dict)


class TokenTrackingHandler(BaseCallbackHandler):
    """토큰 사용량을 모델별로 추적하는 콜백 핸들러"""

    def __init__(self, token_limit: Optional[int] = None) -> None:
        self.stats = TokenUsageStats()
        self.token_limit = token_limit  # 토큰 사용 한도 (선택)

    def on_llm_end(
        self,
        response: LLMResult,
        **kwargs: Any,
    ) -> None:
        # LLMResult에서 토큰 사용 정보 추출
        token_usage = response.llm_output.get("token_usage", {}) if response.llm_output else {}

        input_tokens = token_usage.get("prompt_tokens", 0)
        output_tokens = token_usage.get("completion_tokens", 0)
        total = token_usage.get("total_tokens", 0)

        # 모델명 추출
        model = response.llm_output.get("model_name", "unknown") if response.llm_output else "unknown"

        # 전체 통계 업데이트
        self.stats.total_input_tokens += input_tokens
        self.stats.total_output_tokens += output_tokens
        self.stats.total_tokens += total
        self.stats.call_count += 1

        # 모델별 통계 업데이트
        if model not in self.stats.per_model:
            self.stats.per_model[model] = {
                "input_tokens": 0,
                "output_tokens": 0,
                "total_tokens": 0,
                "calls": 0,
            }
        self.stats.per_model[model]["input_tokens"] += input_tokens
        self.stats.per_model[model]["output_tokens"] += output_tokens
        self.stats.per_model[model]["total_tokens"] += total
        self.stats.per_model[model]["calls"] += 1

        # 토큰 한도 초과 경고
        if self.token_limit and self.stats.total_tokens > self.token_limit:
            print(
                f"⚠️ 토큰 한도 초과! "
                f"사용: {self.stats.total_tokens:,} / 한도: {self.token_limit:,}"
            )

    def get_summary(self) -> str:
        """토큰 사용 요약 문자열 반환"""
        lines = [
            "=== 토큰 사용 요약 ===",
            f"총 호출 횟수: {self.stats.call_count}",
            f"총 입력 토큰: {self.stats.total_input_tokens:,}",
            f"총 출력 토큰: {self.stats.total_output_tokens:,}",
            f"총 토큰: {self.stats.total_tokens:,}",
            "",
            "--- 모델별 ---",
        ]
        for model, usage in self.stats.per_model.items():
            lines.append(
                f"  {model}: {usage['total_tokens']:,} tokens "
                f"({usage['calls']} calls)"
            )
        return "\n".join(lines)
```

> ⚠️ **흔한 오해**: `response.llm_output`의 구조는 프로바이더마다 다릅니다. OpenAI는 `token_usage` 키 안에 `prompt_tokens`, `completion_tokens`를 넣지만, Anthropic이나 다른 프로바이더는 다른 키를 사용할 수 있습니다. 프로덕션에서는 `usage_metadata`(LangChain v0.3.49+)를 활용하거나, 여러 키 경로에 대한 폴백 로직을 넣는 것이 안전합니다.

### 개념 3: 비용 계산 콜백

> 💡 **비유**: 택시 미터기를 생각해보세요. 미터기는 거리(입력 토큰)와 시간(출력 토큰)에 따라 다른 요금을 적용하고, 실시간으로 요금이 올라가는 것을 보여줍니다. 비용 계산 콜백은 LLM 호출의 택시 미터기입니다 — 모델마다 다른 단가를 적용하여 호출할 때마다 누적 비용을 계산합니다.

LangChain에는 이전에 `get_openai_callback()`이라는 OpenAI 전용 비용 추적 컨텍스트 매니저가 있었습니다. 하지만 이것은 OpenAI 모델에서만 동작하고, 커스텀 단가 적용이 어렵다는 한계가 있었죠. 커스텀 핸들러를 만들면 **어떤 프로바이더든** 지원할 수 있고, **기업 할인 단가**나 **캐시 할인**도 반영할 수 있습니다.

```python
from langchain_core.callbacks import BaseCallbackHandler
from langchain_core.outputs import LLMResult

# 모델별 토큰 단가 (USD per 1K tokens, 2025년 기준)
MODEL_PRICING = {
    "gpt-4o": {"input": 0.0025, "output": 0.01},
    "gpt-4o-mini": {"input": 0.00015, "output": 0.0006},
    "gpt-4-turbo": {"input": 0.01, "output": 0.03},
    "gpt-3.5-turbo": {"input": 0.0005, "output": 0.0015},
    "claude-3-5-sonnet": {"input": 0.003, "output": 0.015},
    "claude-3-5-haiku": {"input": 0.0008, "output": 0.004},
}


class CostTrackingHandler(BaseCallbackHandler):
    """LLM 호출 비용을 실시간으로 계산하고 추적하는 콜백 핸들러"""

    def __init__(
        self,
        budget_limit_usd: Optional[float] = None,
        custom_pricing: Optional[Dict[str, Dict[str, float]]] = None,
    ) -> None:
        self.total_cost_usd: float = 0.0
        self.cost_history: List[Dict[str, Any]] = []
        self.budget_limit_usd = budget_limit_usd
        # 커스텀 단가로 기본 단가를 오버라이드
        self.pricing = {**MODEL_PRICING, **(custom_pricing or {})}

    def on_llm_end(
        self,
        response: LLMResult,
        **kwargs: Any,
    ) -> None:
        if not response.llm_output:
            return

        token_usage = response.llm_output.get("token_usage", {})
        model = response.llm_output.get("model_name", "unknown")

        input_tokens = token_usage.get("prompt_tokens", 0)
        output_tokens = token_usage.get("completion_tokens", 0)

        # 모델 단가 조회 (없으면 기본값 사용)
        pricing = self.pricing.get(model, {"input": 0.01, "output": 0.03})

        # 비용 계산: (토큰 수 / 1000) × 1K 토큰당 단가
        input_cost = (input_tokens / 1000) * pricing["input"]
        output_cost = (output_tokens / 1000) * pricing["output"]
        call_cost = input_cost + output_cost

        self.total_cost_usd += call_cost

        # 호출별 비용 기록
        self.cost_history.append({
            "model": model,
            "input_tokens": input_tokens,
            "output_tokens": output_tokens,
            "input_cost_usd": round(input_cost, 6),
            "output_cost_usd": round(output_cost, 6),
            "call_cost_usd": round(call_cost, 6),
            "cumulative_cost_usd": round(self.total_cost_usd, 6),
        })

        # 예산 한도 체크
        if self.budget_limit_usd and self.total_cost_usd > self.budget_limit_usd:
            print(
                f"🚨 예산 초과! "
                f"누적: ${self.total_cost_usd:.4f} / "
                f"한도: ${self.budget_limit_usd:.4f}"
            )

    def get_report(self) -> str:
        """비용 리포트 문자열 반환"""
        lines = [
            "=== 비용 리포트 ===",
            f"총 비용: ${self.total_cost_usd:.4f}",
            f"총 호출: {len(self.cost_history)}회",
            "",
        ]
        for i, record in enumerate(self.cost_history, 1):
            lines.append(
                f"  [{i}] {record['model']}: "
                f"${record['call_cost_usd']:.6f} "
                f"(in:{record['input_tokens']}, out:{record['output_tokens']})"
            )
        return "\n".join(lines)
```

### 개념 4: 실시간 알림 콜백 (비동기)

> 💡 **비유**: 집에 설치된 화재 감지기를 생각해보세요. 감지기는 평소에는 조용히 있다가, 연기를 감지하면 즉시 경보를 울리고, 연결된 시스템을 통해 소방서에 자동으로 신고합니다. 실시간 알림 콜백도 마찬가지예요. 에러가 발생하거나 비용이 임계치를 넘으면, 즉시 Slack이나 웹훅으로 알림을 보냅니다.

외부 서비스에 HTTP 요청을 보내는 작업은 I/O가 발생하므로, `AsyncCallbackHandler`를 사용하는 것이 좋습니다. 동기 핸들러를 사용하면 LLM 응답을 기다리는 동안 알림 전송이 블로킹될 수 있거든요.

```python
import asyncio
import aiohttp
from langchain_core.callbacks import AsyncCallbackHandler
from langchain_core.outputs import LLMResult


class WebhookAlertHandler(AsyncCallbackHandler):
    """에러 발생 시 웹훅으로 실시간 알림을 보내는 비동기 콜백 핸들러"""

    def __init__(self, webhook_url: str, alert_on_error: bool = True) -> None:
        self.webhook_url = webhook_url
        self.alert_on_error = alert_on_error

    async def _send_alert(self, payload: Dict[str, Any]) -> None:
        """웹훅 URL로 알림 전송"""
        try:
            async with aiohttp.ClientSession() as session:
                async with session.post(
                    self.webhook_url,
                    json=payload,
                    timeout=aiohttp.ClientTimeout(total=5),
                ) as resp:
                    if resp.status != 200:
                        print(f"알림 전송 실패: HTTP {resp.status}")
        except Exception as e:
            # 알림 실패가 메인 로직을 방해하면 안 됨
            print(f"알림 전송 중 에러: {e}")

    async def on_llm_error(
        self,
        error: BaseException,
        *,
        run_id: UUID,
        **kwargs: Any,
    ) -> None:
        if self.alert_on_error:
            await self._send_alert({
                "type": "llm_error",
                "error": str(error),
                "error_type": type(error).__name__,
                "run_id": str(run_id),
                "severity": "critical",
            })

    async def on_llm_end(
        self,
        response: LLMResult,
        *,
        run_id: UUID,
        **kwargs: Any,
    ) -> None:
        # 토큰 사용량이 비정상적으로 높을 때 경고
        if response.llm_output:
            token_usage = response.llm_output.get("token_usage", {})
            total_tokens = token_usage.get("total_tokens", 0)

            if total_tokens > 10000:  # 임계치: 10K 토큰
                await self._send_alert({
                    "type": "high_token_usage",
                    "total_tokens": total_tokens,
                    "run_id": str(run_id),
                    "severity": "warning",
                })
```

> 🔥 **실무 팁**: 알림 콜백에서 가장 중요한 원칙은 **"알림 실패가 메인 로직을 방해하지 않아야 한다"**는 것입니다. `_send_alert` 메서드에서 모든 예외를 잡아서 조용히 처리하는 이유가 바로 이것입니다. Slack 서버가 다운됐다고 LLM 호출까지 실패하면 안 되니까요.

### 개념 5: 여러 콜백 핸들러 조합하기

앞에서 만든 핸들러들은 독립적으로 동작하기 때문에, 여러 개를 동시에 사용할 수 있습니다. [16.1](./01-콜백-시스템-이해.md)에서 배운 것처럼, `config`의 `callbacks` 리스트에 여러 핸들러를 함께 전달하면 됩니다.

```python
from langchain_openai import ChatOpenAI

# 핸들러 인스턴스 생성
logging_handler = StructuredLoggingHandler()
token_handler = TokenTrackingHandler(token_limit=100_000)
cost_handler = CostTrackingHandler(budget_limit_usd=5.0)

# 모든 핸들러를 함께 사용
llm = ChatOpenAI(model="gpt-4o", temperature=0)

response = llm.invoke(
    "LangChain의 콜백 시스템을 설명해주세요.",
    config={
        "callbacks": [logging_handler, token_handler, cost_handler]
    },
)

# 각 핸들러에서 수집한 정보 확인
print(token_handler.get_summary())
print(cost_handler.get_report())
```

각 핸들러는 같은 이벤트를 독립적으로 수신합니다. 로깅 핸들러는 JSON 로그를 파일에 쓰고, 토큰 핸들러는 사용량을 집계하고, 비용 핸들러는 비용을 계산하죠. 마치 공항에서 같은 비행기를 관제탑, 정비팀, 승무원이 각자 다른 관점에서 동시에 모니터링하는 것과 비슷합니다.

## 실습: 직접 해보기

이제 앞에서 배운 모든 개념을 하나의 완전한 코드로 합쳐봅시다. LLM에 여러 번 질문을 보내면서 토큰과 비용을 실시간으로 추적하는 프로그램입니다.

```python
"""
LangChain 커스텀 콜백 핸들러 실습
- 토큰 사용량 추적
- 비용 계산
- 구조화된 로깅
"""
import json
import time
from dataclasses import dataclass, field
from typing import Any, Dict, List, Optional
from uuid import UUID

from dotenv import load_dotenv
from langchain_core.callbacks import BaseCallbackHandler
from langchain_core.outputs import LLMResult
from langchain_openai import ChatOpenAI

load_dotenv()  # .env 파일에서 OPENAI_API_KEY 로드

# ──────────────────────────────────────────────
# 1. 모델별 토큰 단가 정의
# ──────────────────────────────────────────────
MODEL_PRICING = {
    "gpt-4o": {"input": 0.0025, "output": 0.01},
    "gpt-4o-mini": {"input": 0.00015, "output": 0.0006},
}


# ──────────────────────────────────────────────
# 2. 통합 모니터링 핸들러 (로깅 + 토큰 + 비용)
# ──────────────────────────────────────────────
class MonitoringHandler(BaseCallbackHandler):
    """로깅, 토큰 추적, 비용 계산을 하나로 통합한 모니터링 핸들러"""

    def __init__(self, budget_limit_usd: Optional[float] = None) -> None:
        self._start_times: Dict[UUID, float] = {}
        self.total_input_tokens: int = 0
        self.total_output_tokens: int = 0
        self.total_cost_usd: float = 0.0
        self.call_count: int = 0
        self.call_logs: List[Dict[str, Any]] = []
        self.budget_limit_usd = budget_limit_usd

    def on_llm_start(
        self,
        serialized: Dict[str, Any],
        prompts: List[str],
        *,
        run_id: UUID,
        **kwargs: Any,
    ) -> None:
        self._start_times[run_id] = time.time()
        print(f"🟢 LLM 호출 시작 (run_id: {str(run_id)[:8]}...)")

    def on_llm_end(
        self,
        response: LLMResult,
        *,
        run_id: UUID,
        **kwargs: Any,
    ) -> None:
        elapsed = time.time() - self._start_times.pop(run_id, time.time())
        self.call_count += 1

        # 토큰 사용량 추출
        token_usage = (
            response.llm_output.get("token_usage", {})
            if response.llm_output
            else {}
        )
        input_tokens = token_usage.get("prompt_tokens", 0)
        output_tokens = token_usage.get("completion_tokens", 0)
        model = (
            response.llm_output.get("model_name", "unknown")
            if response.llm_output
            else "unknown"
        )

        # 누적 집계
        self.total_input_tokens += input_tokens
        self.total_output_tokens += output_tokens

        # 비용 계산
        pricing = MODEL_PRICING.get(model, {"input": 0.01, "output": 0.03})
        cost = (input_tokens / 1000) * pricing["input"] + \
               (output_tokens / 1000) * pricing["output"]
        self.total_cost_usd += cost

        # 호출 로그 기록
        log_entry = {
            "call_number": self.call_count,
            "model": model,
            "input_tokens": input_tokens,
            "output_tokens": output_tokens,
            "cost_usd": round(cost, 6),
            "elapsed_sec": round(elapsed, 2),
        }
        self.call_logs.append(log_entry)

        print(
            f"🔵 LLM 호출 완료 #{self.call_count} | "
            f"모델: {model} | "
            f"토큰: {input_tokens}+{output_tokens} | "
            f"비용: ${cost:.6f} | "
            f"시간: {elapsed:.2f}s"
        )

        # 예산 한도 체크
        if self.budget_limit_usd and self.total_cost_usd > self.budget_limit_usd:
            print(
                f"🚨 예산 한도(${self.budget_limit_usd}) 초과! "
                f"누적: ${self.total_cost_usd:.4f}"
            )

    def on_llm_error(
        self,
        error: BaseException,
        *,
        run_id: UUID,
        **kwargs: Any,
    ) -> None:
        elapsed = time.time() - self._start_times.pop(run_id, time.time())
        print(f"🔴 LLM 에러 발생: {error} ({elapsed:.2f}s)")

    def print_report(self) -> None:
        """최종 리포트 출력"""
        print("\n" + "=" * 50)
        print("📊 모니터링 리포트")
        print("=" * 50)
        print(f"총 호출 횟수: {self.call_count}")
        print(f"총 입력 토큰: {self.total_input_tokens:,}")
        print(f"총 출력 토큰: {self.total_output_tokens:,}")
        print(f"총 토큰: {self.total_input_tokens + self.total_output_tokens:,}")
        print(f"총 비용: ${self.total_cost_usd:.4f}")
        if self.call_count > 0:
            avg_cost = self.total_cost_usd / self.call_count
            print(f"평균 비용/호출: ${avg_cost:.6f}")
        print("=" * 50)


# ──────────────────────────────────────────────
# 3. 실행
# ──────────────────────────────────────────────
def main() -> None:
    # 모니터링 핸들러 생성 (예산 한도: $1.00)
    monitor = MonitoringHandler(budget_limit_usd=1.0)

    # LLM 인스턴스 생성
    llm = ChatOpenAI(model="gpt-4o-mini", temperature=0.7)

    # 여러 질문을 순차적으로 호출
    questions = [
        "Python의 GIL(Global Interpreter Lock)이 뭔가요?",
        "비동기 프로그래밍과 멀티스레딩의 차이를 간단히 설명해주세요.",
        "FastAPI와 Flask의 주요 차이점은 무엇인가요?",
    ]

    for question in questions:
        print(f"\n💬 질문: {question}")
        response = llm.invoke(
            question,
            config={"callbacks": [monitor]},
        )
        # 응답의 첫 100자만 출력
        print(f"📝 응답: {response.content[:100]}...")

    # 최종 리포트 출력
    monitor.print_report()


if __name__ == "__main__":
    main()
```

## 더 깊이 알아보기

### 옵서버 패턴의 기원과 콜백의 진화

LangChain의 콜백 시스템이 채택한 **옵서버(Observer) 패턴**은 1994년 GoF(Gang of Four)의 『Design Patterns』에서 체계화된 유서 깊은 디자인 패턴입니다. 원래는 GUI 프로그래밍에서 버튼 클릭 같은 이벤트를 처리하기 위해 탄생했는데요, "관찰 대상(Subject)의 상태가 변하면 등록된 관찰자(Observer)에게 자동으로 알려준다"는 단순한 아이디어입니다.

LangChain 초기 버전(v0.0.x)에서는 콜백 시스템이 꽤 단순했습니다. `CallbackManager`에 핸들러를 등록하는 방식이었죠. 하지만 LCEL과 Runnable 인터페이스가 도입되면서, 콜백도 `config` 딕셔너리를 통해 전달하는 방식으로 진화했습니다. LangChain v0.3에서는 `CallbackManager`에 직접 핸들러를 붙이는 방식이 공식적으로 deprecated 되었고, `config={"callbacks": [...]}` 패턴이 표준이 되었습니다.

재미있는 점은, LangChain v0.3.49에서 추가된 `UsageMetadataCallbackHandler`의 탄생 배경입니다. 커뮤니티에서 "왜 `get_openai_callback()`은 OpenAI만 지원하나요?"라는 불만이 계속 제기되었고, LangChain 팀은 결국 **모든 프로바이더에서 동작하는 범용 토큰 추적 콜백**을 만들었습니다. 이처럼 오픈소스의 장점은 사용자 피드백이 직접 기능 개선으로 이어진다는 점이죠.

### 프로덕션에서의 관찰 가능성(Observability) 계층

실무에서 콜백 핸들러는 보통 **3계층 관찰 가능성 스택**의 일부로 설계됩니다:

1. **수집 계층 (Collection)**: 커스텀 콜백 핸들러가 이벤트를 수집
2. **저장 계층 (Storage)**: 수집된 데이터를 시계열 DB(InfluxDB, TimescaleDB)나 로그 시스템(Elasticsearch)에 저장
3. **시각화 계층 (Visualization)**: Grafana, Kibana 등으로 대시보드 구성

이 세션에서 만든 핸들러들은 1계층에 해당합니다. 2~3계층은 이후 세션에서 LangSmith와 함께 다룰 예정입니다.

## 흔한 오해와 팁

> ⚠️ **흔한 오해**: "콜백 핸들러에서 무거운 작업을 해도 괜찮다." — 아닙니다! 동기 콜백 핸들러의 코드는 LLM 호출과 **같은 스레드**에서 실행됩니다. 데이터베이스 쿼리나 HTTP 요청 같은 I/O 작업을 동기 핸들러에 넣으면 LLM 응답 시간이 느려집니다. 무거운 I/O 작업은 반드시 `AsyncCallbackHandler`를 사용하거나, 큐에 넣고 별도 워커에서 처리하세요.

> 💡 **알고 계셨나요?**: LangChain의 `get_openai_callback()`은 사실 컨텍스트 매니저 패턴으로 구현된 콜백 핸들러입니다. 내부적으로는 `OpenAICallbackHandler`를 `with` 블록 진입 시 등록하고, 블록을 빠져나올 때 제거합니다. 이 패턴을 참고하면 "특정 코드 블록에서만 활성화되는 콜백"을 직접 만들 수도 있습니다.

> 🔥 **실무 팁**: 토큰 비용을 추적할 때, API 프로바이더의 단가는 자주 변경됩니다. 단가를 코드에 하드코딩하지 말고, 환경 변수나 설정 파일에서 읽어오도록 설계하세요. 또한 `cached_tokens`(OpenAI의 Prompt Caching)처럼 할인이 적용되는 토큰도 있으므로, 정확한 비용 계산에는 이를 고려해야 합니다.

## 핵심 정리

| 개념 | 설명 |
|------|------|
| `StructuredLoggingHandler` | `run_id`로 시작/끝을 매칭하여 JSON Lines 형식으로 구조화된 로그를 기록하는 핸들러 |
| `TokenTrackingHandler` | `on_llm_end`에서 `llm_output.token_usage`를 추출하여 모델별 토큰 사용량을 집계하는 핸들러 |
| `CostTrackingHandler` | 모델별 단가 테이블을 참조하여 호출마다 비용을 실시간 계산하고 예산 한도를 체크하는 핸들러 |
| `WebhookAlertHandler` | `AsyncCallbackHandler`를 상속받아 에러나 이상 징후 발생 시 웹훅으로 알림을 전송하는 비동기 핸들러 |
| `UsageMetadataCallbackHandler` | LangChain v0.3.49+에서 제공하는 범용 토큰 추적 내장 콜백 (모든 프로바이더 지원) |
| 핸들러 조합 | `config={"callbacks": [h1, h2, h3]}` 형태로 여러 핸들러를 동시에 사용 가능 |
| 동기 vs 비동기 | I/O 작업이 포함된 핸들러는 `AsyncCallbackHandler`를 사용하여 블로킹 방지 |

## 다음 섹션 미리보기

이번 세션에서는 커스텀 콜백 핸들러를 **직접 만드는 방법**에 집중했습니다. 하지만 프로덕션 환경에서 콜백만으로 모든 관찰 가능성 요구사항을 충족하기는 어렵습니다. 다음 세션 **16.3: LangSmith 트레이싱**에서는 LangChain의 공식 관찰 가능성 플랫폼인 LangSmith를 사용하여 트레이스를 시각적으로 분석하고, 실행 흐름을 디버깅하고, 체인의 각 단계별 입출력을 상세하게 추적하는 방법을 배웁니다. 직접 만든 콜백과 LangSmith를 함께 사용하면 어떤 시너지가 나는지도 살펴볼 예정이니 기대해주세요.
