# 파싱 에러 복구와 자동 수정

> LLM 출력 파싱이 실패했을 때 자동으로 복구하고, 프로덕션 환경에서 안정적으로 구조화된 출력을 보장하는 전략을 배웁니다.

## 개요

이 섹션에서는 LLM의 출력이 기대한 형식과 다를 때 어떻게 자동으로 감지하고 복구하는지를 학습합니다. 단순히 에러를 던지는 것이 아니라, LLM 스스로가 자신의 실수를 고치도록 하거나, 여러 단계의 폴백(fallback) 전략을 구성하는 방법을 익힙니다.

**선수 지식**: [4.1 출력 파서 기초](./01-출력-파서-기초.md)에서 배운 OutputParser와 format_instructions, [4.2 Pydantic 기반 구조화 출력](./02-pydantic-기반-구조화-출력.md)에서 배운 PydanticOutputParser와 검증 로직, [4.3 with_structured_output 활용](./03-with-structured-output-활용.md)에서 배운 include_raw 옵션, [4.4 스트리밍과 부분 파싱](./04-스트리밍과-부분-파싱.md)에서 배운 스트리밍 환경의 파싱 제약

**학습 목표**:
- OutputFixingParser로 잘못된 출력을 LLM이 자동 수정하게 할 수 있다
- RetryWithErrorOutputParser로 원본 프롬프트와 에러 정보를 활용한 재시도 전략을 구현할 수 있다
- with_fallbacks()를 활용한 다단계 폴백 체인을 설계할 수 있다
- 프로덕션 환경에서 파싱 실패를 로깅하고 모니터링하는 체계를 구축할 수 있다

## 왜 알아야 할까?

실제 서비스에서 LLM을 운영해보면, 놀라울 정도로 자주 파싱 에러를 마주하게 됩니다. LLM은 본질적으로 확률적(stochastic)이기 때문에 아무리 정교한 프롬프트를 작성해도 100%의 형식 준수를 보장할 수 없거든요.

프로덕션 환경에서 파싱 에러가 발생하면 어떻게 될까요? 사용자에게 "서버 오류" 메시지가 뜨고, API 요금은 이미 청구되었고, 귀중한 LLM 응답은 버려집니다. 한 번의 API 호출에 수십 원에서 수백 원이 드는데, 결과를 살리지 못하고 버리는 건 정말 아까운 일이죠.

이 섹션에서 배울 에러 복구 전략들은 마치 **자동차의 에어백과 ABS**와 같습니다. 사고(파싱 실패)가 나지 않도록 예방하는 것이 최선이지만, 사고가 났을 때 피해를 최소화하는 안전장치도 반드시 필요합니다. LangChain은 이를 위한 여러 도구를 제공하는데, 상황에 맞게 적절한 전략을 선택하는 것이 핵심입니다.

## 핵심 개념

### 개념 1: OutputParserException — 파싱 에러의 시작점

> 💡 **비유**: OutputParserException은 병원의 **진단서**와 같습니다. 단순히 "아프다"가 아니라, 어디가 아픈지(observation), 검사 결과는 어땠는지(llm_output), 전문의에게 의뢰할지(send_to_llm)까지 기록되어 있죠.

LangChain의 모든 출력 파서는 파싱 실패 시 일반 `Exception`이 아닌 `OutputParserException`을 발생시킵니다. 이 예외에는 복구에 필요한 정보가 담겨 있어서, 단순한 에러 메시지와는 차원이 다릅니다.

```python
from langchain_core.exceptions import OutputParserException

# OutputParserException의 핵심 속성들
try:
    parser.parse(malformed_output)
except OutputParserException as e:
    print(f"에러 메시지: {e}")                    # 무엇이 잘못되었는지
    print(f"LLM 원본 출력: {e.llm_output}")       # LLM이 실제로 뱉은 텍스트
    print(f"관찰 내용: {e.observation}")           # 에러 상세 설명
    print(f"LLM에 전달 여부: {e.send_to_llm}")    # 에이전트 환경에서 LLM에 피드백할지
```

`OutputParserException`이 일반 `ValueError`나 `Exception`과 다른 점은, **파싱 에러를 다른 종류의 에러와 구분**할 수 있다는 것입니다. 네트워크 장애나 인증 실패 같은 에러와 파싱 에러는 복구 전략이 완전히 다르니까요.

### 개념 2: OutputFixingParser — "이거 고쳐줘"

> 💡 **비유**: OutputFixingParser는 **교정 편집자**와 같습니다. 작가(LLM)가 쓴 원고에 문법 오류가 있으면, 편집자(또 다른 LLM)가 원고를 받아서 문법 규칙에 맞게 고쳐주는 거죠. 원고 내용은 이해하지만, 작가가 왜 그렇게 썼는지(원래 프롬프트)는 모릅니다.

OutputFixingParser는 가장 단순한 에러 복구 전략입니다. 파싱이 실패하면, **잘못된 출력**과 **에러 메시지**를 다른 LLM에 보내서 형식을 수정하도록 합니다.

```python
from langchain.output_parsers import OutputFixingParser
from langchain_core.output_parsers import PydanticOutputParser
from langchain_openai import ChatOpenAI
from pydantic import BaseModel, Field

# 1. 원본 파서 정의
class MovieReview(BaseModel):
    title: str = Field(description="영화 제목")
    rating: float = Field(description="평점 (0.0~10.0)", ge=0, le=10)
    pros: list[str] = Field(description="장점 목록")
    cons: list[str] = Field(description="단점 목록")

parser = PydanticOutputParser(pydantic_object=MovieReview)

# 2. OutputFixingParser로 감싸기
fixing_parser = OutputFixingParser.from_llm(
    parser=parser,        # 원본 파서
    llm=ChatOpenAI(model="gpt-4o-mini", temperature=0),  # 수정용 LLM
    max_retries=3         # 최대 재시도 횟수 (기본값: 1)
)

# 3. 잘못된 형식의 출력 수정 시도
malformed = """
{
    "title": "인셉션",
    "rating": "9점",
    "pros": "뛰어난 스토리, 놀라운 시각효과",
    "cons": "다소 복잡한 전개"
}
"""
# rating이 문자열이고, pros/cons가 리스트가 아님 → 파싱 실패
# fixing_parser가 이를 감지하고 LLM에게 수정 요청

result = fixing_parser.parse(malformed)
print(result)
# MovieReview(title='인셉션', rating=9.0, pros=['뛰어난 스토리', '놀라운 시각효과'], cons=['다소 복잡한 전개'])
```

**동작 원리를 자세히 살펴보면:**

1. `fixing_parser.parse(malformed)` 호출
2. 내부의 `parser.parse(malformed)`가 실패 → `OutputParserException` 발생
3. 잘못된 출력 + 에러 메시지 + format_instructions를 조합하여 수정용 프롬프트 생성
4. 수정용 LLM에 전달 → 수정된 출력 반환
5. 수정된 출력을 다시 `parser.parse()`로 파싱
6. 성공하면 결과 반환, 실패하면 max_retries까지 반복

> ⚠️ **흔한 오해**: OutputFixingParser가 **원래 프롬프트를 알지 못한다**는 점을 기억하세요. 이것은 단순히 "이 JSON을 올바른 형식으로 고쳐줘"라고 요청하는 것이지, "사용자가 영화 리뷰를 요청했으니 빠진 필드를 채워줘"라고 할 수는 없습니다. 형식 오류만 고칠 수 있고, 내용 오류는 고칠 수 없습니다.

### 개념 3: RetryWithErrorOutputParser — "왜 틀렸는지 알려줄 테니 다시 해봐"

> 💡 **비유**: RetryWithErrorOutputParser는 **시험 재응시 기회**를 주는 것과 같습니다. 단순히 "오답"이라고만 알려주는 게 아니라, 문제지(원래 프롬프트), 당신의 답안(원래 출력), 채점 결과(에러 메시지)를 모두 돌려주면서 "다시 풀어보세요"라고 하는 거죠.

OutputFixingParser보다 한 단계 더 정교한 전략입니다. 핵심 차이는 **원본 프롬프트를 함께 전달**한다는 점입니다.

```python
from langchain.output_parsers import RetryWithErrorOutputParser
from langchain_core.prompts import ChatPromptTemplate
from langchain_openai import ChatOpenAI

# 1. 기본 설정
llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)
parser = PydanticOutputParser(pydantic_object=MovieReview)

# 2. RetryWithErrorOutputParser 생성
retry_parser = RetryWithErrorOutputParser.from_llm(
    parser=parser,
    llm=llm,
    max_retries=3  # 최대 재시도 횟수
)

# 3. 프롬프트 템플릿 정의
prompt = ChatPromptTemplate.from_messages([
    ("system", "당신은 영화 평론가입니다. 주어진 형식에 맞춰 리뷰를 작성하세요."),
    ("human", "{query}\n\n{format_instructions}")
])

# 4. 원본 프롬프트를 포함한 재시도
formatted_prompt = prompt.invoke({
    "query": "인셉션 리뷰를 작성해주세요",
    "format_instructions": parser.get_format_instructions()
})

# 형식이 잘못된 LLM 응답
bad_response = "인셉션은 정말 대단한 영화입니다. 10점 만점에 9점!"

# parse_with_prompt: 원본 프롬프트 + 잘못된 응답 + 에러를 LLM에 전달
result = retry_parser.parse_with_prompt(
    bad_response,
    formatted_prompt  # 원본 프롬프트를 함께 전달!
)
print(result)
```

**OutputFixingParser vs RetryWithErrorOutputParser 비교:**

| 특성 | OutputFixingParser | RetryWithErrorOutputParser |
|------|-------------------|---------------------------|
| 원본 프롬프트 참조 | ❌ 불가 | ✅ 가능 |
| 호출 메서드 | `parse(text)` | `parse_with_prompt(text, prompt)` |
| 복구 능력 | 형식 수정만 | 형식 + 내용 보완 |
| API 비용 | 낮음 (짧은 프롬프트) | 높음 (원본 프롬프트 재전달) |
| 적합한 상황 | JSON 형식 깨짐 | 필드 누락, 내용 불완전 |

### 개념 4: with_fallbacks() — 다단계 폴백 체인

> 💡 **비유**: with_fallbacks()는 **여행 보험의 다단계 보상**과 같습니다. 비행기가 결항(1차 실패)되면 다른 항공사로 재예약(1차 폴백), 그것도 안 되면 KTX로 대체(2차 폴백), 그것마저 안 되면 환불(최종 안전망)이 순서대로 시도되는 거죠.

[4.3절](./03-with-structured-output-활용.md)에서 배운 `with_structured_output()`의 `include_raw` 옵션과 함께, LCEL의 `with_fallbacks()`를 조합하면 강력한 다단계 복구 체인을 구성할 수 있습니다.

```python
from langchain_openai import ChatOpenAI
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import PydanticOutputParser, StrOutputParser
from langchain_core.runnables import RunnableLambda

# 1. 스키마 정의
class AnalysisResult(BaseModel):
    topic: str = Field(description="분석 주제")
    sentiment: str = Field(description="감성: positive, negative, neutral")
    confidence: float = Field(description="확신도 (0.0~1.0)", ge=0, le=1)
    key_points: list[str] = Field(description="핵심 포인트 목록")

# 2. 전략별 체인 구성
prompt = ChatPromptTemplate.from_messages([
    ("system", "텍스트를 분석하여 결과를 JSON으로 반환하세요."),
    ("human", "{text}\n\n{format_instructions}")
])

parser = PydanticOutputParser(pydantic_object=AnalysisResult)

# 전략 1: GPT-4o로 구조화 출력 (가장 정확)
primary_chain = prompt | ChatOpenAI(model="gpt-4o").with_structured_output(AnalysisResult)

# 전략 2: GPT-4o-mini + PydanticOutputParser (비용 절감)
fallback_chain = (
    prompt | ChatOpenAI(model="gpt-4o-mini", temperature=0) | parser
)

# 전략 3: 최소한의 텍스트 응답 (최후의 수단)
def create_default_result(input_dict: dict) -> AnalysisResult:
    """파싱 완전 실패 시 기본값 반환"""
    return AnalysisResult(
        topic="분석 실패",
        sentiment="neutral",
        confidence=0.0,
        key_points=["파싱 에러로 인해 분석 결과를 생성하지 못했습니다"]
    )

safety_net = RunnableLambda(create_default_result)

# 3. 폴백 체인 조합
robust_chain = primary_chain.with_fallbacks(
    [fallback_chain, safety_net]
)

# 사용
result = robust_chain.invoke({
    "text": "이 제품은 정말 훌륭합니다. 배송도 빠르고 품질도 좋아요.",
    "format_instructions": parser.get_format_instructions()
})
```

> 🔥 **실무 팁**: `with_fallbacks()`와 `include_raw=True`를 함께 쓸 때 주의하세요! `include_raw=True`는 파싱 에러를 예외로 던지지 않고 결과에 포함시키기 때문에, `with_fallbacks()`가 에러를 감지하지 못합니다. 폴백을 사용하려면 `include_raw=False`(기본값)로 두어야 합니다.

### 개념 5: 커스텀 에러 핸들러 — 직접 설계하는 복구 전략

실제 프로덕션에서는 LangChain의 내장 파서만으로는 부족할 때가 많습니다. 에러 유형에 따라 다른 전략을 적용하고, 모든 실패를 로깅하며, 점진적으로 약화(graceful degradation)시키는 커스텀 핸들러가 필요하거든요.

```python
import logging
from typing import TypeVar, Optional
from langchain_core.exceptions import OutputParserException
from langchain_core.output_parsers import BaseOutputParser
from pydantic import ValidationError

logger = logging.getLogger(__name__)
T = TypeVar("T")

class ResilientParser:
    """다단계 에러 복구를 지원하는 래퍼 파서"""
    
    def __init__(
        self,
        parser: BaseOutputParser[T],
        fixing_parser: Optional[OutputFixingParser] = None,
        retry_parser: Optional[RetryWithErrorOutputParser] = None,
        default_factory: Optional[callable] = None,
        max_retries: int = 2
    ):
        self.parser = parser
        self.fixing_parser = fixing_parser
        self.retry_parser = retry_parser
        self.default_factory = default_factory
        self.max_retries = max_retries
        
        # 에러 통계 (모니터링용)
        self.stats = {
            "total_calls": 0,
            "direct_success": 0,
            "fix_success": 0,
            "retry_success": 0,
            "fallback_used": 0,
            "total_failures": 0
        }
    
    def parse(self, text: str, prompt_value=None) -> T:
        """다단계 복구를 시도하며 파싱"""
        self.stats["total_calls"] += 1
        
        # 1단계: 직접 파싱 시도
        try:
            result = self.parser.parse(text)
            self.stats["direct_success"] += 1
            return result
        except OutputParserException as e:
            logger.warning(f"직접 파싱 실패: {e}")
        
        # 2단계: OutputFixingParser 시도
        if self.fixing_parser:
            try:
                result = self.fixing_parser.parse(text)
                self.stats["fix_success"] += 1
                logger.info("OutputFixingParser로 복구 성공")
                return result
            except OutputParserException as e:
                logger.warning(f"OutputFixingParser 복구 실패: {e}")
        
        # 3단계: RetryWithErrorOutputParser 시도 (프롬프트 필요)
        if self.retry_parser and prompt_value:
            try:
                result = self.retry_parser.parse_with_prompt(text, prompt_value)
                self.stats["retry_success"] += 1
                logger.info("RetryWithErrorOutputParser로 복구 성공")
                return result
            except OutputParserException as e:
                logger.warning(f"RetryWithErrorOutputParser 복구 실패: {e}")
        
        # 4단계: 기본값 폴백
        if self.default_factory:
            self.stats["fallback_used"] += 1
            logger.error(f"모든 복구 실패, 기본값 사용. 원본: {text[:200]}")
            return self.default_factory()
        
        # 모든 전략 실패
        self.stats["total_failures"] += 1
        raise OutputParserException(
            f"모든 복구 전략이 실패했습니다. 원본 출력: {text[:500]}"
        )
    
    def get_stats(self) -> dict:
        """파싱 성공률 통계 반환"""
        total = self.stats["total_calls"]
        if total == 0:
            return self.stats
        return {
            **self.stats,
            "success_rate": (total - self.stats["total_failures"]) / total,
            "direct_rate": self.stats["direct_success"] / total,
            "recovery_rate": (
                self.stats["fix_success"] + self.stats["retry_success"]
            ) / total
        }
```

## 실습: 직접 해보기

실제 프로덕션 시나리오를 시뮬레이션하며, 각 복구 전략이 어떤 상황에서 효과적인지 확인해봅시다.

```python
"""
파싱 에러 복구 실습 — 프로덕션 레벨 에러 핸들링
"""
import json
import logging
from typing import Optional
from pydantic import BaseModel, Field, field_validator
from langchain_openai import ChatOpenAI
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import PydanticOutputParser
from langchain_core.exceptions import OutputParserException
from langchain.output_parsers import OutputFixingParser, RetryWithErrorOutputParser

# 로깅 설정

# 1. Pydantic 모델 정의
# ============================================================
class ProductAnalysis(BaseModel):
    """상품 리뷰 분석 결과"""
    product_name: str = Field(description="상품명")
    overall_sentiment: str = Field(description="전체 감성: positive, negative, mixed")
    score: float = Field(description="점수 (1.0~5.0)", ge=1, le=5)
    strengths: list[str] = Field(description="장점 목록 (최소 1개)", min_length=1)
    weaknesses: list[str] = Field(description="단점 목록")
    recommendation: bool = Field(description="추천 여부")
    
    @field_validator("overall_sentiment")
    @classmethod
    def validate_sentiment(cls, v: str) -> str:
        allowed = {"positive", "negative", "mixed"}
        v_lower = v.lower().strip()
        if v_lower not in allowed:
            raise ValueError(f"sentiment는 {allowed} 중 하나여야 합니다. 받은 값: {v}")
        return v_lower

# ============================================================
llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)
parser = PydanticOutputParser(pydantic_object=ProductAnalysis)

# OutputFixingParser: 형식 오류 자동 수정
fixing_parser = OutputFixingParser.from_llm(
    parser=parser,
    llm=llm,
    max_retries=2
)

# RetryWithErrorOutputParser: 프롬프트 컨텍스트 기반 재시도
retry_parser = RetryWithErrorOutputParser.from_llm(
    parser=parser,
    llm=llm,
    max_retries=2
)

    print(f"[OutputFixingParser] {label}")

def test_retry_parser(text: str, label: str):
    """RetryWithErrorOutputParser 테스트"""
    print(f"\n{'='*60}")
    print(f"[RetryWithErrorOutputParser] {label}")
    print(f"{'='*60}")

# 5. 프로덕션용 폴백 체인 구성

from langchain_core.runnables import RunnableLambda, RunnableParallel

prompt = ChatPromptTemplate.from_messages([
    ("system", "상품 리뷰를 분석하여 지정된 JSON 형식으로 결과를 반환하세요."),
    ("human", "리뷰:\n{review}\n\n{format_instructions}")
])

# 전략 1: with_structured_output (가장 안정적)
chain_structured = (
    prompt 
    | llm.with_structured_output(ProductAnalysis)
)

# 전략 2: PydanticOutputParser (스트리밍 가능)
chain_parser = prompt | llm | parser

# 전략 3: 안전망 — 최소한의 결과 보장
def safe_fallback(inputs: dict) -> ProductAnalysis:
    logger.error(f"모든 전략 실패. 기본값으로 대체합니다.")
    return ProductAnalysis(
        product_name="파싱 실패",
        overall_sentiment="mixed",
        score=3.0,
        strengths=["분석 결과를 생성하지 못했습니다"],
        weaknesses=["파싱 에러 발생"],
        recommendation=False
    )

# 폴백 체인 조합
production_chain = chain_structured.with_fallbacks(
    [chain_parser, RunnableLambda(safe_fallback)]
)

# 실행
print("\n" + "="*60)
print("🚀 프로덕션 폴백 체인 테스트")
print("="*60)

    "format_instructions": parser.get_format_instructions()
})
print(f"결과: {result}")

# 6. 에러 로깅과 모니터링
# ============================================================

class ParsingMonitor:
    """파싱 성공/실패를 추적하는 모니터 클래스"""
    
    def __init__(self):
        self.records: list[dict] = []
    
    def log_attempt(self, strategy: str, success: bool, 
                    error: Optional[str] = None,
                    raw_output: Optional[str] = None):
        """파싱 시도를 기록"""
        record = {
            "strategy": strategy,
            "success": success,
            "error": error,
            "raw_output_preview": raw_output[:200] if raw_output else None
        }
        self.records.append(record)
        
        if not success:
            logger.warning(
                f"파싱 실패 [{strategy}]: {error}"
            )
    
    def summary(self) -> dict:
        """모니터링 요약 통계"""
        total = len(self.records)
        successes = sum(1 for r in self.records if r["success"])
        by_strategy = {}
        for r in self.records:
            s = r["strategy"]
            if s not in by_strategy:
                by_strategy[s] = {"total": 0, "success": 0}
            by_strategy[s]["total"] += 1
            if r["success"]:
                by_strategy[s]["success"] += 1
        
        return {
            "total_attempts": total,
            "success_rate": successes / total if total > 0 else 0,
            "by_strategy": by_strategy
        }

# 모니터 사용 예시
monitor = ParsingMonitor()

# 여러 번의 파싱 시도를 시뮬레이션
monitor.log_attempt("direct_parse", success=True)
monitor.log_attempt("direct_parse", success=False, 
                    error="JSONDecodeError", raw_output=broken_json)
monitor.log_attempt("output_fixing", success=True)
monitor.log_attempt("direct_parse", success=False,
                    error="ValidationError", raw_output=incomplete_output)
monitor.log_attempt("retry_with_error", success=True)

print("\n📊 파싱 모니터링 요약:")
print(json.dumps(monitor.summary(), indent=2, ensure_ascii=False))
```

## 더 깊이 알아보기

### LLM "자기 교정"의 기원

파싱 에러를 LLM 스스로에게 고치게 한다는 아이디어는 사실 LangChain이 처음 만든 개념이 아닙니다. 2022년 말, OpenAI의 ChatGPT가 등장하면서 개발자들은 LLM의 출력을 프로그래밍적으로 활용하고 싶어했지만, 형식 준수가 불안정해서 골머리를 앓았거든요.

Harrison Chase가 2022년 10월 LangChain을 처음 공개했을 때, 초기 버전부터 OutputFixingParser가 포함되어 있었습니다. "LLM이 잘못된 출력을 만들었으면, 또 다른 LLM에게 고치라고 시키면 되지 않나?"라는 단순하면서도 효과적인 직관에서 탄생한 것이죠.

이 접근법은 이후 학계에서 **Self-Refinement** 또는 **Self-Correction** 패턴으로 발전했습니다. 2023년 논문 "Self-Refine: Iterative Refinement with Self-Feedback"(Madaan et al.)에서 LLM이 자신의 출력을 평가하고 반복적으로 개선하는 일반적 프레임워크를 제시했는데, LangChain의 OutputFixingParser와 RetryWithErrorOutputParser는 이 아이디어를 출력 파싱이라는 실용적 영역에 먼저 적용한 셈입니다.

### Function Calling의 등장과 패러다임 전환

흥미롭게도, 2023년 6월 OpenAI가 Function Calling을 발표하면서 파싱 에러 복구의 필요성이 크게 줄었습니다. [4.3절](./03-with-structured-output-활용.md)에서 배운 `with_structured_output()`이 바로 그 결과물이죠. 모델이 프롬프트의 지시를 해석해서 JSON을 "생성"하는 대신, API 수준에서 스키마를 강제하니까 형식 오류가 거의 발생하지 않게 된 겁니다.

그럼에도 OutputFixingParser와 RetryWithErrorOutputParser가 여전히 유용한 이유가 있습니다:
- Function Calling을 지원하지 않는 모델(오픈소스 모델, 로컬 LLM)을 사용할 때
- `strict: true` 모드에서도 Pydantic validator 수준의 커스텀 검증이 실패할 때
- 스트리밍 환경에서 부분 파싱이 필요할 때([4.4절](./04-스트리밍과-부분-파싱.md) 참조)

## 흔한 오해와 팁

> ⚠️ **흔한 오해**: "OutputFixingParser를 쓰면 모든 파싱 에러가 해결된다"고 생각하기 쉽지만, 실제로는 그렇지 않습니다. OutputFixingParser는 형식 오류(작은따옴표, 누락된 쉼표 등)에는 강하지만, **의미적 오류**(필드 내용이 틀림)나 **구조적 누락**(필수 필드가 아예 없음)에는 한계가 있습니다. 내용까지 고쳐야 한다면 RetryWithErrorOutputParser를 사용하세요.

> 💡 **알고 계셨나요?**: LangChain의 OutputFixingParser 내부에서 수정용 LLM에 보내는 프롬프트를 보면, 실은 아주 간단합니다 — "다음 지시에 맞게 이 텍스트를 고쳐주세요"가 전부죠. 이 프롬프트를 `OutputFixingParser.from_llm(prompt=custom_prompt)`로 커스터마이징하면, 도메인 특화된 수정 지시를 내릴 수 있습니다. 예를 들어 의료 데이터라면 "의학 용어의 정확성을 유지하면서 JSON 형식만 수정하세요"라고 지시할 수 있죠.

> 🔥 **실무 팁**: 프로덕션에서는 **모든 파싱 실패를 반드시 로깅**하세요. 실패한 원본 출력, 사용된 프롬프트, 에러 메시지, 복구 성공 여부를 기록하면 다음과 같은 인사이트를 얻을 수 있습니다: (1) 어떤 유형의 에러가 가장 빈번한지 → 프롬프트 개선에 활용, (2) 복구 성공률이 낮은 패턴 → 스키마 재설계 고려, (3) 특정 모델 버전 업데이트 후 에러율 변화 → 회귀 감지. LangSmith를 사용하면 이런 모니터링을 체계적으로 할 수 있습니다.

> 🔥 **실무 팁**: 에러 복구에 사용하는 LLM은 **원래 호출보다 더 강력한(혹은 최소 동급) 모델**을 사용하세요. GPT-4o-mini가 실패한 출력을 GPT-4o-mini에게 다시 고치라고 하면 같은 실수를 반복할 가능성이 높습니다. 수정용 LLM은 GPT-4o 같은 상위 모델을 쓰거나, 최소한 temperature를 0으로 낮춰서 결정론적 응답을 유도하세요.

## 핵심 정리

| 개념 | 설명 |
|------|------|
| OutputParserException | 파싱 에러 전용 예외 — llm_output, observation, send_to_llm 속성으로 복구에 필요한 정보를 담고 있다 |
| OutputFixingParser | 잘못된 출력 + 에러를 LLM에 보내 형식을 자동 수정. 원본 프롬프트는 참조하지 않음 |
| RetryWithErrorOutputParser | 원본 프롬프트 + 잘못된 출력 + 에러를 모두 LLM에 전달하여 재시도. parse_with_prompt() 사용 |
| with_fallbacks() | LCEL의 폴백 체인으로 여러 복구 전략을 순차적으로 시도 |
| include_raw | with_structured_output(include_raw=True)는 에러를 예외로 던지지 않으므로 with_fallbacks()와 함께 쓸 때 주의 |
| max_retries | OutputFixingParser와 RetryWithErrorOutputParser 모두 최대 재시도 횟수 지정 가능 (기본값: 1) |
| 모니터링 | 파싱 실패율, 복구 성공률, 에러 유형별 통계를 추적하여 프롬프트와 스키마를 지속 개선 |
