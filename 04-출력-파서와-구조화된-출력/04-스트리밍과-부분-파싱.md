# 스트리밍과 부분 파싱

> LLM이 구조화된 데이터를 생성하는 동안 실시간으로 부분 결과를 받아 처리하는 스트리밍 파싱 기법을 학습합니다.

## 개요

이 섹션에서는 LLM이 JSON이나 구조화된 출력을 토큰 단위로 생성하는 과정에서, 완성되지 않은 부분 JSON을 점진적으로 파싱하고 활용하는 방법을 학습합니다. [세션 4.1: 출력 파서 기초](ch04/session_4_1.md)에서 배운 `JsonOutputParser`의 스트리밍 모드를 심화하고, 다양한 파서의 스트리밍 동작 차이를 이해합니다.

**선수 지식**:
- [세션 4.1](ch04/session_4_1.md)에서 배운 `JsonOutputParser`와 `format_instructions` 개념
- [세션 4.2](ch04/session_4_2.md)에서 배운 `PydanticOutputParser`와 Pydantic 모델 정의
- [세션 4.3](ch04/session_4_3.md)에서 배운 `with_structured_output()`의 동작 원리

**학습 목표**:
- `JsonOutputParser`의 스트리밍 모드에서 부분 JSON이 점진적으로 파싱되는 원리를 이해한다
- `diff` 파라미터를 활용하여 JSONPatch 기반의 증분 업데이트를 구현할 수 있다
- 동기(`stream`)와 비동기(`astream`) 스트리밍 패턴의 차이를 이해하고 적절히 선택할 수 있다
- 스트리밍 파싱 결과를 실시간 UI 업데이트에 연동하는 패턴을 구현할 수 있다

## 왜 알아야 할까?

ChatGPT를 사용해본 적 있다면, 글자가 한 글자씩 화면에 나타나는 경험이 익숙하실 겁니다. 사용자는 전체 응답이 완성될 때까지 빈 화면을 보며 기다리는 것보다, 불완전하더라도 실시간으로 결과가 나타나는 것을 훨씬 선호합니다.

그런데 여기서 문제가 생깁니다. 일반 텍스트 스트리밍은 토큰이 올 때마다 그대로 화면에 붙이면 되지만, **JSON과 같은 구조화된 출력**은 이야기가 다릅니다. `{"name": "홍길` 같은 불완전한 JSON 문자열을 `json.loads()`로 파싱하면 어떻게 될까요? 당연히 에러가 발생합니다.

실무에서 이 문제가 특히 중요한 이유는 다음과 같습니다:

- **UX 품질**: 구조화된 응답(상품 목록, 분석 결과 표 등)을 생성할 때 사용자가 5~10초를 기다리면 이탈률이 급증합니다
- **실시간 대시보드**: LLM이 분석 결과를 JSON으로 반환할 때, 각 필드가 채워지는 것을 실시간으로 보여줘야 합니다
- **프로그레시브 렌더링**: 긴 리스트를 생성할 때, 항목이 하나씩 추가되는 것을 보여주는 것이 자연스럽습니다

LangChain은 이 문제를 `parse_partial_json`이라는 내부 유틸리티와 스트리밍 가능한 파서들로 우아하게 해결합니다.

## 핵심 개념

### 개념 1: 부분 JSON 파싱의 원리

> 💡 **비유**: 퍼즐 맞추기를 생각해보세요. 퍼즐 조각이 하나씩 도착할 때, 완성된 부분까지만 보여주는 것과 같습니다. `{"name": "홍길`이 도착하면 아직 `name` 값이 완성되지 않았지만, `{"name": "홍길동", "age": 30`까지 도착하면 `name`은 확정되었으니 이 부분만 먼저 보여줄 수 있죠.

LangChain 내부에는 `parse_partial_json`이라는 유틸리티 함수가 있습니다. 이 함수는 불완전한 JSON 문자열에서 닫히지 않은 괄호(`}`, `]`)나 미완성 문자열(`"`)을 자동으로 보완하여 파싱을 시도합니다.

```python
from langchain_core.utils.json import parse_partial_json

# 완전한 JSON — 당연히 파싱 성공
complete = '{"name": "홍길동", "age": 30}'
print(parse_partial_json(complete))
# 출력: {'name': '홍길동', 'age': 30}

# 불완전한 JSON — 닫는 괄호 없음
partial_1 = '{"name": "홍길동", "age": 30'
print(parse_partial_json(partial_1))
# 출력: {'name': '홍길동', 'age': 30}

# 더 불완전한 JSON — 값이 잘린 상태
partial_2 = '{"name": "홍길'
print(parse_partial_json(partial_2))
# 출력: {'name': '홍길'}

# 배열이 중간에 잘린 경우
partial_3 = '{"items": [1, 2, 3'
print(parse_partial_json(partial_3))
# 출력: {'items': [1, 2, 3]}
```

이 함수가 `JsonOutputParser`의 스트리밍 모드를 지탱하는 핵심입니다. LLM에서 토큰이 도착할 때마다 전체 누적 문자열에 대해 `parse_partial_json`을 호출하고, 이전 결과와 비교하여 변경이 있으면 새 청크를 yield하는 구조입니다.

### 개념 2: JsonOutputParser 스트리밍 모드

> 💡 **비유**: 실시간 스포츠 중계를 떠올려보세요. 경기가 끝날 때까지 기다렸다가 결과를 한 번에 보여주는 게 아니라, 점수가 바뀔 때마다 즉시 업데이트합니다. `JsonOutputParser`의 스트리밍도 마찬가지로, JSON 객체의 각 필드가 확정될 때마다 현재 상태를 전달합니다.

`JsonOutputParser`를 LCEL 체인에 연결하고 `.stream()`을 호출하면, 파서가 자동으로 스트리밍 모드로 동작합니다.

```python
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import JsonOutputParser
from langchain_openai import ChatOpenAI
from pydantic import BaseModel, Field

# 출력 스키마 정의
class MovieReview(BaseModel):
    title: str = Field(description="영화 제목")
    rating: float = Field(description="평점 (1-10)")
    summary: str = Field(description="한 줄 요약")
    pros: list[str] = Field(description="장점 목록")
    cons: list[str] = Field(description="단점 목록")

# 파서 생성
parser = JsonOutputParser(pydantic_object=MovieReview)

# 프롬프트 구성
prompt = ChatPromptTemplate.from_messages([
    ("system", "당신은 영화 평론가입니다. 다음 형식으로 응답하세요:\n{format_instructions}"),
    ("human", "{movie}에 대한 리뷰를 작성해주세요.")
])

# 모델 생성
model = ChatOpenAI(model="gpt-4o", temperature=0.7)

# LCEL 체인 구성
chain = prompt | model | parser

# 스트리밍 실행 — 부분 JSON이 점진적으로 도착
for chunk in chain.stream({
    "movie": "인터스텔라",
    "format_instructions": parser.get_format_instructions()
}):
    print(chunk)
    print("---")
```

실행하면 다음과 같은 출력이 점진적으로 나타납니다:

```
{}
---
{'title': ''}
---
{'title': '인터스텔라'}
---
{'title': '인터스텔라', 'rating': 9.2}
---
{'title': '인터스텔라', 'rating': 9.2, 'summary': ''}
---
{'title': '인터스텔라', 'rating': 9.2, 'summary': '시간과 사랑을 넘나드는 우주 대서사시'}
---
{'title': '인터스텔라', 'rating': 9.2, 'summary': '시간과 사랑을 넘나드는 우주 대서사시', 'pros': ['놀라운 시각 효과']}
---
{'title': '인터스텔라', 'rating': 9.2, 'summary': '시간과 사랑을 넘나드는 우주 대서사시', 'pros': ['놀라운 시각 효과', '한스 짐머의 음악'], 'cons': []}
---
...
```

주목할 점은 **매번 전체 객체가 반환**된다는 것입니다. 이전 값은 보존되면서 새 키-값 쌍이 추가되거나, 기존 값이 업데이트됩니다. 이 동작 방식은 UI 업데이트에 매우 유리한데, 매 청크마다 전체 상태를 받으므로 별도의 상태 관리 없이 바로 화면에 렌더링할 수 있기 때문입니다.

### 개념 3: diff 모드와 JSONPatch

기본 스트리밍 모드에서는 매번 전체 객체를 반환하지만, 객체가 커질수록 이는 비효율적입니다. `diff=True`를 사용하면 이전 상태와 현재 상태의 **차이만** JSONPatch 형식으로 받을 수 있습니다.

> 💡 **비유**: 문서 편집의 "변경 추적(Track Changes)" 기능과 같습니다. 전체 문서를 매번 보내는 대신, "3페이지 2번째 문단에 '중요한'이라는 단어를 추가했습니다"라는 변경 사항만 전달하는 거죠.

```python
from langchain_core.output_parsers import JsonOutputParser
from langchain_core.prompts import ChatPromptTemplate
from langchain_openai import ChatOpenAI
from pydantic import BaseModel, Field

class BookInfo(BaseModel):
    title: str = Field(description="책 제목")
    author: str = Field(description="저자")
    year: int = Field(description="출판 연도")
    genres: list[str] = Field(description="장르 목록")

parser = JsonOutputParser(pydantic_object=BookInfo, diff=True)

prompt = ChatPromptTemplate.from_messages([
    ("system", "다음 형식으로 응답하세요:\n{format_instructions}"),
    ("human", "해리포터 첫 번째 책에 대해 알려주세요.")
])

model = ChatOpenAI(model="gpt-4o", temperature=0)
chain = prompt | model | parser

# diff=True일 때 JSONPatch 형식으로 반환
for patch in chain.stream({
    "format_instructions": parser.get_format_instructions()
}):
    print(patch)
    print("---")
```

출력 예시:

```
[{'op': 'replace', 'path': '', 'value': {}}]
---
[{'op': 'add', 'path': '/title', 'value': ''}]
---
[{'op': 'replace', 'path': '/title', 'value': '해리 포터와 마법사의 돌'}]
---
[{'op': 'add', 'path': '/author', 'value': 'J.K. 롤링'}]
---
[{'op': 'add', 'path': '/year', 'value': 1997}]
---
[{'op': 'add', 'path': '/genres', 'value': []}]
---
[{'op': 'add', 'path': '/genres/0', 'value': '판타지'}]
---
[{'op': 'add', 'path': '/genres/1', 'value': '모험'}]
---
```

각 패치에는 `op`(연산 종류), `path`(대상 경로), `value`(값)가 포함됩니다. 이 형식은 [RFC 6902 JSON Patch](https://datatracker.ietf.org/doc/html/rfc6902) 표준을 따르며, 프론트엔드에서 `jsonpatch` 라이브러리를 사용하여 효율적으로 상태를 업데이트할 수 있습니다.

### 개념 4: 비동기 스트리밍 — astream

웹 서버나 비동기 애플리케이션에서는 `stream()` 대신 `astream()`을 사용합니다. 문법은 거의 동일하지만, `async for`를 사용한다는 점이 다릅니다.

```python
import asyncio
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import JsonOutputParser
from langchain_openai import ChatOpenAI
from pydantic import BaseModel, Field

class TaskPlan(BaseModel):
    goal: str = Field(description="목표")
    steps: list[str] = Field(description="단계별 계획")
    estimated_hours: float = Field(description="예상 소요 시간")

parser = JsonOutputParser(pydantic_object=TaskPlan)

prompt = ChatPromptTemplate.from_messages([
    ("system", "다음 형식으로 응답하세요:\n{format_instructions}"),
    ("human", "{task}를 완료하기 위한 계획을 세워주세요.")
])

model = ChatOpenAI(model="gpt-4o", temperature=0.7)
chain = prompt | model | parser

async def stream_plan():
    """비동기 스트리밍으로 계획을 실시간 수신"""
    async for chunk in chain.astream({
        "task": "Python으로 웹 스크래퍼 만들기",
        "format_instructions": parser.get_format_instructions()
    }):
        # chunk는 부분 파싱된 딕셔너리
        if "goal" in chunk:
            print(f"🎯 목표: {chunk['goal']}")
        if "steps" in chunk:
            print(f"📋 단계: {len(chunk['steps'])}개 확인")
            for i, step in enumerate(chunk["steps"], 1):
                print(f"   {i}. {step}")
        if "estimated_hours" in chunk:
            print(f"⏱️ 예상 시간: {chunk['estimated_hours']}시간")
        print()

# 비동기 실행
asyncio.run(stream_plan())
```

`astream()`은 FastAPI, Starlette 같은 비동기 웹 프레임워크와 자연스럽게 통합됩니다. 동기 코드에서는 `stream()`, 비동기 환경에서는 `astream()`을 사용하면 됩니다.

### 개념 5: with_structured_output과 스트리밍의 관계

[세션 4.3](ch04/session_4_3.md)에서 배운 `with_structured_output()`은 강력하지만, 스트리밍과의 조합에서는 주의가 필요합니다. 현재 대부분의 프로바이더에서 `with_structured_output()`은 **스트리밍을 지원하지 않습니다**. Function Calling/Tool Use 기반의 구조화 출력은 전체 응답이 완성된 후에야 파싱이 가능하기 때문입니다.

```python
from langchain_openai import ChatOpenAI
from pydantic import BaseModel, Field

class SimpleOutput(BaseModel):
    answer: str = Field(description="답변")
    confidence: float = Field(description="신뢰도")

# with_structured_output — 스트리밍이 제한적
structured_llm = ChatOpenAI(model="gpt-4o").with_structured_output(SimpleOutput)

# stream()을 호출해도 단일 청크로 전체 결과가 도착
for chunk in structured_llm.stream("파이썬의 장점을 하나만 알려주세요"):
    print(type(chunk), chunk)
    # 전체 SimpleOutput 객체가 한 번에 반환됨

# ✅ 스트리밍이 필요하면 JsonOutputParser를 사용
# JsonOutputParser + stream() → 부분 JSON이 점진적으로 도착
```

> ⚠️ **흔한 오해**: `with_structured_output()` + `stream()`을 쓰면 구조화된 데이터가 스트리밍될 거라고 기대하는 분이 많습니다. 실제로는 전체 응답이 완성된 뒤 한 번에 전달되는 경우가 대부분입니다. **실시간 부분 파싱이 필요하다면 `JsonOutputParser`를 사용하세요.**

따라서 구조화된 출력의 스트리밍이 필요한 경우, 선택 전략은 다음과 같습니다:

| 요구사항 | 추천 방식 |
|---------|----------|
| 부분 파싱 + 실시간 UI | `JsonOutputParser` + `stream()` |
| 타입 안전 + 실시간 불필요 | `with_structured_output()` |
| 비동기 + 부분 파싱 | `JsonOutputParser` + `astream()` |
| 이벤트 기반 세밀한 제어 | `astream_events()` |

## 실습: 직접 해보기

실시간으로 상품 분석 결과가 채워지는 시나리오를 구현해봅시다. 사용자가 상품명을 입력하면, LLM이 분석 결과를 JSON으로 생성하는 동안 각 필드가 점진적으로 표시됩니다.

```python
"""
실습: 실시간 상품 분석 스트리밍
- JsonOutputParser의 스트리밍 모드로 부분 JSON을 점진적으로 파싱
- 각 필드가 추가될 때마다 화면을 업데이트
"""
import asyncio
import os
from dotenv import load_dotenv
from pydantic import BaseModel, Field
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import JsonOutputParser
from langchain_openai import ChatOpenAI

load_dotenv()

# --- 1단계: 출력 스키마 정의 ---
class ProductAnalysis(BaseModel):
    """상품 분석 결과 스키마"""
    product_name: str = Field(description="상품명")
    category: str = Field(description="상품 카테고리")
    target_audience: str = Field(description="타겟 고객층")
    strengths: list[str] = Field(description="상품 강점 목록")
    weaknesses: list[str] = Field(description="상품 약점 목록")
    marketing_angle: str = Field(description="추천 마케팅 방향")
    price_range: str = Field(description="추천 가격대")
    overall_score: float = Field(description="종합 점수 (1-10)")

# --- 2단계: 파서와 체인 구성 ---
parser = JsonOutputParser(pydantic_object=ProductAnalysis)

prompt = ChatPromptTemplate.from_messages([
    ("system", 
     "당신은 상품 분석 전문가입니다. 아래 형식에 맞춰 분석 결과를 JSON으로 응답하세요.\n"
     "{format_instructions}"),
    ("human", "다음 상품을 분석해주세요: {product}")
])

model = ChatOpenAI(model="gpt-4o", temperature=0.7)
chain = prompt | model | parser

# --- 3단계: 실시간 스트리밍 표시 함수 ---
def display_streaming_analysis(product: str) -> dict:
    """동기 스트리밍으로 상품 분석 결과를 실시간 표시"""
    print(f"\n{'='*50}")
    print(f"📦 상품 분석 중: {product}")
    print(f"{'='*50}\n")
    
    # 이전 청크의 키 수를 추적하여 새 필드 감지
    prev_keys = set()
    result = {}
    
    for chunk in chain.stream({
        "product": product,
        "format_instructions": parser.get_format_instructions()
    }):
        result = chunk  # 매번 전체 객체가 옴
        new_keys = set(chunk.keys()) - prev_keys
        
        for key in new_keys:
            value = chunk[key]
            # 필드별 포매팅
            if isinstance(value, list):
                print(f"  ✅ {key}:")
                for item in value:
                    print(f"     • {item}")
            elif isinstance(value, float):
                print(f"  ✅ {key}: {'⭐' * int(value)} ({value})")
            else:
                print(f"  ✅ {key}: {value}")
        
        # 리스트 필드는 항목이 추가될 수 있으므로 업데이트 감지
        for key in prev_keys:
            if key in chunk and isinstance(chunk[key], list):
                prev_len = len(result.get(key, []))  
                curr_len = len(chunk[key])
                if curr_len > prev_len:
                    for item in chunk[key][prev_len:]:
                        print(f"     • {item}")
        
        prev_keys = set(chunk.keys())
    
    print(f"\n{'='*50}")
    print("✨ 분석 완료!")
    print(f"{'='*50}")
    return result

# --- 4단계: 비동기 스트리밍 버전 (웹 서버용) ---
async def stream_analysis_async(product: str):
    """
    비동기 스트리밍 — FastAPI/웹소켓 연동에 적합
    yield하는 각 청크를 SSE나 WebSocket으로 클라이언트에 전달 가능
    """
    async for chunk in chain.astream({
        "product": product,
        "format_instructions": parser.get_format_instructions()
    }):
        # 실제 웹 앱에서는 이 chunk를 SSE/WebSocket으로 전송
        yield chunk

# --- 5단계: diff 모드 데모 ---
def demo_diff_mode(product: str):
    """diff 모드로 JSONPatch 스트리밍 시연"""
    diff_parser = JsonOutputParser(pydantic_object=ProductAnalysis, diff=True)
    diff_chain = prompt | model | diff_parser
    
    print(f"\n📊 [diff 모드] JSONPatch 스트리밍:")
    print(f"{'-'*40}")
    
    for patches in diff_chain.stream({
        "product": product,
        "format_instructions": diff_parser.get_format_instructions()
    }):
        for patch in patches:
            op = patch.get("op", "?")
            path = patch.get("path", "")
            value = patch.get("value", "")
            print(f"  [{op:>7}] {path} → {value}")

# --- 실행 ---
if __name__ == "__main__":
    # 기본 스트리밍 실행
    final_result = display_streaming_analysis("에어팟 프로 3")
    print(f"\n최종 결과 타입: {type(final_result)}")
    print(f"최종 결과 키: {list(final_result.keys())}")
    
    # diff 모드 시연
    demo_diff_mode("에어팟 프로 3")
```

이 코드를 실행하면 분석 결과가 한 번에 쏟아지는 것이 아니라, 각 필드(`product_name` → `category` → `strengths` → ...)가 순차적으로 화면에 나타나는 것을 확인할 수 있습니다.

### 보너스: FastAPI SSE(Server-Sent Events) 연동

실무에서 가장 흔한 패턴인 FastAPI + SSE 연동 예제입니다:

```python
"""
FastAPI + SSE를 활용한 실시간 구조화 출력 스트리밍
클라이언트(브라우저)에서 EventSource로 연결하여 실시간 업데이트를 받을 수 있음
"""
import json
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import JsonOutputParser
from langchain_openai import ChatOpenAI
from pydantic import BaseModel, Field

app = FastAPI()

class AnalysisResult(BaseModel):
    summary: str = Field(description="요약")
    key_points: list[str] = Field(description="핵심 포인트")
    recommendation: str = Field(description="추천 사항")

parser = JsonOutputParser(pydantic_object=AnalysisResult)
prompt = ChatPromptTemplate.from_messages([
    ("system", "다음 형식으로 응답하세요:\n{format_instructions}"),
    ("human", "{query}")
])
model = ChatOpenAI(model="gpt-4o", temperature=0.7)
chain = prompt | model | parser

@app.get("/analyze")
async def analyze(query: str):
    """SSE 엔드포인트 — 부분 JSON을 실시간 스트리밍"""
    async def event_generator():
        async for chunk in chain.astream({
            "query": query,
            "format_instructions": parser.get_format_instructions()
        }):
            # SSE 형식: data: {json}\n\n
            yield f"data: {json.dumps(chunk, ensure_ascii=False)}\n\n"
        yield "data: [DONE]\n\n"
    
    return StreamingResponse(
        event_generator(),
        media_type="text/event-stream"
    )

# 클라이언트 사용 예시 (JavaScript):
# const evtSource = new EventSource("/analyze?query=AI 트렌드");
# evtSource.onmessage = (e) => {
#     if (e.data === "[DONE]") { evtSource.close(); return; }
#     const partial = JSON.parse(e.data);
#     updateUI(partial);  // 부분 JSON으로 UI 업데이트
# };
```

## 더 깊이 알아보기

### 부분 JSON 파싱의 기술적 도전

불완전한 JSON을 파싱하는 것은 사실 쉽지 않은 문제입니다. 표준 JSON 파서(Python의 `json` 모듈)는 완전한 JSON만 처리할 수 있도록 설계되었기 때문입니다. LangChain의 `parse_partial_json` 함수는 이 문제를 해결하기 위해 닫히지 않은 문자열, 객체, 배열을 자동으로 보완하는 전략을 사용합니다.

흥미로운 점은 이 접근 방식이 성능 문제를 안고 있다는 것입니다. 토큰이 하나 추가될 때마다 **전체 누적 JSON 문자열을 처음부터 다시 파싱**합니다. 작은 JSON에서는 문제가 없지만, 큰 JSON(수천 토큰)에서는 O(n²) 복잡도가 됩니다. LangChain 커뮤니티에서는 `ijson` 같은 이벤트 기반 JSON 파서를 사용하여 이 성능 문제를 개선하자는 [논의](https://github.com/langchain-ai/langchain/discussions/27911)가 진행되기도 했습니다.

### JSONPatch — 효율적인 증분 업데이트의 역사

diff 모드에서 사용되는 JSONPatch는 [RFC 6902](https://datatracker.ietf.org/doc/html/rfc6902)로 표준화된 형식입니다. 2013년에 IETF에서 표준으로 채택되었으며, "JSON 문서에 대한 수정 사항을 표현하는 형식"으로 정의됩니다. 원래는 REST API에서 PATCH 요청의 본문으로 사용하기 위해 만들어졌는데, LLM 스트리밍이라는 전혀 다른 맥락에서 재활용되고 있는 것이 흥미롭습니다. 기술이 원래 의도와 다른 곳에서 빛을 발하는 좋은 사례죠.

### 스트리밍의 기원 — Server-Sent Events

LLM 스트리밍에서 널리 사용되는 SSE(Server-Sent Events)는 사실 2004년경 Opera 브라우저 개발자인 Ian Hickson이 제안한 기술입니다. HTML5 명세의 일부로 2006년에 처음 구현되었으며, 원래는 주식 시세나 뉴스 피드를 실시간으로 전달하기 위한 용도였습니다. WebSocket보다 단순하고 HTTP 인프라와 호환성이 좋다는 장점 때문에, OpenAI가 ChatGPT의 스트리밍 API에 SSE를 채택하면서 LLM 생태계의 사실상 표준이 되었습니다.

## 흔한 오해와 팁

> ⚠️ **흔한 오해**: "스트리밍 모드에서 각 청크는 새로 추가된 부분만 포함한다" — 아닙니다. `diff=False`(기본값)일 때 `JsonOutputParser`의 각 청크는 **지금까지 파싱된 전체 객체**를 포함합니다. 이전 필드의 값도 모두 포함되어 있으므로, 매 청크를 독립적으로 사용할 수 있습니다. 차이만 받으려면 명시적으로 `diff=True`를 설정해야 합니다.

> 💡 **알고 계셨나요?**: `PydanticOutputParser`도 내부적으로 `JsonOutputParser`를 상속합니다. 따라서 스트리밍 모드에서 부분 **딕셔너리**를 yield합니다(Pydantic 모델 인스턴스가 아닌!). 불완전한 JSON으로는 Pydantic 검증을 통과할 수 없기 때문입니다. 최종 청크에서만 완전한 JSON이 되며, 이때 `parser.invoke()`를 호출하면 Pydantic 모델로 변환됩니다.

> 🔥 **실무 팁**: 큰 JSON을 스트리밍할 때 성능이 걱정된다면, LLM에게 최상위 키의 순서를 지정하세요. 예를 들어 "가장 중요한 필드인 `title`, `status`를 먼저, 긴 텍스트인 `description`을 마지막에 배치하라"고 프롬프트에 명시하면, 사용자가 가장 궁금해하는 정보를 먼저 볼 수 있습니다. 프롬프트에 키 순서를 지정하면 대부분의 LLM이 이를 따릅니다.

> 🔥 **실무 팁**: 스트리밍 체인을 디버깅할 때는 `astream_events()`를 사용하세요. 체인 내부의 각 컴포넌트(프롬프트, 모델, 파서)에서 발생하는 이벤트를 개별적으로 추적할 수 있어, 어디서 병목이 생기는지 정확하게 파악할 수 있습니다.

## 핵심 정리

| 개념 | 설명 |
|------|------|
| `parse_partial_json` | 불완전한 JSON 문자열을 파싱하는 LangChain 내부 유틸리티. 닫히지 않은 괄호와 문자열을 자동 보완 |
| `JsonOutputParser` + `stream()` | 기본 모드: 매 청크마다 **지금까지 파싱된 전체 객체**를 반환 |
| `diff=True` | JSONPatch(RFC 6902) 형식으로 이전 상태와의 **차이만** 반환 |
| `astream()` | 비동기 스트리밍. FastAPI 등 비동기 웹 프레임워크에서 사용 |
| `with_structured_output` + 스트리밍 | 대부분의 프로바이더에서 실시간 부분 파싱 불가. 전체 완성 후 반환 |
| SSE (Server-Sent Events) | LLM 스트리밍 결과를 브라우저에 전달하는 사실상 표준 프로토콜 |
| `astream_events()` | 체인 내부의 세밀한 이벤트를 추적하는 고급 스트리밍 API |
